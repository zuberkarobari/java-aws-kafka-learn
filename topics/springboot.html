🌱 Spring Boot

Spring Boot is an opinionated, convention-over-configuration framework built on top of the Spring ecosystem. It lets you create stand-alone, production-grade Spring applications with minimal setup — no XML, no separate servlet container required. You write business logic; Spring Boot handles the boilerplate.

What Is Spring Boot and Why Use It?

Plain Spring gives you a powerful, modular toolkit for building Java applications, but historically required a significant amount of configuration. Spring Boot solves this with three big ideas:

  • Auto-configuration — detects libraries on your classpath and configures them automatically.
  • Opinionated defaults — sensible choices are made for you; you override only what you need.
  • Embedded server — packages Tomcat, Jetty, or Undertow directly inside your JAR so you run with java -jar app.jar.

The result: a new REST service can be running in under five minutes, with no separate server installation and no XML configuration files.

Spring vs Spring Boot

Understanding the difference helps you know what Spring Boot is actually doing under the hood.

Aspect Spring Framework Spring Boot
Configuration Manual XML or Java @Configuration Auto-configured based on classpath
Server Deploy WAR to external Tomcat/Jetty Embedded Tomcat/Jetty/Undertow inside JAR
Startup Needs web.xml or WebApplicationInitializer Single main() method
Dependencies Manually pick compatible versions Starters manage compatible dependency sets
Production features Build yourself Actuator health checks, metrics out of the box
Learning curve Steeper — many moving parts Gentler — sensible defaults everywhere

Spring Boot is Spring — it does not replace the core IoC container, AOP, or any other Spring module. It simply provides a faster, opinionated way to wire them all together.

Your First Spring Boot Application

The canonical entry point for any Spring Boot app is a single class annotated with @SpringBootApplication and a standard Java main() method. 

package com.example.demo;

import org.springframework.boot.SpringApplication;
import org.springframework.boot.autoconfigure.SpringBootApplication;

@SpringBootApplication   // ← does the heavy lifting (see below)
public class DemoApplication {

  public static void main(String[] args) {
    SpringApplication.run(DemoApplication.class, args);
  }
}

That is a complete, runnable Spring Boot application. When you run it, Spring Boot:

  1. Starts the embedded Tomcat server (default port 8080).
  2. Scans the package and sub-packages for Spring-managed components.
  3. Applies all relevant auto-configurations.

Key Annotations Explained

Spring Boot is annotation-driven. Learning these six annotations will take you very far.

@SpringBootApplication

A convenience meta-annotation that combines three annotations:

  • @SpringBootConfiguration — marks the class as a source of @Bean definitions.
  • @EnableAutoConfiguration — tells Spring Boot to start adding beans based on classpath settings.
  • @ComponentScan — enables scanning of the current package and sub-packages for components.
// These two declarations are equivalent:

// ① Short form (recommended)
@SpringBootApplication
public class MyApp { ... }

// ② Explicit form (shows what's happening)
@SpringBootConfiguration
@EnableAutoConfiguration
@ComponentScan(basePackages = "com.example")
public class MyApp { ... }

@Component

The generic stereotype annotation. Any class annotated with @Component is picked up by component scanning and registered as a Spring bean. The more specific annotations below are all specialisations of @Component. 

import org.springframework.stereotype.Component;

@Component
public class EmailValidator {

  public boolean isValid(String email) {
    return email != null && email.contains("@");
  }
}

@Service

Marks a class as belonging to the service layer — where business logic lives. Functionally identical to @Component, but communicates intent to other developers and enables AOP pointcuts that target the service layer specifically. 

import org.springframework.stereotype.Service;

@Service
public class ProductService {

  private final ProductRepository productRepository;

  // Constructor injection (preferred — see Dependency Injection section)
  public ProductService(ProductRepository productRepository) {
    this.productRepository = productRepository;
  }

  public Product findById(Long id) {
    return productRepository.findById(id)
        .orElseThrow(() -> new RuntimeException("Product not found: " + id));
  }
}

@Repository

Marks a class as a data access object (DAO). Beyond marking intent, Spring translates persistence-layer exceptions (e.g. SQLException) into Spring's unified DataAccessException hierarchy automatically when this annotation is present. 

import org.springframework.stereotype.Repository;
import java.util.HashMap;
import java.util.Map;
import java.util.Optional;

@Repository
public class InMemoryProductRepository {

  private final Map<Long, Product> store = new HashMap<>();
  private long nextId = 1L;

  public Product save(Product product) {
    product.setId(nextId++);
    store.put(product.getId(), product);
    return product;
  }

  public Optional<Product> findById(Long id) {
    return Optional.ofNullable(store.get(id));
  }
}

@RestController

A convenience annotation that combines @Controller and @ResponseBody. Every method return value in a @RestController class is automatically serialized to JSON (via Jackson) and written directly into the HTTP response body. 

import org.springframework.web.bind.annotation.GetMapping;
import org.springframework.web.bind.annotation.RestController;

@RestController
public class GreetingController {

  @GetMapping("/hello")
  public String hello() {
    return "Hello, Spring Boot!";  // serialized as: Hello, Spring Boot!
  }

  @GetMapping("/status")
  public Map<String, String> status() {
    return Map.of("status", "ok", "version", "1.0.0");
    // serialized as: {"status":"ok","version":"1.0.0"}
  }
}

@Autowired

Tells Spring to inject a dependency automatically. Can be placed on constructors, fields, or setter methods. Field injection is discouraged — prefer constructor injection (shown in the Dependency Injection section below). 

import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.stereotype.Service;

@Service
public class NotificationService {

  // ❌ Field injection — avoid (hard to test, hides dependencies)
  @Autowired
  private EmailSender emailSender;

  // ✅ Constructor injection — preferred (see next section)
}

Dependency Injection and the IoC Container

Inversion of Control (IoC) means you don't create objects with new — instead, you declare what you need and Spring's IoC container creates and wires dependencies for you. Dependency Injection (DI) is the mechanism Spring uses to fulfil those declarations.

The Three Styles of Injection

// ① Constructor injection — RECOMMENDED
// Spring injects through the constructor; no @Autowired needed when
// there is exactly one constructor (Spring 4.3+).
@Service
public class OrderService {

  private final PaymentService paymentService;
  private final InventoryService inventoryService;

  public OrderService(PaymentService paymentService,
                      InventoryService inventoryService) {
    this.paymentService = paymentService;
    this.inventoryService = inventoryService;
  }
}

// ② Setter injection — optional/nullable dependencies
@Service
public class ReportService {

  private EmailSender emailSender;

  @Autowired(required = false)
  public void setEmailSender(EmailSender emailSender) {
    this.emailSender = emailSender;
  }
}

// ③ Field injection — easy to write, hard to test, avoid in production code
@Service
public class BadPracticeService {

  @Autowired
  private SomeDependency someDependency;  // ← not recommended
}

How the IoC Container Works

Spring's ApplicationContext is the IoC container. At startup it:

  1. Scans for classes annotated with @Component, @Service, @Repository, @Controller, etc.
  2. Reads @Bean factory methods in @Configuration classes.
  3. Builds a bean graph, resolving all dependencies.
  4. Instantiates beans in the correct order and injects them.
// You can also register beans manually using @Configuration + @Bean
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;

@Configuration
public class AppConfig {

  // Spring calls this method and stores the returned object as a bean
  @Bean
  public ObjectMapper objectMapper() {
    ObjectMapper mapper = new ObjectMapper();
    mapper.disable(SerializationFeature.WRITE_DATES_AS_TIMESTAMPS);
    return mapper;
  }
}

Auto-Configuration Explained

Auto-configuration is the mechanism behind Spring Boot's "just works" behaviour. When @EnableAutoConfiguration is active, Spring Boot reads a list of candidate configuration classes from META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports (Spring Boot 3.x) inside each Spring Boot starter JAR.

Each auto-configuration class is guarded by @Conditional annotations, so it only activates when certain conditions are met: 

// Simplified view of what DataSourceAutoConfiguration looks like internally
@AutoConfiguration
@ConditionalOnClass({ DataSource.class, EmbeddedDatabaseType.class })
@ConditionalOnMissingBean(type = "io.r2dbc.spi.ConnectionFactory")
@EnableConfigurationProperties(DataSourceProperties.class)
public class DataSourceAutoConfiguration {

  @Bean
  @ConditionalOnMissingBean
  public DataSource dataSource(DataSourceProperties properties) {
    // Creates a DataSource bean only if you haven't defined your own
    return properties.initializeDataSourceBuilder().build();
  }
}

Key @Conditional annotations you'll encounter:

  • @ConditionalOnClass — activates only if a class is on the classpath.
  • @ConditionalOnMissingBean — activates only if you haven't defined your own bean.
  • @ConditionalOnProperty — activates based on an application.properties value.
  • @ConditionalOnWebApplication — activates only in a web context.

This is why adding spring-boot-starter-data-jpa to your pom.xml is enough to get a fully configured JPA environment — the starter JAR brings the right classes onto the classpath and triggers the relevant auto-configuration.

Spring Boot Starters

Starters are curated dependency descriptors — a single Maven/Gradle dependency that brings in everything you need for a given use case, with guaranteed-compatible versions.

Building a REST API

Let's build a complete CRUD REST API for a Product resource to see all the pieces working together.

1. The Model

package com.example.demo.model;

public class Product {

  private Long id;
  private String name;
  private double price;

  // Constructors
  public Product() {}

  public Product(Long id, String name, double price) {
    this.id = id;
    this.name = name;
    this.price = price;
  }

  // Getters and setters
  public Long getId() { return id; }
  public void setId(Long id) { this.id = id; }

  public String getName() { return name; }
  public void setName(String name) { this.name = name; }

  public double getPrice() { return price; }
  public void setPrice(double price) { this.price = price; }
}

2. The Service Layer

package com.example.demo.service;

import com.example.demo.model.Product;
import org.springframework.stereotype.Service;

import java.util.*;
import java.util.concurrent.atomic.AtomicLong;

@Service
public class ProductService {

  // In-memory store for demo purposes
  private final Map<Long, Product> store = new HashMap<>();
  private final AtomicLong idCounter = new AtomicLong(1);

  public List<Product> findAll() {
    return new ArrayList<>(store.values());
  }

  public Optional<Product> findById(Long id) {
    return Optional.ofNullable(store.get(id));
  }

  public Product create(Product product) {
    product.setId(idCounter.getAndIncrement());
    store.put(product.getId(), product);
    return product;
  }

  public Optional<Product> update(Long id, Product updated) {
    if (!store.containsKey(id)) return Optional.empty();
    updated.setId(id);
    store.put(id, updated);
    return Optional.of(updated);
  }

  public boolean delete(Long id) {
    return store.remove(id) != null;
  }
}

3. The REST Controller

This is where HTTP verbs meet Java methods. The mapping annotations tell Spring which HTTP method + URL path should invoke which method. 

package com.example.demo.controller;

import com.example.demo.model.Product;
import com.example.demo.service.ProductService;
import org.springframework.http.HttpStatus;
import org.springframework.http.ResponseEntity;
import org.springframework.web.bind.annotation.*;

import java.util.List;

@RestController
@RequestMapping("/api/products")   // base path for all methods in this controller
public class ProductController {

  private final ProductService productService;

  // Constructor injection — no @Autowired needed (single constructor)
  public ProductController(ProductService productService) {
    this.productService = productService;
  }

  // GET /api/products
  @GetMapping
  public List<Product> getAllProducts() {
    return productService.findAll();
  }

  // GET /api/products/{id}
  // @PathVariable binds the {id} segment of the URL to the method parameter
  @GetMapping("/{id}")
  public ResponseEntity<Product> getProduct(@PathVariable Long id) {
    return productService.findById(id)
        .map(ResponseEntity::ok)                              // 200 OK
        .orElse(ResponseEntity.notFound().build());           // 404 Not Found
  }

  // POST /api/products
  // @RequestBody deserializes the JSON request body into a Product object
  @PostMapping
  public ResponseEntity<Product> createProduct(@RequestBody Product product) {
    Product created = productService.create(product);
    return ResponseEntity.status(HttpStatus.CREATED).body(created);  // 201 Created
  }

  // PUT /api/products/{id}
  @PutMapping("/{id}")
  public ResponseEntity<Product> updateProduct(
      @PathVariable Long id,
      @RequestBody Product product) {

    return productService.update(id, product)
        .map(ResponseEntity::ok)
        .orElse(ResponseEntity.notFound().build());
  }

  // DELETE /api/products/{id}
  @DeleteMapping("/{id}")
  public ResponseEntity<Void> deleteProduct(@PathVariable Long id) {
    if (productService.delete(id)) {
      return ResponseEntity.noContent().build();    // 204 No Content
    }
    return ResponseEntity.notFound().build();       // 404 Not Found
  }
}

Reading Configuration in Your Code

import org.springframework.beans.factory.annotation.Value;
import org.springframework.stereotype.Component;

// ① @Value — inject a single property
@Component
public class AppInfo {

  @Value("${app.welcome-message}")
  private String welcomeMessage;

  @Value("${app.max-page-size:20}")  // 20 is the default if property is missing
  private int maxPageSize;

  public String getWelcomeMessage() { return welcomeMessage; }
  public int getMaxPageSize() { return maxPageSize; }
}

import org.springframework.boot.context.properties.ConfigurationProperties;
import org.springframework.stereotype.Component;

// ② @ConfigurationProperties — bind a whole group of properties to a POJO
// Binds all properties that start with "app."
@Component
@ConfigurationProperties(prefix = "app")
public class AppProperties {

  private int maxPageSize;
  private String welcomeMessage;

  // Spring needs getters AND setters to bind the values
  public int getMaxPageSize() { return maxPageSize; }
  public void setMaxPageSize(int maxPageSize) { this.maxPageSize = maxPageSize; }

  public String getWelcomeMessage() { return welcomeMessage; }
  public void setWelcomeMessage(String welcomeMessage) {
    this.welcomeMessage = welcomeMessage;
  }
}

Profiles

Use Spring profiles to maintain different configurations for different environments without changing code. 

# application-dev.properties  (active when spring.profiles.active=dev)
spring.datasource.url=jdbc:h2:mem:testdb
logging.level.com.example=DEBUG

# application-prod.properties  (active when spring.profiles.active=prod)
spring.datasource.url=jdbc:postgresql://prod-server:5432/mydb
logging.level.com.example=WARN

// Activate a profile programmatically or via CLI:
//   java -jar app.jar --spring.profiles.active=prod
// or in application.properties:
//   spring.profiles.active=dev

// Beans can also be profile-scoped:
import org.springframework.context.annotation.Profile;

@Service
@Profile("dev")
public class MockEmailSender implements EmailSender {
  public void send(String to, String body) {
    System.out.println("[DEV] Email to " + to + ": " + body);
  }
}

@Service
@Profile("prod")
public class SmtpEmailSender implements EmailSender {
  public void send(String to, String body) {
    // real SMTP implementation
  }
}

Project Structure Convention

A well-organized Spring Boot project follows a layered package structure that mirrors the separation of concerns: 

// Recommended package layout

├── DemoApplication.java          // @SpringBootApplication entry point
│
├── controller/                   // @RestController — HTTP layer
│   └── ProductController.java
│
├── service/                      // @Service — business logic
│   └── ProductService.java
│
├── repository/                   // @Repository — data access
│   └── ProductRepository.java
│
├── model/                        // Plain POJOs / JPA entities

│
├── dto/                          // Data Transfer Objects (request/response shapes)

│   └── ProductResponse.java
│
└── config/                       // @Configuration classes
    └── AppConfig.java

A Complete Minimal Example

Here's a self-contained Spring Boot application — a single file that demonstrates all the key concepts together. In a real project these classes would live in separate files. 

package com.example.hello;

import org.springframework.boot.SpringApplication;
import org.springframework.boot.autoconfigure.SpringBootApplication;
import org.springframework.stereotype.Service;
import org.springframework.web.bind.annotation.*;

import java.util.List;
import java.util.concurrent.CopyOnWriteArrayList;
import java.util.concurrent.atomic.AtomicLong;

// ─── Entry Point ────────────────────────────────────────────────────────────
@SpringBootApplication
public class HelloApp {
  public static void main(String[] args) {
    SpringApplication.run(HelloApp.class, args);
  }
}

// ─── Model ──────────────────────────────────────────────────────────────────
record Note(Long id, String text) {}

// ─── Service ────────────────────────────────────────────────────────────────
@Service
class NoteService {

  private final List<Note> notes = new CopyOnWriteArrayList<>();
  private final AtomicLong ids = new AtomicLong(1);

  List<Note> getAll() { return notes; }

  Note add(String text) {
    Note note = new Note(ids.getAndIncrement(), text);
    notes.add(note);
    return note;
  }
}

// ─── Controller ─────────────────────────────────────────────────────────────
@RestController
@RequestMapping("/notes")
class NoteController {

  private final NoteService noteService;

  NoteController(NoteService noteService) {
    this.noteService = noteService;
  }

  @GetMapping
  List<Note> list() {
    return noteService.getAll();
  }

  @PostMapping
  Note create(@RequestBody String text) {
    return noteService.add(text);
  }
}

Run this app and test it with curl:

# Create a note
curl -X POST http://localhost:8080/notes \
     -H "Content-Type: application/json" \
     -d '"Learn Spring Boot"'

# Response: {"id":1,"text":"Learn Spring Boot"}

# Get all notes
curl http://localhost:8080/notes

# Response: [{"id":1,"text":"Learn Spring Boot"}]

Key Takeaways

  • Spring Boot = Spring Framework + auto-configuration + embedded server + starters.
  • @SpringBootApplication activates component scanning, auto-configuration, and bean definition.
  • Prefer constructor injection over field injection — it's testable and explicit.
  • Auto-configuration is conditional: it backs off if you define your own bean.
  • Starters manage compatible dependency sets so you don't version-manage individually.
  • @RestController + mapping annotations (@GetMapping, @PostMapping, etc.) are all you need to build a REST API.
  • application.properties / application.yml is your single configuration file; profiles let you swap config per environment.
