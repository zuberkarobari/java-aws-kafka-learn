Collections Framework Hierarchy
The Java Collections Framework (JCF) provides an architecture to store and manipulate groups of objects. It reduces programming effort by providing high-performance, thoroughly tested data structures.

Core Methods Guide
To master the Collections framework, you must know the exact methods defined by the primary interfaces. Because interfaces enforce contracts, these methods are guaranteed to exist across all their implementing classes.
Collection<E> Methods
Applies to all Lists, Sets, and Queues.

boolean add(E e) - Adds element.
boolean remove(Object o) - Removes instance.
boolean contains(Object o) - Checks existence.
int size() - Number of elements.
boolean isEmpty() - True if size is 0.
void clear() - Removes all elements.
Iterator<E> iterator() - Returns an iterator.
Object[] toArray() - Converts to a raw array.
List<E> Specific Methods
Lists are indexed, unlocking positional methods.

E get(int index) - Retrieves at index.
E set(int index, E element) - Replaces at index.
void add(int index, E element) - Inserts at index.
E remove(int index) - Removes at index.
int indexOf(Object o) - Finds first index of obj.
List<E> subList(int from, int to) - Returns a view.
Map<K, V> Methods
Maps handle Key-Value pairs.

V put(K key, V value) - Inserts/updates mapping.
V get(Object key) - Retrieves by key.
V remove(Object key) - Deletes mapping.
boolean containsKey(Object key) - Checks key.
boolean containsValue(Object val) - Checks val.
Set<K> keySet() - Returns all keys.
Collection<V> values() - Returns all values.
Java 8+ Map Additions
putIfAbsent(K key, V value) - Only inserts if missing or null.
getOrDefault(Object key, V default) - Safe retrieval.
computeIfAbsent(...) - Builds value if key missing.
forEach(BiConsumer) - Lambda iteration.
The List Interface
An ordered collection (also known as a sequence). The user of this interface has precise control over where in the list each element is inserted. Allows duplicate elements.
ArrayList
Backed by a dynamic resizable array.

Access get(i): O(1) - Extremely fast.
Insertion add(e): O(1) amortized.
Middle Insertion add(i, e): O(N) - Slow, requires shifting elements to the right.

Verdict: Best for 95% of use cases where you mostly read data and add to the end of the list.
LinkedList
Backed by a doubly-linked list (nodes with pointers to previous and next).

Access get(i): O(N) - Slow, must traverse nodes.
Insertion add(e): O(1).
Middle Insertion: O(1) if you already have the Node reference via an Iterator, otherwise O(N) to find it.

Verdict: Use when you are heavily adding/removing elements from the beginning or middle of a large list.
The ConcurrentModificationException Trap If you iterate through a List using a for-each loop and try to .remove() an element from the list inside the loop, Java will throw a ConcurrentModificationException.

Fix: Use an explicit Iterator and call iterator.remove(), OR use Java 8's list.removeIf(condition).
The Set Interface
A collection that contains no duplicate elements. It models the mathematical set abstraction.
HashSet
Backed by a HashMap instance internally.

Ordering: None. Elements are scattered randomly based on their Hash Code.
Performance: O(1) for add, remove, and contains.

Verdict: The go-to default when you just need to ensure uniqueness and do fast lookups.
LinkedHashSet
Backed by a Hash table and a linked list running through it.

Ordering: Insertion-order. Iterating over it returns elements in the exact order they were added.
Performance: Slightly slower O(1) due to maintaining the linked list.

Verdict: Use when uniqueness AND order of creation matter.
TreeSet
Backed by a Red-Black Tree.

Ordering: Natural ordering (alphabetical/numerical) or by a custom Comparator.
Performance: O(log N) for add, remove, and contains.

Verdict: Use when you need to maintain a constantly sorted list without duplicate values.
java — Set Uniqueness Example
Set<String> uniqueNames = new HashSet<>();
uniqueNames.add("Alice");
uniqueNames.add("Bob");
boolean isAdded = uniqueNames.add("Alice"); 

System.out.println(isAdded); // Output: false (Alice was already there)
System.out.println(uniqueNames.size()); // Output: 2
Queue & Deque
Queues typically process elements in a FIFO (First-In-First-Out) manner. Deques (Double Ended Queues) allow insertion and removal at both ends.

The Queue interface provides special methods for handling capacity constraints. For example, add() throws an exception if the queue is full, whereas offer() gracefully returns false.

Removal: remove() throws exception if empty; poll() returns null.
Inspection: element() throws exception if empty; peek() returns null.

PriorityQueue
Elements are ordered based on their natural ordering or a supplied Comparator, NOT by insertion order.

The element at the head of the queue is always the "least" element according to the specified ordering.

Use Case: Task scheduling systems where high-priority tasks must jump ahead of low-priority tasks.
ArrayDeque
A resizable-array implementation of the Deque interface.

Can be used as a Queue (FIFO) or a Stack (LIFO).

Use Case: It is significantly faster than Stack when used as a stack, and faster than LinkedList when used as a queue. This is the modern go-to for Stack/Queue operations.
The Map Interface
An object that maps keys to values. A map cannot contain duplicate keys; each key can map to at most one value.
HashMap
Does not maintain any order.
Allows one null key and multiple null values.
O(1) time complexity for get() and put().
The default map to use in almost all situations.
LinkedHashMap
Maintains insertion order (or access order if configured via constructor).
Great for building simple LRU (Least Recently Used) caches.
TreeMap
Sorted according to the natural ordering of its keys, or by a Comparator.
O(log N) time for operations.
Does NOT allow null keys.
How HashMap works under the hood (Crucial Interview Topic)
1. A HashMap is backed by an array of Node<K,V> objects (called "buckets").
2. When you call put(Key, Value), Java calls hashCode() on the Key.
3. A mathematical formula converts the hash code into an array index (e.g., Index 4).
4. The Node is placed in bucket 4.
5. Hash Collision: If another Key hashes to bucket 4, it is added to a Linked List at that bucket.
6. Java 8 Optimization: If the Linked List at one bucket grows beyond 8 elements, Java converts it into a Red-Black Tree, improving lookup from O(N) to O(log N).
The equals() and hashCode() Contract If you use a custom Object as a Map Key, you MUST override both equals() and hashCode(). If two objects are equals(), they must return the same hashCode(). If you don't override them, put() and get() will fail to find your object in the Map.
Interview Prep
Collections are the most frequently asked topic in Java interviews. You must understand internal structures and concurrency implications.
Q1 ArrayList vs LinkedList: When to use which? Easy ▼

Use ArrayList (dynamic array) for 95% of use cases. It provides fast O(1) random access. Adding to the end is O(1). However, inserting/removing from the middle is O(N) because elements must be shifted.

Use LinkedList (doubly linked nodes) only when you are constantly inserting or deleting elements from the beginning or middle of the list. Random access in LinkedList is O(N) because it must traverse the nodes from the start/end to find the index.

Q2 What is the difference between Fail-Fast and Fail-Safe Iterators? Hard ▼

Fail-Fast iterators operate directly on the collection itself. If the collection is structurally modified (an element is added or removed) while iterating, it immediately throws a ConcurrentModificationException. Examples: ArrayList, HashMap, HashSet.

Fail-Safe iterators operate on a clone of the collection. Therefore, structural modifications to the original collection during iteration are ignored, and no exception is thrown. Examples: ConcurrentHashMap, CopyOnWriteArrayList.

Q3 HashMap vs Hashtable vs ConcurrentHashMap? Medium ▼

HashMap: Not thread-safe. Fast. Allows one null key and multiple null values.

Hashtable: Legacy class. Thread-safe, but poorly designed because it synchronizes (locks) the entire map during every read/write, causing a massive performance bottleneck. No null keys/values allowed.

ConcurrentHashMap: The modern thread-safe alternative. It uses lock-striping at the bucket level. Multiple threads can read/write simultaneously as long as they are operating on different buckets. Highly performant in concurrent environments.
