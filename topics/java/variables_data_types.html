Primitives & Memory Allocation
Java is a statically typed language. Every variable must have a declared type before it can be used. Java provides 8 primitive data types, which are the fundamental building blocks of data manipulation. Unlike Objects, primitives are not instantiated with the new keyword and do not share state.
Static Typing Stack Memory Performance

Primitives store raw values directly in memory. They are significantly faster and consume less memory than their Object wrapper equivalents (like Integer vs int).

java — The 8 Primitives
// --- Integer Types ---
byte  b = 100;         // 1 byte  (-128 to 127)
short s = 30000;       // 2 bytes (-32,768 to 32,767)
int   i = 2000000000;  // 4 bytes (-2^31 to 2^31-1) - Default for whole numbers
long  l = 9000000000L; // 8 bytes (-2^63 to 2^63-1) - Notice the 'L' suffix

// --- Floating-Point Types ---
float  f = 19.99f;     // 4 bytes (Single precision) - Notice the 'f' suffix
double d = 19.99999;   // 8 bytes (Double precision) - Default for decimals

// --- Character & Boolean ---
char    c = 'A';       // 2 bytes (Unsigned, represents a single 16-bit Unicode character)
boolean isJavaFun = true; // 1 bit of information (Actual memory size varies by JVM)
Underscores in Numeric Literals Since Java 7, you can use underscores to make large numbers readable. The compiler simply strips them out. E.g., int oneMillion = 1_000_000; or long creditCard = 1234_5678_9012_3456L;.
Reference Types vs Primitives
The distinction between how Java stores primitives and how it stores Objects is the source of 90% of beginner bugs, especially involving equality checks and method parameter passing.
Primitives (Stack Memory)
Stored directly on the Stack.

When you declare int x = 5;, the value 5 is placed exactly where x lives in memory.

Assigning a primitive creates an independent copy: int y = x; copies the value 5. Changing y later does not change x.
Reference Types (Heap Memory)
Objects are stored on the Heap. The variable on the Stack only holds a memory address (pointer) to the object.

When you declare User u = new User();, u stores an address like 0x1AB2.

Assigning a reference copies the address: User u2 = u;. Both variables now point to the exact same object. Modifying u2 modifies u.
The == Trap for Reference Types Using == on primitives compares their actual values. Using == on Reference Types compares their memory addresses.

String a = new String("Java");
String b = new String("Java");
System.out.println(a == b); // Prints false!

Always use .equals() to compare the logical contents of an Object: a.equals(b) is true.
Type Conversion & Casting Traps
Assigning a value of one type to a variable of another type triggers type conversion. This can be implicit (safe) or explicit (potentially destructive).

Widening Casting (Implicit) happens automatically when passing a smaller size type to a larger size type. There is no loss of information.

Sequence: byte → short → char → int → long → float → double

java
int myInt = 9;
double myDouble = myInt; // Automatic casting: int to double
System.out.println(myDouble); // Outputs 9.0

Narrowing Casting (Explicit) must be done manually by placing the type in parentheses in front of the value. This forces the compiler to squeeze a larger type into a smaller one, which can cause overflow or loss of precision.

java
double myDouble = 9.78;
int myInt = (int) myDouble; // Manual casting: double to int
System.out.println(myInt);    // Outputs 9 (Fractional part is completely chopped off, not rounded)

// Integer Overflow Trap
int max = Integer.MAX_VALUE; // 2147483647
max = max + 1; 
System.out.println(max);     // Outputs -2147483648 (Overflow wraps around to the lowest negative!)

Floating point numbers (float and double) use the IEEE 754 standard to represent fractions in binary. Because some fractions cannot be represented cleanly in binary (just like 1/3 cannot be cleanly written in decimal as 0.333...), they introduce minute precision errors.

java — The 0.3 Trap
double a = 0.1;
double b = 0.2;
System.out.println(a + b); // Outputs: 0.30000000000000004

if (a + b == 0.3) {
    System.out.println("Math works!"); // This line will NEVER print.
}
Never use float/double for Currency! If you use double to calculate financial transactions, pennies will silently appear or disappear due to binary rounding errors. Always use java.math.BigDecimal for money or precise scientific calculations.
Default Values & Initialization
Java's rules for initializing variables depend heavily on where the variable is declared.
Instance & Static Fields
Variables declared at the class level get default values automatically if not initialized:

• Numbers (byte, short, int, long): 0
• Floating points (float, double): 0.0
• booleans: false
• char: '\u0000' (null character)
• All Reference Types: null
Local Variables
Variables declared inside a method do not get default values.

If you declare a local variable and try to use it before initializing it, the Java compiler will throw an error: variable might not have been initialized.

This forces developers to be explicit and prevents undefined behavior.
Local Variable Type Inference (var)
Introduced in Java 10, the var keyword allows the compiler to infer the type of a local variable based on the value assigned to it. This reduces boilerplate, especially with complex Generics.

It is crucial to understand that Java remains strongly, statically typed. var is not a runtime dynamic type like in JavaScript. The type is baked in at compile time.

java
// Pre-Java 10: Repetitive typing
Map<String, List<UserProfile>> cache = new HashMap<String, List<UserProfile>>();

// Java 10+: Clean and readable
var cache = new HashMap<String, List<UserProfile>>();

// The compiler knows 'message' is a String.
var message = "Hello World"; 
message = 100; // COMPILE ERROR: incompatible types. A String cannot hold an int.
Strict Rules for var
1. You can only use it for local variables inside methods. You cannot use var for class instance fields, method parameters, or method return types.
2. It must be initialized immediately on the same line. var name; name = "John"; will not compile.
3. You cannot initialize it to null. var user = null; fails because the compiler has no clue what type of object `null` is supposed to represent.
Interview Prep
Questions on variables and memory are designed to expose developers who rely entirely on frameworks and don't understand the JVM's underlying mechanics.
Q1 Why does Java use 16 bits (2 bytes) for a `char`, whereas C/C++ traditionally use 8 bits? Medium ▼

Java was designed from the ground up to support internationalization. Instead of using ASCII (which only requires 8 bits to represent 256 characters), Java uses the Unicode (UTF-16) encoding system.

16 bits allow Java to natively represent 65,536 distinct characters, accommodating almost all standard characters from global languages. (Note: Supplementary characters beyond the Basic Multilingual Plane require surrogate pairs, which take two 16-bit `char`s).

Q2 What is the "Integer Cache" in Java, and how does it affect `==` comparisons? Hard ▼

To save memory, Java internally caches Integer wrapper objects for values between -128 and 127.

If you write Integer a = 100; Integer b = 100;, Java reuses the cached object. Therefore, a == b returns true because they point to the exact same memory address.

However, if you write Integer x = 128; Integer y = 128;, these fall outside the cache. Java creates two separate objects on the heap. Therefore, x == y returns false.

This is why you must always use .equals() to compare wrapper classes, regardless of the value!
