Multithreading in Java

Multithreading allows a Java program to perform multiple tasks concurrently, making better use of CPU resources and improving responsiveness. Java has built-in, first-class support for threads through the java.lang.Thread class, the java.lang.Runnable interface, and a rich set of concurrency utilities in java.util.concurrent.

1. Process vs Thread

Before diving into threads, it is important to understand the difference between a process and a thread.

Feature Process Thread
Definition An independent program in execution A lightweight unit of execution within a process
Memory Has its own memory space (heap, stack, etc.) Shares the heap of its owning process
Communication Inter-process communication (IPC) is expensive Shared memory — fast, but requires synchronization
Creation cost Heavy — OS must allocate resources Lightweight — created inside an existing process
Crash impact Crash is isolated to the process An uncaught exception can kill the entire process

A single JVM instance is a process. Every Java application starts with one thread — the main thread. You can spawn additional threads to run code in parallel within that same process.

2. Thread Lifecycle

Every thread in Java moves through a well-defined set of states, defined by the Thread.State enum.

The Six States

  • NEW — The thread object has been created but start() has not yet been called.
  • RUNNABLE — The thread is eligible to run. It may be actively executing on a CPU core or waiting for the scheduler to pick it up.
  • BLOCKED — The thread is waiting to acquire an intrinsic (monitor) lock held by another thread (e.g., trying to enter a synchronized block).
  • WAITING — The thread is waiting indefinitely for another thread to perform a specific action (e.g., after calling Object.wait() or Thread.join() with no timeout).
  • TIMED_WAITING — Like WAITING, but with a specified timeout (e.g., Thread.sleep(1000), Object.wait(500)).
  • TERMINATED — The thread has finished execution (either normally or due to an unhandled exception). 
public class LifecycleDemo {
    public static void main(String[] args) throws InterruptedException {
        Thread t = new Thread(() -> {
            try {
                Thread.sleep(500); // TIMED_WAITING
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            }
        });

        System.out.println("After new:   " + t.getState()); // NEW
        t.start();
        System.out.println("After start: " + t.getState()); // RUNNABLE or TIMED_WAITING
        Thread.sleep(100);
        System.out.println("While sleep: " + t.getState()); // TIMED_WAITING
        t.join();
        System.out.println("After join:  " + t.getState()); // TERMINATED
    }
}

3. Creating Threads

There are two classic ways to define the work a thread will execute.

3.1 Extending Thread

Subclass Thread and override the run() method. This is the simplest approach but limits flexibility because Java does not support multiple class inheritance. 

public class CounterThread extends Thread {

    private final String name;

    public CounterThread(String name) {
        this.name = name;
    }

    @Override
    public void run() {
        for (int i = 1; i <= 5; i++) {
            System.out.println(name + " -> " + i);
        }
    }

    public static void main(String[] args) {
        CounterThread t1 = new CounterThread("Thread-A");
        CounterThread t2 = new CounterThread("Thread-B");

        t1.start(); // starts a new OS thread — do NOT call run() directly!
        t2.start();
    }
}
Common Mistake: Calling thread.run() instead of thread.start() executes the method on the current thread — no new thread is spawned.

3.2 Implementing Runnable

Separate the task from the thread by implementing Runnable. This is the preferred approach because your class can still extend another class. 

public class PrintTask implements Runnable {

    private final String message;

    public PrintTask(String message) {
        this.message = message;
    }

    @Override
    public void run() {
        for (int i = 0; i < 3; i++) {
            System.out.println(Thread.currentThread().getName() + ": " + message);
        }
    }

    public static void main(String[] args) {
        Runnable task = new PrintTask("Hello from Runnable");

        Thread t1 = new Thread(task, "Worker-1");
        Thread t2 = new Thread(task, "Worker-2");

        t1.start();
        t2.start();
    }
}

3.3 Using Lambda Expressions (Java 8+)

Because Runnable is a functional interface, you can pass a lambda directly — the most concise modern style. 

public class LambdaThreadDemo {
    public static void main(String[] args) {
        Thread t = new Thread(() -> {
            System.out.println("Running in: " + Thread.currentThread().getName());
        }, "LambdaThread");

        t.start();
    }
}

4. Key Thread Methods

start()

Requests the JVM to schedule a new OS-level thread and invoke run() on it. Calling start() twice on the same Thread object throws IllegalThreadStateException.

run()

Contains the task logic. Calling this directly (without start()) just runs the method synchronously on the calling thread.

sleep(long millis)

Pauses the current thread for at least the specified duration. The thread transitions to TIMED_WAITING. It does not release any locks it holds. 

public class SleepDemo {
    public static void main(String[] args) throws InterruptedException {
        System.out.println("Start");
        Thread.sleep(2000); // pause for 2 seconds
        System.out.println("Resumed after 2s");
    }
}

join()

Causes the calling thread to wait until the target thread terminates. Useful for coordinating tasks that depend on each other. 

public class JoinDemo {
    public static void main(String[] args) throws InterruptedException {
        Thread worker = new Thread(() -> {
            System.out.println("Worker: computing...");
            try { Thread.sleep(1000); } catch (InterruptedException e) { Thread.currentThread().interrupt(); }
            System.out.println("Worker: done.");
        });

        worker.start();
        worker.join(); // main thread waits here until worker finishes

        System.out.println("Main: worker has finished, continuing.");
    }
}

interrupt()

Sets the thread's interrupt flag. If the thread is in a blocking operation (sleep, wait, join), it throws InterruptedException. Otherwise, it is the thread's responsibility to check Thread.currentThread().isInterrupted() and stop gracefully. 

public class InterruptDemo {
    public static void main(String[] args) throws InterruptedException {
        Thread worker = new Thread(() -> {
            while (!Thread.currentThread().isInterrupted()) {
                System.out.println("Working...");
                try {
                    Thread.sleep(500);
                } catch (InterruptedException e) {
                    System.out.println("Interrupted during sleep — shutting down.");
                    Thread.currentThread().interrupt(); // restore interrupt flag
                    break;
                }
            }
            System.out.println("Worker stopped cleanly.");
        });

        worker.start();
        Thread.sleep(1800); // let it run for a bit
        worker.interrupt();
    }
}

5. Synchronization

When multiple threads share mutable state, race conditions can produce incorrect results. Synchronization ensures only one thread accesses a critical section at a time.

5.1 The Race Condition Problem

public class UnsafeCounter {
    private int count = 0;

    public void increment() {
        count++; // NOT atomic: read → increment → write (3 steps)
    }

    public int getCount() { return count; }

    public static void main(String[] args) throws InterruptedException {
        UnsafeCounter counter = new UnsafeCounter();

        Runnable task = () -> {
            for (int i = 0; i < 10_000; i++) counter.increment();
        };

        Thread t1 = new Thread(task);
        Thread t2 = new Thread(task);
        t1.start(); t2.start();
        t1.join();  t2.join();

        // Expected: 20000 — but you will often see less due to lost updates!
        System.out.println("Final count: " + counter.getCount());
    }
}

5.2 The synchronized Keyword

Java's synchronized keyword acquires the intrinsic lock (monitor) of an object. Only one thread can hold a given object's monitor at a time; others BLOCK until it is released.

Synchronized method:

public class SafeCounter {
    private int count = 0;

    // Acquires the lock on `this` instance
    public synchronized void increment() {
        count++;
    }

    public synchronized int getCount() {
        return count;
    }
}

Synchronized block (finer-grained control):

public class SafeCounterBlock {
    private int count = 0;
    private final Object lock = new Object(); // dedicated lock object

    public void increment() {
        synchronized (lock) { // only this block is protected
            count++;
        }
        // code here runs without holding the lock
    }

    public int getCount() {
        synchronized (lock) {
            return count;
        }
    }
}
Tip: Prefer synchronized blocks over synchronized methods when only a small portion of a method needs protection. Smaller critical sections mean less contention and better performance.

5.3 wait() and notify()

These methods on Object allow threads to communicate through a shared monitor. They must always be called within a synchronized block on the same object. 

public class ProducerConsumer {
    private final Object monitor = new Object();
    private int data = -1;
    private boolean ready = false;

    public void produce(int value) throws InterruptedException {
        synchronized (monitor) {
            data = value;
            ready = true;
            System.out.println("Produced: " + value);
            monitor.notifyAll(); // wake up waiting consumers
        }
    }

    public void consume() throws InterruptedException {
        synchronized (monitor) {
            while (!ready) {         // always use a loop, not if!
                monitor.wait();      // releases lock, waits for notify
            }
            System.out.println("Consumed: " + data);
            ready = false;
        }
    }
}

6. The volatile Keyword

Without volatile, each thread may cache a variable's value in its local CPU cache. Changes made by one thread may not be visible to others. Marking a field volatile guarantees visibility: every read sees the latest write from any thread. 

public class VolatileDemo {
    // Without volatile, the worker thread may never see the updated value
    private static volatile boolean running = true;

    public static void main(String[] args) throws InterruptedException {
        Thread worker = new Thread(() -> {
            System.out.println("Worker started.");
            while (running) {
                // spin — kept intentionally simple for demonstration
            }
            System.out.println("Worker stopped.");
        });

        worker.start();
        Thread.sleep(1000);
        running = false; // guaranteed to be visible to worker due to volatile
        System.out.println("Main: set running = false");
    }
}
Important: volatile guarantees visibility but NOT atomicity. A volatile int counter still has a race condition on counter++. For atomic compound operations, use java.util.concurrent.atomic classes (e.g., AtomicInteger) or synchronized.

AtomicInteger Example

import java.util.concurrent.atomic.AtomicInteger;

public class AtomicCounter {
    private final AtomicInteger count = new AtomicInteger(0);

    public void increment() {
        count.incrementAndGet(); // atomic: no synchronization needed
    }

    public int getCount() {
        return count.get();
    }
}

7. Executor Framework

Manually creating threads is error-prone and inefficient at scale. The Executor framework in java.util.concurrent manages a pool of reusable worker threads, separating task submission from execution policy.

7.1 ExecutorService

ExecutorService is the main interface. The Executors factory class provides common pool configurations. 

import java.util.concurrent.ExecutorService;
import java.util.concurrent.Executors;

public class ExecutorDemo {
    public static void main(String[] args) {
        // Fixed pool: at most 3 threads run concurrently
        ExecutorService executor = Executors.newFixedThreadPool(3);

        for (int i = 1; i <= 7; i++) {
            final int taskId = i;
            executor.submit(() -> {
                System.out.println("Task " + taskId + " running on "
                        + Thread.currentThread().getName());
                try { Thread.sleep(500); } catch (InterruptedException e) { Thread.currentThread().interrupt(); }
            });
        }

        executor.shutdown(); // no new tasks; existing tasks finish
        System.out.println("All tasks submitted.");
    }
}

7.2 Common Thread Pool Types

Factory Method Behaviour Best For
newFixedThreadPool(n) Fixed number of threads; tasks queue when all busy CPU-bound tasks with known parallelism
newCachedThreadPool() Grows on demand; idle threads are reused or expired after 60 s Many short-lived I/O tasks
newSingleThreadExecutor() Exactly one thread; tasks execute sequentially Ordered background processing
newScheduledThreadPool(n) Supports delayed and periodic task scheduling Timers, polling, recurring jobs

7.3 Returning Results with Callable and Future

Unlike Runnable, Callable<V> can return a value and throw checked exceptions. ExecutorService.submit() returns a Future<V> that you can use to retrieve the result. 

import java.util.concurrent.*;

public class CallableFutureDemo {
    public static void main(String[] args) throws ExecutionException, InterruptedException {
        ExecutorService executor = Executors.newFixedThreadPool(2);

        Callable<Integer> sumTask = () -> {
            int sum = 0;
            for (int i = 1; i <= 100; i++) sum += i;
            return sum;
        };

        Future<Integer> future = executor.submit(sumTask);

        System.out.println("Task submitted; doing other work...");
        // future.get() blocks until the result is ready
        System.out.println("Sum 1-100 = " + future.get());

        executor.shutdown();
    }
}

8. Common Concurrency Problems

8.1 Race Condition

A race condition occurs when the correctness of a program depends on the relative timing of threads. The counter example above demonstrated this: two threads interleaving their read-modify-write steps produce an incorrect result.

Fix: Use synchronized, AtomicInteger, or other thread-safe constructs.

8.2 Deadlock

Deadlock happens when two or more threads each hold a lock the other needs, and none can proceed. 

public class DeadlockDemo {
    private static final Object LOCK_A = new Object();
    private static final Object LOCK_B = new Object();

    public static void main(String[] args) {
        Thread t1 = new Thread(() -> {
            synchronized (LOCK_A) {
                System.out.println("T1 holds A, waiting for B...");
                try { Thread.sleep(50); } catch (InterruptedException e) { Thread.currentThread().interrupt(); }
                synchronized (LOCK_B) { // waits forever — T2 holds B
                    System.out.println("T1 holds A and B");
                }
            }
        });

        Thread t2 = new Thread(() -> {
            synchronized (LOCK_B) {
                System.out.println("T2 holds B, waiting for A...");
                synchronized (LOCK_A) { // waits forever — T1 holds A
                    System.out.println("T2 holds B and A");
                }
            }
        });

        t1.start();
        t2.start();
        // Both threads are now deadlocked!
    }
}

Prevention strategies:

  • Lock ordering: always acquire locks in the same global order across all threads.
  • Try-lock with timeout: use ReentrantLock.tryLock(timeout) so a thread can back off and retry.
  • Minimize lock scope: hold locks for the shortest time possible.
  • Use higher-level abstractions: java.util.concurrent collections and utilities avoid manual locking altogether.
import java.util.concurrent.locks.ReentrantLock;
import java.util.concurrent.TimeUnit;

public class DeadlockPrevention {
    private static final ReentrantLock lockA = new ReentrantLock();
    private static final ReentrantLock lockB = new ReentrantLock();

    public static void acquireBoth() throws InterruptedException {
        while (true) {
            boolean gotA = lockA.tryLock(50, TimeUnit.MILLISECONDS);
            boolean gotB = lockB.tryLock(50, TimeUnit.MILLISECONDS);

            if (gotA && gotB) {
                try {
                    System.out.println("Both locks acquired — doing work.");
                    return;
                } finally {
                    lockA.unlock();
                    lockB.unlock();
                }
            }

            // Release whatever we got and retry
            if (gotA) lockA.unlock();
            if (gotB) lockB.unlock();
            Thread.sleep(10); // brief back-off
        }
    }
}

8.3 Starvation

Starvation occurs when a thread is perpetually denied access to a resource because other threads are always given priority. For example, a thread with low priority may never get CPU time if high-priority threads keep running.

Fix:

  • Use ReentrantLock(true) (fair mode) to ensure waiting threads are served in FIFO order.
  • Avoid holding locks for long periods that freeze out other threads.
  • Be cautious when using Thread.setPriority() — rely on it only as a hint, not a guarantee.
import java.util.concurrent.locks.ReentrantLock;

public class FairLockDemo {
    // true = fair mode: threads acquire the lock in the order they requested it
    private final ReentrantLock fairLock = new ReentrantLock(true);

    public void criticalSection(String threadName) {
        fairLock.lock();
        try {
            System.out.println(threadName + " in critical section.");
        } finally {
            fairLock.unlock();
        }
    }
}

8.4 Livelock

A livelock is similar to a deadlock, except the threads are not blocked — they continuously react to each other and change state but make no real progress. Think of two people in a corridor both stepping aside in the same direction, forever.

Fix: Introduce randomness or a back-off strategy so threads do not always react identically.

9. Thread-Safe Collections

Standard collections (ArrayList, HashMap) are not thread-safe. Use the concurrent alternatives from java.util.concurrent. 

import java.util.concurrent.*;

public class ConcurrentCollectionsDemo {
    public static void main(String[] args) throws InterruptedException {
        // Thread-safe map — fine-grained segment locking
        ConcurrentHashMap<String, Integer> map = new ConcurrentHashMap<>();
        map.put("alpha", 1);
        map.computeIfAbsent("beta", k -> 2);

        // Thread-safe queue — ideal for producer-consumer
        BlockingQueue<String> queue = new LinkedBlockingQueue<>(10);

        Thread producer = new Thread(() -> {
            try {
                queue.put("task-1");
                queue.put("task-2");
            } catch (InterruptedException e) { Thread.currentThread().interrupt(); }
        });

        Thread consumer = new Thread(() -> {
            try {
                System.out.println("Consumed: " + queue.take()); // blocks until available
                System.out.println("Consumed: " + queue.take());
            } catch (InterruptedException e) { Thread.currentThread().interrupt(); }
        });

        producer.start();
        consumer.start();
        producer.join();
        consumer.join();
    }
}

10. Best Practices

  • Prefer immutability. Immutable objects need no synchronization because their state never changes after construction. Declare fields final wherever possible.
  • Use the Executor framework. Never manage raw threads in production code. Use ExecutorService, CompletableFuture, or higher-level abstractions.
  • Keep critical sections small. The less code runs under a lock, the less contention there is and the faster your application runs.
  • Always handle InterruptedException properly. Either re-throw it or restore the interrupt flag with Thread.currentThread().interrupt().
  • Document thread-safety guarantees. Use annotations like @ThreadSafe, @NotThreadSafe, and @GuardedBy from jcip-annotations to communicate intent.
  • Test with concurrency tools. Use tools like jcstress and thread-safety analysis in your IDE to catch subtle bugs.
