Object-Oriented Programming in Java

Object-Oriented Programming (OOP) is the backbone of Java. Every Java program is built around classes and objects, and the four pillars — Encapsulation, Inheritance, Polymorphism, and Abstraction — guide how you structure, reuse, and scale code. This topic walks you through all the core OOP concepts with practical examples.

1. The Four Pillars of OOP

OOP is a programming paradigm that organises software around objects rather than functions and logic. Java enforces OOP through four core principles:

🔒 Encapsulation

Bundle data and the methods that operate on it inside a class, and hide internal state from the outside world using access modifiers.

🧬 Inheritance

A child class inherits fields and methods from a parent class, promoting code reuse and establishing an "is-a" relationship.

🔀 Polymorphism

One interface, many implementations. The same method call behaves differently depending on the actual object type at runtime.

🎭 Abstraction

Hide complex implementation details and expose only what the user needs through abstract classes and interfaces.

2. Classes and Objects

What is a Class?

A class is a blueprint or template for creating objects. It defines the state (fields/attributes) and behaviour (methods) that all objects of that type will share. 

// Blueprint for a Car object
public class Car {
  // Fields — represent state
  String brand;
  String model;
  int year;
  double speed;

  // Method — represents behaviour
  public void accelerate(double amount) {
    speed += amount;
    System.out.println(brand + " is now going " + speed + " km/h");
  }

  public void brake(double amount) {
    speed = Math.max(0, speed - amount);
    System.out.println(brand + " slowed to " + speed + " km/h");
  }
}

What is an Object?

An object is a concrete instance of a class. When you create an object, Java allocates memory on the heap for its fields. You create objects using the new keyword. 

public class Main {
  public static void main(String[] args) {
    // Creating objects (instances) of the Car class
    Car myCar = new Car();
    myCar.brand = "Toyota";
    myCar.model = "Corolla";
    myCar.year  = 2022;

    Car anotherCar = new Car();
    anotherCar.brand = "Honda";
    anotherCar.model = "Civic";
    anotherCar.year  = 2023;

    // Each object has its own independent state
    myCar.accelerate(60);      // Toyota is now going 60.0 km/h
    anotherCar.accelerate(80); // Honda is now going 80.0 km/h
  }
}
Class vs Object: Think of a class like a cookie cutter and an object like an actual cookie. The cutter (class) defines the shape; each cookie (object) is an independent instance with its own state.

3. Constructors

A constructor is a special method that runs automatically when an object is created. It has the same name as the class and no return type. Constructors are used to initialise fields to meaningful values.

Default Constructor

If you write no constructor, Java provides a no-arg default constructor that sets all fields to their default values (0, null, false). Once you define any constructor, Java no longer provides this implicit one. 

public class Student {
  String name;
  int age;

  // Explicit no-arg constructor — sets safe defaults
  public Student() {
    name = "Unknown";
    age  = 0;
  }
}

// Usage
Student s = new Student();
System.out.println(s.name); // Unknown
System.out.println(s.age);  // 0

Parameterized Constructor

A parameterized constructor accepts arguments so you can provide initial values at the moment of object creation. 

public class Student {
  String name;
  int    age;
  double gpa;

  // Parameterized constructor
  public Student(String name, int age, double gpa) {
    this.name = name;  // 'this' disambiguates field vs parameter
    this.age  = age;
    this.gpa  = gpa;
  }

  public void display() {
    System.out.printf("Student: %s | Age: %d | GPA: %.2f%n", name, age, gpa);
  }
}

// Usage
Student alice = new Student("Alice", 20, 3.85);
Student bob   = new Student("Bob",   22, 3.50);

alice.display(); // Student: Alice | Age: 20 | GPA: 3.85
bob.display();   // Student: Bob   | Age: 22 | GPA: 3.50

Constructor Overloading

You can define multiple constructors in the same class, each with a different parameter list. Java selects the correct one based on the arguments you pass — this is called constructor overloading. 

public class Rectangle {
  double width;
  double height;

  // No-arg: creates a 1×1 square
  public Rectangle() {
    this(1.0, 1.0);  // Delegates to two-arg constructor
  }

  // One-arg: creates a square
  public Rectangle(double side) {
    this(side, side);
  }

  // Two-arg: creates a rectangle
  public Rectangle(double width, double height) {
    this.width  = width;
    this.height = height;
  }

  public double area() {
    return width * height;
  }
}

// Usage
Rectangle r1 = new Rectangle();       // 1.0 × 1.0
Rectangle r2 = new Rectangle(5.0);    // 5.0 × 5.0
Rectangle r3 = new Rectangle(4.0, 6.0); // 4.0 × 6.0

System.out.println(r1.area()); // 1.0
System.out.println(r2.area()); // 25.0
System.out.println(r3.area()); // 24.0

Copy Constructor

A copy constructor takes an object of the same type as its parameter and creates a new, independent copy. Java does not provide one automatically — you must write it yourself. 

public class Point {
  int x;
  int y;

  public Point(int x, int y) {
    this.x = x;
    this.y = y;
  }

  // Copy constructor
  public Point(Point other) {
    this.x = other.x;
    this.y = other.y;
  }
}

// Usage
Point original = new Point(3, 7);
Point copy     = new Point(original); // deep copy

copy.x = 99; // modifying copy does NOT affect original

System.out.println(original.x); // 3  ← unchanged
System.out.println(copy.x);     // 99
Shallow vs Deep Copy: The copy constructor above works for primitive fields. If your class contains reference-type fields (arrays, other objects), you must manually copy those too — otherwise both objects share the same reference (shallow copy).

4. The this Keyword

Inside any instance method or constructor, this is a reference to the current object — the object on which the method was called. It has four main uses:

4.1 Disambiguate Fields from Parameters

public class Circle {
  double radius;

  public Circle(double radius) {
    // Without 'this', 'radius = radius' would assign the param to itself
    this.radius = radius; // left = field, right = parameter
  }
}

4.2 Calling Another Constructor — this()

public class Book {
  String title;
  String author;
  int    pages;

  public Book(String title) {
    this(title, "Unknown", 0); // must be the FIRST statement
  }

  public Book(String title, String author, int pages) {
    this.title  = title;
    this.author = author;
    this.pages  = pages;
  }
}

4.3 Passing the Current Object as an Argument

public class Printer {
  public void print(Circle c) {
    System.out.println("Radius: " + c.radius);
  }
}

public class Circle {
  double radius;

  public Circle(double radius) { this.radius = radius; }

  public void printSelf(Printer p) {
    p.print(this); // pass the current Circle to Printer
  }
}

4.4 Returning the Current Object (Builder Pattern)

public class QueryBuilder {
  private String table;
  private String condition;
  private int    limit;

  public QueryBuilder from(String table) {
    this.table = table;
    return this; // enables method chaining
  }

  public QueryBuilder where(String condition) {
    this.condition = condition;
    return this;
  }

  public QueryBuilder limit(int limit) {
    this.limit = limit;
    return this;
  }

  public String build() {
    return "SELECT * FROM " + table + " WHERE " + condition + " LIMIT " + limit;
  }
}

// Fluent / builder-style usage
String query = new QueryBuilder()
    .from("users")
    .where("age > 18")
    .limit(50)
    .build();

System.out.println(query);
// SELECT * FROM users WHERE age > 18 LIMIT 50

5. Access Modifiers

Access modifiers control visibility — who can see and use a class, field, method, or constructor. Java has four levels:

Modifier Same Class Same Package Subclass Everywhere
private ✅ ❌ ❌ ❌
default (no keyword) ✅ ✅ ❌ ❌
protected ✅ ✅ ✅ ❌
public ✅ ✅ ✅ ✅

Encapsulation with private + Getters/Setters

The best practice is to make fields private and expose controlled access through public getter and setter methods. This is the essence of encapsulation. 

public class BankAccount {
  private String owner;
  private double balance; // hidden — cannot be accessed directly

  public BankAccount(String owner, double initialBalance) {
    this.owner   = owner;
    this.balance = Math.max(0, initialBalance); // enforced on creation
  }

  // Getter — read-only access
  public double getBalance() {
    return balance;
  }

  public String getOwner() {
    return owner;
  }

  // No direct setter for balance — use domain methods instead
  public void deposit(double amount) {
    if (amount <= 0) throw new IllegalArgumentException("Deposit must be positive");
    balance += amount;
  }

  public void withdraw(double amount) {
    if (amount <= 0)       throw new IllegalArgumentException("Amount must be positive");
    if (amount > balance)  throw new IllegalStateException("Insufficient funds");
    balance -= amount;
  }
}

// Usage
BankAccount account = new BankAccount("Alice", 1000.0);
account.deposit(500.0);
account.withdraw(200.0);
System.out.println(account.getBalance()); // 1300.0

// account.balance = -9999; // COMPILE ERROR — private field is protected

protected — Inheritance-Friendly Access

public class Animal {
  protected String name; // accessible by subclasses

  public Animal(String name) {
    this.name = name;
  }

  protected void breathe() {
    System.out.println(name + " is breathing.");
  }
}

public class Dog extends Animal {
  public Dog(String name) {
    super(name);
  }

  public void bark() {
    breathe();                             // can access protected method
    System.out.println(name + " barks!"); // can access protected field
  }
}

6. The Four Pillars — In Depth

6.1 Encapsulation

Encapsulation means keeping an object's internal state private and providing a public interface to interact with it. The BankAccount example above is a perfect demonstration — balance is hidden and all changes go through methods that enforce business rules.

Benefits of encapsulation:

  • Prevents invalid state (validation in setters/methods)
  • Allows internal implementation to change without breaking callers
  • Makes code easier to reason about and test

6.2 Inheritance

Inheritance allows a child class (subclass) to acquire all non-private fields and methods from a parent class (superclass) using the extends keyword. Use super to call parent constructors or methods. 

// Parent class
public class Shape {
  String color;

  public Shape(String color) {
    this.color = color;
  }

  public double area() {
    return 0; // to be overridden
  }

  public void describe() {
    System.out.printf("A %s %s with area %.2f%n",
        color, getClass().getSimpleName(), area());
  }
}

// Child class
public class Circle extends Shape {
  private double radius;

  public Circle(String color, double radius) {
    super(color); // call parent constructor first
    this.radius = radius;
  }

  @Override
  public double area() {
    return Math.PI * radius * radius;
  }
}

// Another child class
public class Rectangle extends Shape {
  private double width, height;

  public Rectangle(String color, double width, double height) {
    super(color);
    this.width  = width;
    this.height = height;
  }

  @Override
  public double area() {
    return width * height;
  }
}

// Usage
Shape c = new Circle("red", 5);
Shape r = new Rectangle("blue", 4, 6);

c.describe(); // A red Circle with area 78.54
r.describe(); // A blue Rectangle with area 24.00
Java supports single inheritance only. A class can extend exactly one parent class. To achieve multiple-inheritance-like behaviour, use interfaces.

6.3 Polymorphism

Polymorphism ("many forms") lets you treat objects of different types through a common interface. There are two kinds in Java:

  • Compile-time (static) polymorphism — method overloading
  • Runtime (dynamic) polymorphism — method overriding via inheritance

Method Overloading (Compile-Time)

public class MathUtils {
  // Same name, different signatures
  public int add(int a, int b) {
    return a + b;
  }

  public double add(double a, double b) {
    return a + b;
  }

  public int add(int a, int b, int c) {
    return a + b + c;
  }
}

MathUtils m = new MathUtils();
System.out.println(m.add(2, 3));        // 5       — int version
System.out.println(m.add(2.5, 3.5));   // 6.0     — double version
System.out.println(m.add(1, 2, 3));    // 6       — three-arg version

Method Overriding (Runtime)

public class Animal {
  public String sound() {
    return "...";
  }
}

public class Dog extends Animal {
  @Override
  public String sound() { return "Woof"; }
}

public class Cat extends Animal {
  @Override
  public String sound() { return "Meow"; }
}

public class Duck extends Animal {
  @Override
  public String sound() { return "Quack"; }
}

// Runtime polymorphism — the correct sound() is selected at runtime
Animal[] animals = { new Dog(), new Cat(), new Duck() };

for (Animal a : animals) {
  System.out.println(a.sound()); // Woof, Meow, Quack
}

6.4 Abstraction

Abstraction hides implementation complexity and exposes only the essential interface. Java achieves abstraction through abstract classes and interfaces.

Abstract Classes

// Cannot be instantiated — just a blueprint
public abstract class Vehicle {
  protected String brand;

  public Vehicle(String brand) {
    this.brand = brand;
  }

  // Abstract method — subclasses MUST implement this
  public abstract void startEngine();

  // Concrete method — shared by all subclasses
  public void stop() {
    System.out.println(brand + " has stopped.");
  }
}

public class Car extends Vehicle {
  public Car(String brand) { super(brand); }

  @Override
  public void startEngine() {
    System.out.println(brand + ": Vroom! Engine started.");
  }
}

public class ElectricCar extends Vehicle {
  public ElectricCar(String brand) { super(brand); }

  @Override
  public void startEngine() {
    System.out.println(brand + ": Silently powered on.");
  }
}

// Usage
Vehicle v1 = new Car("Toyota");
Vehicle v2 = new ElectricCar("Tesla");

v1.startEngine(); // Toyota: Vroom! Engine started.
v2.startEngine(); // Tesla: Silently powered on.
v1.stop();        // Toyota has stopped.

Interfaces

public interface Flyable {
  int MAX_ALTITUDE = 40_000; // implicitly public static final

  void fly();                // implicitly public abstract

  default void land() {     // default method (Java 8+)
    System.out.println("Landing safely.");
  }
}

public interface Swimmable {
  void swim();
}

// A Duck can both fly and swim — multiple interface implementation
public class Duck extends Animal implements Flyable, Swimmable {
  @Override
  public void fly()  { System.out.println("Duck is flying!"); }

  @Override
  public void swim() { System.out.println("Duck is swimming!"); }
}

7. Static vs Instance Members

Java members (fields and methods) can be either instance-level or class-level (static). The distinction determines what data they hold and how they are accessed.

Aspect Instance Member Static Member
Belongs to Each object separately The class itself (shared)
Memory Allocated per object Allocated once at class loading
Access object.member ClassName.member
Can use this? ✅ Yes ❌ No
Typical use Per-object state/behaviour Shared constants, counters, utilities 
public class Counter {
  // Static field — shared across ALL Counter instances
  private static int totalCount = 0;

  // Instance field — unique to each Counter object
  private int id;
  private String label;

  public Counter(String label) {
    totalCount++;           // increments the SHARED counter
    this.id    = totalCount;
    this.label = label;
  }

  // Instance method — operates on a specific object
  public void display() {
    System.out.println("Counter #" + id + ": " + label);
  }

  // Static method — operates on class-level data only
  public static int getTotalCount() {
    return totalCount;
  }
}

// Usage
Counter a = new Counter("Alpha");
Counter b = new Counter("Beta");
Counter c = new Counter("Gamma");

a.display();                        // Counter #1: Alpha
b.display();                        // Counter #2: Beta
c.display();                        // Counter #3: Gamma

System.out.println(Counter.getTotalCount()); // 3

Static Initializer Blocks

A static block runs once when the class is first loaded, before any object is created. Use it for complex static field initialisation. 

public class Config {
  public static final String DB_URL;
  public static final int    TIMEOUT;

  // Static initializer block
  static {
    DB_URL  = "jdbc:mysql://localhost:3306/mydb";
    TIMEOUT = 30;
    System.out.println("Config loaded.");
  }
}

// Config.DB_URL is ready before any object is created
System.out.println(Config.DB_URL); // jdbc:mysql://localhost:3306/mydb

8. Object Lifecycle

Every Java object goes through a well-defined lifecycle from creation to garbage collection.

Phase 1 — Creation

The new keyword triggers three steps:

  1. Memory allocation on the heap for all fields
  2. Default initialisation — numerics → 0, booleans → false, references → null
  3. Constructor execution — your constructor code runs
// All three steps happen in one line:
Person p = new Person("Alice", 30);

Phase 2 — Use

The object lives on the heap and is reachable via one or more references. You can call methods, read/write fields (if accessible), and pass it to other methods.

Phase 3 — Unreachable

When no reference in the program points to an object, it becomes eligible for garbage collection (GC). 

Person p = new Person("Alice", 30); // object created, reachable

p = new Person("Bob", 25);  // p now points to Bob — Alice is unreachable

p = null;                   // Bob is now unreachable too

Phase 4 — Garbage Collection

Java's Garbage Collector automatically reclaims memory from unreachable objects. You do not need to free memory manually (unlike C/C++). 

public class Resource {
  private String name;

  public Resource(String name) {
    this.name = name;
    System.out.println(name + " created.");
  }

  // Called by GC before collecting the object — avoid relying on this
  @Override
  protected void finalize() throws Throwable {
    System.out.println(name + " is being garbage collected.");
    super.finalize();
  }
}

// Note: In modern Java, use try-with-resources / AutoCloseable
// instead of finalize() for deterministic cleanup.
Avoid finalize(): The finalize() method is deprecated as of Java 9 and removed in Java 18. Use AutoCloseable with try-with-resources for deterministic resource cleanup.

The AutoCloseable Pattern

public class DatabaseConnection implements AutoCloseable {
  private String url;

  public DatabaseConnection(String url) {
    this.url = url;
    System.out.println("Connected to " + url);
  }

  public void query(String sql) {
    System.out.println("Executing: " + sql);
  }

  @Override
  public void close() {
    System.out.println("Connection to " + url + " closed.");
  }
}

// try-with-resources — close() is called automatically
try (DatabaseConnection db = new DatabaseConnection("jdbc:mysql://localhost/mydb")) {
  db.query("SELECT * FROM users");
} // close() runs here, even if an exception was thrown

9. Putting It All Together

Here is a mini class hierarchy that demonstrates all four pillars working in concert: 

// ── Abstraction via abstract class ────────────────────────────────────────
public abstract class Employee {
  private String name;   // Encapsulation — private field
  private double salary;

  public Employee(String name, double salary) {
    this.name   = name;
    this.salary = salary;
  }

  // Controlled access — Encapsulation
  public String getName()   { return name; }
  public double getSalary() { return salary; }

  protected void setSalary(double salary) {
    if (salary < 0) throw new IllegalArgumentException("Salary cannot be negative");
    this.salary = salary;
  }

  // Abstract — every Employee subtype MUST define this
  public abstract double calculateBonus();

  // Concrete shared behaviour
  public void printPayslip() {
    System.out.printf("%-15s Salary: $%,.2f  Bonus: $%,.2f%n",
        name, salary, calculateBonus());
  }
}

// ── Inheritance + Polymorphism ─────────────────────────────────────────────
public class FullTimeEmployee extends Employee {
  public FullTimeEmployee(String name, double salary) {
    super(name, salary);
  }

  @Override
  public double calculateBonus() {
    return getSalary() * 0.15; // 15% of salary
  }
}

public class Contractor extends Employee {
  private int hoursWorked;

  public Contractor(String name, double hourlyRate, int hoursWorked) {
    super(name, hourlyRate);
    this.hoursWorked = hoursWorked;
  }

  @Override
  public double calculateBonus() {
    return hoursWorked > 160 ? 500.0 : 0.0; // flat bonus for extra hours
  }
}

// ── Usage ─────────────────────────────────────────────────────────────────
public class Payroll {
  public static void main(String[] args) {
    Employee[] staff = {
      new FullTimeEmployee("Alice",  90_000),
      new FullTimeEmployee("Bob",    75_000),
      new Contractor("Carol", 50.0, 200),
    };

    // Polymorphism — each printPayslip() calls the correct calculateBonus()
    for (Employee e : staff) {
      e.printPayslip();
    }
  }
}

// Output:
// Alice           Salary: $90,000.00  Bonus: $13,500.00
// Bob             Salary: $75,000.00  Bonus: $11,250.00
// Carol           Salary: $50.00      Bonus: $500.00

10. Summary

  • A class is a blueprint; an object is a live instance created with new.
  • Constructors initialise objects. You can overload them for flexibility, and use copy constructors for independent copies.
  • this refers to the current object — use it to disambiguate fields, chain constructors, or return the current instance.
  • Access modifiers (private, default, protected, public) control visibility and enforce encapsulation.
  • Static members belong to the class and are shared; instance members belong to individual objects.
  • The four pillars work together: Encapsulation protects state, Inheritance reuses code, Polymorphism provides flexibility, and Abstraction hides complexity.
  • Java's GC automatically handles object deallocation; prefer AutoCloseable for deterministic cleanup.
