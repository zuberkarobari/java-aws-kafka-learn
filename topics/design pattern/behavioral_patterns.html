Behavioral Patterns

Mastering the Strategy, Observer, and Command patterns to eliminate giant switch-statements and decouple application logic.

1. Definition

Simple English

Instead of a rigid machine where parts are welded together, think of a team of workers. Behavioral patterns define the rules of how these workers talk to each other, hand off tasks, and react to changes without stepping on each other's toes.

Technical

Behavioral design patterns are concerned with algorithms and the assignment of responsibilities between objects. They characterize complex control flow that's difficult to follow at runtime, shifting focus from control flow to object interconnection.

Purpose

To decouple the sender of a request from the receiver. They help eliminate massive if-else or switch statements, making applications modular, testable, and compliant with the Open/Closed Principle.

2. Deep Dive: The Strategy Pattern

3. Real-World Project Example

E-Commerce Notification System (Observer)

How the Observer Pattern decouples core logic from side effects.

The Problem: Spaghetti Code

When an order is placed, you need to: 1) Email the user, 2) SMS the driver, 3) Update the inventory, and 4) Log analytics. If the OrderService injects the EmailService, SmsService, and AnalyticsService directly, it violates the Single Responsibility Principle and becomes a nightmare to test.

The Observer Solution

The OrderService (Subject) simply publishes an OrderPlacedEvent and finishes its job. The Email, SMS, and Analytics services (Observers) are silently listening for this event. When the event fires, they automatically react. The OrderService doesn't even know they exist.

4. 7 Behavioral Patterns You need to know

1

Strategy

Defines a family of algorithms and makes them interchangeable. Perfect for different payment methods, sorting algorithms, or pricing strategies.

2

Observer (Pub/Sub)

A subscription mechanism where a Subject notifies multiple Observers of state changes. Heavily used in Spring via ApplicationEventPublisher.

3

Command

Turns a request into a stand-alone object. This allows you to parameterize methods, delay execution, queue operations, and easily implement "Undo" functionality.

4

State

Allows an object to alter its behavior when its internal state changes. Instead of massive if(state == "SHIPPED") blocks, the state itself is represented by different classes.

5

Chain of Responsibility

Passes a request along a chain of handlers. Each handler decides either to process the request or pass it to the next. Used inherently by Spring Security's FilterChain.

6

Template Method

Defines the skeleton of an algorithm in a superclass but lets subclasses override specific steps. Spring's JdbcTemplate is named after this pattern.

7

Iterator

Lets you traverse elements of a collection without exposing its underlying representation (list, stack, tree). Deeply embedded in Java's native java.util.Iterator.

6. Code Example: Spring Strategy Pattern

AdvancedStrategyDemo.java
Copied! 
import org.springframework.stereotype.Service;
import java.util.Map;

/* --- 1. The Strategy Interface --- */
public interface PaymentStrategy {
    void pay(double amount);
}

/* --- 2. Concrete Strategies --- */
@Service("stripe") // The bean name acts as the map key
public class StripeStrategy implements PaymentStrategy {
    @Override
    public void pay(double amount) {
        System.out.println("Paid $" + amount + " via Stripe.");
    }
}

@Service("paypal")
public class PayPalStrategy implements PaymentStrategy {
    @Override
    public void pay(double amount) {
        System.out.println("Paid $" + amount + " via PayPal.");
    }
}

/* --- 3. The Context (Checkout Service) --- */
@Service
public class CheckoutService {
    
    // MAGIC: Spring automatically injects all PaymentStrategy beans into this Map.
    // Key = Bean Name ("stripe", "paypal"), Value = The Bean Instance
    private final Map<String, PaymentStrategy> strategyMap;

    public CheckoutService(Map<String, PaymentStrategy> strategyMap) {
        this.strategyMap = strategyMap;
    }

    public void processOrder(double amount, String paymentMethod) {
        // 1. Fetch the correct strategy at runtime
        PaymentStrategy strategy = strategyMap.get(paymentMethod.toLowerCase());
        
        if (strategy == null) {
            throw new IllegalArgumentException("Invalid payment method: " + paymentMethod);
        }
        
        // 2. Execute the strategy
        strategy.pay(amount);
    }
}

7. Senior Developer Answer

"In enterprise Java, behavioral patterns are the antidote to cyclomatic complexity. The moment I see a Service class littered with if/else or switch statements determining business rules, I immediately refactor it using the Strategy Pattern.

Spring Boot makes the Strategy pattern incredibly powerful. By simply declaring an interface and annotating multiple implementations with @Service, I can inject a Map<String, Strategy> into my Context. This allows me to completely adhere to the Open/Closed Principle: if the business requests a new payment provider, I write a new class. I touch zero existing code.

Similarly, for decoupling side-effects like sending emails after a transaction, I avoid hard-coupling services. Instead, I implement the Observer Pattern using Spring's ApplicationEventPublisher and @EventListener (or @TransactionalEventListener to ensure the DB commit succeeded first). This keeps my core domains pristine and highly testable."

8. Quick Revision Notes

Focus: Object communication and assigning responsibilities.
Strategy: Swappable algorithms. Kills switch statements.
Observer: Pub/Sub. Decouples core logic from side-effects.
Command: Wraps request in object. Enables Undo/Redo queues.
Chain of Resp: Sequential handlers (e.g., Security FilterChain).
OCP Compliance: Allows adding new behaviors without altering existing code.
Spring Map Inject: Spring can auto-inject a Map of all Strategies.
Spring Events: Uses @EventListener to implement Observer cleanly.

9. Interview Answer Template

"Behavioral design patterns are focused on how objects communicate and divide responsibilities. The two most critical patterns in my day-to-day enterprise development are the Strategy Pattern and the Observer Pattern.

I use the Strategy Pattern to eliminate massive conditional blocks. If a service needs to process different payment types, I define a PaymentStrategy interface. In Spring, I can leverage Dependency Injection to auto-wire a Map of all available strategies, fetching the correct algorithm at runtime. This guarantees my code adheres to the Open/Closed Principle.

I use the Observer Pattern to handle side effects. If a user registers, I don't want the UserService to be tightly coupled to the EmailService. Instead, the UserService publishes a Spring ApplicationEvent. The EmailService silently observes this event and reacts asynchronously, keeping my core domain logic clean and highly testable."
