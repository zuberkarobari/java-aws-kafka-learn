The Anatomy of a JWT
A JWT is just three Base64Url encoded JSON strings, separated by dots. It is encoded, NOT encrypted. Anyone who intercepts the token can read the payload.

Security Rule: Never put passwords, SSNs, or highly sensitive PII in the JWT payload. It is readable by anyone who finds it in the browser's local storage or network logs.
Step 3
The Trust Model (Signatures)
If anyone can decode and read the token, what stops a hacker from changing their role from "USER" to "ADMIN"? The Cryptographic Signature.
🥷
1. The Hacker Alters the Payload
Hacker decodes the payload, changes "role":"USER" to "role":"ADMIN", re-encodes it to Base64, and sends the manipulated token to the server.
eyJhbGciOi....eyJzdWIiOiJ1c2VyIiwicm9sZSI6IkFETUlOIg....SflKxwR...
🛡️
2. Server Calculates the Expected Signature
The server takes the Header and the tampered Payload, and runs the HMAC/RSA hashing algorithm using its secret key.
❌
3. Signature Mismatch!
Because the payload changed, the calculated hash is entirely different from the signature attached to the end of the token. The server instantly knows it was tampered with and throws a 401 Unauthorized.

Step 5
Storage Security: XSS vs CSRF
Where do you store the JWT on the frontend? This is the most debated topic in web security. You must choose between mitigating XSS (Cross-Site Scripting) or CSRF (Cross-Site Request Forgery).
⚠️ Local Storage (Vulnerable to XSS)
Easy to use for SPAs (React/Vue). But if a hacker injects malicious JavaScript into your site, they can run localStorage.getItem('token') and steal it perfectly.
✅ HttpOnly Cookies (Preferred)
The server sets the JWT inside a cookie with the HttpOnly flag. JavaScript cannot read it, making it immune to XSS. The browser automatically attaches it to API requests.
*Note: Cookies are vulnerable to CSRF. You must use SameSite=Strict attribute to mitigate this.
Step 6
The Flaw: Revocation & Refresh Tokens
Because verification is stateless, the server cannot invalidate a JWT before it expires. If a hacker steals an active JWT, they have access. To minimize risk, Access Tokens must be short-lived (15 mins).
1
Access Token Expires
After 15 minutes, API requests return 401 Expired.
2
Client sends Refresh Token
A long-lived (7 days) refresh_token was given at login. The client sends it to a special /refresh endpoint.
3
Stateful Verification (The Database Hit)
The Auth Server checks the Refresh Token against the Database. Is the user banned? Did they click "Log out of all devices"? If valid, generate a fresh 15-min Access Token.
⚖️
The Perfect Balance
By using this architecture, 99.9% of your API requests are verified statelessly (blazing fast). Only 0.1% (the refresh calls every 15 mins) hit the database, preserving your ability to revoke access.
