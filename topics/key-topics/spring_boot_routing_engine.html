Legacy
The Traditional Pipeline
When your method returns a string like "home", the DispatcherServlet routes that string to the ViewResolver. It finds home.html on the disk and returns a full webpage.
JAVA
@Controller public class WebController { @GetMapping("/home") public String getHome(Model m) { m.addAttribute("user", "Zuber"); return "home"; // -> Routes to ViewResolver } }
Disaster
The API Catastrophe
You try to build an API. You return a User object. The DispatcherServlet blindly fires that object into the ViewResolver. It searches for an HTML file named User@0x12A4, fails, and crashes.
JAVA
@Controller public class ApiController { @GetMapping("/api/user") public User getUser() { return new User("Bob"); // MASSIVE CRASH! } }
Solution
The @ResponseBody Detour
To fix this, we inject @ResponseBody. This is a physical barricade. It screams: 'ABORT! Skip the ViewResolver!' The object is forcefully detoured into Jackson, which serializes it to JSON.
JAVA
@Controller public class ApiController { @GetMapping("/api/user") @ResponseBody // THE DETOUR BARRICADE public User getUser() { return new User("Bob"); } }
Exposed
The Meta-Annotation Secret
Writing @ResponseBody on 50 different methods is bloated. So, Spring engineers created @RestController. It contains ZERO new routing logic. It is literally just syntactic sugar.
Decompiled: RestController.java
@Target(ElementType.TYPE)
@Retention(RetentionPolicy.RUNTIME)
@Documented

@Controller
@ResponseBody
public @interface RestController {
  // Zero routing logic.
  // Pure syntactic sugar wrapping the detour.
}
Content Negotiation
The 406 Error Mystery
What if a client requests XML but you only have Jackson (JSON) loaded? The converter realizes it CANNOT satisfy the request. It throws a 406 Not Acceptable error.
HTTP Client
GET /api/user HTTP/1.1 Accept: application/xml
pom.xml Fix
<dependency> <groupId>com.fasterxml...</groupId> <artifactId>jackson-dataformat-xml</artifactId> </dependency>
