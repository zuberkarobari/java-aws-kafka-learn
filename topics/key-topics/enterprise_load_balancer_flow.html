What is a Load Balancer?
Before a request hits an API Gateway or a microservice, it hits a Load Balancer (LB). The LB acts as a highly available reverse proxy, distributing incoming traffic across multiple backend servers to ensure reliability, scalability, and zero downtime.
High Availability Distribution
🌎 Internet
Millions of Users

⚖️ Load Balancer
AWS ALB / Nginx / Envoy
───┬──→
💻 Node 1 (AZ-a)
Healthy
💻 Node 2 (AZ-b)
Healthy
💻 Node 3 (AZ-c)
Dead (Traffic Blocked)
⚖️ Traffic Distribution
Spreads the load
Prevents any single server from becoming overwhelmed. Automatically scales out horizontally.
🏥 Health Checks
Zero Downtime
Constantly pings backends. If Node 3 crashes, the LB instantly stops sending traffic to it.
🔒 TLS Offloading
Saves CPU Cycles
Decrypts HTTPS traffic at the edge, so backend servers don't waste compute power on cryptography.
🌐 Layer 4 vs Layer 7
TCP vs HTTP/HTTPS
L4 (Network) routes based on IP/Port. L7 (Application) routes based on URL paths and HTTP headers.

Stage 1
TLS Offloading & Connection Setup
The load balancer is the public face of the application. It holds the SSL/TLS certificate, negotiates the secure handshake with the client, and decrypts the traffic.
📱 Client
Public Internet
──🔐 TLS 1.3 ──→
⚖️ Load Balancer
Holds Certificate
── Plain HTTP ──→
⚙️ Backend Pod
VPC Internal
Why offload TLS at the Load Balancer?
1
Centralized Certificate Management
Instead of installing SSL certs on 500 different microservices, you install it ONCE on the load balancer (e.g., AWS Certificate Manager).
2
CPU Optimization
RSA/ECC cryptography is CPU intensive. The LB uses dedicated hardware/optimized software to decrypt, freeing up backend CPU for business logic.
3
Layer 7 Visibility
To route traffic based on URL paths (e.g., /api vs /images), the LB *must* be able to read the HTTP request. It cannot read it if it remains encrypted.
Config LB Listener Configuration (Nginx style)
server { listen 443 ssl http2; server_name api.globalbank.com; # The LB holds the certificate ssl_certificate /etc/ssl/certs/globalbank.crt; ssl_certificate_key /etc/ssl/private/globalbank.key; # Modern TLS config ssl_protocols TLSv1.2 TLSv1.3; ssl_ciphers ECDHE-ECDSA-AES128-GCM-SHA256:...; location / { # Forward decrypted traffic to internal pool via HTTP proxy_pass http://backend_cluster; } }

Fault Isolation
Because Node 10.0.1.42 returned a 500 Internal Server Error three times in a row, the LB removed it from the active pool. Our incoming request is guaranteed to only be routed to a healthy node (like 10.0.1.41).

Stage 4
Load Balancing Algorithms
We know the request goes to the API-Target-Group. But there are 3 healthy servers in that group. How does the LB pick exactly which one gets this specific request?
🔄 Algorithm 1: Round Robin (Default)
Requests are dealt sequentially: Server 1, then 2, then 3, then back to 1.
Req 1 -> S1
Req 2 -> S2
Req 3 -> S3
Flaw: If Req 1 is a heavy DB query, S1 might be overloaded, but Round Robin blindly gives it Req 4 anyway.
⚖️ Algorithm 2: Least Outstanding Connections (Best for APIs)
The LB looks at how many active, unfinished requests each server is currently processing, and sends the new request to the least busy server.
Server Active Conns Decision

10.0.1.44 12 ← SEND HERE
10.0.1.45 42 Wait
Stage 5
Session Persistence (Stickiness)
Sometimes, algorithms shouldn't be used. If an app stores session data in local server memory (not Redis), subsequent requests from the same user MUST go to the same server.
🍪
The Sticky Cookie
When enabled, the LB intercepts the first response and adds a special cookie (e.g., AWSALB). On the next request, the LB reads this cookie and bypasses the load balancing algorithm, routing directly to the server ID inside the cookie.
HTTPLB Reading the incoming cookie
GET /api/v2/transactions HTTP/2 Host: api.globalbank.com Cookie: AWSALB=83j2k1l9... (Encrypted pointer to 10.0.1.44)
✓
Stateless Microservices (Our Flow)
Modern fintech architectures use stateless microservices and external caches (Redis) for session state. Therefore, Sticky Sessions are disabled for this target group. The LB proceeds with the Least Connections algorithm.
Stage 6
Header Injection & Proxying
Because the LB terminates the TCP connection and creates a NEW connection to the backend, the backend sees the LB's IP address, not the user's. The LB must inject headers to preserve client context.
👤 Original Client info
IP: 203.0.113.55 Protocol: HTTPS Port: 443
🏷️ LB Injected Headers
X-Forwarded-For: 203.0.113.55 X-Forwarded-Proto: https X-Forwarded-Port: 443 X-Amzn-Trace-Id: Root=1-67891233...
Connection Pooling (Multiplexing)
Client 1 Client 2 Client 3

⚖️ LB

⚙️ Backend
The LB maintains persistent "Keep-Alive" connections to the backend.
It funnels hundreds of client requests through a handful of warm backend connections, drastically reducing TCP handshake overhead.
Stage 7
Access Logs & Metrics Emission
The backend processes the request and returns the response. The LB passes it back to the client and immediately writes a highly detailed access log and emits metrics to Prometheus/CloudWatch.

⏱️ The 3 Latencies
1. Request Proc: Time LB took to receive data (0.001s)
2. Target Proc: Time Backend took to compute (0.045s)
3. Response Proc: Time to send back to client (0.000s)
