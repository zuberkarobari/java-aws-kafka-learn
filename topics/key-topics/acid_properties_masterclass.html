Architecture
The Multi-Step Problem
To a database, transferring money isn't one step; it's a sequence of highly vulnerable operations. If the server loses power exactly between deducting from Alice and adding to Bob, the money disappears into the void.
1. Alice Balance
$1000 - $500
───→
⚡ POWER OUTAGE ⚡
Server Crashes
───→
2. Bob Balance
$500 + $500 (Never Runs)
Result: Alice lost $500, Bob got nothing. Database corrupted.
A - Atomicity
Transactions are "all or nothing". Partial failures are rolled back automatically.
C - Consistency
Data must strictly abide by rules (e.g., balance cannot drop below zero).
I - Isolation
Concurrent transactions do not interfere with each other. No race conditions.
D - Durability
Once committed, the data is invincible and survives total hardware failure.
Property 1
Atomicity (All or Nothing)
If step 2 fails, the database triggers an automatic 'Rollback'. It uses an UNDO log to physically reverse Alice's subtraction. The entire transaction is treated as a single, indivisible atom.
Step 1: Deduct Alice
✓ Success
─────→
Step 2: Add Bob
❌ Network Error
UNDO Log Applied
Alice restored to $1000
←── ROLLBACK ──
SQLTransaction Block
BEGIN; UPDATE accounts SET balance = balance - 500 WHERE name = 'Alice'; -- System crashes here! ROLLBACK; -- Reverts the previous UPDATE instantly
Property 2
Consistency (Valid States)
Suppose Alice only has $200, but she tries to send $500. This subtraction violates a core database constraint. The database halts the transaction, ensuring data never enters an invalid state.
1
Initial State
Alice Balance: $200
2
Operation Attempted
UPDATE accounts SET balance = balance - 500
❌
Constraint Violation
CHECK (balance >= 0) evaluates to FALSE (New balance would be -$300). Transaction aborted.
SQLEnforcing Rules at the Schema Level
ALTER TABLE accounts ADD CONSTRAINT check_positive_balance CHECK (balance >= 0); -- Prevents bad application logic from ever -- writing corrupt financial data to the disk.
Property 3
Isolation (Concurrency Control)
What if Alice and Charlie both withdraw $500 from a $500 account at the exact same millisecond? Isolation locks the row, forcing concurrent transactions to wait in a queue.
Thread 1 (Alice)
Thread 2 (Charlie)
───→
🔒
Alice's Row
Locked by Thread 1
Thread 2 is forced to WAIT until Thread 1 commits. Prevents race conditions.
SQLPessimistic Locking
BEGIN; -- Locks the row. Other threads must wait! SELECT balance FROM accounts WHERE name = 'Alice' FOR UPDATE; UPDATE accounts SET balance = balance - 500; COMMIT; -- Lock released. Thread 2 may proceed.
Property 4
Durability (Write-Ahead Log)
Once the database says 'Commit Successful', the data is invincible. Before updating the table on the physical hard drive, it writes the exact changes to a sequential 'Write-Ahead Log' (WAL).
1
Transaction Commits in RAM
Data is fast but volatile. Power loss = Data loss.
2
Write-Ahead Log (WAL) Flush
The DB sequentially appends the change string (e.g., "Alice-500") to the WAL file directly on the SSD. Sequential writes are blazing fast.
3
Confirmation & Hard Crash Recovery
Success is returned to the user. If the server explodes 1ms later, upon reboot, the DB reads the WAL and restores the exact state. It survives hardware death.
Conclusion
The RDBMS Guarantee
ACID makes Relational Databases the absolute king of Fintech and billing. But it comes at a steep cost: speed and global scalability (The CAP Theorem).
Atomicity
Transactions completely succeed or fail. No partial data.
Consistency
Enforces strict schema rules and valid data states.
Isolation
Locks prevent concurrent threads from overriding each other.
Durability
WAL guarantees permanence even during hardware failure.
