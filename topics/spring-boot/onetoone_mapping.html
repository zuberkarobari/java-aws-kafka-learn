@OneToOne Mapping

Mastering bidirectional links, the owning side vs inverse side, @MapsId, and the infamous lazy-loading proxy issue.

1. Definition

Simple English

A relationship where one row in a table is linked to exactly one row in another table. For example, a User has exactly one UserProfile, and that UserProfile belongs to exactly one User.

Technical

An association defining a 1:1 multiplicity. At the database level, it is enforced by adding a UNIQUE constraint to a Foreign Key column, or by sharing the identical Primary Key across both tables.

Purpose

To normalize data, secure sensitive columns, or achieve vertical partitioning. It allows you to separate heavy, rarely accessed data (like a BLOB photo or huge Bio text) from the frequently accessed main entity.

2. Internal Working: Owning vs Inverse

Understanding who holds the Foreign Key is the most critical part of bidirectional mappings.

Inverse Side

User Entity

@OneToOne(mappedBy = "user")
private UserProfile profile;
  • Does NOT contain the Foreign Key in DB.
  • Uses mappedBy to tell Hibernate: "Look at the 'user' field in UserProfile to find the mapping rules."
Foreign Key
Owning Side

UserProfile Entity

@OneToOne
@JoinColumn(name = "user_id")
private User user;
  • Contains the physical Foreign Key column (user_id).
  • Is responsible for updating the relationship in the database.
The Golden Rule of Bidirectional Mapping

If you only call user.setProfile(profile) and save, the database Foreign Key will be NULL. You MUST call profile.setUser(user) because the Owning Side is the only side Hibernate looks at when generating SQL UPDATE/INSERT statements for foreign keys.

3. Real-World Project Example

Banking KYC System (Vertical Partitioning)

Why we don't put everything in one table.

The Problem

In a banking app, the Customer table is queried 100 times a second just to show the name and balance. However, the customer also has KYC documents (Passport PDF blob, massive biography text, SSN). If we put the BLOB in the Customer table, every simple query pulls massive amounts of unnecessary data into memory.

The @OneToOne Solution

We split it into two tables: Customer and KycDetails. We link them with a @OneToOne(fetch = FetchType.LAZY). Now, querying the customer is lightning fast. The massive KYC data is only fetched from the database when a bank admin explicitly clicks "View KYC Documents" and the code calls customer.getKycDetails().

4. 7 Key Points For senior-level depth

1

Owning Side Rules

The side with @JoinColumn is the Owner. Hibernate only looks at the owner object to determine if a foreign key should be inserted or updated in the database.

2

@MapsId (Shared PK)

The absolute best practice for 1:1. Instead of creating a separate id and a user_id FK on the child table, @MapsId tells the child to use the exact same Primary Key value as its Parent. It saves an index and enforces strict 1:1.

3

Default FetchType

By default, @OneToOne is EAGER. This means fetching the parent will always execute a SQL JOIN to fetch the child. You should almost always override this to fetch = FetchType.LAZY.

4

CascadeType.ALL

Highly common in 1:1 relationships. If you save the Parent, you usually want the Profile saved automatically. If you delete the Parent, the Profile must be deleted.

5

Orphan Removal

Setting orphanRemoval = true means if you set the child reference to null (e.g., user.setProfile(null)), Hibernate will actually execute a SQL DELETE to remove the orphaned profile row.

6

Infinite Recursion

Bidirectional links cause `StackOverflowError` if you call toString() or serialize to JSON. Use `@JsonIgnore` on the inverse side, or avoid including the relationship in Lombok's @ToString.

7

The Inverse Lazy Loading Flaw

Even if you set fetch = FetchType.LAZY on the Inverse side, it won't work out-of-the-box. Hibernate must query the DB to check if the child exists (to decide whether to assign a Proxy object or null). Because it hits the DB anyway, it eagerly fetches. To fix this, you need build-time bytecode enhancement.

6. Code Example: The Best Practice

OneToOneMappingDemo.java
Copied! 
// 1. The Inverse Side (Parent)
@Entity
@Table(name = "users")
public class User {
    
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    private String username;

    // mappedBy = "user" means: look at the 'user' field in UserProfile
    // CascadeType.ALL means saving/deleting User saves/deletes Profile
    @OneToOne(mappedBy = "user", cascade = CascadeType.ALL, orphanRemoval = true, fetch = FetchType.LAZY)
    private UserProfile profile;

    // Helper method to keep both sides of the bidirectional link in sync!
    public void addProfile(UserProfile profile) {
        this.profile = profile;
        profile.setUser(this); // CRITICAL: Sets the owning side!
    }
}

// 2. The Owning Side (Child) - Using Best Practice @MapsId
@Entity
@Table(name = "user_profiles")
public class UserProfile {
    
    @Id
    // Notice: NO @GeneratedValue here! It shares the User's ID.
    private Long id;

    @Lob
    private String massiveBiography;

    @OneToOne(fetch = FetchType.LAZY)
    @MapsId // Tells Hibernate: Share the PK of the 'user' object as my PK.
    @JoinColumn(name = "user_id") // Optional naming
    private User user;
    
    // Getters & Setters ...
}

7. Senior Developer Answer

"When mapping 1:1 relationships in JPA, junior developers often default to creating two separate auto-generated IDs and linking them with a Foreign Key. As a senior, I mandate the use of @MapsId. This forces the child table to share the exact same Primary Key as the parent, skipping the overhead of a unique index on a foreign key and guaranteeing strict one-to-one integrity at the database level.

Secondly, I always ensure developers override the default FetchType.EAGER to LAZY to prevent N+1 issues. However, I always ask in code reviews if they are aware of the 'Inverse Lazy Loading Flaw'. Because Hibernate cannot know if a child profile exists when loading the parent User, it is forced to run a query anyway to decide if it should inject a proxy or null.

If this flaw causes a performance bottleneck in our system, I either utilize Hibernate bytecode enhancement to enable true lazy loading, or I architect the domain to use Unidirectional mapping from the child to the parent, removing the inverse mapping entirely."

8. Quick Revision Notes

Definition: 1 row in Table A maps to 1 row in Table B.
Owning Side: Holds the FK. Uses @JoinColumn.
Inverse Side: No FK. Uses mappedBy="fieldName".
Default Fetch: Always EAGER. Should be overridden to LAZY.
@MapsId: Best practice. Child table shares Parent's Primary Key.
Sync Method: Always set both sides of relationship in Java memory.
Lazy Flaw: Inverse side LAZY fails and eagerly queries DB anyway.
Recursion: Fix `toString()` and JSON serialization loops (`@JsonIgnore`).

9. Interview Answer Template

"A @OneToOne mapping in JPA defines a strict one-to-one relationship between two entities, often used for vertical table partitioning.

In a bidirectional mapping, one side must be the 'Owner' containing the @JoinColumn (the actual foreign key in the database), while the other is the 'Inverse' side containing the mappedBy attribute. The most crucial rule is that Hibernate only checks the Owning side to update the foreign key, so we must always keep both sides in sync in Java memory.

To optimize this, I prefer using @MapsId on the owning side. This ensures the child table uses the exact same Primary Key as the parent, saving an index and guaranteeing data integrity. Furthermore, I always override the default EAGER fetch type to LAZY, while keeping in mind that the inverse side cannot truly be lazy without bytecode enhancement."
