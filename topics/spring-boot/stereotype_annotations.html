Stereotype Annotations

@Component Hierarchy

Understanding the differences between @Component, @Service, and @Repository in multi-tier Spring architectures.

1. Definition

Simple English

Think of a company. Every person is an "Employee" (@Component). But for clarity, we give them specific titles like "Manager" (@Service) or "Warehouse Worker" (@Repository). They are all employees, but their titles tell you their exact job.

Technical

These are Stereotype Annotations. @Component is the generic base annotation. @Service and @Repository are meta-annotated with @Component, meaning Spring treats them identically during classpath scanning, but applies specific semantic meaning and targeted behaviors.

Purpose

1. To enforce Domain-Driven Design (DDD) by clearly marking the architectural layer of a class.
2. To allow Spring to apply layer-specific logic (e.g., translating SQL exceptions in Repositories).
3. To make AOP pointcuts easier to define.

3. Real-World Project Example

E-Commerce Checkout Flow

Mapping architectural layers to stereoypes.

@Component: StripePaymentUtil

A utility class that formats credit card strings or generates SHA-256 hashes for Stripe API requests. It's not a service, it's not a database, it's just a helpful generic component.

@Service: OrderProcessingService

Holds the core business logic. It checks inventory, calculates tax, calls the StripePaymentUtil, and manages the @Transactional boundary so that if payment fails, the database rolls back.

@Repository: OrderDaoImpl

Directly talks to the PostgreSQL database. If a unique constraint violation occurs (raw SQLException), Spring intercepts it because of this annotation and throws a generic DuplicateKeyException instead.

4. 7 Key Points For senior-level depth

1

The Meta-Annotation

If you look at the source code of @Service or @Repository, you will see they are annotated with @Component. This is why @ComponentScan finds all of them.

2

Exception Translation

@Repository is the only one with unique built-in logic. It translates vendor-specific exceptions (like Hibernate or JDBC exceptions) into Spring's unified DataAccessException.

3

AOP Pointcuts

Using specialized annotations makes Aspect-Oriented Programming (AOP) trivial. You can easily write an aspect that targets "Execution of all methods in classes annotated with @Service" for logging/metrics.

4

Technical Interchangeability

Technically, replacing a @Service with a @Component will not break your application. The app will boot and run perfectly fine. However, it violates design principles.

5

Missing Sibling: @Controller

@Controller (and @RestController) is the fourth major stereotype. It is specialized to register the class with the DispatcherServlet for HTTP request mapping.

6

Future Proofing

Spring documentation explicitly states that @Service may carry specific semantics in future releases. Using it properly now ensures your app doesn't break later.

7

Domain-Driven Design (DDD)

These annotations align perfectly with DDD concepts. Services orchestrate business logic across multiple aggregates, while Repositories abstract the persistence of those aggregates.

// 1. The @Component (Generic Utility Layer)
@Component
public class TaxCalculatorUtil {
    public double calculateTax(double amount) {
        return amount * 0.08;
    }
}

// 2. The @Repository (Data Access Layer)
@Repository
public class UserDaoImpl implements UserDao {
    
    // Without @Repository, this would throw a raw JDBC SQLException.
    // With @Repository, Spring catches it and throws DataAccessException
    public void saveUser(User user) {
        // raw JDBC/Hibernate logic here...
    }
}

// 3. The @Service (Business Logic Layer)
@Service
public class UserService {
    
    private final UserDao userDao;
    private final TaxCalculatorUtil taxUtil;

    // Spring auto-wires all components because they are all in the IoC container
    public UserService(UserDao userDao, TaxCalculatorUtil taxUtil) {
        this.userDao = userDao;
        this.taxUtil = taxUtil;
    }

    @Transactional // Transaction boundaries belong here!
    public void registerUserAndCalculateTax(User user) {
        userDao.saveUser(user);
        double tax = taxUtil.calculateTax(user.getSalary());
        // ... more business logic
    }
}

7. Senior Developer Answer

"Fundamentally, all of these are Stereotype annotations where @Component is the root. If you use @Service instead of @Component, the Spring IoC container treats the bean instantiation exactly the same.

However, as a senior developer, I strictly enforce their correct usage for architectural boundaries. @Service tells other developers 'this is where transaction boundaries belong.' @Repository is the only one with unique technical behavior: it triggers a BeanPostProcessor that wraps the class in a proxy to catch vendor-specific database exceptions (like an Oracle SQLException) and translates them into Spring's agnostic DataAccessException hierarchy.

Furthermore, consistently using these annotations allows our Platform team to easily write global AOP pointcuts. For instance, we can easily write an aspect to intercept all classes annotated with @Service to record method execution times, which wouldn't be possible if developers lazily used @Component for everything."

8. Quick Revision Notes

Hierarchy: @Component is the parent. @Service and @Repository are meta-annotated children.
Scanning: @ComponentScan detects all of them identically to create BeanDefinitions.
@Service: Used for business logic, facades, and defining @Transactional boundaries.
@Component: Used for generic utilities, mappers, schedulers, and custom wrappers.
@Repository: Used for DAOs. Includes specific logic for Exception Translation.
Translation: Converts SQL/Hibernate exceptions to DataAccessException.
AOP: Specialized annotations make Pointcut expressions for cross-cutting concerns easier.
Interchangeability: Swapping @Service for @Component compiles and runs, but breaks DDD semantics.

9. Interview Answer Template

"In Spring, @Component is the root stereotype annotation used to declare a class as a Spring-managed Bean.

@Service and @Repository are simply meta-annotations of @Component. While @Service acts technically identical to @Component today, it serves a semantic purpose to identify business logic layers and makes targeting AOP pointcuts much easier.

@Repository, on the other hand, provides distinct technical behavior. It tells Spring to apply a proxy (via a post-processor) that catches vendor-specific database exceptions—like raw SQLExceptions—and translates them into Spring's unified DataAccessException hierarchy. This prevents our business layer from being tightly coupled to specific database vendor exception types."
