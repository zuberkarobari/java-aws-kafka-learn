SQS — how it actually works
SQS is a fully managed message queue. Understanding the storage model, visibility timeout mechanics, and polling behaviour explains every SQS failure mode and drives the configuration choices that determine whether your system handles load correctly under failure.
at-least-once delivery visibility timeout long polling message retention

SQS stores messages across multiple redundant servers in multiple AZs within a region. Every ReceiveMessage call can return messages from different servers — there is no guarantee that messages are returned in order (for Standard queues), and you may occasionally receive the same message twice. This is the at-least-once delivery model.

At-least-once delivery is not a bug — it is a deliberate trade-off that allows SQS to scale horizontally without distributed coordination. To achieve exactly-once delivery, SQS offers FIFO queues with a different internal architecture at the cost of lower throughput.

Why duplicate delivery happens in Standard queues SQS stores each message on multiple redundant servers. When you call ReceiveMessage, SQS picks a server and returns the message from it. After you successfully process the message and call DeleteMessage, SQS deletes it from all servers. If the DeleteMessage call arrives at server A before server B syncs, a concurrent ReceiveMessage call from another consumer might get the message again from server B before the delete propagates. This window is typically milliseconds, but under high load or network partitions it can be seconds. Your consumer code must be idempotent — processing the same message twice produces the same outcome as processing it once.

Message attributes and limits: Each SQS message can carry up to 256KB of payload. For larger payloads, use the Extended Client Library — it stores the message body in S3 and puts only the S3 reference in the SQS message. Messages are retained for a configurable period (4 days default, 1 minute to 14 days). After the retention period, unprocessed messages are permanently deleted.

Throughput: Standard queues: virtually unlimited throughput (nearly unlimited transactions per second). FIFO queues: 300 TPS default, 3,000 TPS with high throughput mode enabled.

The 4-day message retention that loses data over a long weekend Your consumer application goes down on Friday evening. SQS retains messages for 4 days (default). The outage lasts until Tuesday morning — 4 days. When the consumer recovers, messages queued before Tuesday are already deleted by SQS's retention policy. Data loss without any error — messages were received by SQS successfully, just never processed before expiry. For business-critical operations (payment events, job application notifications): set retention to 14 days. Alert when the queue depth grows (messages older than 1 hour still in queue = consumer is down). The retention period is not a recovery window — it is the last line of defence before data disappears.

The visibility timeout is the most important SQS configuration parameter. When a consumer calls ReceiveMessage, the message becomes invisible to all other consumers for the duration of the visibility timeout. The consumer is expected to process the message and call DeleteMessage within the timeout. If it doesn't — because the consumer crashed, took too long, or had an exception — the message becomes visible again and another consumer picks it up.

The visibility timeout mechanics in a Spring Boot consumer Your Spring Boot service calls receiveMessage and gets a job application event. The visibility timeout is 30 seconds (the default). Processing involves: validating the application (5s), writing to RDS (10s), sending a notification email via SES (8s) = 23 seconds total. Fine. But if the SES call takes 35 seconds due to a SES rate limit, the visibility timeout expires at 30 seconds. Before your consumer finishes, another instance of the same Spring Boot service receives the same message, starts processing it, writes a duplicate application record to RDS, and sends a duplicate notification email. The first consumer then finishes and calls DeleteMessage — but so does the second. You have two records for one job application and two emails sent to the user. Set visibility timeout to at least 6× your average processing time. Use ChangeMessageVisibility to extend the timeout programmatically if a specific message takes longer than expected.

The visibility timeout set too high — the hidden cost Setting visibility timeout to 12 hours "to be safe" has a real cost: if a consumer crashes after receiving a message, that message is invisible for 12 hours. No other consumer can pick it up. The user's job application just disappeared for half a day. Find the balance: long enough that a single message never times out during normal processing, short enough that a crashed consumer releases messages in a reasonable time.

SQS supports two polling modes. Short polling (the default) queries a subset of SQS's distributed servers and returns immediately — even if no messages are available. Long polling queries all servers and waits up to 20 seconds for a message to arrive before returning empty.

Short polling (avoid in production)
Returns immediately. If the queue is empty: returns empty response. Your consumer immediately polls again. At high poll frequency: hundreds of empty API calls per minute per consumer instance. Each API call costs $0.0000004 — negligible individually, but 10 consumers × 100 empty polls/minute × 525,600 min/year = $220/year in wasted API calls. More importantly, empty polls waste CPU and network bandwidth on your EC2/ECS instances.

When short polling occurs: by default on SQS SDK calls, or when WaitTimeSeconds = 0.
Long polling (always use)
Waits up to 20 seconds (WaitTimeSeconds=20) for a message to arrive. If a message arrives within the wait window, returns immediately. If no message arrives in 20 seconds, returns empty. Dramatically reduces empty polls — from hundreds per minute to 3 per minute per idle consumer.

In Spring Cloud AWS: spring.cloud.aws.sqs.listener.max-messages-per-poll=10 and spring.cloud.aws.sqs.listener.poll-timeout-seconds=20.

Result: 95%+ reduction in API calls for idle queues.

A message moves through four states during its lifecycle. Understanding each state explains the SQS metrics that matter for monitoring.

1
Available — visible in queue
Message is in the queue, visible to consumers. CloudWatch metric: ApproximateNumberOfMessagesVisible. Rising count = consumer throughput is less than producer throughput. Critical alarm: this metric growing over time indicates consumer can't keep up — scale out consumers or investigate processing slowdowns.
2
In-flight — being processed
Message received by a consumer, currently invisible. CloudWatch metric: ApproximateNumberOfMessagesNotVisible. High in-flight with low visible = healthy processing. High in-flight and visibility timeouts expiring frequently = consumers are too slow or crashing.
3
Dead-lettered — failed after max retries
Message reached the maxReceiveCount (e.g. 5 retries) without successful processing. Moved to the Dead Letter Queue (DLQ). CloudWatch metric: ApproximateNumberOfMessagesVisible on the DLQ. Any message in the DLQ requires human investigation. Alarm: DLQ depth > 0 triggers a Slack alert.
4
Deleted — processing complete
Consumer called DeleteMessage after successful processing. Message is permanently removed from all SQS servers. The deletion is idempotent — calling DeleteMessage twice on the same receipt handle is safe (second call returns an error but doesn't cause issues).
Standard vs FIFO queues
The most common SQS architectural decision. Standard queues are the default and handle most use cases. FIFO queues solve specific ordering and deduplication requirements but at the cost of significantly lower throughput.
Standard Queue
Ordering: Best-effort — messages may be received in a different order than sent. Under high load, messages from the same producer can arrive in different orders.

Delivery: At-least-once — same message may be delivered more than once. Your consumer must be idempotent.

Throughput: Virtually unlimited — SQS auto-scales internally. No per-account or per-queue TPS limit in practice.

Use when: Processing order doesn't matter (sending emails, resizing images, triggering notifications, async background jobs). The vast majority of SQS use cases. GetWork job match notifications, worker availability updates, image processing.
FIFO Queue
Ordering: Strict first-in, first-out within each message group. Message group ID groups related messages that must be ordered relative to each other.

Delivery: Exactly-once — SQS deduplicates by deduplication ID (content hash or explicit ID) within a 5-minute window. Same message sent twice → delivered once.

Throughput: 300 TPS default. 3,000 TPS with high-throughput mode enabled. Named with .fifo suffix.

Use when: Ordering matters (financial transactions where debit must precede credit), or you need guaranteed exactly-once delivery (audit logs that must not have duplicates, sequential state machine transitions).
FIFO message group IDs — the ordering unit A FIFO queue processes messages in strict order within a message group. Messages from different groups can be processed concurrently. This is the key to FIFO scalability: if you use a single message group ID for all messages, only one consumer processes the queue at a time (strict global ordering, but throughput limited to one concurrent consumer). If you use per-entity group IDs — e.g. workerId as the group ID for a worker's activity events — messages for the same worker are ordered, but messages for different workers are processed concurrently by different consumers. This gets you both ordering (per worker) and scalability (parallel processing across workers).
The FIFO deduplication window — a 5-minute idempotency guarantee, not permanent FIFO deduplication checks if a message with the same deduplication ID has been sent within the last 5 minutes. If yes, the duplicate is discarded. If the same message is sent again after 5 minutes, it is treated as a new unique message and delivered. This is not a database-level deduplication — it's a short-window cache. If exactly-once semantics must hold beyond 5 minutes (e.g. a payment event that must never be processed twice regardless of when it's retried), implement idempotency at the application level using a database unique constraint or DynamoDB conditional write on the transaction ID.
Dead Letter Queues — handling processing failures
A DLQ captures messages that consistently fail to process. It is your safety net against poison pill messages — malformed messages that crash your consumer repeatedly, consuming retries and blocking healthy message processing.

The DLQ is a separate SQS queue you create and configure as the dead-letter destination for your primary queue. When a message's receive count exceeds the maxReceiveCount threshold (typically 3–5), SQS automatically moves it to the DLQ rather than continuing to retry.

A message in the DLQ stays there until you process it or its retention period expires. The DLQ gives you time to: investigate the failed message's content, fix the bug in the consumer that caused failures, and redrive the message back to the source queue after the fix is deployed.

The poison pill message and why maxReceiveCount matters A job application event arrives with a malformed JSON body — the workerCity field contains a SQL injection string that crashes your Jackson ObjectMapper. The consumer throws an exception. The visibility timeout expires. The message becomes visible again. Another consumer picks it up and crashes. After maxReceiveCount (5) attempts: the message moves to the DLQ. Without a DLQ, the message would retry forever, consuming 5 visibility timeout cycles per day, keeping your consumer workers busy processing a message that will never succeed. With a DLQ and maxReceiveCount=5: after 5 attempts, the poison pill is quarantined in the DLQ. Your consumers process the healthy messages. You get an alert (DLQ depth > 0), investigate the DLQ message, fix the bug in production, and redrive.
CLI — DLQ setup and message redrive
# Create the DLQ first
DLQ_URL=$(aws sqs create-queue \
  --queue-name getwork-job-events-dlq \
  --attributes MessageRetentionPeriod=1209600 \  # 14 days — max
  --query QueueUrl --output text --region ap-south-1)

DLQ_ARN=$(aws sqs get-queue-attributes \
  --queue-url $DLQ_URL \
  --attribute-names QueueArn \
  --query Attributes.QueueArn --output text --region ap-south-1)

# Create source queue with DLQ configured
aws sqs create-queue \
  --queue-name getwork-job-events \
  --attributes "{
    \"VisibilityTimeout\": \"120\",
    \"MessageRetentionPeriod\": \"345600\",
    \"ReceiveMessageWaitTimeSeconds\": \"20\",
    \"RedrivePolicy\": \"{\\\"deadLetterTargetArn\\\":\\\"$DLQ_ARN\\\",\\\"maxReceiveCount\\\":\\\"5\\\"}\"
  }" --region ap-south-1

# After fixing the bug — redrive DLQ messages back to source
# SQS console: Start DLQ Redrive, or use the API:
aws sqs start-message-move-task \
  --source-arn $DLQ_ARN \
  --destination-arn SOURCE_QUEUE_ARN \
  --max-number-of-messages-per-second 1 \   # Throttle redrive rate
  --region ap-south-1
The DLQ alarm that wakes your on-call engineer at the right time An alarm on ApproximateNumberOfMessagesVisible > 0 on the DLQ fires immediately when any message fails processing. But at 3AM, a single malformed test message from development shouldn't page the on-call engineer. Better threshold: ApproximateNumberOfMessagesVisible > 5 for 5 consecutive minutes — indicating a systematic failure, not a one-off bad message. Separately, alarm on ApproximateAgeOfOldestMessage > 3600 on the source queue — this fires when the oldest message has been sitting unprocessed for an hour, regardless of depth. This catches the case where the queue has many messages but they're all new, vs a stuck single message from an hour ago.
Spring Boot + SQS — the complete integration
Spring Cloud AWS provides a listener model for SQS that hides the ReceiveMessage/DeleteMessage polling loop. Understanding what it does under the hood determines how you configure timeouts, error handling, and concurrency correctly.
Java — SQS listener in Spring Boot
// pom.xml dependency
// io.awspring.cloud:spring-cloud-aws-starter-sqs:3.x

// application.properties
spring.cloud.aws.sqs.listener.max-messages-per-poll=10
spring.cloud.aws.sqs.listener.poll-timeout-seconds=20    # Long polling
spring.cloud.aws.sqs.listener.acknowledgement-mode=ON_SUCCESS
# ON_SUCCESS: auto-delete message on successful return from @SqsListener method
# MANUAL: you explicitly call Acknowledgement.acknowledge() or fail()

// Listener — receives and processes job application events
@Component
@Slf4j
public class JobApplicationListener {

    private final ApplicationService applicationService;

    // Spring Cloud AWS calls this method for each SQS message
    // Automatic deserialization from JSON to JobApplicationEvent
    @SqsListener(value = "getwork-job-events",
                  acknowledgementMode = SqsListenerAcknowledgementMode.ON_SUCCESS)
    public void handleJobApplication(JobApplicationEvent event,
                                         @Header("MessageId") String messageId) {
        log.info("Processing application: jobId={} workerId={} messageId={}",
            event.getJobId(), event.getWorkerId(), messageId);

        // Idempotency check — what if this message was already processed?
        if (applicationService.alreadyProcessed(messageId)) {
            log.info("Duplicate message {} — skipping", messageId);
            return;  // Return success to delete from queue — we handled it before
        }

        try {
            applicationService.processApplication(event);
            applicationService.markProcessed(messageId);
            log.info("Application processed: jobId={}", event.getJobId());
            // Method returns normally → ON_SUCCESS → message deleted from SQS
        } catch (RetryableException e) {
            // Transient failure — throw to keep message visible for retry
            log.warn("Retryable failure: {} — will retry", e.getMessage());
            throw e;  // Message stays in queue, visibility timeout expires, redelivered
        } catch (PermanentException e) {
            // Permanent failure — log and return success to remove from queue
            // DLQ handles this case after maxReceiveCount retries
            log.error("Permanent failure for message {}: {}", messageId, e.getMessage());
            throw e;  // Let SQS count the receive, eventually route to DLQ
        }
    }
}

// Event class — must match the JSON structure of SQS messages
@Data
public class JobApplicationEvent {
    private Long jobId;
    private Long workerId;
    private String applicationNote;
    private Instant appliedAt;
}
The listener that deletes messages on exception — silent data loss Spring Cloud AWS's ON_SUCCESS mode deletes messages when the listener method returns normally. But some teams configure acknowledgement incorrectly or use try-catch blocks that swallow exceptions and return normally. The message is deleted from SQS. The processing failed silently. No DLQ entry. No retry. No record of the failure. Always: let exceptions propagate out of the listener method for failures you want retried. Only return normally (or explicitly acknowledge) when processing succeeded. The DLQ is only populated when SQS sees enough receive attempts — if you swallow exceptions and return success, the DLQ never fires.
Idempotency implementation with DynamoDB for SQS consumers The MessageId header (a UUID assigned by SQS per message send) is stable — the same physical message always has the same MessageId on every redelivery. Use it as the idempotency key: before processing, write the MessageId to a DynamoDB table with a conditional put (condition: item doesn't exist). If the conditional put succeeds, you're the first to process this message. If it fails with ConditionalCheckFailedException, this message was already processed by a previous attempt — skip it and return success to delete from queue. TTL the DynamoDB entry at message retention period + 1 day. This implements exactly-once processing semantics on top of SQS's at-least-once delivery guarantee.
SNS — publisher/subscriber at scale
SNS is a push-based pub/sub service. A publisher sends a message to an SNS topic once. SNS delivers it to all subscribers simultaneously — SQS queues, Lambda functions, HTTP endpoints, email, SMS, mobile push. Understanding SNS delivery semantics, retry behaviour, and filter policies is what makes SNS reliable in production.

SNS topics are the broadcast channel. A publisher calls sns.publish(topicArn, message) once. SNS fans out the message to all subscribed endpoints simultaneously. The publisher doesn't know how many subscribers there are or whether they succeed — it fires and forgets.

SNS supports the following subscription types: SQS (message placed in a queue — the most reliable combination), Lambda (function invoked directly), HTTP/HTTPS (webhook to any endpoint), Email (sent as email — for human notification, not machine processing), SMS (text message), Kinesis Data Firehose (real-time data streaming), and Application (mobile push notification via APNS, FCM).

SNS message format — what your SQS consumer actually receives When SNS delivers to an SQS subscription, the SQS message body is NOT your original message. It is a JSON envelope from SNS containing: Type, MessageId, TopicArn, Subject, Message (your original message body as a string), Timestamp, and Signature. Your Spring Boot SQS consumer must unwrap this envelope. Configure rawMessageDelivery=true on the SNS-to-SQS subscription to tell SNS to put only your original message in the SQS body — not the JSON wrapper. Without raw message delivery, you need to parse the SNS envelope and extract the Message field before deserializing your actual event payload. Enable raw message delivery for SQS subscriptions unless you specifically need the SNS envelope metadata.

SNS subscription filter policies route messages to specific subscribers based on message attributes. Without filter policies, every subscriber receives every message published to the topic. With filter policies, a subscriber only receives messages whose attributes match its filter.

JSON — SNS filter policy
// Topic: getwork-job-events
// Published with message attributes: {"eventType": "JOB_APPLICATION", "city": "Mumbai"}

// Subscription 1: Analytics SQS queue — receives all events
// Filter policy: none (receives everything)

// Subscription 2: Notification Lambda — only JOB_APPLICATION events
{
  "eventType": ["JOB_APPLICATION", "JOB_ACCEPTED"]
}

// Subscription 3: Mumbai operations SQS — only Mumbai events
{
  "city": ["Mumbai", "Pune", "Nashik"]
}

// Subscription 4: High-priority handler — numeric filter
{
  "jobCount": [{ "numeric": [">", 10] }]
}

// Apply filter policy to subscription:
aws sns set-subscription-attributes \
  --subscription-arn SUB_ARN \
  --attribute-name FilterPolicy \
  --attribute-value '{"eventType":["JOB_APPLICATION","JOB_ACCEPTED"]}' \
  --region ap-south-1
Messages dropped when filter policy has no matching subscriber You publish a message with eventType=JOB_DELETED to the topic. Every subscriber has a filter policy that doesn't match JOB_DELETED. SNS delivers to zero subscribers — the message is silently discarded. No error is returned to the publisher. The publisher gets a success response (the publish to SNS succeeded). The message simply reaches nobody. If you add a new event type, ensure at least one subscriber (e.g. an analytics catch-all queue with no filter) receives every message type, acting as an audit trail. Alternatively, add the new event type to relevant filter policies before publishing any messages of that type.

SNS delivery retries depend on the subscription type. For SQS and Lambda subscribers, SNS retries delivery with exponential backoff for up to 23 days if the endpoint is unavailable. For HTTP/HTTPS endpoints, SNS retries with backoff but gives up after 3 retries within 30 minutes by default (configurable). For email and SMS, there is typically only one delivery attempt.

SNS Dead Letter Queue: For Lambda and SQS subscriptions, you can configure a DLQ on the subscription itself (not the topic). If SNS exhausts all retries and still can't deliver to the subscriber, the undeliverable message goes to the subscription's DLQ. This captures messages that SNS could not deliver at all — different from the SQS DLQ which captures messages the consumer received but failed to process.

The two DLQ tiers in SNS → SQS architectures In a SNS → SQS → Lambda pipeline, there are potentially two DLQ tiers. SNS subscription DLQ: catches messages SNS couldn't deliver to the SQS queue (SQS was down, wrong permissions). SQS queue DLQ: catches messages the Lambda consumer received but failed to process after maxReceiveCount retries. Both DLQs need alarms. The first is rare (SQS is highly available) but catastrophic when it fires — it means messages were permanently lost before even entering your processing pipeline.
Fan-out — SNS + SQS for parallel async processing
The SNS + SQS fan-out pattern decouples producers from multiple consumers, allows each consumer to process at its own pace, and provides durable buffering so a slow or temporarily unavailable consumer doesn't block others or lose messages.
GetWork fan-out — job posting event triggers three independent processing flows

POST /jobs
→ publish →
SNS Topic
getwork-job-events
→ fan-out →
SQS → Worker matching service
SQS → Push notification service

Each SQS queue buffers messages independently. If the analytics service is slow or down, messages queue up in the analytics SQS without affecting the matching or notification services. The Spring Boot API returns success immediately after publishing to SNS — it doesn't wait for any downstream processing.

Why SQS between SNS and the consumer, not direct SNS-to-Lambda? A direct SNS → Lambda subscription invokes Lambda immediately for every message. If Lambda is rate-limited (concurrency exhausted), SNS retries and eventually routes to the Lambda DLQ. There is no buffering — Lambda either handles it now or the message is queued in the SNS retry mechanism. With SQS in between: messages buffer durably in SQS regardless of Lambda's current concurrency. Lambda processes them at a sustainable rate. Backpressure is handled naturally — SQS depth grows when Lambda is slow, and you can scale Lambda concurrency to drain the backlog. SQS also enables batch processing (Lambda reads 10 messages at once from SQS) which is more efficient than SNS's one-message-per-invocation model.

The GetWork job posting flow with fan-out When an employer posts a job in Mumbai: the API handler saves to RDS (synchronous, must succeed) and publishes one event to the getwork-job-events SNS topic. SNS fans out to three SQS queues simultaneously in milliseconds. The API returns 201 Created to the employer. Worker matching runs asynchronously (5-30 seconds to query and score workers). Push notifications fire asynchronously (1-5 seconds via FCM). Analytics records the job posting (1-2 seconds). Total API response time: ~50ms (just RDS write + SNS publish). The employer sees an immediate response. Everything else happens behind the scenes.
EventBridge — event routing at the AWS service level
EventBridge is an event bus that routes events between AWS services, your applications, and SaaS partners. Unlike SQS (queue) and SNS (topic), EventBridge is a router — it evaluates events against rules and routes matching events to targets. It's the backbone of event-driven architecture with AWS-native services.
event bus event routing AWS service events scheduler

EventBridge has three types of event buses.

Default event bus: automatically exists in every AWS account. AWS services (EC2, RDS, CodePipeline, ECS, GuardDuty, Health, and 90+ others) publish their state change events here. You don't publish to this bus directly — AWS does. You create rules to catch specific AWS service events and route them to targets.

Custom event bus: you create and publish application events to it. Your Spring Boot service publishes JobCreated, WorkerRegistered, PaymentProcessed events. Rules route matching events to SQS queues, Lambda functions, Step Functions, or cross-account buses.

Partner event bus: SaaS applications (Stripe, Zendesk, Shopify, GitHub) can publish directly to your account's partner event bus. You create rules to respond to SaaS events — e.g. route Stripe payment events to your payment processing Lambda.

The default bus events that are already flowing — you just need to listen Right now, your AWS account's default EventBridge bus is receiving: every EC2 instance state change (running, stopped, terminated), every RDS instance state change (available, backing-up, failing-over), every ECS task state change (RUNNING, STOPPED), every CodePipeline pipeline execution state change (STARTED, SUCCEEDED, FAILED), every AWS Health event (service degradation notifications), every GuardDuty finding. You pay nothing for these events — they arrive on the default bus automatically. Creating a rule to catch them and route to a Lambda (that posts to Slack) gives you real-time operational awareness at essentially zero cost.

Every EventBridge event is a JSON object with a fixed envelope structure. Understanding the structure is what allows you to write accurate rule patterns.

EventBridge delivery guarantees depend on the target type. For targets that support it, EventBridge retries delivery with exponential backoff for up to 24 hours. For Lambda: invokes asynchronously, retries on failure, routes to DLQ after exhausted retries. For SQS: puts the message in the queue durably. For HTTP endpoints: retries with backoff.

EventBridge targets each have an independent retry policy configurable per rule target. You can configure the maximum retry attempts (0–185) and maximum event age (1 minute to 24 hours). You can also configure a DLQ on the rule target — events that EventBridge fails to deliver after all retries go to this DLQ.

EventBridge "at-least-once" — rules can match the same event twice EventBridge guarantees at-least-once delivery. Under rare circumstances (EventBridge infrastructure retry on transient error), a rule can trigger its targets more than once for the same event. Your Lambda or SQS consumer must be idempotent. The event's id field is a unique identifier for the event — use it as the idempotency key. This is the same pattern as SQS: design consumers to handle duplicate events gracefully rather than assuming each event is delivered exactly once.

Rules evaluate every event on the bus and route matching events to configured targets. The event pattern is the filter. Understanding the pattern language lets you write precise rules that route exactly the right events.

Up to 5 targets per rule — route the same event to multiple destinations A single EventBridge rule can route matched events to up to 5 targets simultaneously. One JobCreated event → SQS matching queue + SQS notification queue + SQS analytics queue + a Step Functions execution for complex matching logic + a Lambda for real-time dashboard update. Each target receives the full event. Each target can have input transformers that reshape the event before delivery — so the analytics SQS receives only the fields the analytics service needs, not the full event payload.
Overly broad rules that match too many events A rule with pattern { "source": ["aws.ec2"] } matches every EC2 event — instance launches, terminations, state changes, volume attachments, security group changes. If your account has 100 EC2 instances with active churn, this rule fires hundreds of times per minute. Each firing invokes its target (Lambda, SQS). Lambda invocations cost $0.0000002 each — negligible per invocation, but 100 invocations/minute × 525,600 minutes/year = $10.51/year per Lambda. More importantly, the Lambda log group in CloudWatch gets flooded with logs for events you don't care about. Always be specific in EventBridge rules. Add detail-level filters to narrow the event set to exactly what you need.

EventBridge Pipes
Connects a source (SQS, Kinesis, DynamoDB Streams, Kafka, MQ) to a target (Lambda, SQS, SNS, EventBridge bus, Step Functions, API Gateway) with optional filtering and enrichment.

Without Pipes: Write a Lambda that polls SQS, optionally calls an enrichment API to augment the message, and publishes to EventBridge. 50+ lines of boilerplate polling, error handling, retry logic.

With Pipes: Configure source=SQS, filter={"status":"ACTIVE"}, enrichment=Lambda (optional), target=EventBridge bus. Zero boilerplate. AWS manages the polling, filtering, and routing. The enrichment Lambda only receives filtered events and transforms them — no infrastructure concerns.

GetWork use case: DynamoDB Streams → Pipe → filter only JOB_STATUS_CHANGED events → enrich with employer details → publish to EventBridge custom bus.
EventBridge Scheduler
Triggers AWS targets (Lambda, SQS, Step Functions, ECS tasks, and 270+ more) on a schedule — one-time or recurring. Replaces: CloudWatch Events cron rules, scheduled Lambda functions, manual EC2 cron jobs.

Supports: Cron expressions, rate expressions (every 5 minutes), one-time schedules with exact timestamp, flexible time windows (fire within a 15-minute window of the scheduled time to distribute load).

At-least-once with retries: If the target fails, Scheduler retries with configurable backoff. DLQ for failed invocations. Unlike Lambda scheduled events which silently drop failed invocations.

GetWork use case: Every night at 11PM IST → start ECS task → run job matching batch → stop task. One Scheduler rule, no running EC2 cron server.
EventBridge Scheduler vs CloudWatch Events cron — the migration CloudWatch Events (the legacy name for EventBridge) supports cron rules that trigger Lambda or other targets. EventBridge Scheduler is the modern replacement with two key improvements: it supports one-time schedules (schedule exactly one invocation at a specific timestamp — useful for "send this reminder in 24 hours"), and it has a dedicated retry policy with DLQ rather than the "best effort" delivery of CloudWatch Events cron. For new scheduled workloads, use Scheduler. For existing CloudWatch cron rules — migrate opportunistically when you're touching the infrastructure anyway.

Event-driven architecture patterns
How SQS, SNS, and EventBridge compose into the event-driven patterns that senior engineers are expected to design — outbox pattern, saga choreography, and event sourcing fundamentals.

The transactional outbox pattern solves the dual-write problem: how do you atomically write to a database AND publish an event to SQS/SNS/EventBridge? If you write to the database first and then publish the event, the publish can fail — your database has the record but no event was published, and downstream services never process it. If you publish first and then write to the database, the write can fail — an event was published for a record that doesn't exist.

The outbox solution: Write to the database AND write to an "outbox" table in the same database transaction. An outbox poller (a separate thread or process) reads from the outbox table and publishes events to SQS/SNS. The outbox entry is deleted after successful publishing. This uses the database's ACID transaction as the atomicity guarantee — either both the record and the outbox entry are committed, or neither is.

Java — transactional outbox in Spring Boot
// In the service layer — one transaction writes both the record and outbox entry
@Transactional
public Job createJob(CreateJobRequest request) {
    Job job = jobRepository.save(new Job(request));

    // Write outbox entry in the SAME transaction
    OutboxEvent outboxEvent = OutboxEvent.builder()
        .aggregateId(job.getId())
        .eventType("JOB_CREATED")
        .payload(objectMapper.writeValueAsString(job))
        .status(OutboxStatus.PENDING)
        .createdAt(Instant.now())
        .build();
    outboxRepository.save(outboxEvent);

    // Both committed atomically, or both rolled back on failure
    return job;
}

// Outbox poller — runs every 5 seconds in a separate @Scheduled thread
@Scheduled(fixedDelay = 5000)
public void pollAndPublishOutbox() {
    List<OutboxEvent> pending = outboxRepository.findByStatus(OutboxStatus.PENDING);
    for (OutboxEvent event : pending) {
        try {
            snsClient.publish(topicArn, event.getPayload());
            outboxRepository.delete(event);
        } catch (Exception e) {
            // Leave in PENDING — will retry on next poll cycle
            log.error("Failed to publish outbox event {}", event.getId(), e);
        }
    }
}
When to use the outbox pattern vs direct publishing Direct publish (write to DB, then publish to SNS) is sufficient for most use cases where losing an event is acceptable — analytics events, logging, non-critical notifications. The outbox pattern is required when: missing an event causes data inconsistency, financial transactions where downstream services (accounting, invoicing) must receive every event, compliance events that must be auditable. For GetWork: job application events → use outbox (missing one could mean a worker's application is never processed). Job view analytics events → direct publish is fine (missing a view count event is acceptable).

The saga pattern manages distributed transactions across multiple services — each step of a multi-service workflow is triggered by an event, and failures trigger compensating events that undo previous steps. Choreography means no central coordinator — services react to events and emit their own events.

GetWork payment saga — choreography via EventBridge
1. API publishes
→ JobAccepted event → EventBridge
2. Payment service
→ receives JobAccepted → charges employer → publishes PaymentSucceeded OR PaymentFailed
3. Worker service
→ receives PaymentSucceeded → confirms worker booking → publishes BookingConfirmed

Each service is autonomous — knows only its event in/event out. No service calls another directly. Failure in payment triggers compensation events. EventBridge routes each event to the appropriate service's SQS queue.
Choreography complexity — the hidden coupling Choreography appears decoupled but creates implicit coupling through event contracts. When the payment service changes its PaymentSucceeded event schema (adds a field, renames a field), every downstream service that consumes it must be updated. With many services in a saga, tracing why a booking was never confirmed requires reconstructing the event flow across multiple service logs. EventBridge's event archive feature (store all events for replay and investigation) is essential for debugging choreographed sagas in production. Enable archive on your custom event bus from day one.

Event sourcing stores state as a sequence of events rather than as the current value. Instead of UPDATE jobs SET status='FILLED' WHERE id=123, you append an event JobFilled{jobId:123, workerId:42, timestamp:...} to an append-only event log. The current state is derived by replaying the event sequence.

Why it matters for messaging: In a pure event-sourced system, SQS/SNS/EventBridge carry the events. The event log (typically DynamoDB Streams or Kinesis) is the source of truth. Services subscribe to the event log and maintain their own read models (materialized views). A new service can replay the entire event history to build its initial state without needing a data migration.

For GetWork, full event sourcing is overkill. But the pattern is worth understanding for the interview: event sourcing provides complete auditability (every state change is recorded as an event with who/what/when), time travel (replay events up to any point in time), and decoupled projections (multiple views of the same data without schema coupling). The complexity cost is significant — event schema evolution, snapshot management, and eventual consistency make it harder to build and reason about than CRUD with direct database writes.

Interview prep
SQS/SNS/EventBridge questions test architectural decision-making and failure mode awareness. They're almost always phrased as design scenarios, not "what is X."
Q1 Your Spring Boot service processes job applications from an SQS queue. After a deployment, you notice the same application is being processed twice, creating duplicate records. What caused this and how do you fix it? Hard ▼

The root cause is almost certainly a visibility timeout misconfiguration combined with non-idempotent processing logic.

Diagnosing the visibility timeout issue: The deployment introduced a slower code path — perhaps a new external API call (Aadhaar verification, payment gateway). Processing now takes longer than the visibility timeout. When processing exceeds the timeout, the message becomes visible again. Another consumer picks it up and starts processing. The original consumer finishes and deletes the message. The second consumer also writes a duplicate record and then tries to delete the same message (getting a harmless error since it was already deleted). You have one application processed twice.

Immediate fix — extend the visibility timeout: Set it to at least 6× the worst-case processing time. If the new code path can take up to 30 seconds, set the timeout to 180 seconds. Use ChangeMessageVisibility to extend dynamically for messages that are taking longer than expected. Monitor the ApproximateNumberOfMessagesNotVisible metric — if it consistently grows, consumers are hitting the timeout.

Permanent fix — implement idempotent processing: The visibility timeout fix reduces duplicates but doesn't eliminate them (SQS Standard delivers at-least-once regardless). The correct permanent solution: before creating the application record, check whether a record with the same messageId already exists. Store the messageId alongside the application record. If it exists, return success (to delete from queue) without writing a duplicate. This makes the consumer idempotent — processing the same message twice produces exactly one record.

Additional monitoring: Add a CloudWatch metric for duplicate detections. If idempotency checks fire more than 0.1% of the time, investigate whether the visibility timeout is still too short or whether the application is restarting and re-processing messages from its startup batch.

Why this works: identifies two distinct problems (visibility timeout causing re-delivery, and non-idempotent code causing duplicate writes), fixes both at the correct layer, and adds monitoring to detect if the fix is holding.
Q2 When would you use SNS + SQS fan-out instead of just publishing directly to multiple SQS queues? Medium ▼

The answer is about coupling and extensibility, not just current functionality.

Direct multi-queue publishing: The publisher knows about every queue it needs to write to. sqsClient.sendMessage(matchingQueue); sqsClient.sendMessage(notificationQueue); sqsClient.sendMessage(analyticsQueue);. Adding a new consumer requires changing the publisher code, redeploying it, and coordinating the deployment. If publishing to one queue fails (permissions error, throttling), you have partial delivery — some consumers got the event, others didn't. The publisher must implement retry logic for each queue independently.

SNS + SQS fan-out: The publisher knows about one SNS topic. It publishes once. SNS delivers to all subscribed SQS queues atomically — either all get the message or none do (within SNS's delivery guarantee). Adding a new consumer means adding a new SQS subscription to the SNS topic — zero change to the publisher code, zero redeployment. A new team can subscribe their SQS queue to the topic without involving the publishing team at all. This is genuine decoupling — the publisher has no knowledge of who consumes its events.

When direct multi-queue is acceptable: if the number of consumers is small, stable, and within your team's control (two queues that you own, unlikely to change). If SNS is adding latency you can't afford (rare — SNS adds ~10ms typically). If you need per-destination ordering (SNS doesn't preserve message order across destinations).

Why this works: frames the comparison around coupling and operational consequences (who must change when a new consumer is added), not just feature list.
Q3 Design the event-driven architecture for GetWork's job posting flow: employer posts a job, and multiple services need to respond (match workers, send notifications, update analytics, and trigger billing). Hard ▼

This is a fan-out + saga scenario. I'd use a combination of SNS fan-out for parallel independent processing and EventBridge for the orchestrated billing saga.

The synchronous API layer: The Spring Boot POST /jobs endpoint does two things: saves the job to RDS in a transaction that also writes to the outbox table, and returns 201 Created to the employer. This is the only synchronous part — the employer sees an immediate response.

The outbox poller publishes to SNS: The outbox poller picks up the JobCreated outbox entry and publishes it to an SNS topic. This guarantees the event is published even if the poller restarts or SNS has a temporary hiccup — the outbox retries until successful. The SNS fan-out:

→ SQS worker-matching-queue: Worker matching service consumes this queue. Runs the matching algorithm, identifies the top 20 available workers for this job. At-least-once is fine here — if matching runs twice, the second run produces the same matches (idempotent). Visibility timeout: 5 minutes (matching is CPU-intensive).

→ SQS notification-queue: Notification service sends push notifications to matched workers. Idempotent via FCM message deduplication ID. Short visibility timeout: 30 seconds (push notification is fast).

→ SQS analytics-queue: Analytics ingestion. At-least-once is fine — duplicate events in analytics is acceptable. No DLQ needed for analytics (data loss is acceptable; duplicate is not a problem).

The billing saga via EventBridge: When the matching service successfully matches workers, it publishes WorkersMatched to the EventBridge custom bus. EventBridge routes this to the billing service's SQS queue. The billing service charges the employer's credit card and publishes either BillingSucceeded or BillingFailed. On BillingFailed: EventBridge routes to the matching service which marks the job matches as cancelled, and to the notification service which sends a "job not confirmed" message to workers.

Key design decisions: Outbox pattern for the initial event to guarantee delivery. SNS fan-out for parallel, independent processing. DLQ on every SQS queue with alarm. Idempotency at the consumer level using MessageId. EventBridge for the billing saga because it provides rich event routing, retry policies, and audit trail for financial events.

Why this works: uses multiple patterns correctly (outbox, fan-out, saga), explains why each service uses a different SQS configuration (visibility timeout, DLQ), and addresses the financial transaction requirement with EventBridge specifically.
Q4 What is the difference between SQS and Kafka? When would you use one over the other? Medium ▼

SQS and Kafka are both message queuing systems but with fundamentally different designs that suit different use cases.

SQS is a consumption queue: Messages are received, processed, and deleted. Once a message is consumed and deleted, it's gone — you can't replay it. Multiple consumer groups reading the same SQS queue compete for messages (each message goes to one consumer). SQS is fully managed — no servers, no broker configuration, no partition management. It scales automatically and is available in every AWS region and account with zero setup.

Kafka is a distributed commit log: Messages are retained for a configurable period (days, weeks, indefinitely) regardless of consumption. Multiple consumer groups each maintain their own offset and read independently — consumer group A and consumer group B both receive every message without competing. Kafka is ideal for event sourcing (replay the log to rebuild state), event streaming (real-time analytics with Kafka Streams), and audit trails (every event is permanently recorded). Kafka requires operational management — brokers, partitions, replication, consumer group lag monitoring — unless you use AWS MSK (managed Kafka).

For your JP Morgan project: Your Kafka usage (from your experience at Deloitte) is for the real-time payment processing pipeline — high throughput, strict ordering per payment entity, message replay capability for reconciliation. SQS is for the simpler async job queuing (background notifications, image processing, email sending) where replay isn't needed and operational simplicity matters. Knowing both tools and the correct use case for each is what makes you valuable — "use Kafka for everything" and "use SQS for everything" are both wrong.

Why this works: the commit log vs consumption queue distinction is the conceptual core. The replay capability as the key differentiator. The JP Morgan connection shows you've applied both in practice.

Day 19 cheatsheet
SQS, SNS, and EventBridge production mechanics at a glance.
SQS at-least-once
Standard queues may deliver the same message twice. Consumer code must be idempotent. Use MessageId as the idempotency key in a DynamoDB conditional write.
Visibility timeout
Set to 6× worst-case processing time. If processing exceeds timeout, message becomes visible again → another consumer picks it up → duplicate processing. Extend dynamically with ChangeMessageVisibility.
Long polling always
WaitTimeSeconds=20 on all ReceiveMessage calls. Reduces empty API calls from hundreds/minute to 3/minute for idle queues. Saves cost and CPU. spring.cloud.aws.sqs.listener.poll-timeout-seconds=20
DLQ maxReceiveCount
Set to 3–5 for most queues. After this many failed receives, message routes to DLQ. Alarm: DLQ depth > 0. DLQ retention: 14 days. Source queue: 4 days.
FIFO deduplication window
5-minute window only. Same deduplication ID sent 6 minutes apart = two deliveries. For permanent exactly-once semantics: implement application-level idempotency, not just FIFO.
SNS raw delivery
Enable RawMessageDelivery=true on SQS subscriptions. Without it, SQS message body is SNS JSON envelope. Consumers must unwrap. Raw delivery puts your message directly in SQS body.
SNS filter → silent drop
If no subscriber's filter matches a published message, it's silently discarded. No error returned to publisher. Always have one catch-all subscriber (no filter) for audit purposes.
SQS policy for SNS/EB
SQS queue must have a resource policy allowing SNS or EventBridge to SendMessage. Without it: SNS/EB publish succeeds, SQS silently rejects, messages disappear. Common misconfiguration.
EventBridge default bus
AWS services publish natively: EC2, RDS, ECS, CodePipeline, GuardDuty, 90+ more. Already flowing in your account — create rules to react. Free to receive, cost per matched event forwarded to targets.
EventBridge InputTransformer
Reshape event before delivery to target. Extract only needed fields. Reduces message size, simplifies consumer code. Best practice: consumers should not need to parse raw AWS event envelopes.
Transactional outbox
Write event to outbox table in the same DB transaction as the record. Poller publishes from outbox to SNS/SQS. Guarantees no event is lost even if publish fails. Required for financial-critical events.
Fan-out vs direct publish
SNS fan-out: publisher writes once, zero knowledge of consumers, add consumers without touching publisher. Direct multi-queue: tight coupling, partial delivery risk on failure, requires publisher code change for new consumers.
Production rules for messaging
1Every SQS consumer must be idempotent. At-least-once delivery is the contract — processing the same message twice must produce the same outcome as processing it once.
2Visibility timeout = 6× worst-case processing time. Monitor ApproximateNumberOfMessagesNotVisible. Rising count = consumers are hitting the timeout.
3Every production SQS queue must have a DLQ configured. Alarm on DLQ depth > 0. DLQ messages require human investigation and action.
4Always use long polling (WaitTimeSeconds=20). Never short poll in production — it wastes API calls, CPU, and money.
5Enable RawMessageDelivery on all SNS → SQS subscriptions. Consumers should not be responsible for unwrapping SNS envelopes.
6Set explicit SQS queue resource policies for SNS and EventBridge senders. The default deny policy silently drops messages — there is no error visible to the sender.
7Use the transactional outbox pattern for events tied to database writes. Direct publish (write then publish) risks message loss if publish fails. The outbox guarantees delivery.
8EventBridge rule patterns must be as specific as possible. Overly broad rules (source: aws.ec2 with no detail filter) fire on every EC2 event, flooding Lambda logs and increasing costs.
9Alarm on ApproximateAgeOfOldestMessage on source queues in addition to depth. A single stuck message from an hour ago needs attention even if the queue only has 1 message total.
10Enable EventBridge event archive on custom event buses. Required for debugging choreographed sagas — without it, tracing a failed workflow across multiple services is nearly impossible.
