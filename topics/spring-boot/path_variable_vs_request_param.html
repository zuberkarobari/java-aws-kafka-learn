Extracting HTTP Data

@PathVariable vs @RequestParam

Mastering the differences between URL Path mapping, Query Strings, and RESTful design conventions.

1. Definition

Simple English

Imagine looking for a specific file in a filing cabinet. @PathVariable is the exact folder name pointing to the file (e.g., folder #123). @RequestParam acts like a search filter applied to that folder (e.g., "only show me active files").

Technical

Both are parameter-binding annotations in Spring MVC. @PathVariable extracts values embedded directly within the URI path segments. @RequestParam extracts values from the query string (the part of the URL after the ? mark).

Purpose

They serve fundamentally different architectural roles. @PathVariable is used to uniquely identify a specific REST resource. @RequestParam is used to filter, sort, paginate, or modify the query for that resource.

2. Internal Working: URL Dissection

When a request arrives, the HandlerMethodArgumentResolver component is responsible for parsing the URL string, converting the string data into correct Java types (like Long or Integer), and injecting them into the method parameters.

Dissecting the HTTP Request

https://api.app.com /users/ 123 ? status=active
@PathVariable("id")
Extracts from: /users/{id}
Long id = 123;
@RequestParam("status")
Parses Query String after ?
String status = "active";

3. Real-World Project Example

E-Commerce Product Catalog

How these annotations handle different parts of a complex API query.

GET /api/products/electronics?sort=price&limit=10

@PathVariable

Extracts "electronics". This identifies the Core Resource we are querying. Without knowing the category, the request makes no sense. It is mandatory for routing.

@RequestParam

Extracts "price" and 10. These Modify the query. The client is filtering and organizing the resource. If they don't provide them, the API falls back to defaults.

4. Key Points

1. RESTful Conventions

Use PathVariables to identify a specific resource (/users/123). Use RequestParams for non-resource actions like sorting, filtering, and pagination (?page=2&sort=asc).

2. Missing Values Behavior

If a @PathVariable is missing, it changes the URI entirely, resulting in a 404 Not Found. If a required @RequestParam is missing, Spring throws a 400 Bad Request.

3. Making them Optional

Both support required = false. However, making a PathVariable optional is highly discouraged as it makes route mapping ambiguous.

4. Default Values

@RequestParam supports a defaultValue attribute (e.g., defaultValue="10"). @PathVariable does not support default values.

5. Dynamic Mapping

If you have multiple unknown query parameters, you can inject them all at once using @RequestParam Map<String, String> allParams.

6. Code Example

SearchController.java 
import org.springframework.web.bind.annotation.*;
import java.util.Optional;
import java.util.Map;

@RestController
@RequestMapping("/api/v1/search")
public class SearchController {

    /* 1. Standard Combination of Both
       URL: GET /api/v1/search/books?sortBy=price&limit=20 */
    @GetMapping("/{category}")
    public String search(
            // Identifies the resource group
            @PathVariable("category") String category, 
            
            // Optional filter with default
            @RequestParam(defaultValue = "relevance") String sortBy,
            
            // Optional filter
            @RequestParam(required = false) Integer limit) {
            
        return "Searching " + category + " sorted by " + sortBy;
    }

    /* 2. Handling Optional Path Variables (The safe way)
       URL: GET /api/v1/search/users  OR  /api/v1/search/users/123 */
    @GetMapping({"/users", "/users/{id}"})
    public String getUserOrAll(
            @PathVariable Optional<Long> id) {
        
        if (id.isPresent()) {
            return "Fetching user " + id.get();
        }
        return "Fetching all users";
    }
}

7. Senior Developer Answer

"The distinction between @PathVariable and @RequestParam goes far beyond Spring Boot mechanics; it dictates your API's alignment with RESTful design principles and directly impacts infrastructure caching.

I enforce a strict rule: if a parameter is required to uniquely identify a resource, it must be a Path Variable. Path variables are cache-friendly at the CDN layer (like Cloudflare or AWS CloudFront) because the URL itself represents a unique object.

Conversely, if a parameter is used to alter the state of the result—such as pagination, sorting, or complex filtering—it must be a Request Param. Furthermore, from a security and logging perspective, query strings are notoriously easy to log inadvertently in access logs. Therefore, sensitive data should never be passed in either, but rather secured inside the HTTP request body."
