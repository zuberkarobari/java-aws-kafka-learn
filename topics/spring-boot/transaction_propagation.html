Transaction Boundaries

Transaction Propagation

Understanding how Spring determines whether to join, suspend, or create new database transactions when methods call each other.

1. Definition

Simple English

When Method A calls Method B, and both are marked as `@Transactional`, what happens? Do they share the same transaction (if one fails, both fail)? Or does Method B get its own independent transaction? Propagation rules decide this.

Technical

Transaction Propagation defines the boundary logic for the PlatformTransactionManager. It dictates how Spring's AOP proxies should handle active connections/transactions bound to the current thread (via ThreadLocal) when transitioning between service layer methods.

Purpose

To provide fine-grained control over atomicity. For example, allowing an audit log to successfully commit to the database even if the parent business transaction fails and rolls back.

2. Internal Working: REQUIRED vs REQUIRES_NEW

Understanding Physical vs Logical transactions at the database connection level.

Default Behavior

REQUIRED

Method A
Physical Tx 1
Method B
Logical Tx inside Tx 1
Result: Spring uses the same database connection. They share one physical transaction. If B rolls back, A is also forced to roll back.
Independent

REQUIRES_NEW

Method A
Physical Tx 1 (SUSPENDED)
Method B
Physical Tx 2 (ACTIVE)
Result: Spring pauses Tx 1, acquires a second database connection, and starts Tx 2. Tx 2 commits independently. Then Tx 1 resumes.

3. Real-World Project Example

E-Commerce Checkout & Audit Logging

Why we need independent transaction scopes.

The Main Transaction (REQUIRED)

A user submits a checkout request. The CheckoutService.processOrder() starts a transaction. It decrements inventory and processes payment.

The Audit Log (REQUIRES_NEW)

During checkout, we call AuditService.logAttempt(). If this used the default REQUIRED, and the payment later failed, the entire transaction would roll back—meaning our audit log of the failed attempt would also be deleted from the DB! By using REQUIRES_NEW, the audit log commits immediately to its own connection. Even if the checkout throws an exception seconds later, the audit log remains safely stored.

4. 7 Key Points For senior-level depth

1

REQUIRED (Default)

If a transaction exists, join it. If not, create a new one. This is used in 95% of standard business logic cases.

2

REQUIRES_NEW Connection Cost

It halts the current transaction and acquires a new connection from the pool. If you iterate a loop calling a REQUIRES_NEW method 100 times, you are slamming your connection pool. Use with caution.

3

NESTED (Savepoints)

Uses a single physical transaction but creates a JDBC "Savepoint". If the nested method fails, it rolls back to the savepoint without killing the outer transaction. (Note: Not all SQL dialects support savepoints).

4

MANDATORY

Throws an exception if NO transaction is currently active. Use this on helper DAOs or private-like services that absolutely must not be invoked independently.

5

SUPPORTS

If a transaction exists, join it. If not, execute non-transactionally. Often used for read-only query methods that don't strictly require a transaction but can participate if one exists.

6

NOT_SUPPORTED & NEVER

NOT_SUPPORTED suspends the active transaction and executes non-transactionally. NEVER throws an exception if an active transaction exists.

7

The Self-Invocation Trap (Proxy Bypass)

Because Spring implements transactions via AOP Proxies around beans, if Method A calls Method B, and both are in the same class, the call to Method B bypasses the proxy. Therefore, whatever Propagation rule you put on Method B is completely ignored!

6. Code Example: Managing Scope

@Service
public class OrderService {

    private final InventoryService inventoryService;
    private final AuditLogService auditService;

    public OrderService(InventoryService inventoryService, AuditLogService auditService) {
        this.inventoryService = inventoryService;
        this.auditService = auditService;
    }

    // 1. Uses DEFAULT propagation (REQUIRED)
    @Transactional
    public void checkout(Order order) {
        
        // This method requires its own connection!
        // If checkout fails later, this audit log STILL COMMIT.
        auditService.logAttempt(order.getId(), "Checkout Started");

        try {
            // This method uses REQUIRED (joins our current transaction)
            inventoryService.deductStock(order.getItems());
            
        } catch (OutOfStockException e) {
            // DANGER! If deductStock is REQUIRED and threw an exception,
            // our transaction is now marked as "Rollback-Only".
            // Catching it here and continuing will cause an UnexpectedRollbackException
            // when this checkout() method finishes!
            // Fix: deductStock must use NESTED or REQUIRES_NEW if we want to swallow the error.
            throw e; 
        }
    }
}

@Service
public class AuditLogService {
    
    // 2. Forces a new physical database connection. 
    // Original transaction is suspended until this finishes.
    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void logAttempt(Long orderId, String msg) {
        // Save log to database...
    }
}

7. Senior Developer Answer

"Understanding propagation is really about understanding database connections. The default REQUIRED is perfectly fine for 90% of business logic because we usually want all-or-nothing atomicity using a single connection.

However, as a senior, I pay extreme attention when a junior developer uses REQUIRES_NEW. While it's great for audit logging, it requires acquiring a second connection from the connection pool (like HikariCP) while holding onto the first. Under high concurrency, this can cause a deadlock where Thread A holds Connection 1 waiting for Connection 2, and Thread B holds Connection 2 waiting for Connection 1, instantly starving the pool.

I also heavily review try-catch blocks inside transactional services. If an inner method participating in a REQUIRED transaction throws an exception, the TransactionInterceptor marks the shared transaction as 'rollback-only'. If the outer method catches that exception and attempts to silently continue, Spring throws an UnexpectedRollbackException at commit time. Developers must understand this logical vs physical transaction boundary to prevent production bugs."

8. Quick Revision Notes

REQUIRED: Join existing or create new. Shares 1 connection. (Default)
REQUIRES_NEW: Suspend current, open a new independent transaction. Uses 2 connections!
NESTED: Uses savepoints within the same connection. Can rollback independently.
MANDATORY: Must be called inside a transaction, else throws Exception.
SUPPORTS: Joins if exists, otherwise runs without one.
NOT_SUPPORTED: Suspends existing, runs non-transactionally.
NEVER: Throws Exception if a transaction exists.
Rollback Trap: Catching an error from a nested REQUIRED method causes `UnexpectedRollbackException`.

9. Interview Answer Template

"Transaction Propagation in Spring determines how boundaries are handled when one transactional method calls another. The default is REQUIRED, which means the inner method joins the outer method's existing database connection and physical transaction. If the inner method fails, the entire transaction is marked for rollback.

If we need independent atomicity—for instance, an audit log that must commit even if the parent business logic fails—we use REQUIRES_NEW. This forces Spring to suspend the current transaction, grab a new database connection from the pool, execute and commit the inner method, and then resume the original transaction.

It's crucial to remember that because propagation relies on Spring AOP proxies, self-invocation (calling a method within the same class) bypasses the proxy entirely, causing the propagation rules to be completely ignored."
