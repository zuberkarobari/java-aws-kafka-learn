Stateless Security

JSON Web Tokens

Mastering stateless authentication, cryptographic signatures, custom security filters, and the "Revocation Problem".

1. Definition

Simple English

It's a digital VIP wristband. You show your ID once (username/password), the server gives you an unforgeable wristband (JWT). For the rest of your visit, the server just checks your wristband instead of asking for your ID again.

Technical

JSON Web Token (RFC 7519) is an open standard that defines a compact, self-contained way of securely transmitting information between parties as a JSON object. This information is verified and trusted because it is cryptographically signed.

Purpose

To achieve stateless scalability. By embedding the user's identity and roles directly into a signed token, the server doesn't need to store session states in memory or query the database on every single API request.

2. Internal Working

In Spring Security, JWT implementation requires intercepting the HTTP request before the standard authorization filters. Here is the full lifecycle.

The Two Phases of JWT

1. Token Issuance (Login)

1. Client POSTs {user, pass}
2. AuthManager verifies in DB
3. JwtService generates JWT
Header + Payload + Signature
4. Return JSON {"token": "eyJ..."}

2. Token Verification (API Call)

1. Client GET /api/data
Authorization: Bearer eyJ...
2. JwtAuthFilter intercepts.
Extracts token, checks mathematically if Signature is valid & Not Expired.
3. SecurityContext
Filter saves UserDetails in ThreadLocal context.
4. DispatcherServlet processes logic.

3. Real-World Project Example

Microservices Architecture

Why JWTs are the industry standard for distributed systems.

The Old Way: Stateful Sessions

Imagine you have 5 Order microservices. If you use Sessions (JSESSIONID), Service A stores your session in memory. If the load balancer routes your next request to Service B, Service B doesn't know who you are. You must set up a shared Redis cluster just to store sessions, adding network latency and infrastructure cost.

The JWT Way: Decentralized Trust

An API Gateway or Auth Service verifies your login and issues a JWT signed with a Private Key. You send that token to the Order Service. The Order Service does not call the database and does not call the Auth Service. It simply uses a Public Key to mathematically verify the signature. If the math checks out, it inherently trusts the token and processes your order. Infinite scalability.

4. 7 Key Points For senior-level depth

1

JWT Anatomy

It consists of three parts separated by dots: Header.Payload.Signature. The Header and Payload are just Base64Url encoded JSON. The Signature is the cryptographic hash of the first two parts.

2

Signed, Not Encrypted

Standard JWTs (JWS) are signed, not encrypted. Anyone who intercepts the token can decode the Base64 payload and read its contents. Never put passwords or sensitive PII inside a JWT payload.

3

HS256 vs RS256

HS256 (Symmetric): Uses one secret key to both sign and verify. Faster, but if the key leaks, the system is compromised. RS256 (Asymmetric): Uses a Private key to sign, and a Public key to verify. Essential for microservices.

4

The Revocation Problem

Because JWT verification is mathematically stateless, if an admin bans a user, their valid JWT will continue to work until it naturally expires. You cannot instantly "logout" a stateless token.

5

Access & Refresh Tokens

To mitigate the Revocation Problem, Access Tokens must be short-lived (e.g., 15 mins). When they expire, the client uses a secure, long-lived Refresh Token (stored in HttpOnly cookies) to ask the Auth server for a new Access Token.

6

OncePerRequestFilter

In Spring, the custom JWT filter must extend OncePerRequestFilter and be injected into the SecurityFilterChain before the UsernamePasswordAuthenticationFilter.

7

Stateless Configuration

You must explicitly configure Spring Security to not create HTTP sessions: sessionManagement.sessionCreationPolicy(SessionCreationPolicy.STATELESS). If you forget this, Spring will fallback to JSESSIONID cookies, defeating the purpose of JWT.

6. Code Example: The JWT Filter

JwtAuthenticationFilter.java
Copied! 
import jakarta.servlet.FilterChain;
import org.springframework.security.core.context.SecurityContextHolder;
import org.springframework.security.authentication.UsernamePasswordAuthenticationToken;
import org.springframework.web.filter.OncePerRequestFilter;
import org.springframework.stereotype.Component;

@Component
public class JwtAuthenticationFilter extends OncePerRequestFilter {

    private final JwtService jwtService;
    private final UserDetailsService userDetailsService;

    // Constructor injection omitted for brevity

    @Override
    protected void doFilterInternal(
            HttpServletRequest request,
            HttpServletResponse response,
            FilterChain filterChain
    ) throws ServletException, IOException {
        
        // 1. Extract Header
        final String authHeader = request.getHeader("Authorization");
        final String jwt;
        final String username;

        // 2. Check if valid Bearer token exists
        if (authHeader == null || !authHeader.startsWith("Bearer ")) {
            filterChain.doFilter(request, response);
            return;
        }

        // 3. Extract Token and Username
        jwt = authHeader.substring(7);
        username = jwtService.extractUsername(jwt); // Throws exception if signature is bad

        // 4. If user isn't already authenticated in this thread
        if (username != null && SecurityContextHolder.getContext().getAuthentication() == null) {
            
            // Load user from DB (or optionally directly from token claims to avoid DB hit)
            UserDetails userDetails = this.userDetailsService.loadUserByUsername(username);

            // 5. Validate Expiration and Username match
            if (jwtService.isTokenValid(jwt, userDetails)) {
                
                // 6. Create Authentication Object
                UsernamePasswordAuthenticationToken authToken = new UsernamePasswordAuthenticationToken(
                        userDetails,
                        null, // No credentials needed anymore
                        userDetails.getAuthorities()
                );
                
                // 7. Save to Security Context
                SecurityContextHolder.getContext().setAuthentication(authToken);
            }
        }
        
        // 8. Continue down the filter chain
        filterChain.doFilter(request, response);
    }
}

7. Senior Developer Answer

"JWT is the backbone of modern, distributed stateless architectures. By embedding user identity and authorities directly into a cryptographically signed payload, we entirely eliminate the need for server-side HTTP Sessions and Redis-backed session clustering.

However, statelessness introduces the 'Revocation Problem'. Because the token is mathematically verified, it remains valid until it expires, making immediate logout or user banning difficult. To mitigate this in enterprise systems, I implement a dual-token strategy: a short-lived Access Token (e.g., 10 minutes) and a long-lived Refresh Token stored securely in an HttpOnly cookie. For immediate revocation, I maintain a high-performance JWT Blacklist in Redis.

Furthermore, for microservices, I transition away from symmetric HS256 signatures and use asymmetric RS256 (Public/Private Key pairs). The Auth Server signs the token with a Private Key, and all downstream microservices verify it using a publicly distributed Public Key (usually fetched via a JWKS endpoint). This guarantees zero-trust security without sharing secrets across teams."

8. Quick Revision Notes

Structure: Header, Payload (Claims), Signature. Joined by dots.
Encoding: Base64Url encoded. NOT encrypted. Do not store secrets.
Stateless: Server stores no session. Validation is purely mathematical.
Symmetric (HS256): One secret key for signing and verifying.
Asymmetric (RS256): Private key signs, Public key verifies. Safe for microservices.
Refresh Tokens: Used to obtain new Access Tokens without forcing re-login.
Filter: Extending OncePerRequestFilter ensures it fires precisely once per API call.
Revocation: Handled via short lifespans or a Redis Blacklist.

9. Interview Answer Template

"A JSON Web Token (JWT) is an open standard for securely transmitting identity and authorization claims between a client and a server in a stateless manner.

In a Spring Boot application, instead of relying on server-side HTTP sessions, the server verifies the user's login credentials once and issues a JWT. This token contains a Base64-encoded header, a payload containing user claims like username and roles, and a cryptographic signature. On subsequent requests, the client sends this token in the Authorization header.

We intercept this request using a custom OncePerRequestFilter placed early in the SecurityFilterChain. The filter mathematically verifies the signature using a Secret or Public Key. If valid, it extracts the user details and populates the SecurityContextHolder, granting access. This stateless approach is highly scalable and perfectly suited for microservice architectures."
