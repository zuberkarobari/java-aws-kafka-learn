The Presentation Layer

@RestController

Building modern REST APIs, mastering content negotiation, and the magic of HttpMessageConverters.

1. Definition

Simple English

Normally, a Spring Controller tries to hand back a web page (like an HTML file). @RestController acts like a drive-thru window: it doesn't give you a dining room (HTML page), it just hands you raw food (data, usually in JSON format) directly.

Technical

It is a convenience annotation that combines @Controller and @ResponseBody. It marks the class as a web controller and instructs the DispatcherServlet to bypass the View Resolver entirely.

Purpose

To drastically simplify the creation of RESTful web services. It completely separates the frontend (React, Angular, Mobile Apps) from the backend by strictly serving data APIs over HTTP.

2. Internal Working: Request Flow

When an HTTP request hits a Spring Boot application, it is intercepted by the Front Controller (DispatcherServlet). Here is how a @RestController processes that request.

REST Request Lifecycle

Client GET /api/users/1
DispatcherServlet
Intercepts request, uses HandlerMapping
@RestController Method
Executes business logic. Returns Java Object (e.g., UserDTO)
HttpMessageConverter (Jackson)
Bypasses ViewResolver. Converts Java Object to JSON string.
{"id": 1, "name": "John"} HTTP 200 OK

3. Real-World Project Example

User Profile Service API

Traditional MVC vs Modern REST approach.

The Old Way (@Controller)

@GetMapping("/profile")
public String getProfile(Model m) {
  m.addAttribute("user", user);
  return "profile-view";
}

Returns a String. The framework searches for an HTML/JSP file named profile-view.html, renders it with the data on the server, and sends a bulky HTML page to the browser.

The Modern Way (@RestController)

@GetMapping("/api/profile")
public UserDTO getProfile() {
  return new UserDTO(user);
}

Returns a pure Java Object. Spring intercepts this, converts it directly to a JSON string {"name":"John", "age":30}, and sends it to the Mobile App or React frontend.

4. Key Points

1. A Convenience Annotation

@RestController is literally just @Controller and @ResponseBody bundled together. You could build a REST API using @Controller if you explicitly put @ResponseBody on every single method.

2. HttpMessageConverters

When @ResponseBody is present, Spring loops through its registered HttpMessageConverters. If it sees Jackson on the classpath, it uses MappingJackson2HttpMessageConverter.

3. Content Negotiation

The controller can return JSON or XML based on what the client asks for. Spring looks at the HTTP Accept header (e.g., Accept: application/xml) and picks the appropriate converter.

4. ResponseEntity Wrapper

Instead of returning pure objects, you should return ResponseEntity<T>. This allows you to explicitly control the HTTP Status Code (e.g., 201 Created) and add custom HTTP headers.

5. DTOs over Entities

Never return JPA Entities directly from a @RestController. This causes LazyInitializationException, exposes sensitive database columns (like passwords), and tightly couples the DB schema to the API contract. Always map to a Data Transfer Object (DTO) first.

6. Code Example

UserController.java 
import org.springframework.web.bind.annotation.*;
import org.springframework.http.ResponseEntity;
import org.springframework.http.HttpStatus;

@RestController // = @Controller + @ResponseBody
@RequestMapping("/api/v1/users")
public class UserController {

    private final UserService userService;

    public UserController(UserService userService) {
        this.userService = userService;
    }

    /* GET - Fetch a resource */
    @GetMapping("/{id}")
    public ResponseEntity<UserDTO> getUser(@PathVariable Long id) {
        UserDTO user = userService.findUserById(id);
        return ResponseEntity.ok(user); // HTTP 200
    }

    /* POST - Create a resource */
    @PostMapping
    public ResponseEntity<UserDTO> createUser(@RequestBody UserCreateRequest request) {
        UserDTO createdUser = userService.createUser(request);
        // Returns HTTP 201 CREATED instead of 200
        return ResponseEntity.status(HttpStatus.CREATED).body(createdUser);
    }
}

7. Senior Developer Answer

"While @RestController removes the boilerplate of adding @ResponseBody to every method, using it effectively in an enterprise application requires strict architectural discipline. First, a Controller should contain absolutely zero business logic; it exists solely to translate HTTP semantics into Java domain language. Second, we never expose JPA Entities through a RestController. This tightly couples the API contract to the DB schema and causes LazyInitializationExceptions. I always map Entities to distinct DTOs or Java Records. Finally, I wrap my return objects in ResponseEntity to ensure our API accurately communicates using proper HTTP status codes."
