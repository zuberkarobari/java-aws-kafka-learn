Global Error Handling

@RestControllerAdvice

Mastering AOP-driven, centralized exception handling to build resilient and user-friendly REST APIs.

1. Definition

Simple English

Imagine an office where every worker (Controller) can panic when something goes wrong. @RestControllerAdvice is the dedicated crisis manager who catches all those panics, organizes them into a clean, polite letter, and hands it to the client.

Technical

It is a composed annotation (@ControllerAdvice + @ResponseBody) based on Aspect-Oriented Programming (AOP). It intercepts exceptions thrown globally by any @RequestMapping method and serializes the returned error object directly into JSON.

Purpose

To separate business logic from error handling. It eliminates massive try-catch blocks scattered across controllers, preventing raw stack traces from leaking to the frontend while ensuring a consistent JSON error contract.

2. Internal Working: Exception Resolution

When a controller throws an exception, Spring's DispatcherServlet catches it and delegates it to a chain of HandlerExceptionResolvers. The primary one that handles @ExceptionHandler methods is the ExceptionHandlerExceptionResolver.

The Exception Interception Flow

3. Real-World Project Example

Banking API - Funds Transfer

Comparing business logic with and without global advice.

The Bad Way

@PostMapping("/transfer") public ResponseEntity<?> transfer() { try { bankService.transfer(amount); return ResponseEntity.ok("Done"); } catch (InsufficientFundsException e) { return ResponseEntity.badRequest().body(new Error("No Money")); } catch (AccountLockedException e) { return ResponseEntity.status(403).body(new Error("Locked")); } }

Controller is 80% error handling. Hard to maintain and copy-pasted everywhere.

The Clean Way

@PostMapping("/transfer") public ResponseEntity<String> transfer() { // Pure, happy-path business logic bankService.transfer(amount); return ResponseEntity.ok("Done"); }

If bankService throws, the method exits. The @RestControllerAdvice class catches it globally and translates it into standard JSON.

4. Key Points

1. AOP Proxying

It operates via Spring AOP. It weaves cross-cutting exception-handling logic around all methods annotated with @RequestMapping.

2. The Matcher Method

Inside the advice class, you define methods annotated with @ExceptionHandler(MyException.class). Spring uses the exception hierarchy to find the closest matching method.

3. Validation Errors

It perfectly pairs with JSR-380 (Hibernate Validator). You can intercept MethodArgumentNotValidException to extract field-level validation errors to return cleanly to the client.

4. Global Catch-All

Always include a fallback method annotated with @ExceptionHandler(Exception.class) to catch unexpected errors and prevent Tomcat's ugly default 500 HTML page from bleeding out.

5. Spring Boot 3: Problem Details

In Spring 3.x, you don't need to build custom ErrorDTOs anymore. You can return Spring's native ProblemDetail object, which implements the RFC 7807 standard for error reporting in HTTP APIs, providing fields like type, title, status, and detail automatically.

6. Code Example

GlobalExceptionHandler.java 
import org.springframework.http.HttpStatus;
import org.springframework.http.ProblemDetail;
import org.springframework.web.bind.annotation.ExceptionHandler;
import org.springframework.web.bind.annotation.RestControllerAdvice;

@RestControllerAdvice
public class GlobalExceptionHandler {

    /* 1. Handling Custom Business Exceptions */
    @ExceptionHandler(ResourceNotFoundException.class)
    public ProblemDetail handleNotFound(ResourceNotFoundException ex) {
        // Using Spring Boot 3 RFC-7807 standard ProblemDetail
        ProblemDetail pd = ProblemDetail.forStatusAndDetail(
                HttpStatus.NOT_FOUND, ex.getMessage());
        pd.setTitle("Resource Not Found");
        return pd;
    }

    /* 2. The Catch-All Fallback (Security: Hide stack traces) */
    @ExceptionHandler(Exception.class)
    public ProblemDetail handleGenericException(Exception ex) {
        // Log the actual exception for developers securely
        // log.error("Unhandled Exception caught: ", ex);
        
        // Return a generic, safe message to the client
        return ProblemDetail.forStatusAndDetail(
                HttpStatus.INTERNAL_SERVER_ERROR, 
                "An unexpected server error occurred.");
    }
}

7. Senior Developer Answer

"In an enterprise Spring Boot application, scattering try-catch blocks throughout the Controller or Service layers is a major anti-pattern. I strictly utilize @RestControllerAdvice to implement global, AOP-driven exception handling. By throwing domain-specific RuntimeExceptions inside the business logic, the code reads cleanly like a 'happy path'.

Starting with Spring Boot 3, I've moved away from building custom ErrorDTOs and now exclusively return Spring's native ProblemDetail object. This guarantees that our APIs comply with the RFC 7807 standard. Finally, a catch-all Exception.class handler is mandatory in production to ensure raw Java stack traces never leak to the client, preventing critical security vulnerabilities."
