Object Lifespans

Spring Bean Scopes

Understanding Singleton, Prototype, Request, Session, and the intricacies of the @Scope annotation.

1. Definition

Simple English

When you ask a library for a book, do they give you the only copy they have (Singleton), or do they print a brand new copy just for you every time you ask (Prototype)? Bean Scope defines this behavior in Spring.

Technical

The @Scope annotation dictates the lifecycle and visibility of a bean instance managed by the IoC Container. It controls whether Spring caches a single instance, creates new instances per injection, or ties instances to specific web boundaries (like an HTTP Request).

Purpose

To optimize memory and ensure thread safety. By defaulting to Singleton, Spring saves memory. By allowing Prototype or Web scopes, Spring accommodates stateful objects that cannot be shared across multiple concurrent threads.

2. Internal Working: Cache Mechanics

Spring handles scopes by managing how it interacts with its internal registries (caches) during the injection phase.

3. Real-World Project Example

E-Commerce Shopping App

When to use which scope in a single architecture.

Singleton: ProductService

Fetching products from the DB doesn't change per user. The logic is stateless. We only need one instance of this service for the entire application to save memory.

Session Scope: ShoppingCart

If the ShoppingCart was a Singleton, User A adding an item would appear in User B's cart! By making it @SessionScope, Spring gives User A and User B their own unique cart instance tied to their browser session.

Prototype Scope: EncryptionCipher

An encryption tool that holds a unique, stateful "salt" string while encrypting a specific file. It cannot be shared. Every time a class asks for a Cipher, Spring creates a brand new, clean instance.

4. 7 Key Points For senior-level depth

1

The 6 Core Scopes

Singleton (default), Prototype, Request, Session, Application (ServletContext), and WebSocket.

2

Statefulness Rule

Singletons must be completely stateless (no mutable instance variables). If a bean needs to hold mutable state, it must be Prototype or a Web Scope to prevent race conditions.

3

Prototype Destruction

Spring does not manage the complete lifecycle of a Prototype bean. It instantiates it, injects it, and then hands it over. Spring will never call @PreDestroy on a Prototype bean; the GC handles it.

4

The Injection Trap

If you inject a Prototype bean into a Singleton bean via Constructor/Field injection, the Prototype is only injected once (when the Singleton is created). It effectively becomes a Singleton.

5

Solving the Trap (@Lookup)

To fix the trap above, use Method Injection (annotating a method with @Lookup). Spring overrides the method at runtime to fetch a brand new Prototype from the Context every time it's called.

6

Web Scope Proxies

When injecting a Session/Request scoped bean into a Singleton controller, Spring actually injects a CGLIB Proxy. The proxy routes calls to the correct instance tied to the active HTTP request.

7

Custom Scopes

You can create entirely custom scopes by implementing the org.springframework.beans.factory.config.Scope interface and registering it with a CustomScopeConfigurer. Example: a "TenantScope" in a SaaS application.

6. Complete Code Example

ScopeTrapDemo.java
Copied! 
import org.springframework.stereotype.Component;
import org.springframework.context.annotation.Scope;
import org.springframework.beans.factory.annotation.Lookup;
import org.springframework.beans.factory.config.ConfigurableBeanFactory;

/* 1. The Prototype Bean (Stateful) */
@Component
@Scope(ConfigurableBeanFactory.SCOPE_PROTOTYPE)
public class TaskWorker {
    private String taskName; // Stateful data

    public void setTask(String name) { this.taskName = name; }
    public void execute() { System.out.println("Executing: " + taskName); }
}

/* -----------------------------------------------------------------
   2. THE TRAP (Bad Practice)
   This is a Singleton. It gets instantiated ONCE.
   Therefore, 'worker' is injected ONCE and becomes a Singleton too!
   ----------------------------------------------------------------- */
@Component
public class BadJobManager {
    private final TaskWorker worker;

    public BadJobManager(TaskWorker worker) {
        this.worker = worker; 
    }

    public void runJobs() {
        // Race condition! Multiple threads are modifying the SAME worker instance.
        worker.setTask("Job A"); 
        worker.execute();
    }
}

/* -----------------------------------------------------------------
   3. THE SOLUTION (@Lookup Method Injection)
   ----------------------------------------------------------------- */
@Component
public class GoodJobManager {

    // Spring uses CGLIB to override this method at runtime.
    // Every time you call it, it fetches a BRAND NEW Prototype from the Context.
    @Lookup
    public TaskWorker getNewWorker() {
        return null; // Implementation provided by Spring proxy
    }

    public void runJobs() {
        // Thread safe! A new instance is created for this specific method call.
        TaskWorker worker = getNewWorker();
        worker.setTask("Job B");
        worker.execute();
    }
}

7. Senior Developer Answer

"Understanding bean scopes is fundamentally about understanding memory management and thread safety. By default, Spring uses the Singleton scope, meaning every request hitting a @RestController or @Service is sharing the exact same object instance in memory. This is highly performant but dictates that our business logic layers must be strictly stateless.

When developers try to introduce stateful beans—like an encryption cipher holding a specific key—they often use the Prototype scope but fall into the classic Injection Trap by autowiring it into a Singleton service via the constructor. This results in the Prototype acting like a Singleton, leading to catastrophic race conditions in production under load.

To solve this, I mandate the use of Method Injection via the @Lookup annotation, which forces the Singleton to fetch a fresh Prototype instance from the container at runtime. Alternatively, if the state is tied to a user session in a monolith, I use @SessionScope, ensuring the proxyMode is correctly set to TARGET_CLASS so the CGLIB proxy can route method calls safely."

8. Quick Revision Notes

Singleton: Default. 1 instance per AppContext. Must be stateless.
Prototype: New instance per request/injection. Good for stateful logic.
Request Scope: 1 instance per HTTP request.
Session Scope: 1 instance per HTTP Session.
The Trap: Prototype injected into Singleton = Acts like Singleton.
The Fix: Use @Lookup method injection to fetch fresh prototypes.
Prototype Cleanup: Spring does NOT call @PreDestroy on Prototypes.
Web Proxies: Web scopes injected into Singletons require proxyMode config.
