Declarative Transactions

@Transactional Internals

Mastering AOP proxies, propagation behaviors, isolation levels, and the infamous self-invocation trap.

1. Definition

Simple English

It's an "All or Nothing" guarantee. If you are doing multiple database updates inside a method, `@Transactional` ensures that if one step fails, everything is undone (rolled back) as if nothing ever happened.

Technical

It provides Declarative Transaction Management. Spring uses AOP (Aspect-Oriented Programming) to create a proxy around the method. The proxy starts a database transaction before the method runs, commits it if successful, and rolls it back if a RuntimeException occurs.

Purpose

To enforce ACID (Atomicity, Consistency, Isolation, Durability) properties without polluting business logic with boilerplate connection.commit() or connection.rollback() JDBC code.

3. Real-World Project Example

Banking System (Money Transfer)

Why ACID properties are non-negotiable.

The Requirement

Alice wants to transfer $100 to Bob. The logic requires two SQL queries: UPDATE account SET balance = balance - 100 WHERE user = 'Alice', followed by UPDATE account SET balance = balance + 100 WHERE user = 'Bob'.

Without @Transactional

If Alice's account is successfully debited, but the database crashes (or throws a NullPointerException) before Bob's account is credited, the $100 disappears into the void. Data is now inconsistent.

With @Transactional

By adding @Transactional to the transfer() method, Spring wraps both updates in one unit of work. If Bob's update fails and throws an exception, Spring immediately rolls back Alice's debit. The data remains consistent.

4. 7 Key Points For senior-level depth

1

The Self-Invocation Trap

Because Spring uses proxies, calling a @Transactional method from another method within the exact same class bypasses the proxy entirely. No transaction will be created!

2

Default Rollback Rules

By default, Spring only rolls back for unchecked exceptions (RuntimeException) and Error. If your method throws a checked exception (like IOException or SQLException), it will commit.

3

Customizing Rollbacks

To fix the above issue, you must explicitly declare: @Transactional(rollbackFor = Exception.class) to guarantee rollbacks on all exception types.

4

Propagation Behaviors

Dictates what happens when a transactional method calls another. Default is REQUIRED (join existing, else create new). REQUIRES_NEW suspends the current one and starts a completely independent transaction.

5

Isolation Levels

Controls how concurrent transactions view each other's changes. Spring uses the underlying DB's default (usually READ_COMMITTED). Can be changed to SERIALIZABLE to prevent dirty, non-repeatable, and phantom reads.

6

readOnly = true

Used for fetch operations. It optimizes performance because Hibernate disables "dirty checking" and doesn't maintain snapshots of the entities in memory, saving significant RAM and CPU.

7

Private Methods

Putting @Transactional on a private or protected method does absolutely nothing (if using default Spring AOP proxies). Proxies can only intercept public method calls. (AspectJ weaving is an exception).

@Service
public class TransferService {

    private final AccountRepository accountRepo;
    private final AuditService auditService; // Injected to avoid self-invocation proxy trap

    public TransferService(AccountRepository accountRepo, AuditService auditService) {
        this.accountRepo = accountRepo;
        this.auditService = auditService;
    }

    // 1. Explicit rollbackFor ensures Checked Exceptions trigger rollback
    // 2. Default propagation is REQUIRED
    @Transactional(rollbackFor = Exception.class)
    public void executeTransfer(Long fromId, Long toId, BigDecimal amount) throws Exception {
        
        // This audit runs in a completely separate transaction!
        // If the transfer fails below, the audit record remains safely in the DB.
        auditService.logAttempt(fromId, toId, amount);

        accountRepo.debit(fromId, amount);
        
        if (amount.compareTo(BigDecimal.valueOf(10000)) > 0) {
            // Simulating a failure. 
            // Because of rollbackFor=Exception.class, the debit is rolled back.
            throw new Exception("Fraud rule triggered! Aborting transfer.");
        }

        accountRepo.credit(toId, amount);
    }
}

7. Senior Developer Answer

"While @Transactional simplifies code, it abstracts away database connection pooling, which is a major risk factor for production outages. As a senior developer, my primary rule is to keep transactions as short as possible.

A transaction holds an active database connection from the HikariCP pool. I regularly see developers making 3rd-party HTTP REST calls or publishing to Kafka inside a @Transactional method. If that external API takes 5 seconds to respond, your DB connection is locked for 5 seconds. Under high load, your connection pool will exhaust instantly, bringing down the entire application.

Secondly, I ensure teams understand the limitations of Spring's AOP proxies—specifically that self-invocation bypasses the proxy entirely, meaning REQUIRES_NEW or custom isolation levels will silently fail. Furthermore, we mandate explicit rollbackFor = Exception.class globally in our architecture to prevent checked exceptions from accidentally committing invalid state."

8. Quick Revision Notes

AOP Proxies: Spring uses proxies to wrap the method with begin/commit logic.
Public Only: Only works on public methods (when using default proxies).
Self-Invocation Trap: Calling a method inside the same class bypasses the proxy!
Default Rollback: ONLY rolls back on Unchecked Exceptions (`RuntimeException`).
Custom Rollback: Use rollbackFor = Exception.class to fix the default rule.
REQUIRED: Default propagation. Joins existing transaction or creates one.
REQUIRES_NEW: Suspends current, opens a new independent transaction (e.g., Audits).
readOnly=true: Use for fetching data. Disables Hibernate dirty checking (saves RAM).

9. Interview Answer Template

"The @Transactional annotation provides Declarative Transaction Management in Spring. It ensures ACID properties by wrapping our methods in database transactions without requiring manual connection handling.

Internally, Spring uses AOP to create a proxy around the target class. When a method is called, the proxy intercepts it, requests a connection from the TransactionManager, starts the transaction, executes our business logic, and finally commits or rolls back based on the outcome.

Crucially, we must be aware of two major traps: First, self-invocation bypasses the AOP proxy completely, meaning transactions won't be applied. Second, Spring only rolls back on Unchecked exceptions by default, so we often explicitly configure rollbackFor = Exception.class. Finally, to ensure database connection pools aren't exhausted, we must keep transactional methods as short as possible, avoiding long-running external API calls within them."
