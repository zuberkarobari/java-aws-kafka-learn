The Boundary Layers

Filters, Interceptors & ResponseEntity

Mastering the HTTP request journey from the Servlet Container to the Spring MVC Controller and back.

1. Definition

Filter

A low-level Servlet concept. It is the outermost shield of your application. It intercepts HTTP requests before they enter the Spring framework, making it perfect for global tasks like CORS, raw request logging, and payload compression.

Interceptor

A Spring MVC concept. It sits inside the Spring context, intercepting the request between the DispatcherServlet and your Controller. It has access to your beans and target methods, ideal for Auth and tenancy checks.

ResponseEntity

The payload sent back. It is a wrapper object representing the entire HTTP response. It grants explicit control over the HTTP Status Code (e.g., 201 Created), custom HTTP Headers, and the final JSON Body.

2. Internal Working: The Onion Architecture

A request travels through multiple layers before executing business logic, and the response travels back through those same layers in reverse. Understanding this "onion" is key to choosing the right tool.

3. Real-World Project Example

Multi-Tenant SaaS Application

How the layers work together securely.

1. The Filter (MDC Logging)

A request arrives. The RequestLoggingFilter intercepts the raw HTTP request, generates a unique traceId, and puts it in the thread's MDC context so all logs across the app include this ID.

2. The Interceptor (Tenant Auth)

The TenantInterceptor extracts a JWT, autowires the TenantService to verify the user, and sets the Tenant Context. If suspended, it immediately returns HTTP 403, blocking the Controller.

3. ResponseEntity (The Controller)

The Controller creates a resource. Instead of just returning the object (HTTP 200), it returns ResponseEntity.created(uri).body(data) to explicitly send HTTP 201 along with a Location header.

4. Key Points

1. Filter Implementation

Filters implement jakarta.servlet.Filter. They must explicitly call chain.doFilter(request, response) to pass control to the next layer.

2. Interceptor Methods

Interceptors implement HandlerInterceptor with 3 hooks: preHandle, postHandle, and afterCompletion.

3. Context Awareness

Interceptors are Spring-aware. preHandle receives the target Object handler (the Controller method), allowing reflection on custom annotations.

4. OncePerRequestFilter

Spring provides OncePerRequestFilter. It guarantees the filter executes exactly once per request, solving internal forward multi-execution issues.

5. Interceptor Registration

Unlike Filters (which use @Component), Interceptors must be explicitly registered via WebMvcConfigurer.addInterceptors().

6. The Body Reading Trap

If you read the HTTP Request body inside a Filter, the InputStream is consumed. The Controller will fail to read it later unless you use a ContentCachingRequestWrapper.

5. Follow-Up Questions

6. Code Example

LifecycleDemo.java 
import org.springframework.web.filter.OncePerRequestFilter;
import org.springframework.web.servlet.HandlerInterceptor;
import org.springframework.web.bind.annotation.*;
import org.springframework.http.ResponseEntity;
import jakarta.servlet.FilterChain;

/* 1. FILTER (Outermost Layer) */
@Component
public class LogFilter extends OncePerRequestFilter {
    @Override
    protected void doFilterInternal(request, response, FilterChain chain) {
        System.out.println("Filter: Incoming request");
        chain.doFilter(request, response); // MUST call this to continue
    }
}

/* 2. INTERCEPTOR (Inner Spring Layer) */
@Component
public class AuthInterceptor implements HandlerInterceptor {
    @Override
    public boolean preHandle(request, response, Object handler) {
        if (request.getHeader("Token") == null) {
            response.setStatus(401);
            return false; // Blocks request
        }
        return true; // Proceeds to controller
    }
}

/* 3. CONTROLLER + RESPONSE ENTITY */
@RestController
public class UserController {
    @PostMapping("/api/users")
    public ResponseEntity<String> create() {
        // Full control over HTTP Status and Headers
        return ResponseEntity
                .status(201)
                .header("Location", "/api/users/99")
                .body("User created.");
    }
}

7. Senior Developer Answer

"I strictly compartmentalize concerns based on the request lifecycle. If I need to log raw HTTP requests, compress responses, or handle CORS, I use a Filter because it operates at the highly efficient Tomcat/Servlet layer, blocking bad traffic before it touches the Spring framework.

If I need to verify user permissions or establish tenant context, I use an Interceptor. Because Interceptors execute inside the DispatcherServlet, they have full access to my Spring Beans and can use Reflection to inspect the specific Controller method being called.

Finally, at the Controller layer, I mandate the use of ResponseEntity. Default 200 OKs are sloppy REST design; ResponseEntity gives us the semantic control needed to return 201 Created for POSTs, 204 No Content for DELETEs, and to inject essential HTTP headers."
