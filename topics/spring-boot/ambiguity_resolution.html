Dependency Injection

@Primary vs @Qualifier

How Spring resolves ambiguity when multiple beans implement the exact same interface.

1. Definition

Simple English

Imagine asking an assistant for "a pen," but there's a red pen and a blue pen. The assistant gets confused and stops. @Primary tells the assistant: "When in doubt, always grab the blue one." @Qualifier tells the assistant: "Right now, I specifically need the red one."

Technical

When Spring's `BeanFactory` finds multiple candidate beans matching an injection point's Type, it throws a `NoUniqueBeanDefinitionException`. @Primary sets a global default bean, while @Qualifier narrows the search by Bean Name at the injection point.

Purpose

To enable the Strategy Design Pattern. It allows you to have multiple implementations of a single interface (e.g., EmailSender, SmsSender) and safely dictate exactly which implementation gets injected where.

2. Internal Working: The Resolution Flow

When Spring encounters an interface dependency, it follows a strict fallback hierarchy to resolve ambiguity.

Spring Ambiguity Resolver

Step 1: Match by Type
Spring finds 2+ beans implementing NotificationService.
Step 2: Check @Qualifier
Does the injection point specify a name?
@Qualifier("sms")
Step 3: Check @Primary
If no Qualifier, is one bean marked Primary?
@Primary
Step 4: Check Variable Name
Does the variable name match a Bean ID?
emailService
If all 4 steps fail, Spring throws NoUniqueBeanDefinitionException and crashes.

3. Real-World Project Example

Payment Processing System

Handling multiple payment gateways seamlessly.

@Primary StripePaymentService

Stripe is the company's default processor. By marking this implementation as @Primary, any controller that asks for a generic PaymentService will automatically receive Stripe without any extra configuration.

@Qualifier PayPalPaymentService

PayPal is only used as a fallback for specific European users. In the specific EuropeanCheckoutController, we inject the service using @Qualifier("payPalPaymentService") to explicitly override the Stripe default.

4. Key Points

1. @Primary sets the Default

Use it when one implementation is the clear winner for 90% of your application's use cases. You can only have ONE @Primary bean per interface.

2. @Qualifier Overrides All

@Qualifier is the ultimate authority. Even if another bean is marked @Primary, an explicit @Qualifier at the injection site wins every time.

3. String Literal Danger

Using `@Qualifier("myBeanName")` relies on string matching. If you rename the class later, refactoring tools might miss the string, causing runtime crashes.

4. Custom Meta-Annotations

To fix the String danger, you can create a custom annotation (e.g., `@PayPal`) that is meta-annotated with `@Qualifier`. This provides strict compile-time type safety.

5. The Fallback: Parameter Naming

If you have neither @Primary nor @Qualifier, Spring will attempt to match the parameter name to the bean name. Example: `PaymentService payPalPaymentService` will inject the PayPal bean by name. (This is generally considered a bad practice as it is fragile).

6. Code Example

ResolutionDemo.java 
import org.springframework.stereotype.Service;
import org.springframework.context.annotation.Primary;
import org.springframework.beans.factory.annotation.Qualifier;

/* 1. The Interface */
public interface NotificationService {
    void send();
}

/* 2. Implementation A (The Default) */
@Service
@Primary
public class EmailNotificationService implements NotificationService {
    public void send() { System.out.println("Sending Email"); }
}

/* 3. Implementation B (The Alternative) */
@Service("smsService")
public class SmsNotificationService implements NotificationService {
    public void send() { System.out.println("Sending SMS"); }
}

/* 4. The Controller injecting the dependencies */
@Service
public class AlertManager {

    private final NotificationService defaultService;
    private final NotificationService urgentService;

    public AlertManager(
            // Receives EmailNotificationService automatically because of @Primary
            NotificationService defaultService,
            
            // Explicitly requests SmsNotificationService bypassing the @Primary
            @Qualifier("smsService") NotificationService urgentService) {
        
        this.defaultService = defaultService;
        this.urgentService = urgentService;
    }
}

7. Senior Developer Answer

"Resolving ambiguity correctly is the cornerstone of implementing the Strategy pattern in Spring. When multiple beans implement an interface, I use @Primary to define the sensible default for the vast majority of use cases.

However, @Qualifier has a major flaw: it relies on brittle String literals which break easily during refactoring. To solve this in an enterprise environment, I completely avoid raw @Qualifier strings. Instead, I create custom Meta-Annotations (for example, creating a custom @SmsProvider annotation that is internally annotated with @Qualifier). This gives us strict compile-time type safety and makes the injection points perfectly immune to variable or class renaming errors."
