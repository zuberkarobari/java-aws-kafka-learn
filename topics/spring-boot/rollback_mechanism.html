Declarative Tx Management

Rollback Mechanism

Understanding how Spring's AOP proxies intercept exceptions to decide between committing or rolling back a database transaction.

1. Definition

Simple English

If something goes wrong (an error occurs) while saving multiple pieces of related data to the database, Spring hits the "undo" button so you don't end up with partially saved, corrupted data.

Technical

It's a feature of Spring's TransactionInterceptor. By default, when a method marked with @Transactional throws an Unchecked Exception (RuntimeException or Error), the interceptor catches it and tells the PlatformTransactionManager to issue a ROLLBACK command to the DB.

Purpose

To guarantee the Atomicity property of ACID. Transactions must be "all or nothing." Rollbacks prevent orphaned records and inconsistent system states when sudden failures occur.

3. Real-World Project Example

Banking Fund Transfer

Protecting money from disappearing into the void.

Step 1: Debit Account A

User initiates a transfer of $500. The system deducts $500 from Account A. This change is written to the database but remains uncommitted (locked in the transaction).

Step 2: The Failure

The system attempts to credit Account B, but Account B is frozen. The code throws an AccountFrozenException (a RuntimeException).

Step 3: The Rollback

Because the exception is Unchecked, Spring intercepts it. Instead of committing the deduction from Account A, Spring issues a ROLLBACK. Account A's balance reverts to its original state. The $500 is saved from disappearing.

4. 7 Key Points For senior-level depth

1

Default Behavior Rule

By default, Spring ONLY rolls back on Unchecked Exceptions (RuntimeException and Error). If a Checked Exception (like IOException or Exception) is thrown, Spring will COMMIT the transaction.

2

rollbackFor Override

You can override the default behavior using @Transactional(rollbackFor = Exception.class). This forces Spring to roll back for ALL exceptions, checked or unchecked. This is a highly recommended best practice.

3

The Try-Catch Trap

If you wrap your database logic in a try-catch block and do not rethrow the exception, the AOP Proxy never sees the exception. Because the proxy sees the method complete normally, it will COMMIT the transaction despite the error occurring.

4

Manual / Programmatic Rollback

If you must catch an exception but still want to roll back, you can manually instruct Spring to do so by calling TransactionAspectSupport.currentTransactionStatus().setRollbackOnly(); inside your catch block.

5

noRollbackFor

Sometimes you want a transaction to commit even if a specific Unchecked Exception occurs (like a non-critical notification failing). You can use @Transactional(noRollbackFor = NotificationException.class) to ignore it.

6

Interaction with Global Exception Handlers

Using a @RestControllerAdvice to handle exceptions globally does NOT prevent rollbacks. The AOP Proxy wraps the Service layer, while the Advice wraps the Controller. The exception hits the Proxy (triggering rollback) before it ever reaches the Controller Advice.

7

The Private Method Proxy Bypass

Because Spring uses AOP Proxies, if you apply @Transactional to a private method, or if a non-transactional method in the same class calls a transactional method, the proxy is bypassed. The transaction is never created, and therefore, no rollback will ever occur.

6. Code Example: Good vs Bad Rollbacks

RollbackDemoService.java
Copied! 
@Service
public class RollbackDemoService {

    private final UserRepository userRepository;

    public RollbackDemoService(UserRepository userRepository) {
        this.userRepository = userRepository;
    }

    // ❌ BAD: Won't rollback for Checked Exceptions!
    @Transactional 
    public void badSave(User user) throws IOException {
        userRepository.save(user);
        throw new IOException("Disk full!"); // This is a Checked Exception. Tx will COMMIT!
    }

    // ✅ GOOD: Best Practice. Rolls back for EVERYTHING.
    @Transactional(rollbackFor = Exception.class)
    public void goodSave(User user) throws Exception {
        userRepository.save(user);
        throw new Exception("Any error"); // Tx will ROLLBACK.
    }

    // ❌ BAD: The Try-Catch Trap
    @Transactional
    public void trappedSave(User user) {
        try {
            userRepository.save(user);
            throw new RuntimeException("DB Error");
        } catch (Exception e) {
            log.error("Error happened: ", e);
            // The exception is eaten. The Proxy thinks the method succeeded.
            // Tx will COMMIT!
        }
    }

    // ✅ FIX: Manual Rollback
    @Transactional
    public void manualRollbackSave(User user) {
        try {
            userRepository.save(user);
            throw new RuntimeException("DB Error");
        } catch (Exception e) {
            log.error("Error happened: ", e);
            // Manually tell the Transaction Manager to Rollback
            TransactionAspectSupport.currentTransactionStatus().setRollbackOnly();
        }
    }
}

7. Senior Developer Answer

"One of the strictest rules I enforce during code reviews is the explicit use of @Transactional(rollbackFor = Exception.class). Spring's default behavior of only rolling back on Unchecked exceptions is a legacy design that catches junior developers off-guard, leading to partial commits if an IOException or custom checked exception is thrown.

Secondly, I constantly watch for the 'Try-Catch Trap'. If a developer wraps a transactional method in a generic try-catch block to suppress errors and doesn't explicitly mark the transaction for rollback via TransactionAspectSupport.currentTransactionStatus().setRollbackOnly(), they unknowingly commit invalid states to the database.

Finally, from an architectural standpoint, I ensure we don't mix slow third-party API calls inside database transaction boundaries. If a transaction is opened, and we then make a 5-second REST call that times out (throwing an exception to trigger a rollback), we have held a database connection hostage for 5 seconds. Transactions should wrap database logic quickly, commit/rollback, and release the connection."

8. Quick Revision Notes

Default: Rolls back ONLY for Unchecked Exceptions (`RuntimeException`, `Error`).
Checked Exceptions: E.g., `IOException`. Transaction WILL COMMIT by default.
Best Practice: Use `@Transactional(rollbackFor = Exception.class)`.
Try-Catch Trap: Catching an error and not re-throwing it prevents rollbacks.
Manual Rollback: Call `TransactionAspectSupport.currentTransactionStatus().setRollbackOnly()`.
noRollbackFor: Instructs Spring to ignore a specific exception and commit anyway.
Controller Advice: Doesn't affect rollbacks. The proxy evaluates exceptions before the controller.
Private Methods: Annotating private methods does nothing. The proxy ignores them.

"In Spring, the Transaction rollback mechanism is controlled by AOP Proxies intercepting method executions. By default, Spring will only roll back a transaction if an Unchecked Exception (a RuntimeException or Error) crosses the transactional boundary. If a Checked Exception is thrown, Spring will assume it's an alternative business state and issue a COMMIT.

Because this default behavior often leads to bugs, the enterprise best practice is to explicitly define @Transactional(rollbackFor = Exception.class) to guarantee atomicity regardless of the exception type.

We must also be careful of the 'try-catch' trap. If a developer catches an exception inside the transactional method and fails to re-throw it or manually call setRollbackOnly(), the AOP Proxy assumes the method completed successfully, and the invalid data will be committed to the database."
