Core Concepts

@Component vs @Service vs @Repository

Understanding Spring's Stereotype Annotations, meta-annotations, exception translation, and semantic layering.

1. Definition

Simple English

They all essentially tell Spring: "Hey, take control of this class." @Component is a generic nametag. @Service and @Repository are special nametags indicating the class's specific job (like "Manager" vs "Database Worker").

Technical

These are Stereotype Annotations. @Component is the root annotation. @Service and @Repository are meta-annotated with @Component, meaning they inherit its bean-registration properties but add semantic domain value.

Purpose

The primary purpose is to clarify architecture (Domain-Driven Design). Additionally, @Repository triggers specific framework features (automatic exception translation), making the distinction mechanically important.

2. Internal Working: Meta-Annotations

During startup, @ComponentScan sweeps your packages. When it encounters @Service or @Repository, it reads their source code and sees they are internally annotated with @Component.

The Special Case: @Repository

While @Service adds no mechanical framework features, @Repository does. Spring registers a PersistenceExceptionTranslationPostProcessor. This proxy intercepts methods inside @Repository beans and translates raw SQL exceptions into standard DataAccessExceptions.

3. Real-World Project Example

User Authentication Module

How annotations separate concerns in a standard architecture.

@Component JwtTokenProvider

A utility class that signs and validates JWT strings. It doesn't perform business logic or touch the database. It's just a helpful tool.

@Service AuthService

The "Brain". It checks if the password is valid, decides if the account is locked, and coordinates the login rules.

@Repository UserRepository

Strictly handles data access. It runs the SQL queries. If the DB is down, it throws a standard Spring exception.

4. Key Points

1. Root Annotation

@Component is the foundation. @Service and @Repository are built on top of it via meta-annotations.

2. Exception Translation

@Repository is the only one with unique framework behavior (translating vendor SQL exceptions to Spring DataAccessExceptions).

3. @Controller Sibling

@Controller is another stereotype with special behavior: it registers the class with the DispatcherServlet.

4. AOP Targeting

Stereotypes make AOP trivial. You can write pointcuts targeting all @Service classes to log business execution times.

5. No Default Transactions

@Service does NOT make methods transactional automatically. You must explicitly add @Transactional.

6. Custom Stereotypes

You can create your own stereotypes by meta-annotating a custom annotation with @Component.

7. Naming Beans

Spring decapitalizes the class name (e.g., AuthService -> authService). Override via @Service("customName").

6. Code Example

StereotypesDemo.java 
import org.springframework.stereotype.Component;
import org.springframework.stereotype.Service;
import org.springframework.stereotype.Repository;

/* 1. @Component - Generic Utility */
@Component
public class PasswordHasher {
    public String hash(String pwd) { return "hash"; }
}

/* 2. @Repository - Exception Translating Data Access */
@Repository
public class UserJdbcRepository {
    public void save() { /* DB Logic */ }
}

/* 3. @Service - Business Logic */
@Service
public class UserService {
    private final PasswordHasher hasher;
    private final UserJdbcRepository repo;
    
    public UserService(PasswordHasher h, UserJdbcRepository r) {
        this.hasher = h; this.repo = r;
    }
}

7. Senior Developer Answer

"Mechanically, they all derive from @Component and register beans. However, I mandate their strict use to enforce Domain-Driven Design. @Repository is critical because it triggers the PersistenceExceptionTranslationPostProcessor, shielding the business layer from vendor-specific SQL exceptions. Furthermore, these stereotypes are essential for AOP pointcuts, allowing me to apply transaction boundaries and logging globally to all @Service classes without touching the code."
