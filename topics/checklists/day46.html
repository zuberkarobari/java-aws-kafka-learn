Section 1 — The Problem Reactive Solves

Thread-per-request bottleneck:

In Spring MVC, every HTTP request gets a thread from Tomcat's thread pool (typically 200 threads). A thread waiting for a database response or an HTTP call to another service is blocked — it cannot do anything else. At 200 concurrent slow requests, the thread pool is exhausted and new requests queue up. Reactive programming uses non-blocking I/O: a small number of event-loop threads handle many concurrent requests by reacting to events (data arrived, error occurred) rather than blocking and waiting.

Aspect

Spring MVC (Imperative) | Spring WebFlux (Reactive)

Threading model

One thread per request (typically 200 threads max) | Small event-loop pool (typically CPUs * 2) handles thousands of concurrent requests

I/O model

Blocking — thread waits while DB/HTTP call completes | Non-blocking — thread is released while waiting; reused for other requests

Best for

CPU-bound work, blocking libraries, JDBC, most CRUD services | High concurrency with many concurrent slow I/O operations

Complexity

Low — sequential code, familiar debugging | High — reactive pipelines, stacktraces are hard to read, context propagation differs

DB support

JPA, JDBC — mature ecosystem | R2DBC — limited driver support, smaller ecosystem

Choose when

Default for most services — simpler and sufficient | Serving thousands of concurrent SSE/WebSocket connections, API gateway, or streaming data

Section 2 — Project Reactor Core

Mono and Flux

// Mono — zero or one element
Mono<PaymentResult> result = paymentService.process(request);
// Flux — zero to many elements (streaming)
Flux<Payment> payments = paymentRepository.findByAccountId(accountId);
// Creating publishers
Mono.just(value)                    // emit one value
Mono.empty()                        // emit nothing, complete
Mono.error(new RuntimeException())  // emit error
Mono.fromCallable(() -> blockingCall())  // wrap a blocking call
Flux.just(1, 2, 3)                  // emit fixed values
Flux.fromIterable(list)             // emit from collection
Flux.interval(Duration.ofSeconds(1)) // emit every second
// Core transformation operators
Flux.just(1, 2, 3, 4, 5)
    .filter(n -> n % 2 == 0)        // keep even numbers
    .map(n -> n * 10)               // transform each element
    .take(2)                        // take first 2 elements
    .subscribe(System.out::println); // 20, 40
// flatMap — async transformation, unordered
// Use when: calling async service per element, order does not matter
Flux<Payment> payments = accountIds
    .flatMap(id -> paymentService.findByAccount(id));  // concurrent
// concatMap — async transformation, ordered (sequential)
// Use when: order must be preserved, elements processed one at a time
Flux<Payment> payments = accountIds
    .concatMap(id -> paymentService.findByAccount(id)); // sequential
// Error handling
Mono<PaymentResult> safe = paymentService.process(request)
    .onErrorReturn(PaymentResult.failed())    // fallback value on error
    .onErrorResume(e -> fallbackService.process(request)) // fallback Mono
    .onErrorMap(GatewayException.class,
        e -> new PaymentServiceException(e)); // translate exception type
// Combining publishers
Mono.zip(userMono, accountMono)     // wait for both, combine
    .map(tuple -> new Dashboard(tuple.getT1(), tuple.getT2()));
Flux.merge(flux1, flux2)            // interleave two streams as they arrive
Flux.concat(flux1, flux2)           // flux1 then flux2, sequential

Schedulers and Blocking Code

The golden rule:

Never block on a reactor thread. If you must call a blocking library (JDBC, a legacy SDK, file I/O), offload it to the boundedElastic scheduler which is designed for blocking work. Blocking on the event-loop thread (the default thread) stalls all reactive processing on that thread.

// Offloading blocking work to the right scheduler
Mono<User> user = Mono.fromCallable(() -> jdbcTemplate.queryForObject(sql, User.class))
    .subscribeOn(Schedulers.boundedElastic()); // execute on a thread pool for blocking I/O
// publishOn — switch scheduler for downstream operators
Flux.fromIterable(largeList)
    .publishOn(Schedulers.parallel())   // downstream operations run on parallel scheduler
    .map(this::cpuIntensiveTransform)   // runs on parallel
    .publishOn(Schedulers.boundedElastic())
    .flatMap(item -> callExternalApi(item)); // runs on boundedElastic
// Scheduler types
// Schedulers.parallel()        — fixed pool = number of CPUs, for CPU-bound work
// Schedulers.boundedElastic()  — elastic pool for blocking I/O, capped to prevent OOM
// Schedulers.single()          — single thread for sequential guaranteed ordering
// Schedulers.immediate()       — same thread as caller (default)
Q1: What is backpressure in reactive programming and why does it matter? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

Backpressure is the mechanism that lets a subscriber control how fast a publisher emits data. Without it, a fast producer and slow consumer leads to the consumer's buffer overflowing. In Project Reactor, a subscriber requests N items from a publisher using the request() method on the Subscription. The publisher only emits up to N items before waiting for another request signal. This demand-driven model prevents out-of-memory errors when consuming large streams. When backpressure cannot be applied — for example, when you are receiving external events you cannot slow down — you use an overflow strategy: buffer a fixed number of items, drop the oldest, or error when the buffer fills. The practical importance: in a payment event stream from Kafka, if the consumer is slower than the producer, without backpressure you would buffer millions of events in memory. With backpressure, the consumer only pulls what it can handle.

Section 3 — WebFlux in Practice

// Reactive REST controller
@RestController
@RequestMapping("/api/payments")
public class PaymentController {
    private final ReactivePaymentService service;
    @PostMapping
    public Mono<ResponseEntity<PaymentResult>> create(@RequestBody Mono<PaymentRequest> request) {
        return request
            .flatMap(service::process)
            .map(result -> ResponseEntity.status(HttpStatus.CREATED).body(result))
            .onErrorResume(ValidationException.class,
                e -> Mono.just(ResponseEntity.badRequest().build()));
    }
    @GetMapping
    public Flux<PaymentSummary> list(@RequestParam String accountId) {
        return service.findByAccount(accountId);
    }
    // Server-Sent Events — streaming to browser
    @GetMapping(value = "/stream", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux<PaymentEvent> stream() {
        return eventService.getPaymentEventStream();
    }
}
// WebClient — non-blocking HTTP client
@Service
public class FraudServiceClient {
    private final WebClient webClient;
    public FraudServiceClient(WebClient.Builder builder) {
        this.webClient = builder
            .baseUrl("https://fraud.internal.example.com")
            .build();
    }
    public Mono<FraudResult> check(PaymentRequest request) {
        return webClient.post()
            .uri("/fraud/check")
            .bodyValue(request)
            .retrieve()
            .onStatus(HttpStatusCode::is5xxServerError,
                resp -> Mono.error(new FraudServiceException("Service error")))
            .bodyToMono(FraudResult.class)
            .timeout(Duration.ofSeconds(2))
            .retryWhen(Retry.backoff(3, Duration.ofMillis(100))
                .filter(e -> e instanceof FraudServiceException));
    }
}

Q2: When would you choose Spring WebFlux over Spring MVC? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

I would choose WebFlux when the service has a high number of concurrent slow I/O operations and thread count is the bottleneck. The clearest cases: an API gateway handling tens of thousands of concurrent connections where most time is spent waiting for downstream responses — a reactive event loop handles this with a handful of threads. A service streaming real-time data via Server-Sent Events or WebSockets to thousands of clients simultaneously. Or a data pipeline reading from Kafka and calling external APIs where maximising I/O concurrency is the goal. For most CRUD services — even high-throughput ones — Spring MVC with a properly sized thread pool and non-blocking connection handling at the OS level is simpler and sufficient. WebFlux introduces real complexity: reactive stacktraces are hard to read, JDBC does not work without R2DBC (which has less mature driver support), debugging is harder, and the mental model requires everyone on the team to understand reactive programming. I would not introduce that complexity unless the concurrency profile genuinely demands it.

Interview Q&A — 12 Questions

How to use:

Q3: What is Project Reactor and how does it relate to Spring WebFlux? [Easy] ▼
💡 Ideal Answer (Amazon/GS Standard)

Project Reactor is the reactive streams library that Spring WebFlux is built on. It provides Mono (for 0-1 items) and Flux (for 0-N items) as the core publisher types, and a rich set of operators for transforming, filtering, combining, and error-handling reactive pipelines. Spring WebFlux is the reactive web framework that uses Reactor as its programming model. When a WebFlux controller returns a Mono or Flux, the framework subscribes to it using Netty's event loop threads and writes the result to the HTTP response without blocking. The relationship: WebFlux is to Reactor what Spring MVC is to Java threads — a framework that manages the execution model, while the underlying model (Reactor or threads) handles the concurrency.

Q4: What is the difference between map and flatMap in Reactor? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

map transforms each element synchronously — it applies a function that returns a plain value, not a publisher. flatMap transforms each element asynchronously — it applies a function that returns a Mono or Flux, then flattens all those publishers into a single stream. Use map when the transformation is pure and immediate, like converting a domain object to a DTO. Use flatMap when the transformation requires an async call — fetching data from a database or calling another service. The important difference between flatMap and concatMap: flatMap subscribes to all inner publishers concurrently and emits results as they arrive (unordered). concatMap subscribes to each inner publisher one at a time and preserves order. If you need all fraud checks to run concurrently and do not care about order, use flatMap. If order matters, use concatMap.

Q5: How do you handle errors in a reactive pipeline? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Reactor has several error-handling operators depending on what you want to do when an error occurs. onErrorReturn provides a static fallback value — if the fraud service throws, return a default FraudResult. onErrorResume provides a fallback publisher — if the primary service fails, switch to a cached or degraded response. onErrorMap translates one exception type to another — convert a GatewayException to a domain-specific PaymentFailedException. doOnError runs a side effect on error without changing the pipeline — useful for logging. The important rule: errors in a reactive pipeline propagate downstream to the subscriber. If you do not handle them, the subscriber receives the error signal. In a WebFlux controller, unhandled errors become 500 responses. Wrapping each async call in its own onErrorResume lets you handle failures at the right granularity rather than catching everything at the top.

Q6: What is subscribeOn and when do you need it? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

subscribeOn sets the scheduler on which the subscription happens — which thread calls the actual source of data. It affects where the source executes, not where downstream operators execute. You need it specifically when the source of your Mono or Flux is blocking — a JDBC call, a legacy synchronous SDK, file I/O. Wrapping the blocking call in Mono.fromCallable() and adding subscribeOn(Schedulers.boundedElastic()) moves the blocking execution off the reactor event loop thread and onto a thread pool designed for blocking work. Without it, a blocking call on a reactor thread stalls all reactive processing on that thread — effectively blocking the event loop for all other requests. publishOn, by contrast, switches the scheduler for the operators that come after it in the pipeline, not for the source itself.

Q7: What is R2DBC and what are its limitations? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

R2DBC is the Reactive Relational Database Connectivity specification — a non-blocking API for relational databases. It allows reactive Spring applications to access PostgreSQL, MySQL, and others without blocking. The Spring Data R2DBC module provides reactive repositories that return Mono and Flux. The limitations are real and worth being honest about. First, driver maturity: R2DBC drivers are newer and have less battle-tested production history than JDBC drivers. Second, no JPA: the R2DBC ecosystem does not have a reactive equivalent of JPA or Hibernate — there is no lazy loading, no first-level cache, no entity lifecycle management. You write explicit SQL or use Spring Data's derived query methods for simple cases. Third, transaction support is different — you work with reactive transaction operators rather than @Transactional. For most services, JDBC with a properly sized connection pool and offloading blocking calls to boundedElastic is simpler and more mature.

Q8: How do you test reactive code? [Easy] ▼
💡 Ideal Answer (Amazon/GS Standard)

StepVerifier from reactor-test is the standard tool. It subscribes to a Mono or Flux and lets you assert on each emitted element, the completion signal, or the error signal using expectNext(), expectNextMatches(), expectComplete(), expectError(). For time-based operators like delay or timeout, StepVerifier.withVirtualTime() lets you advance virtual time without actually waiting. For WebFlux controllers, WebTestClient provides a reactive HTTP test client that integrates with MockMvc-style assertions. You use it to send test requests and assert on the response status, headers, and body. The most common mistake in reactive testing is not calling verify() at the end of a StepVerifier chain — without verify(), the assertions are defined but never executed, and the test always passes.

Q9: When would you NOT use WebFlux? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

The most common and honest answer is: when the team is not familiar with reactive programming and the service is not genuinely I/O-bound with high concurrency. The learning curve is real — reactive stacktraces are difficult to read, debugging is harder, and the programming model is fundamentally different from imperative code. If you are building a CRUD service that talks to a PostgreSQL database using JDBC, WebFlux gives you no throughput benefit because JDBC is blocking anyway. The reactive non-blocking model only provides benefit when the actual I/O layer is non-blocking end to end. Another case to avoid: when you need to integrate with a library that is not reactive — PDF generation, legacy SDKs, complex file operations. These require offloading to boundedElastic schedulers which partially defeats the purpose. For most Spring Boot services at reasonable scale, MVC with a properly configured thread pool is simpler, more debuggable, and fully sufficient.

Q10: What is the reactive security model in WebFlux? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

WebFlux uses a reactive security filter chain — SecurityWebFilterChain instead of SecurityFilterChain. The authentication and authorization filters are implemented as WebFilter rather than javax.servlet.Filter, and they work with Mono and Flux throughout. The security context is stored in a reactive context rather than ThreadLocal, which is why you use ReactiveSecurityContextHolder.getContext() instead of SecurityContextHolder.getContext(). JWT validation uses ReactiveJwtDecoder which returns a Mono. The practical implication: any code that needs to access the current user in a reactive service must use the reactive context API, not ThreadLocal-based SecurityContextHolder — the latter is not thread-safe in a reactive environment where a single logical request may execute on multiple threads.

Q11: How does Flux handle streaming large datasets from a database? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Flux is ideal for streaming large result sets because it allows processing records one at a time as they arrive from the database, rather than loading the entire result into memory. With R2DBC, a repository method returning Flux streams rows from the database to the subscriber using the reactive backpressure protocol — the subscriber requests rows at the rate it can process them. This is the reactive equivalent of scrolling through a JDBC ResultSet. For HTTP streaming, a WebFlux controller returning Flux with text/event-stream media type streams results to the client one by one, which is useful for real-time dashboards or large report downloads. The key benefit: a query returning 1 million rows does not require 1 million objects in memory simultaneously — only the in-flight batch.

Q12: What is the difference between Mono.zip and Mono.flatMap for combining calls? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Mono.zip combines multiple Mono publishers concurrently — it subscribes to all of them simultaneously and emits when all complete. The result is a Tuple containing all the values. If you need a user profile AND an account summary, Mono.zip(userMono, accountMono) fetches both concurrently and the total latency is the maximum of the two, not the sum. Mono.flatMap chains calls sequentially — the second call only starts after the first completes. Use flatMap when the second call depends on the result of the first. Use zip when the calls are independent. The latency difference is significant: if each call takes 100ms, zip takes 100ms total, flatMap takes 200ms. I use zip heavily for dashboard endpoints that aggregate data from multiple independent sources.

Q13: What is Flux.create and when would you use it over Flux.fromIterable? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Flux.fromIterable is for converting an existing synchronous collection into a reactive stream. The data is already in memory. Flux.create is for wrapping a push-based data source — one that calls you when data is available, like a callback-based API, a message listener, or an event emitter. You provide a FluxSink to the create callback; the external source calls sink.next() when data arrives, sink.error() on failure, and sink.complete() when done. Flux.create supports backpressure strategies for when the source produces faster than the subscriber consumes. Flux.generate is the alternative for pull-based sources — the framework calls your generator function each time the subscriber requests an element, giving you full backpressure control without buffering.

Q14: How do you propagate MDC trace context in a reactive pipeline? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

MDC is ThreadLocal-based. In a reactive pipeline, a single request may execute on multiple threads as it moves through operators. MDC values set on the initial thread are lost when the pipeline switches to a different thread. The solution is to use the Reactor Context — a key-value store that is attached to the reactive pipeline and propagates through all operators regardless of thread switches. Micrometer Tracing handles this automatically when you use the Micrometer Tracing bridge for WebFlux — it stores the trace context in the Reactor Context and writes it to MDC on each operator that executes. If you need custom MDC values in a reactive service, use the contextWrite operator to attach them to the pipeline context and a doOnEach operator to write them to MDC before log statements. Without this, log lines from a reactive request will have missing or incorrect traceId values.
