PECS: Producer Extends, Consumer Super. This one rule explains every correct wildcard choice:

If a collection PRODUCES values (you read from it): use ? extends T. The collection gives you T objects — you are a consumer of those objects. If a collection CONSUMES values (you add to it): use ? super T. The collection accepts T objects — you produce objects for it. Reading from ? extends T: you get at most a T (could be a subtype). Writing to ? extends T: FORBIDDEN — the compiler cannot know the actual type. Reading from ? super T: you get at least an Object. Writing to ? super T: you can add T and subtypes safely. In Collections.copy(dest, src): src is ? extends T (you read from it), dest is ? super T (you write to it). This is PECS in action.

// ── BOUNDED WILDCARDS ──
// Upper bound: ? extends T — read-only, get T or subtype
// Use when: you only READ from the collection
public double sumAmounts(List<? extends Number> amounts) {
    double total = 0;
    for (Number n : amounts) total += n.doubleValue(); // OK — reading
    // amounts.add(1.0); // COMPILE ERROR — cannot add: could be List<Integer>
    return total;
}
// Can call with: List<Integer>, List<Double>, List<BigDecimal>
// Lower bound: ? super T — write-safe, accept T and supertypes
// Use when: you only WRITE to the collection
public void addTransactionAmounts(List<? super BigDecimal> sink, List<BigDecimal> amounts) {
    for (BigDecimal amt : amounts) sink.add(amt); // OK — writing BigDecimal
    // BigDecimal val = sink.get(0); // COMPILE ERROR — could be List<Object>
}
// Can call with: List<BigDecimal>, List<Number>, List<Object>
// PECS in action — Collections.copy signature:
// public static <T> void copy(List<? super T> dest, List<? extends T> src)
//   dest is a CONSUMER  → ? super T
//   src  is a PRODUCER  → ? extends T
// Unbounded wildcard: List<?> — you don't care about type
// Can read only as Object, cannot add anything (except null)
public void printAll(List<?> items) {
    items.forEach(item -> System.out.println(item)); // item is Object
}
// ── TYPE ERASURE ──
// At compile time: List<String> and List<Integer> are different
// At runtime:       both become List (raw type) — generic info erased
// Consequence 1: cannot do instanceof with generic type
// if (list instanceof List<String>) { } // COMPILE ERROR
// if (list instanceof List) { }          // OK — raw type check
// Consequence 2: cannot create generic arrays
// T[] array = new T[10]; // COMPILE ERROR — type unknown at runtime
// Workaround: use (T[]) new Object[10] with @SuppressWarnings("unchecked")
// Consequence 3: cannot use primitives as type arguments
// List<int> list; // COMPILE ERROR — use List<Integer>
// What SURVIVES erasure (reifiable types):
// - Raw types: List, Map
// - Non-generic types: String, Integer
// - Unbound wildcards: List<?>
// - Arrays of reifiable: String[], List<?>[]
// ── RECURSIVE GENERIC BOUND: <T extends Comparable<T>> ──
// Means: T must be comparable to itself
public <T extends Comparable<T>> T max(List<T> items) {
    return items.stream().max(Comparator.naturalOrder())
        .orElseThrow(NoSuchElementException::new);
}
// Works with: max(List<Integer>), max(List<String>), max(List<LocalDate>)
// Multiple bounds: class must come first, interfaces after
// <T extends AbstractPayment & Serializable & Auditable>
// ── HEAP POLLUTION ──
// Occurs when a variable of parameterized type refers to an object of wrong type
@SafeVarargs  // suppress unchecked varargs warning when method is safe
@SuppressWarnings("varargs")
public final <T> List<T> asList(T... elements) {
    return Arrays.asList(elements);
}
// @SafeVarargs safe when: method does NOT store into varargs array
// and does NOT allow array reference to escape to untrusted code
