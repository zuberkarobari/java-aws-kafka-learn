Section 1 — Auto-Configuration Internals

@SpringBootApplication — What It Actually Does

It is composed of three annotations:

@SpringBootConfiguration (marks this as a configuration class — same as @Configuration). @EnableAutoConfiguration (triggers the auto-configuration loading mechanism). @ComponentScan (scans the current package and sub-packages for @Component, @Service, @Repository, @Controller). None of these three is magic — they each have a well-defined, traceable behaviour.

How @EnableAutoConfiguration Loads 200+ Auto-Configurations

Spring Boot 2.6 and earlier — spring.factories:

Spring Boot 2.7+ — ImportCandidates:

Moved to META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports — one class per line. More efficient: lazy loading, cleaner override mechanism. The old spring.factories still works for backwards compatibility.

// What happens at startup — simplified
// 1. @EnableAutoConfiguration triggers AutoConfigurationImportSelector
// 2. Selector reads all auto-config candidates from ImportCandidates
// 3. Each candidate is evaluated against its @Conditional annotations
// 4. Only those passing all conditions become beans
// Example: DataSourceAutoConfiguration
@AutoConfiguration
@ConditionalOnClass({ DataSource.class, EmbeddedDatabaseType.class })
@ConditionalOnMissingBean(type = "io.r2dbc.spi.ConnectionFactory")
@EnableConfigurationProperties(DataSourceProperties.class)
@Import({ DataSourcePoolMetadataProvidersConfiguration.class,
         DataSourceCheckpointRestoreConfiguration.class })
public class DataSourceAutoConfiguration {
    @Bean
    @ConditionalOnMissingBean
    @ConditionalOnSingleCandidate(DataSource.class)
    public DataSourceTransactionManager transactionManager(
            Environment environment, DataSource dataSource) {
        // Only created if: DataSource exists AND no TransactionManager bean yet
        return new DataSourceTransactionManager(dataSource);
    }
}
// Writing your own auto-configuration
@AutoConfiguration
@ConditionalOnClass(PaymentGateway.class)
@ConditionalOnProperty(prefix = "payment", name = "enabled", havingValue = "true")
@EnableConfigurationProperties(PaymentProperties.class)
public class PaymentAutoConfiguration {
    @Bean
    @ConditionalOnMissingBean  // user can override by defining their own PaymentGateway bean
    public PaymentGateway paymentGateway(PaymentProperties props) {
        return new DefaultPaymentGateway(props.getApiKey(), props.getBaseUrl());
    }
}
// Registered in: META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports
// com.example.payment.PaymentAutoConfiguration

@Conditional annotation

What it checks | Common use

@ConditionalOnClass

Specified class is on the classpath | Only configure JDBC if DataSource.class is present

@ConditionalOnMissingBean

No bean of the given type exists yet | Provide a default bean but allow user override

@ConditionalOnProperty

application.properties has specified key=value | Feature flags, enable/disable auto-configs

@ConditionalOnBean

A bean of the given type IS present | Configure X only if Y was already configured

@ConditionalOnMissingClass

Specified class is NOT on the classpath | Provide fallback when a library is absent

@ConditionalOnExpression

SpEL expression evaluates to true | Complex multi-property conditions

@ConditionalOnWebApplication

App is a web (servlet) application | Only wire MVC configs in web context

@ConditionalOnCloudPlatform

Running on a specific cloud (AWS, GCP, Azure) | Cloud-specific config

A: When @EnableAutoConfiguration is processed, Spring invokes AutoConfigurationImportSelector, which reads all auto-configuration class names from META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports inside spring-boot-autoconfigure.jar. This gives it roughly 150 candidate classes. Each candidate is then evaluated against its @Conditional annotations — @ConditionalOnClass checks the classpath, @ConditionalOnMissingBean checks existing beans, @ConditionalOnProperty checks application.properties. Only the configurations that pass all their conditions are registered as bean definitions. The rest are discarded. The order matters: auto-configurations run after your own @Configuration classes, so your beans take precedence. The @ConditionalOnMissingBean pattern is how Spring ensures your custom bean always wins over the auto-configured default.

Section 2 — Bean Lifecycle

The 7 Lifecycle Steps

Step

What happens | Your hook

1. Instantiation

Spring calls the constructor (or factory method) | Constructor injection happens here

2. Dependency injection

Spring injects @Autowired fields, setters, and constructor args | All dependencies available after this step

3. BeanNameAware / BeanFactoryAware

Spring sets bean name and factory reference if implemented | Rarely used directly

4. BeanPostProcessor.postProcessBeforeInitialization

All registered BPPs run before init | Spring AOP proxy wrapping, @Autowired validation

5. Initialisation

6. BeanPostProcessor.postProcessAfterInitialization

All registered BPPs run after init | This is where the AOP proxy is actually applied

7. Destroy

@PreDestroy OR DisposableBean.destroy() OR @Bean(destroyMethod) — called on context close | Resource cleanup, connection pool shutdown

// Full lifecycle demonstration
@Component
public class PaymentServiceBean implements InitializingBean, DisposableBean {
    private final PaymentGateway gateway;
    private       ConnectionPool  pool;
    // Step 1+2: constructor — dependency injection happens here
    public PaymentServiceBean(PaymentGateway gateway) {
        this.gateway = gateway;
        System.out.println("Step 1+2: constructed and injected");
    }
    // Step 5a: @PostConstruct — called after injection, before bean is used
    @PostConstruct
    public void postConstruct() {
        this.pool = ConnectionPool.create(gateway.getConfig());
        System.out.println("Step 5a: @PostConstruct — pool initialised");
    }
    // Step 5b: InitializingBean.afterPropertiesSet() — same timing as @PostConstruct
    // Runs AFTER @PostConstruct when both are present
    @Override
    public void afterPropertiesSet() {
        System.out.println("Step 5b: afterPropertiesSet");
    }
    // Step 7a: @PreDestroy — called before bean is removed from context
    @PreDestroy
    public void preDestroy() {
        System.out.println("Step 7a: @PreDestroy");
    }
    // Step 7b: DisposableBean.destroy() — runs after @PreDestroy
    @Override
    public void destroy() {
        if (pool != null) pool.close();
        System.out.println("Step 7b: destroy — pool closed");
    }
}

// They are JSR-250 annotations — no Spring interface coupling

BeanPostProcessor — The Most Powerful Extension Point

What BeanPostProcessor does:

It intercepts every bean in the context at two points: before initialisation (postProcessBeforeInitialization) and after initialisation (postProcessAfterInitialization). The return value replaces the original bean in the context — this is how Spring AOP replaces your @Service with a proxy. Several critical Spring features are implemented as BeanPostProcessors: AutowiredAnnotationBeanPostProcessor processes @Autowired, CommonAnnotationBeanPostProcessor processes @PostConstruct and @PreDestroy, and AbstractAutoProxyCreator is the base for all AOP proxy creation.

// Writing a custom BeanPostProcessor
// Use case: automatically register every service bean with a metrics registry
@Component
public class MetricsRegistrationBeanPostProcessor implements BeanPostProcessor {
    private final MeterRegistry meterRegistry;
    public MetricsRegistrationBeanPostProcessor(MeterRegistry registry) {
        this.meterRegistry = registry;
    }
    @Override
    public Object postProcessAfterInitialization(Object bean, String beanName) {
        // After every bean is fully initialised, check if it's a service
        if (bean.getClass().isAnnotationPresent(Service.class)) {
            // Register a gauge for the bean's class name
            Gauge.builder("spring.bean.active", () -> 1.0)
                .tag("bean", beanName)
                .register(meterRegistry);
        }
        // IMPORTANT: return the bean (or a replacement)
        // Returning null would remove the bean from the context
        return bean;
    }
}
// Built-in BPPs you should know:
// AutowiredAnnotationBeanPostProcessor  → processes @Autowired and @Value
// CommonAnnotationBeanPostProcessor     → processes @PostConstruct, @PreDestroy, @Resource
// PersistenceAnnotationBeanPostProcessor → processes @PersistenceContext
// AbstractAutoProxyCreator              → creates AOP proxies for @Transactional, @Async, etc.

Bean Scopes — The Injection Trap

Scope

Lifecycle | Use case | Injection trap?

singleton (default)

One instance per ApplicationContext. Shared across all requests. | Services, repositories, default | No

prototype

New instance every time the bean is requested. | Stateful beans, user-specific config | Yes — see below

request

One instance per HTTP request (web apps only). | Request-scoped state | Yes — inject via proxy

session

One instance per HTTP session. | User session state | Yes — inject via proxy

application

One instance per ServletContext (web apps). | App-wide shared web state | No

The prototype injection trap:

If you inject a prototype bean into a singleton bean, the prototype bean is created once at injection time and never refreshed — it effectively becomes a singleton. Fix: inject ApplicationContext and call getBean() each time, or use @Lookup method injection, or use ObjectProvider. The request and session scopes have the same issue if injected directly into a singleton — fix by injecting them with @ScopedProxy (creates a proxy that delegates to the correct scoped instance per request/session).

Q1: What is BeanPostProcessor and what are two things Spring uses it for? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

BeanPostProcessor is an extension point that lets you intercept every bean before and after it is initialised. The key detail is that postProcessAfterInitialization can return a different object — this is how Spring replaces your beans with proxies. Two built-in uses: first, AutowiredAnnotationBeanPostProcessor processes @Autowired — it scans each bean for @Autowired fields and performs the injection. Second, AbstractAutoProxyCreator is the BeanPostProcessor that creates AOP proxies — when it sees a bean that has @Transactional, @Async, or @Cacheable methods, postProcessAfterInitialization wraps it in a proxy and returns the proxy instead of the original bean. Every subsequent reference to that bean in the context is actually a reference to the proxy.

Section 3 — Spring AOP

How Spring AOP Works

The core idea:

JDK proxy vs CGLIB:

JDK dynamic proxy: created when the bean implements at least one interface. Uses java.lang.reflect.Proxy. Only interface methods are intercepted. CGLIB proxy: created when the bean has no interface (Spring Boot default behaviour since 2.x, even if interfaces exist). Subclasses the concrete class using bytecode generation. All public methods can be intercepted. The practical implication: CGLIB cannot proxy final classes or final methods — @Transactional on a final method will silently have no effect.

// AOP terminology
// Aspect:      The class that contains cross-cutting logic
// Pointcut:    The expression that selects which methods to intercept
// Advice:      The code that runs at the join point (before, after, around)
// Join point:  A specific method execution matched by the pointcut
// Weaving:     Applying aspects to target objects (Spring does this at startup)
// Writing a custom Aspect: audit logging for all payment operations
@Aspect
@Component
public class PaymentAuditAspect {
    private final AuditService auditService;
    private final Logger log = LoggerFactory.getLogger(getClass());
    // Pointcut: all methods in PaymentService
    @Pointcut("execution(* com.example.payment.PaymentService.*(..))")
    private void paymentOperations() {}
    // Around advice: most powerful — controls method execution entirely
    @Around("paymentOperations()")
    public Object auditPayment(ProceedingJoinPoint pjp) throws Throwable {
        String method    = pjp.getSignature().getName();
        Object[] args    = pjp.getArgs();
        long     start   = System.currentTimeMillis();
        log.info("Payment operation started: method={} args={}", method, args);
        try {
            Object result = pjp.proceed(); // call the real method
            long elapsed  = System.currentTimeMillis() - start;
            auditService.recordSuccess(method, args, result, elapsed);
            return result;
        } catch (Exception e) {
            auditService.recordFailure(method, args, e);
            throw e; // re-throw — do not silently swallow
        }
    }
    // Before advice: runs before the method, cannot stop execution
    @Before("@annotation(com.example.RequiresRole) && args(req,..)")
    public void checkRole(JoinPoint jp, PaymentRequest req) {
        // SecurityContext check...
    }
    // AfterReturning: only runs if method returns normally
    @AfterReturning(pointcut="paymentOperations()", returning="result")
    public void onSuccess(JoinPoint jp, Object result) {
        // Only called when no exception
    }
    // AfterThrowing: only runs if method throws
    @AfterThrowing(pointcut="paymentOperations()", throwing="ex")
    public void onFailure(JoinPoint jp, Exception ex) {
        log.error("Payment operation failed: {}", ex.getMessage());
    }
}
// Pointcut expression syntax:
// execution(* com.example.service.*.*(..))  — all methods in all service classes
// @annotation(org.springframework.transaction.annotation.Transactional) — methods annotated with @Transactional
// within(com.example.payment.*)  — any method in the payment package
// bean(paymentService)  — specific named bean
// args(String, BigDecimal)  — methods with these parameter types

@Transactional — The Full Picture

Propagation

Behaviour | Use when

REQUIRED (default)

Join existing transaction if one exists. Create new one if none. | 99% of cases — standard service method

REQUIRES_NEW

Always create a new transaction. Suspend the existing one if active. | Audit logging — must commit even if outer tx rolls back

NESTED

Create a savepoint within existing transaction. Rollback to savepoint on failure. | Partial rollback within a larger transaction

MANDATORY

NEVER

Must NOT have existing transaction. Throw if one exists. | Explicitly non-transactional operations

NOT_SUPPORTED

Run without transaction. Suspend existing if one exists. | Expensive read with no locking requirements

SUPPORTS

Use existing transaction if one exists. Run without one if none. | Read-only operations that can work either way

// @Transactional gotchas you must know
// GOTCHA 1: Self-invocation — the most common mistake
@Service
public class PaymentService {
    @Transactional
    public void processPayment(PaymentRequest req) {
        // This calls the method directly on 'this', NOT on the proxy
        // The @Transactional on updateAudit is IGNORED
        this.updateAudit(req); // WRONG
    }
    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void updateAudit(PaymentRequest req) {
        // This NEVER starts a new transaction when called via this.updateAudit()
        // because the call bypasses the proxy entirely
    }
}
// GOTCHA 2: @Transactional on a private method — silently ignored
@Transactional
private void internalMethod() { /* proxy cannot override private methods */ }
// GOTCHA 3: Default rollback rule — only rolls back on RuntimeException and Error
// Checked exceptions do NOT trigger rollback by default
@Transactional
public void transfer(String from, String to, BigDecimal amount) throws IOException {
    debit(from, amount);
    // IOException thrown here → transaction COMMITS (data is inconsistent)
    credit(to, amount);
}
// FIX: always specify rollbackFor for checked exceptions
@Transactional(rollbackFor = Exception.class)
public void transfer(String from, String to, BigDecimal amount) throws IOException { }
// GOTCHA 4: @Transactional on a final method (CGLIB)
@Transactional
public final void criticalOperation() {
    // CGLIB cannot override final methods — @Transactional silently ignored
}
// GOTCHA 5: REQUIRES_NEW and the outer transaction
@Transactional(propagation = Propagation.REQUIRES_NEW)
public void auditLog(String message) {
    // This opens its own transaction, commits independently
    // If outer transaction rolls back, the audit log still exists
    // This is exactly what you want for audit — NOT what you want for business logic
    auditRepo.save(new AuditEntry(message));
}
Q2: What are the @Transactional pitfalls that actually matter in production? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

Three that I have seen cause real incidents. First: self-invocation. Calling a @Transactional method from within the same class via 'this' bypasses the proxy — the transaction annotation has no effect. This is silent and will not throw an error, it will just not work. The fix is to inject the service into itself via ApplicationContext or restructure the code. Second: rollback only on RuntimeException by default. Checked exceptions do not trigger a rollback unless you add rollbackFor = Exception.class. I have seen services where a checked IOException left the database in a half-committed state. Third: @Transactional on private or final methods is silently ignored — CGLIB cannot proxy them. Always put @Transactional on public, non-final methods in beans that Spring controls.

Section 4 — @Async and Thread Pool Configuration

How @Async Works

The mechanism:

@Async is implemented via BeanPostProcessor (AsyncAnnotationBeanPostProcessor). It wraps the bean in a proxy. When an @Async method is called on the proxy, the proxy submits the method invocation to a TaskExecutor thread pool and returns immediately. The calling thread is not blocked. The method executes on a different thread.

// Enable async processing
@SpringBootApplication
@EnableAsync
public class PaymentApplication { }
// Simple @Async — uses Spring's default SimpleAsyncTaskExecutor (NOT production-ready)
@Service
public class NotificationService {
    @Async
    public void sendEmailNotification(PaymentResult result) {
        // Runs on a separate thread from the default executor
        emailClient.send(result.getUserEmail(), result.getSummary());
    }
    // @Async with a return value — caller gets a Future
    @Async
    public CompletableFuture<Boolean> sendPushNotification(PaymentResult result) {
        boolean sent = pushClient.send(result.getDeviceToken(), result.getSummary());
        return CompletableFuture.completedFuture(sent);
    }
}
// IMPORTANT: Configure your own ThreadPoolTaskExecutor — never use the default
// SimpleAsyncTaskExecutor creates a new thread per task — no pooling, no limits.
@Configuration
@EnableAsync
public class AsyncConfig implements AsyncConfigurer {
    @Override
    public Executor getAsyncExecutor() {
        ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
        executor.setCorePoolSize(5);          // always-alive threads
        executor.setMaxPoolSize(20);           // max threads under load
        executor.setQueueCapacity(200);        // bounded queue — back-pressure
        executor.setThreadNamePrefix("async-payment-");
        executor.setRejectedExecutionHandler(new ThreadPoolExecutor.CallerRunsPolicy());
        // CallerRunsPolicy: when queue is full, calling thread executes the task
        // This provides back-pressure instead of dropping tasks
        executor.initialize();
        return executor;
    }
    @Override
    public AsyncUncaughtExceptionHandler getAsyncUncaughtExceptionHandler() {
        // Exceptions in void @Async methods are silently swallowed without this
        return (ex, method, params) ->
            log.error("Async method {} threw: {}", method.getName(), ex.getMessage(), ex);
    }
}
// Multiple named executors for different task types
@Bean("paymentExecutor")
public TaskExecutor paymentExecutor() {
    ThreadPoolTaskExecutor e = new ThreadPoolTaskExecutor();
    e.setCorePoolSize(10); e.setMaxPoolSize(50); e.setQueueCapacity(500);
    e.setThreadNamePrefix("payment-"); e.initialize(); return e;
}
@Async("paymentExecutor")  // use specific named executor
public void processPaymentAsync(PaymentRequest req) { }
Q3: What happens to exceptions thrown in @Async void methods and how do you handle them? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Exceptions in @Async void methods are silently swallowed by default — the calling thread has already returned, there is nobody to receive the exception. This is a common source of bugs where notifications fail silently. The fix is to implement AsyncUncaughtExceptionHandler in your AsyncConfigurer and register it via getAsyncUncaughtExceptionHandler(). This handler receives the exception, the method reference, and the arguments, so you can log it properly or trigger an alert. For @Async methods that return CompletableFuture, the exception is captured in the future and the caller can handle it with exceptionally() or when the future is awaited. I always configure a custom uncaught exception handler and add an alert when async tasks fail — silent failures in notification or audit pipelines are the worst kind of bug to diagnose.

@Component
public class PaymentGatewayHealthIndicator implements HealthIndicator {
    private final PaymentGatewayClient client;
    @Override
    public Health health() {
        try {
            PingResponse ping = client.ping();

                return Health.up()
                    .withDetail("latency_ms", ping.getLatencyMs())
                    .withDetail("gateway", ping.getVersion())

                .withDetail("reason", "Cannot reach payment gateway")
                .build();
        }
    }
}

Micrometer — The Metrics API

What Micrometer is:

Micrometer is a vendor-neutral metrics API — like SLF4J but for metrics. You instrument your code with Micrometer and choose the backend (CloudWatch, Prometheus, Datadog, Influx) via a dependency. The same instrumentation code works regardless of backend.

Metric type

Use for | Example

Counter

Things that only increase — request count, error count, messages processed | payment.requests.total, payment.errors.total

Gauge

Current value that can go up or down — queue depth, active connections, cache size | kafka.consumer.lag, db.pool.active

Timer

Duration of an operation — includes count, sum, max, percentiles | payment.processing.time, http.server.requests

DistributionSummary

Distribution of values (not time) — request payload size, response size | payment.amount.distribution

LongTaskTimer

Duration of long-running tasks in progress — still running at query time | batch.job.running

// Micrometer instrumentation in a payment service
@Service
public class PaymentService {
    private final MeterRegistry  registry;
    private final Counter        paymentCounter;
    private final Counter        failureCounter;
    private final Timer          processingTimer;
    private final DistributionSummary amountSummary;
    public PaymentService(MeterRegistry registry) {
        this.registry = registry;
        this.paymentCounter = Counter.builder("payment.requests")
            .description("Total payment requests received")
            .tag("service", "payment")
            .register(registry);
        this.failureCounter = Counter.builder("payment.failures")
            .description("Total payment failures")
            .register(registry);
        this.processingTimer = Timer.builder("payment.processing.time")
            .description("Payment processing latency")
            .publishPercentiles(0.5, 0.95, 0.99)  // expose p50, p95, p99
            .register(registry);
        this.amountSummary = DistributionSummary.builder("payment.amount")
            .description("Distribution of payment amounts in INR")
            .baseUnit("INR")
            .register(registry);
    }
    public PaymentResult process(PaymentRequest req) {
        paymentCounter.increment();
        amountSummary.record(req.getAmount().doubleValue());
        return processingTimer.recordCallable(() -> {
            try {
                return doProcess(req);
            } catch (PaymentException e) {
                failureCounter.increment();  // tag could include failure reason
                throw e;
            }
        });
    }
}
// Gauge for a live value — Kafka consumer lag
Gauge.builder("kafka.consumer.lag", kafkaConsumer,

// @Timed shortcut — auto-instruments method with a Timer
@Timed(value = "payment.processing.time", percentiles = {0.5, 0.95, 0.99})
public PaymentResult process(PaymentRequest req) { return doProcess(req); }

Q4: How would you add p99 latency monitoring to a Spring Boot payment service? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

I use Micrometer's Timer with percentile publishing. Build a Timer with publishPercentiles(0.5, 0.95, 0.99) and register it against the MeterRegistry. Wrap the payment processing call inside timer.recordCallable(). This records the duration of every call and Micrometer automatically computes and publishes the p50, p95, and p99 percentiles. To export to CloudWatch, I add the micrometer-registry-cloudwatch2 dependency and set management.cloudwatch.metrics.export.enabled=true with the namespace. The EC2 instance needs a CloudWatch:PutMetricData IAM permission — the right way to grant this is via an IAM instance profile, not hardcoded credentials. Then I set a CloudWatch alarm on the p99 metric with a threshold of 500ms. In the Deloitte project, this is how we detected the compliance service latency issue — the p99 started climbing before p50 moved, which was the signal that a small fraction of requests were hitting a slow path.

Q5: How does Spring Boot load configuration and what is relaxed binding? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Spring Boot loads configuration from multiple sources in a defined priority order: command-line arguments override environment variables, which override application-{profile}.yml, which override application.yml, which override @PropertySource files. Relaxed binding means Spring is flexible about property name casing — payment.apiKey, payment.api-key, payment.api_key, and PAYMENT_API_KEY all bind to the same field in a @ConfigurationProperties class. This matters because environment variables typically use underscores and caps, but code uses camelCase or kebab-case. With @Value you have to match the exact property name. With @ConfigurationProperties, relaxed binding handles the translation automatically. For production config I always use @ConfigurationProperties over @Value for anything beyond a single property — it gives you type safety, validation with @Validated, IDE autocomplete through generated metadata, and clean testing with @TestPropertySource.

Interview Q&A — 15 Questions

How to use:

Answer out loud before reading. Every answer here is written the way you should speak it in an interview — clear, direct, with the trade-off or gotcha included.

Q6: What does @SpringBootApplication actually do? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

It is a composed annotation of three things. @SpringBootConfiguration marks the class as a configuration class — same as @Configuration, it just signals that this is the primary configuration. @ComponentScan scans the current package and all sub-packages for @Component, @Service, @Repository, and @Controller classes and registers them as beans. @EnableAutoConfiguration is the interesting one — it triggers the auto-configuration mechanism that reads all auto-config candidates from spring-boot-autoconfigure.jar, evaluates their @Conditional annotations, and registers only the ones that pass as bean definitions. Without @EnableAutoConfiguration you would have to manually configure DataSource, JPA, MVC, Jackson, and everything else that Spring Boot gives you automatically.

Q7: How does Spring Boot auto-configuration avoid creating beans you do not want? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

@Conditional annotations gate every auto-configured bean. The most important ones: @ConditionalOnClass only creates the bean if the specified class is on the classpath — so JPA auto-config only runs if Hibernate is present. @ConditionalOnMissingBean only creates the bean if you have not already defined one of the same type — this is the override mechanism that lets you replace any Spring Boot default by just defining your own bean. @ConditionalOnProperty gates on application.properties values — useful for feature flags. This layered approach means Spring Boot only configures what is actually needed, and you always win over the defaults without any explicit exclusions.

Q8: Walk me through the Spring bean lifecycle. [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

It starts with instantiation — Spring calls the constructor, injecting any constructor dependencies at this point. Then it processes field and setter injection for @Autowired. Then it checks for BeanNameAware and BeanFactoryAware and calls those setters. Then BeanPostProcessors run their before-init hook — this is where @Autowired validation happens. Then the initialisation methods run: @PostConstruct first, then afterPropertiesSet() if the bean implements InitializingBean, then any initMethod specified on @Bean. Then BeanPostProcessors run their after-init hook — this is where AOP proxies are created. The bean is now ready for use. On context shutdown, @PreDestroy runs first, then destroy() if the bean implements DisposableBean.

Q9: What is BeanPostProcessor and how does Spring AOP use it? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

BeanPostProcessor is an interface with two methods: postProcessBeforeInitialization and postProcessAfterInitialization. Spring calls these for every single bean in the context. The key detail is that the return value replaces the original bean — returning a different object is how proxies work. AbstractAutoProxyCreator, which is the BeanPostProcessor behind all AOP, looks at each bean in postProcessAfterInitialization: if the bean has methods annotated with @Transactional, @Async, @Cacheable, or matches an AOP pointcut, it creates a proxy wrapping the bean and returns the proxy. Everything in the context that holds a reference to that bean now holds a reference to the proxy, not the original.

Q10: Why does @Transactional sometimes not work? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

Four situations. First: self-invocation — calling a @Transactional method from within the same class via 'this' bypasses the proxy, so no transaction starts. Second: @Transactional on a private or final method — CGLIB proxies cannot override them, so the annotation is silently ignored. Third: the default rollback rule only rolls back on RuntimeException and Error — checked exceptions do not trigger rollback unless you add rollbackFor = Exception.class. I have seen this cause real data inconsistency. Fourth: the bean is not managed by Spring — if you create the object with 'new' instead of getting it from the context, there is no proxy and @Transactional has no effect.

Q11: What is the difference between @Transactional REQUIRED and REQUIRES_NEW? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

REQUIRED, the default, joins an existing transaction if one is active. If there is no active transaction, it starts a new one. All operations run in the same unit of work — if the outer transaction rolls back, everything rolls back. REQUIRES_NEW always starts a brand new transaction, suspending the existing one if any. The new transaction commits or rolls back independently. The classic use case: audit logging. You want the audit record to persist even when the main business transaction rolls back — put @Transactional(propagation = REQUIRES_NEW) on the audit method. If you used REQUIRED, the audit log would be rolled back along with the failed business transaction, and you would lose the audit trail of the failure.

Q12: How does @Async work and what goes wrong if you do not configure a thread pool? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

@Async is implemented via a BeanPostProcessor that wraps the bean in a proxy. When an @Async method is called on the proxy, the proxy submits the invocation to a TaskExecutor and returns immediately — the calling thread is not blocked. If you do not configure a TaskExecutor, Spring uses SimpleAsyncTaskExecutor, which creates a brand new thread for every single @Async call. Under load this means unbounded thread creation — each thread takes roughly 512KB of stack space, and with enough concurrent requests you will run out of memory or hit the OS thread limit. Always define a ThreadPoolTaskExecutor with explicit bounds. The other thing to know: exceptions in void @Async methods are silently swallowed unless you implement AsyncUncaughtExceptionHandler.

Q13: What is the difference between JDK dynamic proxy and CGLIB and when does Spring use each? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

JDK dynamic proxy uses java.lang.reflect.Proxy and requires the target class to implement at least one interface. The proxy implements the same interfaces. Only methods defined in the interfaces are intercepted — concrete methods not in any interface are not proxied. CGLIB generates a subclass of the target class at runtime using bytecode manipulation. It works even when the class has no interface. Since Spring Boot 2.x, CGLIB is used by default even when interfaces are present, because it avoids the interface mismatch issues. The practical limits of CGLIB: final classes cannot be subclassed — Spring will throw an exception if you try to proxy one. Final methods cannot be overridden — @Transactional on a final method will be silently ignored.

Q14: What is Spring Actuator and how do you secure it in production? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Actuator exposes management endpoints over HTTP: /health, /metrics, /info, /beans, /env, /threaddump, and others. It is invaluable in production for monitoring and diagnosis. In production, I configure it on a separate management port (management.server.port=8081) so it is never exposed on the same port as the public API — the management port is only accessible within the VPC or via a bastion host. I expose only health, metrics, and info publicly. I restrict /env, /beans, /loggers, and /heapdump to require authentication. For the health endpoint, I set show-details=when-authorized so the load balancer sees UP/DOWN without exposing internal component details to anyone who probes the endpoint.

Q15: What is Micrometer and what types of metrics does it support? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Micrometer is a vendor-neutral metrics API — you instrument your code once and choose the monitoring backend (CloudWatch, Prometheus, Datadog) by adding the corresponding dependency. The four main types: Counter for things that only increase, like request count or error count. Gauge for a current value that can go up or down, like queue depth or active DB connections. Timer for operation duration, which automatically tracks count, sum, max, and configurable percentiles (p50, p95, p99). DistributionSummary for the distribution of a value that is not time, like payment amount. I use @Timed as a convenient shortcut on methods to get a Timer automatically. Tags allow you to slice metrics by dimensions — I tag every payment timer by type (DEBIT, CREDIT, TRANSFER) so I can see if one type is slower than others.

Q16: How do you write a custom health indicator? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Implement the HealthIndicator interface and annotate the class with @Component. The health() method returns a Health object: Health.up(), Health.down(), or Health.status() for custom statuses. I add details with .withDetail(key, value) to give operators context — I always include the thing I checked, the result, and any relevant numbers like latency. A good custom health indicator checks whether a dependency your service actually needs is reachable — the payment gateway, Kafka broker, Redis. The health endpoint aggregates all indicators — if any returns DOWN, the overall status is DOWN. Load balancers typically call /health on a short interval and remove instances that return DOWN from the target group. So a health indicator that falsely reports DOWN will cause the instance to be deregistered unnecessarily — test your indicators carefully.

Q17: What is @ConfigurationProperties and why is it better than @Value? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

@ConfigurationProperties binds a whole prefix of properties to a POJO. @Value injects a single property. @ConfigurationProperties is better for more than a couple of properties because: it gives you relaxed binding so payment.api-key, payment.apiKey, and PAYMENT_API_KEY all bind to the same field; it supports JSR-303 validation with @Validated so you get startup-time failure if a required property is missing; it generates IDE metadata for autocomplete; and it is much easier to test — you can override a whole block of properties with @TestPropertySource or in @SpringBootTest properties. With @Value, you have to match the exact property name, there is no validation, and overriding individual fields in tests requires mocking or reflection.

Q18: How does Spring decide the order in which beans are created? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Spring builds a dependency graph from constructor injection and @Autowired relationships — if bean A requires bean B, B is created first. You can explicitly control order with @DependsOn('beanName') if there is no direct injection relationship but a creation order still matters, like a database schema migrator that must run before the data access layer. BeanPostProcessors are created early in the lifecycle before ordinary beans. @Configuration classes are processed before component-scanned beans. For auto-configurations, the @AutoConfiguration annotation supports before() and after() attributes to declare ordering relative to other auto-configurations. The general rule: use constructor injection wherever possible because it makes dependencies explicit and Spring will figure out the order automatically.

Q19: What is the prototype scope injection trap? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

If you inject a prototype-scoped bean into a singleton bean, the prototype bean is created once at injection time and then held by the singleton. From that point on, every call to the singleton's injected field gets the same prototype instance — it never gets a fresh one. The prototype effectively becomes a singleton. The fix depends on the use case. If you need a new instance per method call: inject ObjectProvider and call objectProvider.getObject() each time — Spring creates a new instance on every getObject() call. Alternatively use @Lookup method injection — Spring overrides the method to return a new prototype instance each time. The request and session scopes have the same problem when injected into singletons — fix with @ScopedProxy, which creates a proxy that delegates to the correct scoped instance for the current request or session.
