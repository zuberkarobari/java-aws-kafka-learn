# Resilience4j circuit breaker configuration
resilience4j:
  circuitbreaker:
    instances:
      fraud-service:
        # Sliding window: count-based (last N calls) or time-based (last N seconds)
        slidingWindowType: COUNT_BASED
        slidingWindowSize: 10          # evaluate last 10 calls
        minimumNumberOfCalls: 5        # need at least 5 calls before opening
        # Open circuit if failure rate > 50% OR slow calls > 80%
        failureRateThreshold: 50
        slowCallRateThreshold: 80
        slowCallDurationThreshold: 2000ms
        # How long to stay open before trying again
        waitDurationInOpenState: 10s
        # In half-open: allow 3 calls to test if service recovered
        permittedNumberOfCallsInHalfOpenState: 3
        automaticTransitionFromOpenToHalfOpenEnabled: true
        # What counts as failure (default: any exception)
        recordExceptions:
          - java.io.IOException
          - com.example.FraudServiceException
        ignoreExceptions:
          - com.example.FraudBlockedException  # this is a valid response, not a failure
// Usage with @CircuitBreaker
@Service
public class PaymentService {
    @CircuitBreaker(name = "fraud-service", fallbackMethod = "fraudFallback")
    public FraudResult checkFraud(PaymentRequest request) {
        return fraudServiceClient.check(request);
    }
    // Fallback receives the exception that triggered it
    private FraudResult fraudFallback(PaymentRequest request, Exception e) {
        log.warn("Fraud service unavailable, using fallback for paymentId={}", request.getId());
        // Return a conservative result: allow the payment but flag for manual review
        return FraudResult.requiresManualReview(request.getId());
    }
}

Bulkhead Pattern

Bulkhead type

How it works | Use when | Configuration

Semaphore

Limits concurrent calls to a service using a semaphore. If limit is reached, new calls are rejected immediately. | Default — low overhead, same thread. Use for most services. | maxConcurrentCalls, maxWaitDuration

Thread pool

Each decorated service gets its own dedicated thread pool. Calls run on pool threads, not the caller thread. | When you need true isolation — a slow service blocks its own pool threads, not the caller's. Higher overhead. | coreThreadPoolSize, maxThreadPoolSize, queueCapacity

# Semaphore bulkhead — limit concurrent calls to fraud service
resilience4j:
  bulkhead:
    instances:
      fraud-service:
        maxConcurrentCalls: 20     # max 20 concurrent fraud checks
        maxWaitDuration: 500ms     # wait up to 500ms for a permit
  # Thread pool bulkhead — isolated thread pool
  thread-pool-bulkhead:
    instances:
      slow-report-service:
        coreThreadPoolSize: 5
        maxThreadPoolSize: 10
        queueCapacity: 50
// @Bulkhead annotation
@Bulkhead(name = "fraud-service", fallbackMethod = "fraudFallback")
@CircuitBreaker(name = "fraud-service", fallbackMethod = "fraudFallback")
public FraudResult checkFraud(PaymentRequest request) { ... }

Retry with Exponential Backoff and Jitter

# Retry configuration
resilience4j:
  retry:
    instances:
      fraud-service:
        maxAttempts: 3
        waitDuration: 500ms
        enableExponentialBackoff: true
        exponentialBackoffMultiplier: 2   # 500ms, 1000ms, 2000ms
        exponentialMaxWaitDuration: 5s
        # Jitter: add randomness to prevent thundering herd
        # actual wait = waitDuration * (1 + random * jitterFactor)
        # Only retry on transient errors
        retryExceptions:
          - java.io.IOException
          - java.net.SocketTimeoutException
        ignoreExceptions:
          - com.example.ValidationException  # permanent error, do not retry
          - com.example.FraudBlockedException
// Correct decorator order for combined resilience
// Outermost to innermost: Retry → CircuitBreaker → Bulkhead → TimeLimiter → call
// This means:
// 1. Retry wraps everything — if circuit is open, retry backs off
//    Actually: Retry → CircuitBreaker means retry hits the circuit breaker check
//    If circuit is OPEN, circuit breaker throws immediately
//    Retry sees this as a failure and retries (hits circuit again — still open)
//    This is usually WRONG — you want Retry INSIDE CircuitBreaker
// CORRECT ORDER: CircuitBreaker → Retry → Bulkhead → TimeLimiter → call
// 1. Circuit breaker check: is circuit closed?
// 2. Retry: if call fails, retry (within the circuit breaker's window)
// 3. Bulkhead: is there capacity?
// 4. TimeLimiter: does call complete within deadline?
// 5. Actual call
// Using the Decorators builder for explicit ordering
CheckedFunction0<FraudResult> decorated = Decorators.ofCheckedSupplier(() -> fraudClient.check(req))
    .withCircuitBreaker(circuitBreaker)
    .withRetry(retry)
    .withBulkhead(bulkhead)
    .withTimeLimiter(timeLimiter, scheduledExecutorService)
    .decorate();
Q1: What is the correct order for combining resilience patterns and why does order matter? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

The recommended order from outermost to innermost is: CircuitBreaker → Retry → Bulkhead → TimeLimiter → actual call. The reasoning: the circuit breaker is the outermost because it makes the first decision — if the circuit is open, nothing else runs. Retry sits inside the circuit breaker, which means retries count toward the circuit breaker's failure rate. If the call fails and retry also fails, both attempts count as failures in the circuit breaker's sliding window, which is the correct behaviour — the circuit breaker should see the full failure picture. Bulkhead limits concurrency at the point just before the call is made. TimeLimiter enforces the per-call deadline. The wrong order: Retry outside the circuit breaker means a failed call triggers retries that each hit the circuit breaker check — if the circuit opens during the retry sequence, the retry throws immediately. That changes the semantics in subtle ways. When using @annotations in Spring, the annotation that appears first in the annotation list is the outermost decorator.

Interview Q&A — 12 Questions

Q2: How does a semaphore bulkhead differ from a thread pool bulkhead? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

A semaphore bulkhead uses a counting semaphore to limit the number of concurrent calls. When the maximum is reached, new callers block for up to maxWaitDuration and then receive a BulkheadFullException. The calls execute on the caller's thread — no thread context switching, low overhead. A thread pool bulkhead allocates a dedicated thread pool for the protected service. Calls are submitted to the pool and the calling thread returns immediately (asynchronously). The thread pool provides true thread isolation — if the downstream service is slow and fills its pool threads, the calling threads are not blocked. The cost is higher overhead (thread creation, context switching) and the need to handle async results. Use semaphore bulkheads by default. Use thread pool bulkheads when you need the caller to remain responsive even while many calls are in flight.

Q3: Why does retry need jitter and what is the thundering herd problem? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Without jitter, all retries happen at the same time after the same delay. If 1,000 consumers all fail at t=0 and all retry at t=1s, they create another spike at t=1s that might fail again, and then all retry at t=2s — a synchronized wave of retries that can overwhelm a service trying to recover. Jitter adds randomness to the retry delay: instead of all retrying exactly at 1 second, they retry at random intervals between 0.8 and 1.2 seconds (with 20% jitter). The load is spread over the interval instead of spiking. The formula: actual_wait = base_wait * (1 + random * jitter_factor). Jitter is especially important in microservices where many consumer instances all fail simultaneously when a service goes down.

Q4: What is the difference between a TimeLimiter and an HTTP client timeout? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

An HTTP client timeout (like WebClient's responseTimeout or RestTemplate's connectTimeout/readTimeout) is a connection-level timeout — it cancels the outgoing HTTP request after the configured duration. A TimeLimiter is a Resilience4j decorator that applies a deadline to any callable, not just HTTP calls. It submits the work to a ScheduledExecutorService and cancels it if it does not complete within the duration. The key difference: a TimeLimiter wraps any operation (HTTP call, database query, complex computation) and integrates with the Resilience4j metrics and circuit breaker. When it fires, it also records the result as a timeout in the circuit breaker's sliding window, potentially contributing to opening the circuit. An HTTP client timeout only applies to HTTP and does not feed circuit breaker metrics. For HTTP calls, I configure both: the HTTP client timeout for connection-level control, and a TimeLimiter for application-level deadline enforcement with circuit breaker integration.

Q5: What is the difference between failureRateThreshold and slowCallRateThreshold? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

failureRateThreshold opens the circuit when the percentage of failed calls (calls that threw an exception) exceeds the threshold in the sliding window. slowCallRateThreshold opens the circuit when the percentage of slow calls exceeds the threshold — calls that completed successfully but took longer than slowCallDurationThreshold. The slow call threshold is important because a service can be technically responding without errors but still be too slow to be useful. An API that takes 30 seconds per call is functionally failing even if it returns HTTP 200. Configuring both gives the circuit breaker a complete picture of service health: failing fast OR being very slow, either should open the circuit. In practice I set slowCallDurationThreshold to 2-3x the normal p99 latency — enough to catch genuine degradation without triggering on normal variance.

Q6: What should a fallback do and what are the different fallback strategies? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

A fallback is what you return when the circuit is open or the call fails. The strategy depends on what the calling code needs. Return a cached value — the last successful response, stored in Redis. Safe for read operations where slightly stale data is acceptable. Return a degraded result — a partial response with default values. For fraud check, return 'unknown' and flag for manual review rather than blocking the payment. Throw a specific business exception — for operations that cannot proceed without the dependency, fail with a clear error (503, Retry-After header) rather than a confusing null or default. Return nothing (for optional services) — if the notification service is down, the payment still completes. The notification is queued for retry later. The key principle: the fallback should be a deliberate business decision, not just an empty return.

Q7: How do Resilience4j patterns interact with Spring's @Transactional? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

This is a subtle integration question. If @CircuitBreaker is applied to a @Transactional method, the Spring proxy order matters. If the transaction starts before the circuit breaker check and the circuit is open, the transaction is started and then immediately rolled back when the circuit throws CallNotPermittedException. This is wasteful but usually harmless. The more important case: if the circuit breaker fallback method is called, the transaction context depends on where the fallback runs. If the fallback method is in the same class, the @Transactional annotation may or may not apply depending on how the proxy is structured. The recommended approach: put the Resilience4j annotations on a service layer that does not have @Transactional, and put @Transactional on a separate repository or persistence layer. This avoids the ambiguity.

Q8: What is the difference between bulkhead and rate limiter in Resilience4j? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

A bulkhead limits concurrent executions — how many calls can be in flight simultaneously. If 20 calls are already being processed and a 21st arrives, the bulkhead rejects it immediately. A rate limiter limits execution frequency — how many calls can be made per time period. If the limit is 100 calls per second and 101 calls arrive in one second, the 101st is rejected (or waits, depending on the strategy). Bulkhead prevents cascading failure from a slow service (too many concurrent waits). Rate limiter prevents overloading a service with traffic spikes. They address different failure modes and are often used together: rate limiter as the outer control (prevent too many calls per second) and bulkhead as the inner control (prevent too many concurrent in-flight calls).

Q9: When should you NOT retry a failed call? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Do not retry: non-idempotent operations (POST, PATCH with side effects) unless the server has idempotency key support. Do not retry on validation errors (400, 422) — the same request will always fail. Do not retry on authentication errors (401, 403) — a different request with different credentials might succeed, but retrying the same request does not help. Do not retry when the circuit breaker is open — Retry sitting outside CircuitBreaker means retries hit the circuit check; if open, they all fail immediately and count as failures in the next window. Do not retry on business-logic errors from the downstream service — if fraud check returns BLOCKED, retrying will get the same answer. Only retry on transient infrastructure errors: connection timeouts, network failures, 503 Service Unavailable, 504 Gateway Timeout.

Q10: How do you monitor the health of your resilience patterns in production? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Resilience4j exposes Micrometer metrics for all its patterns. For circuit breakers: resilience4j.circuitbreaker.state (a Gauge that is 0=CLOSED, 1=OPEN, 2=HALF_OPEN), resilience4j.circuitbreaker.failure.rate (percentage), resilience4j.circuitbreaker.slow.call.rate. I set up alerts: if circuit state stays at OPEN for more than 60 seconds, page on-call — this means a service has been failing for a full minute. For retries: resilience4j.retry.calls tagged with kind=failed.with.retry (succeeded after retry) and kind=failed.without.retry (all retries exhausted). High failed.without.retry rate signals a persistent problem. For bulkheads: resilience4j.bulkhead.available.concurrent.calls approaching 0 means the bulkhead is nearly full. Spring Actuator also exposes a /actuator/circuitbreakers endpoint showing the current state of all circuit breakers, which is useful for operational dashboards.
