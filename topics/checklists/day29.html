QuickSort — Internals

// QuickSort — in-place, average O(n log n), worst O(n^2) on sorted input
// Pivot choice matters: last element is simple but bad for sorted arrays
// Median-of-three pivot avoids worst case in practice
public void quickSort(int[] nums, int lo, int hi) {
    if (lo >= hi) return;
    int pivot = partition(nums, lo, hi);
    quickSort(nums, lo, pivot - 1);
    quickSort(nums, pivot + 1, hi);
}
// Lomuto partition: pivot is last element
// Everything left of write pointer is <= pivot
private int partition(int[] nums, int lo, int hi) {
    int pivot = nums[hi];
    int write = lo; // write pointer: boundary of elements <= pivot
    for (int read = lo; read < hi; read++) {
        if (nums[read] <= pivot) swap(nums, write++, read);
    }
    swap(nums, write, hi); // place pivot in its final position
    return write;
}
// Why QuickSort is faster than MergeSort in practice despite same Big-O:
// 1. In-place — no extra O(n) allocation, better cache behaviour
// 2. Inner loop is simpler — fewer operations per comparison
// 3. Constant factor in O(n log n) is smaller for QuickSort
// Worst case O(n^2): already sorted array with last-element pivot
// Every partition puts 0 elements on one side, n-1 on the other
// Fix: randomise pivot selection
private int partitionRandom(int[] nums, int lo, int hi) {
    int randIdx = lo + (int)(Math.random() * (hi - lo + 1));
    swap(nums, randIdx, hi); // move random pivot to end
    return partition(nums, lo, hi);
}
private void swap(int[] a, int i, int j) {
    int tmp = a[i]; a[i] = a[j]; a[j] = tmp;
}

MergeSort — Internals

// MergeSort — O(n log n) guaranteed, O(n) space, stable
// Divide array in half recursively, merge sorted halves
public void mergeSort(int[] nums, int lo, int hi) {
    if (lo >= hi) return;
    int mid = lo + (hi - lo) / 2; // avoids integer overflow vs (lo+hi)/2
    mergeSort(nums, lo, mid);
    mergeSort(nums, mid + 1, hi);
    merge(nums, lo, mid, hi);
}
private void merge(int[] nums, int lo, int mid, int hi) {
    int[] left  = Arrays.copyOfRange(nums, lo, mid + 1);
    int[] right = Arrays.copyOfRange(nums, mid + 1, hi + 1);
    int i = 0, j = 0, k = lo;
    while (i < left.length && j < right.length)
        nums[k++] = (left[i] <= right[j]) ? left[i++] : right[j++];
    while (i < left.length)  nums[k++] = left[i++];
    while (j < right.length) nums[k++] = right[j++];
}
// Why Java uses MergeSort for objects but QuickSort for primitives:
// Objects: need STABLE sort (equal objects preserve original order).
//          MergeSort is stable. QuickSort is not.
// Primitives: stability does not matter (int 5 == int 5, no identity).
//             QuickSort's cache-friendliness wins.
// MergeSort application: count inversions
// An inversion is a pair (i, j) where i < j but nums[i] > nums[j]
// Key: during merge, when a right element is placed before left elements,
// each remaining left element forms an inversion with it.
// Time: O(n log n) — same as merge sort, just add a counter during merge
long inversions = 0; // global counter in merge step
// In merge: when right[j] < left[i], add (left.length - i) to inversions
Q1: QuickSort vs MergeSort — when would you choose each? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

QuickSort is faster in practice for in-memory sorting of primitives because it is cache-friendly and has a smaller constant factor. It works in-place, so there is no extra memory allocation. The downside is the worst case — O(n^2) on already sorted data with a naive pivot choice. You fix this by randomising the pivot. MergeSort guarantees O(n log n) regardless of input, is stable (equal elements preserve their original order), and is the right choice when stability matters — sorting objects by one field when they are already sorted by another field. It also works well for external sorting when data does not fit in memory, because you can merge sorted chunks from disk. Java uses TimSort (a MergeSort hybrid) for object arrays for exactly this reason — stability is needed for objects. For primitive arrays, Java uses a dual-pivot QuickSort which avoids the worst case by design.

Section 2 — Binary Search: One Template That Always Works

The Off-By-One Problem and How to Solve It Once

Why binary search is harder than it looks:

The standard binary search looks simple but has subtle bugs — when to use lo <= hi vs lo < hi, when to set mid = lo + (hi - lo) / 2 vs mid = (lo + hi) / 2, when to set hi = mid - 1 vs hi = mid. Different sources use different conventions and mixing them causes infinite loops or missed elements. The approach here uses one consistent template for all variants. Learn this and nothing else.

// THE ONE TEMPLATE — works for exact search and boundary search
// Loop invariant: the answer is always within [lo, hi]
// Terminates when lo > hi
// VARIANT 1: Find exact target — return index or -1
public int binarySearch(int[] nums, int target) {
    int lo = 0, hi = nums.length - 1;
    while (lo <= hi) {
        int mid = lo + (hi - lo) / 2; // avoids overflow; same as (lo+hi)/2
        if      (nums[mid] == target) return mid;
        else if (nums[mid] <  target) lo = mid + 1;
        else                          hi = mid - 1;
    }
    return -1;
}
// VARIANT 2: Find FIRST occurrence (leftmost) of target
// When you find target, do not stop — continue searching LEFT
public int findFirst(int[] nums, int target) {
    int lo = 0, hi = nums.length - 1, result = -1;
    while (lo <= hi) {
        int mid = lo + (hi - lo) / 2;
        if (nums[mid] == target) {
            result = mid;   // record this as a candidate
            hi = mid - 1;   // keep searching LEFT for an earlier occurrence
        } else if (nums[mid] < target) lo = mid + 1;
        else                           hi = mid - 1;
    }
    return result;
}
// VARIANT 3: Find LAST occurrence (rightmost) of target
// When you find target, do not stop — continue searching RIGHT
public int findLast(int[] nums, int target) {
    int lo = 0, hi = nums.length - 1, result = -1;
    while (lo <= hi) {
        int mid = lo + (hi - lo) / 2;
        if (nums[mid] == target) {
            result = mid;   // record this as a candidate
            lo = mid + 1;   // keep searching RIGHT for a later occurrence
        } else if (nums[mid] < target) lo = mid + 1;
        else                           hi = mid - 1;
    }
    return result;
}
// VARIANT 4: Find insertion point (first position where nums[i] >= target)
// Also called lower_bound in C++
// Returns index where target would be inserted to keep array sorted
public int searchInsert(int[] nums, int target) {
    int lo = 0, hi = nums.length; // hi = length, not length-1
    while (lo < hi) {             // lo < hi, not lo <= hi
        int mid = lo + (hi - lo) / 2;
        if (nums[mid] < target) lo = mid + 1;
        else                    hi = mid;  // hi = mid, not mid - 1
    }
    return lo; // lo == hi at termination — this is the insertion point
}
// PROBLEM: Count occurrences of target in sorted array
// findLast(target) - findFirst(target) + 1
// Time: O(log n)
public int countOccurrences(int[] nums, int target) {
    int first = findFirst(nums, target);
    if (first == -1) return 0;
    return findLast(nums, target) - first + 1;
}

Variant

Loop condition | hi update on match | lo update on match | Returns

Exact search

lo <= hi | hi = mid - 1 | lo = mid + 1 | mid when found, -1 if not

First occurrence

lo <= hi | result=mid; hi=mid-1 | lo = mid + 1 | result (-1 if not found)

Last occurrence

lo <= hi | result=mid; lo=mid+1 | hi = mid - 1 | result (-1 if not found)

Insertion point

lo < hi | hi = mid (not mid-1) | lo = mid + 1 | lo (always valid index)

Q2: Why do you write mid = lo + (hi - lo) / 2 instead of (lo + hi) / 2? [Easy] ▼
💡 Ideal Answer (Amazon/GS Standard)

With lo + hi, if both lo and hi are large positive integers, their sum can overflow a 32-bit int. For a 1 billion element array, lo and hi could both be around 500 million. Their sum would be 1 billion, which exceeds Integer.MAX_VALUE of about 2.1 billion — actually it would not overflow in this specific example but it becomes an issue when both are near Integer.MAX_VALUE. Writing lo + (hi - lo) / 2 computes the same midpoint but keeps the arithmetic within bounds because hi - lo is always less than hi. In competitive programming and system code with large arrays this matters. In Java interviews, stating this shows you are thinking about correctness in edge cases, which is exactly the signal a senior interview is looking for.

Section 3 — Binary Search on Rotated and Irregular Arrays

Search in Rotated Sorted Array

The key insight:

When you split a rotated sorted array at any mid point, at least one half is always normally sorted. Identify which half is sorted by comparing nums[lo] with nums[mid]. Once you know which half is sorted, you can determine in O(1) whether the target is in that half or must be in the other half.

// PROBLEM: Search in Rotated Sorted Array
// Array: [4, 5, 6, 7, 0, 1, 2], target = 0 → index 4
// Time: O(log n)  Space: O(1)
public int searchRotated(int[] nums, int target) {
    int lo = 0, hi = nums.length - 1;
    while (lo <= hi) {
        int mid = lo + (hi - lo) / 2;
        if (nums[mid] == target) return mid;
        // Determine which half is sorted
        if (nums[lo] <= nums[mid]) {
            // LEFT half [lo..mid] is sorted
            if (target >= nums[lo] && target < nums[mid])
                hi = mid - 1; // target is in the sorted left half
            else
                lo = mid + 1; // target must be in the right half
        } else {
            // RIGHT half [mid..hi] is sorted
            if (target > nums[mid] && target <= nums[hi])
                lo = mid + 1; // target is in the sorted right half
            else
                hi = mid - 1; // target must be in the left half
        }
    }
    return -1;
}
// PROBLEM: Find Minimum in Rotated Sorted Array
// The minimum is always in the unsorted half
// Time: O(log n)  Space: O(1)
public int findMin(int[] nums) {
    int lo = 0, hi = nums.length - 1;
    while (lo < hi) {
        int mid = lo + (hi - lo) / 2;
        if (nums[mid] > nums[hi])
            lo = mid + 1; // mid is in the left sorted half — min is to the right
        else
            hi = mid;     // mid could be the minimum — do not exclude it
    }
    return nums[lo]; // lo == hi — this is the minimum
}

Find Peak Element

Why binary search works on an unsorted array:

A peak element is one that is strictly greater than its neighbours. There is always at least one peak — if you walk from the higher of nums[mid] and nums[mid+1], you always move toward a peak. This monotonicity is enough to apply binary search even though the array is not sorted.

// PROBLEM: Find Peak Element
// nums[i] != nums[i+1]. Return any peak index.
// Time: O(log n)  Space: O(1)
public int findPeakElement(int[] nums) {
    int lo = 0, hi = nums.length - 1;
    while (lo < hi) {
        int mid = lo + (hi - lo) / 2;
        if (nums[mid] > nums[mid + 1])
            hi = mid;     // peak is at mid or to the left — do not exclude mid
        else
            lo = mid + 1; // nums[mid+1] > nums[mid] — peak is to the right
    }
    return lo; // lo == hi — this is a peak
}
// PROBLEM: Search a 2D Matrix (rows and columns each sorted)
// Start from top-right corner: if target < current, go left; if target > current, go down
// This is not binary search — it is the staircase search
// Time: O(m + n)  Space: O(1)
public boolean searchMatrix(int[][] matrix, int target) {
    int row = 0, col = matrix[0].length - 1; // start at top-right
    while (row < matrix.length && col >= 0) {
        if      (matrix[row][col] == target) return true;
        else if (matrix[row][col] >  target) col--;  // too big — move left
        else                                 row++;  // too small — move down
    }
    return false;
}
// PROBLEM: Search a 2D Matrix where each row is sorted AND
// the first element of each row > last element of previous row
// This is effectively a sorted 1D array — use direct binary search
// Time: O(log(m*n))  Space: O(1)
public boolean searchMatrixStrict(int[][] matrix, int target) {
    int m = matrix.length, n = matrix[0].length;
    int lo = 0, hi = m * n - 1;
    while (lo <= hi) {
        int mid  = lo + (hi - lo) / 2;
        int val  = matrix[mid / n][mid % n]; // convert 1D index to 2D
        if      (val == target) return true;
        else if (val <  target) lo = mid + 1;
        else                    hi = mid - 1;
    }
    return false;
}
Q3: How do you search in a rotated sorted array? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

The key observation is that when you split a rotated sorted array at any midpoint, at least one of the two halves is guaranteed to be normally sorted. I compare nums[lo] with nums[mid] to figure out which half is sorted. If nums[lo] is less than or equal to nums[mid], the left half is sorted. I then check whether the target falls within the left half's range. If yes, I search there. If no, I search the right half. If nums[lo] is greater than nums[mid], the right half is sorted, and I do the same check. This works in O(log n) because at each step I eliminate half the array. The condition to check whether the target is in the sorted half is important: for the left half, target must be between nums[lo] and nums[mid - 1] inclusive. Missing the boundary conditions is where most people get this wrong.

Section 4 — Binary Search on Answer Space

The Most Underused Binary Search Pattern

The idea:

Not all binary search problems ask you to find a value in an array. Some ask you to find the minimum or maximum value that satisfies a condition. If you can write a function that checks whether a given answer is feasible, and the feasibility is monotonic (once feasible, always feasible above a threshold), you can binary search on the answer itself.

// PROBLEM: Koko Eating Bananas
// Piles of bananas. Must eat all in h hours.
// Find minimum eating speed k (bananas per hour).
// Key: if speed k works, any speed > k also works — monotonic.
// Binary search on k between 1 and max(piles).
// Time: O(n log(max_pile))  Space: O(1)
public int minEatingSpeed(int[] piles, int h) {
    int lo = 1, hi = Arrays.stream(piles).max().getAsInt();
    while (lo < hi) {
        int mid = lo + (hi - lo) / 2;
        if (canFinish(piles, h, mid)) hi = mid;  // mid works — try smaller
        else                          lo = mid + 1; // mid too slow — go faster
    }
    return lo;
}
private boolean canFinish(int[] piles, int h, int speed) {
    int hours = 0;
    for (int pile : piles)
        hours += (pile + speed - 1) / speed; // ceiling division
    return hours <= h;
}
// PROBLEM: Find minimum capacity to ship packages in D days
// Same pattern — binary search on the answer (capacity)
// lo = max(weights), hi = sum(weights)
public int shipWithinDays(int[] weights, int days) {
    int lo = Arrays.stream(weights).max().getAsInt(); // must fit heaviest package
    int hi = Arrays.stream(weights).sum();            // ship all in one day
    while (lo < hi) {
        int mid = lo + (hi - lo) / 2;
        if (canShip(weights, days, mid)) hi = mid;
        else                             lo = mid + 1;
    }
    return lo;
}
private boolean canShip(int[] weights, int days, int capacity) {
    int daysNeeded = 1, currentLoad = 0;
    for (int w : weights) {
        if (currentLoad + w > capacity) { daysNeeded++; currentLoad = 0; }
        currentLoad += w;
    }
    return daysNeeded <= days;
}
// PROBLEM: Kth Smallest Element in a Sorted Matrix
// Binary search on the answer value (not the index)
// For a given value mid, count how many elements <= mid
// Time: O(n log(max - min))  Space: O(1)
public int kthSmallest(int[][] matrix, int k) {
    int n = matrix.length;
    int lo = matrix[0][0], hi = matrix[n-1][n-1];
    while (lo < hi) {
        int mid   = lo + (hi - lo) / 2;
        int count = countLessEqual(matrix, mid, n);
        if (count < k) lo = mid + 1;
        else           hi = mid;
    }
    return lo;
}
private int countLessEqual(int[][] matrix, int mid, int n) {
    int count = 0, row = n - 1, col = 0;
    while (row >= 0 && col < n) {
        if (matrix[row][col] <= mid) { count += row + 1; col++; }
        else                          row--;
    }
    return count;
}
Q4: What is binary search on the answer space and when does it apply? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

Binary search on the answer space applies when the problem asks for a minimum or maximum value satisfying some condition, and that condition is monotonic — meaning if value k satisfies it, every value above k also satisfies it, or vice versa. Instead of searching an array, you search the range of possible answers. You define the search bounds as the minimum and maximum plausible answers. At each midpoint you ask: is this answer feasible? If yes, try smaller (or larger, depending on whether you want minimum or maximum). If no, go the other direction. The signal in the problem is usually phrasing like find the minimum speed, find the minimum capacity, find the smallest X such that some condition holds. The trick is writing the feasibility check function correctly. Once you have that function, the binary search wrapper is always the same shape.

Section 5 — Kth Element and Heap Patterns

Three Approaches to Kth Largest

Approach

Time | Space | Choose when

Sort, return index

O(n log n) | O(1) | Simple but overkill — sorts the whole array when you only need one element

Min-heap of size k

O(n log k) | O(k) | Best for streaming data or very large arrays — keeps only k elements in memory

QuickSelect (partition)

O(n) average, O(n^2) worst | O(1) | Best average performance for in-memory arrays where worst case is acceptable

// APPROACH 1: Min-heap of size k
// Keep a min-heap of the k largest elements seen so far
// The top of the heap is always the kth largest
// Time: O(n log k)  Space: O(k)
public int findKthLargest(int[] nums, int k) {
    PriorityQueue<Integer> minHeap = new PriorityQueue<>(); // min-heap by default
    for (int num : nums) {
        minHeap.offer(num);
        if (minHeap.size() > k) minHeap.poll(); // remove smallest if over k
    }
    return minHeap.peek(); // top is kth largest
}
// APPROACH 2: QuickSelect — average O(n)
// Same partition logic as QuickSort, but only recurse on ONE side
// We want the element at index (n - k) in sorted order
public int findKthLargestQuickSelect(int[] nums, int k) {
    int target = nums.length - k; // target index in sorted order
    return quickSelect(nums, 0, nums.length - 1, target);
}
private int quickSelect(int[] nums, int lo, int hi, int target) {
    int pivot = partition(nums, lo, hi);
    if      (pivot == target) return nums[pivot];
    else if (pivot <  target) return quickSelect(nums, pivot + 1, hi, target);
    else                      return quickSelect(nums, lo, pivot - 1, target);
}
private int partition(int[] nums, int lo, int hi) {
    int pivot = nums[hi], write = lo;
    for (int i = lo; i < hi; i++)
        if (nums[i] <= pivot) swap(nums, write++, i);
    swap(nums, write, hi);
    return write;
}
// PROBLEM: Top K Frequent Elements
// Count frequencies, then use a min-heap of size k on (frequency, element)
// Time: O(n log k)  Space: O(n)
public int[] topKFrequent(int[] nums, int k) {
    Map<Integer, Integer> freq = new HashMap<>();
    for (int num : nums) freq.merge(num, 1, Integer::sum);
    // Min-heap ordered by frequency — keep only top k
    PriorityQueue<int[]> minHeap =
        new PriorityQueue<>(Comparator.comparingInt(a -> a[1]));
    for (Map.Entry<Integer, Integer> e : freq.entrySet()) {
        minHeap.offer(new int[]{e.getKey(), e.getValue()});
        if (minHeap.size() > k) minHeap.poll();
    }
    return minHeap.stream().mapToInt(a -> a[0]).toArray();
}
private void swap(int[] a, int i, int j) {
    int tmp = a[i]; a[i] = a[j]; a[j] = tmp;
}
Q5: When would you use a heap versus QuickSelect for the Kth largest problem? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

The choice depends on the context. QuickSelect is O(n) average with O(1) extra space and is the fastest option for an in-memory array where you can modify the data. The downside is O(n^2) worst case with bad pivot choices, and it modifies the original array. A min-heap of size k is O(n log k) but handles streaming data naturally — you do not need all elements in memory at once, and you do not modify the input. If I am processing a live event stream and need to track the top 100 amounts seen so far, the heap approach is the right model because elements arrive one at a time. If I have a fixed array in memory and want the fastest average case without worrying about modifying the input, QuickSelect wins. Java's PriorityQueue is a min-heap by default, so to find the kth largest you keep a heap of size k — the root is always the kth largest element seen so far.

Section 6 — Interval Problems

Merge Intervals and Overlap Detection

The two-step pattern for interval problems:

// PROBLEM: Merge Overlapping Intervals
// Input:  [[1,3],[2,6],[8,10],[15,18]]
// Output: [[1,6],[8,10],[15,18]]
// Time: O(n log n)  Space: O(n)
public int[][] merge(int[][] intervals) {
    Arrays.sort(intervals, Comparator.comparingInt(a -> a[0])); // sort by start
    List<int[]> result = new ArrayList<>();
    for (int[] interval : intervals) {
        if (result.isEmpty() || result.get(result.size()-1)[1] < interval[0]) {
            result.add(interval); // no overlap — add as new interval
        } else {
            // Overlap — extend the end of the last interval if needed
            result.get(result.size()-1)[1] =
                Math.max(result.get(result.size()-1)[1], interval[1]);
        }
    }
    return result.toArray(new int[0][]);
}
// PROBLEM: Meeting Rooms II — minimum rooms for all meetings
// Equivalent to: maximum number of meetings happening at the same time
// Approach: sort start times and end times separately.
// Walk through meetings in start order. If next meeting starts before
// earliest-ending current meeting ends, we need a new room.
// Time: O(n log n)  Space: O(n)
public int minMeetingRooms(int[][] intervals) {
    int n = intervals.length;
    int[] starts = new int[n], ends = new int[n];
    for (int i = 0; i < n; i++) {
        starts[i] = intervals[i][0];
        ends[i]   = intervals[i][1];
    }
    Arrays.sort(starts);
    Arrays.sort(ends);
    int rooms = 0, endPtr = 0;
    for (int i = 0; i < n; i++) {
        if (starts[i] < ends[endPtr])
            rooms++;          // new meeting starts before earliest ends — need new room
        else
            endPtr++;         // a meeting has ended — reuse its room
    }
    return rooms;
}
// PROBLEM: Insert Interval
// Insert a new interval into a sorted non-overlapping list
// Time: O(n)  Space: O(n)
public int[][] insert(int[][] intervals, int[] newInterval) {
    List<int[]> result = new ArrayList<>();
    int i = 0, n = intervals.length;
    // Add all intervals that end before the new interval starts
    while (i < n && intervals[i][1] < newInterval[0])
        result.add(intervals[i++]);
    // Merge all overlapping intervals with the new interval
    while (i < n && intervals[i][0] <= newInterval[1]) {
        newInterval[0] = Math.min(newInterval[0], intervals[i][0]);
        newInterval[1] = Math.max(newInterval[1], intervals[i][1]);
        i++;
    }
    result.add(newInterval);
    // Add remaining intervals
    while (i < n) result.add(intervals[i++]);
    return result.toArray(new int[0][]);
}

Exercise 7 — BOSS: Median of Two Sorted Arrays

Hard problem — binary search on partition point:

Given two sorted arrays nums1 and nums2, find the median of the combined sorted array in O(log(min(m, n))) time. The key insight: finding the median is equivalent to finding the correct partition point that splits the combined array into two halves of equal size. Binary search on the partition point of the smaller array. At the correct partition, maxLeft1 <= minRight2 and maxLeft2 <= minRight1. If maxLeft1 > minRight2, move the partition left; if maxLeft2 > minRight1, move it right. Handle edge cases: empty left or right halves by using Integer.MIN_VALUE and Integer.MAX_VALUE as sentinels. This is one of the hardest binary search problems in interviews — work through a concrete example before coding.

Interview Q&A — 12 Questions

How to use:

State the pattern and complexity before reading the answer. Every answer here is written the way you would say it out loud in a room — direct, with the trade-off included.

Q6: What is the worst case for QuickSort and how do you avoid it? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

The worst case is O(n^2) and it happens when every partition is maximally unbalanced — one side gets zero elements and the other gets n-1. This occurs with a sorted or reverse-sorted array when the pivot is always the first or last element. Each partition produces only one useful split, giving n levels of recursion instead of log n. The fix is to randomise pivot selection. Pick a random index in the current range, swap it with the last element, then partition normally. With a random pivot, the probability of consistently bad splits is exponentially small. A more deterministic approach is median-of-three: take the median of the first, middle, and last elements as the pivot. Java's built-in dual-pivot QuickSort handles this by using two pivots and three-way partitioning, which also handles arrays with many duplicate elements much better than single-pivot QuickSort.

Q7: Why does Java use different sorting algorithms for primitives vs objects? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

For primitive arrays, Java uses a dual-pivot QuickSort. For object arrays, it uses TimSort which is a hybrid of MergeSort and InsertionSort. The reason for the split is stability. When you sort int arrays, two equal integers are indistinguishable — there is no identity to preserve, so stability has no meaning. QuickSort wins on raw speed because it is cache-friendly and in-place. When you sort object arrays, equal objects still have identity. If you sort a list of payments by amount and two payments have the same amount, a stable sort preserves their original relative order — which might reflect their insertion order or a previous sort by date. Breaking stability would be a subtle bug. MergeSort is stable by construction: when merging two sorted halves, equal elements from the left half always go before equal elements from the right half.

Q8: What is binary search and what are the conditions for using it? [Easy] ▼
💡 Ideal Answer (Amazon/GS Standard)

Binary search finds a value (or a boundary) in a sorted collection in O(log n) by repeatedly halving the search space. The conditions: the data must be sorted, or the search space must have a monotonic property. For classic binary search that is a sorted array. For binary search on answer space, the condition is that the feasibility function is monotonic — once an answer becomes feasible, all larger (or smaller) answers are also feasible. The most common mistake is getting the boundary conditions wrong — using lo < hi versus lo <= hi, and updating hi to mid versus mid - 1. I keep one template that I use consistently for all variants, adjusting only the match condition and which side to record the result on.

Q9: How do you find the first and last occurrence of a target in a sorted array with duplicates? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Two separate binary searches, each O(log n). For the first occurrence: run standard binary search, but when you find the target at mid, do not stop. Record mid as the current best answer and continue searching left by setting hi to mid minus one. For the last occurrence: same approach but when you find the target, record mid and continue searching right by setting lo to mid plus one. The key insight is that finding the target is not the termination condition — reaching a state where lo exceeds hi is. The total count of a value in the array is findLast minus findFirst plus one, computed in two log n passes.

Q10: How does searching in a rotated sorted array differ from a standard binary search? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

In a standard sorted array, you know exactly which half to search based on whether the target is above or below the midpoint. In a rotated array, you cannot make that determination directly from the midpoint alone. The key insight is that when you split a rotated array at any midpoint, at least one half is always normally sorted. You identify which half is sorted by comparing nums[lo] with nums[mid]. Once you know which half is sorted, you can check in O(1) whether the target falls in that range. If it does, search there; otherwise search the other half. Every step still eliminates half the array, so the total complexity is O(log n).

Q11: Explain binary search on the answer space with a concrete example. [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

The pattern applies when you need the minimum or maximum value satisfying a condition, and that condition is monotonic. Take the ship packages problem: find the minimum capacity to ship all packages within D days. The minimum possible capacity is the heaviest single package (you cannot ship it otherwise). The maximum is the total weight (ship everything in one day). For any capacity in between, you can greedily simulate whether it is feasible. Crucially, if capacity k works, any capacity above k also works. So the feasibility function is monotonic — false for small capacities, true above some threshold. Binary search finds that threshold. I set lo and hi to the bounds, and at each midpoint I run the simulation. If feasible, try smaller by setting hi to mid. If not feasible, try larger by setting lo to mid plus one. When lo meets hi, I have the minimum feasible capacity.

Q12: When would you use counting sort or radix sort instead of comparison-based sorting? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Counting sort applies when all values are non-negative integers within a known bounded range k. It counts the frequency of each value and reconstructs the sorted array in O(n + k) time. If you are sorting exam scores from 0 to 100, k is 101 and the sort is effectively O(n). If k is large relative to n, the space and time cost of the count array makes it worse than comparison sort. Radix sort extends this to larger integers by sorting digit by digit from least significant to most significant, using counting sort at each digit level. Both algorithms break the O(n log n) lower bound for comparison-based sorting because they use the structure of the values, not just comparisons. The practical cases where I reach for counting sort in a backend system: sorting elements with a small number of possible values like status codes or priority levels, or building a frequency histogram when the range is bounded.

Q13: How do you find a peak element using binary search? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

A peak element is one strictly greater than its neighbours. The naive approach checks every element in O(n). Binary search works in O(log n) because of a key property: if nums[mid] is less than nums[mid + 1], then a peak must exist to the right of mid. Either nums[mid + 1] is itself a peak, or it rises further and eventually a peak exists before the array ends. Similarly, if nums[mid] is greater than nums[mid + 1], a peak must exist at or to the left of mid. This monotonicity — the guarantee that moving toward the higher neighbour always leads to a peak — is what makes binary search applicable even though the array is not sorted. The loop uses lo < hi with hi = mid (not mid - 1) when nums[mid] > nums[mid + 1], because mid itself could be the peak.

Q14: How do you merge overlapping intervals and what is the key step? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Sort by start time first. After sorting, intervals that overlap are always adjacent in the list, which means you can resolve all overlaps in a single left-to-right pass. For each interval, compare its start with the end of the last interval in the result list. If the start is after the previous end, there is no overlap — add it as a new interval. If the start is at or before the previous end, they overlap — update the end of the last result interval to the maximum of the two ends. The sort is O(n log n) and the merge pass is O(n), giving O(n log n) total. The follow-up worth knowing is insert interval — you do not need to re-sort because the existing list is already sorted. You scan through three phases: intervals entirely before the new one, intervals that overlap with it, and intervals entirely after it.

Q15: What is the time complexity of building a heap and why is it O(n) not O(n log n)? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

Adding n elements to a heap one at a time is O(n log n) — each insert does one sift-up operation of O(log n). But building a heap from an existing array using heapify is O(n). The reason is that most of the nodes are near the bottom of the heap and have very little sifting to do. Half the nodes are leaves — they require zero sifting. A quarter are at height 1 and require at most one sift. An eighth are at height 2 and require at most two sifts. Summing up all the work across all levels gives a geometric series that converges to O(n). In Java, PriorityQueue with a collection constructor uses this O(n) heapify, which is why building a heap of n elements is O(n) but adding elements one at a time is O(n log n).

Q16: What is an inversion and how do you count inversions efficiently? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

An inversion is a pair of indices (i, j) where i is less than j but nums[i] is greater than nums[j] — an out-of-order pair. Counting inversions tells you how far an array is from being sorted: zero inversions means fully sorted, the maximum n*(n-1)/2 inversions means reverse sorted. The naive O(n^2) approach checks every pair. The efficient O(n log n) approach piggybacks on MergeSort. During the merge step, when a right-half element is placed before remaining left-half elements, each of those remaining left elements forms an inversion with the right element. You add the count of remaining left elements to a global counter. The total merge work is O(n log n) and the inversion counting adds zero extra complexity.

Q17: How do you find the median of a running data stream? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

Maintain two heaps: a max-heap for the smaller half of numbers seen so far, and a min-heap for the larger half. Every new number goes into one of the two heaps, then you rebalance so the sizes differ by at most one. The median is either the top of the larger heap, or the average of both tops if the heaps are equal size. Adding a number is O(log n) and reading the median is O(1). The invariant to maintain: every element in the max-heap (smaller half) must be less than or equal to every element in the min-heap (larger half). If adding a number violates this, you move the top of one heap to the other. This is a classic design problem that tests whether you can think about data structures beyond their basic use case.
