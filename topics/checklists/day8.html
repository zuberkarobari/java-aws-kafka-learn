S3 — The Storage Service You Must Know Deeply

S3 changed its consistency model in December 2020 — this comes up in interviews:

Before December 2020: S3 was eventually consistent for overwrite PUTs and DELETEs — you could get stale data immediately after writing. Since December 2020: S3 provides strong read-after-write consistency for all operations — PUT then GET always returns the latest version. This applies to all regions, all object types, free of charge. Interviewers from companies that built on the old model still ask about this. Your answer: 'S3 now provides strong read-after-write consistency since 2020 — I designed my Deloitte pipeline assuming this and it simplified a lot of the retry logic we previously needed.'

// S3 Storage Classes — cost vs retrieval speed vs availability
// S3 Standard:         frequently accessed, millisecond retrieval, highest cost
//                      Use: active application data, frequently read files
// S3 Standard-IA:      infrequently accessed, millisecond retrieval, lower storage cost
//                      minimum 30-day storage, retrieval fee per GB
//                      Use: disaster recovery backups, compliance archives accessed occasionally
// S3 One Zone-IA:      same as IA but stored in ONE AZ only — cheaper, less durable
//                      Use: re-creatable data (thumbnails, transcoded video)
// S3 Glacier Instant:  archive, millisecond retrieval, much lower cost
//                      Use: medical images, news archives, genomic data
// S3 Glacier Flexible: archive, 1-12 hour retrieval (Expedited: 1-5 min, extra cost)
//                      Use: long-term compliance archives, accessed rarely
// S3 Glacier Deep Archive: cheapest, 12-48 hour retrieval
//                      Use: 7-10 year regulatory retention requirements
// Presigned URL — grant temporary access without sharing credentials
// Use case: allow client to upload directly to S3 without going through your server
// Your server generates the URL, client uses it — server never handles the file bytes
// Spring Boot S3 presigned URL generation
@Service
public class S3StorageService {
    private final S3Presigner presigner;
    private final String bucketName;
    // Generate presigned URL for client to upload directly to S3
    public String generateUploadUrl(String key, String contentType, Duration expiry) {
        PutObjectRequest putRequest = PutObjectRequest.builder()
            .bucket(bucketName)
            .key(key)
            .contentType(contentType)
            .build();
        PresignedPutObjectRequest presigned = presigner.presignPutObject(req ->
            req.signatureDuration(expiry).putObjectRequest(putRequest));
        return presigned.url().toString();
        // Client PUTs file directly to this URL — no credentials needed
        // URL expires after 'expiry' duration
        // Benefit: offload bandwidth from your server, reduce latency
    }
    // Generate presigned URL for client to download
    public String generateDownloadUrl(String key, Duration expiry) {
        GetObjectRequest getRequest = GetObjectRequest.builder()
            .bucket(bucketName).key(key).build();
        PresignedGetObjectRequest presigned = presigner.presignGetObject(req ->
            req.signatureDuration(expiry).getObjectRequest(getRequest));
        return presigned.url().toString();
    }
    // S3 event notification trigger — in application.yml (SAM/CDK/Terraform manages this)
    // When new object created in bucket: trigger Lambda / send to SQS
    // Your Deloitte pipeline: CloudWatch log export to S3 → S3 event → Lambda → process
}

Interview Questions — Topic 1

Q2: What is an S3 presigned URL and when would you use it in a backend service? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

A presigned URL grants temporary access to a specific S3 object without requiring the requester to have AWS credentials. The server signs the URL using its IAM credentials (with an expiry time), hands it to the client, and the client uses it directly. Two main use cases: 1) Secure download — serve private S3 files to authenticated users without proxying through your server (saves bandwidth and latency). 2) Direct upload — let clients upload large files directly to S3 without the bytes passing through your application server. In Deloitte: used for CloudWatch log exports — Lambda generates presigned download URL and returns it to the requesting service.

Q3: What are S3 storage classes and when would you choose Glacier over Standard-IA? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Standard: frequent access, millisecond latency, highest cost. Standard-IA: infrequent access, same millisecond latency, lower storage cost but retrieval fee — best for data accessed a few times per month. Glacier Instant: archive with millisecond retrieval — lower cost than IA for data accessed once per quarter. Glacier Flexible: 1-12 hour retrieval (Expedited 1-5 min for extra cost) — for archives accessed rarely. Deep Archive: cheapest, 12-48 hour retrieval — for 7-10 year regulatory retention. Choose Glacier over Standard-IA when: data is accessed less than once per month, retrieval delay of minutes/hours is acceptable, and cost reduction justifies the retrieval latency.

Q4: What is S3 versioning and how does deleting an object work in a versioned bucket? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Versioning keeps all versions of an object. Every PUT creates a new version with a unique version ID. 'Deleting' a versioned object without specifying version ID inserts a delete marker — a special version that makes the object appear deleted but all previous versions are preserved. To truly delete: specify the exact version ID. To restore: delete the delete marker. Use versioning for: accidental delete protection, audit history, rolling back bad file uploads. Combine with lifecycle policies to auto-expire old versions and control storage costs.

IAM Roles vs Users — The Most Misunderstood IAM Concept

Never give a long-term access key to an EC2 instance, Lambda, or ECS task — use roles:

IAM Users have long-term credentials (access key ID + secret access key). If stored in code or environment variables: credentials can be stolen, rotated manually, and have no expiry. IAM Roles have NO long-term credentials — they use STS to issue temporary credentials (max 12 hours) that auto-rotate. EC2 instance profile: attaches an IAM role to an EC2 instance. The AWS SDK running on the instance automatically calls the EC2 metadata service (169.254.169.254) to get temporary credentials and refresh them before expiry — zero credential management code needed. Lambda execution role: same pattern. This is what you use in Deloitte. Long-term access keys in code is a critical security vulnerability.

Interview Questions — Topic 2

Q5: What is an IAM Instance Profile and why should you use it instead of access keys? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

Instance Profile: a container that attaches an IAM Role to an EC2 instance. The AWS SDK running on the instance automatically calls the EC2 metadata service (http://169.254.169.254/latest/meta-data/iam/security-credentials/) to retrieve temporary credentials (AWS_ACCESS_KEY_ID, AWS_SECRET_ACCESS_KEY, AWS_SESSION_TOKEN) that auto-rotate every few hours. Zero credential management code needed — the SDK credential provider chain handles it. Access keys in environment variables or config files: can be leaked via logs, code commits, or container image inspection. Stolen access keys have no automatic expiry. Instance profiles are the mandatory pattern for any AWS-hosted service.

Q6: What is the principle of least privilege and how do you implement it in IAM? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Grant only the permissions needed to perform the specific task — nothing more. Implementation: start with zero permissions, add only what the code actually calls. Use specific Actions (s3:GetObject, not s3:*). Scope Resources to specific ARNs (specific bucket/prefix, not *). Use Conditions (aws:RequestedRegion, aws:SourceIp). Review with IAM Access Analyzer — it identifies unused permissions. Use Permission Boundaries to cap maximum permissions a role can have (useful when granting teams ability to create roles). In production: regularly audit unused permissions with IAM Access Advisor (shows last used date per service).

Q7: What is the difference between AWS Secrets Manager and SSM Parameter Store? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

SSM Parameter Store: key-value store for config and secrets. Free tier for standard parameters. SecureString type encrypts with KMS. Good for non-secret config (DB URLs, feature flags) and lower-sensitivity secrets. Secrets Manager: purpose-built for secrets. Automatic rotation (Lambda-based, built-in for RDS/Redshift). Cross-account access. Versioning with staging labels (AWSCURRENT, AWSPENDING). More expensive per secret. Use Secrets Manager for: DB passwords (auto-rotate), API keys, JWT signing keys — anything that rotates. Use SSM Parameter Store for: app configuration, non-rotating credentials, feature flags. In Spring Boot: use AWS SDK to fetch at startup or use Spring Cloud AWS auto-injection.

Auto Scaling Group — How Your Deloitte Infrastructure Handled Load

ASG is not just 'add more servers when busy' — it is a state machine with health management:

Desired capacity: the number of instances ASG is trying to maintain. Min/Max: bounds for scaling. When an instance fails its health check: ASG terminates it and launches a new one — self-healing. Scaling OUT (increase capacity): when metric exceeds threshold (e.g., CPU > 70%), ASG launches new instances from the Launch Template, waits for the Instance Warmup Period (default 300s), then adds them to the Target Group. Scaling IN (decrease capacity): when metric drops below threshold, ASG terminates instances via connection draining — waits for in-flight requests to complete before terminating. In your JP Morgan project: the multi-region ASG would have used target tracking on CPU/request count to handle transaction volume spikes.

Interview Questions — Topic 3

Q8: What is a Launch Template and how does it differ from a Launch Configuration? [Easy] ▼
💡 Ideal Answer (Amazon/GS Standard)

Launch Template (newer, recommended): versioned, supports all modern EC2 features (multiple instance types, Spot + On-Demand mix, capacity reservations, T4g ARM instances). Can be updated — ASG uses a specific version. Launch Configuration (legacy): immutable — to change any setting you create a new one and update the ASG. Does not support all modern features. Always use Launch Templates. Key fields: AMI ID, instance type, user data script (startup commands), IAM instance profile, security groups, EBS volumes, key pair.

Q9: What are the three types of Auto Scaling policies and when do you use each? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Target Tracking: specify a target metric value (e.g., CPU = 70%), ASG automatically adjusts capacity to maintain it. Simplest and most common — handles both scale-out and scale-in. Use for most production workloads. Step Scaling: define step adjustments — if CPU 70-80% add 1 instance, if CPU 80-90% add 2, if CPU > 90% add 4. More control, more complex. Use when scale-out speed matters differently at different thresholds. Scheduled Scaling: change capacity on a schedule (e.g., double instances every morning at 8 AM for known traffic spike). Use for predictable load patterns (business hours, batch jobs). Combine: scheduled for predictable baseline, target tracking for dynamic adjustment.

// Spring Boot custom metrics to CloudWatch via Micrometer
@Service
public class PaymentMetricsService {
    private final MeterRegistry meterRegistry;
    private final Counter paymentSuccessCounter;
    private final Counter paymentFailureCounter;
    private final Timer paymentProcessingTimer;
    public PaymentMetricsService(MeterRegistry meterRegistry) {
        this.meterRegistry = meterRegistry;
        this.paymentSuccessCounter = Counter.builder("payment.processed")
            .tag("status", "success")
            .description("Total successful payments processed")
            .register(meterRegistry);
        this.paymentFailureCounter = Counter.builder("payment.processed")
            .tag("status", "failure")
            .register(meterRegistry);
        this.paymentProcessingTimer = Timer.builder("payment.processing.duration")
            .description("Time to process a payment end-to-end")
            .percentiles(0.5, 0.95, 0.99)  // publish p50, p95, p99 latency
            .register(meterRegistry);
    }
    public PaymentResult processWithMetrics(PaymentRequest req) {
        return paymentProcessingTimer.record(() -> {
            try {
                PaymentResult result = paymentEngine.process(req);
                paymentSuccessCounter.increment();
                return result;
            } catch (Exception e) {
                paymentFailureCounter.increment();
                // Tag by error type for granular monitoring
                meterRegistry.counter("payment.errors",
                    "type", e.getClass().getSimpleName()).increment();
                throw e;
            }
        });
    }
}

Q10: How do you publish custom application metrics to CloudWatch from Spring Boot? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Add micrometer-registry-cloudwatch2 dependency. Configure in application.yml: management.metrics.export.cloudwatch.namespace and step (publish interval). Q11: What is CloudWatch Logs Insights and how is it different from basic log filtering? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

CloudWatch Logs Insights: purpose-built query language for log data — query across multiple log groups simultaneously, aggregate, filter, parse structured logs (JSON auto-parsed), visualize results on dashboards. Key operators: filter (WHERE clause), stats (GROUP BY + aggregations: avg, sum, count, percentile), fields (SELECT), parse (regex field extraction), sort, limit. Unlike basic log filter (simple text search): Logs Insights runs analytical queries across gigabytes of logs in seconds, returns structured results, and can be saved to dashboards. In production: used for real-time incident investigation, SLA compliance reporting, error trend analysis.

Interview Questions — Topic 5

Q12: What is the difference between reserved concurrency and provisioned concurrency in Lambda? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

Reserved concurrency: caps maximum concurrent executions for a function — prevents it from consuming all regional concurrency and starving other functions. Does NOT keep instances warm. If limit is 10 and 11th concurrent request arrives: throttled (429). Provisioned concurrency: pre-initializes N execution environments — they are always warm and ready. Eliminates cold starts for those N instances. Costs money even with no invocations. Use reserved for: protecting other functions from a noisy neighbor. Use provisioned for: latency-sensitive APIs where cold starts are unacceptable. Combine: reserve 100 total, provision 5 always-warm instances.

Terraform State — The Foundation of Everything

State is the most important concept in Terraform — and the most common source of problems in teams:

terraform.tfstate is Terraform's understanding of the real-world infrastructure it manages. On every plan: Terraform compares desired config (your .tf files) with current state AND actual AWS resources — produces a diff. On apply: executes the diff, updates state. The problem: if two engineers run apply simultaneously, they corrupt each other's state. State also contains sensitive values (RDS passwords, certificate keys) — never commit it to git. Production solution: S3 remote backend (state stored in S3) + DynamoDB lock table (prevents concurrent applies). Your Deloitte enterprise registry used this pattern — when you ran terraform apply, it acquired a DynamoDB lock, preventing your colleague from running apply at the same time.

Q15: What is the difference between count and for_each in Terraform? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

count: integer — creates N identical resources, referenced by index (resource, resource). Problem: if you delete index 1 of 3, Terraform reindexes — resource becomes resource and is destroyed/recreated. for_each: map or set — creates one resource per key, referenced by key (resource['dev'], resource['prod']). Removing one key only destroys that specific resource, others unchanged. Always prefer for_each over count when creating multiple related resources — avoids the destructive reindexing problem. Use count only for truly identical resources where order doesn't matter (e.g., 3 identical replica instances).

Security Groups vs NACLs — Know This Cold

This is asked in every AWS interview and most candidates get the stateful vs stateless distinction wrong:

Security Groups are stateful — if you allow inbound traffic from port 443, the return traffic is AUTOMATICALLY allowed even without an outbound rule. Security Groups are at the instance/ENI level. You can reference other security groups as source/destination (e.g., 'allow traffic from the ALB security group') — this is more maintainable than CIDR-based rules. NACLs are stateless — you must explicitly allow BOTH inbound AND outbound traffic. NACLs are at the subnet level. Rules evaluated in numeric order — first match wins. NACLs are good for: subnet-level IP blocking (block a specific CIDR range), adding a second layer of defense. In practice: Security Groups handle 95% of traffic control. NACLs for subnet-level guardrails.

Interview Questions — Topic 7

Q16: What is the difference between a Security Group and a NACL? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Security Group: stateful — return traffic is automatically allowed. Instance/ENI level. Allow rules only (no explicit deny). Can reference other security groups as source. Applied last (after NACL). NACL: stateless — must explicitly allow both inbound AND outbound. Subnet level — applies to all instances in the subnet. Allow AND deny rules. Rules evaluated in order (lower number first). Default NACL allows all traffic. Custom NACL denies all until rules added. Use Security Groups for: most traffic control (best practice). Use NACLs for: subnet-level IP blocking, adding defense-in-depth, blocking specific CIDRs that Security Groups cannot deny.

Q17: What is a NAT Gateway and why does a private subnet need one? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Private subnet instances have no public IP and no route to the internet. NAT Gateway (Network Address Translation): sits in a public subnet, provides outbound internet access for private subnet instances while blocking all inbound connections from the internet. Private instances route outbound traffic to NAT Gateway, which translates their private IP to its public Elastic IP for the request, and returns the response to the private instance. Use case: EC2 in private subnet needs to pull updates from apt/yum, call external APIs, access S3 (or use VPC Endpoint for S3 — no NAT needed), push to ECR. Cost: NAT Gateway charges per GB processed — use VPC Endpoints for AWS services (S3, DynamoDB, Secrets Manager) to avoid NAT costs.
