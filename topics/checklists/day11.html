Interview Questions — Topic 1

Q1: What is the internal structure of ArrayList and how does it grow? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

ArrayList backs elements in an Object[] array. Default initial capacity: 10 (lazy in Java 8+). On add() exceeding capacity: newCapacity = oldCapacity + (oldCapacity >> 1) — ~50% growth. Growth calls Arrays.copyOf() (System.arraycopy JVM intrinsic). add(E) at end: O(1) amortised. add(int, E): O(n) — shifts elements. get(int): O(1) direct index. remove(int): O(n) — shifts elements. Iteration is cache-friendly — contiguous memory.

Q2: What is the difference between Arrays.asList() and List.of()? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Arrays.asList(): fixed-size List backed by original array. set() works AND modifies original array. add/remove throw UnsupportedOperationException. Allows null. List.of() (Java 9+): truly immutable — ALL mutations throw UOE. Null throws NPE. More compact implementation. Use Arrays.asList() when you still need element replacement but not structural changes. Use List.of() for true constants.

Q3: When would you choose LinkedList over ArrayList? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

LinkedList wins for: frequent insertions/deletions at the HEAD (O(1) vs O(n) shift). Queue/Deque operations with frequent head/tail modification. Mid-list insertion via ListIterator.add() (O(1) vs O(n)). ArrayList wins for everything else — random access O(1) vs O(n), cache locality (contiguous memory vs scattered Nodes). In practice: ArrayDeque outperforms LinkedList even for queue operations. Real-world LinkedList use is rare in modern Java.

🟢 Topic 2: Map — HashMap Full Internals, LinkedHashMap, TreeMap, EnumMap

Checklist

  • HashMap: array of Node[], each bucket is linked list → tree at 8 nodes (TREEIFY_THRESHOLD)
  • Full hash chain: hashCode() → (h ^ h>>>16) → (n-1) & hash → bucket index
  • Load factor 0.75: resize triggers at capacity × 0.75, doubles capacity, rehashes all
  • Java 8 treeification: threshold 8 → Red-Black tree, untreeify at 6
  • Map.compute(), merge(), computeIfAbsent() — atomic put-if-absent operations
  • LinkedHashMap: insertion-order vs access-order LRU cache with removeEldestEntry
  • TreeMap: Red-Black tree, all 8 NavigableMap methods, null key restriction

HashMap Internal Hash Chain — The Full Picture:

Step 1: h = key.hashCode(). Step 2: hash = h ^ (h >>> 16) — spreads high bits into low bits. Without this, poor hashCodes differing only in high bits would all hit the same bucket. Step 3: index = (n-1) & hash — n is always power of 2, bitwise AND is hash % n but faster. Step 4: walk bucket's linked list (or TreeNode), comparing hash first then equals(). Treeification at 8 nodes: O(n) → O(log n) lookup — also defends against hash collision DoS attacks.

// HashMap pre-sizing: avoid resize for N entries
// threshold = capacity * 0.75, so capacity needed = N / 0.75 + 1
Map<String, BigDecimal> balances = new HashMap<>(134); // holds 100 without resize
// merge: atomic upsert with combining function
totals.merge(accountId, txAmount, BigDecimal::add);
// absent → puts txAmount. present → replaces with add(existing, txAmount)
// null return → removes key
// computeIfAbsent: insert lazily only if absent
grouped.computeIfAbsent("ACC-001", k -> new ArrayList<>()).add(tx);
// LinkedHashMap LRU cache (accessOrder=true)
Map<String, Session> lruCache = new LinkedHashMap<String, Session>(100, 0.75f, true) {
    protected boolean removeEldestEntry(Map.Entry<String,Session> e) {
        return size() > 100;  // evict LRU when over capacity
    }
};
// TreeMap: sorted, NavigableMap operations
TreeMap<LocalDateTime, Transaction> byTime = new TreeMap<>();
// floorKey(t): largest key <= t
// ceilingKey(t): smallest key >= t
// headMap(t): all keys < t
// tailMap(t): all keys >= t
// subMap(lo, hi): [lo, hi)
// firstKey() / lastKey(): min / max

Interview Questions — Topic 2

Q4: Explain the full hash chain in HashMap from key.hashCode() to bucket index. [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

Step 1: h = key.hashCode(). Step 2: hash = h ^ (h >>> 16) — XOR spreads high 16 bits into low 16 so bucket index (which only uses low bits) is influenced by the full hash. Step 3: index = (n-1) & hash where n is always power of 2 — equivalent to hash % n but using fast bitwise AND. Step 4: examine bucket[index] — if empty insert; if occupied walk linked list comparing hash (cheap) then equals() (correct match). Java 8+: if bucket >= 8 nodes, convert to Red-Black tree for O(log n) lookup.

Q5: What is treeification in HashMap and why was it introduced? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

When a bucket accumulates 8+ nodes, HashMap converts it from a linked list to a Red-Black tree (TreeNode). Lookup: O(n) list → O(log n) tree. Untreeify at 6 nodes (hysteresis). MIN_TREEIFY_CAPACITY = 64: below 64 total buckets, resize instead (cheaper). Before Java 8: hash collision attacks could craft keys all landing in same bucket — O(n) per operation could DoS a web server. Treeification caps worst-case at O(log n) regardless of collision count.

Q6: What does Map.merge() do and when would you use it? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

merge(key, value, remappingFn): if key absent or null → put value. If present → replace with remappingFn(existing, value). If fn returns null → remove key. Classic uses: totals.merge(k, amount, BigDecimal::add) for aggregation; counts.merge(word, 1, Integer::sum) for counting. Replaces the verbose getOrDefault + put pattern with a single atomic-intent call. NOT atomic in HashMap — use ConcurrentHashMap for thread-safe merge.

Interview Questions — Topic 3

Q7: How is HashSet implemented and what are its performance characteristics? [Easy] ▼
💡 Ideal Answer (Amazon/GS Standard)

HashSet is backed by a HashMap where all values are a shared PRESENT dummy object. add(e) calls map.put(e, PRESENT). contains(e) calls map.containsKey(e). remove(e) calls map.remove(e). All performance: O(1) amortised — directly from HashMap. One null allowed. No guaranteed order. Not thread-safe. All HashMap characteristics (load factor, treeification, growth) apply. This design means improvements to HashMap automatically improve HashSet.

Q8: What is EnumSet and why is it more efficient than HashSet for enum values? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

EnumSet uses a bit vector — for enums with <= 64 constants, a single long (RegularEnumSet). Each constant's ordinal() is a bit position. add/contains/remove are single bitwise operations — O(1), no allocation, no hashing, no collision. 5-10x faster and dramatically less memory than HashSet. Iteration in enum declaration order. Always use EnumSet when element type is an enum — it is always the right choice.

🟣 Topic 4: Queue & Deque — ArrayDeque, PriorityQueue, BlockingQueue

Checklist

  • Queue API: offer/poll/peek (special value) vs add/remove/element (exception)
  • ArrayDeque: circular array, faster than LinkedList AND Stack for all ops
  • Why not java.util.Stack: extends Vector, synchronized on every operation
  • PriorityQueue: min-heap array, O(log n) add/poll, O(1) peek, no null
  • Top-K pattern: min-heap of size K — O(n log k) vs O(n log n) full sort
  • BlockingQueue: put()/take() blocking, LinkedBlockingQueue two-lock design
// ArrayDeque: circular array — faster than LinkedList for all ops
Deque<String> stack = new ArrayDeque<>();
stack.push("first");   // addFirst
stack.push("second");
stack.pop();           // "second" — LIFO. No sync overhead unlike Stack.
// PriorityQueue: min-heap. peek()=O(1), add/poll=O(log n)
// Top-K largest amounts using a MIN-heap of size K
int K = 5;
PriorityQueue<BigDecimal> topK = new PriorityQueue<>(K);
for (Transaction tx : transactions) {
    topK.offer(tx.getAmount());
    if (topK.size() > K) topK.poll(); // evict smallest — keeps K largest
}
// Why min-heap? The minimum IN the heap is the one to evict next
// O(n log K) — far better than O(n log n) sort for small K, huge n
// Max-heap (largest first):
PriorityQueue<BigDecimal> maxHeap =
    new PriorityQueue<>(Comparator.reverseOrder());
// BlockingQueue producer-consumer
BlockingQueue<PaymentJob> queue = new ArrayBlockingQueue<>(1000);
// Producer: queue.put(job)  — blocks if full (back-pressure)
// Consumer: queue.take()    — blocks if empty
// LinkedBlockingQueue: two locks (head + tail) → higher concurrent throughput
// SynchronousQueue: capacity 0 — direct hand-off, used by newCachedThreadPool

Interview Questions — Topic 4

Q9: What is PriorityQueue and how does it implement a min-heap? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

PriorityQueue uses a binary heap in an Object[] array. Heap property: parent at index i <= children at 2i+1 and 2i+2 (min-heap). peek(): O(1) — return heap. add(): O(log n) — append, sift up. poll(): O(log n) — remove root, move last element to root, sift down. Iteration does NOT give sorted order — only poll() sequence is sorted. No null allowed. Not thread-safe. For concurrent use: PriorityBlockingQueue.

Q10: What is the difference between LinkedBlockingQueue and ArrayBlockingQueue? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

ArrayBlockingQueue: fixed capacity array, single ReentrantLock shared by producers and consumers, fair mode available. LinkedBlockingQueue: linked nodes, separate locks for head (consumers) and tail (producers) — producers and consumers can proceed simultaneously → higher throughput. Can be unbounded (risk: OOM if consumers slow). Use ArrayBlockingQueue for bounded back-pressure. Use LinkedBlockingQueue for higher-throughput pipelines.

🔴 Topic 5: Iterators — fail-fast, fail-safe, CME, Safe Patterns

Checklist

  • fail-fast iterator: modCount check → ConcurrentModificationException on mismatch
  • What counts as structural modification vs non-structural (set/get is NOT structural)
  • 3 safe remove-during-iteration patterns: Iterator.remove(), removeIf(), stream filter
  • fail-safe iterators: CopyOnWriteArrayList snapshot, ConcurrentHashMap weak-consistency
  • ListIterator: bidirectional, add() and set() safe during iteration
  • Spliterator: parallel-friendly, trySplit(), characteristics flags
// modCount mechanism: structural mods increment modCount
// Iterator stores expectedModCount at creation
// next() checks: modCount != expectedModCount → CME
// BAD: modify collection during for-each
for (Transaction tx : transactions) {
    if (tx.isFailed()) transactions.remove(tx); // CME
}
// PATTERN 1: Iterator.remove() — updates modCount + expectedModCount
Iterator<Transaction> it = transactions.iterator();
while (it.hasNext()) {
    if (it.next().isFailed()) it.remove(); // safe
}
// PATTERN 2: removeIf() — single-pass, efficient, cleanest
transactions.removeIf(Transaction::isFailed);
// PATTERN 3: stream filter — new list (non-destructive)
List<Transaction> ok = transactions.stream()
    .filter(tx -> !tx.isFailed()).collect(Collectors.toList());
// fail-safe: ConcurrentHashMap — weakly consistent, never CME
// CopyOnWriteArrayList — snapshot at iterator creation, never CME

Interview Questions — Topic 5

Q11: What causes ConcurrentModificationException and what are two safe ways to remove during iteration? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

CME: fail-fast iterator throws when collection's modCount (incremented on any structural modification) does not match iterator's expectedModCount stored at creation. Even single-threaded modification during iteration triggers it. Safe options: 1) Iterator.remove() — updates both counters atomically. 2) Collection.removeIf(predicate) — single-pass, internally safe. 3) Build new filtered list via stream. 4) Use CopyOnWriteArrayList (snapshot iterator) or ConcurrentHashMap (weakly consistent).

Q12: What is the difference between a fail-fast and a fail-safe iterator? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Fail-fast: throws CME if collection structurally modified after iterator creation — detected via modCount. Best-effort (not guaranteed in all race conditions). Used by ArrayList, HashMap, HashSet. Fail-safe: never throws CME. CopyOnWriteArrayList: iterator works on snapshot — never sees concurrent modifications, may see stale data. ConcurrentHashMap: weakly consistent — may or may not reflect concurrent modifications. Use fail-safe when iteration must complete without exception even with concurrent writes.

Hold this thought:

ConcurrentHashMap's Java 8 design uses CAS for empty buckets (no lock at all) and synchronized on the first node of an occupied bucket (only locks one bucket at a time). Reads use volatile reads — zero locking. This allows fully concurrent reads and parallel writes to different buckets simultaneously. The subtle trap: compound operations (check-then-act) are still not atomic unless you use compute/merge/computeIfAbsent.

Topic 6: Concurrent Collections — ConcurrentHashMap, COWAL, Atomic Ops

Checklist

  • ConcurrentHashMap Java 8: CAS for empty bucket, synchronized on first node for occupied
  • Reads: volatile reads — zero locking, fully concurrent
  • size() approximate — LongAdder array of cells, summed on size()
  • Compound operation trap: get-then-put race condition
  • Atomic methods: putIfAbsent, computeIfAbsent, merge, compute — truly atomic
  • Collections.synchronizedMap: coarse single lock, iteration needs external sync
// ConcurrentHashMap Java 8: fine-grained, no global lock
// READ: volatile reads of Node.val — no lock at all
// WRITE empty bucket: CAS — atomic CPU instruction, no lock
// WRITE occupied bucket: synchronized on first node of that bucket only
// → reads never block writes, writes to different buckets never block each other
// THE COMPOUND OPERATION TRAP
// BAD — two separate operations, race condition between them:
if (!balances.containsKey(id)) {          // Thread A: absent
    balances.put(id, BigDecimal.ZERO);     // Thread B also sees absent
}                                          // Both put ZERO — one silently lost
// GOOD — single atomic operation:
balances.putIfAbsent(id, BigDecimal.ZERO);
balances.computeIfAbsent(id, k -> loadFromDB(k));
// computeIfAbsent: function runs at most ONCE per key — guaranteed
// Atomic counter pattern
ConcurrentHashMap<String, LongAdder> counts = new ConcurrentHashMap<>();
counts.computeIfAbsent(category, k -> new LongAdder()).increment();
// merge: atomic aggregation
totals.merge(category, 1L, Long::sum); // atomic in ConcurrentHashMap
// synchronizedMap: coarse lock, iteration STILL needs external sync
Map<String, V> syncMap = Collections.synchronizedMap(new HashMap<>());
synchronized (syncMap) {  // must sync on map for iteration
    for (Map.Entry<String,V> e : syncMap.entrySet()) { ... }
}
// → Always prefer ConcurrentHashMap in new code

Interview Questions — Topic 6

Q13: How does ConcurrentHashMap allow concurrent reads and writes without a global lock? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

Java 8 ConcurrentHashMap: empty bucket writes use CAS (compare-and-swap CPU instruction — atomic, no lock). Occupied bucket writes synchronize on that bucket's first Node only — other buckets unaffected. Reads use volatile reads of Node.val and Node.next — zero locking. Multiple threads can read any key simultaneously. Threads writing to different buckets run truly concurrently. Near-linear scalability with thread count for workloads hitting different keys.

Q14: What is the compound operation trap in ConcurrentHashMap and how do you avoid it? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

Two individually atomic operations are not together atomic — a thread can interleave between them. Example: containsKey(k) returns false, then another thread inserts k, then put(k,v) overwrites it. Fix: use CHM's atomic methods — putIfAbsent(k,v), computeIfAbsent(k, fn), merge(k, v, fn), compute(k, fn). All execute within the bucket's lock scope — the read-and-write happen as one indivisible operation. Rule: if your logic is 'read then conditionally write', always use a single CHM atomic method.

80-Card Collections Interview Q&A — Full Drill

Drill Format:

Q1 ArrayList default initial capacity and growth formula? [Easy]

▶ Default 10 (lazy-created on first add). Growth: newCap = old + (old>>1) → ×1.5 rounded down. 10→15→22→33→49→73→109...

Q2 What does System.arraycopy do in ArrayList and when is it called? [Medium]

▶ JVM intrinsic (native memcpy). Called: add(index,e) shifts elements right, remove(index) shifts left, grow() copies to new larger array.

Q3 Time complexity: ArrayList add(E), add(int,E), get(int), remove(int)? [Easy]

▶ add(E) end: O(1) amortised. add(int,E): O(n) shifts. get(int): O(1) direct index. remove(int): O(n) shifts. remove(Object): O(n) scan + shift.

Q4 LinkedList structure and time complexity? [Medium]

▶ Doubly-linked Node{prev,item,next}. addFirst/addLast: O(1). get(index): O(n) traversal. remove(index): O(n) to find + O(1) unlink. contains: O(n).

Q5 Why does ArrayList outperform LinkedList for iteration? [Medium]

▶ Cache locality: ArrayList elements are contiguous in memory — CPU prefetches next cache line. LinkedList: each Node is a separate heap object — each step is a potential cache miss.

Q6 Arrays.asList() vs List.of() key differences? [Easy]

▶ Arrays.asList: fixed-size, backed by original array, set() works, allows null. List.of: truly immutable (all mutations throw UOE), null throws NPE, more compact.

Q7 Collections.unmodifiableList() hidden danger? [Medium]

▶ The original mutable list is still reachable — changes to the original are visible through the unmodifiable view. For true immutability: List.copyOf() or List.of().

Q8 CopyOnWriteArrayList: when to use and when to avoid? [Medium]

▶ Use: read-heavy, rarely-written shared lists (event listeners, handler registries). Avoid: frequent writes — O(n) per write is expensive. Reads: lock-free, zero contention.

Q9 subList() danger and the fix? [Hard]

▶ Returns a live view referencing the parent. Parent structural mod → CME on subList. subList modifications affect parent. Fix: new ArrayList<>(list.subList(from, to)) — defensive copy.

Q10 RandomAccess marker interface — what does it signal? [Easy]

▶ Marker (no methods). Implemented by ArrayList. Algorithms (binarySearch, shuffle) check instanceof RandomAccess to choose O(1) random-access strategy vs sequential for non-RandomAccess lists.

Q11 List.copyOf() vs new ArrayList<>(source)? [Medium]

▶ List.copyOf(): unmodifiable, reuses backing if source already unmodifiable, rejects nulls. new ArrayList<>(source): fully mutable copy, allows nulls. Use copyOf() for defensive API returns.

Q12 What happens if you call sort() on a List during iteration? [Easy]

▶ ConcurrentModificationException — sort() increments modCount (structural modification). Never sort inside an active iteration loop.

Q13 How does ArrayList.removeIf() outperform iterator removal? [Hard]

▶ Single pass using a bitset to mark indices to remove, then one arraycopy to compact. O(n) total vs O(n) per remove with iterator (shifting after each removal). Far faster for bulk removal.

Q14 ArrayList vs int[] — when do you use a plain array? [Medium]

▶ int[]: no boxing, contiguous primitive storage, 4 bytes/element vs ~20 bytes for Integer. For millions of numeric elements (statistics, signal processing) use int[] or IntStream. Use ArrayList when: dynamic size, type safety, collection API needed.

Q15 Time complexity of Collections.sort()? [Easy]

▶ O(n log n) — TimSort (adaptive merge sort + insertion sort). Stable. Works on List via list.sort(). LinkedList: converted to array, sorted, written back — O(n log n) + O(n) extra memory.

Q16 How do you implement a stack in modern Java? [Easy]

▶ Deque implemented by ArrayDeque: push(e)=addFirst, pop()=removeFirst, peek()=peekFirst. Never use java.util.Stack — extends Vector, synchronized every operation even single-threaded.

Q17 What is list.clear() vs list = new ArrayList<>()? [Medium]

▶ clear(): O(n) nulls each ref, sets size=0, keeps backing array (capacity preserved). new ArrayList<>(): new object, capacity 10. clear() is better for reuse — avoids GC + realloc.

Q18 ArrayList pre-sizing: when does it matter most? [Medium]

▶ When final size is known upfront. Without: log₁.₅(n) resize+copy operations. For 1M elements: ~33 resizes. new ArrayList<>(n) → zero resizes and no wasted capacity.

Q19 How does Collections.binarySearch() work? [Medium]

▶ Requires sorted list (ascending natural or Comparator). Checks RandomAccess — binary search for ArrayList, sequential index scan for LinkedList. Returns index if found, -(insertionPoint+1) if not. Undefined result on unsorted list.

Q20 What is Collections.nCopies()? [Easy]

▶ Returns immutable List with n copies of obj. Zero allocation per call after creation. Use for: pre-filling, initializing fixed-size result templates. nCopies(10, BigDecimal.ZERO) for report templates.

Q21 HashMap default capacity, load factor, resize trigger? [Easy]

▶ Capacity 16 (power of 2). Load factor 0.75. Resize at 13th entry (16×0.75=12). New capacity = old×2, all entries rehashed. Pre-size: new HashMap<>(expectedSize/0.75+1).

Q22 Why must HashMap capacity always be a power of 2? [Hard]

▶ Bucket index = (n-1) & hash. Only works as hash%n when n is power of 2 — (n-1) becomes bitmask of all 1s in low bits. Bitwise AND is much faster than integer division. Non-power-of-2 constructor argument is rounded up.

Q23 What is the hash spreading step h ^ (h>>>16)? [Hard]

▶ XOR raw hashCode with its upper 16 bits shifted into lower 16. Purpose: poor hashCodes varying only in high bits (discarded by bucket index) would all collide without spreading. Mixes high and low bits — reduces collisions.

Q24 HashMap treeification threshold and why? [Hard]

▶ TREEIFY_THRESHOLD=8 (list→tree), UNTREEIFY_THRESHOLD=6 (tree→list). MIN_TREEIFY_CAPACITY=64 (resize preferred below 64 buckets). Converts O(n) bucket search to O(log n). Also defends against hash collision DoS attacks.

Q25 HashMap.getOrDefault() vs get() — which to prefer? [Easy]

▶ getOrDefault(k, def): returns def without map modification if absent — no null ambiguity. get(k): returns null if absent — ambiguous if null is a valid value. Always use containsKey(k) for true absence check when null is a valid value.

Q26 Map.computeIfAbsent(): when is the function NOT called? [Medium]

▶ If key already present with non-null value: function is NOT called, existing value returned. Function only called when key absent or mapped to null. In ConcurrentHashMap: function runs within bucket lock — guaranteed at most once per key.

Q27 Map.merge() — what happens when the function returns null? [Medium]

▶ Removes the key (tombstone). absent/null → put value. present → remappingFn(existing, value). Uses: totals.merge(k, amount, BigDecimal::add) for aggregation, counts.merge(word, 1, Integer::sum) for counting.

Q28 LinkedHashMap two ordering modes? [Medium]

▶ Insertion order (default): iteration reflects first-insertion order. Access order (constructor true): get() moves accessed entry to tail (MRU). Use access-order for LRU cache with removeEldestEntry override.

Q29 TreeMap structure, order, null key handling? [Medium]

▶ Red-Black BST. O(log n) all ops. Sorted by natural order or Comparator. No null keys (compareTo(null) throws NPE). Null values allowed. NavigableMap: floor/ceiling/lower/higher/headMap/tailMap/subMap.

Q30 TreeMap's 8 NavigableMap methods? [Hard]

▶ floorKey(k): largest<=k. ceilingKey(k): smallest>=k. lowerKey(k): strictlyk. headMap(k): keys=k. subMap(lo,hi): [lo,hi). firstKey()/lastKey(): min/max.

Q31 EnumMap vs HashMap? [Medium]

▶ EnumMap: dense array indexed by ordinal(). No hashing, no collision, direct array lookup. Iteration in enum declaration order. 5-10x faster, dramatically less memory. Always use EnumMap for enum-keyed maps.

Q32 Map.Entry.setValue() — when is it useful? [Medium]

▶ Modifies value in-place during entrySet() iteration — safe (not structural). More efficient than calling map.put() inside loop. Note: adding/removing entries during iteration still causes CME.

Q33 HashMap vs Hashtable? [Easy]

▶ Hashtable: legacy, synchronized every method, no null keys/values, not fail-fast. HashMap: not synchronized, one null key, null values allowed, fail-fast. Never use Hashtable — use ConcurrentHashMap for thread-safety.

Q34 What is IdentityHashMap? [Hard]

▶ Uses == (reference equality) instead of equals() for key comparison. hashCode uses System.identityHashCode(). Use: graph traversal visited-state, object serialization (avoid revisiting same object). Very rare — only when reference identity is semantically correct.

Q35 What is WeakHashMap and its use case? [Hard]

▶ Keys are WeakReferences — when key has no strong references elsewhere, entry is auto-removed by GC on next access/size call. Use: cache where entries expire when key is no longer used (prevents memory leaks). Not thread-safe.

Q36 Count frequency of elements in a list using a Map? [Easy]

▶ list.stream().collect(Collectors.groupingBy(Function.identity(), Collectors.counting())). Or: freq.merge(e, 1, Integer::sum) in a forEach loop. merge approach is cleanest — single atomic-intent call per element.

Q37 Map.of() and Map.copyOf() restrictions? [Medium]

▶ Both: immutable (put/remove throw UOE), no null keys/values (NPE), unspecified iteration order. Map.of() supports up to 10 pairs — use Map.ofEntries() for more. Map.copyOf(): snapshot, rejects nulls. For mutable copy: new HashMap<>(source).

Q38 map.put() vs map.putIfAbsent()? [Medium]

▶ put(k,v): always inserts/replaces, returns previous (null if absent). putIfAbsent(k,v): inserts ONLY if absent, returns existing value (null if inserted). Atomic in ConcurrentHashMap. Use computeIfAbsent when value is expensive to compute — function only called when insertion will occur.

Q39 What happens on get(null) in HashMap vs TreeMap? [Medium]

▶ HashMap: null key always maps to bucket 0 (special case), get(null) works. TreeMap: compareTo(null) throws NullPointerException — null key never allowed.

Q40 HashMap with two keys having same hashCode? [Medium]

▶ Hash collision: both land in same bucket. HashMap walks linked list (or tree), comparing hash then equals() for each node. Multiple objects with same hash but different equals() coexist as separate nodes. Performance: O(1)→O(n) per bucket — mitigated by treeification at 8.

Q41 HashSet implemented as what? Performance? [Easy]

▶ HashMap with shared PRESENT dummy. add=map.put, contains=containsKey, remove=map.remove. All O(1) amortised. One null allowed. No order. All HashMap performance characteristics apply directly.

Q42 TreeSet vs HashSet — when TreeSet? [Easy]

▶ TreeSet when: sorted iteration needed, range ops (subSet/headSet/tailSet), min/max access (first/last). O(log n) all ops, no null. HashSet: O(1), one null, no ordering — 3-10x faster. Choose HashSet for membership-only tests.

Q43 EnumSet two variants? [Hard]

▶ RegularEnumSet: <=64 enum constants — single long bitmask. JumboEnumSet: >64 constants — long[] array. EnumSet.of() picks automatically. All operations are single bitwise op — fastest possible Set.

Q44 LinkedHashSet vs HashSet? [Medium]

▶ LinkedHashSet = LinkedHashMap. Maintains insertion-order iteration. Same O(1) as HashSet. Slight memory overhead (linked list pointers). Use when deduplication AND insertion order preservation both required.

Q45 TreeSet equality — the hidden trap? [Hard]

▶ TreeSet uses compareTo/compare for equality — NOT equals(). If compareTo returns 0: element treated as duplicate, NOT inserted, even if equals() returns false. This can cause 'missing' elements when Comparable is inconsistent with equals.

Q46 Queue API throw-exception vs return-special-value? [Easy]

▶ Throws exception: add(full), remove(empty), element(empty). Returns null/false: offer, poll, peek. Prefer offer/poll/peek — no exception handling in normal flow.

Q47 ArrayDeque vs LinkedList — which is faster? [Medium]

▶ ArrayDeque always: resizable circular array, contiguous memory, no per-node heap allocation. LinkedList: each Node is separate heap object — allocation overhead, cache misses. ArrayDeque wins for stack, queue, and deque operations.

Q48 PriorityQueue peek() vs poll()? [Easy]

▶ peek(): O(1) — returns minimum without removal (heap). Returns null if empty. poll(): O(log n) — removes and returns minimum, restores heap property by sifting. Iteration does NOT yield sorted order.

Q49 How to make a max-heap PriorityQueue? [Medium]

▶ new PriorityQueue<>(Comparator.reverseOrder()) for natural-order types. Or: Comparator.comparing(Obj::getField).reversed(). poll() returns largest element.

Q50 BlockingQueue put/offer and take/poll differences? [Medium]

▶ put(e): blocks indefinitely until space (bounded). offer(e,timeout,unit): waits up to timeout, returns false. take(): blocks indefinitely until element. poll(timeout,unit): waits up to timeout, returns null. Use take/put for blocking producer-consumer; offer/poll for bounded-wait.

Q51 What is SynchronousQueue? [Hard]

▶ Capacity 0 — every put() blocks until a take() is ready (and vice versa). Direct hand-off channel between threads. Used by Executors.newCachedThreadPool(): hands off tasks to waiting threads without queuing. Ideal when producer should not proceed until consumer is ready.

Q52 ArrayBlockingQueue internal structure? [Hard]

▶ Fixed capacity circular array. Single ReentrantLock + two Conditions: notEmpty (consumers await) and notFull (producers await). Fair mode available. put(): lock, await notFull if full, insert, signal notEmpty. take(): lock, await notEmpty if empty, remove, signal notFull.

Q53 Deque addFirst/offerFirst vs addLast/offerLast? [Easy]

▶ addFirst/offerFirst: insert at head. addLast/offerLast: insert at tail. Stack: push=addFirst, pop=removeFirst. Queue: offer=addLast, poll=removeFirst.

Q54 What is DelayQueue? [Hard]

▶ Unbounded blocking queue for Delayed elements. Elements only become available after delay expires (getDelay()<=0). take() blocks until earliest element expires. Use: scheduled task execution, session timeout, retry with exponential backoff, cache expiry.

Q55 PriorityBlockingQueue vs PriorityQueue? [Medium]

▶ PriorityBlockingQueue: thread-safe with ReentrantLock, unbounded (take() blocks when empty only, put() never blocks). Use in concurrent producer-consumer needing priority ordering. PriorityQueue: single-threaded, faster.

Q56 What is modCount and how does it cause CME? [Medium]

▶ modCount: counter incremented on every structural modification (add/remove/clear). Iterator stores expectedModCount at creation. Every next() checks modCount==expectedModCount. Mismatch → throw CME. Best-effort detection.

Q57 3 safe ways to remove during List iteration? [Easy]

▶ 1) Iterator.remove() after next() — updates expectedModCount. 2) removeIf(predicate) — single pass, efficient. 3) stream.filter().collect() — new list. Also: collect indices then remove in reverse.

Q58 Fail-safe iterator — 2 collections that use it? [Easy]

▶ CopyOnWriteArrayList: iterator on snapshot — never sees concurrent modifications, never CME. ConcurrentHashMap: weakly consistent — may or may not reflect concurrent mods, never CME. Trade-off: may see stale data.

Q59 ListIterator vs Iterator — additional capabilities? [Medium]

▶ ListIterator: hasPrevious()/previous() (bidirectional), nextIndex()/previousIndex(), add(e) at current position (safe), set(e) to replace last returned. Only for Lists.

Q60 What is a Spliterator? [Hard]

▶ Parallel-friendly iterator. trySplit(): splits into two halves for parallel processing. tryAdvance(Consumer): process one element. Characteristics: SIZED, ORDERED, SORTED, DISTINCT, etc. Used by parallel streams for ForkJoin work decomposition.

Q61 ConcurrentHashMap reads — how implemented? [Hard]

▶ Volatile reads of Node.val and Node.next — no locking at all. Value field declared volatile — writes immediately visible to all readers. Full concurrent reads across all buckets simultaneously. Reads never block writers.

Q62 putIfAbsent() guarantee vs put()? [Hard]

▶ putIfAbsent(k,v): check-and-insert is atomic — no race between check and insert. put(k,v): always replaces. In ConcurrentHashMap: putIfAbsent and computeIfAbsent are truly atomic. get() + put() in sequence is NOT atomic — race condition possible.

Q63 Weakly consistent iteration in ConcurrentHashMap? [Medium]

Q64 Collections.synchronizedMap() vs ConcurrentHashMap? [Medium]

▶ synchronizedMap: single mutex, one thread at a time, iteration needs external sync. ConcurrentHashMap: fine-grained, concurrent reads (no lock), parallel writes to different buckets. Choose synchronizedMap only to wrap non-HashMap implementations (e.g., TreeMap). Otherwise always prefer CHM.

Q65 CopyOnWriteArrayList performance trade-offs? [Medium]

▶ Write (add/set/remove): copies entire array under ReentrantLock — O(n), expensive. Read: lock-free snapshot — O(1), zero contention. Iterator: snapshot at creation, no CME. Best: read-heavy, rarely-written. Worst: high-write (use ConcurrentLinkedQueue instead).

Q66 ConcurrentSkipListMap vs ConcurrentHashMap? [Hard]

▶ ConcurrentSkipListMap: skip list, O(log n), sorted, NavigableMap (range queries), non-blocking CAS. Use when sorted order + concurrency both needed. ConcurrentHashMap: O(1), no order. Choose CHM for raw throughput; ConcurrentSkipListMap for sorted + concurrent.

Q67 ConcurrentHashMap size() — why approximate? [Hard]

▶ Uses LongAdder — array of counter cells, each thread updates its cell (reduces CAS contention). size() sums all cells. Between summing cells, other threads may insert/remove — result may be slightly stale. Fine for metrics/logging.

Q68 Collections.synchronizedList() iteration safety? [Hard]

▶ NOT safe by default — must manually synchronize on the list: synchronized(list){ for(E e : list){...} }. Without this: CME from concurrent modification. CopyOnWriteArrayList: no external sync needed — snapshot guarantees safety.

Q69 Atomic compound operation in ConcurrentHashMap — what does atomic mean? [Medium]

▶ Read AND write execute within the same bucket lock — no thread can interleave between them. Atomic methods: putIfAbsent, computeIfAbsent, compute, merge, replace(k,old,new). Contrast: get() + put() are two separate lock acquisitions — race possible between them.

Q70 Name 5 thread-safe collections in java.util.concurrent? [Easy]

▶ ConcurrentHashMap, CopyOnWriteArrayList, ConcurrentLinkedQueue, ArrayBlockingQueue, LinkedBlockingQueue, PriorityBlockingQueue, ConcurrentSkipListMap, ConcurrentSkipListSet, LinkedTransferQueue, DelayQueue.

Q71 Decision table: HashMap vs LinkedHashMap vs TreeMap vs ConcurrentHashMap? [Medium]

▶ HashMap: O(1), no order, single-threaded. LinkedHashMap: O(1), insertion/access order, LRU cache. TreeMap: O(log n), sorted, range queries. ConcurrentHashMap: O(1), no order, multi-threaded. EnumMap: O(1) direct array, enum keys only.

Q72 HashMap.putAll() time complexity and resize behaviour? [Medium]

▶ O(n) where n = source size. Before inserting: if current+source.size() > threshold → resize first (one upfront resize vs multiple incremental). Then inserts each entry.

Q73 Why is using a mutable object as a HashMap key dangerous? [Hard]

▶ If mutable fields used in hashCode/equals change after insertion: object is in wrong bucket. Lookups compute new bucket index — object not found (it's in old bucket). Object 'lost' in map — cannot retrieve or remove. Rule: only use immutable objects as Map keys.

Q74 Memory overhead of a single HashMap entry? [Hard]

▶ Each Node: ~32 bytes (16B object header + 4B hash + 4B key pointer + 4B value pointer + 4B next pointer). Plus key and value objects. Empty HashMap(16): 16×pointer_size bytes for array. For millions of small entries: consider flat map libraries or primitive arrays.

Q75 Why return Collections.emptyList() instead of null? [Easy]

▶ Shared immutable singleton — zero allocation. Callers can iterate safely without null check — eliminates NPE at call site. Contract: methods returning Collection should never return null. emptyList/emptyMap/emptySet are always the same object.

Q76 Collections.frequency() vs disjoint() vs nCopies()? [Medium]

▶ frequency(col,obj): count occurrences with equals(). disjoint(c1,c2): true if no common elements. nCopies(n,obj): immutable List of n copies of obj — for pre-filling/initializing.

Q77 PriorityQueue in a payment system — 3 use cases? [Medium]

▶ 1) Process high-priority transactions first (VIP/time-sensitive). 2) Retry queue: process soonest-due failed transaction first (min-heap by scheduledTime). 3) Top-K reporting: efficiently maintain top 10 highest-value transactions. For concurrent: PriorityBlockingQueue.

Q78 TreeMap thread-safe options? [Hard]

▶ TreeMap NOT thread-safe. Options: 1) Collections.synchronizedSortedMap(new TreeMap<>()) — coarse lock. 2) ConcurrentSkipListMap — lock-free, sorted, O(log n), better concurrent performance. 3) External ReadWriteLock if reads vastly outnumber writes.

Q79 stream() vs parallelStream() — when is parallel actually faster? [Medium]

▶ Parallel wins: large dataset (100k+ elements), CPU-bound per-element ops, stateless independent ops, easily splittable source (ArrayList, arrays). Parallel loses: small data, IO-bound ops, stateful/shared-state ops, LinkedList (poor Spliterator split). Always benchmark before using parallel.

Q80 Collections Framework hierarchy — top-level interfaces? [Hard]

▶ Iterable→Collection→{List,Set,Queue/Deque}. List: ArrayList,LinkedList,COWAL. Set→SortedSet→NavigableSet: HashSet,LinkedHashSet,TreeSet,EnumSet. Queue→BlockingQueue: ArrayBlockingQueue,LinkedBlockingQueue,PBQ. Map (NOT Collection): HashMap,LinkedHashMap,TreeMap,EnumMap,CHM,ConcurrentSkipListMap.
