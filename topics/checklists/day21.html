Section 1 — Producer Configuration

The acks Setting — The Most Important Producer Decision

acks value

What it means | Data loss risk | Latency | Use when

acks=0

Fire and forget. Producer does not wait for any acknowledgement. | Highest — broker crash = lost message | Lowest | Metrics, telemetry — data loss is acceptable

acks=1

Leader broker acknowledges once it writes to its local log. Replicas have not confirmed yet. | Medium — leader crashes before replication = lost message | Medium | Rarely the right choice — false sense of safety

acks=all (or -1)

All in-sync replicas (ISR) must acknowledge before the producer gets a response. | Lowest — requires min.insync.replicas brokers to agree | Highest | Payments, orders, any data that cannot be lost

The acks=1 trap:

Many teams use acks=1 thinking it is a good balance. It is not. If the leader crashes after acknowledging but before replicating to followers, the message is lost. The new leader elected from the followers does not have it. For anything important, use acks=all with min.insync.replicas=2. This ensures at least two brokers have confirmed the write before success is reported to the producer.

Idempotent Producer

The problem it solves:

// Producer configuration for a payment service
spring:
  kafka:
    producer:
      bootstrap-servers: ${KAFKA_BROKERS}
      key-serializer:   org.apache.kafka.common.serialization.StringSerializer
      value-serializer: org.springframework.kafka.support.serializer.JsonSerializer
      # Durability — do not lose data
      acks: all                    # wait for all ISR replicas
      retries: 2147483647          # retry indefinitely (bounded by delivery.timeout.ms)
      enable-idempotence: true     # deduplication via PID + sequence number
      # Throughput — batch messages before sending
      linger-ms: 5                 # wait up to 5ms to fill a batch
      batch-size: 65536            # 64KB batch (default 16KB)
      compression-type: snappy     # good balance of CPU and ratio
      buffer-memory: 33554432      # 32MB send buffer
      # Ordering — required with retries
      max-in-flight-requests-per-connection: 5  # safe with idempotence enabled
      # Without idempotence, set this to 1 to preserve order under retries
// Sending a message with explicit error handling
@Service
public class PaymentEventPublisher {
    private final KafkaTemplate<String, PaymentEvent> kafkaTemplate;
    public void publish(PaymentEvent event) {
        kafkaTemplate.send("payments.completed", event.getPaymentId(), event)
            .whenComplete((result, ex) -> {
                if (ex != null) {
                    log.error("Failed to publish payment event [id={}]: {}",
                        event.getPaymentId(), ex.getMessage(), ex);
                    // Consider writing to outbox for retry
                } else {
                    log.info("Published to partition {} offset {}
                        result.getRecordMetadata().partition(),
                        result.getRecordMetadata().offset());
                }
            });
    }
}

Batching and Throughput

Config

What it does | Throughput impact | Latency impact

linger.ms=0 (default)

Send immediately when a record is ready | Low — one record per network round trip | Lowest possible latency

linger.ms=5

Wait up to 5ms to accumulate records into a batch | High — many records per round trip | Up to 5ms extra

batch.size=16384 (16KB)

Max batch size per partition per request | Larger = fewer requests = better throughput | No latency penalty unless linger.ms is 0

compression.type=snappy

Compress batch before sending | Less bandwidth, more broker I/O capacity | Small CPU cost, worth it at scale

compression.type=lz4

Faster decompression than snappy, similar ratio | Better for high-throughput consumer scenarios | Slightly better than snappy on consumer side

Q1: What is the difference between acks=1 and acks=all, and which would you use for a payment service? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

With acks=1, the producer waits for the partition leader to write the message to its own log and acknowledge. Replicas have not confirmed anything yet. If the leader crashes after acknowledging but before the followers replicate the message, the message is gone. A new leader is elected from the followers, and they do not have the message. With acks=all, every in-sync replica must acknowledge before the producer gets a success response. Combined with min.insync.replicas=2 on the broker, you need at least two brokers to have written the message before the producer proceeds. For a payment service, acks=all is the only acceptable setting. Losing a payment event because the leader crashed after acknowledging is not a trade-off we make. The latency cost is real but small compared to the consequences of data loss in a financial system.

Section 2 — Consumer Groups and Rebalancing

How Partition Assignment Works

The core rule:

Each partition is assigned to exactly one consumer within a group at any time. Multiple consumer groups can read the same partition independently. If a topic has 6 partitions and your group has 3 consumers, each consumer gets 2 partitions. If you add a 7th consumer, one consumer will be idle — you cannot have more active consumers than partitions.

// Consumer group assignment — visual
// Topic: payments (6 partitions: P0-P5)
// Group: payment-processor (3 consumers: C0, C1, C2)
//
// C0 → P0, P1
// C1 → P2, P3
// C2 → P4, P5
// If C2 crashes:
// REBALANCE triggered
// C0 → P0, P1, P4   (C0 gets one of C2's partitions)
// C1 → P2, P3, P5   (C1 gets the other)
// If we add C3:
// REBALANCE triggered again
// C0 → P0, P1
// C1 → P2, P3
// C2 → P4
// C3 → P5
// If we add C7 (more consumers than partitions):
// C0 → P0, C1 → P1, C2 → P2, C3 → P3, C4 → P4, C5 → P5
// C6 → idle (no partition to assign)
// Never useful to have more consumers than partitions in a group

Rebalancing — The Throughput Killer

What happens during rebalance:

When any consumer in the group joins, leaves, or fails to heartbeat within session.timeout.ms, the group coordinator triggers a rebalance. During eager rebalancing (the default before Kafka 2.4), ALL consumers in the group stop processing and revoke ALL their partitions. They then wait for the group leader to assign new partitions to everyone. During this window, the entire group is idle. For a busy consumer group, this means measurable gaps in processing and consumer lag spikes.

Rebalance type

How it works | Processing pause | Use when

Eager (EAGER)

All consumers stop. All partitions revoked. Group leader reassigns everything from scratch. | Full group stops for entire rebalance | Legacy — avoid for latency-sensitive consumers

Cooperative (COOPERATIVE_STICKY)

Only the partitions that need to move are revoked. Other consumers keep processing. | Minimal — only consumers gaining or losing partitions pause briefly | Default since Kafka 3.1 — use this

Static membership

// Spring Kafka consumer config — cooperative rebalancing + static membership
spring:
  kafka:
    consumer:
      bootstrap-servers: ${KAFKA_BROKERS}
      group-id: payment-processor
      key-deserializer:   org.apache.kafka.common.serialization.StringDeserializer
      value-deserializer: org.springframework.kafka.support.serializer.JsonDeserializer
      auto-offset-reset: earliest   # start from beginning if no committed offset
      enable-auto-commit: false      # manual commit — never trust auto-commit
      properties:
        partition.assignment.strategy: org.apache.kafka.clients.consumer.CooperativeStickyAssignor
        group.instance.id: ${POD_NAME}  # static membership — Kubernetes pod name
        session.timeout.ms: 45000       # how long before coordinator declares consumer dead
        heartbeat.interval.ms: 15000    # must be < session.timeout.ms / 3
        max.poll.interval.ms: 300000    # max time between polls before consumer is removed
    listener:
      ack-mode: MANUAL_IMMEDIATE   # call Acknowledgment.acknowledge() explicitly
      concurrency: 3               # 3 listener threads — one per partition in a 3-partition topic

Offset Commit Strategies

Strategy

When offset is committed | Risk | Use when

Auto-commit (enable.auto.commit=true)

Every auto.commit.interval.ms (default 5s) regardless of processing state | Commits before processing completes → message lost if consumer crashes | Only for truly at-most-once scenarios

Manual commitSync()

After processing, blocks until broker confirms | Slower — synchronous round trip per batch | When you need guaranteed at-least-once with ordered commits

Manual commitAsync()

After processing, does not block — callback on completion | If commit fails and is not retried, offset may not advance | High throughput with careful error handling

Commit after batch (default Spring)

After all records in a poll batch are processed | Reprocesses the whole batch on crash — consumers must be idempotent | Most Spring Kafka services — good default

Q2: What happens during a Kafka consumer group rebalance and how do you minimise the impact? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

During an eager rebalance, which was the default until recently, every consumer in the group stops processing and revokes all its partition assignments. They wait for the group coordinator to redistribute partitions. For a payment consumer with 6 consumers processing a busy topic, this can mean several seconds of zero processing — lag spikes, delayed events, and timeouts upstream. Three things reduce the impact. First, switch to cooperative-sticky rebalancing. Only the partitions that actually need to move are revoked. Consumers that keep their partitions continue processing throughout. Second, use static membership by setting group.instance.id to a stable identifier like the Kubernetes pod name. When a consumer restarts, it re-joins with the same ID and the coordinator gives it back its previous partitions without triggering a full rebalance. Third, tune session.timeout.ms and max.poll.interval.ms carefully. If processing is slow and you are not polling frequently enough, the coordinator wrongly declares the consumer dead and triggers an unnecessary rebalance.

Section 3 — Delivery Semantics and Exactly-Once

The Three Delivery Guarantees

Guarantee

How it is achieved | Risk | Real-world use

At-most-once

At-least-once

Exactly-once (EOS)

Idempotent producer + transactional API. Producer writes records and commits consumer offsets in a single atomic transaction. A record is either processed exactly once or not at all — no duplicates, no loss. | Higher latency, lower throughput — transactional overhead | Financial ledgers, inventory, anywhere duplicates cause business harm

The honest position on exactly-once:

Exactly-once within Kafka is real and works well. Exactly-once across Kafka plus a database is harder — you need the Outbox pattern or transactional outbox to tie them together. Most systems use at-least-once with idempotent consumers because it is simpler to operate and the cost of an occasional duplicate is lower than the complexity of maintaining transactional producers across all services.

Exactly-Once Semantics — How It Works

// Exactly-once: idempotent producer + transactions
// Step 1: Configure transactional producer
spring:
  kafka:
    producer:
      transaction-id-prefix: payment-processor-  # enables transactional mode
      enable-idempotence: true                   # required for transactions
      acks: all                                  # required for transactions
// Step 2: Use KafkaTemplate within a transaction
@Service
public class PaymentProcessor {
    private final KafkaTemplate<String, PaymentEvent> kafkaTemplate;
    // Process a record and publish result — exactly once
    @Transactional("kafkaTransactionManager")
    public void processAndPublish(PaymentCommand command) {
        // All sends and the offset commit happen atomically
        PaymentEvent event = processPayment(command);
        kafkaTemplate.send("payments.completed", event.getId(), event);
        kafkaTemplate.send("audit.log",         event.getId(), AuditEvent.from(event));
        // If anything fails, both sends are rolled back
        // Consumer offset is NOT committed if the transaction aborts
    }
}
// Step 3: Consumer must set isolation.level=read_committed
// This ensures consumers only see messages from committed transactions
// Messages from aborted transactions are invisible to read_committed consumers
spring:
  kafka:
    consumer:
      properties:
        isolation.level: read_committed  # never read messages from aborted transactions
// The atomic unit:
// BEGIN TRANSACTION
//   produce record to payments.completed
//   produce record to audit.log
//   commit consumer offset for payments.commands
// COMMIT TRANSACTION (or ABORT if anything fails)
// Consumer sees nothing until COMMIT — no partial results
Q3: What is exactly-once delivery in Kafka and when would you actually use it? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

Exactly-once in Kafka means a message is processed and its effects are visible exactly one time — not zero, not twice. It is achieved by combining an idempotent producer (which prevents duplicate writes via PID and sequence numbers) with Kafka transactions (which atomically commit the produced records and the consumer offset together). If the consumer crashes mid-processing, the transaction aborts. When it restarts, it reprocesses from the uncommitted offset and the transaction runs again cleanly. Consumers with isolation.level=read_committed only see messages from committed transactions, so aborted messages are invisible. I would use it when duplicates cause real business harm — a financial ledger where processing a payment twice creates incorrect balances, or an inventory system where a duplicate message over-reserves stock. For most services, at-least-once with idempotent consumers is simpler to operate and sufficient. The transactional overhead roughly halves throughput, so it is not a default choice.

The Problem with Failing Messages

What happens without a DLT:

A consumer throws an exception processing message at offset 100. The offset is not committed. Kafka keeps delivering the same message. The consumer keeps failing. All messages after offset 100 are stuck waiting. This is called a poison pill — one bad message blocks the entire partition. A Dead Letter Topic (DLT) gives you a way to move poison pills out of the way and keep processing, then investigate and reprocess them separately.

// Spring Kafka @RetryableTopic — declarative retry + DLT
@Component
public class PaymentEventConsumer {
    @RetryableTopic(
        attempts = "4",                    // 1 original + 3 retries
        backoff = @Backoff(
            delay = 1000,                   // 1s first retry
            multiplier = 2.0,               // exponential: 1s, 2s, 4s
            maxDelay = 10000                // cap at 10s
        ),
        dltTopicSuffix = "-dlt",
        retryTopicSuffix = "-retry-",
        include = { TransientPaymentException.class },  // only retry transient errors
        exclude = { InvalidPaymentException.class }     // do NOT retry — send straight to DLT
    )
    @KafkaListener(topics = "payments.commands", groupId = "payment-processor")
    public void handlePayment(PaymentCommand command, Acknowledgment ack) {
        try {
            paymentService.process(command);
            ack.acknowledge();
        } catch (InvalidPaymentException e) {
            // Non-retryable — goes directly to DLT
            log.error("Non-retryable payment error [id={}]: {}", command.getId(), e.getMessage());
            throw e;
        } catch (TransientPaymentException e) {
            // Retryable — Spring will retry on retry topics before DLT
            log.warn("Transient error [id={}], will retry: {}", command.getId(), e.getMessage());
            throw e;
        }
    }
    // Separate listener for the DLT — inspect, alert, potentially reprocess
    @DltHandler
    public void handleDlt(PaymentCommand command, @Header KafkaHeaders.RECEIVED_TOPIC String topic) {
        log.error("Payment landed in DLT [id={}, sourceTopic={}]",
                  command.getId(), topic);
        // Alert on-call, write to a dead_letter_payments table for manual review
        alertingService.sendAlert("DLT message", command);
        dltRepository.save(DeadLetterRecord.from(command, topic));
    }
}
// Spring creates these topics automatically:
// payments.commands-retry-0   (1s delay)
// payments.commands-retry-1   (2s delay)
// payments.commands-retry-2   (4s delay)
// payments.commands-dlt       (final resting place)
// The message flow on failure:
// payments.commands → fail → payments.commands-retry-0
// payments.commands-retry-0 → fail → payments.commands-retry-1
// payments.commands-retry-1 → fail → payments.commands-retry-2
// payments.commands-retry-2 → fail → payments.commands-dlt

Decision

Choice | Why

Which errors to retry

Transient errors: network timeouts, gateway unavailable, DB connection lost | Retrying fixes them

Which errors NOT to retry

Permanent errors: invalid schema, business rule violation, data corruption | Retrying will never fix them — go straight to DLT

DLT consumer action

Alert on-call + write to dead_letter table for manual investigation + expose /admin/dead-letters endpoint to resubmit | DLT messages need human attention

Resubmitting from DLT

Produce the DLT message back to the original topic with a corrected payload | After fixing the root cause, replay the message

DLT retention

Long — days or weeks | You need time to investigate and fix before messages expire

Q4: What is a dead letter topic and how do you decide which errors should go there immediately vs which should be retried? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

A dead letter topic is where messages go after all retry attempts are exhausted, or immediately if the error is non-retryable. Without it, a message that cannot be processed blocks the partition — the consumer keeps failing on the same offset and nothing after it can move forward. The key design decision is the retry vs dead letter split. Transient errors — network timeouts, a downstream service being temporarily unavailable, a database connection failure — are worth retrying because the underlying condition will likely clear. I use exponential backoff: 1 second, 2 seconds, 4 seconds. Permanent errors — malformed schema, business rule violations, data that is fundamentally invalid — should never be retried because repeating the same operation on bad data will always produce the same failure. These go directly to the DLT. In Spring, @RetryableTopic handles this cleanly with the include and exclude parameters. The DLT listener alerts on-call and writes the message to a database table so it can be investigated and potentially resubmitted after the root cause is fixed.

Section 5 — Kafka Streams

KStream vs KTable

KStream | KTable

Represents

An unbounded stream of events — each record is an independent event | A changelog of the latest value per key — like a materialised view

Key semantics

Each record is processed independently regardless of key | A new record with the same key overwrites the previous value for that key

Example

A stream of payment events — each event matters | Account balances — only the latest balance per account matters

Join semantics

Stream-stream join: both sides are events in a time window | Stream-table join: enrich every stream event with the current table value

State

Stateless by default (stateful with aggregations) | Always stateful — maintains the latest value per key in a RocksDB store

// Kafka Streams: real-time payment fraud detection
@Configuration
@EnableKafkaStreams
public class FraudDetectionStreams {
    @Bean
    public KStream<String, PaymentEvent> fraudDetectionStream(StreamsBuilder builder) {
        // KStream of all payment events
        KStream<String, PaymentEvent> payments = builder
            .stream("payments.completed",
                    Consumed.with(Serdes.String(), paymentSerde()));
        // KTable of account risk profiles — enrichment table
        KTable<String, AccountRisk> riskProfiles = builder
            .table("account.risk.profiles",
                   Consumed.with(Serdes.String(), accountRiskSerde()),
                   Materialized.as("account-risk-store"));
        // Enrich each payment with the account's risk profile
        KStream<String, EnrichedPayment> enriched = payments.join(
            riskProfiles,
            (payment, risk) -> new EnrichedPayment(payment, risk),
            Joined.with(Serdes.String(), paymentSerde(), accountRiskSerde())
        );
        // Count payments per account in a 5-minute tumbling window
        KTable<Windowed<String>, Long> paymentCounts = enriched
            .groupByKey()
            .windowedBy(TimeWindows.ofSizeWithNoGrace(Duration.ofMinutes(5)))
            .count(Materialized.as("payment-counts-store"));
        // Flag accounts with more than 10 payments in 5 minutes
        paymentCounts
            .toStream()
            .filter((windowedKey, count) -> count > 10)
            .map((windowedKey, count) -> KeyValue.pair(
                windowedKey.key(),
                new FraudAlert(windowedKey.key(), count, windowedKey.window())))
            .to("fraud.alerts", Produced.with(Serdes.String(), fraudAlertSerde()));
        return payments;
    }
}
Q5: What is the difference between KStream and KTable in Kafka Streams? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

KStream represents an unbounded sequence of independent events — every record matters and nothing overwrites anything else. KTable represents the current state per key — if two records arrive with the same key, the second one replaces the first. Think of KStream as a log of events and KTable as a database table that is continuously updated from that log. The practical difference shows up in joins. Joining two KStreams means matching events that fall within the same time window — both sides are records in time. Joining a KStream against a KTable means enriching each event with the current value for that key in the table. In our payment example: the stream of payment events is a KStream, and the account risk profiles are a KTable. For every payment event, I look up the current risk score for that account from the table and use it to decide whether to flag the payment. KTable state is stored in RocksDB locally on each Kafka Streams instance, with a changelog topic in Kafka as the backup for fault tolerance.

Section 6 — Partition Sizing and Consumer Lag

How to Size Partitions

Why partition count matters:

Partitions are the unit of parallelism in Kafka. More partitions means more consumer threads can process in parallel. But partitions also have overhead — each partition requires memory on the broker, leader election time during failover, and coordination overhead. You cannot easily reduce partition count later — only increase. Partition count also sets the maximum concurrency for a consumer group.

Factor

Guidance

Target throughput

Measure: one producer and one consumer thread each handle roughly 10-50MB/s. Partition count = max(target_producer_throughput / per_producer_throughput, target_consumer_throughput / per_consumer_throughput)

Consumer parallelism

You need at least as many partitions as your maximum consumer thread count. 12 partitions = maximum 12 concurrent consumers. Plan for future growth — over-partition slightly.

Retention and storage

Each partition retains up to retention.bytes of data. Total storage = partitions × retention.bytes × replication.factor. Factor this into broker disk planning.

Adding partitions later

Consumer Lag — The Most Important Kafka Metric

What consumer lag means:

Consumer lag is the difference between the latest offset produced to a partition and the last offset committed by the consumer group. If the topic has offset 1000 and the consumer has committed offset 850, the lag is 150. A lag of zero means the consumer is keeping up. Growing lag means the consumer is falling behind — it is processing slower than messages are arriving.

// Monitoring lag from the command line
kafka-consumer-groups.sh --bootstrap-server broker:9092 \
  --describe --group payment-processor
// Output:
// GROUP              TOPIC             PARTITION  CURRENT-OFFSET  LOG-END-OFFSET  LAG
// payment-processor  payments.commands  0          850             1000            150
// payment-processor  payments.commands  1          920             920             0
// payment-processor  payments.commands  2          100             800             700  ← problem
// Partition 2 has lag 700 — one consumer thread is stuck or slow
// Micrometer Gauge for consumer lag — publish to CloudWatch
@Component
public class KafkaLagMonitor {
    private final AdminClient  adminClient;
    private final MeterRegistry registry;
    @Scheduled(fixedDelay = 30000)   // check every 30 seconds
    public void reportLag() {
        Map<TopicPartition, OffsetAndMetadata> committed =
            adminClient.listConsumerGroupOffsets("payment-processor")
                .partitionsToOffsetAndMetadata().get();
        Map<TopicPartition, Long> endOffsets =
            adminClient.listOffsets(endOffsetSpec(committed.keySet())).all().get()
                .entrySet().stream()
                .collect(toMap(Map.Entry::getKey, e -> e.getValue().offset()));
        committed.forEach((tp, offsetMeta) -> {
            long lag = endOffsets.get(tp) - offsetMeta.offset();
            Gauge.builder("kafka.consumer.lag", () -> lag)
                .tag("topic",     tp.topic())
                .tag("partition", String.valueOf(tp.partition()))
                .tag("group",     "payment-processor")
                .register(registry);
        });
    }
}
// CloudWatch alarm: alert when lag > 10,000 for 5 minutes
// This means consumers are ~10,000 messages behind — investigate immediately

Lag symptom

Likely cause | Fix

Lag growing on all partitions

Producer throughput exceeds consumer capacity | Add consumer instances (up to partition count); scale up consumer resources

Lag growing on one partition only

That consumer thread is stuck, slow, or processing a poison pill | Check that consumer's logs; look for exceptions; check if DLT has messages

Lag spikes then recovers

Rebalancing — all consumers paused during rebalance | Switch to cooperative-sticky; use static membership to reduce rebalance frequency

Lag at zero but processing latency high

Consumer is keeping up in terms of offsets but each message takes too long | Profile the processing code; check downstream service latency; add parallelism within consumer

Lag jumps after deployment

Q6: How do you monitor Kafka consumer health in production and what does growing lag tell you? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Consumer lag is the primary health signal. It tells you whether consumers are keeping up with the rate of incoming messages. A lag trending toward zero is healthy. A lag that grows consistently means producers are outpacing consumers. I expose lag as a Micrometer Gauge per partition and publish it to CloudWatch. I set an alarm when lag exceeds 10,000 on any partition for more than five minutes — that is the signal to look at what is happening. When lag is growing on all partitions, it usually means consumer throughput is the bottleneck — either add more consumer instances or make processing faster. When lag grows on one partition only, I look at that specific consumer thread: is it stuck on a poison pill message, is it throwing exceptions, is a downstream service it depends on slow. The lag number alone tells you there is a problem. The partition-level breakdown and the consumer logs tell you where.

Q7: What does the acks setting control and what are the trade-offs between acks=1 and acks=all? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

acks controls how many brokers must confirm a write before the producer considers it successful. With acks=1, only the partition leader needs to write and acknowledge. It is faster but has a gap: if the leader crashes after acknowledging but before the followers replicate, the message is gone and the new leader will not have it. This is silent data loss — the producer got a success but the message is not there. With acks=all, every in-sync replica must confirm. Combined with min.insync.replicas=2 on the broker side, you need at least two brokers to have the message before success is reported. This protects against leader failures. The cost is higher latency on each send. For payments or any financial data, acks=all is not optional — data loss is worse than extra milliseconds.

Q8: What is an idempotent producer and what problem does it solve? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

The problem: with retries enabled, a producer sends a message, the broker writes it, but the network drops the acknowledgement. The producer retries and the broker receives a duplicate. Without idempotence, the topic has the message twice — downstream consumers process a payment twice. With enable.idempotence=true, the broker assigns the producer a PID (Producer ID) and tracks a sequence number per partition. If a message arrives with a PID and sequence number it has already seen, the broker drops the duplicate silently. The producer gets a success response and has no idea a duplicate was received and discarded. The result is exactly-once writes per producer session within a single partition. Idempotence is required when you enable transactions and is strongly recommended any time you have retries enabled with data that must not be duplicated.

Q9: Why does Kafka rebalancing hurt throughput and how do you mitigate it? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

During a rebalance with eager rebalancing (the old default), the entire consumer group stops. Every consumer revokes its partitions, waits for the coordinator to run the assignment algorithm, and then starts fresh. For a group with heavy processing, this can mean several seconds of zero throughput — and because messages are still arriving from producers, lag accumulates during that window. Three mitigations. Cooperative-sticky rebalancing only revokes and reassigns the partitions that actually need to move. Consumers that keep their partitions continue processing during the rebalance. Static membership, configured via group.instance.id, lets a consumer restart and rejoin with the same identity, getting its previous partitions back without triggering a rebalance. And careful tuning of max.poll.interval.ms prevents unnecessary rebalances caused by a consumer that is processing a large batch and appears dead to the coordinator.

Q10: What is the difference between at-least-once and exactly-once delivery? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

At-least-once means a message will be delivered one or more times. You get this by committing offsets after processing — if the consumer crashes after processing but before committing, the message is redelivered and processed again. The consumer must be idempotent to handle duplicates correctly. Exactly-once means a message is processed and its effects are applied exactly one time. In Kafka, this is achieved by combining idempotent producers with Kafka transactions. The producer writes its output records and commits the input consumer offset in a single atomic operation. If the transaction aborts, neither the output records nor the offset commit are applied — the consumer restarts and tries again from the same input offset. The cost is roughly half the throughput of at-least-once because of the transactional overhead. Most services use at-least-once with idempotent consumers because it is simpler and sufficient. Exactly-once is for cases where duplicates would cause real business harm that cannot be handled by idempotency logic.

Q11: What is a Dead Letter Topic and how does Spring's @RetryableTopic implement the pattern? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

A Dead Letter Topic is where messages go after all retry attempts fail, or immediately for errors that are not worth retrying. Without a DLT, a message that consistently fails blocks its partition — the consumer keeps failing on the same offset and nothing after it can be processed. The DLT takes the poison pill out of the main flow so the rest of the queue can continue. Spring's @RetryableTopic automatically creates a chain of retry topics with configurable delays between attempts. The message moves from the original topic to retry-0, then retry-1, then retry-2, and finally to the DLT. You configure which exceptions trigger a retry and which go directly to the DLT. A @DltHandler method on the same class handles DLT messages — typically alerting the on-call team, writing to a database for investigation, and exposing a way to resubmit after the root cause is fixed.

Q12: How do you choose between retrying an error and sending it to the DLT immediately? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

The split is between transient and permanent errors. Transient errors are conditions that will likely resolve with time — a downstream service that is temporarily unavailable, a network timeout, a database connection pool exhausted. Retrying after a delay gives these a chance to clear. Exponential backoff prevents hammering a struggling service. Permanent errors are conditions that will never fix themselves by retrying — a payment with an invalid structure, a business rule violation, a schema mismatch, or data that is simply wrong. Retrying a message with an invalid account ID four times does not make the account ID valid. It just delays landing in the DLT by however long the retry backoff takes. I use the include and exclude parameters in @RetryableTopic to classify exception types. The classification matters operationally: a high DLT rate on transient errors tells you a dependency is having issues. A high DLT rate on permanent errors tells you something upstream is producing bad data.

Q13: What does consumer lag mean and what does it tell you about your system? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Consumer lag is the number of messages sitting in the topic that the consumer group has not yet processed. Specifically, it is the difference between the latest produced offset and the last committed consumer offset per partition. Zero lag means the consumer is processing messages as fast as they arrive. Growing lag means producers are outpacing consumers. The diagnosis depends on where the lag is growing. If lag grows on all partitions equally, the consumer group as a whole is too slow — add more consumer instances or optimise processing. If lag grows on one partition only, that specific consumer thread is the problem — look for exceptions, a blocking downstream call, or a poison pill. Lag spikes that recover quickly usually indicate rebalancing. I track lag as a Micrometer Gauge per partition and alert when any partition exceeds a threshold for more than a few minutes, because prolonged lag means event processing is falling behind real time.

Q14: How many partitions should a Kafka topic have? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Partition count determines the maximum parallelism for consumers and cannot be reduced after creation. I think about it from two angles. From the consumer side: how many consumer threads will I run at peak? If I want 12 consumer threads, I need at least 12 partitions. I usually create 1.5 to 2 times the expected peak consumer count to allow for future growth without schema changes. From the throughput side: measure how much throughput one producer thread and one consumer thread can sustain for your message size and processing logic, then divide the target throughput by that number. For a payment topic where I expect 6 consumers initially but may scale to 12, I would start with 18 partitions. The practical advice: it is better to over-partition than under-partition. Adding partitions later breaks key-to-partition mapping for keyed messages, which can break ordering guarantees for consumers that depend on them.

Q15: What is the role of isolation.level=read_committed and when must it be set? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

isolation.level=read_committed controls what messages a consumer sees when producers are using Kafka transactions. With the default read_uncommitted, a consumer sees all messages as soon as they are written, including messages from transactions that have not committed yet and messages from transactions that will eventually abort. If a transactional producer writes a record and then aborts the transaction, a read_uncommitted consumer has already processed a message that was supposed to never exist. With read_committed, consumers only see messages from committed transactions. Messages from in-progress or aborted transactions are invisible until either the transaction commits or the consumer advances past them. This must be set any time you use Kafka transactions for exactly-once semantics. Without it, the consumer-side half of exactly-once does not hold — consumers can read phantom records from aborted transactions.

Q16: What is linger.ms and how does it affect throughput vs latency? [Easy] ▼
💡 Ideal Answer (Amazon/GS Standard)

linger.ms tells the producer how long to wait for more records before sending a batch. With the default of 0, the producer sends a record immediately as soon as it is ready — one record per network round trip in the low-throughput case. With linger.ms=5, the producer waits up to 5 milliseconds to accumulate more records into the same batch. If 50 records arrive in that 5ms window, they all go in one network request instead of 50 separate ones. This dramatically improves throughput and reduces broker CPU load. The trade-off is up to 5ms of additional latency for each record. For a payment service where end-to-end latency matters but is measured in hundreds of milliseconds, adding 5ms at the Kafka layer is usually invisible. For a real-time bidding system where latency is measured in single-digit milliseconds, linger.ms=0 is the right choice. I pair linger.ms=5 with batch.size=65536 to get good batching for high-throughput scenarios.

Q17: What is static membership in Kafka and why does it matter in Kubernetes? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Static membership lets you assign a stable identity — group.instance.id — to each consumer. Without it, every time a consumer restarts it looks like a new member to the group coordinator, which triggers a rebalance. In Kubernetes, pods restart frequently — rolling deploys, OOM restarts, node evictions. Each restart without static membership triggers a full group rebalance, pausing all processing during the reassignment. With static membership, a restarting pod rejoins with the same identity and the coordinator gives it back its previous partitions without a rebalance, as long as it rejoins within session.timeout.ms. I set group.instance.id to the Kubernetes pod name using the Downward API. This is one of the most impactful operational improvements for Kafka in Kubernetes — it eliminates most of the unnecessary rebalances that happen during normal pod lifecycle events.

Q18: How do you handle a Kafka consumer that is processing a message too slowly and timing out? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

The symptom: the consumer takes longer than max.poll.interval.ms to process a message, so the coordinator declares it dead and triggers a rebalance. Two approaches depending on root cause. If the processing itself is genuinely slow — a large payload, a complex calculation, a slow downstream call — increase max.poll.interval.ms to match realistic processing time, and reduce max.poll.records so each poll fetches fewer records and the processing time per poll batch is bounded. If the slow path is in a specific downstream call, add a timeout and circuit breaker on that call so slow dependencies do not cascade into consumer timeouts. The second approach is to move the slow processing off the consumer thread: the consumer writes the message to a work queue and immediately acknowledges, while a separate thread pool does the heavy processing. This keeps the poll loop fast and prevents coordinator timeouts, at the cost of adding at-least-once semantics to the work queue as well.

Q19: What is the relationship between partition count, consumer count, and throughput? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Partitions are the unit of parallelism. One consumer thread processes one partition at a time within a consumer group. If you have 6 partitions and 3 consumers, each consumer handles 2 partitions — you have 3 threads of parallelism. Adding a 4th consumer means one of the existing consumers gives up one partition — you get 4 threads of parallelism. If you add a 7th consumer to a 6-partition topic, the 7th consumer sits idle — you cannot have more active consumer threads than partitions. To increase throughput beyond what 6 consumers can achieve, you need more partitions. This is why partition count should be planned generously up front. The relationship with throughput is roughly linear until other bottlenecks appear — broker I/O, network, or downstream service capacity.

Q20: How do you replay Kafka messages from a specific point in time? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Kafka retains messages for a configurable period (default 7 days, configurable per topic). To replay from a specific point: reset the consumer group's committed offsets to the desired position. The kafka-consumer-groups.sh tool with --reset-offsets can reset to earliest (beginning of retention), latest (current end), a specific offset, a specific datetime, or a duration ago (--shift-by). With Spring Kafka, you can also do this programmatically via the AdminClient API or by listening to the ConsumerRebalanceListener callback and calling consumer.seek() on partition assignment. One important operational note: before replaying, stop the consumer group so there are no active consumers. Resetting offsets while consumers are running is either rejected or races with their commits. After the reset, restart the consumers — they will pick up from the new offset. Replay is one of the genuinely powerful properties of Kafka over traditional message queues — the log is persistent and replayable, not consumed and discarded.

Q21: What is the difference between a Kafka topic and a traditional message queue? [Easy] ▼
💡 Ideal Answer (Amazon/GS Standard)

A traditional message queue — RabbitMQ, SQS — is designed around consume-and-delete. A consumer reads a message, processes it, and the message is gone. If you need to reprocess or replay, you have to store the messages somewhere else yourself. Kafka is a distributed log. Messages are retained for a configured period regardless of whether they have been consumed. Multiple independent consumer groups can read the same topic from different positions. One group can be at offset 5000 while another is at offset 100 catching up. Consumer groups can be reset to replay from the beginning. New groups start from wherever they choose. This changes the architecture of event-driven systems: producers publish events to a topic, and any number of consumers can subscribe independently without the producer needing to know about them. The durability and replayability also enable event sourcing patterns and audit trails naturally. The trade-off: Kafka is operationally heavier than a simple queue, and the log model means you must manage retention periods carefully to control disk usage.
