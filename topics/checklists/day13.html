Section 1 — Scaling

Vertical vs Horizontal Scaling

Vertical Scaling | Horizontal Scaling

What it means

Bigger machine — more CPU, RAM, faster disk | More machines — add nodes to the cluster

When it works

Quick wins, early stage, stateful apps | Stateless apps, high traffic, fault tolerance

Hard ceiling

Yes — biggest available machine has a limit | No — theoretically infinite

Cost

Expensive at the top end (diminishing returns) | Commodity hardware, linear cost

Failure impact

Single point of failure | One node fails, others continue

Application change

Real example

Upgrade RDS from db.t3.medium to db.r5.4xlarge | Add EC2 instances behind an ALB

The real answer to 'when do you scale horizontally?':

When your single machine's vertical ceiling is approaching, OR when you need fault tolerance, OR when your load is bursty and you want auto-scaling. In AWS: EC2 Auto Scaling Groups + Application Load Balancer is horizontal scaling. RDS read replicas are horizontal read scaling. Aurora serverless is automatic vertical scaling within a range. The shift from vertical to horizontal is also a shift in application architecture — you cannot horizontally scale a stateful app that stores sessions in memory. You externalise state to Redis or a database first.

Load Balancing

Algorithm

How it works | Best for

Round Robin

Requests go to servers in rotation, equally | Homogeneous servers, similar request cost

Weighted Round Robin

Same but heavier servers get more requests | Mixed instance types

Least Connections

Route to the server with fewest active connections | Varying request durations (long-running vs fast)

IP Hash

Hash client IP to always hit the same server | Session stickiness without a session store

Random

Random selection | Simple, works fine for stateless uniform load

Least Response Time

Route to the server with lowest latency + fewest connections | Latency-sensitive applications

L4 vs L7 Load Balancing:

L4 (Transport Layer) — routes based on IP and TCP port. Fast, low overhead, no inspection of payload. AWS NLB is L4. L7 (Application Layer) — routes based on HTTP headers, URL path, cookies, or request body. Slower but smarter: can route /api/payments to the payment service and /api/users to the user service. AWS ALB is L7. Health checks: L4 checks TCP connection, L7 checks HTTP response code. In microservices you almost always want L7 so you can do path-based and header-based routing.

Q1: When would you use IP hash load balancing and what is its drawback? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

IP hash is useful when you want the same client to always hit the same server — session stickiness without needing a shared session store. The drawback is that it can create uneven load distribution: if a small number of clients generate a disproportionate volume of traffic, those clients always hit the same server while others are underutilised. The other issue is that when you add or remove a server, the hash changes and clients get routed to different servers — breaking any local session state. In modern stateless microservices we prefer externalising sessions to Redis and using round-robin or least-connections instead.

Section 2 — CAP Theorem

The Three Properties

Consistency (C):

Every read returns the most recent write, or an error. All nodes see the same data at the same time. Strong consistency. Not to be confused with the C in ACID — they are different.

Availability (A):

Every request receives a response — not necessarily the most recent data, but a response. The system never rejects a request with an error due to a node being down.

Partition Tolerance (P):

The system continues operating even when network messages between nodes are lost or delayed. In any real distributed system, network partitions will happen — this is non-negotiable. You cannot sacrifice P in a real distributed system.

What CAP actually means in practice:

Since P is mandatory in any real distributed system (network partitions happen whether you want them to or not), the real choice is between C and A during a partition. When a partition occurs: do you return an error (choose C — consistency) or return potentially stale data (choose A — availability)? This is not a permanent architecture choice — most modern databases let you tune this per-operation.

PACELC — The More Complete Model

What PACELC adds:

CAP only describes behaviour during a partition. PACELC extends this: 'If Partition, choose between Availability and Consistency. Else (normal operation), choose between Latency and Consistency.' Most real-world debates are about the Else case — when the network is healthy, do you want low latency (return local data immediately) or strong consistency (wait for all replicas to confirm)? DynamoDB default: PA/EL — available during partition, low latency during normal operation. DynamoDB with strong consistency: PA/EC — available during partition, but consistent (waits for all replicas) during normal op.

Q2: A candidate says 'we chose Cassandra for our payment system because it is highly available.' What concerns would you raise? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

Cassandra is AP — it prioritises availability over consistency. In a payment system, double-spending and lost transactions are catastrophic. If two nodes cannot communicate during a partition and both accept a debit, you have a real financial problem. I would push back with: what is your consistency level per operation? Cassandra supports QUORUM and ALL reads and writes — at QUORUM you get a much stronger consistency guarantee at the cost of some availability. For payment ledger writes specifically I would want at least LOCAL_QUORUM. The other concern is last-write-wins conflict resolution — payments are not idempotent by nature, so you need explicit idempotency keys and application-level deduplication on top. For a payments system I would lean toward CP — Aurora, CockroachDB, or Spanner — and accept the occasional timeout over the risk of inconsistency.

Redis — The Practical Details

Data structures you must know:

String (get/set/incr/expire), Hash (hgetall/hset — good for objects), List (lpush/rpop — queue), Set (sadd/sismember — unique membership), Sorted Set (zadd/zrange — leaderboard, rate limiting), Bitmap (setbit/bitcount — presence flags), HyperLogLog (pfadd/pfcount — approximate unique counts).
// ── Cache-Aside in Spring Boot (most common pattern) ──
@Service
public class AccountService {
    private final AccountRepository repo;
    private final RedisTemplate<String, Account> redis;
    private static final Duration TTL = Duration.ofMinutes(10);
    public Account getAccount(String accountId) {
        String key = "account:" + accountId;
        // 1. Check cache
        Account cached = redis.opsForValue().get(key);
        if (cached != null) return cached;
        // 2. Cache miss — load from DB
        Account account = repo.findById(accountId)
            .orElseThrow(() -> new AccountNotFoundException(accountId));
        // 3. Populate cache with TTL
        redis.opsForValue().set(key, account, TTL);
        return account;
    }
    public void updateBalance(String accountId, BigDecimal newBalance) {
        repo.updateBalance(accountId, newBalance);
        // Invalidate cache — next read will reload fresh data
        redis.delete("account:" + accountId);
    }
}
// ── Spring @Cacheable (Read-Through style, declarative) ──
@Cacheable(value = "accounts", key = "#accountId", unless = "#result == null")

    return repo.findById(accountId).orElseThrow();
}
@CacheEvict(value = "accounts", key = "#accountId")
public void updateBalance(String accountId, BigDecimal newBalance) {

}
// ── Distributed Lock with Redis (Redlock pattern) ──
// Prevent duplicate payment processing across multiple instances
public PaymentResult processOnce(PaymentRequest req) {
    String lockKey  = "lock:payment:" + req.getIdempotencyKey();
    String lockValue = UUID.randomUUID().toString(); // unique per instance
    // Try to acquire lock: SET NX EX (set if not exists, with expiry)
    Boolean acquired = redis.opsForValue()
        .setIfAbsent(lockKey, lockValue, Duration.ofSeconds(30));
    if (!acquired) {
        throw new DuplicatePaymentException("Payment already in progress");
    }
    try {
        return doProcess(req);
    } finally {
        // Release only if we still own the lock (compare-and-delete)
        // Use Lua script for atomicity — GET then DEL is not atomic
        String currentValue = redis.opsForValue().get(lockKey);
        if (lockValue.equals(currentValue)) redis.delete(lockKey);
    }
}
// ── Redis Sorted Set for rate limiting ──
// Sliding window: track timestamps of requests in a sorted set
public boolean isAllowed(String userId, int maxRequests, Duration window) {
    String key     = "ratelimit:" + userId;
    long   now     = System.currentTimeMillis();
    long   cutoff  = now - window.toMillis();
    // Remove timestamps older than the window
    redis.opsForZSet().removeRangeByScore(key, 0, cutoff);
    // Count requests in current window
    Long count = redis.opsForZSet().zCard(key);
    if (count != null && count >= maxRequests) return false;
    // Add current request timestamp
    redis.opsForZSet().add(key, String.valueOf(now), now);
    redis.expire(key, window);
    return true;
}

Cache Invalidation — the hard problem:

Phil Karlton famously said there are only two hard problems in computer science: cache invalidation and naming things. The difficulty: you have multiple app instances, each with a local or shared cache. When data changes, how do you ensure all caches see the fresh data? Options: TTL expiry (simple, but stale window), write-through (invalidate/update on every write, consistent but slower writes), event-driven invalidation (publish a 'data changed' event via Kafka or Redis pub/sub, all instances invalidate on receipt — decoupled and near real-time). The thundering herd problem: many concurrent cache misses all hit the DB at once. Fix: probabilistic early expiration, or a lock (only one thread refreshes while others wait).

Q3: Walk me through how you would cache account balances in a payment service. [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

I would use cache-aside with Redis. On a balance read: check Redis first with key 'account:{id}:balance'. Hit means return immediately. Miss means load from the database, store in Redis with a TTL of around 30 seconds, then return. The TTL choice depends on how stale you can tolerate — for balance displays 30 seconds is usually fine since we show the last known balance. On a balance update — any debit or credit — I delete the Redis key immediately so the next read sees fresh data from the database. I would not use write-through here because I do not want a cache failure to block a payment write to the database. The two critical concerns: thundering herd on a cache miss for a popular account, which I handle with a per-key Redis lock so only one request refreshes; and the distributed lock for payment processing itself, so two concurrent payment requests for the same account do not both pass a balance check against stale cache.

Section 4 — Database Sharding

What Sharding Is and When You Need It

The problem sharding solves:

A single database node has limits: disk size, write throughput, connection count, query concurrency. When you exceed what the biggest available machine can handle, you split the data horizontally across multiple database nodes — each node owns a subset of the rows. This is horizontal partitioning at the database layer. You should consider sharding only after: read replicas are saturated, connection pooling is maxed, vertical scaling is too expensive, and query optimisation is done. Sharding introduces significant operational complexity — exhaust simpler options first.

Shard Key Selection — The Most Important Decision

Shard Key Choice

What happens

accountId (good)

Even distribution if accountIds are uniformly distributed. Most queries for one account hit one shard.

timestamp / date (bad)

All new writes go to the latest shard — hotspot. Old shards are cold. This is the time-series hotspot problem.

country (can be bad)

If 80% of users are in one country, 80% of writes go to one shard. Know your data distribution first.

userId (common, good if uniform)

composite key (advanced)

Hash(tenantId + entityId) — balances load across tenants. Common in multi-tenant SaaS.

Consistent Hashing

hash(key) % N assigns each key to a node. If you add or remove a node, N changes and almost every key maps to a different node — massive cache invalidation, hot spots, or full data rebalance. With 10 nodes, removing 1 remaps approximately 90% of keys.

How consistent hashing works:

Imagine a ring from 0 to 2^32 - 1. Each node is placed at a position on the ring by hashing its identifier. Each key is placed at hash(key) on the ring and is owned by the first node clockwise from that position. When you add a node: only keys between the new node and its predecessor on the ring need to move — roughly 1/N of all keys. When you remove a node: only that node's keys move to the next clockwise node. Virtual nodes (vnodes): each physical node occupies multiple positions on the ring. This ensures even distribution even when physical nodes have different capacities. Cassandra, DynamoDB, and Redis Cluster all use consistent hashing.

// Simplified consistent hashing ring
public class ConsistentHashRing {
    private final TreeMap<Long, String> ring    = new TreeMap<>();
    private final int                   vnodes;  // virtual nodes per physical node
    private final MessageDigest         md;
    public ConsistentHashRing(int vnodes) throws Exception {
        this.vnodes = vnodes;
        this.md     = MessageDigest.getInstance("MD5");
    }
    public void addNode(String nodeId) {
        for (int i = 0; i < vnodes; i++) {
            long hash = hash(nodeId + "-vnode-" + i);
            ring.put(hash, nodeId);
        }
    }
    public void removeNode(String nodeId) {
        for (int i = 0; i < vnodes; i++) {
            ring.remove(hash(nodeId + "-vnode-" + i));
        }
    }
    public String getNode(String key) {
        if (ring.isEmpty()) throw new IllegalStateException("No nodes in ring");
        long hash  = hash(key);
        // Find the first node at or after this hash position
        Map.Entry<Long, String> entry = ring.ceilingEntry(hash);
        // Wrap around the ring if we are past the last node
        return entry != null ? entry.getValue() : ring.firstEntry().getValue();
    }
    private long hash(String key) {
        byte[] digest = md.digest(key.getBytes());
        // Use first 4 bytes as an unsigned int
        return ((long)(digest[0] & 0xFF) << 24) | ((long)(digest[1] & 0xFF) << 16)
             | ((long)(digest[2] & 0xFF) << 8)  |  (long)(digest[3] & 0xFF);
    }
}
Q4: What is the hotspot problem in database sharding and how do you prevent it? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

A hotspot occurs when a disproportionate amount of traffic hits one shard — usually because the shard key correlates with request frequency. Classic example: sharding by date. All new writes go to today's shard; historical shards sit idle. Another example: sharding by customer tier where your top 10 enterprise customers generate 80% of traffic and all land on the same shard. Prevention: choose a shard key that is high-cardinality and uniformly distributed. UUID-based account IDs work well. If you must use a natural key that skews, add a random prefix or suffix to spread it. Consistent hashing with virtual nodes helps distribute a skewed key distribution across physical nodes. In DynamoDB, you explicitly design partition keys to avoid hot partitions — the recommended pattern is to add a random suffix and query all suffixed partitions in parallel.

Section 5 — Rate Limiting

The Three Main Algorithms

Algorithm

How it works | Burst handling | Best for

Token Bucket

Bucket holds N tokens. Each request consumes one. Tokens refill at rate R/sec. Burst allowed up to bucket capacity. | Yes — bucket accumulates tokens during idle, allows short bursts | APIs that should allow occasional bursts, most common for public APIs

Leaky Bucket

Requests enter a queue (the bucket). Processed at a fixed outflow rate. If queue full, reject. | No — smooths all traffic to constant rate | Payment processing where strict steady rate is required, protecting downstream

Sliding Window Log

Track exact timestamps of all requests in a window. Count in window = current rate. | Accurate, no edge case at window boundaries | Most accurate, but high memory (store all timestamps)

Sliding Window Counter

Split window into smaller buckets (e.g., 60 × 1s buckets for a 60s window). Approximate count. | Good enough, low memory | Practical balance — used by most production rate limiters including Redis-based ones

Fixed Window Counter

Simple counter per time window (per minute, per hour). Resets at window boundary. | Boundary attack: 100 reqs at :59 + 100 at :01 = 200 in 2 seconds | Simple cases where boundary edge case is acceptable

// ── Token Bucket — in-memory (single instance) ──
public class TokenBucketRateLimiter {
    private final long   capacity;         // max tokens
    private final double refillRate;        // tokens per millisecond
    private       double tokens;
    private       long   lastRefillTime;
    public TokenBucketRateLimiter(long capacity, long refillPerSecond) {
        this.capacity      = capacity;
        this.refillRate    = (double) refillPerSecond / 1000.0;
        this.tokens        = capacity;
        this.lastRefillTime = System.currentTimeMillis();
    }
    public synchronized boolean tryConsume() {
        refill();
        if (tokens >= 1) { tokens--; return true; }
        return false;
    }
    private void refill() {
        long  now     = System.currentTimeMillis();
        double earned = (now - lastRefillTime) * refillRate;
        tokens         = Math.min(capacity, tokens + earned);
        lastRefillTime = now;
    }
}
// ── Sliding Window with Redis Sorted Set (distributed) ──
// Works across multiple instances — the rate limiter state lives in Redis
@Component
public class RedisRateLimiter {
    private final RedisTemplate<String, String> redis;
    public boolean isAllowed(String userId, int maxRequests, Duration window) {
        String key    = "ratelimit:" + userId;
        long   now    = System.currentTimeMillis();
        long   cutoff = now - window.toMillis();
        // Use a Lua script for atomicity across these three operations
        List<Object> results = redis.executePipelined(conn -> {
            conn.zRemRangeByScore(key.getBytes(), 0, cutoff);  // remove stale
            conn.zCard(key.getBytes());                         // count current
            conn.zAdd(key.getBytes(), now, (now+"").getBytes()); // add this req
            conn.expire(key.getBytes(), (int) window.toSeconds()); // set TTL
            return null;
        });
        Long currentCount = (Long) results.get(1); // count before adding this req
        return currentCount == null || currentCount < maxRequests;
    }
}

Q5: What is the difference between the token bucket and leaky bucket algorithms? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Token bucket allows bursts — the bucket accumulates tokens during quiet periods and a burst of requests can consume them all at once, as long as the bucket has enough. This is good for APIs where you want to allow short bursts without penalising the user for being bursty. Leaky bucket smooths traffic to a constant output rate regardless of how bursty the input is — it queues incoming requests and drains them at a fixed pace. This is better for protecting a downstream service that cannot handle load spikes. For a public REST API I would use token bucket — it gives a better user experience because you can do up to your burst limit instantly. For an internal payment processor that calls a third-party gateway with a strict SLA, leaky bucket makes more sense — I do not want to slam it with a burst even if my own limiter would technically allow it.

Section 6 — Idempotency and Back-Pressure

Idempotency in Distributed Systems

Why it matters:

In a distributed system, any request can fail ambiguously. The network times out — did the server receive it? Did it process it? Did the response get lost? The safe response is to retry. But if the operation is not idempotent, retrying causes duplicates — double charges, double inventory decrements, double emails. Idempotency means: calling the same operation N times has the same effect as calling it once.

// ── Idempotency Key Pattern ──
// Client generates a unique idempotency key per logical operation.
// Server stores the result of the first execution keyed by that ID.
// On retry: detect the key, return the stored result — do not re-execute.
@RestController
@RequestMapping("/payments")
public class PaymentController {
    @PostMapping
    public ResponseEntity<PaymentResult> createPayment(
            @RequestHeader("Idempotency-Key") String idempotencyKey,
            @RequestBody PaymentRequest req) {
        return paymentService.createIdempotent(idempotencyKey, req);
    }
}
@Service
public class PaymentService {
    private final IdempotencyStore      store;  // Redis or DB-backed
    private final PaymentGateway         gateway;
    @Transactional
    public ResponseEntity<PaymentResult> createIdempotent(
            String key, PaymentRequest req) {
        // 1. Check if we have already processed this key
        Optional<PaymentResult> existing = store.find(key);
        if (existing.isPresent()) {
            return ResponseEntity.ok(existing.get()); // Return cached result
        }
        // 2. Lock the key to prevent concurrent duplicate execution
        if (!store.tryLock(key, Duration.ofSeconds(30))) {
            throw new ConcurrentRequestException("Duplicate request in progress");
        }
        try {
            // 3. Process and store result atomically
            PaymentResult result = gateway.charge(req);
            store.save(key, result, Duration.ofDays(7)); // keep for 7 days
            return ResponseEntity.status(HttpStatus.CREATED).body(result);
        } finally {
            store.unlock(key);
        }
    }
}
// ── Idempotent Kafka Consumer ──
// Messages can be delivered more than once (at-least-once semantics).
// The consumer must be idempotent.

}

Q6: How would you prevent your payment service from overwhelming a third-party gateway? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

Three complementary approaches. First, a rate limiter in front of outbound gateway calls — token bucket with the gateway's stated rate limit as the refill rate. Second, Resilience4j circuit breaker on the gateway client: if more than 50% of calls fail in a 10-second window, open the circuit for 30 seconds and return a fast failure. This prevents cascading timeouts from exhausting thread pools. Third, an outbound queue: payment requests go into a bounded blocking queue, a fixed thread pool drains it at a controlled pace, and the queue length is your back-pressure signal. If the queue fills, HTTP 429 back to the caller. I have used this pattern in the Deloitte JP Morgan project — we had a third-party compliance service with a strict rate limit and this combination kept us well within it while still processing all requests.

Q7: Walk me through how you would design a URL shortener for 100k reads per second. [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

I would start by separating reads and writes — reads are 95% of traffic, writes are 5%. The write path: accept a long URL, generate a 6-7 character base-62 ID (62^7 = 3.5 trillion unique IDs, enough for a long time), store in a single PostgreSQL table with the ID as primary key, return the short URL. This is a simple key-value write, one DB call. The read path is where the work is. For 100k RPS, hitting the database directly would require a very large database cluster. Instead: put a Redis cache in front with the short code as key and the long URL as value. Cache hit ratio for URL shorteners is typically very high because popular links get clicked many times — a small cache covers most traffic. On cache miss, load from DB and cache with a TTL of 24 hours. For the infrastructure: stateless Spring Boot services behind an ALB, EC2 Auto Scaling based on CPU, Redis Cluster for the cache. The HTTP response is a 301 (permanent redirect, browser caches it) or 302 (temporary redirect, browser always asks). I would use 302 if I want accurate click analytics — 301 means the browser never calls our service again for that URL.
