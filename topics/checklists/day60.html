Master Reference — One-Liner Per Topic

Pre-interview read:

Read this table once the morning of the interview. Do not try to re-read any detailed content. One clear concept per topic is the goal.

Topic

The most important single thing to remember

HashMap

(n-1)&hash bucket index, treeify at 8, ConcurrentHashMap reads are lock-free volatile.

@Transactional

Four traps: self-invocation, private/final, checked exceptions, REQUIRES_NEW inside test.

Spring AOP

CGLIB subclasses the bean. Proxy intercepts public methods. Final methods are not proxied.

Auto-configuration

@ConditionalOnMissingBean is the override — your bean always wins over the default.

Bean lifecycle

BPP.postProcessAfterInitialization returns a replacement — this is how proxies are created.

Kafka acks

acks=all + min.insync.replicas=2 = no data loss on single broker failure.

Kafka rebalance

Outbox pattern

Write event in same DB transaction. Relay publishes at-least-once. Consumers must be idempotent.

BOLA

Always scope queries to authenticated user. @PostAuthorize ownership check. Test: user A cannot access user B's data.

JWT validation

exp + iss + aud + alg (never accept none). RS256: services hold public key only.

OAuth2 PKCE

code_verifier prevents interception. Use for all clients, not just SPAs.

400/422/409

Malformed / business rule fail / state conflict. Never 200 with error body.

Cursor pagination

WHERE id > last — index. OFFSET N — scans rows. Cursor is stable on inserts.

CAP theorem

P is mandatory. Real choice: C (error on partition) vs A (stale data on partition).

Consistent hashing

Add node: 1/N keys move. vs hash%N: nearly all keys move.

Saga orchestration

Use when > 3 steps, complex compensation, or you need to know current state at any time.

CQRS

Write model enforces invariants. Read model is denormalised. Projection keeps them in sync.

Liveness vs readiness

JVM in containers

UseContainerSupport + MaxRAMPercentage=75. 75% leaves room for metaspace, threads, direct memory.

Terraform state

S3 + DynamoDB. Never edit manually. terraform state mv for refactoring without destroying resources.

for_each vs count

for_each keyed by name — removing one key removes one resource. count keyed by index — removal shifts all indices.

GC choice

G1GC default for most. ZGC for sub-ms latency SLAs. Set Xms=Xmx to avoid resizing pauses.

N+1 fix

JOIN FETCH or @EntityGraph or batch_fetch_size. Enable SQL logging in dev to count queries.

CLOSED → OPEN at failure threshold → HALF_OPEN after wait → CLOSED if test calls pass.

Interview Day — What to Do

Phase

What to do

Morning before

Read the Master Reference table once. Do not study anything new. Review your career narrative. Eat a proper meal.

Before the interview

Arrive or log on 5 minutes early. Have water. Have a notepad for system design sketching.

When you receive a technical question

Pause for 2-3 seconds before answering. Say the pattern name or category first. Then explain.

When you receive a coding problem

State your approach before writing code. Name the pattern, state the time/space complexity. Then implement.

When you do not know the answer

Say: 'I have not worked with that specific scenario, but based on what I know about X, I would approach it by...' Never silence, never guess randomly.

For system design questions

Ask 2-3 clarifying questions first. Estimate scale. Draw the happy path. Then add failure handling, caching, and scale.

After each answer

Check: did I state the trade-off? Did I connect it to real experience? Add it if not.

At the end — your questions

Ask the questions from Day 56. Listen carefully to the answers — they tell you about the team culture.

How to Think Out Loud

The habit that gets offers:

Thinking out loud is not narrating every keystroke. It is telling the interviewer where you are in your reasoning before you commit to an approach. 'This looks like a sliding window problem because we need the longest subarray with a constraint' — that single sentence shows more than 10 lines of code.

Situation

What to say

Recognising a DSA pattern

'This looks like a [pattern] problem because [signal in the problem]. My approach is...'

Choosing between two designs

'I see two options here. Option A gives us [benefit] but costs [trade-off]. Option B is simpler but [limitation]. Given the scale we discussed, I would go with...'

Hitting a bug while coding

'I see the issue — I'm not handling [edge case]. Let me fix that here...'

Genuinely stuck

'I am working through this. The constraint I am struggling with is [specific thing]. Let me think about [related concept]...'

'That is my solution. Let me walk through edge cases: empty input, single element, maximum size, null values...'

Evaluating and Negotiating an Offer

Factor

What to consider

Compensation

Base salary, variable pay, equity (vesting schedule, cliff), joining bonus. Negotiate the total package, not just base.

Technical growth

Will you work on problems that make you better? Is the codebase a place you can learn from?

Team culture

Oncall burden, postmortem culture, how decisions are made. Ask the Day 56 questions seriously.

Career trajectory

Does this role open doors to what you want in 3-5 years? Senior to what? In what domain?

Negotiating

Always negotiate. Counter with a specific number based on market data (LinkedIn Salary, Glassdoor, levels.fyi). Reason: 'Based on my research and the scope of this role, I was expecting closer to X'. Be polite but direct.

Timeline

You can ask for 1-2 weeks to decide. Reasonable companies respect this. Pressure to decide in 24 hours is a signal.
