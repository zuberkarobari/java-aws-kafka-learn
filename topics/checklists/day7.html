The Architecture That Makes Kafka Fast

Kafka is not a message queue — it is a distributed commit log. That distinction matters:

Traditional message queues (RabbitMQ, ActiveMQ): messages consumed and deleted. Each message delivered to one consumer. Kafka: messages written to an append-only log and retained for a configurable period. MULTIPLE independent consumer groups can read the same message — each at their own pace, each with their own offset. A new consumer group can replay the entire history from offset 0. In your VISA payment system: the same transaction event could be consumed by the fraud detection service, the analytics service, and the notification service — each independently, each at their own speed, none affecting the others.

// Kafka Architecture — the key relationships
// Cluster:    multiple Brokers (e.g., 3 brokers for HA)
// Topic:      logical category for messages (e.g., 'payment.transactions')
// Partition:  ordered, immutable sequence of messages within a topic
//             each partition is stored on ONE broker (leader) + replicated to others
// Offset:     sequential integer per partition — uniquely identifies each message
//             consumers track their position via offsets
// Topic with 3 partitions across 3 brokers (replication factor = 2)
// Broker 1: Partition 0 (Leader), Partition 2 (Follower)
// Broker 2: Partition 1 (Leader), Partition 0 (Follower)
// Broker 3: Partition 2 (Leader), Partition 1 (Follower)
// If Broker 1 fails: Kafka elects a new leader for Partition 0 from ISR
// ISR (In-Sync Replicas): replicas that are fully caught up with the leader
// A replica is in ISR if it has replicated all messages within replica.lag.time.max.ms
// acks=all means: producer waits for ALL ISR replicas to acknowledge the write
// If ISR only contains the leader: acks=all = same as acks=1 (single point of failure)
// min.insync.replicas=2: require at least 2 ISR before accepting writes
// Together: acks=all + min.insync.replicas=2 = true durability
// Partition key determines message routing
// No key: round-robin across partitions
// With key: hash(key) % numPartitions → always same partition for same key
// Why it matters: messages with the same key land in the same partition
//                 = ordering guaranteed per key
// VISA payment: use accountId as key → all events for one account are ordered
// Log compaction — keep only LATEST value per key
// Regular topic: all messages retained until retention period expires
// Compacted topic: for each key, only the most recent message is kept
// Use case: user profile changelog, configuration state, CDC (Change Data Capture)
// Delete a key: publish a tombstone message (key + null value)

Interview Questions — Topic 1

Q2: What is the difference between a Kafka topic, partition, and offset? [Easy] ▼
💡 Ideal Answer (Amazon/GS Standard)

Topic: logical category/channel for messages (e.g., payment.transactions). Partition: an ordered, immutable, append-only log — a topic is split into N partitions stored across brokers. Each partition's messages are strictly ordered and each has a sequential offset. Offset: monotonically increasing integer per partition — uniquely identifies each message. Consumer groups track their position via (topic, partition, offset) tuples. Key insight: offsets are per-partition, not per-topic — ordering is guaranteed within a partition, not across partitions.

Q3: What is the ISR (In-Sync Replicas) and why does it matter for durability? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

ISR: the set of partition replicas fully caught up with the leader (within replica.lag.time.max.ms). For a write to be durable: acks=all means producer waits for all ISR replicas to acknowledge. min.insync.replicas=2 ensures at least 2 replicas (including leader) must be in ISR before accepting writes. If only the leader is in ISR and it crashes before replication: data is lost. Together: acks=all + min.insync.replicas=2 + replication.factor=3 provides true durability — tolerates 1 broker failure without data loss.

Q4: Why does the partition key matter for ordering guarantees? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Kafka only guarantees ordering WITHIN a partition. With no key: messages round-robin across partitions — no ordering for related messages. With a key: hash(key) % numPartitions always routes to the same partition — all messages with that key are ordered. In a payment system: use accountId as key — all transactions for account 'ACC-001' are in the same partition, in the order they were produced. Without this: event for 'payment reversed' could arrive before 'payment initiated' — catastrophic for financial processing.

Producer acks — The Most Important Config for Durability

acks controls the trade-off between write latency and durability — get this wrong and you lose data or pay unnecessary latency:

acks=0 (fire-and-forget): producer sends and moves on immediately. No acknowledgment. Highest throughput, but if the broker crashes after receiving the message but before writing it: message lost. Use for: metrics, logs where occasional loss is acceptable. acks=1 (leader ack): producer waits for the leader to write to its log. If leader crashes AFTER acking but BEFORE follower replication: message lost. Use for: moderate durability needs. acks=all (or -1): producer waits for ALL in-sync replicas to write. No data loss as long as min.insync.replicas are alive. Use for: financial transactions, anything where data loss is unacceptable. Your VISA payment system must use acks=all.

// Spring Boot Kafka Producer configuration
// application.yml
// spring:
//   kafka:
//     bootstrap-servers: localhost:9092
//     producer:
//       key-serializer: org.apache.kafka.common.serialization.StringSerializer
//       value-serializer: org.springframework.kafka.support.serializer.JsonSerializer
//       acks: all                    # wait for all ISR replicas
//       retries: 10                  # retry on transient failures
//       enable-idempotence: true     # exactly-once at producer level
//       batch-size: 32768            # 32KB batch — buffer before sending
//       linger-ms: 5                 # wait 5ms for more messages to batch
//       compression-type: snappy     # fast compression, good ratio
//       max-in-flight-requests-per-connection: 5  # required for idempotence
@Service
public class PaymentEventProducer {
    private final KafkaTemplate<String, PaymentEvent> kafkaTemplate;
    // Send with explicit key — ensures ordering per account
    public void publishPaymentProcessed(PaymentEvent event) {
        ProducerRecord<String, PaymentEvent> record = new ProducerRecord<>(
            "payment.transactions",
            event.getAccountId(),  // KEY: routes to same partition for same account
            event
        );
        // Add tracing headers for distributed tracing across services
        record.headers().add("X-Correlation-Id",
            event.getTransactionId().getBytes(StandardCharsets.UTF_8));
        CompletableFuture<SendResult<String, PaymentEvent>> future =
            kafkaTemplate.send(record);
        future.whenComplete((result, ex) -> {
            if (ex != null) {
                log.error("Failed to publish payment event {}: {}",
                    event.getTransactionId(), ex.getMessage());
                // Alert, retry, or use Outbox Pattern fallback
            } else {
                log.debug("Published to partition {} offset {}",
                    result.getRecordMetadata().partition(),
                    result.getRecordMetadata().offset());
            }
        });
    }
}
// Idempotent producer internals:
// enable.idempotence=true assigns each producer a unique PID (Producer ID)
// Each message gets a sequence number per (PID, partition)
// Broker detects and deduplicates messages with the same (PID, partition, seqNum)
// On retry: producer resends with same sequence number → broker discards duplicate
// Guarantee: exactly-once writes to a single partition within a single session

Interview Questions — Topic 2

Q5: What is the difference between acks=1 and acks=all in a Kafka producer? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

acks=1: producer waits for only the LEADER to write and acknowledge. If leader crashes after acking but before replication to followers: data lost (followers may not have the message). acks=all (or -1): producer waits for all in-sync replicas (ISR) to acknowledge. Survives leader failure as long as at least one ISR replica has the data. Always combine with min.insync.replicas=2 (require at least 2 ISR). Latency: acks=1 faster, acks=all ~2-3x slower. For financial transactions: always acks=all. For metrics/logs: acks=1 or acks=0 acceptable.

Q6: What does enable.idempotence=true do in a Kafka producer? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

Assigns the producer a unique PID (Producer ID) per broker session. Each message gets a monotonically increasing sequence number per (PID, partition). On network failure + retry: producer resends with the same sequence number. Broker detects the duplicate (already saw this seqNum for this PID/partition) and discards it — no duplicate in the log. Automatically sets acks=all, retries=Integer.MAX_VALUE, max.in.flight.requests=5. Guarantees exactly-once writes to a single partition within one producer session. Limitation: does NOT survive producer restart (new session = new PID). For cross-session exactly-once: use Kafka transactions.

Q7: What is the trade-off between linger.ms and latency? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

linger.ms=0 (default): producer sends messages immediately — lowest latency, small batches (often 1 message). linger.ms=5: producer waits up to 5ms for more messages to fill the batch — higher throughput (fewer, larger network requests), slightly higher latency. In high-throughput systems (your VISA payment processing): linger.ms=5-20 significantly improves throughput with minimal latency impact. batch.size (default 16KB) also caps batch size. A batch is sent when EITHER batch.size is full OR linger.ms expires — whichever comes first.

Consumer Groups — Kafka's Scalability Model

The most important rule in Kafka consumer design:

Within a consumer group, each partition is assigned to AT MOST ONE consumer. This means: with 6 partitions and 3 consumers, each consumer handles 2 partitions — perfect parallelism. With 6 partitions and 6 consumers: each consumer handles exactly 1 partition — maximum parallelism. With 6 partitions and 7 consumers: one consumer is IDLE — you cannot get more parallelism than you have partitions. This is why partition count is the maximum parallelism ceiling. If you need to scale your VISA payment processor to 20 consumers, you need at least 20 partitions. You cannot add partitions to an existing topic without re-keying data — plan partition count upfront.

// Consumer group parallelism model
// Topic: payment.transactions — 6 partitions
// Consumer Group: payment-processor
// 3 consumers → each gets 2 partitions
// Consumer A: Partition 0, Partition 1
// Consumer B: Partition 2, Partition 3
// Consumer C: Partition 4, Partition 5
// Scale to 6 consumers → each gets 1 partition (max parallelism)
// Scale to 7 consumers → 7th consumer is idle (no partition left)
// DIFFERENT consumer groups can ALL read the same partitions independently
// payment.transactions consumed by:
//   payment-processor group  (real-time fraud check)
//   analytics-pipeline group (batch reporting — can be hours behind)
//   audit-logger group       (compliance — must not miss any message)
// Each group tracks its own offsets independently
// ── OFFSET COMMIT STRATEGIES ──
// ❌ Auto-commit (default, enable.auto.commit=true) — DANGEROUS for payment systems
// Commits offset every auto.commit.interval.ms (default 5000ms)
// Problem: consumer processes message → offsets auto-committed → consumer crashes
//          → messages between last commit and crash are LOST (committed but failed to process)
// OR:      consumer crashes after poll() but before processing
//          → offsets NOT committed → messages redelivered → DUPLICATES
// NEVER use auto-commit for financial transaction processing
// ✅ Manual synchronous commit — process THEN commit
@KafkaListener(topics = "payment.transactions", groupId = "payment-processor")
public void processPayment(ConsumerRecord<String, PaymentEvent> record,
                           Acknowledgment ack) {
    try {
        paymentProcessor.process(record.value()); // process first
        ack.acknowledge();                         // commit AFTER success
        // If crash BEFORE ack: message redelivered — consumer must be idempotent
    } catch (Exception e) {
        log.error("Processing failed for offset {}: {}",
            record.offset(), e.getMessage());
        // Do NOT ack — message will be redelivered
        // After max retries: route to Dead Letter Topic
    }
}
// application.yml for manual ack mode
// spring.kafka.listener.ack-mode: manual_immediate
// spring.kafka.consumer.enable-auto-commit: false
// Consumer poll loop timing — critical to understand
// max.poll.interval.ms (default 5min): max time between poll() calls
//   If consumer doesn't poll within this window → considered dead → rebalance
//   Long-running processing? Increase this OR process async
// session.timeout.ms (default 45s): broker considers consumer dead if no heartbeat
// heartbeat.interval.ms (default 3s): how often consumer sends heartbeat
// Rule: session.timeout.ms > 3 * heartbeat.interval.ms

Rebalancing — The Painful but Necessary Disruption

Rebalancing stops all consumption — understand what triggers it:

Eager rebalancing (default before Kafka 2.4): when rebalance triggers, ALL consumers stop, ALL partition assignments revoked, then redistributed. During rebalance: zero throughput, offsets not committed, messages not processed. Triggers: consumer joins or leaves the group, broker failure, partition count change, session timeout exceeded. Cooperative incremental rebalancing (Kafka 2.4+, CooperativeStickyAssignor): only affected partitions are revoked and reassigned — other consumers keep their partitions and continue processing. Massive improvement for large consumer groups.

// Consumer configuration for minimal rebalance impact
// spring.kafka.consumer:
//   partition-assignment-strategy:
//     - org.apache.kafka.clients.consumer.CooperativeStickyAssignor
//   # Sticky: try to reassign same partitions to same consumers after rebalance
//   # Cooperative: only revoke partitions that need to move, not all
// Seek for replay — re-process messages from a specific point
@Autowired
private KafkaListenerEndpointRegistry endpointRegistry;
public void replayFromBeginning(String topic, int partition) {
    kafkaTemplate.getConsumerFactory().createConsumer("replay-group", null)
        .seekToBeginning(Collections.singleton(new TopicPartition(topic, partition)));
}

Interview Questions — Topic 3

Q8: What happens if you have more consumers in a group than partitions? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

The excess consumers are idle — they receive no partition assignments and process no messages. Kafka enforces that each partition is assigned to at most one consumer in a group. With 6 partitions and 8 consumers: 6 consumers are active (one partition each), 2 are idle standby. The idle consumers do serve as automatic failover — if an active consumer crashes, a partition is reassigned to an idle consumer during the next rebalance. Bottom line: partition count is your maximum parallelism ceiling. Plan partitions at topic creation time — changing partition count on an existing topic breaks key-based ordering.

Q9: What is the danger of auto-commit in a Kafka consumer? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

enable.auto.commit=true commits offsets on a timer (default every 5 seconds), regardless of processing status. Two failure modes: 1) Lost messages: consumer polls batch, auto-commit fires, consumer crashes before processing completes — committed offsets indicate messages were 'processed' but they were not. 2) Duplicates: less common but if crash happens before auto-commit fires — messages redelivered. For financial systems: disable auto-commit, use manual acknowledgment (AckMode.MANUAL_IMMEDIATE in Spring Kafka), call ack.acknowledge() only after successful processing. Consumers must be idempotent to handle the at-least-once redelivery.

Q10: What triggers a consumer group rebalance and what is its impact? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

Triggers: consumer joins or leaves the group, consumer crashes (session.timeout.ms exceeded without heartbeat), consumer stops polling (max.poll.interval.ms exceeded — long processing), broker failure, topic partition count change. Impact with eager rebalancing: ALL consumers stop, ALL partition assignments revoked, zero throughput during rebalance. With CooperativeStickyAssignor (Kafka 2.4+): only affected partitions revoked, unaffected consumers continue — dramatically reduces impact. Best practice: process messages quickly (or async), tune session.timeout.ms and max.poll.interval.ms appropriately, use cooperative rebalancing.

The Three Delivery Guarantees — Explained With Real Consequences

Guarantee

How to Achieve It | Production Consequence

At-most-once

Producer: acks=0 or acks=1 with retries=0. Consumer: commit offset BEFORE processing. Message may be lost if processing fails after commit. Use for: metrics, logs, analytics where occasional loss is acceptable. NEVER for payments.

At-least-once

Exactly-once

Producer: enable.idempotence=true + Kafka transactions. Consumer: read-process-write within a transaction (consume → process → produce to output topic → commit all atomically). Highest overhead. Use for: financial processing where duplicate charges are catastrophic.

Exactly-once does NOT mean the business logic runs exactly once — it means the Kafka records are produced exactly once:

Exactly-once semantics (EOS) in Kafka means: within a Kafka transaction, the consumer offset commit and the producer's output message are committed atomically. Either both happen or neither happens. The business logic inside your consumer CAN still run multiple times (e.g., if the consumer crashes after processing but before the transaction commits). Your business logic must be designed to be idempotent OR wrapped in a separate idempotency check (deduplication table). In the VISA payment system: we stored transaction IDs in a deduplication table — if we saw the same transaction ID twice, we skipped it. That is the at-least-once + idempotent consumer pattern — simpler and more reliable than full EOS for most payment use cases.

// Kafka Transactions — exactly-once producer
// Producer config:
// enable.idempotence: true
// transactional.id: payment-producer-1  # unique per producer instance
// Spring Kafka exactly-once configuration
// spring.kafka.producer.transaction-id-prefix: payment-tx-
@Service
public class ExactlyOncePaymentProcessor {
    private final KafkaTemplate<String, Object> transactionalTemplate;
    // Consume from input topic, process, produce to output topic — atomically
    @Transactional  // Spring Kafka + @Transactional = EOS
    @KafkaListener(topics = "payment.raw", groupId = "payment-eos-processor")
    public void processWithEOS(ConsumerRecord<String, RawPayment> record) {
        // All within Kafka transaction:
        // 1. Process the payment
        ProcessedPayment result = paymentEngine.process(record.value());
        // 2. Produce output message (within same transaction)
        transactionalTemplate.send("payment.processed", result.getAccountId(), result);
        // 3. Produce audit event (within same transaction)
        transactionalTemplate.send("payment.audit", result.getTransactionId(), new AuditEvent(result));
        // When method returns: Spring Kafka commits:
        // - Consumer offset for 'payment.raw'
        // - Both produced messages to 'payment.processed' and 'payment.audit'
        // All atomically. On crash: all three operations roll back.
    }
}
// ── Simpler alternative: at-least-once + idempotent consumer ──
// More practical for most payment systems
@KafkaListener(topics = "payment.transactions", groupId = "payment-processor")
public void processWithIdempotency(ConsumerRecord<String, PaymentEvent> record,
                                    Acknowledgment ack) {
    String txId = record.value().getTransactionId();
    // Check deduplication table FIRST
    if (processedTxRepo.existsByTransactionId(txId)) {
        log.warn("Duplicate transaction detected: {}, skipping", txId);
        ack.acknowledge(); // commit offset — don't reprocess
        return;
    }
    // Process and mark as processed atomically
    paymentService.processAndRecord(record.value()); // saves result + txId in one TX
    ack.acknowledge(); // commit Kafka offset after successful processing
}

Interview Questions — Topic 4

Q11: What is the difference between at-least-once and exactly-once delivery in Kafka? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

At-least-once: acks=all + retries + manual offset commit after processing. Message delivered at least once — duplicates possible on retry or consumer restart. Consumers must be idempotent. Most practical for payment systems with a deduplication table. Exactly-once: Kafka transactions (transactional.id + enable.idempotence). Consumer offset commit and producer output are committed atomically — no duplicates in Kafka records. Higher latency and complexity. Note: EOS applies to Kafka record delivery, not business logic execution — your processing code can still run twice; EOS prevents duplicate Kafka output records.

Q12: How do you design an idempotent Kafka consumer for a payment system? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

Store a processed transaction ID in a deduplication table (Redis or DB) when a message is processed. Before processing: check if transactionId already exists. If yes: skip (already processed), commit offset. If no: process the payment and insert transactionId into the deduplication table within the SAME database transaction. Only commit the Kafka offset after the DB transaction commits. This provides at-least-once delivery + idempotent business logic = effectively exactly-once business outcomes. TTL the deduplication records (keep for 7 days) to bound storage growth.

🔴 Topic 5: Spring Kafka — @KafkaListener, Error Handling, Dead Letter Topics

Checklist

  • Error handling: DefaultErrorHandler, BackOff strategies, SeekToCurrentErrorHandler
  • Dead Letter Topic (DLT): DeadLetterPublishingRecoverer — what goes there and why
  • @RetryableTopic — automatic retry with DLT, backoff, non-retryable exceptions

A message that cannot be processed must go SOMEWHERE — otherwise it is silently lost or blocks the entire partition:

Without a DLT strategy: consumer throws exception → message retried forever → partition blocked (no other messages in this partition processed). Or: exception swallowed → message lost. Dead Letter Topic pattern: after N retries, route the failed message to a separate DLT topic (e.g., payment.transactions.DLT). DLT messages retain the original payload, partition, offset, and exception details as Kafka headers. A separate DLT consumer monitors the DLT, alerts on-call engineers, persists failures for analysis, and retries after manual investigation. In your VISA payment system, any failed transaction that cannot be retried must land in a DLT — missing a payment is a regulatory issue.

// @RetryableTopic — the easiest way to add retry + DLT in Spring Kafka 2.7+
@Component
public class PaymentEventConsumer {
    @RetryableTopic(
        attempts = "4",         // 1 original + 3 retries
        backoff = @Backoff(
            delay = 1000,        // 1s initial delay
            multiplier = 2.0,    // exponential: 1s, 2s, 4s
            maxDelay = 10000     // cap at 10s
        ),
        dltTopicSuffix = ".DLT",         // payment.transactions.DLT
        autoCreateTopics = "false",        // create DLT topic manually in prod
        exclude = { BusinessValidationException.class }  // don't retry business errors
    )
    @KafkaListener(topics = "payment.transactions", groupId = "payment-processor")
    public void consume(PaymentEvent event) {
        paymentService.process(event);
        // On exception: Spring retries with exponential backoff
        // After all retries exhausted: routes to payment.transactions.DLT
    }
    // DLT handler — receives failed messages
    @DltHandler
    public void handleDlt(PaymentEvent event,
                          @Header(KafkaHeaders.RECEIVED_TOPIC) String topic,
                          @Header(KafkaHeaders.EXCEPTION_MESSAGE) String exceptionMsg) {
        log.error("Message sent to DLT from topic {}: event={}, error={}",
            topic, event.getTransactionId(), exceptionMsg);
        // Save to DB for investigation, alert on-call, trigger incident
        dltAlertService.alert(event, exceptionMsg);
    }
}
// Manual ErrorHandler configuration (Spring Kafka < 2.7)
@Bean
public DefaultErrorHandler errorHandler(
        DeadLetterPublishingRecoverer recoverer) {
    FixedBackOff backOff = new FixedBackOff(1000L, 3); // 3 retries, 1s between
    DefaultErrorHandler handler = new DefaultErrorHandler(recoverer, backOff);
    // Don't retry these — send directly to DLT
    handler.addNotRetryableExceptions(
        BusinessValidationException.class,
        SerializationException.class
    );
    return handler;
}
@Bean
public DeadLetterPublishingRecoverer recoverer(KafkaTemplate<?, ?> template) {
    // DLT topic name: original topic + '.DLT'
    // DLT partition: same as source partition (for ordering)
    return new DeadLetterPublishingRecoverer(template,
        (record, ex) -> new TopicPartition(record.topic() + ".DLT", record.partition()));
}
// Batch listener — consume multiple records at once
@KafkaListener(topics = "payment.bulk", groupId = "bulk-processor",
               containerFactory = "batchListenerContainerFactory")
public void processBatch(List<ConsumerRecord<String, PaymentEvent>> records,
                         Acknowledgment ack) {
    log.info("Processing batch of {} records", records.size());
    paymentService.processBatch(records.stream()
        .map(ConsumerRecord::value).collect(Collectors.toList()));
    ack.acknowledge(); // commit after entire batch processed
}

Interview Questions — Topic 5

Q13: What is a Dead Letter Topic and when should a message be sent to it? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

A Dead Letter Topic (DLT) is a separate Kafka topic that receives messages which cannot be processed after exhausting all retry attempts. Send to DLT when: business logic throws a non-transient exception (business rule violation — retrying will always fail), max retry attempts exhausted for transient errors, deserialization fails (message is malformed). DLT preserves: original message payload, source topic/partition/offset, exception details as Kafka headers. A separate DLT consumer monitors and alerts. Never silently discard failed messages — in a payment system, a lost message is a missing transaction.

Q14: What is the difference between retryable and non-retryable exceptions in @RetryableTopic? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Retryable exceptions: transient failures expected to resolve with time — ConnectException, KafkaException, transient DB exceptions. These get exponential backoff retries before going to DLT. Non-retryable exceptions: failures that will ALWAYS fail regardless of retries — BusinessValidationException (invalid transaction data), DeserializationException (malformed message), ConstraintViolationException. These go DIRECTLY to DLT without retrying — retrying them wastes time and blocks partition progress. Configure with @RetryableTopic(exclude={BusinessValidationException.class}).

Kafka Streams — Building the Real-Time Fraud Detection Engine

How your VISA real-time payment system could use Kafka Streams for fraud detection:

Traditional approach: consume payment events, call fraud service via HTTP (synchronous, fragile). Kafka Streams approach: build a fraud detection topology that processes each payment event as it arrives, maintains running account velocity state (how much spent in last 5 minutes) in RocksDB, uses a windowed aggregation to detect spending spikes, and immediately routes suspicious transactions to a fraud.alerts topic — all within the Kafka Streams topology, with no external service calls, no shared state across instances, automatically partitioned and parallelized. This is exactly the kind of real-time, stateful, low-latency processing that payment companies need.

// Kafka Streams topology for payment fraud detection
@Configuration
public class FraudDetectionTopology {
    @Bean
    public KStream<String, PaymentEvent> paymentStream(StreamsBuilder builder) {
        // Source: consume from payment.transactions
        KStream<String, PaymentEvent> payments =
            builder.stream("payment.transactions");
        // ── Stateless operations: filter and transform ──
        KStream<String, PaymentEvent> cardPayments = payments
            .filter((accountId, event) -> event.getMethod() == PaymentMethod.CARD)
            .mapValues(event -> enrichWithRiskScore(event)); // add risk metadata
        // ── Windowed aggregation: total spend per account in 5-minute windows ──
        // Tumbling window: non-overlapping fixed windows
        KTable<Windowed<String>, Double> spendPerWindow = cardPayments
            .groupByKey()
            .windowedBy(TimeWindows.ofSizeWithNoGrace(Duration.ofMinutes(5)))
            .aggregate(
                () -> 0.0,                                        // initializer
                (accountId, event, total) -> total + event.getAmount(), // aggregator
                Materialized.<String, Double, WindowStore<Bytes, byte[]>>as("spend-windows")
                    .withValueSerde(Serdes.Double())              // stored in RocksDB
            );
        // ── Alert on velocity: > 5000 in 5 minutes ──
        spendPerWindow.toStream()
            .filter((window, total) -> total > 5000.0)
            .map((window, total) -> KeyValue.pair(
                window.key(),  // accountId
                new FraudAlert(window.key(), total, window.window().startTime())))
            .to("fraud.alerts"); // sink: write alerts to output topic
        // ── KTable: reference data join ──
        // Load account risk profiles as a KTable (compacted changelog topic)
        KTable<String, AccountProfile> profiles =
            builder.table("account.profiles",
                Materialized.as("account-profiles-store")); // queryable state store
        // Enrich payment stream with account profile data
        KStream<String, EnrichedPayment> enriched = cardPayments
            .join(profiles,
                (payment, profile) -> new EnrichedPayment(payment, profile),
                Joined.with(Serdes.String(), paymentSerde, profileSerde));
        enriched.to("payment.enriched");
        return payments;
    }
}
// Window types summary:
// Tumbling:  [0-5min] [5-10min] [10-15min] — fixed, non-overlapping
//            use for: 5-minute spend totals, hourly aggregates
// Hopping:   [0-10min] [5-15min] [10-20min] — fixed, overlapping (advance < size)
//            use for: sliding-window moving averages
// Sliding:   windows that start whenever a new event arrives
//            use for: 'fraud in any 5 minutes around this event'
// Session:   activity-based — gaps > inactivity gap close the window
//            use for: user session analytics (gap > 30 min = new session)

Interview Questions — Topic 6

Q15: What is the difference between KStream and KTable in Kafka Streams? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

KStream: unbounded sequence of events — each record is an independent event. map, filter, flatMap operate record-by-record. Represents event log (INSERT-only semantics). KTable: change log — for each key, only the LATEST value is materialized. New record with same key UPDATES the previous value. Represents current state (UPSERT semantics). Backed by RocksDB state store. Join: KStream-KTable join enriches each stream event with the latest table value — like a real-time lookup against a continuously updated reference table. Use case: enrich payment events with latest account risk profile from a KTable.

Q16: What is a tumbling window vs a hopping window in Kafka Streams? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

Tumbling window: fixed size, non-overlapping — each event belongs to exactly one window. Example: 5-minute windows [0-5], [5-10], [10-15]. Events at t=3min go in window [0-5], events at t=7min go in [5-10]. Use for: aggregate totals per period (hourly revenue, 5-minute fraud velocity). Hopping window: fixed size but overlapping — advance interval < window size. Example: size=10min, advance=5min: [0-10], [5-15], [10-20]. Each event can appear in multiple windows. Use for: moving averages, smoothed trend lines. Session window: activity-based — a gap of N minutes closes the window. Use for: user session analytics.

Consumer Lag — The One Metric That Tells You Everything

Consumer lag is the most important operational Kafka metric — it is the early warning of every production issue:

Lag = (latest offset in partition) - (consumer committed offset). Lag = 0: consumer is caught up in real time. Lag growing slowly: consumer throughput slightly below producer rate — optimize consumer or scale out. Lag growing rapidly: processing bottleneck, consumer crash, rebalancing storm, or downstream dependency (DB) is slow. Lag stuck at a fixed high number: consumer paused or crashed. In your VISA payment system: lag on the fraud detection consumer directly translates to delay between a fraudulent transaction happening and the fraud alert being raised. Lag > threshold = on-call alert.

Q17: How do you monitor Kafka consumer health in production? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Primary metric: consumer lag per (group, topic, partition) — lag growing means consumer is falling behind, lag = 0 means caught up. Q18: What is AWS MSK and how does it differ from self-managed Kafka? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Amazon Managed Streaming for Apache Kafka — fully managed Kafka on AWS. Handles: broker provisioning, OS patching, Kafka upgrades, storage scaling, monitoring (CloudWatch metrics), multi-AZ replication. Authentication: IAM-based auth (SASL/SCRAM + TLS), integrates with AWS IAM roles — no separate credential management. MSK Connect: managed Kafka Connect workers for connectors (Debezium, S3 Sink). Advantages over self-managed: no operational burden, automatic broker replacement on failure. Limitations: limited Kafka version choice, cannot access ZooKeeper directly, less flexibility in broker configuration. Your Deloitte AWS experience maps directly here.
