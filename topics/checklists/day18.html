Section 1 — HTTP Methods, Safety and Idempotency

The Two Properties That Drive Method Choice

Safe:

A method is safe if it does not modify server state. Safe methods can be cached, prefetched, and retried freely. GET, HEAD, and OPTIONS are safe. POST, PUT, PATCH, DELETE are not.

Idempotent:

A method is idempotent if calling it multiple times produces the same server state as calling it once. It does not mean the response is the same — a second DELETE on the same resource might return 404, but the state (resource does not exist) is the same. GET, HEAD, PUT, and DELETE are idempotent. POST and PATCH are not idempotent by default.

Method

Safe? | Idempotent? | Use for | Common mistake

GET

Yes | Yes | Read a resource or collection | Using GET for state-changing operations (URL links, webhooks)

POST

No | No | Create a resource, trigger an action | Using it when PUT is correct; not handling duplicate submissions

PUT

No | Yes | Full replace of a resource | Partial update — use PATCH instead; forgetting to send all fields

PATCH

No | No (by default) | Partial update of a resource | Treating it as idempotent when it is not (e.g., increment operations)

DELETE

No | Yes | Remove a resource | Returning 404 on second call and confusing clients — second DELETE should still return 200 or 204

HEAD

Yes | Yes | Get headers of a GET response without body | Rarely implemented — useful for checking resource existence cheaply

OPTIONS

Yes | Yes | Discover allowed methods, CORS preflight | Not implementing it — required for CORS

Making POST Idempotent with Idempotency Keys

Why this matters:

Creating a payment is a POST. Networks fail. Clients retry. Without an idempotency key, a retry creates a second payment. With one, the server detects the duplicate and returns the original result. Stripe, PayPal, and most financial APIs use this pattern.

// Client sends Idempotency-Key header on every POST that creates a resource
POST /api/payments
Idempotency-Key: a8098c1a-f86e-11da-bd1a-00112444be1e
Content-Type: application/json
{ "amount": 500.00, "currency": "INR", "accountId": "ACC-001" }
// Server implementation
@RestController
@RequestMapping("/api/payments")
public class PaymentController {
    private final PaymentService     service;
    private final IdempotencyStore   store;   // backed by Redis with TTL
    @PostMapping
    public ResponseEntity<PaymentResult> createPayment(
            @RequestHeader("Idempotency-Key") String idempotencyKey,
            @RequestBody @Valid PaymentRequest request) {
        // Check if we have already processed this key
        Optional<PaymentResult> cached = store.get(idempotencyKey);
        if (cached.isPresent()) {
            // Return the original result — do NOT process again
            return ResponseEntity.status(HttpStatus.CREATED).body(cached.get());
        }
        // Process and store atomically
        PaymentResult result = service.process(request);
        store.put(idempotencyKey, result, Duration.ofHours(24));
        return ResponseEntity.status(HttpStatus.CREATED)
            .header("Idempotency-Key", idempotencyKey)
            .body(result);
    }
}
Q1: What is the difference between safe and idempotent? Why does it matter when designing an API? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Safe means the method does not change server state — GET, HEAD, OPTIONS. Clients, proxies, and browsers can cache or prefetch safe methods freely. Idempotent means you can call it multiple times and end up in the same state — GET, PUT, DELETE. The practical value: idempotent methods are safe to retry automatically on network failures without side effects. When I designed our payment webhook endpoints at Deloitte, we made sure DELETE was truly idempotent — the second call on an already-deleted resource returns 200 with the last known state, not 404. A client that gets a network timeout on a DELETE does not know if the request succeeded, so it retries. If the retry gets a 404, the client may treat it as an error and raise an alert for something that already worked correctly. Idempotency at the API level is a contract with your clients.

Bad Gateway | Upstream service returned invalid response (your service is the gateway)

503 Service Unavailable

Unavailable | Server is up but temporarily unable to handle the request; include Retry-After

Gateway Timeout | Upstream service timed out — your service is the gateway

400 vs 422 vs 409 — The Common Confusion

Code

Situation | Example

400 Bad Request

Structure or syntax is wrong — the server could not even parse the intent | Missing required JSON field, malformed date string, invalid JSON

422 Unprocessable Entity

Structure is fine but business rules fail — server understood but cannot act | amount is -500, transfer to your own account, currency not supported

409 Conflict

Request is valid but conflicts with current server state | Creating a user with an email that already exists, version mismatch on update

Q2: When do you return 401 vs 403, and what is the practical difference? [Easy] ▼
💡 Ideal Answer (Amazon/GS Standard)

401 means authentication failed — the request had no credentials, or the token was expired, or the signature was wrong. The client should authenticate and try again. 403 means the client is authenticated — we know who they are — but they do not have permission to do what they asked. The client should not retry with the same credentials. The mistake I see often is returning 403 for everything auth-related because it feels safer. The problem is you lose the signal. A client getting 401 knows to refresh the token. A client getting 403 knows there is a permission issue and should escalate rather than loop. There is also a privacy nuance: some APIs return 404 instead of 403 for private resources to not reveal they exist. I have done this for admin endpoints — an unauthenticated request to /api/admin/users gets 404, not 403, so the endpoint structure is not exposed to API scanners.

Section 3 — Resource Design and URI Conventions

URI Design Rules

Rule

Good example | Bad example | Why

Nouns not verbs

/api/payments | /api/createPayment | The HTTP method IS the verb

Plural collections

/api/payments/TXN-123 | /api/payment/TXN-123 | Consistent — /payments is a collection, /payments/{id} is an item

Lowercase + hyphens

/api/payment-methods | /api/paymentMethods | Case-insensitive by RFC; hyphens are more readable than underscores

Hierarchy for relationships

/api/accounts/{id}/payments | /api/account-payments?accountId=x | Sub-resources read naturally

No trailing slash

/api/payments | /api/payments/ | Two URIs should not refer to the same resource

No file extensions

/api/payments/TXN-123 | /api/payments/TXN-123.json | Use Content-Type and Accept headers instead

Nested vs flat resource design:

POST /api/accounts/{accountId}/payments is clean for creation where the parent context is important. But deeply nested routes like /api/accounts/{accountId}/payments/{paymentId}/refunds/{refundId} become hard to work with. A practical rule: nest at most one level. For deep associations, use a top-level resource with a filter: GET /api/refunds?paymentId=TXN-123 is cleaner than /api/accounts/ACC-001/payments/TXN-123/refunds.

Versioning Strategies

Strategy

How it looks | Pros | Cons | Best for

URI path versioning

Request header

Accept: application/vnd.myapi.v2+json | Clean URIs; versioning is a negotiation detail | Invisible in browser, harder to test, harder to route in API Gateway | Mature APIs with sophisticated clients

Query parameter

/api/payments?version=2 | Easy to test in browser | Caching issues (version is a cache dimension); mixes metadata with resource params | Internal or developer-facing APIs

Subdomain

// URI versioning — the pragmatic choice for most teams
@RestController
@RequestMapping("/api/v1/payments")
public class PaymentControllerV1 { }
@RestController
@RequestMapping("/api/v2/payments")
public class PaymentControllerV2 {
    // V2 adds pagination, different response shape
}
// Header versioning — cleaner URIs but less obvious
@GetMapping("/api/payments/{id}")
public ResponseEntity<?> getPayment(
        @PathVariable String id,
        @RequestHeader(value="API-Version", defaultValue="1") int version) {
    if (version >= 2) {
        return ResponseEntity.ok(paymentService.getV2(id));
    }
    return ResponseEntity.ok(paymentService.getV1(id));
}
// Content negotiation versioning (most RESTful)
// Accept: application/vnd.payments.v2+json
@GetMapping(value="/api/payments/{id}",
            produces="application/vnd.payments.v2+json")
public PaymentResponseV2 getPaymentV2(@PathVariable String id) {
    return paymentService.getV2(id);
}
Q3: How do you approach API versioning and what strategy did you use in your projects? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

I have used URI path versioning in practice and would choose it again for most teams. The reason is operability — /api/v1/payments is visible in logs, easy to route in Spring Cloud Gateway with a simple path predicate, easy to test in a browser or curl, and immediately obvious to any developer reading the code. The argument against it — that it violates REST because version is not a resource property — is theoretically correct but practically less important than being easy to operate. I keep backward compatibility for at least one version. When we needed to change the payment response structure at Deloitte, we added /v2/payments with the new shape and kept /v1 working for existing consumers with a deprecation header. I remove old versions on a published schedule. The thing I avoid is flag-day breaking changes where all consumers have to migrate at once.

Section 4 — Pagination, Filtering and Sorting

Pagination Patterns

Pattern

How it works | Strengths | Weaknesses | Use when

Offset / Limit

GET /payments?offset=40&limit=20 | Simple, random access to any page | Expensive on large tables (OFFSET scans rows); page drift when new items inserted | Small to medium datasets, admin UIs that need page numbers

Page / Size

GET /payments?page=3&size=20 | Familiar to users; easy to build page controls | Same issues as offset — OFFSET = page * size under the hood | Admin UIs, reports

Cursor-based

GET /payments?after=cursor_xyz&limit=20 | Stable: new inserts do not shift pages; O(log n) with index | No random page access; cursor must be opaque to clients | Feeds, timelines, large datasets, infinite scroll

Keyset

GET /payments?after_id=TXN-500&after_date=2024-01 | Same stability as cursor; human-readable | Requires compound index on sort fields; tricky with ties | APIs with natural sort keys like date+id

// Offset pagination response — include total so client can build page controls
{
  "data": [...],
  "pagination": {
    "total": 1250,
    "page": 3,
    "size": 20,
    "totalPages": 63
  }
}
// Cursor pagination response — next/prev cursors are opaque base64 tokens
{
  "data": [...],
  "pagination": {
    "nextCursor": "eyJpZCI6IlROWC01MDAiLCJkYXRlIjoiMjAyNC0wMSJ9",
    "prevCursor": "eyJpZCI6IlROWC00ODAiLCJkYXRlIjoiMjAyNC0wMSJ9",
    "hasNext": true,
    "hasPrev": true
  }
}
// Spring Data cursor pagination with keyset
@GetMapping
public PagedResponse<PaymentResponse> list(
        @RequestParam(defaultValue = "20") @Max(100) int limit,
        @RequestParam(required = false) String after) {
    Cursor cursor = after != null ? Cursor.decode(after) : Cursor.start();
    List<Payment> page = repository.findAfterCursor(cursor, limit + 1);
    boolean hasNext = page.size() > limit;
    List<Payment> data = hasNext ? page.subList(0, limit) : page;
    String nextCursor = hasNext ? Cursor.encode(data.get(data.size()-1)) : null;
    return PagedResponse.of(data.stream().map(mapper::toResponse).toList(),
                            nextCursor, hasNext);
}
// Filtering and sorting — query parameter conventions
GET /api/payments?status=COMPLETED&currency=INR&createdAfter=2024-01-01
GET /api/payments?sort=createdAt,desc&sort=amount,asc   // multi-field sort
GET /api/payments?fields=id,amount,status               // sparse fieldset
Q4: What is cursor-based pagination and when would you use it over offset? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Cursor-based pagination uses an opaque token derived from the last item on the page to fetch the next set. Instead of OFFSET 100 LIMIT 20, which the database handles by scanning and discarding 100 rows, you do WHERE id > last_seen_id LIMIT 20, which hits the index directly. Two practical advantages over offset. First, it is stable — if 5 new payments are inserted while a user is on page 3, offset pagination would show the same records again on the next page load. Cursor pagination is not affected because it anchors to a specific record. Second, it performs better on large tables — deep offsets in PostgreSQL are slow because the database still scans all the rows it is skipping. The trade-off is you cannot jump to an arbitrary page, which means you cannot build page number controls. For a customer-facing transaction feed, I use cursor pagination. For an admin dashboard where someone needs to jump to page 47 of results, offset is fine.

Section 5 — API Error Response Design

RFC 7807 Problem Details — The Standard

Why a standard format matters:

Without a consistent error format, every endpoint in your API returns errors differently. Clients have to handle each one specially. RFC 7807 Problem Details defines a standard JSON structure for HTTP error responses. Spring Boot 3 implements it natively via ProblemDetail.

// RFC 7807 Problem Details structure
{
  "type":     "https://api.example.com/errors/insufficient-funds",  // URI identifying error type
  "title":    "Insufficient Funds",                                  // human-readable summary
  "status":   422,                                                    // HTTP status code
  "detail":   "Account ACC-001 has balance INR 300 but transfer requires INR 500.",
  "instance": "/api/payments/TXN-123",                              // URI of the specific occurrence
  "traceId":  "4bf92f3577b34da6",                                   // extension field — correlates logs
  "timestamp": "2024-01-15T10:30:00Z"                               // extension field
}
// Validation error — field-level details as an extension
{
  "type":    "https://api.example.com/errors/validation-failed",
  "title":   "Request Validation Failed",
  "status":  400,
  "detail":  "3 fields failed validation.",
  "errors": [
    { "field": "amount",     "code": "MUST_BE_POSITIVE",  "message": "Amount must be greater than 0" },
    { "field": "currency",   "code": "INVALID_CURRENCY",  "message": "INR, USD, EUR are supported" },
    { "field": "accountId",  "code": "NOT_BLANK",         "message": "Account ID is required" }
  ]
}
// Spring Boot 3 implementation — ProblemDetail
@RestControllerAdvice
public class GlobalExceptionHandler {
    @ExceptionHandler(MethodArgumentNotValidException.class)
    public ProblemDetail handleValidation(MethodArgumentNotValidException ex) {
        ProblemDetail problem = ProblemDetail.forStatusAndDetail(
            HttpStatus.BAD_REQUEST, "Request validation failed");
        problem.setType(URI.create("https://api.example.com/errors/validation-failed"));
        problem.setTitle("Request Validation Failed");
        List<FieldError> errors = ex.getBindingResult().getFieldErrors()
            .stream()
            .map(fe -> new FieldError(
                fe.getField(),
                fe.getCode(),
                fe.getDefaultMessage()))
            .toList();
        problem.setProperty("errors", errors);
        problem.setProperty("traceId", MDC.get("traceId"));
        return problem;
    }
    @ExceptionHandler(InsufficientFundsException.class)
    public ProblemDetail handleInsufficientFunds(InsufficientFundsException ex) {
        ProblemDetail problem = ProblemDetail.forStatusAndDetail(
            HttpStatus.UNPROCESSABLE_ENTITY, ex.getMessage());
        problem.setType(URI.create("https://api.example.com/errors/insufficient-funds"));
        problem.setTitle("Insufficient Funds");
        problem.setProperty("accountId", ex.getAccountId());
        problem.setProperty("traceId", MDC.get("traceId"));
        return problem;
    }
    @ExceptionHandler(Exception.class)
    public ProblemDetail handleUnexpected(Exception ex, HttpServletRequest req) {
        // Never expose the exception message or stack trace externally
        log.error("Unexpected error on {}: {}", req.getRequestURI(), ex.getMessage(), ex);
        ProblemDetail problem = ProblemDetail.forStatusAndDetail(
            HttpStatus.INTERNAL_SERVER_ERROR,
            "An unexpected error occurred. Please contact support.");
        problem.setProperty("traceId", MDC.get("traceId"));
        return problem;
    }
}
Q5: How do you design a consistent error response format for a REST API? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

I use RFC 7807 Problem Details, which Spring Boot 3 supports natively with ProblemDetail. The format gives you type (a URI identifying the error category), title (human summary), status (HTTP code), detail (specific message for this occurrence), and instance (the request URI). I extend it with two fields that are non-negotiable in production: traceId from MDC so support can correlate the error to the exact log chain, and for validation errors, a list of field-level errors with the field name, a machine-readable code, and a human message. The machine-readable code matters because clients can display localised messages based on the code rather than parsing the English message string. One rule I always enforce: the generic 500 handler never exposes the exception message or stack trace in the response body. The detail field says 'an unexpected error occurred' and nothing more. All the real information goes to the server logs keyed by traceId.

Section 6 — HTTP Caching Headers

The Headers That Control Caching

Header

Direction | Purpose | Example

Cache-Control

Response | Primary cache control directive | Cache-Control: max-age=3600, public

ETag

Response | A fingerprint of the response body (hash or version) | ETag: "v3-a8f2c1"

Last-Modified

Response | When the resource was last changed | Last-Modified: Mon, 15 Jan 2024 10:00:00 GMT

If-None-Match

Request | Send the cached ETag back; server returns 304 if unchanged | If-None-Match: "v3-a8f2c1"

If-Modified-Since

Request | Send the cached date; server returns 304 if unchanged | If-Modified-Since: Mon, 15 Jan 2024 10:00:00 GMT

Vary

Response | Tell caches which request headers affect the response | Vary: Accept-Encoding, Authorization

Expires

Response | Legacy alternative to Cache-Control max-age | Expires: Tue, 16 Jan 2024 10:00:00 GMT

// Cache-Control directives
Cache-Control: public, max-age=3600         // cacheable by anyone for 1 hour
Cache-Control: private, max-age=300         // only the browser caches (not CDN) — user-specific data
Cache-Control: no-cache                     // must revalidate before using cached copy (ETag)
Cache-Control: no-store                     // never cache — sensitive data, financial transactions
Cache-Control: immutable, max-age=31536000  // versioned static assets — never revalidate
// ETag conditional request flow
// 1. First request
GET /api/payments/TXN-123
→ 200 OK  ETag: "v1-abc"  Cache-Control: no-cache
// 2. Client caches response and ETag. Next request sends it back.
GET /api/payments/TXN-123
If-None-Match: "v1-abc"
→ 304 Not Modified  (no body — client uses cached copy)
// 3. Resource has changed — server returns new body and new ETag
GET /api/payments/TXN-123
If-None-Match: "v1-abc"
→ 200 OK  ETag: "v2-def"  (full body returned)
// Spring Boot implementation
@GetMapping("/api/payments/{id}")
public ResponseEntity<PaymentResponse> getPayment(
        @PathVariable String id,
        @RequestHeader(value="If-None-Match", required=false) String ifNoneMatch) {
    Payment payment = service.findById(id);
    String  etag    = "\"v" + payment.getVersion() + "-" + payment.getId() + "\"";
    if (etag.equals(ifNoneMatch)) {
        return ResponseEntity.status(HttpStatus.NOT_MODIFIED)
            .eTag(etag).build();
    }
    return ResponseEntity.ok()
        .eTag(etag)
        .cacheControl(CacheControl.noCache().mustRevalidate())
        .body(mapper.toResponse(payment));
}
Q6: What is an ETag and how does it reduce bandwidth in a REST API? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

An ETag is a fingerprint of a response — typically a hash of the content or a version number. The server includes it in the response. The client stores it alongside the cached response. On the next request for the same resource, the client sends the ETag back in an If-None-Match header. If the resource has not changed, the server returns 304 Not Modified with no body — the client uses its cached copy. The bandwidth saving is the full response body, which matters for large payloads or high-frequency polling. At Deloitte, we used ETags on CloudWatch log group listings that clients polled every 30 seconds. The list rarely changed, so most requests got a 304 and we dramatically reduced both bandwidth and DB query load. The implementation is straightforward: hash the response body or use the database version field. I prefer using the database version because it is O(1) to compute and does not require serialising the response twice.

Section 7 — HATEOAS, OpenAPI and Rate Limiting

HATEOAS — Honest Assessment

What HATEOAS is:

Hypermedia As The Engine Of Application State. Responses include links to valid next actions so clients do not need to hard-code URLs or know which operations are valid in the current state. A PENDING payment response includes a 'cancel' link but no 'refund' link. A COMPLETED payment response includes a 'refund' link but no 'cancel' link.

When it is worth it and when it is not:

// HATEOAS response with Spring HATEOAS
// GET /api/payments/TXN-123  →  payment in PENDING state
{
  "id": "TXN-123",
  "amount": 500.00,
  "status": "PENDING",
  "_links": {
    "self":   { "href": "/api/payments/TXN-123" },
    "cancel": { "href": "/api/payments/TXN-123/cancel", "method": "POST" },
    "account": { "href": "/api/accounts/ACC-001" }
  }
}
// GET /api/payments/TXN-123  →  same payment now COMPLETED
{
  "id": "TXN-123",
  "status": "COMPLETED",
  "_links": {
    "self":   { "href": "/api/payments/TXN-123" },
    "refund": { "href": "/api/payments/TXN-123/refund", "method": "POST" }
    // 'cancel' link is gone — not valid in COMPLETED state
  }
}

Rate Limiting Headers

// Standard rate limiting headers in responses
X-RateLimit-Limit: 100          // max requests allowed in the window
X-RateLimit-Remaining: 42       // requests remaining in the current window
X-RateLimit-Reset: 1705315200   // Unix timestamp when the window resets
// When rate limit is exceeded — 429 Too Many Requests
HTTP/1.1 429 Too Many Requests
Retry-After: 30                 // seconds to wait before retrying
X-RateLimit-Limit: 100
X-RateLimit-Remaining: 0
// Spring Boot — adding rate limit headers to responses
@Component
public class RateLimitFilter extends OncePerRequestFilter {
    private final RateLimiterService rateLimiter;
    @Override
    protected void doFilterInternal(HttpServletRequest request,
                                    HttpServletResponse response,
                                    FilterChain chain) throws IOException, ServletException {
        String apiKey = request.getHeader("X-API-Key");
        RateLimitResult result = rateLimiter.check(apiKey);
        response.setHeader("X-RateLimit-Limit",     String.valueOf(result.getLimit()));
        response.setHeader("X-RateLimit-Remaining", String.valueOf(result.getRemaining()));
        response.setHeader("X-RateLimit-Reset",     String.valueOf(result.getResetAt().getEpochSecond()));
        if (!result.isAllowed()) {
            response.setStatus(HttpStatus.TOO_MANY_REQUESTS.value());
            response.setHeader("Retry-After", String.valueOf(result.getRetryAfterSeconds()));
            response.getWriter().write("{\"status\":429,\"detail\":\"Rate limit exceeded\"}");
            return;
        }
        chain.doFilter(request, response);
    }
}
Q7: What makes a good OpenAPI specification and what is the difference between contract-first and code-first? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Contract-first means you write the OpenAPI YAML before writing any code. The spec becomes the agreement between the team building the API and the team consuming it. Both sides can generate stubs and mocks from it and work in parallel. Code-first means you write the code and generate the spec from annotations — Spring's springdoc-openapi does this. The pragmatic difference: contract-first forces you to think about the API design before implementation details constrain you. Code-first is faster to start but the spec often reflects internal implementation choices rather than what the client actually needs. A good spec includes: clear descriptions on every operation and parameter, example request and response bodies, all possible error response schemas (not just 200), authentication requirements, and rate limit information. The thing most specs get wrong is only documenting the happy path. Clients need to know what 422 and 409 look like as much as they need to know what 200 looks like.

Interview Q&A — 15 Questions

How to use:

Cover the answer, say it out loud, then check. These answers are written to sound like a senior engineer talking — direct, grounded in real experience, with the trade-off included.

Q8: What makes a REST API truly RESTful? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Honestly, most APIs that call themselves RESTful are not — they are HTTP APIs with JSON. True REST has six constraints: uniform interface (consistent resource-based URIs, standard HTTP verbs), stateless (server holds no session state between requests), client-server separation, cacheable responses, layered system (client does not need to know about load balancers, gateways), and optionally code on demand (return executable code). In practice, the constraints that matter most are statelessness — sessions in the server break horizontal scaling — and the uniform interface, which means using HTTP methods for what they were designed for and not building RPC over POST. I would not describe most of what I have built as fully RESTful. I would describe it as pragmatic HTTP API design that follows REST conventions where they give clear value.

Q9: Why is idempotency important for payment APIs specifically? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

Payments involve money, which means the failure modes matter more than in most systems. When a client sends a POST to create a payment and the network fails before the response arrives, the client does not know if the payment was created. If it retries without an idempotency mechanism, you get a duplicate charge. With an idempotency key — a UUID the client generates and sends with the request — the server stores the result for 24 hours keyed on that UUID. If the same key comes in again, the server returns the original result without processing again. The client gets the correct answer and no duplicate charge. Stripe, PayPal, and Razorpay all mandate idempotency keys for payment creation. In my own payment service I implemented this with a Redis store. The important detail: you store the full response, not just a boolean. The client on retry gets the exact same response they would have gotten on the first successful call.

Q10: How do you decide which HTTP status code to return? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

I work through a mental decision tree. First: was the request syntactically malformed — missing fields, wrong types, unparseable JSON? That is 400. Was it well-formed but semantically invalid — negative amount, unsupported currency, account does not exist? That is 422. Did the request conflict with current state — duplicate record, optimistic lock mismatch, concurrent modification? That is 409. Did the user lack permission? 401 for unauthenticated, 403 for authenticated but unauthorised. Did it succeed with a created resource? 201 with Location header. Did it succeed with no body to return? 204. Did an upstream dependency fail? 502 if it returned garbage, 504 if it timed out. Is my own service temporarily overloaded? 503 with a Retry-After. The most common mistakes I see are using 400 for everything bad (losing the semantic distinction), using 200 for errors with an error flag in the body (breaks HTTP semantics and client error handling), and returning 500 for validation errors (scares clients into thinking the server crashed).

Q11: What is the difference between PUT and PATCH? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

PUT is a full replacement — you send the complete resource and the server stores exactly that. If you omit a field in a PUT, that field is cleared. PATCH is a partial update — you send only the fields you want to change. The practical difference matters for clients. With PUT, the client must first GET the resource, modify the field they care about, then PUT the full object. With PATCH, they send only the changed field. PUT is idempotent by definition — sending the same full resource twice leaves the same state. PATCH is not inherently idempotent: a PATCH that says increment amount by 100 applied twice gives a different result than once. A PATCH that sets amount to 500 is idempotent. The JSON Patch RFC 6902 and JSON Merge Patch RFC 7396 define standard formats for PATCH operations. In practice most teams use JSON Merge Patch — send the fields you want to update, null to clear a field — because it is simpler.

Q12: What is the Accept header and when does it matter? [Easy] ▼
💡 Ideal Answer (Amazon/GS Standard)

The Accept header tells the server what content types the client can handle. The server responds with the best matching format and puts the actual format in Content-Type. If the server cannot satisfy the Accept header, it returns 406 Not Acceptable. In practice, most APIs only serve application/json and most clients send Accept: */* (accept anything), so content negotiation rarely comes up. Where it does matter: versioning via content type — Accept: application/vnd.payments.v2+json routes to the V2 handler. Serving multiple formats — some APIs serve both JSON and XML based on Accept. Rate limit responses that need to be machine-readable — always return application/json even when the request Accept was something else, because 429 errors need to be parseable. I always set Content-Type: application/json explicitly on all responses rather than relying on framework defaults, because it removes any ambiguity.

Q13: How do you paginate a resource collection and what are the trade-offs? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

I usually start with offset pagination because it is simple and supports random page access — useful for admin UIs with page number controls. When the dataset grows large (say, over 100k records) or the data changes frequently, I switch to cursor pagination. With offset, OFFSET 10000 LIMIT 20 makes the database scan and discard 10,000 rows to give you 20. With cursor pagination, WHERE id > last_seen_id LIMIT 20 uses the index directly. Cursor is also stable — new inserts do not cause the same record to appear on two pages. The downside of cursor: no random page access, and you need a monotonic sort key to derive the cursor from. For the response format, I always include whether there is a next page and the cursor for it — I do not make clients guess based on page size.

Q14: What does CORS do and when do you need it? [Easy] ▼
💡 Ideal Answer (Amazon/GS Standard)

CORS — Cross-Origin Resource Sharing — is a browser security mechanism. Browsers block JavaScript from making requests to a different origin (different protocol, domain, or port) unless the server explicitly allows it. When a browser makes a cross-origin request, it either sends a preflight OPTIONS request asking what is allowed, or for simple requests it sends the actual request and checks the response headers. The server must return Access-Control-Allow-Origin with the allowed origin (or *), and for credentials-based requests, the exact origin, never *. Where it matters for backend developers: when you are building an API that a web frontend at a different domain calls. You configure CORS in Spring with @CrossOrigin or a global CorsConfigurationSource bean. The mistake I see is setting Access-Control-Allow-Origin: * when the requests use cookies or Authorization headers — that combination is rejected by the browser. You must use the exact origin and set Access-Control-Allow-Credentials: true.

Q15: When should a DELETE return 200 vs 204, and should the second DELETE return 404? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

204 No Content is the most common choice for DELETE — the operation succeeded and there is nothing to return. 200 OK with a response body is appropriate when you want to return the final state of the deleted resource, which some clients find useful for confirmation or audit trail. On the second DELETE — the idempotency question: technically it should return 200 or 204 because the end state is the same (resource does not exist), and the operation was successful from the client's perspective. The argument for 404 on the second call is that the resource genuinely does not exist. In practice, I return 200 with the last known state on the second DELETE if I store it, or 204 if the record is hard-deleted. I avoid 404 on the second call because clients handling DELETE failures will retry, get a 404, and may treat it as an error when the original DELETE succeeded. Idempotency should mean the client gets the same success signal every time.

Q16: How do you handle long-running operations in a REST API? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

The pattern I use is the job resource pattern. The client sends a POST to initiate the operation and gets back 202 Accepted immediately with a job ID and a Location header pointing to /api/jobs/{jobId}. The operation runs asynchronously. The client polls GET /api/jobs/{jobId} to check progress — the job resource returns status: PENDING, IN_PROGRESS, COMPLETED, or FAILED. When COMPLETED, it includes the result or a link to the result. I add Cache-Control: no-cache on the job resource so clients do not cache stale status. The polling interval should be communicated — I include a Retry-After header on 200 responses from the job endpoint suggesting when to poll next. A more sophisticated alternative is webhooks — the client registers a callback URL and you POST the result to it when done. Webhooks are push and more efficient, but they require the client to have a reachable endpoint, which is not always the case.

Q17: What is the Location header and when should you use it? [Easy] ▼
💡 Ideal Answer (Amazon/GS Standard)

Location tells the client where a newly created resource can be found. After a 201 Created response to POST /api/payments, the Location header should be /api/payments/TXN-123. This lets clients bookmark or access the created resource without parsing the response body. It is also used in 301 and 302 redirects — Location points to the new URL. The practical value: if a client posts a payment and the response body is lost in a network failure, the client can still retrieve the created resource using the Location header it may have captured from the response headers. I always include Location on 201 responses — it is part of the HTTP spec for created resources and clients that follow it correctly benefit from it.

Q18: How do you document an API and what makes documentation actually useful? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

I use OpenAPI with springdoc-openapi for automatic spec generation from code, plus manual enrichment for things the annotations do not capture — error response schemas, business rule explanations, example values. Useful documentation has four things that most generated docs lack. First, real examples for both request and response — not just the schema, but an actual filled-in example that a developer can copy and run. Second, all error responses documented, not just the 200 — what does a 422 look like, what fields are in the errors array. Third, business context — what does it mean for a payment to be in PENDING state, what transitions are allowed. Fourth, authentication documented clearly — what the token looks like, where to get one, how long it lives. The thing that makes documentation useless is generating it and then never updating it. Contract-first development — write the spec, generate the server stub — keeps the spec and the implementation in sync.

Q19: What is the difference between 400 and 422? [Easy] ▼
💡 Ideal Answer (Amazon/GS Standard)

400 Bad Request is for when the request is structurally malformed — the server could not understand what was being asked. Missing required JSON fields, wrong data type for a field, unparseable date string. 422 Unprocessable Entity is for when the request is perfectly well-formed — the server understood it completely — but it violates business rules. A payment request with amount: -500 is 422, not 400. The JSON is valid, the field is present, the type is correct, but the value makes no business sense. A transfer to your own account is 422. An unsupported currency code is 422. The distinction matters for clients: a 400 means fix your request format; a 422 means your request format is fine but the business logic rejected it, and the error message explains why.

Q20: How do you rate limit an API and communicate it to clients? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Rate limiting happens at the API Gateway layer first — Spring Cloud Gateway's RequestRateLimiter filter with a Redis token bucket, keyed by API key or user ID. When the limit is exceeded, the gateway returns 429 Too Many Requests. Every response — not just 429s — should include the rate limit headers: X-RateLimit-Limit (the window limit), X-RateLimit-Remaining (how many left), and X-RateLimit-Reset (when the window resets as a Unix timestamp). On a 429, add Retry-After in seconds. This gives clients the information they need to implement exponential backoff without guessing. Clients that behave well — respect Retry-After — are rate-limited but functional. Clients that ignore it and hammer the API get blacklisted. I also differentiate rate limits by tier: free tier gets 100 requests/minute, paid tier gets 1,000. This is easier to implement if your rate limiter is keyed on a combined (apiKey, tier) rather than just IP.

Q21: What is HATEOAS and would you actually use it? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

HATEOAS means including hypermedia links in API responses so clients can navigate the API without hard-coding URLs. A payment in PENDING state includes a cancel link but not a refund link. A COMPLETED payment includes a refund link. Clients follow links rather than constructing URLs. The theory is sound — it decouples clients from URL structure and makes state transitions explicit. In practice, I have not used it in production and would not recommend it by default. Most API clients ignore the links and hard-code URLs regardless. The response model becomes significantly more complex. State-based link inclusion requires careful testing. And the primary benefit — URL decoupling — is better solved by stable versioned URIs. Where I would consider it: a very public API consumed by hundreds of integrators over a long lifetime, where you genuinely want clients to be resilient to URL changes. For internal APIs or developer-facing APIs with a normal lifecycle, it adds complexity without payoff.
