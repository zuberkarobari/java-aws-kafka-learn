Q1: Explain how HashMap works internally including what happens at treeify threshold. [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

HashMap uses an array of buckets. Each bucket is a linked list of entries that hashed to the same bucket index. The bucket index is computed as (n-1) & hash(key) where n is always a power of 2. The hash() method applies h ^ (h>>>16) to spread the high 16 bits into the low 16, reducing collisions for keys with similar low-order bits. When a bucket's list reaches 8 entries (TREEIFY_THRESHOLD), the linked list is converted to a Red-Black tree, capping worst-case lookup at O(log n) instead of O(n). When entries drop below 6 (UNTREEIFY_THRESHOLD) due to removal, it converts back to a list. Resize happens when size exceeds capacity * loadFactor (default 0.75). The new capacity is doubled and all entries are rehashed.

Q2: What is the happens-before relationship in Java memory model? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

Happens-before defines the visibility guarantee between threads. If action A happens-before action B, all effects of A are visible to B. Key rules: within a thread, each action happens-before subsequent ones. Monitor unlock happens-before every subsequent lock of that monitor. Writing to a volatile field happens-before every subsequent read of that field. Thread.start() on a thread happens-before any action in that thread. A thread completing happens-before Thread.join() returning in another thread. Without happens-before, the JVM and CPU are free to reorder operations for performance. This is why double-checked locking without volatile is broken — the assignment of the instance field can be reordered before the constructor completes, exposing a partially-constructed object.

Q3: What is CompletableFuture and how does it differ from Future? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Future represents an asynchronous computation — you can call get() to block and retrieve the result. But you cannot chain operations or handle completion callbacks. CompletableFuture extends Future and adds functional composition: thenApply (transform the result), thenCompose (chain another async operation), thenCombine (combine two futures), exceptionally (handle errors), allOf/anyOf (combine multiple futures). This allows building non-blocking pipelines: fetch user profile, then fetch their payments, then combine both into a dashboard response — all without blocking threads. CompletableFuture's thenApplyAsync runs the transformation on a ForkJoinPool thread; thenApply runs it on the thread that completed the future. The key advantage over Future: no blocking unless you explicitly call join() or get().

Q4: What are the rules for @Transactional rollback and name three ways it can silently fail. [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

By default @Transactional rolls back on RuntimeException and Error. Checked exceptions do not trigger rollback — the transaction commits even if a checked exception propagates. The rollbackFor attribute extends this: rollbackFor = Exception.class makes all exceptions trigger rollback. Three silent failures: first, self-invocation — calling a @Transactional method from within the same class via 'this' bypasses the proxy, so no transaction starts. Second, private or final methods — CGLIB cannot proxy them, the annotation is ignored. Third, @Transactional on a test method prevents testing REQUIRES_NEW propagation — the inner transaction cannot start independently inside the test transaction, so REQUIRES_NEW behaves like REQUIRED.

Q5: Explain Spring AOP proxy types and the self-invocation problem. [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

Spring AOP is proxy-based. For classes implementing at least one interface, Spring creates a JDK dynamic proxy that implements the same interfaces. For classes without interfaces (Spring Boot's default since 2.x), Spring creates a CGLIB proxy that subclasses the target class. The proxy intercepts method calls and applies advice before delegating to the real object. Self-invocation: when a bean calls its own @Transactional method via this.method(), the call goes directly to the real object, not through the proxy. The proxy is never involved. No advice runs, no transaction starts. The fix: inject the bean into itself via ApplicationContext.getBean() or @Autowired, or restructure the code so the @Transactional method is called from a different class.

Q6: How does Spring Boot auto-configuration work? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

@SpringBootApplication includes @EnableAutoConfiguration which triggers AutoConfigurationImportSelector. It reads all auto-configuration class names from META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports. Each candidate is evaluated against its @Conditional annotations: @ConditionalOnClass checks the classpath, @ConditionalOnMissingBean checks for existing beans, @ConditionalOnProperty checks application.properties. Only configurations that pass all conditions are registered. Auto-configurations run after your own @Configuration classes — your beans take precedence. @ConditionalOnMissingBean is the override mechanism: Spring creates the default DataSource only if you have not defined your own.

Q7: What is the volatile keyword and when is it necessary? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

volatile ensures that reads and writes to a variable are always from/to main memory rather than a CPU cache. It provides two guarantees: visibility (all threads see the latest written value) and ordering (writes to volatile happen-before subsequent reads). It does NOT guarantee atomicity for compound operations. When necessary: the classic case is a flag checked by one thread and set by another, like a 'running' boolean that controls a processing loop. Without volatile, the reading thread may cache the value and never see the update. Also necessary for double-checked locking: the instance field must be volatile to prevent the reorder of object reference assignment and constructor execution. For increment operations, use AtomicInteger — volatile boolean++ is still not atomic.

Q8: What is a BeanPostProcessor and name two that Spring uses internally. [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

BeanPostProcessor is an interface with two methods: postProcessBeforeInitialization and postProcessAfterInitialization. Spring calls both for every bean. The return value can replace the original bean — this is how AOP proxies are created. postProcessAfterInitialization is where AbstractAutoProxyCreator wraps @Transactional, @Async, @Cacheable beans in proxies. Two internal examples: AutowiredAnnotationBeanPostProcessor processes @Autowired and @Value injection. CommonAnnotationBeanPostProcessor processes @PostConstruct and @PreDestroy. PersistenceAnnotationBeanPostProcessor handles @PersistenceContext. Writing a custom BeanPostProcessor is how you implement cross-cutting concerns that apply to all beans at startup.

Q9: What is the ReentrantLock and when would you use it over synchronized? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

ReentrantLock is an explicit lock implementation providing capabilities that synchronized blocks do not. First: tryLock() — try to acquire the lock without blocking; fail fast if unavailable. Second: lockInterruptibly() — acquire the lock but allow the thread to be interrupted while waiting. Third: fair locking — ReentrantLock(true) gives the lock to the longest-waiting thread rather than an arbitrary one. Fourth: multiple condition variables — a single ReentrantLock can have multiple Condition objects for different wait conditions, more flexible than Object.wait/notify. Use synchronized for simple mutual exclusion — it is less code and JVM-optimised. Use ReentrantLock when you need tryLock for deadlock avoidance, interruptibility, or fair ordering.

Q10: Explain the difference between @Async void and @Async CompletableFuture. [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

@Async void: the method executes on a separate thread, the caller returns immediately. Exceptions thrown in the method are silently swallowed unless you implement AsyncUncaughtExceptionHandler. The caller has no way to know if the operation succeeded. @Async CompletableFuture: the method executes on a separate thread and the caller gets a CompletableFuture. Exceptions are captured in the future — the caller can chain .exceptionally() or call .join() to get the result and handle errors. When to use void: fire-and-forget operations like sending notifications where the caller does not need the result or error. When to use CompletableFuture: when the caller needs to react to the result or handle failures — parallel API calls that are combined with thenCombine or allOf.

Q11: What does 'thread-safe' mean and why is ArrayList not thread-safe? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Thread-safe means that concurrent access from multiple threads produces correct results without external synchronisation. ArrayList is not thread-safe because its internal array and size field are updated without synchronisation. Two threads calling add() simultaneously can both read the same size, both assign to the same index, and one write is lost. Or one thread calls get() while another is in the middle of a resize (copying to a new array), reading from a partially-copied array. CopyOnWriteArrayList is thread-safe for reads at the cost of expensive writes (copies the entire array on every write). ConcurrentLinkedQueue provides a lock-free concurrent queue. Collections.synchronizedList() wraps each method in synchronized but requires external synchronisation for iteration. The choice depends on the read/write ratio and whether you need iteration to be safe.
