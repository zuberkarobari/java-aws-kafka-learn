🗣 The Mentor Explains: var — Clarity vs Laziness

👨‍💻

Senior Mentor Tip

This is misunderstood constantly — even by seniors:

var is NOT dynamic typing. It is NOT like JavaScript's var. Java is still statically typed. var just asks the COMPILER to infer the type from the right-hand side. The type is fixed at compile time. You cannot reassign a var to a different type later. Think of it as syntactic sugar that removes redundancy: List list = new ArrayList() — why write Employee twice? var list = new ArrayList() — same thing, cleaner.

// var — compiler infers the type from the initializer
var name     = "Zuber";              // inferred: String
var count    = 42;                   // inferred: int
var list     = new ArrayList<String>(); // inferred: ArrayList<String>
var employee = new Employee();       // inferred: Employee
// var in for-each loops — very clean
for (var employee : employees) {
    System.out.println(employee.getName());
}
// var in try-with-resources
try (var conn = dataSource.getConnection()) {
    // conn is inferred as Connection
}
// ❌  When NOT to use var (readability suffers)
var x = getResult();  // What type is x? Can't tell without reading getResult()
var map = createMapping();  // Is this Map<String, List<Employee>>? Unknown.
// ✅  When var HELPS (type is obvious from right side)
var users = new HashMap<String, User>();  // crystal clear
var response = client.send(request, HttpResponse.BodyHandlers.ofString());

Java 11 String Methods — Know Them All

👨‍💻

Senior Mentor Tip

These get asked in coding rounds more than you'd expect:

Junior devs still write str.trim() in 2025. Senior devs know strip() uses Unicode-aware whitespace detection — trim() only handles ASCII spaces (chars <= 32). In a global payment system like Visa where you process international names and addresses, strip() is the correct choice. Interviewers at Goldman notice this.

String s = "  Hello World  ";
// strip() — Unicode-aware whitespace removal (PREFERRED over trim())
s.strip();           // "Hello World"
s.stripLeading();    // "Hello World  "
s.stripTrailing();   // "  Hello World"
// isBlank() — true if empty or only whitespace (better than isEmpty())
"   ".isBlank();    // true   — isEmpty() would return false here!
"".isBlank();       // true
"hi".isBlank();     // false
// lines() — splits on line terminators, returns Stream<String>
"line1\nline2\nline3".lines().count();  // 3 — great for parsing multi-line text
// repeat() — string multiplication
"=-".repeat(10);   // "=-=-=-=-=-=-=-=-=-="  — separator lines, padding
// Java 12: indent(n) — adds n spaces to each line
"hello\nworld".indent(4);  // "    hello\n    world\n"
// Java 12: transform(fn) — apply a function to the string inline
"  hello  ".transform(String::strip).transform(String::toUpperCase);  // "HELLO"

Java 14: Switch Expressions — The New Syntax

👨‍💻

Senior Mentor Tip

The old switch was a bug factory. The new one is a value:

// OLD switch — verbose, fall-through danger, can't be used as value
String result;
switch (day) {
    case MONDAY:
    case TUESDAY: result = "Weekday"; break;  // forget break = bug
    case SATURDAY:
    case SUNDAY:  result = "Weekend"; break;
    default:      result = "Unknown"; break;
}
// NEW switch expression — arrow syntax, no fall-through, IS a value
String result = switch (day) {
    case MONDAY, TUESDAY, WEDNESDAY, THURSDAY, FRIDAY -> "Weekday";
    case SATURDAY, SUNDAY -> "Weekend";
};  // compiler enforces exhaustiveness for enums — no need for default!
// Multi-line case with yield
int fee = switch (accountType) {
    case PREMIUM -> 0;
    case STANDARD -> {
        int base = 50;
        int discount = calculateDiscount(customer);
        yield base - discount;  // 'yield' returns value from block
    }
    case BASIC -> 100;
};

Q3: What is var in Java 11? Is Java now dynamically typed? [Easy] ▼
💡 Ideal Answer (Amazon/GS Standard)

Ideal Answer: No — Java remains statically typed. var is local variable type inference. The compiler infers the type from the right-hand side at compile time — the type is fixed. You can't reassign a var to a different type later. It's purely syntactic sugar to reduce redundancy. Limitations: cannot use var for method parameters, return types, or fields. Cannot use with null initializer (var x = null — compiler can't infer type). Best for local variables where the type is obvious from context.

Q4: What is the difference between String.trim() and String.strip() in Java 11? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Ideal Answer: Both remove leading and trailing whitespace but with different definitions of 'whitespace'. trim() removes characters with ASCII value <= 32 (limited, ASCII-only). strip() uses Character.isWhitespace() which is Unicode-aware — it handles Unicode whitespace like non-breaking spaces, em-spaces used in international text. In a global payment system processing international customer data, strip() is always the correct choice. isBlank() similarly uses Unicode definition.

Q5: What is a yield keyword in Java 14+ switch expressions? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Ideal Answer: yield returns a value from a multi-line block inside a switch expression. Similar to return but specific to switch blocks. Single-line cases use the arrow (->) and directly provide the value. Multi-line cases need a block ({}) and must use yield to specify which value to return. Without yield in a block, the compiler errors. It cannot be used outside of switch expressions.

🗣 Records — The DTO Revolution

👨‍💻

Senior Mentor Tip

Before Records, writing a DTO in Java was painful:

A proper immutable value object needed: private final fields, constructor with validation, getters, equals(), hashCode(), toString() — easily 50+ lines for 3 fields. Nobody wants to write that. Nobody wants to maintain it. Lombok @Value helped, but it was a library dependency. Records are Java's native answer. You declare the components, Java generates EVERYTHING else. And here's the key: Records communicate intent — this class IS a data carrier, nothing more.

// BEFORE Records — 50 lines for a simple DTO
public final class PaymentRequest {
    private final String transactionId;
    private final double amount;
    private final String currency;
    // constructor, 3 getters, equals, hashCode, toString... 50+ lines
}
// AFTER Records — same thing, 1 line
public record PaymentRequest(String transactionId, double amount, String currency) {}
// What the compiler auto-generates:
// ✅  private final fields for each component
// ✅  canonical constructor (all-args)
// ✅  accessors: transactionId(), amount(), currency()  (NOT getXxx!)
// ✅  equals() based on all components
// ✅  hashCode() based on all components
// ✅  toString() → PaymentRequest[transactionId=TX-1, amount=5000.0, currency=INR]
// Using the record
var req = new PaymentRequest("TX-001", 5000.0, "INR");
System.out.println(req.amount());      // 5000.0 — accessor, not getAmount()
System.out.println(req.currency());    // INR
// Compact Constructor — for validation (no parameter list needed)
public record PaymentRequest(String transactionId, double amount, String currency) {
    public PaymentRequest {   // compact constructor — parameters are implicit
        if (amount <= 0)      throw new IllegalArgumentException("Amount must be positive");
        if (currency == null)  throw new IllegalArgumentException("Currency required");
        transactionId = transactionId.trim();  // you can normalize inputs here
    }
}
// Records CAN have: static fields, static methods, instance methods, implement interfaces
// Records CANNOT have: mutable instance state, extend other classes (implicitly extend Record)

Sealed Classes — Bringing Type Hierarchy Under Control

👨‍💻

Senior Mentor Tip

Why would you ever want to RESTRICT who can extend your class?

Classic example: Shape hierarchy. You define Shape, and Circle, Rectangle, Triangle extend it. In classic Java, ANYONE — even code in other packages — can create a new Shape subclass. Your switch statement that handles all shapes can suddenly get a subclass it doesn't know about. Sealed classes say: 'These and ONLY these classes are permitted to extend me.' The compiler now KNOWS the complete list of subclasses — enabling exhaustive pattern matching without a default case. Used in compilers, expression trees, domain models. Your Spring Boot result types (Success/Failure) are perfect sealed class use cases.

// Sealed class — defines the permitted hierarchy
public sealed class PaymentResult
    permits SuccessResult, FailureResult, PendingResult {}
// Each subclass must be: final, sealed, or non-sealed
public final record SuccessResult(String transactionId, double amount) extends PaymentResult {}
public final record FailureResult(String reason, String errorCode) extends PaymentResult {}
public final record PendingResult(String transactionId, Instant retryAfter) extends PaymentResult {}
// The payoff: EXHAUSTIVE switch with NO default needed
// Compiler verifies all subtypes are handled
String message = switch (result) {  // Java 17 pattern matching switch
    case SuccessResult s -> "Payment " + s.transactionId() + " succeeded: " + s.amount();
    case FailureResult f -> "Payment failed: " + f.reason() + " [" + f.errorCode() + "]";
    case PendingResult p -> "Pending — retry after: " + p.retryAfter();
    // No default needed — compiler knows these 3 cover all cases
};
// Sealed interface (more common with records)
public sealed interface Shape permits Circle, Rectangle, Triangle {}
public record Circle(double radius) implements Shape {}
public record Rectangle(double width, double height) implements Shape {}
public record Triangle(double base, double height) implements Shape {}
double area = switch (shape) {
    case Circle c    -> Math.PI * c.radius() * c.radius();
    case Rectangle r -> r.width() * r.height();
    case Triangle t  -> 0.5 * t.base() * t.height();
};

Pattern Matching for instanceof — Clean Up the Cast Boilerplate

// BEFORE Pattern Matching instanceof (Java 15 and earlier)
if (shape instanceof Circle) {
    Circle c = (Circle) shape;  // redundant cast — you JUST checked it's a Circle
    System.out.println(c.radius());
}
// AFTER Pattern Matching instanceof (Java 16+)
if (shape instanceof Circle c) {  // test AND bind in one expression
    System.out.println(c.radius());  // c is in scope here
}
// The binding variable scope is smart:
if (!(shape instanceof Circle c)) {
    return;  // early return
}
// c is in scope here — the compiler knows shape IS a Circle past this point
System.out.println(c.radius());
// In switch (Java 17 preview, Java 21 finalized)
String describe = switch (obj) {
    case Integer i   -> "Integer: " + i;
    case String s    -> "String: " + s.toUpperCase();
    case null        -> "null value";  // explicit null handling!
    default          -> "Other: " + obj.getClass().getSimpleName();
};

Q7: What are Java Records? What do they auto-generate? What can't they do? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Ideal Answer: Records are immutable data carriers. Auto-generated: canonical constructor, accessors (component name without get prefix), equals(), hashCode(), toString(). Q8: What are Sealed Classes and what problem do they solve? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

Ideal Answer: Sealed classes restrict which classes can extend or implement them using the 'permits' clause. Problem solved: uncontrolled type hierarchies where any class can subclass yours, breaking exhaustive switch/if-instanceof logic. With sealed classes, the compiler knows ALL possible subtypes — enabling exhaustive pattern matching without a default case. Use cases: algebraic data types, domain model hierarchies (Result/Success/Failure), expression trees in compilers. Each permitted subclass must be final, sealed, or non-sealed.

Q9: What is Pattern Matching instanceof and how does it differ from traditional instanceof? [Easy] ▼
💡 Ideal Answer (Amazon/GS Standard)

Ideal Answer: Traditional instanceof requires two steps: if (obj instanceof String) { String s = (String) obj; — redundant cast. Pattern Matching (Java 16) combines test and bind: if (obj instanceof String s) — s is automatically bound and available in scope. The binding variable's scope is narrowed by the compiler — only accessible where the condition is provably true. Eliminates ClassCastException risk and removes boilerplate. Pairs beautifully with switch expressions for clean polymorphic dispatch.

Race Condition — The Most Common Multithreading Bug

👨‍💻

Senior Mentor Tip

A race condition is when the outcome depends on the timing/order of threads:

Classic example: two threads both read a counter (value=5), both add 1 to their local copy, both write back 6. Expected: 7. Actual: 6. One increment is lost. This happens because counter++ is NOT atomic — it's three operations: READ, MODIFY, WRITE. Any thread can interleave between these steps. In your Infosys VISA payment system, if two threads processed the same transaction ID simultaneously without synchronization, you could double-process a payment. That's a real consequence of race conditions.

// THE RACE CONDITION — looks harmless, isn't
public class Counter {
    private int count = 0;
    // This is NOT thread-safe!
    public void increment() {
        count++;  // 3 operations: READ count, ADD 1, WRITE back
    }  // Two threads can interleave between these 3 steps
    // FIX 1: synchronized method — only one thread at a time
    public synchronized void incrementSafe() {
        count++;  // now atomic — monitor lock acquired on 'this'
    }
    // FIX 2: synchronized block — lock only the critical section
    public void incrementBlock() {
        synchronized (this) {
            count++;
        }
        // non-critical work here runs concurrently
    }
    // FIX 3: AtomicInteger — CAS-based, no locking, faster than synchronized
}
// Best fix: AtomicInteger
public class Counter {
    private final AtomicInteger count = new AtomicInteger(0);
    public void increment() { count.incrementAndGet(); }  // atomic, lock-free
    public int getCount()   { return count.get(); }
}

Deadlock — The Scenario That Freezes Production Systems

👨‍💻

Senior Mentor Tip

This is the most asked multithreading theory question at FAANG and banks:

// CLASSIC DEADLOCK — Thread A and Thread B each hold one lock
// and are waiting for the other's lock. System freezes.
Object lockA = new Object();
Object lockB = new Object();
// Thread 1: acquires lockA, then tries to acquire lockB
Thread t1 = new Thread(() -> {
    synchronized (lockA) {
        System.out.println("T1 holds lockA, waiting for lockB");
        synchronized (lockB) {  // BLOCKED — T2 holds lockB
            System.out.println("T1 holds both");
        }
    }
});
// Thread 2: acquires lockB, then tries to acquire lockA
Thread t2 = new Thread(() -> {
    synchronized (lockB) {
        System.out.println("T2 holds lockB, waiting for lockA");
        synchronized (lockA) {  // BLOCKED — T1 holds lockA
            System.out.println("T2 holds both");
        }
    }
});
// SOLUTION: Always acquire locks in the SAME ORDER
// Both threads should acquire lockA first, then lockB.
// Now neither can be blocked by the other.

wait() / notify() / notifyAll() — The Object Monitor Protocol

👨‍💻

Senior Mentor Tip

Why are wait/notify on Object and NOT on Thread?

Because the lock is on the OBJECT, not the thread. When a thread calls obj.wait(), it releases the lock on 'obj' and goes to WAITING state. When another thread calls obj.notify(), one waiting thread is woken and must re-acquire the lock before continuing. The OBJECT is the communication channel. That's why these methods are on Object — every object can be a lock, every object can be a signal. You MUST call wait/notify inside a synchronized block on the SAME object.

// Producer-Consumer with wait/notify (classic pattern)
public class MessageQueue {
    private final Queue<String> queue = new LinkedList<>();
    private final int MAX_SIZE = 10;
    public synchronized void produce(String msg) throws InterruptedException {
        while (queue.size() == MAX_SIZE) {  // while, not if — guard against spurious wakeup
            wait();    // release lock, go WAITING
        }
        queue.add(msg);
        notifyAll(); // wake ALL waiting threads (prefer notifyAll over notify)
    }
    public synchronized String consume() throws InterruptedException {
        while (queue.isEmpty()) {  // while loop is CRITICAL here
            wait();    // release lock, go WAITING
        }
        String msg = queue.poll();
        notifyAll(); // producer might be waiting — wake them
        return msg;
    }
}
// WHY while and not if? Spurious wakeups — a thread can wake from wait()
// without being notified. while re-checks the condition and waits again if needed.

ThreadLocal — Per-Thread Data Without Synchronization

👨‍💻

Senior Mentor Tip

In Spring Boot every request is handled by a thread. This pattern keeps request-scoped data clean:

ThreadLocal stores a SEPARATE value for EACH thread. Thread A sets userId=101, Thread B sets userId=202 — they don't interfere. Spring Security uses ThreadLocal to store the SecurityContext (authenticated user). MDC (Mapped Diagnostic Context) in logging uses ThreadLocal for correlation IDs. In your Deloitte JP Morgan project, the thread-per-request model means ThreadLocal is the right place to store request-scoped context like a request ID for log tracing.

public class RequestContext {
    // Each thread gets its own copy — zero synchronization needed
    private static final ThreadLocal<String> correlationId = new ThreadLocal<>();
    private static final ThreadLocal<String> userId        = new ThreadLocal<>();
    public static void set(String corrId, String uid) {
        correlationId.set(corrId);
        userId.set(uid);
    }
    public static String getCorrelationId() { return correlationId.get(); }
    public static String getUserId()         { return userId.get(); }
    // CRITICAL: Always remove() when done — prevents memory leaks in thread pools!
    // In thread pools, threads are REUSED — old ThreadLocal values can leak to next request
    public static void clear() {
        correlationId.remove();
        userId.remove();
    }
}
// In Spring — use OncePerRequestFilter:
// set() in doFilter() → process request → clear() in finally block
Q10: Interview Questions — Topic 3 ▼
💡 Ideal Answer (Amazon/GS Standard)

Q11: What are the 4 conditions required for a deadlock? How do you prevent it? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

Ideal Answer: Coffman conditions: 1) Mutual Exclusion — resources held exclusively. 2) Hold and Wait — holding one lock while requesting another. 3) No Preemption — cannot forcibly take a lock. 4) Circular Wait — A waits for B's lock, B waits for A's lock. Prevention strategies: 1) Lock ordering — always acquire locks in same global order (breaks circular wait). 2) tryLock() with timeout — abandon and retry if lock not acquired (ReentrantLock). 3) Lock-free data structures — AtomicInteger, ConcurrentHashMap. 4) Minimize lock scope — don't hold locks longer than needed.

Q12: Why must you use a while loop instead of if when calling wait()? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

Ideal Answer: Spurious wakeups — a thread can be woken from wait() without any notify() call, due to OS-level signals. This is documented in the Java spec. If you use if: the thread wakes up, the condition is not actually met (queue is still empty), but the thread proceeds anyway — corrupting the program state. With while: the thread wakes up, re-checks the condition, and if not met, calls wait() again. The while loop is the ONLY correct pattern for wait()/notify().

Q13: What is ThreadLocal and what is the critical danger of using it in thread pools? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

Ideal Answer: ThreadLocal stores a separate value per thread — each thread accessing it gets its own isolated copy. No synchronization needed. Danger in thread pools: threads are REUSED across requests. If you set a ThreadLocal value for Request 1 and don't remove() it, the SAME thread handling Request 2 will see Request 1's stale value. Solution: always call ThreadLocal.remove() in a finally block after processing. Spring's RequestContextHolder handles this automatically — which is a good example of why you should use framework-provided context holders.

🗣 Why Manual Thread Creation Is Wrong

👨‍💻

Senior Mentor Tip

A real story from a production incident:

// ❌ WRONG — unbounded thread creation
for (Transaction t : transactions) {
    new Thread(() -> processTransaction(t)).start();  // 10,000 transactions = 10,000 threads
}
// ✅ CORRECT — bounded thread pool
ExecutorService pool = Executors.newFixedThreadPool(10);  // max 10 concurrent threads
for (Transaction t : transactions) {
    pool.submit(() -> processTransaction(t));  // tasks queue when all 10 threads busy
}
pool.shutdown();  // stop accepting new tasks, wait for running ones to finish

When to Use + Internal Behaviour

newFixedThreadPool(n)

Fixed n threads. Queue is unbounded (LinkedBlockingQueue). Predictable resource usage. Use for CPU-bound tasks where n = CPU cores. Your batch processing, report generation.

newCachedThreadPool()

0 to unbounded threads — creates new as needed, reuses idle ones (60s keepalive). NO queue — tasks execute immediately. Use for short-lived async tasks with variable load. DANGER: traffic spikes can create thousands of threads.

newSingleThreadExecutor()

1 thread, unbounded queue. Tasks run in FIFO order, one at a time. Guarantees sequential execution. Use for event loops, ordered background processing, writing to a single file.

newScheduledThreadPool(n)

For delayed or periodic tasks. schedule(task, delay, unit). scheduleAtFixedRate(), scheduleWithFixedDelay(). Use for heartbeats, polling, cache refresh, cron-like tasks.

// ExecutorService — the right way to submit work
ExecutorService executor = Executors.newFixedThreadPool(10);
// execute(): Runnable — fire and forget, no return value, exception is swallowed
executor.execute(() -> System.out.println("Background task"));
// submit(): returns Future<T> — track completion, get result, handle exception
Future<String> future = executor.submit(() -> {
    return fetchUserFromDatabase(userId);  // Callable<String>
});
// Future usage
System.out.println(future.isDone());          // is task complete?
String result = future.get();                 // BLOCKS until done
String result2 = future.get(5, TimeUnit.SECONDS); // blocks max 5 sec, throws TimeoutException
future.cancel(true);                          // attempt to cancel (true = interrupt if running)
// invokeAll — submit multiple tasks, wait for ALL to complete
List<Callable<String>> tasks = ...
List<Future<String>> futures = executor.invokeAll(tasks);  // blocks until all done
// invokeAny — submit multiple tasks, return FIRST completed result
String first = executor.invokeAny(tasks);  // great for race-to-first-answer pattern
// Shutdown — ALWAYS do this, or the JVM won't exit (non-daemon threads)
executor.shutdown();         // graceful: finishes in-progress tasks, rejects new ones
executor.shutdownNow();      // forceful: tries to interrupt running tasks, returns pending
executor.awaitTermination(60, TimeUnit.SECONDS);  // wait for shutdown completion
Q14: Interview Questions — Topic 4 ▼
💡 Ideal Answer (Amazon/GS Standard)

Q15: What is the difference between execute() and submit() in ExecutorService? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Ideal Answer: execute(Runnable): fire-and-forget. Returns void. If the task throws an unchecked exception, it's swallowed (handled by the thread's UncaughtExceptionHandler — often silently lost). submit(Callable/Runnable): returns a Future. Exceptions are captured inside the Future — thrown when you call future.get(). This is CRITICAL: use submit() when you need the result OR when you need to know if the task failed. execute() is fine for pure fire-and-forget logging, metrics, etc. where failure is acceptable.

Q16: What is the difference between shutdown() and shutdownNow()? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Ideal Answer: shutdown(): graceful stop. Stops accepting NEW tasks. Allows already-submitted and currently-running tasks to complete. The pool drains naturally. shutdownNow(): immediate stop attempt. Sends interrupt signals to all running threads. Returns a List of submitted-but-not-yet-executed tasks. Running tasks may or may not stop depending on whether they check Thread.interrupted(). Use shutdown() for normal application shutdown, shutdownNow() for emergency stop or testing. Always follow with awaitTermination() to wait for completion.

🗣 The Problem with Future.get()

👨‍💻

Senior Mentor Tip

Future.get() blocks the calling thread. You haven't gained anything:

Imagine your Deloitte microservice needs to call 3 downstream APIs — user service, account service, transaction service. With Future: you submit 3 tasks, then call get() on each one sequentially. While calling get(), your current thread is BLOCKED and can't do anything else. If each API takes 200ms, you wait 200ms per get() call. CompletableFuture lets you say 'when all 3 are done, THEN process the results' — without blocking. Your thread is free to do other work while waiting. This is non-blocking async programming.

// CompletableFuture — non-blocking async pipeline
// supplyAsync: run in ForkJoinPool.commonPool() by default, returns value
CompletableFuture<User> userFuture =
    CompletableFuture.supplyAsync(() -> userService.getUser(userId));
// Custom executor — always use one in production!
ExecutorService pool = Executors.newFixedThreadPool(10);
CompletableFuture<User> userFuture2 =
    CompletableFuture.supplyAsync(() -> userService.getUser(userId), pool);
// CHAINING — thenApply (transform), thenAccept (consume), thenRun (trigger)
CompletableFuture<String> emailFuture = userFuture
    .thenApply(user -> user.getEmail())        // User -> String (transform)
    .thenApply(email -> email.toLowerCase())   // String -> String (transform)
    .thenApply(email -> "mailto:" + email);   // String -> String
// thenAccept — consume result, returns CompletableFuture<Void>
userFuture.thenAccept(user -> auditLog.record(user.getId()));
// thenRun — run after completion, no access to result
userFuture.thenRun(() -> metrics.increment("user.fetched"));
// COMBINING — thenCombine: wait for two futures, merge results
CompletableFuture<User> user    = CompletableFuture.supplyAsync(() -> getUser(id));
CompletableFuture<Account> acct = CompletableFuture.supplyAsync(() -> getAccount(id));
CompletableFuture<String> combined = user.thenCombine(acct,
    (u, a) -> u.getName() + " owns account: " + a.getNumber());
// allOf — wait for ALL, no combined result (use join() on each after)
CompletableFuture<Void> allDone = CompletableFuture.allOf(user, acct);
allDone.thenRun(() -> System.out.println("Both done"));
// anyOf — complete when ANY one completes (fastest wins)
CompletableFuture<Object> fastest = CompletableFuture.anyOf(user, acct);
// ERROR HANDLING — the right way to handle failures in async chains
// exceptionally — handle exception, return fallback value
CompletableFuture<User> safe = userFuture
    .exceptionally(ex -> {
        log.error("Failed to fetch user: {}", ex.getMessage());
        return User.anonymous();  // fallback
    });
// handle — called for BOTH success and failure
CompletableFuture<String> handled = userFuture
    .handle((user, ex) -> {
        if (ex != null) return "Error: " + ex.getMessage();
        return user.getName();
    });
// whenComplete — side effect for both, does NOT transform the result
userFuture.whenComplete((user, ex) -> {
    if (ex != null) metrics.increment("user.fetch.error");
    else            metrics.increment("user.fetch.success");
});  // result passes through unchanged
// The real-world pattern: call 3 services in parallel, combine results
CompletableFuture<User>        userF    = supplyAsync(() -> getUser(id),    pool);
CompletableFuture<Account>     accountF = supplyAsync(() -> getAccount(id), pool);
CompletableFuture<List<Txn>>   txnsF    = supplyAsync(() -> getTxns(id),   pool);
CompletableFuture.allOf(userF, accountF, txnsF).thenRun(() -> {
    User u = userF.join();   // join() = get() but wraps checked in unchecked
    Account a = accountF.join();
    List<Txn> t = txnsF.join();
    dashboard.build(u, a, t);
});
Q17: Interview Questions — Topic 5 ▼
💡 Ideal Answer (Amazon/GS Standard)

Q18: What is the difference between CompletableFuture.get() and join()? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Ideal Answer: Both block until the future completes. get() throws two checked exceptions: InterruptedException and ExecutionException — you must handle them with try-catch. join() throws CompletionException (unchecked) on failure and rethrows unchecked exceptions directly — no try-catch needed. In stream pipelines (which don't allow checked exceptions), join() is preferred. get() is better when you need to handle InterruptedException explicitly (e.g., allow graceful cancellation).

Q19: What is the difference between thenApply() and thenCompose()? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

Ideal Answer: thenApply(fn): fn takes a value, returns a plain value — CompletableFuture wraps it. Like Stream.map(). thenCompose(fn): fn takes a value, returns ANOTHER CompletableFuture — thenCompose unwraps/flattens it. Like Stream.flatMap(). Use thenCompose when the next step is itself an async operation. Example: getUser(id).thenCompose(user -> getAccount(user.getAccountId())) — if you used thenApply, you'd get CompletableFuture>.

ReentrantLock — synchronized's More Powerful Sibling

👨‍💻

Senior Mentor Tip

When is ReentrantLock better than synchronized?

synchronized is simpler and JVM-optimised — great for 90% of cases. ReentrantLock shines when you need: 1) tryLock(timeout) — attempt lock, bail out if not acquired (deadlock prevention). 2) lockInterruptibly() — allow the waiting thread to be interrupted. 3) Multiple Condition objects — separate wait queues for different conditions (vs one queue with synchronized). 4) Fairness policy — threads acquire in order (prevents starvation). The most common interview scenario: 'How do you prevent deadlock with ReentrantLock?' Answer: tryLock with timeout.

import java.util.concurrent.locks.*;
ReentrantLock lock = new ReentrantLock(true);  // true = fair mode
// Basic usage
lock.lock();
try {
    // critical section
} finally {
    lock.unlock();  // ALWAYS unlock in finally — never forget this
}
// tryLock() — deadlock prevention
if (lock.tryLock(500, TimeUnit.MILLISECONDS)) {
    try { /* critical section */ }
    finally { lock.unlock(); }
} else {
    // couldn't acquire — fallback, retry, log
}
// Multiple Conditions — producer/consumer with separate wait queues
ReentrantLock bufLock  = new ReentrantLock();
Condition notFull      = bufLock.newCondition();  // producer waits here when full
Condition notEmpty     = bufLock.newCondition();  // consumer waits here when empty
// Producer:
bufLock.lock();
try {
    while (isFull())  notFull.await();   // wait on notFull condition
    addItem(item);
    notEmpty.signal();                   // wake ONE consumer
} finally { bufLock.unlock(); }
// Consumer:
bufLock.lock();
try {
    while (isEmpty())  notEmpty.await();  // wait on notEmpty condition
    String item = removeItem();
    notFull.signal();                     // wake ONE producer
} finally { bufLock.unlock(); }

Atomic Classes — Lock-Free Thread Safety

👨‍💻

Senior Mentor Tip

CAS — Compare And Swap — the magic behind Atomic classes:

Atomic classes don't use locks. They use CPU-level CAS instructions. CAS says: 'If the current value is EXPECTED, update it to NEW and report success. Otherwise report failure.' AtomicInteger.compareAndSet(expected, update) — if value == expected, set to update atomically. Under low contention, this is MUCH faster than synchronized (no thread blocking/unblocking). Under HIGH contention (many threads fighting), performance degrades — use LongAdder instead for counters.

AtomicInteger counter  = new AtomicInteger(0);
AtomicLong    longCnt  = new AtomicLong(0L);
AtomicBoolean flag     = new AtomicBoolean(false);
// Common operations — all atomic (thread-safe without synchronized)
counter.incrementAndGet();    // ++counter atomic
counter.getAndIncrement();    // counter++ atomic
counter.addAndGet(5);         // counter += 5 atomic
counter.compareAndSet(10, 20); // if(counter==10) counter=20 — returns boolean
// AtomicReference — for atomic object references
AtomicReference<User> userRef = new AtomicReference<>(currentUser);
User oldUser = userRef.get();
User newUser = new User("newName");
boolean updated = userRef.compareAndSet(oldUser, newUser);  // atomic swap
// LongAdder — better than AtomicLong for high-contention counters
// Distributes increments across multiple cells, reduces CAS contention
LongAdder hits = new LongAdder();
hits.increment();       // from any thread
long total = hits.sum(); // get total (not exact if still updating)
Q20: Interview Questions — Topic 6 ▼
💡 Ideal Answer (Amazon/GS Standard)

Q21: What is the difference between synchronized and ReentrantLock? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Ideal Answer: synchronized: JVM-managed, automatic unlock on exit (even exception), no timeout, one implicit Condition queue, JVM can optimize (biased locking). ReentrantLock: explicit lock/unlock (must use try/finally), tryLock(timeout) for deadlock prevention, lockInterruptibly(), multiple Condition objects (separate wait queues), fairness option. Rule: use synchronized for simple mutual exclusion. Use ReentrantLock when you need timeouts, interruptibility, or multiple conditions. Never forget to unlock in a finally block.

Q22: What is ConcurrentHashMap and how is it thread-safe without locking the entire map? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

Ideal Answer: In Java 8+, ConcurrentHashMap uses bucket-level synchronization — only the affected bucket (linked list or tree node) is locked during write. Reads are fully non-blocking using volatile. This allows many concurrent reads and several concurrent writes (to different buckets). Null keys/values NOT allowed (unlike HashMap). compute(), merge(), computeIfAbsent() are atomic operations — safe for read-modify-write without external synchronization. For counters per key: map.merge(key, 1, Integer::sum) is atomic.
