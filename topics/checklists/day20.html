Question

Answer

How does @SpringBootApplication trigger auto-configuration?

@EnableAutoConfiguration → AutoConfigurationImportSelector → reads ImportCandidates → evaluates @Conditional annotations → registers passing configs as beans.

What does @ConditionalOnMissingBean do?

Only creates the auto-configured bean if no bean of that type already exists in the context. This is how users override Spring Boot defaults.

Bean lifecycle: what runs between constructor injection and @PostConstruct?

BeanPostProcessor.postProcessBeforeInitialization() runs for all registered BPPs. Then @PostConstruct runs.

JDK proxy vs CGLIB — when does CGLIB fail silently?

@Transactional or @Async on a final method. CGLIB subclasses the bean but cannot override final methods. The annotation is silently ignored.

The four @Transactional traps:

1) Self-invocation bypasses proxy. 2) Checked exceptions do not rollback by default. 3) Private/final methods ignored. 4) REQUIRES_NEW inside a test @Transactional does not work.

@WebMvcTest vs @SpringBootTest — what does @WebMvcTest exclude?

Excludes @Service, @Repository, JPA, and all non-web beans. Only loads the web layer. Dependencies must be @MockBean.

Why use Testcontainers instead of H2?

H2 does not support PostgreSQL-specific features (JSONB, array types, pg_trgm). Tests pass on H2 and fail on real PostgreSQL. Testcontainers runs the actual database.

What does @DynamicPropertySource do?

How do you test a void @Async method?

Awaitility — poll every 100ms until the expected side effect (DB record, counter) appears. Or replace TaskExecutor with SyncTaskExecutor in a @TestConfiguration.

GET vs POST: one is idempotent, one is not — which?

GET is idempotent (and safe). POST is neither safe nor idempotent by default.

400 vs 422 — one sentence each:

400: request is structurally malformed, server cannot parse it. 422: structure is fine but business rules reject it.

Cursor pagination vs offset — the performance difference:

RFC 7807 — what does the 'type' field contain?

A URI (not necessarily resolvable) that identifies the error category. e.g., https://api.example.com/errors/insufficient-funds.

JWT: what claims must you validate beyond the signature?

exp (not expired), iss (correct issuer), aud (correct audience), alg (whitelist — never accept 'none').

HS256 vs RS256 — which for microservices and why?

RS256. Private key stays on the auth server. Services only need the public key. A compromised service cannot forge tokens.

Where to store a refresh token in a browser app?

HttpOnly, Secure, SameSite=Strict cookie. HttpOnly means JavaScript cannot read it. Secure means HTTPS only. SameSite=Strict blocks CSRF.

What is BOLA?

Broken Object Level Authorisation — checking authentication but not ownership. User A accesses user B's resource by changing the ID in the URL.

Mass assignment — what causes it and the fix?

Using a JPA entity as @RequestBody — attacker sets internal fields like isAdmin. Fix: dedicated request DTO with only the fields the client is allowed to set.

Spring Security: JWT filter returns 401 on bad token — correct or not?

Not correct. The JWT filter clears the SecurityContext and calls chain.doFilter. The 401 comes from ExceptionTranslationFilter when the request hits a secured endpoint.

OAuth2 Client Credentials — what is it for?

Machine-to-machine auth. Service A authenticates as itself (no user) to call Service B. Uses client ID and secret to get a scoped access token from the auth server.

Day 16 — Spring Boot Internals (Questions 1–15)

1. What three annotations compose @SpringBootApplication?

@SpringBootConfiguration, @EnableAutoConfiguration, @ComponentScan.

2. What file does Spring Boot 2.7+ use to register auto-configurations?

META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports — one class per line.

3. What does @ConditionalOnClass do?

Only creates the bean if the specified class is on the classpath. Prevents auto-config from running when the dependency is absent.

4. Name the 7 bean lifecycle steps in order.

Instantiation → injection → BeanNameAware/BeanFactoryAware → BPP before-init → initialisation (@PostConstruct / afterPropertiesSet) → BPP after-init → destroy (@PreDestroy / destroy).

5. What does postProcessAfterInitialization return and why does it matter?

It can return a different object. This is how Spring AOP replaces beans with proxies — BPP returns the proxy, not the original bean.

6. Why does @Transactional on a private method do nothing?

CGLIB creates a subclass proxy and overrides methods. Private methods cannot be overridden in a subclass, so the proxy cannot intercept them.

7. What is the self-invocation proxy trap?

8. What does @Transactional(rollbackFor = Exception.class) do that the default does not?

The default only rolls back on RuntimeException and Error. Adding rollbackFor = Exception.class also rolls back on checked exceptions.

9. What is the prototype injection trap?

Injecting a prototype bean into a singleton. The prototype is created once at injection time and never refreshed — it behaves like a singleton. Fix: use ObjectProvider.

10. What does SimpleAsyncTaskExecutor do differently from ThreadPoolTaskExecutor?

SimpleAsyncTaskExecutor creates a brand new thread for every @Async call — no pooling, no bounds. Under load it causes unbounded thread creation and OOM.

11. What is AsyncUncaughtExceptionHandler for?

Catches exceptions thrown by void @Async methods. Without it, exceptions are silently swallowed. The handler receives the exception, method reference, and arguments.

12. Which Actuator endpoints must never be publicly accessible?

/env (exposes config), /heapdump (memory contents), /beans (architecture), /loggers (can change log levels at runtime).

13. Micrometer: Counter vs Gauge vs Timer — one sentence each.

Counter: only increases (request count). Gauge: current value that can go up or down (queue depth). Timer: duration distribution with count, sum, and percentiles.

14. What does @ConfigurationProperties relaxed binding mean?

payment.api-key, payment.apiKey, payment.api_key, and PAYMENT_API_KEY all bind to the same field. Spring translates casing automatically.

Day 17 — Testing (Questions 16–30)

16. @Mock vs @MockBean — one concrete difference.

@Mock creates a Mockito mock without Spring. @MockBean creates a mock AND replaces the bean in the Spring ApplicationContext. Use @Mock for pure unit tests, @MockBean in slice tests.

17. What does @WebMvcTest load and what does it not load?

Loads: controllers, filters, @ControllerAdvice, MockMvc, Jackson. Does NOT load: @Service, @Repository, JPA, @Component beans outside the web layer.

18. What does @AutoConfigureTestDatabase(replace=NONE) do?

Tells @DataJpaTest not to replace the DataSource with an embedded DB (H2). Used with Testcontainers so the test runs against a real database.

19. What does @DynamicPropertySource require to function?

20. WireMock: how do you test that a client retries exactly 3 times?

Use inScenario with state transitions: first two calls return 500 and advance state, third returns 200. Then verify(3, postRequestedFor(urlEqualTo(...))).

22. Awaitility vs CompletableFuture.get() — when each?

CompletableFuture.get() for @Async methods that return a future. Awaitility for void @Async methods where you poll a side effect (DB write, counter increment).

23. What does verifyNoMoreInteractions() assert?

That no further Mockito interactions occurred on the mock beyond what was explicitly verified. Catches unexpected calls.

24. What is the Object Mother pattern?

A class with named factory methods for common test states (pendingPayment(), completedPayment()). Centralises test data creation so adding a field only requires one change.

25. How do you assert JSON response body in a MockMvc test?

.andExpect(jsonPath("$.transactionId").value("TXN-123")) or .andExpect(content().json(expectedJson)) for full body comparison.

26. What is @SpyBean and one risk of using it?

28. What is TestEntityManager used for in @DataJpaTest?

Setting up test data by persisting entities directly, bypassing the repository layer. Also flushes to force DB writes within the test transaction.

29. How do you share a Testcontainers container across all tests in a suite?

Declare the container as a static field. Spring will reuse the context across tests with the same configuration. Add withReuse(true) for reuse across test runs.

30. What is contract-first API development?

Write the OpenAPI spec before writing any code. Both producer and consumer generate stubs from the spec and develop in parallel. Spec is the source of truth, not the implementation.

Day 18 — REST API Design (Questions 31–45)

31. Which HTTP methods are both safe AND idempotent?

GET, HEAD, OPTIONS. All three are safe (no state change) and idempotent (multiple calls = same state).

32. Is DELETE idempotent? What should the second DELETE return?

Yes, DELETE is idempotent. The second call should return 200 or 204 — not 404. The state (resource does not exist) is the same after both calls.

33. Is PATCH idempotent?

Not by default. A PATCH that sets a field to a value is idempotent. A PATCH that increments a field is not. Depends on the operation, not the method.

34. What status code for a successfully created resource? What header must accompany it?

201 Created. Must include Location header pointing to the newly created resource URI.

35. What does 202 Accepted mean?

Request received and will be processed asynchronously. Return a job ID or Location pointing to a job resource. The actual processing has not completed yet.

36. 409 Conflict — give two real examples.

Attempting to create a user with an email that already exists. Attempting to update a record with an outdated version number (optimistic locking conflict).

37. URI naming: two rules you must follow.

Nouns not verbs (/payments not /createPayment). Plural collections (/payments not /payment). Lowercase with hyphens (/payment-methods not /paymentMethods).

38. What is the main operational reason to choose URI versioning?

It is visible in logs, easy to route in API Gateway with a path predicate, and easy to test in a browser or curl. Simple to operate for teams.

39. Cursor pagination: what does the cursor encode and why is it opaque?

Encodes the last seen record's sort key (e.g., id + createdAt), Base64 encoded. Opaque so clients cannot construct or manipulate it — only the server defines cursor format.

40. What is the RFC 7807 'title' field vs the 'detail' field?

title is a human-readable summary of the error type — the same for all occurrences of that error. detail is specific to this particular occurrence — includes the actual values that caused the error.

41. What does ETag: no-cache mean in Cache-Control?

no-cache does not mean 'do not cache'. It means 'you may cache it but must revalidate before using the cached copy' — typically via If-None-Match with the ETag.

42. What HTTP status does a matching ETag return, and what is in the response body?

304 Not Modified. The response body is empty — the client uses its cached copy.

43. What is idempotency key used for in POST and where is it stored server-side?

Prevents duplicate processing on client retries. The processed result is stored in Redis keyed on the UUID for 24 hours. Duplicate key returns the original result without reprocessing.

44. Rate limit exceeded — correct status code and which header tells the client when to retry?

429 Too Many Requests. Include Retry-After header with the number of seconds to wait.

Day 19 — Security (Questions 46–60)

46. JWT structure: what are the three parts?

header (algorithm + token type), payload (claims: iss, sub, aud, exp, iat, jti, custom), signature (computed over header + payload with signing key).

47. What does the 'aud' claim in a JWT do and why must you validate it?

It specifies the intended audience — which service the token was issued for. Without validation, a token issued for the mobile app can be used against the payment backend.

48. Why is accepting alg: none a critical JWT vulnerability?

alg: none means no signature. An attacker can craft any payload, remove the signature, and set alg: none. Vulnerable libraries accept it. Always whitelist algorithms.

49. Access token TTL: why 15 minutes specifically?

Short enough to limit the damage window if stolen — an attacker has at most 15 minutes. Long enough for normal user sessions where the refresh token silently issues new access tokens.

50. What are the three practical ways to revoke a JWT?

1) Short TTL — damage window is small. 2) jti blocklist in Redis — instant revocation, O(1) lookup per request. 3) Opaque tokens — server-side lookup on every request, full revocation control.

51. OAuth2 Authorization Code Flow: why is the authorization code single-use and short-lived?

If the code is intercepted in the redirect, the attacker has a narrow window to use it. Single-use ensures the code cannot be replayed even if captured.

52. What does PKCE prevent?

Authorization code interception on mobile/SPA. An attacker who intercepts the code cannot exchange it for tokens because they do not have the code_verifier that was used to generate the code_challenge.

53. Spring Security: what does SessionCreationPolicy.STATELESS do?

Tells Spring not to create or use HTTP sessions. Every request must be authenticated independently via the JWT. Essential for stateless REST APIs.

54. Why does the JWT filter call chain.doFilter even when the token is invalid?

The filter's job is to populate the SecurityContext, not to reject requests. Rejection happens downstream in ExceptionTranslationFilter when an unauthenticated request hits a secured endpoint.

55. @PreAuthorize vs @PostAuthorize — when use each?

@PreAuthorize checks before the method runs — use for role/authority checks. @PostAuthorize checks the return value after the method — use for object ownership checks where you need to see what was returned.

56. BOLA — the two layers where you prevent it.

1) Query layer: always include the authenticated user's ID in the WHERE clause. 2) Security layer: @PostAuthorize or a custom @PreAuthorize expression verifying ownership.

57. Mass assignment — what is the root cause and the correct fix?

Root cause: using a JPA entity as @RequestBody so all JSON fields map to entity fields including security-sensitive ones. Fix: dedicated request DTO containing only client-settable fields.

58. CSRF: when does a stateless JWT API need CSRF protection?

It does not. CSRF attacks rely on cookies being sent automatically by the browser. A JWT in the Authorization header cannot be sent by a cross-site form post — the browser blocks it.

59. Service-to-service auth: JWT Client Credentials vs mTLS — one advantage each.

Client Credentials: works with existing OAuth2 infrastructure, tokens carry scopes, no certificate management. mTLS: identity proven at TLS layer, no token to steal, stronger for high-security environments.

60. What must you never log in a production backend service?

Authorization headers (contains tokens), request bodies for payment endpoints (financial data / PII), passwords, API keys, stack traces in responses, and any sensitive user data (emails, phone numbers) without masking.

Access control

BOLA prevention: repository query always includes WHERE owner_id = ? OR EXISTS (SELECT 1 FROM file_shares WHERE file_id = ? AND shared_with_user_id = ?) | Database enforces access, not application logic alone

Virus scanning

On confirm: publish to Kafka topic files.uploaded. Virus scanner consumer processes file, updates virusScanStatus. Downloads blocked until status = CLEAN. | Async scanning does not block upload confirmation. Downloads are safe.

Delete

PATCH /api/files/{fileId} with status=DELETED (soft delete). Scheduled job hard-deletes from S3 after 30 days. | Accidental deletion is recoverable. S3 lifecycle rules handle final cleanup.

Security

JWT filter validates token. @PreAuthorize checks ownership or share on every file operation. Pre-signed URLs include S3 bucket policy restricting content-type and max size. | Defence in depth: JWT + Spring Security + DB query scope + S3 policy

Trade-offs worth calling out:

Pre-signed URLs mean the client uploads directly to S3 — you lose the ability to inspect the file content before it lands in S3. Virus scanning must happen after upload and before download. The two-step (get-URL then confirm) adds a round trip. Soft delete adds storage cost but gives recovery capability. Sharing via database join means every download needs a DB query — you could cache the share check in Redis but that adds invalidation complexity when shares are revoked.

Part 4 — Mock Interview Round

Rules:

Read one question at a time. Give a full verbal answer before reading the model response. These six questions replicate a real 45-minute senior interview.

Q1: Explain how @Transactional works in Spring and describe three ways it can silently fail. [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

Spring implements @Transactional using a BeanPostProcessor that wraps the target bean in a CGLIB proxy at startup. When you call a @Transactional method on the proxy, the proxy starts a transaction, invokes the real method, and commits or rolls back based on the outcome. Three ways it silently fails. First: self-invocation. If you call a @Transactional method via 'this' from within the same class, you are calling the real object directly — the proxy is not involved, so no transaction starts. Second: private or final methods. CGLIB proxies work by subclassing the bean. Private methods cannot be inherited, final methods cannot be overridden, so the proxy cannot intercept them. The annotation is present but does nothing. Third: checked exceptions. By default, Spring only rolls back for RuntimeException and Error. If a checked exception is thrown mid-transaction, the transaction commits and the database is left in a partially-changed state. You must add rollbackFor = Exception.class to cover checked exceptions.

Q2: You have a Spring Boot service with 200 tests. The test suite takes 8 minutes to run. How do you approach improving it? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

I start by profiling — run with -Dtest.timings or use Maven Surefire's summary to see which tests are slowest. Usually the problem falls into two categories. The first is context loading. If many tests use @SpringBootTest when they should be @WebMvcTest or plain unit tests, each one starts the full application context. I audit and downgrade every test to the narrowest slice it needs. @WebMvcTest for controller tests, @DataJpaTest for repository tests, no Spring context at all for pure domain logic. The second is resource setup. If each test class spins up a fresh Testcontainers database, startup time multiplies. I move containers to static fields shared across the class or use a shared base class so the container starts once per JVM. Parallel test execution helps too — Maven Surefire supports parallel with forkCount or JUnit 5 parallel execution with junit.jupiter.execution.parallel.enabled=true. The goal is to run fast feedback tests (unit tests, slice tests) frequently and reserve the full @SpringBootTest integration suite for CI only.

Q3: A junior developer on your team returns a 200 OK response with an error field in the body when a payment fails. What do you tell them and why does it matter? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

I tell them this breaks HTTP semantics and creates real problems for clients. HTTP status codes are a machine-readable contract. Every HTTP client, load balancer, monitoring system, and API gateway treats 2xx as success. A 200 with an error body means: Resilience4j's circuit breaker sees a success and does not open; CloudWatch alarms on 5xx do not fire; client code that checks status before parsing the body proceeds to read a success body that is not there; API Gateway caches the response because 200 is cacheable. The fix is to map errors to the correct status codes — 400 for validation failures, 422 for business rule violations, 503 for gateway failures — with a consistent RFC 7807 error body. The client then gets a clear signal at the HTTP layer and can handle it without parsing the body. This is not a style preference. It is the difference between a system that is observable and operable and one that fails silently.

Q4: Describe how you would implement JWT authentication in a Spring Boot microservices system, from token issuance to validation. [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

Token issuance lives in a dedicated auth service, or we delegate entirely to a managed identity provider like Keycloak or Auth0. For service-issued tokens: user authenticates, auth service signs a JWT with a private key using RS256, and returns an access token (15 minute TTL) and a refresh token (7 days, stored server-side). Each microservice validates tokens independently. It fetches the auth server's JWKS endpoint once at startup and caches the public keys — keys include a kid header so the service knows which public key to use for verification. The JWT filter runs before the access control filters. It extracts the Bearer token, validates the signature, checks exp, iss, and aud, extracts roles from the claims, and puts an Authentication object in the SecurityContext. If validation fails, it clears the context and continues — ExceptionTranslationFilter handles the 401 for unauthenticated requests that reach secured endpoints. For service-to-service calls, each service uses OAuth2 Client Credentials to get its own scoped token. Spring's OAuth2AuthorizedClientManager handles token caching and refresh, so the application code just makes the WebClient call.

Q5: What is BOLA, can you give a concrete example from a payment system, and how do you test for it? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

BOLA — Broken Object Level Authorisation — is when the API checks that you are authenticated but does not check that you are authorised to access the specific resource you requested. In a payment system: user A is logged in with a valid JWT. They call GET /api/payments/TXN-456. TXN-456 belongs to user B. The API verifies the JWT, fetches the payment by ID, and returns user B's payment details — because it only checked 'are you authenticated', not 'does this payment belong to you'. The fix has two parts. At the query layer: always scope data queries to the authenticated user — repository.findByIdAndAccountId(id, currentUserId) returns nothing if the resource does not belong to the caller. At the security layer: a @PostAuthorize expression that verifies the returned object belongs to the caller. For testing: log in as user A, capture the ID of a payment belonging to user B, make the API call as user A, assert that you receive 404 — not the payment data. This test must exist for every resource endpoint in the system.

Q6: How would you design an API endpoint that handles a long-running operation — say, generating a large financial report? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

I use the job resource pattern. The client sends POST /api/reports with the report parameters. The server validates the request, creates a job record in the database with status PENDING, publishes a job message to Kafka, and immediately returns 202 Accepted with the job ID and a Location header pointing to GET /api/reports/jobs/{jobId}. The actual report generation runs asynchronously on a Kafka consumer — it processes the data, stores the report in S3, and updates the job status to COMPLETED with the download URL. The client polls GET /api/jobs/{jobId} to check progress. The job endpoint returns the current status and, when COMPLETED, a link to the report file. I add Cache-Control: no-cache on the job endpoint so intermediaries do not cache stale status. For the retry interval I include a Retry-After header on the job response suggesting how long to wait before the next poll — for a report that takes 30 seconds, I return Retry-After: 10 to avoid hammering. An alternative is webhooks: the client registers a callback URL and we POST the result when done. Webhooks are more efficient but require the client to have a reachable public endpoint, which is not always the case for server-to-server integrations.
