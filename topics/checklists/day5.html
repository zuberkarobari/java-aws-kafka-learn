The Mentor Explains: @SpringBootApplication Internals

This is asked at EVERY senior Spring Boot interview — know all three layers:

// @SpringBootApplication is shorthand for these three:
@SpringBootConfiguration   // = @Configuration — this class defines beans
@EnableAutoConfiguration   // = scan META-INF/spring/auto-configuration files
@ComponentScan             // = scan current package + sub-packages for beans
// How auto-configuration works (the full chain):
// 1. @EnableAutoConfiguration activates AutoConfigurationImportSelector
// 2. It reads META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports
//    (Spring Boot 2.7+) — lists hundreds of *AutoConfiguration classes
// 3. Each AutoConfiguration class has @Conditional annotations:
@Configuration
@ConditionalOnClass(DataSource.class)          // only if DataSource on classpath
@ConditionalOnMissingBean(DataSource.class)    // only if no DataSource bean yet defined
@ConditionalOnProperty(name="spring.datasource.url") // only if property set
public class DataSourceAutoConfiguration {
    @Bean
    public DataSource dataSource() { ... } // created ONLY when conditions met
}
// Result: if you add spring-boot-starter-data-jpa to pom.xml,
// Spring Boot automatically creates DataSource, EntityManagerFactory,
// TransactionManager — zero configuration from you.
// Bean scopes
@Component
@Scope("singleton")   // default — one instance per ApplicationContext
public class PaymentService { }
@Component
@Scope("prototype")   // new instance on every getBean() call
public class ReportBuilder { }
@Component
@Scope("request")     // new instance per HTTP request (web contexts only)
public class RequestContext { }

Dependency Injection — Constructor Injection Is the Right Choice

Why constructor injection wins over field injection:

// Field injection — convenient but problematic
@Service
public class PaymentService {
    @Autowired private UserRepository userRepo;   // hidden dependency
    @Autowired private TransactionRepository txRepo; // cannot test without Spring
}
// Constructor injection — the correct approach
@Service
public class PaymentService {
    private final UserRepository userRepo;        // final = immutable after creation
    private final TransactionRepository txRepo;
    // @Autowired optional if only one constructor (Spring Boot 2.0+)
    public PaymentService(UserRepository userRepo, TransactionRepository txRepo) {
        this.userRepo = userRepo;
        this.txRepo   = txRepo;
    }
    // Now testable: new PaymentService(mockUserRepo, mockTxRepo)
}
// @Qualifier — resolve ambiguity when multiple beans of same type exist
@Service
public class NotificationService {
    private final MessageSender sender;
    public NotificationService(@Qualifier("emailSender") MessageSender sender) {
        this.sender = sender;
    }
}
// Bean lifecycle hooks
@Component
public class CacheManager {
    @PostConstruct   // called after bean fully constructed and injected
    public void init() { loadCacheFromDatabase(); }
    @PreDestroy      // called before bean is destroyed (app shutdown)
    public void cleanup() { flushCacheToDisk(); }
}

Interview Questions — Topic 1

Q2: What are the three annotations that @SpringBootApplication combines? [Easy] ▼
💡 Ideal Answer (Amazon/GS Standard)

@SpringBootConfiguration (marks as configuration class), @EnableAutoConfiguration (triggers auto-configuration reading from META-INF), and @ComponentScan (scans current package for Spring components). The most important: @EnableAutoConfiguration — it reads auto-configuration candidates and applies @Conditional-guarded beans based on classpath, properties, and existing beans.

Q3: How does Spring Boot auto-configuration decide which beans to create? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

1) @EnableAutoConfiguration activates AutoConfigurationImportSelector. 2) It reads META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports listing all *AutoConfiguration classes. 3) Each AutoConfiguration class has @Conditional annotations — @ConditionalOnClass (only if class exists on classpath), @ConditionalOnMissingBean (only if no bean of that type already defined), @ConditionalOnProperty (only if property set). You can see which auto-configs applied and why: run with --debug flag.

Q4: What is the difference between singleton and prototype bean scope? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Singleton (default): one instance per ApplicationContext, created at startup (eager) or first request (lazy). Shared across all injections. Thread safety is YOUR responsibility. Prototype: new instance created every time the bean is requested. Not managed after creation — Spring does not call @PreDestroy on prototype beans. Use prototype for stateful beans where each caller needs its own instance (report builders, conversation objects). Injecting prototype into singleton is a trap — singleton gets one prototype instance forever.

Q5: What is the difference between @Component, @Service, @Repository, and @Controller? [Easy] ▼
💡 Ideal Answer (Amazon/GS Standard)

All are @Component specializations — functionally identical for component scanning. The difference is semantic and AOP-based: @Repository adds automatic exception translation (JPA exceptions translated to Spring DataAccessException hierarchy). @Controller marks MVC controllers for DispatcherServlet. @Service marks business logic — by convention only, no extra Spring behavior currently. Semantic annotations improve readability and allow targeted AOP pointcuts.

Q6: What is the self-invocation trap with Spring AOP proxies? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

Spring AOP works via proxies. When you call @Transactional or @Cacheable method from OUTSIDE the class, the call goes through the proxy — works as expected. When a method within the SAME class calls another method in the same class, it bypasses the proxy (direct method call on 'this') — @Transactional or @Cacheable on the called method has NO EFFECT. Fix: inject the bean into itself using @Autowired (or @Self in Quarkus), or use AspectJ weaving instead of proxy-based AOP.

The Complete REST Controller Pattern

// @RestController = @Controller + @ResponseBody on every method
@RestController
@RequestMapping("/api/v1/payments")
@Validated  // enables method-level validation
public class PaymentController {
    private final PaymentService paymentService;
    public PaymentController(PaymentService paymentService) {
        this.paymentService = paymentService;
    }
    // GET with path variable + query parameter
    @GetMapping("/{id}")
    public ResponseEntity<PaymentResponse> getPayment(
            @PathVariable @NotNull Long id,
            @RequestParam(required=false, defaultValue="false") boolean includeDetails) {
        return paymentService.findById(id, includeDetails)
            .map(ResponseEntity::ok)                   // 200 OK with body
            .orElse(ResponseEntity.notFound().build()); // 404 No body
    }
    // POST with request body validation
    @PostMapping
    public ResponseEntity<PaymentResponse> createPayment(
            @RequestBody @Valid PaymentRequest request) { // @Valid triggers Bean Validation
        PaymentResponse response = paymentService.process(request);
        URI location = URI.create("/api/v1/payments/" + response.getId());
        return ResponseEntity
            .created(location)    // 201 Created with Location header
            .body(response);
    }
    // Paginated list
    @GetMapping
    public Page<PaymentResponse> listPayments(
            @PageableDefault(size=20, sort="createdAt", direction=Sort.Direction.DESC)
            Pageable pageable,
            @RequestParam(required=false) String status) {
        return paymentService.findAll(status, pageable);
    }
}

Global Exception Handling with @ControllerAdvice

This is how production Spring Boot apps handle errors uniformly:

Without @ControllerAdvice: every controller catches its own exceptions — duplicated code, inconsistent error responses. With @ControllerAdvice: one class intercepts all exceptions across all controllers, formats them consistently, maps them to the right HTTP status code. This is the standard production pattern. Your error response should always include: timestamp, status, error code, message, and path. Never expose stack traces in production responses.

@ControllerAdvice  // intercepts exceptions from ALL controllers
public class GlobalExceptionHandler {
    // Validation failures from @Valid
    @ExceptionHandler(MethodArgumentNotValidException.class)
    @ResponseStatus(HttpStatus.BAD_REQUEST)
    public ErrorResponse handleValidationErrors(MethodArgumentNotValidException ex) {
        List<String> errors = ex.getBindingResult().getFieldErrors().stream()
            .map(fe -> fe.getField() + ": " + fe.getDefaultMessage())
            .collect(Collectors.toList());
        return new ErrorResponse(400, "VALIDATION_FAILED", errors.toString());
    }
    // Business rule violations
    @ExceptionHandler(PaymentException.class)
    public ResponseEntity<ErrorResponse> handlePaymentException(PaymentException ex) {
        return ResponseEntity
            .status(ex.getHttpStatus())
            .body(new ErrorResponse(ex.getHttpStatus().value(), ex.getCode(), ex.getMessage()));
    }
    // Resource not found
    @ExceptionHandler(EntityNotFoundException.class)
    @ResponseStatus(HttpStatus.NOT_FOUND)
    public ErrorResponse handleNotFound(EntityNotFoundException ex) {
        return new ErrorResponse(404, "NOT_FOUND", ex.getMessage());
    }
    // Last resort — any unhandled exception
    @ExceptionHandler(Exception.class)
    @ResponseStatus(HttpStatus.INTERNAL_SERVER_ERROR)
    public ErrorResponse handleGeneral(Exception ex, HttpServletRequest req) {
        log.error("Unhandled exception for {}: {}", req.getRequestURI(), ex.getMessage(), ex);
        return new ErrorResponse(500, "INTERNAL_ERROR", "An unexpected error occurred");
    }
}

Interview Questions — Topic 2

Q7: What is the difference between @Controller and @RestController? [Easy] ▼
💡 Ideal Answer (Amazon/GS Standard)

@RestController is a convenience annotation combining @Controller and @ResponseBody. @Controller marks a class as an MVC controller — by default, methods return view names (template resolution). @ResponseBody on a method tells Spring to serialize the return value directly to the HTTP response body (JSON/XML). @RestController applies @ResponseBody to ALL methods — every method returns response body directly, no view resolution. Use @Controller for Thymeleaf/JSP apps, @RestController for REST APIs.

Q8: What happens when @Valid fails? How does a @ControllerAdvice catch it? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

When @Valid or @Validated fails, Spring throws MethodArgumentNotValidException before the controller method body executes. Without @ControllerAdvice: Spring returns 400 with its default error body. With @ControllerAdvice: @ExceptionHandler(MethodArgumentNotValidException.class) intercepts it globally. You then extract field errors from ex.getBindingResult().getFieldErrors() and return a structured error response. This applies to all controllers automatically.

Q9: What HTTP status codes should a REST API return for each scenario? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

200 OK: successful GET/PUT. 201 Created: successful POST (include Location header). 204 No Content: successful DELETE or PUT with no body. 400 Bad Request: validation failure, malformed request. 401 Unauthorized: no valid credentials. 403 Forbidden: authenticated but insufficient permissions. 404 Not Found: resource does not exist. 409 Conflict: duplicate resource or business rule violation. 422 Unprocessable Entity: well-formed but semantically invalid. 500 Internal Server Error: unexpected server failure.

The N+1 Problem — The Most Asked JPA Question

N+1 is one of the most common production performance bugs in Spring Boot apps:

You have 1 query to fetch N departments. Then for each department, JPA fires 1 query to fetch its employees. Total: 1 + N queries — O(N) database round trips instead of O(1). With 1000 departments, that is 1001 queries. Application seems fine in development with 10 departments. Deployed to production with 100,000 departments: 100,001 queries per request, database melts. The fix: use JOIN FETCH or @EntityGraph to load everything in one query.

@Entity
public class Department {
    @Id @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;
    private String name;
    // FetchType.LAZY (default for @OneToMany) = load on access
    @OneToMany(mappedBy = "department", fetch = FetchType.LAZY)
    private List<Employee> employees;
}
// N+1 PROBLEM — 1 query for departments + N queries for employees
List<Department> depts = deptRepo.findAll(); // SELECT * FROM departments — 1 query
for (Department d : depts) {
    d.getEmployees().size(); // SELECT * FROM employees WHERE dept_id = ? — N queries!
}
// FIX 1: JOIN FETCH in JPQL — one query with JOIN
// In repository:
@Query("SELECT DISTINCT d FROM Department d JOIN FETCH d.employees")
List<Department> findAllWithEmployees();
// Generates: SELECT d.*, e.* FROM departments d JOIN employees e ON e.dept_id = d.id
// Everything loaded in ONE query
// FIX 2: @EntityGraph — declarative JOIN FETCH (no JPQL needed)
@EntityGraph(attributePaths = {"employees"})
@Query("SELECT d FROM Department d")
List<Department> findAllWithEmployeesGraph();
// FIX 3: @BatchSize — not one query but fewer queries (batch loading)
@OneToMany(mappedBy = "department", fetch = FetchType.LAZY)
@BatchSize(size = 50) // load 50 departments' employees in one IN query
private List<Employee> employees;
// Instead of N queries: SELECT * FROM employees WHERE dept_id IN (?,?,?...?) batches of 50

Fetch Type Defaults — Know These by Heart

Relationship

Default FetchType and Why

@OneToMany

LAZY — could be millions of child records; loading eagerly by default would be catastrophic

@ManyToMany

LAZY — same reason; join table could produce massive result sets

@ManyToOne

EAGER — typically one parent record; loading it is cheap and almost always needed

@OneToOne

EAGER — one related record; usually needed alongside the owner

LazyInitializationException — The Most Confusing JPA Runtime Error:

Occurs when you access a LAZY collection or association OUTSIDE a Hibernate session. Example: service method loads entity in one transaction, closes session, then controller accesses entity.getOrders() — session is gone, Hibernate cannot load the lazy collection. Three fixes: 1) Use @Transactional on the service method to keep session open. 2) Use JOIN FETCH to eagerly load what you need. 3) Use DTO projection — select only the fields you need, avoiding entities altogether. Option 3 is the production-grade choice for read-heavy endpoints.

// Entity state lifecycle
// TRANSIENT: new object, not tracked by Hibernate
Employee emp = new Employee("Zuber");  // TRANSIENT
// MANAGED: associated with persistence context (open session)
em.persist(emp);          // MANAGED — Hibernate tracks every field change
emp.setName("Md Zuber");  // dirty checking — Hibernate will auto-UPDATE on flush
// DETACHED: session closed, object still in memory but not tracked
em.close();               // emp is now DETACHED
emp.setName("New Name"); // no tracking — change will NOT be persisted
// REMOVED: scheduled for DELETE
em.remove(em.merge(emp)); // re-attach then remove — DELETE on flush
// Spring Data Repository patterns — know all three query methods
// 1. Derived queries — Spring parses method name to generate JPQL
List<Employee> findByDepartmentNameAndSalaryGreaterThan(String dept, double salary);
Optional<Employee> findByEmailIgnoreCase(String email);
// 2. @Query with JPQL
@Query("SELECT e FROM Employee e WHERE e.dept.name = :dept AND e.salary > :min")
List<Employee> findHighEarners(@Param("dept") String dept, @Param("min") double min);
// 3. Native SQL (use sparingly — breaks portability)
@Query(value="SELECT * FROM employees WHERE JSON_CONTAINS(skills, :skill)",
       nativeQuery=true)
List<Employee> findBySkill(@Param("skill") String skill);
// DTO Projection — no entity, no lazy loading issues
public interface EmployeeSummary {
    String getName();
    String getDepartmentName();
    BigDecimal getSalary();
}
List<EmployeeSummary> findProjectedByDepartmentName(String dept);

Interview Questions — Topic 3

Q10: What is the N+1 problem and how do you fix it? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

N+1: fetching N parent records, then firing N separate queries to load each parent's lazy collection — 1 + N database round trips instead of 1. Caused by accessing LAZY associations in a loop. Fix 1: JOIN FETCH in JPQL query — loads everything in one query. Fix 2: @EntityGraph — declarative JOIN FETCH. Fix 3: @BatchSize — batches N queries into ceil(N/batchSize) IN queries. Detection: enable Hibernate SQL logging and count queries per request. Production tool: Hibernate Statistics or p6spy proxy.

Q11: What is LazyInitializationException and how do you prevent it? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

Thrown when accessing a LAZY association after the Hibernate session is closed. Example: entity loaded in service method, session closes, controller accesses lazy collection — session gone. Prevention: 1) @Transactional on service method keeps session open for the method's duration. 2) JOIN FETCH or @EntityGraph to eagerly load what you need. 3) Use DTO projections (interfaces or classes) — select exactly the fields needed, no entity, no lazy loading risk. Option 3 is best for read-heavy endpoints.

Q12: What are the JPA cascade types and what does CascadeType.ALL do? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

CascadeType.PERSIST, MERGE, REMOVE, REFRESH, DETACH — each propagates that operation from parent to children. CascadeType.ALL = all five combined. Use on @OneToMany owned collections: saving a Department automatically saves its Employees. Danger: CascadeType.REMOVE on a shared entity (e.g., ManyToMany) deletes shared records unexpectedly. NEVER use CascadeType.ALL on @ManyToMany — deletes will cascade to records shared by other parents. Use PERSIST and MERGE only on many-to-many.

Propagation Levels — The Heart of @Transactional

Propagation controls what happens when a @Transactional method calls another @Transactional method:

This is the most nuanced @Transactional topic. Think of it as: 'what is the relationship between the outer transaction and the inner transaction?' The vast majority of real code uses REQUIRED (default). REQUIRES_NEW is used for audit logging (must commit even if outer transaction rolls back). NESTED is used for savepoints. Know all six by name — interviewers ask you to describe the difference between REQUIRED and REQUIRES_NEW in a scenario.

Propagation

Behaviour — What Happens to the Transaction

REQUIRED (default)

Join existing transaction if one exists. Create new one if none exists. Both methods run in the SAME transaction — one rollback rolls both back.

REQUIRES_NEW

Always creates a NEW transaction. Suspends any existing transaction. The new transaction commits or rolls back INDEPENDENTLY. Use for: audit logging, event publishing that must persist even if caller rolls back.

NESTED

Creates a savepoint within the existing transaction. Inner method can roll back to savepoint without rolling back the outer transaction. Requires JDBC savepoints — not all databases support this.

SUPPORTS

Runs in a transaction if one exists. Runs non-transactionally if none exists. Use for read-only methods that can work either way.

NOT_SUPPORTED

Always runs non-transactionally. Suspends existing transaction if present. Use for operations that must not participate in a transaction (e.g., sending an external API call that cannot be rolled back).

NEVER

Must NOT run in a transaction. Throws IllegalTransactionStateException if a transaction exists. Strict enforcement of non-transactional execution.

MANDATORY

MUST run within an existing transaction. Throws if no transaction present. Use to enforce that callers always provide a transaction.

// @Transactional internals — AOP proxy wraps your bean
@Service
public class PaymentService {
    @Transactional  // REQUIRED by default
    public void processPayment(PaymentRequest req) {
        // Spring creates a proxy. When this method is called from outside:
        // 1. Proxy intercepts call
        // 2. Begins transaction (or joins existing)
        // 3. Delegates to real method
        // 4. On success: commit
        // 5. On RuntimeException: rollback
        deductBalance(req);
        auditLog(req);   // if this throws, EVERYTHING rolls back
    }
    // REQUIRES_NEW — audit log must persist even if processPayment rolls back
    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void auditLog(PaymentRequest req) {
        // runs in its OWN transaction — commits independently
    }
    // Rollback rules — by default only RuntimeException triggers rollback
    @Transactional(rollbackFor = Exception.class) // rollback on ANY exception
    public void processWithChecked() throws IOException {
        // IOException (checked) would NOT rollback by default
        // rollbackFor = Exception.class makes it rollback on ALL exceptions
    }
    // THE SELF-INVOCATION TRAP
    public void outer() {
        inner(); // DOES NOT go through proxy — @Transactional on inner() IGNORED
    }
    @Transactional
    public void inner() { /* transaction here has NO EFFECT when called from outer() */ }
    // FIX: inject self and call through the proxy
    @Autowired
    private PaymentService self; // Spring injects the PROXY, not 'this'
    public void outer() {
        self.inner(); // goes through proxy — @Transactional works
    }
}

Interview Questions — Topic 4

Q13: What is the difference between REQUIRED and REQUIRES_NEW propagation? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

REQUIRED (default): joins existing transaction if present, creates new one if not. Both inner and outer methods share one transaction — any rollback affects both. REQUIRES_NEW: always creates a fresh transaction, suspends any existing one. Inner method commits or rolls back independently of outer. Use case: audit logging that must persist even if the main business operation rolls back. REQUIRED for business logic. REQUIRES_NEW for side effects that must be durable regardless of main operation outcome.

Q14: Which exceptions trigger @Transactional rollback by default? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

By default: unchecked exceptions only — RuntimeException and its subclasses, and Error. Checked exceptions (IOException, SQLException) do NOT trigger rollback by default — this surprises many developers. Override with: @Transactional(rollbackFor = Exception.class) to rollback on any exception, or @Transactional(noRollbackFor = SpecificRuntimeException.class) to suppress rollback for specific runtime exceptions.

Q15: What is the self-invocation problem with @Transactional? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

@Transactional works through Spring AOP proxies. External calls to a @Transactional method go through the proxy which starts/joins transactions. When method A in the same class calls method B (also @Transactional) using 'this', it bypasses the proxy — no transaction management for B. Fixes: 1) Inject the bean into itself (@Autowired PaymentService self) — Spring injects the proxy. 2) Move method B to a separate Spring bean. 3) Use AspectJ compile-time weaving (no proxies needed). Most teams prefer option 2 — separating concerns is good design anyway.

The Full JWT Authentication Flow

Know this sequence end-to-end — interviewers walk through it step by step:

Login request arrives, authentication filter intercepts it, UserDetailsService loads user from DB, passwords are compared with PasswordEncoder, on success UsernamePasswordAuthenticationToken is created and stored in SecurityContext, JWT is generated and returned to client. On subsequent requests: OncePerRequestFilter extracts JWT from Authorization header, validates signature and expiry, loads UserDetails, sets Authentication in SecurityContext, DispatcherServlet dispatches normally. This entire flow is what runs on EVERY protected request.

// STEP 1: Security configuration — define filter chain
@Configuration
@EnableWebSecurity
public class SecurityConfig {
    private final JwtAuthFilter jwtAuthFilter;  // our custom filter
    private final UserDetailsService userDetailsService;
    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        return http
            .csrf(csrf -> csrf.disable())           // stateless JWT — no CSRF needed
            .sessionManagement(sm -> sm
                .sessionCreationPolicy(SessionCreationPolicy.STATELESS)) // no session
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/api/auth/**").permitAll()  // login/register open
                .requestMatchers("/api/admin/**").hasRole("ADMIN")
                .anyRequest().authenticated())          // everything else needs auth
            .addFilterBefore(jwtAuthFilter, UsernamePasswordAuthenticationFilter.class)
            .build();
    }
    @Bean
    public PasswordEncoder passwordEncoder() {
        return new BCryptPasswordEncoder(); // NEVER store plain passwords
    }
}
// STEP 2: JWT filter — runs on every request
@Component
public class JwtAuthFilter extends OncePerRequestFilter {
    @Override
    protected void doFilterInternal(HttpServletRequest req,
            HttpServletResponse res, FilterChain chain)
            throws ServletException, IOException {
        String authHeader = req.getHeader("Authorization");
        if (authHeader == null || !authHeader.startsWith("Bearer ")) {
            chain.doFilter(req, res); // no JWT — let other filters handle
            return;
        }
        String jwt = authHeader.substring(7);
        String username = jwtService.extractUsername(jwt);
        // Only set auth if not already authenticated
        if (username != null && SecurityContextHolder.getContext().getAuthentication() == null) {
            UserDetails userDetails = userDetailsService.loadUserByUsername(username);
            if (jwtService.isTokenValid(jwt, userDetails)) {
                // Create authentication token and store in SecurityContext
                UsernamePasswordAuthenticationToken authToken =
                    new UsernamePasswordAuthenticationToken(userDetails, null,
                        userDetails.getAuthorities());
                authToken.setDetails(new WebAuthenticationDetailsSource().buildDetails(req));
                SecurityContextHolder.getContext().setAuthentication(authToken);
            }
        }
        chain.doFilter(req, res); // continue filter chain
    }
}
// STEP 3: UserDetailsService — where you load user from YOUR database
@Service
public class UserDetailsServiceImpl implements UserDetailsService {
    private final UserRepository userRepo;
    @Override
    public UserDetails loadUserByUsername(String email) throws UsernameNotFoundException {
        return userRepo.findByEmail(email)
            .map(user -> User.builder()  // Spring Security User builder
                .username(user.getEmail())
                .password(user.getPasswordHash()) // BCrypt hash from DB
                .roles(user.getRoles().toArray(new String[0]))
                .build())
            .orElseThrow(() -> new UsernameNotFoundException("User not found: " + email));
    }
}

Interview Questions — Topic 5

Q16: Walk me through the full JWT request flow in Spring Security. [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

1) Client sends request with 'Authorization: Bearer ' header. 2) JwtAuthFilter (extends OncePerRequestFilter) intercepts every request. 3) Filter extracts JWT from header, calls JwtService.extractUsername(). 4) Loads UserDetails from UserDetailsService (database call). 5) Validates token: signature valid? not expired? username matches? 6) Creates UsernamePasswordAuthenticationToken with user's authorities. 7) Sets it in SecurityContextHolder — marks request as authenticated. 8) Calls chain.doFilter() — request proceeds to controller. 9) SecurityContextHolder is cleared after response. Each step is stateless — no session stored server-side.

Q17: Why disable CSRF for JWT-based REST APIs? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

CSRF protection is needed for session-based apps where browsers automatically send session cookies (attacker can forge requests using victim's cookie). With JWT: the token is in the Authorization header, not a cookie — browsers don't automatically send it. Cross-site malicious pages cannot access the JWT (same-origin policy blocks headers). So CSRF attacks are not possible with header-based JWT authentication. Session management is set to STATELESS — no server-side sessions created or used.

Q18: What is @PreAuthorize and how does it differ from URL-based security? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

@PreAuthorize uses Spring EL to evaluate authorization BEFORE the method executes. More powerful than URL patterns: can access method parameters (#paramName), check complex conditions, verify the requesting user is the resource owner (authentication.principal.id == #userId). URL patterns in SecurityFilterChain are coarser-grained and path-based only. Best practice: use URL patterns for coarse access control (public vs authenticated), @PreAuthorize for fine-grained business rule authorization.

Interview Questions — Topic 6

Q19: What is the difference between @Mock and @MockBean? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

@Mock (Mockito): creates a Mockito mock without Spring. Used in unit tests with @ExtendWith(MockitoExtension.class). No Spring context — fast. @MockBean (Spring): creates a Mockito mock AND registers it as a Spring bean in the ApplicationContext, replacing any existing bean of that type. Used in @SpringBootTest and @WebMvcTest when you need to mock a dependency that Spring needs to inject. @InjectMocks (Mockito): injects @Mock fields into the annotated class — for pure unit tests without Spring.

Q20: When would you use @WebMvcTest vs @SpringBootTest? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

@WebMvcTest loads only the web layer (controllers, filters, security). Service and repository beans are not created — mock them with @MockBean. Fast, focused. Use for testing controller logic, request mapping, validation, and error handling in isolation. @SpringBootTest loads the full application context — all beans, all configurations. Slow. Use for integration tests verifying full request-to-database flow, auto-configuration correctness, or end-to-end behavior. Rule: unit tests and @WebMvcTest for most tests. @SpringBootTest for a few critical integration tests.

Q21: How does MockMvc work and what can it assert? [Easy] ▼
💡 Ideal Answer (Amazon/GS Standard)

MockMvc simulates HTTP requests to your controllers without starting a real HTTP server — it calls your controller directly through the Spring MVC dispatch infrastructure. perform(request): builds and executes the request. andExpect: chain assertions — status().isOk(), jsonPath(expression).value(expected), content().contentType(mediaType), header().string(name, value), cookie().exists(name). andDo(print()): logs the full request/response for debugging. Works with @WebMvcTest or @SpringBootTest with MockMvc autowired.
