    @MockBean PaymentService paymentService;
    @MockBean PaymentValidator validator;
    @Test
    void processPayment_validRequest_returns201() throws Exception {

        PaymentRequest req = PaymentRequest.builder()
            .amount(new BigDecimal("500.00"))
            .currency("INR")
            .accountId("ACC-001")
            .build();
        PaymentResult result = PaymentResult.success("TXN-123", req.getAmount());
        given(paymentService.process(any(PaymentRequest.class))).willReturn(result);
        // Act + Assert

                .contentType(MediaType.APPLICATION_JSON)
                .content(objectMapper.writeValueAsString(req))
                .header("X-API-Key", "valid-key"))
            .andExpect(status().isCreated())
            .andExpect(jsonPath("$.transactionId").value("TXN-123"))
            .andExpect(jsonPath("$.amount").value(500.00))

        String invalidBody = "{\"currency\":\"INR\",\"accountId\":\"ACC-001\"}";
        mockMvc.perform(post("/api/payments")
                .contentType(MediaType.APPLICATION_JSON)
                .content(invalidBody))
            .andExpect(status().isBadRequest())
            .andExpect(jsonPath("$.errors[0].field").value("amount"));

Q1: When would you use @WebMvcTest vs @SpringBootTest and why does it matter? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

@WebMvcTest loads only the web layer — the controller, filters, @ControllerAdvice, and MockMvc — and excludes services, repositories, and JPA. It is fast because the context is small, and focused because it only tests what a controller should be tested for: request mapping, input validation, response serialisation, and exception handling. @SpringBootTest loads the full context and is appropriate for end-to-end integration tests where you want to verify that all layers work together. The cost is speed — full context startup can take seconds. The other reason to care about the distinction: with @WebMvcTest your service is a @MockBean, so you are testing the controller in complete isolation. If the service has a bug, the controller test still passes. That is the point — each layer is tested independently. Mixing them by using @SpringBootTest for controller tests that should be @WebMvcTest gives you a slower test suite and tests that fail for the wrong reasons.

Section 2 — @MockBean, @SpyBean and Mockito Patterns

@MockBean vs @SpyBean

@MockBean | @SpyBean

What it creates

A full Mockito mock — all methods return null/0/false by default | A Mockito spy wrapping the real Spring bean

Real method execution

Never — all calls are stubbed or return defaults | Yes — unstubbed calls execute real code

Effect on context

Replaces the bean in the ApplicationContext | Wraps the existing bean in the context

Use when

Isolating the layer under test from dependencies | Verifying calls to a real dependency without replacing its behaviour

Risk

Over-mocking — tests that pass even when integration is broken | Real methods run — test can fail for reasons unrelated to the unit under test

Common use

@MockBean PaymentService in @WebMvcTest | @SpyBean EmailService to verify send() was called but still execute it

// Mockito patterns you must know
// 1. Basic stubbing
given(paymentService.process(any(PaymentRequest.class)))
    .willReturn(PaymentResult.success("TXN-123", new BigDecimal("500")));
// 2. Stubbing with argument matching
given(paymentService.findById(eq("TXN-123"))).willReturn(Optional.of(payment));
given(paymentService.findById(argThat(id -> id.startsWith("TXN-"))))
    .willReturn(Optional.of(payment));
// 3. Throwing exceptions
given(paymentService.process(any())).willThrow(new PaymentGatewayException("timeout"));
// For void methods:
doThrow(new NotificationException("SMS failed"))
    .when(notificationService).sendSms(anyString(), anyString());
// 4. Verify interactions
verify(paymentService, times(1)).process(any(PaymentRequest.class));
verify(notificationService, never()).sendEmail(anyString());
verifyNoMoreInteractions(paymentService);
verifyNoInteractions(fraudService);  // assert zero calls
// 5. ArgumentCaptor — capture what was actually passed
@Test
void processPayment_capturesCorrectAuditRecord() {
    ArgumentCaptor<AuditRecord> captor = ArgumentCaptor.forClass(AuditRecord.class);
    paymentService.process(validRequest());
    verify(auditService).record(captor.capture());
    AuditRecord captured = captor.getValue();
    assertThat(captured.getAmount()).isEqualTo(new BigDecimal("500.00"));
    assertThat(captured.getStatus()).isEqualTo("COMPLETED");
    assertThat(captured.getTimestamp()).isCloseTo(Instant.now(), within(5, SECONDS));
}
// 6. Stubbing sequential calls — first call succeeds, second throws
given(gatewayClient.submit(any()))
    .willReturn(SubmitResponse.success("TXN-123"))
    .willThrow(new GatewayException("rate limited"));
// 7. Answer — compute return value from the actual argument
given(paymentRepository.save(any(Payment.class)))
    .willAnswer(invocation -> {
        Payment p = invocation.getArgument(0);
        p.setId(UUID.randomUUID().toString()); // simulate DB-assigned ID
        return p;
    });
// 8. @SpyBean — real bean, verifiable calls
@SpyBean NotificationService notificationService;
@Test
void processPayment_notifiesUserOnSuccess() {
    paymentService.process(validRequest()); // calls real NotificationService
    // Verify the real method was called with the right args
    verify(notificationService).sendEmail(eq("user@example.com"), contains("SUCCESS"));
}
Q2: What is the difference between @MockBean and @SpyBean, and when would you reach for @SpyBean? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

@MockBean replaces a bean in the Spring context with a full Mockito mock — all methods return null or zero by default and nothing real executes. @SpyBean wraps the real Spring bean in a Mockito spy — unstubbed calls execute the real code, but you can verify calls and stub specific methods. I reach for @SpyBean when I want to assert that a method was called with specific arguments, but I still want the real behaviour to execute. The most common case in production code: I have a @SpyBean on an audit or notification service where I want to verify that it was called with the right parameters, but I also want the actual email or record to be created so the test catches any downstream failures. The risk with @SpyBean is that real code runs — if the real code has external dependencies, the test can fail for reasons unrelated to what you are testing. In those cases, @MockBean with explicit verification is cleaner.

Q3: Why use Testcontainers instead of H2 for repository tests? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

H2 is not PostgreSQL. It has different SQL dialect rules, it does not support PostgreSQL-specific features like JSONB columns, array types, or pg_trgm full-text search, and its query planner behaviour is completely different. A suite that passes on H2 can fail against real PostgreSQL in a way that is very hard to debug — the test gave you false confidence. Testcontainers starts a real Docker container of the actual database version you deploy. The tests run against the same engine, same extensions, and same behaviour as production. The cost is speed — Docker startup adds a few seconds per test class. The fix is to share a single container across the test suite with a static @Container field. I always use Testcontainers for anything that touches the database, Kafka, or Redis. H2 is acceptable for pure unit tests of domain logic that does not involve any database-specific queries.

Why WireMock

// pom.xml: wiremock-spring-boot (or wiremock-standalone)
// ── WireMock with @SpringBootTest ──
@SpringBootTest(webEnvironment = RANDOM_PORT)
@WireMockTest(httpPort = 9090)          // starts WireMock on port 9090
class PaymentGatewayClientTest {
    // Point your client at WireMock instead of the real gateway
    // application-test.properties: payment.gateway.url=http://localhost:9090
    @Autowired PaymentGatewayClient gatewayClient;
    @Test
    void submitPayment_success_returnsTransactionId() {
        // Stub: when POST /v1/charge is called, return success JSON
        stubFor(post(urlEqualTo("/v1/charge"))
            .withHeader("Authorization", matching("Bearer .*"))
            .withRequestBody(matchingJsonPath("$.amount", equalTo("500.00")))
            .willReturn(aResponse()
                .withStatus(200)
                .withHeader("Content-Type", "application/json")
                .withBody("{\"id\":\"ch_123\",\"status\":\"succeeded\"}")));
        PaymentResult result = gatewayClient.submit(PaymentRequest.of("500.00", "INR"));
        assertThat(result.getTransactionId()).isEqualTo("ch_123");
        assertThat(result.getStatus()).isEqualTo("succeeded");
        // Verify WireMock received exactly one request
        verify(1, postRequestedFor(urlEqualTo("/v1/charge")));
    }
    @Test
    void submitPayment_gatewayTimeout_triggersCircuitBreaker() {
        // Simulate a timeout
        stubFor(post(urlEqualTo("/v1/charge"))
            .willReturn(aResponse()
                .withFixedDelay(5000)   // 5 second delay — exceeds client timeout
                .withStatus(200)));
        // Client has a 2-second timeout — should fall back
        assertThatThrownBy(() -> gatewayClient.submit(PaymentRequest.of("500", "INR")))
            .isInstanceOf(GatewayTimeoutException.class);
    }
    @Test
    void submitPayment_gateway500_retriesThreeTimes() {
        // First two calls fail, third succeeds
        stubFor(post(urlEqualTo("/v1/charge"))
            .inScenario("retry-scenario")
            .whenScenarioStateIs(STARTED)
            .willReturn(serverError())
            .willSetStateTo("attempt-2"));
        stubFor(post(urlEqualTo("/v1/charge"))
            .inScenario("retry-scenario")
            .whenScenarioStateIs("attempt-2")
            .willReturn(serverError())
            .willSetStateTo("attempt-3"));
        stubFor(post(urlEqualTo("/v1/charge"))
            .inScenario("retry-scenario")
            .whenScenarioStateIs("attempt-3")
            .willReturn(okJson("{\"id\":\"ch_retry\",\"status\":\"succeeded\"}")));
        PaymentResult result = gatewayClient.submit(PaymentRequest.of("500", "INR"));
        assertThat(result.getTransactionId()).isEqualTo("ch_retry");
        verify(3, postRequestedFor(urlEqualTo("/v1/charge")));
    }
}

Section 5 — Testing @Async, @Transactional and @Cacheable

Testing @Async Code

The challenge:

@Async methods run on a different thread. If your test asserts immediately after calling the async method, the async work may not have completed yet. The test will either fail intermittently or require Thread.sleep() — both are bad.

// ── Option 1: CompletableFuture.get() — blocks until result is available ──
@Test
void sendNotification_completesSuccessfully() throws Exception {
    CompletableFuture<Boolean> future = notificationService.sendEmail(
        "user@example.com", "Your payment succeeded");
    // get() with timeout — test fails fast if async work hangs
    Boolean result = future.get(5, TimeUnit.SECONDS);
    assertThat(result).isTrue();
    verify(emailClient, times(1)).send(eq("user@example.com"), anyString());
}
// ── Option 2: Awaitility — poll until condition is true ──
// Best for void @Async methods with side effects (DB write, Kafka publish)
@Test
void processEventAsync_persistsToDatabase() {
    eventService.processAsync(new PaymentEvent("TXN-123", "COMPLETED"));
    // Poll every 100ms, timeout after 5s
    await().atMost(5, SECONDS)
           .pollInterval(100, MILLISECONDS)
           .untilAsserted(() ->
               assertThat(eventRepository.findByTransactionId("TXN-123")).isPresent());
}
// ── Option 3: SyncTaskExecutor — replace async executor in tests ──
// All @Async methods run synchronously on the calling thread
@TestConfiguration
public class SynchronousAsyncConfig {
    @Bean
    @Primary
    public TaskExecutor taskExecutor() {
        return new SyncTaskExecutor(); // executes immediately on calling thread
    }
}
@SpringBootTest
@Import(SynchronousAsyncConfig.class)
class NotificationServiceTest {
    // All @Async methods are now synchronous — no waiting needed
    @Test
    void asyncNotification_isCalledSynchronously() {
        notificationService.sendEmail("user@example.com", "message");
        verify(emailClient, times(1)).send(any(), any());
    }
}

Testing @Transactional — Rollback Behaviour

Default behaviour in @DataJpaTest and @SpringBootTest:

@DataJpaTest wraps each test in a transaction and rolls it back after. This means test data does not persist between tests — clean isolation. @SpringBootTest does NOT add transaction management by default — test data persists unless you add @Transactional yourself.

// @DataJpaTest: auto-rollback by default
@DataJpaTest
class PaymentRepositoryTest {
    // Each @Test runs in a transaction that rolls back after.
    // Data written in one test is NOT visible to the next test.
    @Test
    @Rollback(false)  // or @Commit — force commit for tests that verify DB state externally
    void payment_isPersistedToDatabaseForInspection() {
        repository.save(Payment.builder().accountId("ACC-001").amount(new BigDecimal("100")).build());
        // NOT rolled back — useful for debugging, not for regular test suites
    }
}
// Testing @Transactional service behaviour — verify rollback happens
@SpringBootTest
@Transactional  // each test rolls back
class TransferServiceTransactionTest {
    @Autowired TransferService transferService;
    @Autowired AccountRepository accountRepository;
    @Test
    void transfer_failsOnInsufficientFunds_rollsBackDebit() {
        Account from = accountRepository.save(Account.withBalance(new BigDecimal("100")));
        Account to   = accountRepository.save(Account.withBalance(new BigDecimal("0")));
        assertThatThrownBy(() ->
            transferService.transfer(from.getId(), to.getId(), new BigDecimal("200")))
            .isInstanceOf(InsufficientFundsException.class);
        // Verify debit was rolled back — from account balance unchanged
        Account reloaded = accountRepository.findById(from.getId()).orElseThrow();
        assertThat(reloaded.getBalance()).isEqualByComparingTo(new BigDecimal("100"));
    }
    @Test
    void transfer_successPath_bothAccountsUpdated() {
        Account from = accountRepository.save(Account.withBalance(new BigDecimal("500")));
        Account to   = accountRepository.save(Account.withBalance(new BigDecimal("100")));
        transferService.transfer(from.getId(), to.getId(), new BigDecimal("200"));
        assertThat(accountRepository.findById(from.getId()).get().getBalance())
            .isEqualByComparingTo(new BigDecimal("300"));
        assertThat(accountRepository.findById(to.getId()).get().getBalance())
            .isEqualByComparingTo(new BigDecimal("300"));
    }
}

Testing @Cacheable

// Testing cache behaviour — verify hits, misses, and eviction
@SpringBootTest
class PaymentCacheTest {
    @Autowired PaymentService  paymentService;
    @Autowired CacheManager    cacheManager;
    @MockBean  PaymentRepository repository; // count actual DB calls
    @BeforeEach
    void clearCache() {
        cacheManager.getCache("payments").clear();
    }
    @Test
    void getPayment_secondCall_hitsCache_notRepository() {
        Payment payment = Payment.builder().id("TXN-123").build();
        given(repository.findById("TXN-123")).willReturn(Optional.of(payment));
        // First call — cache miss, hits repository
        Payment first = paymentService.getById("TXN-123");
        // Second call — cache hit, repository NOT called again
        Payment second = paymentService.getById("TXN-123");
        assertThat(first).isEqualTo(second);
        // Repository called exactly once despite two service calls
        verify(repository, times(1)).findById("TXN-123");
    }
    @Test
    void updatePayment_evictsCache_nextCallHitsRepository() {
        Payment original = Payment.builder().id("TXN-123").status("PENDING").build();
        Payment updated  = Payment.builder().id("TXN-123").status("COMPLETED").build();
        given(repository.findById("TXN-123"))
            .willReturn(Optional.of(original))
            .willReturn(Optional.of(updated));
        paymentService.getById("TXN-123");        // populate cache
        paymentService.updateStatus("TXN-123", "COMPLETED"); // @CacheEvict
        Payment afterEvict = paymentService.getById("TXN-123"); // must hit DB again
        assertThat(afterEvict.getStatus()).isEqualTo("COMPLETED");
        verify(repository, times(2)).findById("TXN-123"); // called twice
    }
}
Q4: How do you test a void @Async method that writes to the database? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Void @Async methods do not return a future you can await, so you cannot use .get(). The right approach is Awaitility — a library specifically for testing asynchronous code. Call the async method, then use await().atMost(5, SECONDS).pollInterval(100, MILLISECONDS).untilAsserted(() -> assertThat(repository.findById(id)).isPresent()). Awaitility polls your assertion every 100ms until it passes or the timeout is reached. This avoids Thread.sleep() which is flaky (too short = false fail, too long = slow suite). The other option for unit tests where you do not need real async behaviour is to replace the TaskExecutor with SyncTaskExecutor in a @TestConfiguration — then @Async methods run synchronously on the calling thread and you can assert immediately. I use SyncTaskExecutor for unit-level tests and Awaitility for integration tests where the real async threading matters.

Section 6 — Test Data Builders and Object Mother

The Problem with Inline Test Data

Why test data management matters:

When every test creates its own data with 20-field constructors, test files become unreadable and changing a field requires updating dozens of tests. The Object Mother and Builder patterns centralise test data creation so tests express intent, not construction noise.

// ── Object Mother: factory for common test objects ──
public class PaymentMother {
    // Named factory methods — self-documenting
    public static Payment pendingPayment() {
        return Payment.builder()
            .id(UUID.randomUUID().toString())
            .accountId("ACC-001")
            .amount(new BigDecimal("500.00"))
            .currency("INR")
            .status(PaymentStatus.PENDING)
            .createdAt(Instant.now())
            .build();
    }
    public static Payment completedPayment() {
        return pendingPayment().toBuilder()
            .status(PaymentStatus.COMPLETED)
            .completedAt(Instant.now())
            .build();
    }
    public static Payment failedPayment(String reason) {
        return pendingPayment().toBuilder()
            .status(PaymentStatus.FAILED)
            .failureReason(reason)
            .build();
    }
    // Overridable factory — tests customise only the fields they care about
    public static Payment.PaymentBuilder aPayment() {
        return Payment.builder()
            .id(UUID.randomUUID().toString())
            .accountId("ACC-DEFAULT")
            .amount(new BigDecimal("100.00"))
            .currency("INR")
            .status(PaymentStatus.PENDING)
            .createdAt(Instant.now());
    }
}
// ── Using Object Mother in tests ──
@Test
void refund_failedPayment_throwsInvalidStateException() {
    // Intent is clear — no construction noise
    Payment failed = PaymentMother.failedPayment("INSUFFICIENT_FUNDS");
    assertThatThrownBy(() -> paymentService.refund(failed))
        .isInstanceOf(InvalidPaymentStateException.class)
        .hasMessageContaining("Cannot refund a failed payment");
}
@Test
void findHighValuePayments_returnsOnlyAboveThreshold() {
    // Customise only the field this test cares about
    Payment highValue = PaymentMother.aPayment()
        .amount(new BigDecimal("10000.00")).build();
    Payment lowValue  = PaymentMother.aPayment()
        .amount(new BigDecimal("50.00")).build();
    repository.saveAll(List.of(highValue, lowValue));
    List<Payment> results = repository.findByAmountGreaterThan(new BigDecimal("1000"));
    assertThat(results).hasSize(1)
        .first().satisfies(p ->
            assertThat(p.getAmount()).isEqualByComparingTo(new BigDecimal("10000.00")));
}
Q5: How do you structure test data in a large Spring Boot codebase? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

I use the Object Mother pattern combined with builder overrides. The Object Mother class has named factory methods for each meaningful state — pendingPayment(), completedPayment(), failedPayment(reason). It also has a generic builder method aPayment() that returns a preconfigured builder so individual tests can customise only the fields relevant to what they are testing. This means a test for high-value payment filtering only sets the amount field, and the test reads like documentation. Without this, tests are 30 lines of constructor noise before getting to the assertion. For database integration tests, I pair this with a @BeforeEach that uses the test entity manager to persist the Mother objects, and I rely on the auto-rollback from @DataJpaTest or @Transactional to clean up. For Kafka integration tests, I use a shared Testcontainer and explicitly produce and consume messages with a unique test-run identifier to avoid cross-test contamination.

Interview Q&A — 15 Questions

How to use:

Cover the answer, say it out loud, then check. Focus on the reasoning — why, not just what.

Q6: What is the difference between @MockBean and @Mock? [Easy] ▼
💡 Ideal Answer (Amazon/GS Standard)

@Mock is a pure Mockito annotation — it creates a mock but does not involve Spring at all. It is used in unit tests with @ExtendWith(MockitoExtension.class) where you are testing a single class with no Spring context. @MockBean is a Spring Test annotation — it creates a Mockito mock AND registers it as a bean in the Spring ApplicationContext, replacing any existing bean of that type. Use @Mock for fast unit tests with no Spring context startup. Use @MockBean in @WebMvcTest or @SpringBootTest when you need the mock to be injected into a real Spring-managed bean.

Q7: Why does @Transactional on a test method cause problems when testing REQUIRES_NEW? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

When you put @Transactional on a test method, the entire test runs inside a transaction that Spring creates for the test. When your service code calls a method with REQUIRES_NEW propagation, Spring is supposed to suspend the existing transaction and start a new independent one. But the new transaction cannot see data committed by the suspended test transaction, and the test transaction's rollback at the end undoes everything — including what the REQUIRES_NEW transaction committed. The test gives you incorrect results. The fix: remove @Transactional from the test and manage cleanup manually (delete test data in @AfterEach), or use a separate DataSource transaction in your assertion. This is one of the subtler testing gotchas because the test still passes — it just does not test the right thing.

Q8: What does @DynamicPropertySource do and why is it needed with Testcontainers? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Testcontainers starts a Docker container on a random available port — you cannot know the port ahead of time to put in application.properties. @DynamicPropertySource is a Spring Test mechanism that lets you add or override properties after the container starts but before the ApplicationContext loads. You point it at the container's actual port and host. Without it, Spring would try to connect to the database URL from your static application-test.properties, which points to the wrong port, and context startup would fail. The static method annotated with @DynamicPropertySource receives a DynamicPropertyRegistry and you call registry.add() with lambdas that read from the container object at context-load time.

Q9: How do you test that a @KafkaListener processes a message correctly? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

I write a @SpringBootTest with a Testcontainers Kafka container, wired via @DynamicPropertySource. Then I use KafkaTemplate to produce a test message to the topic and use Awaitility to poll until the expected side effect appears — a DB record, a counter increment, or an outbox entry. Awaitility.await().atMost(10, SECONDS).untilAsserted(() -> assertThat(repository.findByTransactionId(id)).isPresent()). For unit testing the listener method in isolation — without Kafka infrastructure — I call the listener method directly with a constructed message object, since it is just a regular method. The distinction matters: unit test the logic, integration test the Kafka plumbing.

Q10: How do you assert JSON response bodies in MockMvc tests? [Easy] ▼
💡 Ideal Answer (Amazon/GS Standard)

MockMvc has built-in JSON path support via andExpect(jsonPath()). For simple values: .andExpect(jsonPath('$.transactionId').value('TXN-123')). For array size: .andExpect(jsonPath('$.errors', hasSize(2))). For nested paths: .andExpect(jsonPath('$.payment.amount').value(500.00)). For complex assertions I use .andExpect(content().json(expectedJson)) with JSONAssert's lenient or strict mode. For response body capture I use .andReturn().getResponse().getContentAsString() and then deserialise with ObjectMapper and use AssertJ for full object equality. The andDo(print()) is invaluable during development — it logs the full request and response to console so you can see exactly what was sent and received.

Q11: What is the Object Mother pattern and why does it matter for test maintainability? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Object Mother is a class (or set of factory methods) that creates common test objects in meaningful named states — pendingPayment(), completedPayment(), failedPayment(reason). It is the test data equivalent of the Builder pattern. Without it, every test constructs its own objects inline with 20-field constructors, and when a new required field is added to the domain object, you update it in 50 test files. With Object Mother, you update one factory method. It also makes tests readable — assertThatThrownBy(() -> paymentService.refund(PaymentMother.failedPayment())) reads as a specification, not construction code. I combine it with a builder method like aPayment() that returns a partially-built builder so individual tests can customise only the fields they care about.

Q12: How do you prevent test pollution when using Testcontainers with a shared database? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Test pollution happens when one test leaves data that affects another. Three strategies. First: @Transactional on each test method — rolls back all changes after each test, cheapest option, works for most cases. Second: @BeforeEach cleanup — delete test-specific data using a known prefix or ID range. This is necessary when testing REQUIRES_NEW or when rollback would hide the actual DB state you want to assert on. Third: create test-specific unique identifiers — every test generates a UUID prefix for its data so even if cleanup fails, tests do not interfere. For Kafka, I use unique topic names or consumer group IDs per test to avoid consuming messages from other tests. The shared container pattern makes startup fast; the cleanup strategy makes tests reliable.

Q13: When would you use @SpringBootTest over @WebMvcTest? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

@WebMvcTest is for controller tests where you want to verify HTTP behaviour in isolation from the service layer. Everything below the controller is a @MockBean. @SpringBootTest is for integration tests where you want all layers working together — real service, real repository, real database (via Testcontainers). The cost of @SpringBootTest is startup time and test complexity. I use @WebMvcTest for the majority of controller tests — they run fast and test exactly what a controller should be responsible for: request validation, response serialisation, HTTP status codes, and exception handling. I use @SpringBootTest for a smaller number of end-to-end integration tests that verify the full flow from HTTP to database, including transaction behaviour and event publishing.

Q14: How do you verify an email was sent with specific content in a Spring Boot test? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

If the email client is a @MockBean, I use ArgumentCaptor to capture the arguments passed to the send method and assert on them — captor.getValue().getSubject().contains('TXN-123'). If I need to verify the actual email content with a template rendered, I use a test email server like GreenMail or Fake SMTP, which runs in-process and captures sent emails. GreenMail integrates with JavaMail and lets you assert on received messages: assertThat(greenMail.getReceivedMessages()).hasSize(1) and then inspect the message body. For notification services that call an external API, WireMock captures the exact HTTP request body, so I can verify the rendered content in the stub's request body matching.

Q15: What does @TestPropertySource do and when do you use it? [Easy] ▼
💡 Ideal Answer (Amazon/GS Standard)

@TestPropertySource overrides properties for a specific test class — without it, you would have to modify application.properties or create a separate test profile. I use it when: a test needs to override a single property without affecting other tests (different rate limit threshold, feature flag on/off), when testing @ConditionalOnProperty auto-configurations (test that the bean IS created when the property is present and is NOT created when absent), and when pointing at a different external service URL. It can load a properties file or specify inline properties. For Testcontainers, @DynamicPropertySource is better because properties are computed after container startup; @TestPropertySource is static and evaluated before the context loads.

Q16: How do you test a Spring Security filter or authentication in a @WebMvcTest? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

@WebMvcTest includes the full security filter chain by default if Spring Security is on the classpath. For tests that need an authenticated user I use @WithMockUser(username='user', roles='USER') — Spring Security's test support. For JWT-based auth I either use @WithMockUser with the same roles the JWT would grant, or I configure a test SecurityFilterChain that accepts a test-specific header. To test that an unauthenticated request is rejected: perform the request without credentials and andExpect(status().isUnauthorized()). For testing the JWT validation logic itself — token expiry, invalid signature, missing claims — I write a unit test for the JwtValidator class directly without involving MockMvc.

Q17: What is @SpyBean and when should you not use it? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

@SpyBean wraps a real Spring bean in a Mockito spy so you can verify calls and selectively stub methods while the real code still runs. You should not use it when: the real code makes external network calls (tests become environment-dependent and slow), the real code has non-deterministic behaviour (random IDs, timestamps), or when the real code has complex setup that the spy tears through. The right use is for in-process side effects where you want both real execution and call verification — an internal event publisher, an in-memory cache, or a service that writes to the database and you want to verify it was called with correct arguments while also asserting the DB state.

Q18: How do you test a Testcontainers-based test suite in a CI pipeline without Docker? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

Testcontainers requires Docker. In CI pipelines (GitHub Actions, Jenkins, GitLab CI) Docker is available by default — Testcontainers detects it automatically through the DOCKER_HOST environment variable or the Docker socket. For environments without Docker (some restrictive CI environments), there are two options: Testcontainers Cloud, which runs containers remotely; or using conditional test tags so Testcontainers tests are only run in environments where Docker is available via JUnit 5's @EnabledIfEnvironmentVariable or a custom @EnabledIfDockerAvailable annotation. The other common issue is the Ryuk container that Testcontainers uses for cleanup — some CI environments restrict it, and you disable it with TESTCONTAINERS_RYUK_DISABLED=true and handle cleanup yourself.

Q19: What makes a good unit test vs a bad one? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

A good unit test: tests one specific behaviour, has a clear arrange-act-assert structure, fails for exactly one reason (the behaviour it tests), runs fast (milliseconds), does not depend on external resources, and has a name that reads as a specification (processPayment_insufficientFunds_throwsException). A bad unit test: tests implementation rather than behaviour (asserting that a specific private method was called), has setup so complex you cannot understand the intent, is order-dependent (passes only when run after another test), uses Thread.sleep(), mocks everything including the class under test (you are testing the mock), or has assertions so broad that unrelated bugs would pass. The question I ask before writing a test: if this test passes, what exactly do I know? If the answer is vague, the test is bad.
