🔵 Topic 1: ArrayList and LinkedList — Internals and Trade-offs

Checklist

  • ArrayList: initial capacity 10, growth factor 1.5, System.arraycopy cost
  • LinkedList: doubly-linked nodes, Deque implementation, memory overhead per node
  • Big-O comparison: get, add-end, add-middle, remove, contains
  • Why LinkedList is almost never the right choice in modern Java
  • Arrays.asList() vs List.of() vs new ArrayList() — the 3 gotchas
  • Q1 through Q12 — answer each before reading the answer

The Growth Algorithm is a Real Interview Differentiator:

// ArrayList capacity vs size
ArrayList<String> list = new ArrayList<>();       // capacity 10, size 0
ArrayList<String> list2 = new ArrayList<>(1000); // pre-allocated — no resizing
list2.ensureCapacity(2000);                       // manual pre-allocation
list2.trimToSize();                               // release unused capacity to GC
// The 3 List creation methods — each behaves differently
List<String> a = Arrays.asList("x","y","z"); // fixed size — set() works, add/remove throw
List<String> b = List.of("x","y","z");       // fully immutable — no nulls allowed
List<String> c = new ArrayList<>(List.of("x","y","z")); // mutable copy — safest
// LinkedList memory cost — why cache hates it
// Each node: object header(16B) + prev(8B) + data reference(8B) + next(8B) = 40B minimum
// Plus the Integer object itself = ~56B per int vs ~4B in ArrayList
// CPU prefetching fails on pointer chasing — ArrayList contiguous memory wins
Q1: What is the default initial capacity of ArrayList and its growth factor? [Easy] ▼
💡 Ideal Answer (Amazon/GS Standard)

Default capacity: 10. Growth: old + (old >> 1) = 1.5x. Adding 11th element to capacity-10 list creates array of size 15, copies via System.arraycopy(), adds element. Amortised O(1) add at end. Pre-allocate with new ArrayList<>(n) when size is known.

Q2: What is the difference between size() and capacity() in ArrayList? [Easy] ▼
💡 Ideal Answer (Amazon/GS Standard)

size() = number of elements stored. Capacity = internal array length (no public getter). new ArrayList<>(100).size() == 0, but holds 100 without resizing. Matters for pre-allocation when adding known quantities.

Q3: When would you choose LinkedList over ArrayList? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Almost never for general use. LinkedList is appropriate only when used purely as a Deque (addFirst/pollFirst — O(1) at head). For random access: ArrayList O(1) vs LinkedList O(n). For modern workloads, ArrayList beats LinkedList due to CPU cache locality from contiguous memory.

Q4: What is the time complexity of ArrayList.add(int index, E element)? [Easy] ▼
💡 Ideal Answer (Amazon/GS Standard)

O(n) — elements from index to end are shifted right via System.arraycopy(). Plus potential resize O(n). add(E e) at end is amortised O(1). Frequent mid-list insertions on large lists are expensive — consider deque or different structure.

Q5: What does Arrays.asList() return and what are its limitations? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Returns a fixed-size List backed by the array — a view, not a copy. set() works to replace elements. add() and remove() throw UnsupportedOperationException. Changes to original array reflect in the list. For mutable list: new ArrayList<>(Arrays.asList(arr)).

Q6: What is the difference between List.of() and Collections.unmodifiableList()? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

List.of() creates a new immutable list. Nulls not allowed. unmodifiableList(existing) wraps an existing list — it is a view, so the underlying list can still be modified by holders of the original reference. List.copyOf() makes a truly independent immutable copy. Use List.of() for new immutable lists, List.copyOf() to defensively snapshot a mutable list.

Q7: How does ArrayList.contains() work and what is its time complexity? [Easy] ▼
💡 Ideal Answer (Amazon/GS Standard)

O(n) — iterates the array calling equals() on each element. For O(1) contains(), use HashSet. Checking contains() in a loop on ArrayList is O(n^2) overall — a common performance mistake in code reviews.

Q8: What is the autoboxing trap with ArrayList.remove()? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

remove(int index) removes by position. remove(Object o) removes by value. On List, list.remove(5) calls remove(int index=5), NOT remove(Integer value 5). To remove the value 5: list.remove(Integer.valueOf(5)). This is a silent bug — no compile error, wrong element removed.

Q9: What is the memory overhead of a LinkedList node vs ArrayList element? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

LinkedList node: object header(16B) + prev pointer(8B) + data reference(8B) + next pointer(8B) = 40B per node, plus Integer object ~16B = 56B+ per int. ArrayList stores a reference (8B) plus the Integer object. LinkedList has roughly 7-14x memory overhead per element and poor CPU cache locality.

Q10: What sorting algorithm does Collections.sort() use? [Easy] ▼
💡 Ideal Answer (Amazon/GS Standard)

TimSort — adaptive stable merge sort. O(n log n) worst case, O(n) for nearly-sorted data. Stable: equal elements maintain relative original order. Since Java 8: list.sort(Comparator) is the direct preferred method. Arrays.sort() on objects uses TimSort. Arrays.sort() on primitives uses dual-pivot QuickSort (not stable, but irrelevant for primitives).

Q11: What is subList() and what is the danger with it? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

list.subList(from, to) returns a VIEW of [from, to) — not a copy. Modifying the subList modifies the original. Danger: if the original list is structurally modified after calling subList() (add/remove), the subList becomes invalid and throws ConcurrentModificationException on next access. Safe usage: new ArrayList<>(list.subList(from, to)) creates an independent copy.

Q12: How do you safely remove elements matching a condition during iteration? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Three safe ways: 1) list.removeIf(predicate) — single O(n) pass, no CME. 2) Iterator.remove() — call after iterator.next(). 3) Collect indices/elements to remove, then remove after iteration completes. Never use list.remove() inside a for-each loop — throws ConcurrentModificationException.

🟢 Topic 2: HashMap Internals — The Most Asked Collection Topic

Checklist

  • Hashing: hashCode() to hash() to index — the full chain
  • Collision: chaining, Java 8 treeification at 8 nodes (TREEIFY_THRESHOLD)
  • Load factor 0.75: why this value, what happens on resize
  • Null key/null value behaviour in HashMap vs Hashtable vs ConcurrentHashMap
  • LinkedHashMap insertion order vs access order — LRU cache pattern
  • Q13 through Q30 — answer before reading

HashMap Internal Process — Know Every Step:

put(key, value): 1) Compute hash: h = key.hashCode(), then hash = h ^ (h >>> 16) spreading to reduce collisions from poor implementations. 2) Bucket index = (table.length - 1) & hash — fast because table length is always power-of-2. 3a) If bucket empty: insert new Node. 3b) If occupied: walk chain comparing equals(). Update if found, append if not. If chain exceeds 8 AND table >= 64: treeify to Red-Black Tree. 4) If ++size > capacity * 0.75: resize — double table, rehash all entries.

// HashMap constants
// INITIAL_CAPACITY  = 16   (always power of 2)
// LOAD_FACTOR       = 0.75 (resize at 12 entries with capacity 16)
// TREEIFY_THRESHOLD = 8    (linked list to Red-Black Tree)
// UNTREEIFY_THRESHOLD = 6  (tree back to list on shrink)
// MIN_TREEIFY_CAPACITY = 64 (table must be 64+ before treeifying)
// Pre-size to avoid resizing — for known entry count
int expected = 1000;
Map<String, Employee> map = new HashMap<>((int)(expected / 0.75f) + 1);
// Creates capacity 1334 — 1000 entries will never trigger a resize
// LinkedHashMap LRU Cache pattern
int CAPACITY = 100;
Map<String, User> lru = new LinkedHashMap<>(CAPACITY, 0.75f, true) {
    protected boolean removeEldestEntry(Map.Entry<String,User> e) {
        return size() > CAPACITY; // evict least-recently-accessed when full
    }
};
// Every get() moves entry to the end — oldest unused entry stays at head
Q13: How does HashMap compute the bucket index for a key? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

1) key.hashCode() gives raw h. 2) Spreading: hash = h ^ (h >>> 16) — XOR with upper 16 bits reduces collision from poor hashCode distributions. 3) Index = (table.length - 1) & hash — bitwise AND works because table is always power-of-2, equivalent to hash % length but much faster.

Q14: What happens when two keys hash to the same bucket? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Collision — stored as linked list chain in the bucket. equals() distinguishes keys within the chain. Java 8 optimization: if chain length exceeds TREEIFY_THRESHOLD (8) AND table size >= 64, chain converts to Red-Black Tree for O(log n) worst case instead of O(n).

Q15: What is the HashMap load factor and what is the default? [Easy] ▼
💡 Ideal Answer (Amazon/GS Standard)

Load factor = resize threshold ratio. Default 0.75. Resize triggers when size > capacity * loadFactor. At capacity 16, resize at 12 entries (doubles to 32). Higher load factor: more collisions, less memory. Lower: fewer collisions, more memory. 0.75 is empirically optimal balance.

Q16: What happens during HashMap resize? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

New array of double capacity created. All entries rehashed. Java 8 optimization: for power-of-2 doubling, each entry either stays at old index OR moves to oldIndex + oldCapacity — determined by checking one new bit. O(n) operation that temporarily doubles memory usage. Avoid with pre-sized constructor.

Q17: Why must you override both equals() and hashCode() together? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

HashMap contract: if a.equals(b) then a.hashCode() == b.hashCode() MUST hold. If you override equals() but not hashCode(): two equal objects get different hash codes, land in different buckets, and get() never finds the stored value. Always override both together. IDEs and Lombok handle this automatically.

Q18: What is the difference between HashMap, Hashtable, and ConcurrentHashMap? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

HashMap: not thread-safe, one null key, fast. Hashtable: fully synchronized single lock, no nulls, legacy — essentially obsolete, never use in new code. ConcurrentHashMap: bucket-level locking (Java 8: CAS + node-level sync), no null keys/values, high concurrent throughput, atomic compute/merge. Use HashMap single-threaded, ConcurrentHashMap multi-threaded.

Q19: Can HashMap have null keys and null values? [Easy] ▼
💡 Ideal Answer (Amazon/GS Standard)

Yes — exactly one null key (stored at bucket 0 with special null-check code) and any number of null values. Hashtable and ConcurrentHashMap both reject null keys and values with NullPointerException — deliberate design to eliminate null ambiguity in concurrent contexts.

Q20: What is HashMap.get() time complexity in best, average, and worst case? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Best/Average: O(1) — hash to bucket, one equals() check. Worst case pre-Java 8 (all keys one bucket): O(n) linked list. Worst case Java 8+ (treeified bucket): O(log n) Red-Black Tree. In practice with good hashCode(): always O(1). Degenerate hashCode() causes O(n) or O(log n).

Q21: What is LinkedHashMap and what are its two ordering modes? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Extends HashMap with a doubly-linked list through all entries. Insertion-order mode (default): iterates in insertion order. Access-order mode (constructor flag true): moves accessed entry to end on every get() call — most recently used last, least recently used first. Access-order + removeEldestEntry() override = LRU cache in ~10 lines.

Q22: What does computeIfAbsent() do and when is it useful? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

computeIfAbsent(key, fn): if key absent, calls fn to compute value, STORES it, returns it. If key present, returns existing value without calling fn. Use case: map.computeIfAbsent(userId, id -> new ArrayList<>()).add(event) — atomically creates the list if absent. In ConcurrentHashMap this entire operation is atomic — no external synchronization needed.

Q23: What does HashMap.merge() do? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

merge(key, value, BiFunction): if key absent, stores value. If key present, applies BiFunction(oldValue, newValue) and stores result. If function returns null, removes key. Classic use: frequency counter — map.merge(word, 1, Integer::sum) increments or initializes to 1. Fully atomic in ConcurrentHashMap.

Q24: What is the difference between merge() and compute()? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

Both do atomic read-modify-write. merge(key, value, fn): you provide a new value; fn receives (old, new). compute(key, BiFunction): fn receives (key, current or null), you compute the new value from scratch. compute() is more flexible when result depends on the key itself or when you want to remove on null return.

Q25: Why did HashMap cause infinite loops in pre-Java 8 multi-threaded code? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

Java 6-7 resize used a head-insertion rehashing algorithm. With two threads resizing simultaneously, circular references could form in the linked list (Node A points to Node B, Node B points back to Node A). Any subsequent get() traversing that bucket loops forever. Java 8 fixed this with a new resize algorithm. Lesson: never use HashMap in multi-threaded code — use ConcurrentHashMap.

Q26: What does putIfAbsent() return vs put()? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

put(k,v): always stores. Returns PREVIOUS value (null if new key). putIfAbsent(k,v): stores only if absent. Returns CURRENT value — existing value if key was present, null if just stored. In HashMap both are non-atomic compound operations. In ConcurrentHashMap, putIfAbsent IS atomic — safe for concurrent conditional initialization.

Q27: How would you implement a thread-safe word frequency counter? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

ConcurrentHashMap freq = new ConcurrentHashMap<>(); For each word: freq.merge(word, 1, Integer::sum). The merge() call is atomic in CHM — no external synchronization needed. Alternative: freq.compute(word, (k,v) -> v==null ? 1 : v+1). Both are O(1) amortised and fully thread-safe.

Q28: What happens if you modify a HashMap while iterating? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Throws ConcurrentModificationException — HashMap iterator is fail-fast, tracks modCount. Any structural modification (add/remove) outside the iterator increments modCount. Iterator checks on each next() call — mismatch throws immediately. Safe alternatives: iterator.remove(), entrySet().removeIf(), or collect keys to remove then remove after iteration.

Q29: What is EnumMap and when should you use it? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

EnumMap is backed by an array indexed by enum ordinal — zero hashing, zero collision, O(1) all operations with minimal memory. Iterates in enum declaration order. Always use EnumMap when keys are enums — faster and more memory-efficient than HashMap. EnumSet similarly uses a bit-vector for enum sets.

Q30: How does HashMap behave when hashCode() always returns the same value? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

All keys land in one bucket — degenerates to linked list O(n). Java 8+: after 8 entries in bucket with table >= 64, treeifies to Red-Black Tree O(log n). This is a hash flooding DoS vulnerability. Fix: strong hashCode() distribution. Java randomizes String hash seeds since Java 7u6 to prevent this attack vector.

🟠 Topic 3: HashSet, LinkedHashMap, TreeMap, TreeSet

Checklist

  • HashSet is a HashMap — understand the internal delegation
  • TreeMap: Red-Black Tree, NavigableMap methods (floor/ceiling/headMap/tailMap)
  • TreeSet: sorted unique elements, compareTo contract
  • Q31 through Q40
// HashSet is literally just a HashMap with dummy values
// HashSet.add(e) calls map.put(e, PRESENT) where PRESENT = new Object()
// HashSet.contains(e) calls map.containsKey(e)
// All HashMap rules: needs equals()+hashCode(), O(1), one null allowed
// TreeMap NavigableMap methods — know all of these
TreeMap<String, Integer> scores = new TreeMap<>();
scores.put("Charlie", 85); scores.put("Alice", 90); scores.put("Bob", 75);
scores.firstKey();              // Alice  — smallest key
scores.lastKey();               // Charlie — largest key
scores.floorKey("Bob");         // Bob    — largest key <= 'Bob'
scores.ceilingKey("Ben");       // Bob    — smallest key >= 'Ben'
scores.lowerKey("Bob");         // Alice  — strictly less than 'Bob'
scores.higherKey("Bob");        // Charlie — strictly greater than 'Bob'
scores.headMap("Bob");          // {Alice=90} keys strictly < 'Bob'
scores.tailMap("Bob");          // {Bob=75, Charlie=85} keys >= 'Bob'
scores.subMap("Alice","Charlie");// {Alice=90, Bob=75} — [from, to) exclusive end
scores.descendingMap();         // reverse-order view — no copy
// All TreeMap operations: O(log n) — Red-Black Tree
Q31: How is HashSet internally implemented? [Easy] ▼
💡 Ideal Answer (Amazon/GS Standard)

Backed by a HashMap. All values share a single dummy PRESENT object. add(e) = map.put(e, PRESENT). contains(e) = map.containsKey(e). All HashMap properties apply: O(1) operations, needs hashCode() + equals(), one null element allowed, no ordering.

Q32: What is the difference between HashSet, LinkedHashSet, and TreeSet? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

HashSet: O(1) ops, no order guarantee, backed by HashMap. LinkedHashSet: O(1) ops, insertion order maintained, backed by LinkedHashMap. TreeSet: O(log n) ops, sorted order (natural or Comparator), backed by TreeMap (Red-Black Tree). Default: HashSet. Need insertion order: LinkedHashSet. Need sorted/range ops: TreeSet.

Q33: What happens if you mutate an object that is a HashSet element? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

hashCode() changes after mutation. Set can no longer find it — looks in wrong bucket. contains() returns false even though object is in the set. The bucket holds it orphaned — breaking Set contract and causing memory leak. Never mutate objects used as Set elements or Map keys. Use immutable keys — Records, String, Integer are safe.

Q34: What is TreeMap and how is it ordered? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Backed by Red-Black Tree. Keys ordered by natural ordering (Comparable) or provided Comparator. All ops O(log n). Null keys NOT allowed (can't compare null). NavigableMap methods: firstKey, lastKey, floorKey, ceilingKey, headMap, tailMap, subMap — essential for range queries and nearest-key lookups.

Q35: HashMap vs TreeMap — when to use which? [Easy] ▼
💡 Ideal Answer (Amazon/GS Standard)

HashMap: O(1) ops, no ordering, needs hashCode()+equals(). Use for most key-value storage. TreeMap: O(log n) ops, sorted by key, needs Comparable or Comparator. Use when you need sorted iteration, range queries (all keys between X and Y), or nearest-key lookups. Do not use TreeMap just for sorted output — sort HashMap entries instead.

Q36: How does TreeSet handle duplicates and what is the contract? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

TreeSet uses compareTo() (or Comparator) for equality — NOT equals(). If compareTo() returns 0, element is considered duplicate and rejected. Critical: if compareTo() is inconsistent with equals() (returns 0 for unequal objects), behaviour is undefined. Elements must implement Comparable or a Comparator must be provided — else ClassCastException at runtime.

Q37: What does NavigableMap add over SortedMap? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

SortedMap: firstKey, lastKey, headMap, tailMap, subMap. NavigableMap adds: floorKey(k) largest key <= k. ceilingKey(k) smallest key >= k. lowerKey(k) strictly less. higherKey(k) strictly greater. pollFirstEntry/pollLastEntry remove and return extremes. descendingMap() reverse-order view. TreeMap implements NavigableMap.

Q38: How would you find all keys in a range [from, to] inclusive in a TreeMap? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

treeMap.subMap(from, true, to, true) — two boolean params control inclusive/exclusive for each bound. Returns a live view — modifications to the view affect the original map. For independent snapshot: new TreeMap<>(original.subMap(from, true, to, true)).

Q39: What is EnumSet and why is it more efficient than HashSet for enums? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

EnumSet uses a bit-vector (single long for enums <= 64 values). Each enum constant maps to a bit position. All operations are pure bit manipulation — O(1) with zero object allocation. Iterates in enum declaration order. No hashing overhead. Always use EnumSet for enum collections — it is the most memory-efficient and fastest Set implementation possible.

Q40: How do you implement a case-insensitive sorted set? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

new TreeSet<>(String.CASE_INSENSITIVE_ORDER) — uses the built-in case-insensitive Comparator. 'Apple' and 'apple' are treated as equal and only one is stored. TreeSet uses the Comparator for both ordering AND equality determination — consistent behaviour throughout.

🟣 Topic 4: Queue, Deque, PriorityQueue, BlockingQueue

Checklist

  • Queue: the 3 pairs — add/offer, remove/poll, element/peek
  • Deque and ArrayDeque as stack and queue replacement, why not Stack class
  • PriorityQueue: min-heap internals, custom Comparator, top-K pattern
  • BlockingQueue: put/take blocking semantics, ArrayBlockingQueue vs LinkedBlockingQueue
  • Q41 through Q50

Operation

Throws Exception vs Returns null/false

Insert

add(e) — throws if full. offer(e) — returns false if full. Use offer() for bounded queues.

Remove head

remove() — throws if empty. poll() — returns null if empty. Always use poll().

Peek head

element() — throws if empty. peek() — returns null if empty. Use peek() for safe read.

// ArrayDeque — preferred replacement for both Stack and Queue
Deque<String> deque = new ArrayDeque<>();
deque.offerLast("A"); deque.offerLast("B"); // Queue: add to tail
deque.pollFirst();  // Queue: remove from head — FIFO
deque.push("X");   // Stack: addFirst
deque.pop();        // Stack: removeFirst — LIFO
// PriorityQueue — min-heap: smallest polled first
PriorityQueue<Integer> minH = new PriorityQueue<>();
PriorityQueue<Integer> maxH = new PriorityQueue<>(Comparator.reverseOrder());
// Top-K largest elements pattern — O(n log k) time, O(k) space
PriorityQueue<Integer> topK = new PriorityQueue<>(); // min-heap of size k
for (int num : array) {
    topK.offer(num);
    if (topK.size() > k) topK.poll(); // remove smallest — keeps K largest
}
// topK now contains the K largest elements
Q41: What is the difference between add() and offer() in Queue? [Easy] ▼
💡 Ideal Answer (Amazon/GS Standard)

add(e): throws IllegalStateException if capacity-bounded and full. offer(e): returns false if full — non-exceptional. For unbounded queues both behave identically. Best practice: use offer() for bounded queues to handle full gracefully. Use add() only when you want an exception on overflow as a programming error signal.

Q42: Why use ArrayDeque instead of Stack class? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Stack extends Vector which is synchronized — you pay locking cost in single-threaded code. ArrayDeque has zero synchronization overhead, is faster, more memory-efficient (circular array). Java documentation explicitly recommends Deque over Stack: 'A more complete and consistent set of LIFO operations is provided by Deque interface.'

Q43: How does PriorityQueue work internally? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

Binary min-heap backed by an array. Root (index 0) always contains minimum. Heap property: parent <= children. offer(e): add at end, sift up O(log n). poll(): remove root, move last to root, sift down O(log n). peek(): return root O(1). Iteration does NOT give sorted order — only poll() gives elements in priority order.

Q44: What is the time complexity of PriorityQueue operations? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

offer()/add(): O(log n) sift up. poll()/remove(): O(log n) sift down. peek(): O(1). contains(): O(n) — no index structure. remove(Object): O(n) find + O(log n) fix. Building PriorityQueue from collection: O(n) heapify — linear, not n log n.

Q45: ArrayDeque vs LinkedList as a Queue — which is faster? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

ArrayDeque is faster in practice for most queue/stack use cases. It uses a circular resizable array — better CPU cache locality, lower memory overhead (no node objects). LinkedList has higher per-element memory (node objects with prev/next pointers) and pointer-chasing hurts cache performance. LinkedList advantage: also implements List — supports random access by index.

Q46: What is BlockingQueue and when would you use it? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Extends Queue with blocking: put(e) blocks if full, take() blocks if empty. Designed for producer-consumer patterns — eliminates explicit wait/notify. Implementations: ArrayBlockingQueue (bounded, array), LinkedBlockingQueue (optionally bounded, node-based with separate put/take locks), PriorityBlockingQueue (unbounded priority), SynchronousQueue (zero capacity — direct handoff).

Q47: How do you implement producer-consumer with BlockingQueue? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

LinkedBlockingQueue queue = new LinkedBlockingQueue<>(100). Producer: queue.put(task) — blocks when full. Consumer: Task t = queue.take() — blocks when empty. No synchronized or wait/notify needed — BlockingQueue handles all synchronization. Use a sentinel/poison-pill value to signal consumers to stop.

Q48: What is a Deque and how does it differ from Queue? [Easy] ▼
💡 Ideal Answer (Amazon/GS Standard)

Double-Ended Queue — supports insert and remove at BOTH ends. Queue only inserts at tail and removes from head. Deque can act as Stack (push/pop at head), Queue (offer at tail, poll from head), or double-ended buffer. Implementations: ArrayDeque (preferred), LinkedList (also a Deque).

Q49: Why does PriorityQueue iteration not return elements in sorted order? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

Heap property only guarantees root is minimum — other elements are in partial order only. The backing array is a heap, not a sorted array. Iterating visits elements in heap array order which is neither insertion order nor sorted order. To retrieve all elements in sorted order: repeatedly call poll() until empty — O(n log n) total.

Q50: What is the difference between ArrayBlockingQueue and LinkedBlockingQueue? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

ArrayBlockingQueue: fixed bounded capacity, array-backed, single lock for both put and take. LinkedBlockingQueue: optionally bounded (or unbounded), node-backed, SEPARATE locks for put and take — higher concurrent throughput. LinkedBlockingQueue is preferred for most producer-consumer use cases due to better concurrency from separate locks.

🔴 Topic 5: Iterators, Comparable vs Comparator, Fail-Fast vs Fail-Safe

Checklist

  • Iterator: hasNext(), next(), remove() — correct usage pattern
  • Fail-fast vs fail-safe — which collections, CME cause and fix
  • Comparable vs Comparator — natural ordering vs external strategy
  • Comparator chaining: comparing().thenComparing().reversed()
  • Q51 through Q60

Fail-Fast vs Fail-Safe — Asked in Every Collections Interview:

Fail-fast (ArrayList, HashMap, HashSet): tracks modCount. Any structural modification outside iterator increments modCount. Iterator checks on each next() — mismatch throws ConcurrentModificationException immediately. Best-effort bug detection, not thread safety. Fail-safe (CopyOnWriteArrayList, ConcurrentHashMap): iterate over snapshot or allow concurrent modification. No CME — may see stale data. Use fail-fast collections and fix your code. Use fail-safe only when concurrent iteration is truly needed.

// Fail-fast — CME thrown
List<String> list = new ArrayList<>(List.of("a","b","c"));
for (String s : list) {
    if (s.equals("b")) list.remove(s); // THROWS ConcurrentModificationException
}
// Fix 1: Iterator.remove()
Iterator<String> it = list.iterator();
while (it.hasNext()) {
    if (it.next().equals("b")) it.remove(); // safe
}
// Fix 2: removeIf — cleanest
list.removeIf(s -> s.equals("b"));
// Comparable — natural ordering (implemented by the class)
public class Employee implements Comparable<Employee> {
    public int compareTo(Employee o) {
        return Integer.compare(this.salary, o.salary); // natural: by salary asc
    }
}
Collections.sort(employees); // uses compareTo()
// Comparator — external strategy (multiple orderings possible)
employees.sort(
    Comparator.comparing(Employee::getDept)           // dept ascending
        .thenComparing(Employee::getSalary,           // then salary
            Comparator.reverseOrder())                // descending
        .thenComparing(Employee::getName));           // then name ascending
Q51: What is the fail-fast behaviour of ArrayList iterator? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

ArrayList iterator tracks modCount. Every structural modification increments modCount. On each next() call, iterator checks if its saved expectedModCount == list's current modCount. Mismatch throws ConcurrentModificationException. Best-effort detection tool — not guaranteed in all edge cases, not a thread-safety mechanism.

Q52: What is the difference between Comparable and Comparator? [Easy] ▼
💡 Ideal Answer (Amazon/GS Standard)

Comparable implemented by the class itself — defines natural ordering. One ordering per class. Used by Collections.sort() and TreeMap/TreeSet by default. Comparator is external — defines ordering without modifying the class. Multiple comparators possible per type. Use Comparable for the most natural default order. Use Comparator for alternative orderings or for classes you cannot modify.

Q53: What is the compareTo() contract? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

Anti-symmetry: sign(a.compareTo(b)) == -sign(b.compareTo(a)). Transitivity: a>b and b>c implies a>c. Consistency with equals (strongly recommended): a.compareTo(b)==0 if and only if a.equals(b). If inconsistent: TreeSet/TreeMap use compareTo for equality, causing wrong behaviour — items considered equal that are not.

Q54: Is Collections.sort() stable? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Yes — uses TimSort (adaptive stable merge sort). Stable means equal elements maintain their relative order from before the sort. This matters for multi-criteria sorting: sort by salary first, then by name — elements with equal salary keep their relative name order. Arrays.sort() on objects is also stable. Arrays.sort() on primitives uses dual-pivot QuickSort (not stable, irrelevant for primitives).

Q55: What is ListIterator and what additional capabilities does it have? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

ListIterator extends Iterator for List only. Additional methods: hasPrevious()/previous() — bidirectional traversal. nextIndex()/previousIndex() — current cursor position. add(e) — inserts at current position. set(e) — replaces last returned element. Useful for reverse traversal or in-place modification beyond simple removal.

Q56: How does ConcurrentHashMap iterator behave during concurrent modification? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

Weakly consistent — NOT fail-fast, will NOT throw ConcurrentModificationException. Reflects some but not necessarily all modifications made after iterator creation. Never returns corrupted or incorrect data. May miss recently added entries or include recently removed ones. This is by design — enables lock-free concurrent iteration at the cost of snapshot freshness.

Q57: What does Comparator.nullsFirst() and nullsLast() do? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Wrap another Comparator to handle null values explicitly. nullsFirst(comparator): nulls sort before all non-null values. nullsLast(comparator): nulls sort after all non-null values. Without these wrappers, comparing a null value typically throws NullPointerException. Use when your data may contain null and you need predictable null ordering.

Q58: What is Spliterator and how does it differ from Iterator? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

Spliterator is designed for parallel decomposition. tryAdvance(Consumer) processes one element. trySplit() splits into two Spliterators for parallel processing. characteristics() reports ORDERED, DISTINCT, SORTED, SIZED, IMMUTABLE flags that help Stream API optimize. Stream API uses Spliterators internally. Iterator has no parallel support.

Q59: How do you sort a Map by value? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

map.entrySet().stream().sorted(Map.Entry.comparingByValue()).collect(Collectors.toMap(Map.Entry::getKey, Map.Entry::getValue, (e1,e2)->e1, LinkedHashMap::new)). Use LinkedHashMap as destination to preserve sort order — plain HashMap does not maintain order.

Q60: What is the difference between Iterator.remove() and Collection.remove()? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Iterator.remove(): removes last element returned by next() from the underlying collection through the iterator — does not trigger CME. Can only be called once per next() call. Collection.remove(obj): directly removes, increments modCount, causes CME if an iterator is active. During iteration: always use iterator.remove() or removeIf().

Topic 6: Concurrent Collections — ConcurrentHashMap, COWAL, BlockingQueues

Checklist

  • ConcurrentHashMap: bucket CAS, node-level sync, compute/merge atomicity, no nulls
  • CopyOnWriteArrayList: snapshot semantics, write cost, when reads >> writes
  • Collections.synchronizedXxx() wrappers: the compound operation trap
  • ConcurrentSkipListMap and ConcurrentLinkedQueue: when to use
  • Q61 through Q80 — 20 questions

ConcurrentHashMap Java 8 Architecture — Why It Is Brilliant:

Java 7 split the table into 16 segments, each independently locked — 16 concurrent writers maximum. Java 8 went further: no segments. Each BUCKET HEAD NODE is individually synchronized. Reads are fully non-blocking using volatile. Multiple threads can write to DIFFERENT buckets simultaneously — as many concurrent writers as non-empty buckets. First insert into empty bucket uses CAS (no locking). Contended buckets use synchronized on just that node. This scales to modern many-core CPUs in a way Hashtable's single lock never could.

// ConcurrentHashMap atomic operations — use these instead of external sync
ConcurrentHashMap<String, Integer> map = new ConcurrentHashMap<>();
map.putIfAbsent("key", 1);                          // atomic conditional insert
map.compute("c", (k, v) -> v == null ? 1 : v + 1); // atomic read-modify-write
map.merge("word", 1, Integer::sum);                 // atomic frequency counter
map.computeIfAbsent("id", k -> new ArrayList<>()).add(item); // atomic lazy init
// The compound operation trap with synchronizedMap:
Map<String,Integer> syncMap = Collections.synchronizedMap(new HashMap<>());
// NOT safe — two separate synchronized calls with race in between:
if (!syncMap.containsKey(key)) { syncMap.put(key, value); } // RACE CONDITION
// Safe: syncMap.putIfAbsent(key, value) or external synchronized block
// CopyOnWriteArrayList — zero-contention reads, O(n) writes
CopyOnWriteArrayList<EventListener> listeners = new CopyOnWriteArrayList<>();
listeners.add(listener); // creates full array copy — O(n)
// Iteration is ALWAYS safe — uses immutable snapshot
for (EventListener l : listeners) { l.onEvent(e); } // no CME possible
// Ideal for: event listener lists, observer pattern, rarely-changing config
Q61: How is ConcurrentHashMap thread-safe without locking the whole map? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Java 8: reads are non-blocking (volatile). First insert into empty bucket uses CAS. Contended inserts/updates use synchronized on individual bucket head node only. Multiple threads writing to different buckets operate fully independently. compute/merge/computeIfAbsent are atomic read-modify-write. Much better than Hashtable's single lock covering all operations.

Q62: Why does ConcurrentHashMap disallow null keys and values? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

Deliberate design. In concurrent context, map.get(key)==null is ambiguous — does it mean key maps to null, or key is absent? In HashMap you can call containsKey() to distinguish, but another thread could remove the key between get() and containsKey(). Disallowing null eliminates this ambiguity — null return from get() always means key absent.

Q63: ConcurrentHashMap vs Collections.synchronizedMap() — key differences? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

synchronizedMap(): single mutex — all operations including reads are serialized. Compound operations not atomic. Simple but slow. ConcurrentHashMap: bucket-level granularity, reads non-blocking, atomic compute/merge/putIfAbsent. CHM is always the better choice for concurrent access. synchronizedMap() is legacy — only use it if you need to wrap an arbitrary Map implementation for occasional thread safety.

Q64: What is CopyOnWriteArrayList and when should you use it? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Every write creates a fresh copy of the entire backing array. Readers iterate an immutable snapshot — zero blocking, zero ConcurrentModificationException. Use when reads vastly outnumber writes: event listener lists, observer pattern, rarely-updated configuration. Avoid when writes are frequent — O(n) per write makes it unsuitable for high-write workloads.

Q65: CopyOnWriteArrayList vs synchronizedList — which to choose? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

synchronizedList: all ops locked including reads. Manual external synchronization required for iteration. CopyOnWriteArrayList: reads lock-free (snapshot). Writes create array copy. Iteration always safe — no external sync needed. synchronizedList wins when writes are frequent and reader concurrency is not important. COWAL wins when read throughput matters and writes are rare.

Q66: What is ConcurrentLinkedQueue? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Unbounded thread-safe Queue using lock-free CAS algorithm. All ops non-blocking. No thread ever blocks waiting for a lock. Does not support blocking (no take/put) — use LinkedBlockingQueue if threads should wait when empty/full. Good for work queues where producers/consumers check and move on without blocking.

Q67: What is the compound operation problem with synchronized collections? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

Compound operations like check-then-act involve two separate synchronized calls. Another thread can modify the collection between them. Each individual method is thread-safe but the compound operation is not. Solutions: ConcurrentHashMap's putIfAbsent/computeIfAbsent (atomic). Or external block: synchronized(map) { if(!map.containsKey(k)) map.put(k,v); }.

Q68: What is SynchronousQueue? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

Zero-capacity queue — no internal buffer. Each put() must rendezvous with a take(). Direct thread-to-thread handoff channel. Executors.newCachedThreadPool() uses it internally — tasks handed directly to waiting threads with zero queuing. Useful for zero-latency task transfer when you want immediate handoff or fail-fast if no consumer is ready.

Q69: ArrayBlockingQueue vs LinkedBlockingQueue — key differences? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

ArrayBlockingQueue: fixed bounded capacity set at creation, array-backed, single shared lock for put and take. LinkedBlockingQueue: optionally bounded, node-backed, SEPARATE locks for put and take — producers and consumers can operate concurrently. LinkedBlockingQueue preferred for most producer-consumer scenarios due to better concurrency from independent locks.

Q70: What are the limitations of Collections.synchronizedList()? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

1) All reads are locked — no reader concurrency. 2) Compound operations not atomic. 3) Iteration requires MANUAL external synchronization: synchronized(list) { for(each) } — easy to forget, causes CME or data corruption if forgotten. 4) Performance degrades under concurrent load. For modern concurrent code: CopyOnWriteArrayList (read-heavy) or concurrent data structure is preferred.

Q71: Is ConcurrentHashMap.size() always accurate? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

Not guaranteed exact under concurrency. Multiple threads adding/removing means count can be stale. mappingCount() (Java 8) gives a more accurate long count by summing internal LongAdder cells. In practice, exact real-time count is rarely needed. If you need accurate real-time count, maintain a separate AtomicLong counter externally.

Q72: How would you implement a thread-safe LRU cache? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

Simple: Collections.synchronizedMap(new LinkedHashMap<>(cap, 0.75f, true) { removeEldestEntry returns size>cap }). Wraps entire map. Better for high concurrency: Caffeine or Guava Cache — lock-striped concurrent LRU with expiry, statistics, and async loading. Production: always use Caffeine — it is the state-of-the-art Java cache library used by Spring Boot's default caching abstraction.

Q73: What is WeakHashMap? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

Map where keys are held with weak references. If no other strong reference to a key exists, GC can collect it and the entry is automatically removed. Used for: caches where entries should vanish when the key object is no longer referenced elsewhere. Warning: entries can disappear at any GC cycle. Never use as general-purpose map. Useful for extra metadata on objects without preventing their garbage collection.

Q74: What is IdentityHashMap? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

Uses reference equality (==) instead of equals() and System.identityHashCode() instead of hashCode(). Two keys are the same ONLY if they are the exact same object reference. Use cases: serialization frameworks tracking visited objects, graph traversal where equals-equal objects are distinct nodes, proxy caches. Very rare in application code but signals depth of knowledge in interviews.

Q75: BlockingQueue.offer() vs put() — when to use which? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

put(e): blocks indefinitely until space available — use in dedicated producer threads where blocking is acceptable. offer(e): non-blocking, returns false if full — for fire-and-forget with no blocking. offer(e, timeout, unit): waits up to timeout, returns false if still full — best balance for bounded-wait scenarios. Use put() for producer loops, offer(timeout) for time-bounded backpressure.

Q76: How does ConcurrentHashMap handle two threads putting the same key simultaneously? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

If bucket empty: both attempt CAS. One succeeds, other's CAS fails and it retries (now finds bucket non-empty, falls into synchronized path). If bucket has existing nodes: bucket head node is synchronized — one thread waits while other walks chain and inserts/updates. Only threads hitting the SAME bucket contend — different buckets fully independent.

Q77: What is PriorityBlockingQueue? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Unbounded BlockingQueue ordering elements by priority. take() blocks when empty but put() NEVER blocks — unbounded so never fills. For priority-based task scheduling: high-priority tasks processed before low-priority ones. Warning: unbounded — can grow without limit if consumers are slower than producers. Internally uses a binary heap like PriorityQueue.

Q78: ConcurrentSkipListMap vs ConcurrentHashMap — when to use which? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

ConcurrentSkipListMap: concurrent sorted map implementing NavigableMap. O(log n) ops (skip list). Thread-safe sorted iteration, floor/ceiling/range views — all concurrent. ConcurrentHashMap: O(1) ops but unordered. Rule: if you need thread-safety AND sorted key order, ConcurrentSkipListMap. Otherwise ConcurrentHashMap for better throughput.

Q79: Which collections do NOT throw ConcurrentModificationException? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Fail-safe collections: CopyOnWriteArrayList (snapshot iterator), ConcurrentHashMap (weakly consistent iterator), ConcurrentLinkedQueue (lock-free), ConcurrentSkipListMap/Set. Also: any collection iterated while holding the collection's own lock externally. Fail-fast collections (ArrayList, HashMap, HashSet, LinkedList) throw CME — fix code to use iterator.remove() or removeIf().

Q80: volatile vs ConcurrentHashMap for sharing state between threads — when to use which? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

volatile variable: single variable, visibility guaranteed, no atomicity for compound ops. Right for: single flag, single reference update. ConcurrentHashMap: full data structure with atomic compound operations. Right for: shared mutable key-value state with concurrent reads/writes. Neither handles multi-variable atomicity — that requires locks or higher-level coordination like AtomicReference with immutable state objects.
