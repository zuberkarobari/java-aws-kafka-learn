Creational Patterns

What creational patterns do:

They abstract the object creation process — decoupling the client from what gets created and how. The goal is to give the calling code a clean interface to get objects without knowing the construction details.

Pattern 1 — Singleton

Intent:

Ensure a class has exactly one instance and provide a global access point to it. In a Spring Boot application, every @Bean is a singleton by default — but understanding the raw pattern is still expected in interviews.

5 Implementations — Know All of Them

// ── IMPLEMENTATION 1: Eager initialisation ──
// Instance created at class load time. Simple, thread-safe by JVM class loading.
// Drawback: created even if never used.
public class PaymentEngine {
    private static final PaymentEngine INSTANCE = new PaymentEngine();
    private PaymentEngine() {}
    public static PaymentEngine getInstance() { return INSTANCE; }
}
// ── IMPLEMENTATION 2: Lazy — NOT thread-safe ──
// Created on first use. Broken in multithreaded env — two threads can both
// pass the null check and create two instances.
public class PaymentEngine {
    private static PaymentEngine instance;
    private PaymentEngine() {}
    public static PaymentEngine getInstance() {
        if (instance == null) instance = new PaymentEngine(); // RACE CONDITION
        return instance;
    }
}
// ── IMPLEMENTATION 3: Double-checked locking (correct) ──
// volatile prevents partial construction reordering.
// Lazy + thread-safe + fast after first init (no lock after creation).
public class PaymentEngine {
    private static volatile PaymentEngine instance;
    private PaymentEngine() {}
    public static PaymentEngine getInstance() {
        if (instance == null) {
            synchronized (PaymentEngine.class) {
                if (instance == null) {
                    instance = new PaymentEngine();
                }
            }
        }
        return instance;
    }
}
// ── IMPLEMENTATION 4: Bill Pugh / Holder class ──
// Best of all worlds: lazy, thread-safe, no synchronisation overhead.
// JVM guarantees the holder class is loaded only when getInstance() is called.
// This is the preferred implementation for most production code.
public class PaymentEngine {
    private PaymentEngine() {}
    private static class Holder {
        static final PaymentEngine INSTANCE = new PaymentEngine();
    }
    public static PaymentEngine getInstance() { return Holder.INSTANCE; }
}
// ── IMPLEMENTATION 5: Enum singleton ──
// JVM guarantees one instance. Handles serialisation and reflection attacks.
// Drawback: cannot extend another class.
public enum PaymentEngine {
    INSTANCE;
    public void process(PaymentRequest req) { /* ... */ }
}
// Usage: PaymentEngine.INSTANCE.process(req);

Implementation

Thread-safe? | Lazy? | Serialisation-safe? | When to use

Eager

Yes (JVM) | No | Needs readResolve | Simple, low cost to create

Lazy (no sync)

No | Yes | No | Single-threaded only — avoid

Double-checked locking

Yes | Yes | Needs readResolve | When you need lazy + performance

Bill Pugh / Holder

Yes | Yes | Needs readResolve | Best general-purpose choice

Enum

Yes | No | Yes (automatic) | When serialisation + reflection safety matter

Q1: What is the Singleton pattern and how do you implement it correctly in Java? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Singleton ensures only one instance of a class exists. The naive lazy version has a race condition — two threads can both pass the null check before either creates the instance. The correct approach is either: Bill Pugh holder class (JVM guarantees the inner class loads only on first getInstance() call — lazy, thread-safe, no locks), double-checked locking with volatile (volatile prevents partial construction being visible to other threads), or an enum (serialisation and reflection safe). In Spring Boot, you rarely implement Singleton manually — @Bean is singleton scope by default. I would use the enum approach when serialisation matters, and Bill Pugh for everything else.

Q2: Why is the naive double-checked locking broken without volatile? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

Object creation is not atomic — new SomeClass() involves allocating memory, initialising fields, and assigning the reference. A CPU can reorder the last two steps: assign the reference first, then initialise fields. Without volatile, a second thread can see a non-null reference and return a partially-constructed object. volatile adds a memory barrier that prevents this reordering — the assignment is only visible to other threads after initialisation completes.

Pattern 2 — Factory Method + Abstract Factory

Intent:

Factory Method defines an interface for creating an object but lets subclasses decide which class to instantiate. Abstract Factory groups related factories together, creating families of objects without specifying concrete classes.

// ── FACTORY METHOD ──
// Define a creation interface; let subclasses or implementations decide what to create.
public interface PaymentGateway {
    PaymentResult process(PaymentRequest req);
}
public class StripeGateway  implements PaymentGateway { /* ... */ }
public class RazorpayGateway implements PaymentGateway { /* ... */ }
public class PaytmGateway   implements PaymentGateway { /* ... */ }
// Factory method — returns the right implementation based on context
public class PaymentGatewayFactory {
    public static PaymentGateway create(String provider) {
        return switch (provider.toUpperCase()) {
            case "STRIPE"   -> new StripeGateway();
            case "RAZORPAY" -> new RazorpayGateway();
            case "PAYTM"    -> new PaytmGateway();
            default -> throw new IllegalArgumentException("Unknown provider: " + provider);
        };
    }
}
// Callers only know PaymentGateway — completely decoupled from implementations.
// ── ABSTRACT FACTORY ──
// Creates families of related objects. Example: payment infra per region.
public interface PaymentInfraFactory {
    PaymentGateway   createGateway();
    FraudDetector    createFraudDetector();
    NotificationSvc  createNotificationService();
}
// Indian market: Razorpay + ML fraud model + SMS notifications
public class IndiaPaymentFactory implements PaymentInfraFactory {
    public PaymentGateway  createGateway()             { return new RazorpayGateway(); }
    public FraudDetector   createFraudDetector()       { return new IndiaFraudDetector(); }
    public NotificationSvc createNotificationService() { return new SmsNotificationSvc(); }
}
// US market: Stripe + rule-based fraud + email notifications
public class USPaymentFactory implements PaymentInfraFactory {
    public PaymentGateway  createGateway()             { return new StripeGateway(); }
    public FraudDetector   createFraudDetector()       { return new RuleBasedFraudDetector(); }
    public NotificationSvc createNotificationService() { return new EmailNotificationSvc(); }
}
// Spring Boot usage: @ConditionalOnProperty or @Profile to select factory
@Bean
@ConditionalOnProperty(name="payment.region", havingValue="india")
public PaymentInfraFactory indiaFactory() { return new IndiaPaymentFactory(); }
Q3: What is the difference between Factory Method and Abstract Factory? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Factory Method is about one product — you define an interface for creating a single type of object and let subclasses or implementations decide the concrete type. Abstract Factory is about families of related products — it groups multiple factory methods together so all created objects are compatible with each other. If you have one thing to create (a payment gateway), use Factory Method. If you need a consistent set of collaborating objects (gateway + fraud detector + notifier all configured for the same region), use Abstract Factory. In Spring, @Bean methods in a @Configuration class are essentially Abstract Factory.

Pattern 3 — Builder

Intent:

Construct a complex object step by step. The builder pattern shines when an object has many optional parameters, when telescoping constructors become unreadable, or when you need immutable objects with a clean construction API.

// The problem Builder solves — telescoping constructors are unreadable:
// new PaymentRequest("ACC-001", "ACC-002", new BigDecimal("500"), "INR", null, null, false, null)
// You cannot tell which argument is which just by reading the call site.
// Builder pattern: named, fluent, readable construction
public class PaymentRequest {
    private final String        fromAccount;   // required
    private final String        toAccount;     // required
    private final BigDecimal    amount;        // required
    private final String        currency;      // required
    private final String        description;   // optional
    private final String        referenceId;   // optional
    private final boolean       idempotent;    // optional, default false
    private final Instant       scheduledAt;   // optional
    private PaymentRequest(Builder b) {
        this.fromAccount = b.fromAccount;
        this.toAccount   = b.toAccount;
        this.amount      = b.amount;
        this.currency    = b.currency;
        this.description = b.description;
        this.referenceId = b.referenceId;
        this.idempotent  = b.idempotent;
        this.scheduledAt = b.scheduledAt;
    }
    // ─ Inner Builder ─
    public static class Builder {
        // Required fields
        private final String     fromAccount;
        private final String     toAccount;
        private final BigDecimal amount;
        private final String     currency;
        // Optional fields with defaults
        private String  description;
        private String  referenceId;
        private boolean idempotent = false;
        private Instant scheduledAt;
        public Builder(String from, String to, BigDecimal amount, String currency) {
            this.fromAccount = Objects.requireNonNull(from, "fromAccount required");
            this.toAccount   = Objects.requireNonNull(to,   "toAccount required");
            this.amount      = Objects.requireNonNull(amount, "amount required");
            this.currency    = Objects.requireNonNull(currency, "currency required");
        }
        public Builder description(String d)  { this.description = d;  return this; }
        public Builder referenceId(String r)  { this.referenceId = r;  return this; }
        public Builder idempotent(boolean i)  { this.idempotent  = i;  return this; }
        public Builder scheduledAt(Instant t) { this.scheduledAt = t;  return this; }
        public PaymentRequest build() {
            // Validate before building — fail fast
            if (amount.compareTo(BigDecimal.ZERO) <= 0)
                throw new IllegalStateException("Amount must be positive");
            return new PaymentRequest(this);
        }
    }
}
// Clean, readable, self-documenting construction:
PaymentRequest req = new PaymentRequest.Builder("ACC-001", "ACC-002",
        new BigDecimal("500.00"), "INR")
    .description("Rent payment")
    .referenceId("REF-2024-001")
    .idempotent(true)
    .build();
// Lombok @Builder generates all of this — but interviewers want to see
// you understand what Lombok is generating under the hood.
// Spring WebClient — Builder pattern in production
WebClient client = WebClient.builder()
    .baseUrl("https://api.payment-gateway.com")
    .defaultHeader(HttpHeaders.CONTENT_TYPE, MediaType.APPLICATION_JSON_VALUE)
    .filter(loggingFilter())
    .build();
Q4: What problem does the Builder pattern solve and when would you use it? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

It solves two problems: telescoping constructors (objects with many optional parameters become impossible to read), and inconsistent partially-constructed objects (you want to enforce that an object is only created in a valid state). I use it when a class has more than 3-4 parameters, when some are optional, or when the object should be immutable after creation. In our payment service, PaymentRequest has about 8 fields — without Builder you end up with several overloaded constructors and it becomes unclear at the call site what each argument represents. Lombok's @Builder does this automatically in Spring Boot projects, but I understand what it generates.

Pattern 4 — Prototype

Intent:

// Prototype via Cloneable — avoid this approach.
// clone() is protected in Object, Cloneable is a marker with no methods,
// and it does a shallow copy by default — mutable fields are shared.
// ── PREFERRED: Copy constructor ──
public class TransactionTemplate {
    private final String  accountId;
    private final String  currency;
    private final String  category;
    private final Map<String, String> metadata; // mutable — needs deep copy
    // Copy constructor — explicit, clear, deep copy where needed
    public TransactionTemplate(TransactionTemplate source) {
        this.accountId = source.accountId;
        this.currency  = source.currency;
        this.category  = source.category;
        this.metadata  = new HashMap<>(source.metadata); // deep copy of mutable field
    }
    public TransactionTemplate withAccount(String newAccountId) {
        TransactionTemplate copy = new TransactionTemplate(this);
        // Not quite — field is final. Use Builder with prototype instead:
        return new TransactionTemplate.Builder(this).accountId(newAccountId).build();
    }
}
// ── PROTOTYPE + BUILDER — most practical combination ──
// Create a template for recurring salary credits, then copy with minor changes each month.
PaymentRequest salaryTemplate = new PaymentRequest.Builder("CORP-001", "EMP-101",
        new BigDecimal("85000.00"), "INR")
    .description("Monthly salary")
    .category("SALARY")
    .build();
// Next month: same template, just a new reference ID
PaymentRequest monthlyRun = PaymentRequest.Builder.from(salaryTemplate)
    .referenceId("SAL-2024-12")
    .scheduledAt(Instant.parse("2024-12-01T00:00:00Z"))
    .build();
Q5: What is the difference between shallow copy and deep copy in the Prototype pattern? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Shallow copy copies field values — for primitives and immutable types (String, Integer) that is fine. For mutable reference types (List, HashMap), a shallow copy means both the original and the copy point to the same underlying object. Changing the list through the copy changes it for the original too. Deep copy creates new instances of all mutable fields recursively. In Java, Cloneable.clone() does a shallow copy by default, which is why it causes subtle bugs. The copy constructor approach is cleaner because it is explicit about what gets copied deeply.

Structural Patterns

What structural patterns do:

They deal with how classes and objects are composed to form larger structures. The key insight is that structural patterns make it easy to add new capabilities to existing code without modifying it — which directly supports the Open/Closed Principle.

Pattern 5 — Adapter

Intent:

Convert the interface of a class into the interface a client expects. Makes incompatible interfaces work together. This is one of the most commonly used patterns in real backend code — any time you integrate a third-party library with your domain model.

// Scenario: your domain uses PaymentGateway interface,
// but you are integrating a third-party Razorpay SDK that has its own API.
// Your domain interface
public interface PaymentGateway {
    PaymentResult charge(String accountId, BigDecimal amount, String currency);
    RefundResult  refund(String transactionId, BigDecimal amount);
}
// Third-party SDK — you cannot change this
public class RazorpayClient {
    public RazorpayOrder createOrder(double amountPaise, String currencyCode) { /* ... */ return null; }
    public RazorpayPayment capturePayment(String orderId, double amountPaise)  { /* ... */ return null; }
    public RazorpayRefund  initiateRefund(String paymentId, double amountPaise){ /* ... */ return null; }
}
// Adapter: wraps RazorpayClient, exposes PaymentGateway interface
@Component
public class RazorpayAdapter implements PaymentGateway {
    private final RazorpayClient razorpay;
    public RazorpayAdapter(RazorpayClient razorpay) {
        this.razorpay = razorpay;
    }
    @Override
    public PaymentResult charge(String accountId, BigDecimal amount, String currency) {
        // Adapt: BigDecimal INR → double paise (Razorpay uses smallest unit)
        double amountPaise = amount.multiply(BigDecimal.valueOf(100)).doubleValue();
        RazorpayOrder   order   = razorpay.createOrder(amountPaise, currency);
        RazorpayPayment payment = razorpay.capturePayment(order.getId(), amountPaise);
        return new PaymentResult(payment.getId(), payment.getStatus());
    }
    @Override
    public RefundResult refund(String transactionId, BigDecimal amount) {
        double amountPaise  = amount.multiply(BigDecimal.valueOf(100)).doubleValue();
        RazorpayRefund ref  = razorpay.initiateRefund(transactionId, amountPaise);
        return new RefundResult(ref.getId(), ref.getStatus());
    }
}
// Service uses PaymentGateway — has no idea Razorpay is behind it.
// Swapping to Stripe: write StripeAdapter, update one @Bean. No service changes.
Q6: When would you use the Adapter pattern in a real Spring Boot project? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Any time you are wrapping a third-party library that has its own API and you want to keep your domain code clean. In my Deloitte project I used this to wrap AWS SDK calls — the service layer calls a CloudWatchPort interface, and the adapter translates those calls to the AWS SDK. This means if we ever migrate to another cloud, only the adapter changes. The same approach works for payment gateways, SMS providers, and any external dependency. It also makes testing much easier — in tests I can inject a mock implementation of my interface without touching the AWS SDK at all.

Pattern 6 — Decorator

Intent:

Attach additional responsibilities to an object dynamically. Decorators provide a flexible alternative to subclassing for extending behaviour. Java I/O streams are the classic example — BufferedInputStream decorates FileInputStream without changing it.

// Core interface
public interface PaymentService {
    PaymentResult process(PaymentRequest req);
}
// Core implementation
@Service
public class BasicPaymentService implements PaymentService {
    public PaymentResult process(PaymentRequest req) {
        // actual payment processing
        return processInternal(req);
    }
}
// Decorator 1: adds logging
public class LoggingPaymentDecorator implements PaymentService {
    private final PaymentService delegate;
    private final Logger log = LoggerFactory.getLogger(getClass());
    public LoggingPaymentDecorator(PaymentService delegate) {
        this.delegate = delegate;
    }
    @Override
    public PaymentResult process(PaymentRequest req) {
        log.info("Processing payment: accountId={} amount={}", req.getAccountId(), req.getAmount());
        PaymentResult result = delegate.process(req);
        log.info("Payment complete: transactionId={} status={}", result.getId(), result.getStatus());
        return result;
    }
}
// Decorator 2: adds metrics
public class MetricsPaymentDecorator implements PaymentService {
    private final PaymentService delegate;
    private final MeterRegistry  meterRegistry;
    public MetricsPaymentDecorator(PaymentService delegate, MeterRegistry registry) {
        this.delegate      = delegate;
        this.meterRegistry = registry;
    }
    @Override
    public PaymentResult process(PaymentRequest req) {
        Timer.Sample sample = Timer.start(meterRegistry);
        try {
            PaymentResult result = delegate.process(req);
            sample.stop(meterRegistry.timer("payment.process", "status", "success"));
            return result;
        } catch (Exception e) {
            sample.stop(meterRegistry.timer("payment.process", "status", "error"));
            throw e;
        }
    }
}
// Decorator 3: adds retry
// Wrap them together — order matters: logging wraps metrics wraps retry wraps core
@Bean
public PaymentService paymentService(BasicPaymentService core, MeterRegistry registry) {
    return new LoggingPaymentDecorator(
               new MetricsPaymentDecorator(
                   new RetryPaymentDecorator(core, 3),
               registry));
}
// In practice, Spring AOP does this declaratively — but the Decorator pattern
// is what AOP is implementing under the hood.
Q7: How does the Decorator pattern differ from inheritance? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Inheritance is static — you decide at compile time what behaviour a class has. Decorator is dynamic — you compose behaviour at runtime by wrapping objects. With inheritance, adding three cross-cutting concerns (logging, metrics, retry) to ten service classes means thirty subclasses. With Decorator, you write three decorators and wrap any service. Decorator also follows the Single Responsibility Principle better — each decorator does one thing. Java I/O is the canonical example: BufferedInputStream, GZIPInputStream, and CipherInputStream are all decorators that wrap any InputStream. Spring AOP is Decorator applied automatically via proxies.

Pattern 7 — Facade

Intent:

Provide a simplified interface to a complex subsystem. The facade does not add new functionality — it makes an existing system easier to use by hiding complexity behind a clean, focused API.

// Complex subsystem — many independent services
@Service public class AccountService     { /* validate, debit, credit */ }
@Service public class FraudService       { /* run fraud checks */ }
@Service public class NotificationService{ /* send SMS, email, push */ }
@Service public class AuditService       { /* write audit trail */ }
@Service public class LedgerService      { /* double-entry bookkeeping */ }
// Without Facade: every controller and consumer must orchestrate all 5 services.
// With Facade: one entry point that handles the orchestration.
@Service
public class PaymentFacade {
    private final AccountService      accounts;
    private final FraudService        fraud;
    private final NotificationService notifications;
    private final AuditService        audit;
    private final LedgerService       ledger;
    // Constructor injection omitted for brevity
    public PaymentResult transfer(PaymentRequest req) {
        // Clients call ONE method — Facade orchestrates everything
        fraud.check(req);                              // 1. Fraud check
        accounts.debit(req.getFrom(), req.getAmount()); // 2. Debit source
        accounts.credit(req.getTo(), req.getAmount());  // 3. Credit dest
        ledger.record(req);                            // 4. Ledger entry
        audit.log(req, "TRANSFER_COMPLETE");          // 5. Audit trail
        notifications.sendConfirmation(req);           // 6. Notify user
        return PaymentResult.success(generateTxId());
    }
}
// Controller just calls the facade — has no idea about the 5 internal services.
@RestController
public class PaymentController {
    private final PaymentFacade facade;
    @PostMapping("/transfer")
    public ResponseEntity<PaymentResult> transfer(@RequestBody PaymentRequest req) {
        return ResponseEntity.ok(facade.transfer(req));
    }
}
Q8: What is the Facade pattern and how does it help in a microservices architecture? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Facade provides a single simplified entry point to a complex subsystem. In microservices, the API Gateway is a Facade — clients call one service and it routes, aggregates, and transforms responses from multiple downstream services. Within a single service, the service layer is a Facade over the repository, fraud, notification, and audit layers. The benefit is that the controller only knows about the facade — if the internal orchestration changes, only the facade changes. It also makes testing easier: to test the controller you mock the facade, not five separate services.

Pattern 8 — Proxy

Intent:

Provide a surrogate that controls access to another object. The proxy has the same interface as the real object — the client cannot tell the difference. Common uses: lazy loading, access control, caching, remote access, and logging.

// ── STATIC PROXY ──
public interface AccountRepository {
    Account findById(String accountId);
    void    save(Account account);
}
// Caching proxy — sits in front of the real repository
public class CachingAccountRepository implements AccountRepository {
    private final AccountRepository    real;
    private final Map<String, Account> cache = new ConcurrentHashMap<>();
    public CachingAccountRepository(AccountRepository real) { this.real = real; }
    @Override
    public Account findById(String accountId) {
        return cache.computeIfAbsent(accountId, real::findById);
    }
    @Override
    public void save(Account account) {
        real.save(account);
        cache.put(account.getId(), account); // keep cache consistent
    }
}
// ── DYNAMIC PROXY (java.lang.reflect) ──
// Creates a proxy at runtime without a pre-written proxy class.
// This is exactly how Spring AOP and Spring Data JPA work.
AccountRepository proxy = (AccountRepository) Proxy.newProxyInstance(
    AccountRepository.class.getClassLoader(),
    new Class[]{ AccountRepository.class },
    (proxyObj, method, args) -> {
        long start = System.currentTimeMillis();
        Object result = method.invoke(realRepository, args); // delegate to real
        long elapsed = System.currentTimeMillis() - start;
        log.info("{}() took {}ms", method.getName(), elapsed);
        return result;
    }
);
// ── SPRING AOP IS A PROXY ──
// @Transactional, @Cacheable, @Async, @Retryable all work by wrapping
// your @Bean in a proxy at application startup.
// This is why self-invocation (calling @Transactional from within the same class)
// does not work — you bypass the proxy when calling this.method().
// ── JPA LAZY LOADING IS A PROXY ──
// account.getTransactions() returns a Hibernate proxy — the actual SELECT
// is deferred until you first access the collection.
Q9: What is the Proxy pattern and how does Spring use it? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

A proxy wraps the real object with the same interface and adds behaviour around method calls. Spring uses JDK dynamic proxies or CGLIB byte-code proxies for almost everything AOP-related. When you annotate a @Bean method with @Transactional, Spring replaces the bean with a proxy that opens a transaction before the method and commits/rolls back after. Same for @Cacheable, @Async, and @Retryable. The practical consequence is the self-invocation problem: if you call a @Transactional method from within the same class using this.method(), you bypass the proxy and the transaction annotation has no effect. The fix is to inject the proxy via the application context or restructure the code.

Behavioural Patterns

What behavioural patterns do:

They define how objects communicate and share responsibility. The recurring theme is separating what you do from how and when you do it — which directly enables extensibility, testability, and loose coupling.

Pattern 9 — Observer

Intent:

Define a one-to-many dependency between objects so that when one object changes state, all dependents are notified and updated automatically. This is the foundation of event-driven architecture.

// ── PURE JAVA OBSERVER ──
public interface PaymentEventListener {
    void onPaymentCompleted(PaymentCompletedEvent event);
}
public class PaymentEventPublisher {
    private final List<PaymentEventListener> listeners = new CopyOnWriteArrayList<>();
    public void subscribe(PaymentEventListener listener)   { listeners.add(listener); }
    public void unsubscribe(PaymentEventListener listener) { listeners.remove(listener); }
    public void publish(PaymentCompletedEvent event) {
        listeners.forEach(l -> l.onPaymentCompleted(event));
    }
}
// Concrete listeners — each handles its own concern
@Component
public class AuditEventListener implements PaymentEventListener {
    public void onPaymentCompleted(PaymentCompletedEvent e) {
        auditService.record(e.getTransactionId(), e.getAmount());
    }
}
// ── SPRING ApplicationEvent — Spring's built-in Observer ──
// 1. Define the event
public class PaymentCompletedEvent extends ApplicationEvent {
    private final String transactionId;
    private final BigDecimal amount;
    public PaymentCompletedEvent(Object source, String txId, BigDecimal amount) {
        super(source);
        this.transactionId = txId;
        this.amount        = amount;
    }
}
// 2. Publish from service
@Service
public class PaymentService {
    @Autowired private ApplicationEventPublisher publisher;
    public PaymentResult process(PaymentRequest req) {
        PaymentResult result = processInternal(req);
        publisher.publishEvent(new PaymentCompletedEvent(this, result.getId(), req.getAmount()));
        return result;
    }
}
// 3. Listen with @EventListener
@Component
public class NotificationEventHandler {
    @EventListener
    public void handle(PaymentCompletedEvent event) {
        notificationService.sendConfirmation(event.getTransactionId());
    }
}
// @Async on the listener makes it non-blocking — handler runs in a separate thread.
// @TransactionalEventListener: fires only after the publishing transaction commits.
// Kafka is Observer at distributed scale — producers publish, consumer groups observe.
Q10: What is the Observer pattern and how does Spring implement it? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Observer defines a one-to-many relationship where a publisher notifies all registered subscribers when something happens. The publisher does not know or care who is listening — it just fires the event. Spring implements this with ApplicationEventPublisher and @EventListener. The practical benefit is decoupling: when a payment completes, the PaymentService does not need to call the NotificationService, AuditService, and LedgerService directly — it just fires an event and each service listens independently. This means you can add a new listener (say, a fraud alert service) without touching the PaymentService. For async listeners I use @Async so the payment service is not blocked waiting for notifications to send.

Pattern 10 — Strategy

Intent:

Define a family of algorithms, encapsulate each one, and make them interchangeable. Strategy lets the algorithm vary independently from the clients that use it.

// Strategy interface
public interface PaymentRoutingStrategy {
    PaymentGateway selectGateway(PaymentRequest req, List<PaymentGateway> available);
}
// Strategy 1: lowest fee routing
@Component("lowestFeeRouting")
public class LowestFeeRoutingStrategy implements PaymentRoutingStrategy {
    @Override
    public PaymentGateway selectGateway(PaymentRequest req, List<PaymentGateway> available) {
        return available.stream()
            .min(Comparator.comparing(g -> g.getFeePercentage(req.getAmount())))
            .orElseThrow();
    }
}
// Strategy 2: highest reliability routing
@Component("highReliabilityRouting")
public class HighReliabilityRoutingStrategy implements PaymentRoutingStrategy {
    @Override
    public PaymentGateway selectGateway(PaymentRequest req, List<PaymentGateway> available) {
        return available.stream()
            .max(Comparator.comparing(PaymentGateway::getUptimePercentage))
            .orElseThrow();
    }
}
// Strategy 3: round-robin for load balancing
@Component("roundRobinRouting")
public class RoundRobinRoutingStrategy implements PaymentRoutingStrategy {
    private final AtomicInteger counter = new AtomicInteger(0);
    @Override
    public PaymentGateway selectGateway(PaymentRequest req, List<PaymentGateway> available) {
        int idx = counter.getAndIncrement() % available.size();
        return available.get(idx);
    }
}
// Context: uses whichever strategy is injected
@Service
public class PaymentRouter {
    private final List<PaymentGateway>    gateways;
    private final PaymentRoutingStrategy  strategy;
    // @Qualifier selects which strategy Spring injects
    public PaymentRouter(List<PaymentGateway> gateways,
                         @Qualifier("lowestFeeRouting") PaymentRoutingStrategy strategy) {
        this.gateways = gateways;
        this.strategy = strategy;
    }
    public PaymentResult route(PaymentRequest req) {
        PaymentGateway gateway = strategy.selectGateway(req, gateways);
        return gateway.process(req);
    }
}
// In Java 8+, simple strategies are often just lambdas:
PaymentRoutingStrategy cheapest = (req, gateways) ->
    gateways.stream().min(Comparator.comparing(g -> g.getFeePercentage(req.getAmount()))).orElseThrow();
Q11: How does the Strategy pattern relate to the Open/Closed Principle? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Open/Closed says a class should be open for extension but closed for modification. Strategy achieves this: when you need a new payment routing algorithm you add a new strategy class — you do not touch the PaymentRouter. The router is closed for modification. The strategy interface is open for extension. In Spring, this maps naturally to @Component + @Qualifier: each strategy is a separate bean and you inject the right one by qualifier or with @ConditionalOnProperty. Without Strategy, you would have a big if-else block in the router — adding a new routing algorithm means modifying the router and risking breaking existing ones.

Pattern 11 — Template Method

Intent:

Define the skeleton of an algorithm in a base class and defer some steps to subclasses. Template Method ensures the overall structure is fixed while allowing subclasses to override specific steps without changing the overall flow.

// The algorithm skeleton is fixed — what varies is the specific steps.
public abstract class PaymentProcessor {
    // Template method — defines the invariant flow, marked final
    public final PaymentResult process(PaymentRequest req) {
        validate(req);             // step 1: always runs
        String txId = initiateTransaction(req); // step 2: varies by processor
        PaymentResult result = executePayment(req, txId); // step 3: varies
        if (result.isSuccessful()) {
            postProcess(req, result); // step 4: optional hook
        }
        audit(req, result);        // step 5: always runs
        return result;
    }
    // Fixed steps — shared logic
    private void validate(PaymentRequest req) {
        if (req.getAmount().compareTo(BigDecimal.ZERO) <= 0)
            throw new IllegalArgumentException("Amount must be positive");
    }
    private void audit(PaymentRequest req, PaymentResult result) {
        // shared audit logging
    }
    // Abstract steps — subclasses must implement
    protected abstract String        initiateTransaction(PaymentRequest req);
    protected abstract PaymentResult executePayment(PaymentRequest req, String txId);
    // Hook — subclasses may optionally override (empty default)
    protected void postProcess(PaymentRequest req, PaymentResult result) {}
}
// Stripe implementation — fills in the gaps
public class StripePaymentProcessor extends PaymentProcessor {
    @Override
    protected String initiateTransaction(PaymentRequest req) {
        return stripeClient.createPaymentIntent(req.getAmount(), req.getCurrency());
    }
    @Override
    protected PaymentResult executePayment(PaymentRequest req, String txId) {
        return stripeClient.confirm(txId);
    }
}
// ── SPRING TEMPLATE METHOD IN THE WILD ──
// JdbcTemplate.query(): fixed skeleton (get connection, prepare statement,
//   execute, map results, close) — you provide the SQL and RowMapper.
// AbstractMessageListenerContainer: fixed lifecycle (start, poll, stop) —
//   you provide handleMessage().
// Spring Security's AbstractAuthenticationProcessingFilter: fixed auth flow —
//   you implement attemptAuthentication().
Q12: What is the Template Method pattern and where does Spring use it? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Template Method defines a fixed algorithm skeleton in a base class and lets subclasses fill in the varying steps. The key is the template method is final — subclasses cannot change the overall flow, only the specific steps. Spring uses this everywhere: JdbcTemplate.query() handles connection management, statement preparation, and resource cleanup — you only provide the SQL and how to map rows. AbstractMessageListenerContainer in Spring Kafka handles the polling loop, error handling, and offset management — you just implement the message handling logic. I used this pattern in my payment processor hierarchy to ensure every payment type goes through the same validation and audit steps, regardless of the underlying gateway.

Pattern 12 — Command

Intent:

Encapsulate a request as an object. This lets you queue, log, and undo operations, and makes it easy to implement retry logic, scheduling, and transactional behaviour around any action.

// Command interface
public interface PaymentCommand {
    PaymentResult execute();
    void          undo();  // optional — for reversible operations
}
// Concrete command: encapsulates everything needed to execute a debit
public class DebitCommand implements PaymentCommand {
    private final AccountService accountService;
    private final String         accountId;
    private final BigDecimal     amount;
    private       boolean        executed = false;
    public DebitCommand(AccountService svc, String accountId, BigDecimal amount) {
        this.accountService = svc;
        this.accountId      = accountId;
        this.amount         = amount;
    }
    @Override
    public PaymentResult execute() {
        accountService.debit(accountId, amount);
        executed = true;
        return PaymentResult.success(accountId);
    }
    @Override
    public void undo() {
        if (executed) accountService.credit(accountId, amount); // reverse the debit
    }
}
// Invoker: runs commands, can queue them, retry them, or roll them back
public class PaymentCommandInvoker {
    private final Deque<PaymentCommand> history = new ArrayDeque<>();
    public PaymentResult execute(PaymentCommand cmd) {
        PaymentResult result = cmd.execute();
        history.push(cmd); // remember for potential undo
        return result;
    }
    public void undoLast() {
        if (!history.isEmpty()) history.pop().undo();
    }
}
// ── COMMAND AS MESSAGE — connecting to Kafka ──
// In an event-driven system, Kafka messages ARE commands:
// A ProcessPaymentCommand is serialised to Kafka, consumed by the processor.
// This gives you: async processing, retry (consumer reprocesses on failure),
// audit trail (Kafka log IS the command history), and replay.
public record ProcessPaymentCommand(
    String commandId,
    String accountId,
    BigDecimal amount,
    String currency,
    Instant issuedAt
) {}
@KafkaListener(topics = "payment.commands")
public void handle(ProcessPaymentCommand command) {
    paymentService.process(command); // execute the encapsulated request
}
Q13: What is the Command pattern and how does it relate to Kafka message processing? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Command encapsulates a request — everything needed to perform an action — as an object. This means you can queue it, log it, retry it, and undo it. The connection to Kafka is direct: a Kafka message is essentially a Command object serialised onto a topic. The producer encapsulates the request as a message, Kafka queues it, and consumers execute it. You get retry for free (consumer re-reads on failure), audit trail for free (the Kafka log is the history of all commands), and replay for free (reset the consumer offset). In my VISA payment system at Infosys, every payment instruction was a message on Kafka — the Command pattern applied at distributed scale.

Interview Q&A — 15 Questions

How to use:

Read the question, pause and say your answer out loud before reading the model answer. The model answers are written in a natural, conversational style — the kind of answer that lands well in a real interview, not a textbook recitation.

Q14: What are the three categories of design patterns from the Gang of Four? [Easy] ▼
💡 Ideal Answer (Amazon/GS Standard)

Creational, Structural, and Behavioural. Creational patterns deal with how objects are created — Singleton, Factory, Builder, Prototype. Structural patterns deal with how objects are composed — Adapter, Decorator, Facade, Proxy. Behavioural patterns deal with how objects communicate — Observer, Strategy, Template Method, Command. The practical split I think about is: Creational = control what you get, Structural = control how things fit together, Behavioural = control how things talk to each other.

Q15: What is the Singleton pattern and how do you implement it correctly in Java? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Singleton guarantees exactly one instance. The naive lazy version has a race condition. I use the Bill Pugh holder class in most cases — the inner static class loads lazily via JVM class loading guarantees, no locks needed, no volatile. For anything that needs serialisation safety I use an enum. In Spring, I rarely write Singleton manually because @Bean is singleton scope by default, but understanding the pattern matters when the interviewer wants to explore thread safety and JVM internals.

Q16: Why is the naive double-checked locking broken without volatile? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

Object creation is not atomic. new SomeClass() involves allocate memory, initialise fields, and assign the reference. The CPU can reorder the last two: assign first, initialise later. Without volatile, another thread can see a non-null reference and return a partially-constructed object. volatile adds a memory barrier — the assignment is visible to other threads only after initialisation completes.

Q17: What problem does the Builder pattern solve? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Telescoping constructors and partially-constructed objects. When a class has many optional parameters you end up with multiple overloaded constructors, or one giant constructor where you cannot tell what each argument means at the call site. Builder gives every parameter a name and lets you skip optionals. It also ensures the object is validated and complete before build() returns — you cannot get a half-baked PaymentRequest into the system. Lombok's @Builder generates this automatically, but I understand what it produces.

Q18: When would you use Abstract Factory over Factory Method? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

When you need a family of related objects that must be compatible with each other. Factory Method creates one type of object. Abstract Factory creates multiple related types together. Example: a payment system for different regions — India needs Razorpay + SMS notifications + India fraud rules, the US needs Stripe + email + US fraud rules. If you mixed them you would get an inconsistent system. Abstract Factory ensures the entire family is internally consistent. In Spring, a @Configuration class is essentially an Abstract Factory.

Q19: What is the difference between Adapter and Facade? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Adapter makes an existing interface compatible with what the client expects — it adapts without hiding. The client still uses a rich interface, just one that maps to the underlying library. Facade simplifies — it hides a complex subsystem behind a much simpler interface, usually exposing far fewer methods than the underlying components. Adapter is about compatibility, Facade is about simplicity. In practice: RazorpayAdapter makes the Razorpay SDK look like my PaymentGateway interface (same richness, different shape). PaymentFacade hides five internal services behind one transfer() method (radically simpler).

Q20: How does the Decorator pattern differ from subclassing? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Subclassing is static — decided at compile time, tightly coupled to the base class. Decorator is dynamic — you wrap objects at runtime and can combine decorators freely. Adding logging, metrics, and retry to ten services via subclassing means thirty subclasses. With Decorator you write three wrappers and compose them. Decorators also follow Single Responsibility better — each one does exactly one thing. Java I/O is the classic example. Spring AOP is Decorator applied automatically via proxies.

Q21: What is a Proxy and how does Spring use it? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

A proxy has the same interface as the real object and adds behaviour around method calls. Spring creates JDK dynamic proxies or CGLIB proxies for @Transactional, @Cacheable, @Async, @Retryable. The practical consequence is the self-invocation trap: calling a @Transactional method from within the same class via this bypasses the proxy, so the transaction does not start. The fix is to either inject the bean into itself via ApplicationContext or restructure the code so the transactional method is on a separate bean. JPA lazy loading is also a proxy — Hibernate creates a subclass proxy that defers the SELECT until you first access the collection.

Q22: Explain the Observer pattern and give a Spring Boot example. [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Observer is one-to-many: when something happens, all registered listeners are notified. The publisher does not know who is listening. Spring's ApplicationEventPublisher and @EventListener implement this. The benefit is decoupling — my PaymentService fires a PaymentCompletedEvent and never directly calls NotificationService or AuditService. Each service listens independently. Adding a new listener means a new @EventListener class, no changes to PaymentService. For async I add @Async on the listener. For audit I use @TransactionalEventListener so the audit record only writes if the payment transaction actually commits.

Q23: What is the Strategy pattern and how does it relate to Spring @Qualifier? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Strategy defines a family of interchangeable algorithms behind an interface. In Spring, multiple implementations of the same interface are multiple strategies — @Qualifier picks which one to inject. My PaymentRouter accepts a PaymentRoutingStrategy. There are three implementations: lowest fee, highest reliability, round-robin. The router does not care which one it has. I can switch strategies by changing the @Qualifier or making it config-driven. This is the Open/Closed Principle in practice — add a new routing algorithm without touching the router.

Q24: What is the Template Method pattern and where does Spring use it? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Template Method defines the invariant skeleton of an algorithm in a base class, marked final, and lets subclasses fill in the varying steps. The overall flow is fixed — only the specific steps change. Spring uses this everywhere: JdbcTemplate.query() handles all JDBC boilerplate and asks you for the SQL and RowMapper. Spring Kafka's AbstractMessageListenerContainer handles polling, thread management, and error handling — you provide handleMessage(). I used it in my payment processor hierarchy to ensure every payment goes through the same validation, audit, and error handling regardless of the gateway underneath.

Q25: How does the Command pattern relate to event-driven systems and Kafka? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Command encapsulates a request as an object — which makes it serialisable, queueable, and replayable. A Kafka message is a Command object serialised onto a topic. The producer encapsulates the request, Kafka queues it durably, and consumers execute it. You get retry for free (re-read on failure), audit trail for free (the log IS the history), and replay for free (reset consumer offset). In my VISA payment system at Infosys, every payment instruction was a ProcessPaymentCommand on Kafka. The undo operation maps to a compensation message on a separate topic.

Q26: What is the Open/Closed Principle and which patterns implement it best? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Open for extension, closed for modification — you should be able to add new behaviour without changing existing code. Strategy implements it most directly: add a new algorithm by adding a new class, never touching the context. Decorator adds new behaviour by wrapping without modifying. Observer adds new listeners without touching the publisher. In my payment service, every time a new post-payment action was needed (say, loyalty points) I added a new @EventListener — never touched the PaymentService. That is Open/Closed in practice.

Q27: Can you give an example of a design pattern you used in your Deloitte project? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

In the Bitbucket event processing system I used Adapter to wrap the Bitbucket webhook payload into our internal domain events. The raw JSON from Bitbucket is quite different from what downstream consumers expect — the adapter normalised it. I also used Strategy for the event routing: depending on the event type (push, PR opened, PR merged) a different handler strategy was invoked. For the CloudWatch log lookup feature I used Facade — the service layer exposed a single getLogs(accountId, timeRange) method and internally coordinated CloudWatch Logs, S3 archival, and the response assembly. The controllers never knew about the underlying AWS complexity.

Q28: What is the difference between Command and Strategy? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

Strategy encapsulates an algorithm — it is about HOW something is done. You swap strategies to change the implementation. Command encapsulates a request — it is about WHAT should be done, along with all the context to do it. Commands are typically one-time, stateful (they carry the data), and can be queued, logged, and undone. Strategies are stateless (usually) and swappable. In a payment system: LowestFeeRoutingStrategy is a Strategy. ProcessPaymentCommand(accountId, amount, currency, timestamp) is a Command. You might use a Strategy to decide HOW to execute a Command.
