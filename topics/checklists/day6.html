The Database-Per-Service Pattern — The Rule That Changes Everything

Every microservices problem flows from this one mandatory rule:

Concern

Monolith vs Microservices Trade-off

Development speed

Monolith: faster for small teams (one codebase, one deploy). Microservices: faster at scale (independent teams deploy independently).

Deployment

Monolith: one artifact, simple but all-or-nothing deploy. Microservices: independent deploys enable zero-downtime, per-service scaling.

Data consistency

Monolith: ACID transactions across all data trivially. Microservices: eventual consistency only across service boundaries — much harder.

Operational complexity

Monolith: simple — one process to monitor. Microservices: requires service discovery, distributed tracing, centralized logging, orchestration (K8s).

Fault isolation

When to use microservices

// Synchronous: REST (HTTP/JSON) vs gRPC (HTTP/2 + Protobuf)
// REST: human-readable, widely supported, browser-friendly
// gRPC: binary, 5-10x smaller payload, strongly typed contracts, streaming support
// Rule: gRPC for internal service-to-service, REST for public APIs and browser clients
// Asynchronous: Kafka message — decouples producer from consumer
// Producer publishes PaymentProcessed event
// Consumer (NotificationService) subscribes and sends email — independently
// Producer does NOT wait for consumer. Consumer can be down and catch up later.
// Synchronous call — tight coupling, cascading failure risk
// OrderService calls InventoryService via HTTP:
// OrderService → [HTTP] → InventoryService
// If InventoryService is slow: OrderService threads pile up → OrderService dies
// Asynchronous via Kafka — loose coupling, resilient
// OrderService → [Kafka: order.created] → InventoryService
// If InventoryService is down: events queue in Kafka, processed when it recovers
// OrderService never blocked

Interview Questions — Topic 1

Q2: What is the database-per-service pattern and why is it mandatory in microservices? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

Each microservice owns its own database — no other service can access it directly. Mandatory because: shared databases create coupling at the schema level (one team's change breaks another team's queries), prevent independent scaling, and cause lock contention. The consequence: no joins across services, no distributed ACID transactions. Cross-service data operations must use events and eventual consistency. This is the root cause of Saga, CQRS, and the Outbox Pattern.

Q3: When would you NOT use microservices? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Small teams (fewer than 2 pizza teams per service), early-stage products where requirements change rapidly (microservices make refactoring expensive), simple domains with no independent scaling needs, when the team lacks operational maturity (K8s, distributed tracing, service mesh are complex). Martin Fowler's rule: start with a modular monolith, extract microservices when you feel the pain that justifies the complexity. Premature microservices are one of the most expensive architectural mistakes.

Q4: What is the difference between synchronous and asynchronous inter-service communication? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Synchronous (REST, gRPC): caller waits for response. Simple, immediate consistency, easy debugging. Problems: tight temporal coupling (both services must be available simultaneously), cascading failure risk (slow downstream blocks upstream threads). Asynchronous (Kafka, RabbitMQ): producer publishes event, returns immediately. Consumer processes independently. Benefits: temporal decoupling, resilience to downstream failures, natural backpressure. Cost: eventual consistency, harder debugging (correlation IDs required), at-least-once delivery semantics require idempotent consumers.

🟢 Topic 2: Resilience Patterns — Circuit Breaker, Retry, Bulkhead, Rate Limiter

Checklist

  • Retry pattern: exponential backoff, jitter, max attempts — when NOT to retry
  • Bulkhead: thread pool isolation — prevent one slow service from consuming all threads
  • Rate Limiter: token bucket vs leaky bucket algorithm — RateLimiter in Resilience4j
  • Timeout: the most important resilience pattern — always set timeouts

// Circuit Breaker: 3 states
// CLOSED: normal operation — requests flow through, failures counted
// OPEN: failure threshold exceeded — all requests fail immediately (no waiting)
//       returns fallback without calling the actual service
// HALF-OPEN: after wait duration — allows limited requests through to test recovery
//            if they succeed: transitions back to CLOSED
//            if they fail: back to OPEN
// State transitions:
// CLOSED → OPEN:      when failure rate >= threshold (e.g., 50% of last 10 calls fail)
// OPEN → HALF-OPEN:   after waitDurationInOpenState (e.g., 60 seconds)
// HALF-OPEN → CLOSED: when permittedNumberOfCallsInHalfOpenState succeed
// HALF-OPEN → OPEN:   when any call in half-open fails
// Resilience4j Spring Boot integration
// pom.xml: spring-boot-starter-aop + resilience4j-spring-boot3
// application.yml configuration
// resilience4j:
//   circuitbreaker:
//     instances:
//       paymentGateway:
//         slidingWindowSize: 10           # evaluate last 10 calls
//         failureRateThreshold: 50        # open if 50%+ fail
//         waitDurationInOpenState: 60s    # wait 60s before half-open
//         permittedNumberOfCallsInHalfOpenState: 3
//         slowCallDurationThreshold: 3s   # calls > 3s count as slow
//         slowCallRateThreshold: 80       # open if 80%+ are slow
@Service
public class PaymentGatewayClient {
    @CircuitBreaker(name = "paymentGateway", fallbackMethod = "paymentFallback")
    @Retry(name = "paymentGateway")
    @Bulkhead(name = "paymentGateway")
    public PaymentResult processPayment(PaymentRequest req) {
        // actual call to external payment gateway
        return externalGateway.process(req);
    }
    // Fallback — called when circuit is OPEN or all retries exhausted
    // Must have same signature + Throwable parameter
    private PaymentResult paymentFallback(PaymentRequest req, Throwable t) {
        log.warn("Payment gateway unavailable, using fallback: {}", t.getMessage());
        return PaymentResult.pending(req.transactionId(), "Gateway temporarily unavailable");
    }
}
// Retry with exponential backoff
// resilience4j:
//   retry:
//     instances:
//       paymentGateway:
//         maxAttempts: 3
//         waitDuration: 500ms
//         enableExponentialBackoff: true
//         exponentialBackoffMultiplier: 2   # 500ms, 1000ms, 2000ms
//         retryExceptions:
//           - java.net.ConnectException
//           - java.util.concurrent.TimeoutException
//         ignoreExceptions:
//           - com.example.BusinessValidationException  # don't retry business errors
// Bulkhead — thread pool isolation
// resilience4j:
//   bulkhead:
//     instances:
//       paymentGateway:
//         maxConcurrentCalls: 10   # only 10 threads can call payment gateway
//         maxWaitDuration: 1s      # wait 1s for a slot before rejecting
// Effect: even if payment gateway hangs, only 10 threads blocked
// Other endpoints still have full thread pool available

Interview Questions — Topic 2

💡 CLOSED: normal — calls pass through, failures tracked. OPEN: failure threshold exceeded — calls fail immediately without reaching downstream, fallback returned. HALF-OPEN: after wait period — limited calls allowed to test recovery. CLOSED→OPEN: when failure rate (or slow call rate) exceeds threshold over sliding window. OPEN→HALF-OPEN: after waitDurationInOpenState. HALF-OPEN→CLOSED: if permitted test calls succeed. HALF-OPEN→OPEN: if test call fails. Purpose: prevent thread exhaustion from slow/failing downstream services, give failing services time to recover.

Q5: When should you NOT retry a failed request? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

Never retry: 4xx HTTP errors (400, 401, 403, 404, 422) — these are client errors, retrying will always fail. Business validation exceptions — same input will always fail the same validation. Non-idempotent operations without idempotency keys — retrying a POST that charges a credit card without an idempotency key causes double-charge. Retry only: transient infrastructure failures (ConnectException, SocketTimeoutException), 503 Service Unavailable, 429 Too Many Requests (with backoff). Always add jitter to prevent retry storms (multiple services retrying simultaneously after a failure).

The Saga Pattern — Distributed Consistency Without 2PC

The problem Saga solves — the hotel booking scenario:

Book flight + Book hotel + Charge credit card — all must succeed together or all must undo. With a monolith and one DB: wrap in one ACID transaction, done. With microservices (FlightService, HotelService, PaymentService each with own DB): cannot use a single transaction. 2PC requires all services to lock resources while coordinator waits for all to vote — if coordinator crashes, all services wait forever with resources locked. Saga instead: each service does its local transaction, emits an event, next service reacts. If step N fails, execute compensating transactions from step N-1 back to step 1.

// ── CHOREOGRAPHY SAGA — events drive the flow, no central coordinator ──
// Each service listens for events and publishes events
// Step 1: OrderService creates order
@KafkaListener(topics = "order.create.requested")
public void handleOrderRequest(OrderCreateEvent event) {
    Order order = orderRepo.save(new Order(event, OrderStatus.PENDING));
    kafkaTemplate.send("order.created", new OrderCreatedEvent(order.getId()));
}
// Step 2: InventoryService reserves stock
@KafkaListener(topics = "order.created")
public void handleOrderCreated(OrderCreatedEvent event) {
    try {
        inventoryService.reserve(event.getOrderId(), event.getItems());
        kafkaTemplate.send("inventory.reserved", new InventoryReservedEvent(event.getOrderId()));
    } catch (InsufficientStockException e) {
        // Publish failure event — triggers compensating transactions upstream
        kafkaTemplate.send("inventory.reservation.failed",
            new InventoryFailedEvent(event.getOrderId(), e.getMessage()));
    }
}
// Step 3: PaymentService charges card
@KafkaListener(topics = "inventory.reserved")
public void handleInventoryReserved(InventoryReservedEvent event) {
    try {
        paymentService.charge(event.getOrderId());
        kafkaTemplate.send("payment.processed", new PaymentProcessedEvent(event.getOrderId()));
    } catch (PaymentFailedException e) {
        kafkaTemplate.send("payment.failed", new PaymentFailedEvent(event.getOrderId()));
    }
}
// Compensating transaction — OrderService listens for failures
@KafkaListener(topics = "payment.failed")
public void handlePaymentFailed(PaymentFailedEvent event) {
    orderRepo.updateStatus(event.getOrderId(), OrderStatus.CANCELLED); // compensate
    kafkaTemplate.send("inventory.release", new ReleaseInventoryEvent(event.getOrderId()));
}
// ── ORCHESTRATION SAGA — central orchestrator drives the flow ──
// One class coordinates all steps, knows the full workflow
@Component
public class OrderSagaOrchestrator {
    public void startOrderSaga(OrderCreateRequest req) {
        SagaState state = sagaRepo.save(new SagaState(req, SagaStep.CREATE_ORDER));
        commandBus.send(new CreateOrderCommand(state.getId(), req));
    }
    @SagaEventHandler
    public void onOrderCreated(OrderCreatedEvent event) {
        sagaRepo.updateStep(event.getSagaId(), SagaStep.RESERVE_INVENTORY);
        commandBus.send(new ReserveInventoryCommand(event.getSagaId(), event.getOrderId()));
    }
    @SagaEventHandler
    public void onInventoryReserved(InventoryReservedEvent event) {
        sagaRepo.updateStep(event.getSagaId(), SagaStep.PROCESS_PAYMENT);
        commandBus.send(new ProcessPaymentCommand(event.getSagaId(), event.getOrderId()));
    }
    @SagaEventHandler
    public void onPaymentFailed(PaymentFailedEvent event) {
        // Orchestrator knows exactly what to compensate
        commandBus.send(new ReleaseInventoryCommand(event.getOrderId()));
        commandBus.send(new CancelOrderCommand(event.getOrderId()));
        sagaRepo.updateStep(event.getSagaId(), SagaStep.FAILED);
    }
}

Choreography vs Orchestration — The Trade-offs

Aspect

Choreography vs Orchestration

Structure

Choreography: services react to events — decentralized. Orchestration: one class/service drives the flow — centralized.

Coupling

Choreography: services only know about events — loosely coupled. Orchestration: orchestrator knows all steps — tightly coupled to workflow.

Visibility

Choreography: flow is implicit — hard to see the whole picture. Orchestration: flow is explicit in one place — easy to understand.

Error handling

Choreography: complex — compensations trigger chains of events. Orchestration: centralized — orchestrator knows exactly what to compensate.

Testing

Choreography: test each service independently. Orchestration: need to test the orchestrator's state machine.

When to use

Choreography: simple linear flows, small team, pure event-driven. Orchestration: complex multi-step workflows, many compensating paths, when you need full visibility.

The Outbox Pattern — Guarantee Event Publication

The dual-write problem: saving to DB and publishing to Kafka are two separate operations — one can fail:

Without Outbox: service saves order to DB (commit), then publishes Kafka event (fails) — order exists in DB but downstream services never know about it. Data inconsistency. With Outbox: in the SAME local transaction, service saves order AND an OutboxEvent record to the same DB. A separate Outbox processor polls the outbox table and publishes events to Kafka, deleting them on success. Since both writes are in one ACID transaction, they either both commit or both rollback — atomicity guaranteed. Debezium (CDC) is the production alternative: capture DB changes and stream to Kafka automatically.

// Outbox Pattern — atomic event publishing with DB transaction
@Transactional
public Order createOrder(OrderRequest req) {
    Order order = orderRepo.save(new Order(req));
    // In SAME transaction — both commit or both rollback
    OutboxEvent event = new OutboxEvent(
        "order.created",
        objectMapper.writeValueAsString(new OrderCreatedEvent(order.getId())),
        OutboxStatus.PENDING
    );
    outboxRepo.save(event); // saved to DB in same TX as order
    return order;
    // TX commits: both Order and OutboxEvent are persisted
}
// Outbox Processor — separate thread, polls and publishes
@Scheduled(fixedDelay = 500) // every 500ms
@Transactional
public void processOutbox() {
    List<OutboxEvent> pending = outboxRepo.findByStatus(OutboxStatus.PENDING);
    for (OutboxEvent event : pending) {
        kafkaTemplate.send(event.getTopic(), event.getPayload());
        event.setStatus(OutboxStatus.PUBLISHED);
        outboxRepo.save(event);
    }
}
// Idempotent consumers required — outbox may publish same event twice on crash/restart

Interview Questions — Topic 3

Q6: What is the Saga pattern and when do you need it? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

Saga manages distributed transactions across microservices without 2PC. Each step does a local transaction and publishes an event. If a step fails, compensating transactions undo previous steps in reverse order. Needed when: a business operation spans multiple microservices each with their own database (database-per-service pattern makes single ACID transaction impossible). Two variants: Choreography (event-driven, decentralized) and Orchestration (central coordinator). Your JP Morgan multi-service payment flow is a perfect real-world example.

Q7: What is a compensating transaction and what is its key limitation? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

A compensating transaction semantically undoes a previously committed local transaction. Example: if payment fails, the compensating transaction for inventory reservation is 'release reserved stock'. Key limitation: some operations cannot be truly compensated after they are committed. Sending an email (cannot unsend), external API calls (already executed), regulatory audit logs (must remain). These are called 'pivot transactions' — design the saga so these happen LAST. For truly non-compensatable actions: design workflows to avoid or accept the inconsistency with explicit reconciliation.

Q8: What is the Outbox Pattern and why do you need it with Sagas? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

Without Outbox: saving to DB and publishing to Kafka are separate operations — DB save can succeed and Kafka publish can fail, creating inconsistency. The Outbox Pattern: within the same local DB transaction, write the business entity AND an OutboxEvent record. A separate processor publishes the event to Kafka and marks it published. Atomicity guaranteed because both writes are in one ACID transaction. Consumers must be idempotent because the outbox processor may publish duplicates after a crash. Production alternative: Debezium CDC — captures DB WAL changes and streams to Kafka.

CQRS — Command Query Responsibility Segregation

CQRS exists because reads and writes have fundamentally different characteristics:

Writes: typically affect one aggregate, require strong consistency, low volume. Reads: often need data from multiple aggregates, can tolerate eventual consistency, very high volume. In a traditional architecture, the same model serves both — the read queries are complex joins pulling from a write-optimized schema. CQRS separates them: write side uses a normalized write-optimized model (Commands). Read side uses a denormalized read-optimized model (Queries) — could be a different database entirely. Write events update the read model asynchronously. Reads are fast because the read model is pre-joined and pre-computed.

// CQRS — separate Command and Query models
// COMMAND SIDE — write operations, domain model, business rules
@Service
public class OrderCommandService {
    @Transactional
    public OrderId createOrder(CreateOrderCommand cmd) {
        // Validate, apply business rules
        Order order = new Order(cmd);
        orderWriteRepo.save(order); // normalized write model
        eventBus.publish(new OrderCreatedEvent(order));
        return order.getId();
    }
}
// QUERY SIDE — read operations, denormalized view, optimized for reads
@Service
public class OrderQueryService {
    // Read model: denormalized table with order + customer + items pre-joined
    // Could be: same DB different table, Redis cache, Elasticsearch, read replica
    public OrderDetailView getOrderDetail(OrderId id) {
        return orderReadRepo.findDetailById(id); // single fast query, no joins needed
    }
    public Page<OrderSummaryView> getOrdersByCustomer(CustomerId customerId, Pageable p) {
        return orderReadRepo.findSummariesByCustomer(customerId, p);
    }
}
// Read model updater — keeps read model in sync with write model via events
@EventHandler
public class OrderReadModelUpdater {
    @KafkaListener(topics = "order.created")
    public void onOrderCreated(OrderCreatedEvent event) {
        // Denormalize: join order + customer + items data and persist to read model
        OrderDetailView view = buildDenormalizedView(event);
        orderReadRepo.save(view); // read model updated asynchronously
    }
}
// EVENT SOURCING — store events, not current state
// Instead of: UPDATE accounts SET balance = 500 WHERE id = 1
// Store:  AccountDebited { accountId:1, amount:500, timestamp:... }
//         AccountCredited { accountId:1, amount:200, timestamp:... }
// Event store — append-only
public class AccountEventStore {
    public void append(String aggregateId, DomainEvent event) {
        // INSERT INTO events (aggregate_id, event_type, payload, version)
        eventRepo.save(new StoredEvent(aggregateId, event));
    }
    public List<DomainEvent> getEvents(String aggregateId) {
        return eventRepo.findByAggregateIdOrderByVersion(aggregateId);
    }
}
// Reconstruct current state by replaying events
public Account loadAccount(String accountId) {
    List<DomainEvent> events = eventStore.getEvents(accountId);
    Account account = new Account(); // empty state
    events.forEach(account::apply); // replay each event to rebuild state
    return account; // current state derived from event history
}
// Snapshot — optimization: don't replay 1M events every time
// Periodically save current state as snapshot
// On load: restore from latest snapshot + replay only events AFTER snapshot
// Benefits of Event Sourcing:
// Complete audit log for free — every state change is recorded
// Time travel — reconstruct state at any point in history
// Event replay — rebuild read models from scratch, fix bugs retroactively
// Decoupled consumers — new services can replay entire history

Interview Questions — Topic 4

Q9: What is CQRS and why would you use it? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

Command Query Responsibility Segregation: separate the write model (Commands) from the read model (Queries). Use when: read and write workloads scale very differently (high read, lower write — e.g., product catalog), read queries need complex joins that hurt write-optimized schema, need to optimize read performance with denormalized projections or different storage (Redis, Elasticsearch for reads, PostgreSQL for writes). Costs: eventual consistency between write and read models, added complexity, separate update path for read model. Not for: simple CRUD apps where read/write patterns are similar.

Q10: What is Event Sourcing and how does it differ from traditional state-based storage? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

Traditional: store current state — UPDATE overwrites previous state, history lost. Event Sourcing: store sequence of events that led to current state. State derived by replaying events. Benefits: complete audit trail (critical for financial systems), time-travel queries (state at any past moment), event replay to build new projections, natural fit with event-driven systems. Costs: event replay performance (mitigated by snapshots), eventual consistency of projections, schema evolution of past events is complex, operational complexity. Ideal for: financial transactions (your VISA work), audit-critical systems, collaborative editing.

API Gateway — The Front Door of Your Microservices

The API Gateway pattern solves cross-cutting concerns at the edge:

// Spring Cloud Gateway configuration
// application.yml
// spring:
//   cloud:
//     gateway:
//       routes:
//         - id: payment-service
//           uri: lb://PAYMENT-SERVICE   # lb:// = load balanced via service discovery
//           predicates:
//             - Path=/api/payments/**
//           filters:
//             - StripPrefix=1           # remove /api prefix before routing
//             - name: CircuitBreaker
//               args:
//                 name: paymentCB
//                 fallbackUri: forward:/fallback/payment
//             - name: RequestRateLimiter
//               args:
//                 redis-rate-limiter.replenishRate: 100
//                 redis-rate-limiter.burstCapacity: 200
// Global JWT filter — validates token before ANY route
@Component
public class JwtGatewayFilter implements GlobalFilter, Ordered {
    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        String token = extractToken(exchange.getRequest());
        if (!jwtService.isValid(token)) {
            exchange.getResponse().setStatusCode(HttpStatus.UNAUTHORIZED);
            return exchange.getResponse().setComplete();
        }
        // Propagate user context to downstream services via headers
        ServerHttpRequest mutatedReq = exchange.getRequest().mutate()
            .header("X-User-Id", jwtService.extractUserId(token))
            .header("X-User-Roles", jwtService.extractRoles(token))
            .build();
        return chain.filter(exchange.mutate().request(mutatedReq).build());
    }
}
// Distributed Tracing — Micrometer Tracing (Spring Boot 3)
// Every request gets a trace ID. Each service hop adds a span.
// Trace ID propagates via HTTP headers: traceparent / b3 / X-B3-TraceId
// Spring Boot auto-instruments:
// - Incoming HTTP requests (new span per request)
// - @Async, @Scheduled, @NewSpan
// - Kafka consumers/producers
// - RestTemplate, WebClient outgoing calls
// MDC for correlation in logs
@RestController
public class PaymentController {
    public ResponseEntity<PaymentResponse> createPayment(@RequestBody PaymentRequest req) {
        // Micrometer auto-puts traceId into MDC
        log.info("Processing payment {}", req.transactionId());
        // Log output: [traceId=abc123 spanId=def456] Processing payment TX-001
        return ResponseEntity.ok(paymentService.process(req));
    }
}

Interview Questions — Topic 5

Q11: What does an API Gateway do and what are the cross-cutting concerns it handles? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

API Gateway is the single entry point for all client requests. Handles: Authentication/Authorization (validate JWT before routing — downstream services trust the gateway), Rate Limiting (per client, per endpoint), SSL Termination (HTTPS between client and gateway, HTTP between gateway and services — simpler cert management), Request Routing (path-based, header-based), Load Balancing, Circuit Breaking at edge, Request/Response transformation, Logging/Tracing correlation ID injection. Frees downstream services from implementing these concerns individually. Common implementations: Spring Cloud Gateway, AWS API Gateway, Kong, NGINX.

Q12: What is distributed tracing and how does trace ID propagation work? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Distributed tracing tracks a request as it flows across multiple services. Each request gets a unique trace ID. Each service call within that request creates a span (sub-unit of work with timing). Trace ID propagates via HTTP headers (W3C traceparent, or B3 headers for Zipkin). Micrometer Tracing (Spring Boot 3) auto-instruments HTTP clients, incoming requests, Kafka. Spans reported to Zipkin/Jaeger which assembles them into a timeline showing exactly which service took how long. Critical for diagnosing latency in multi-service flows — your JP Morgan multi-region architecture would have used this.

Multi-Stage Dockerfile for Spring Boot — Production Best Practice

Single-stage builds include the JDK in your final image — that is 300+ MB of tools you do not need at runtime:

A single-stage Spring Boot image using eclipse-temurin:17-jdk is typically 400-600 MB. A multi-stage build: stage 1 uses the full JDK to compile and build the JAR, stage 2 copies only the JAR into a JRE-only (or distroless) base image. Result: 180-250 MB final image with no compiler, no build tools, smaller attack surface. Spring Boot layered jars go further: split the JAR into layers (dependencies, spring-boot-loader, application classes) that Docker caches separately — rebuilds only changed layers.

# ─── Multi-Stage Dockerfile for Spring Boot ───
# Stage 1: BUILD — use full JDK to build the application
FROM eclipse-temurin:21-jdk-alpine AS builder
WORKDIR /app
# Copy Maven wrapper and pom.xml first — Docker caches this layer
# Only invalidated when pom.xml changes (dependency changes)
COPY mvnw pom.xml ./
COPY .mvn .mvn
RUN ./mvnw dependency:go-offline -q  # pre-download all dependencies
# Copy source and build — this layer invalidated when source changes
COPY src ./src
RUN ./mvnw package -DskipTests -q
# Stage 2: Extract layered jar (Spring Boot 2.3+ layered jar support)
FROM eclipse-temurin:21-jdk-alpine AS extractor
WORKDIR /app
COPY --from=builder /app/target/*.jar app.jar
RUN java -Djarmode=layertools -jar app.jar extract
# Stage 3: RUNTIME — use JRE only, copy layers from extractor
FROM eclipse-temurin:21-jre-alpine
WORKDIR /app
# Create non-root user — security best practice
RUN addgroup -S appgroup && adduser -S appuser -G appgroup
USER appuser
# Copy layers in order of change frequency (least → most)
# Docker caches each COPY layer — only changed layers rebuild
COPY --from=extractor /app/dependencies/ ./
COPY --from=extractor /app/spring-boot-loader/ ./
COPY --from=extractor /app/snapshot-dependencies/ ./
COPY --from=extractor /app/application/ ./  # changes every build
EXPOSE 8080
ENTRYPOINT ["java", "org.springframework.boot.loader.JarLauncher"]
# docker-compose.yml — local development with all dependencies
version: '3.9'
services:
  app:
    build: .
    ports:

    depends_on:
      postgres:

      kafka:
        condition: service_started
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8080/actuator/health"]
      interval: 10s

      POSTGRES_DB: payments

Interview Questions — Topic 6

Q13: What is a multi-stage Docker build and why is it important for production? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Multi-stage build uses multiple FROM instructions in one Dockerfile. Each stage can be based on different images. The final stage copies only the artifacts needed from earlier stages — build tools, compiler, source code are left behind. For Spring Boot: Stage 1 (JDK): compile, run Maven, produce JAR. Stage 2 (JRE or distroless): copy JAR only. Result: final image 180-250 MB instead of 400-600 MB. Benefits: smaller attack surface (no compiler in production image), faster pulls and pushes, no secrets or build credentials in final image. Also: layered JARs let Docker cache dependency layers separately from application code.

Q14: What is the difference between CMD and ENTRYPOINT in a Dockerfile? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

ENTRYPOINT: defines the command that always runs — cannot be overridden by docker run arguments (only with --entrypoint flag). CMD: provides default arguments — can be overridden by arguments passed to docker run. Best practice: ENTRYPOINT for the executable (java), CMD for default arguments that users might want to override. For Spring Boot: ENTRYPOINT ["java"] CMD ["-jar", "app.jar"] allows running docker run image -jar other.jar to override. Exec form (JSON array) preferred over shell form — exec form runs the process as PID 1, receives Unix signals correctly (important for graceful shutdown).

# Deployment — manages ReplicaSet which manages Pods
apiVersion: apps/v1
kind: Deployment
metadata:
  name: payment-service
spec:
  replicas: 3
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1        # create 1 extra pod before removing old ones
      maxUnavailable: 0  # never reduce capacity below desired (zero downtime)
  selector:
    matchLabels:
      app: payment-service
  template:
    metadata:
      labels:
        app: payment-service
    spec:
      containers:
      - name: payment-service
        image: myregistry/payment-service:1.2.3
        ports:
        - containerPort: 8080
        envFrom:
        - configMapRef:
            name: payment-config    # inject all ConfigMap values as env vars
        - secretRef:
            name: payment-secrets   # inject secrets (DB password, JWT key)
        resources:
          requests:
            memory: "256Mi"
            cpu: "250m"
          limits:
            memory: "512Mi"
            cpu: "500m"
        # Startup probe — give app time to start before liveness kicks in
        startupProbe:
          httpGet:
            path: /actuator/health
            port: 8080
          failureThreshold: 30   # allow 30 * 10s = 5 minutes to start
          periodSeconds: 10
        # Liveness probe — is the app alive? Failing = restart the pod
        livenessProbe:
          httpGet:
            path: /actuator/health/liveness
            port: 8080
          initialDelaySeconds: 0  # startup probe handles the delay
          periodSeconds: 10
          failureThreshold: 3     # 3 consecutive failures = restart
        # Readiness probe — is the app ready to receive traffic?
        # Failing = pod removed from Service endpoints (no traffic), NOT restarted
        readinessProbe:
          httpGet:
            path: /actuator/health/readiness
            port: 8080
          periodSeconds: 5
          failureThreshold: 3

Interview Questions — Topic 7

Q15: What is the difference between liveness and readiness probes in Kubernetes? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

Liveness probe: 'Is the application alive?' Failing = pod is stuck/deadlocked, cannot recover itself. K8s restarts the pod. Use for detecting: infinite loops, deadlocks, memory leaks causing the app to stop processing. Readiness probe: 'Is the application ready to receive traffic?' Failing = pod is temporarily unable to serve requests (warming up cache, circuit breaker open, DB connection lost). K8s removes the pod from Service endpoints (no traffic routed to it) but does NOT restart it. Critical distinction: readiness failure means 'do not send traffic', liveness failure means 'kill and restart'. Spring Boot Actuator provides /health/liveness and /health/readiness endpoints.

Q16: What is Horizontal Pod Autoscaler and how does it scale? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

HPA automatically adjusts the number of Pod replicas based on observed metrics. CPU utilization (most common): when average CPU across all pods exceeds target percentage, HPA scales up. Custom metrics: request rate, queue depth, response time via Prometheus metrics. HPA queries metrics every 15 seconds, calculates desired replicas = ceil(current * current_metric / target_metric). Limits: minReplicas and maxReplicas bound the range. Scale-up is aggressive (immediate), scale-down is gradual (default 5-minute cooldown). Important: resource requests must be set on containers — HPA calculates utilization as used/requested.

Q17: What is a rolling update strategy in Kubernetes and why is it important? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Rolling update gradually replaces old pods with new ones without downtime. Parameters: maxSurge (how many extra pods can be created above desired count during update) and maxUnavailable (how many pods can be unavailable). With maxSurge=1 and maxUnavailable=0: K8s creates 1 new pod, waits for it to pass readiness probe, then terminates 1 old pod — capacity never drops below desired replicas. If new pods fail readiness, rollout pauses and can be rolled back with kubectl rollout undo. This is why readiness probes are critical — they gate traffic to new pods until they are actually ready.
