The Payment System Schema — Used Throughout Day 9

All exercises today use this schema — inspired by your VISA payment work and Deloitte JP Morgan project:

This schema represents a simplified payment processing system. Every query you write today should be mentally connected to real payment business questions: 'Which accounts had the highest transaction volume last month?', 'Find all failed transactions that were retried successfully within 1 hour', 'Rank customers by spending per category'. These are the questions payment companies actually ask in interviews.

-- Payment System Schema (PostgreSQL / Aurora PostgreSQL)
CREATE TABLE customers (
    customer_id   BIGSERIAL PRIMARY KEY,
    name          VARCHAR(100) NOT NULL,
    email         VARCHAR(150) UNIQUE NOT NULL,
    tier          VARCHAR(20) DEFAULT 'STANDARD',  -- STANDARD, PREMIUM, VIP
    created_at    TIMESTAMP DEFAULT NOW()
);
CREATE TABLE accounts (
    account_id    BIGSERIAL PRIMARY KEY,
    customer_id   BIGINT NOT NULL REFERENCES customers(customer_id),
    account_type  VARCHAR(20) NOT NULL,  -- SAVINGS, CURRENT, CREDIT
    balance       DECIMAL(18,2) DEFAULT 0.00,
    currency      CHAR(3) DEFAULT 'INR',
    status        VARCHAR(20) DEFAULT 'ACTIVE',
    created_at    TIMESTAMP DEFAULT NOW()
);
CREATE TABLE transactions (
    transaction_id   BIGSERIAL PRIMARY KEY,
    account_id       BIGINT NOT NULL REFERENCES accounts(account_id),
    transaction_type VARCHAR(20) NOT NULL,  -- CREDIT, DEBIT
    amount           DECIMAL(18,2) NOT NULL,
    category         VARCHAR(50),  -- FOOD, TRAVEL, SHOPPING, UTILITIES, etc.
    status           VARCHAR(20) DEFAULT 'PENDING',  -- PENDING, SUCCESS, FAILED
    reference_id     VARCHAR(100),  -- external payment gateway reference
    created_at       TIMESTAMP DEFAULT NOW(),
    processed_at     TIMESTAMP
);
CREATE TABLE transaction_fees (
    fee_id         BIGSERIAL PRIMARY KEY,
    transaction_id BIGINT NOT NULL REFERENCES transactions(transaction_id),
    fee_type       VARCHAR(50),  -- PROCESSING, FOREX, LATE_PAYMENT
    fee_amount     DECIMAL(10,2) NOT NULL,
    created_at     TIMESTAMP DEFAULT NOW()
);
-- JOIN types — all on the payment schema
-- INNER JOIN: only rows with a match in BOTH tables
-- Find all transactions with customer name
SELECT c.name, t.transaction_id, t.amount, t.status
FROM transactions t
INNER JOIN accounts a  ON t.account_id = a.account_id
INNER JOIN customers c ON a.customer_id = c.customer_id
WHERE t.status = 'SUCCESS';
-- LEFT JOIN: all rows from LEFT table, NULL for non-matching RIGHT rows
-- Find ALL customers, including those with no transactions
SELECT c.name, COUNT(t.transaction_id) AS transaction_count
FROM customers c
LEFT JOIN accounts a  ON c.customer_id = a.customer_id
LEFT JOIN transactions t ON a.account_id = t.account_id
GROUP BY c.customer_id, c.name
ORDER BY transaction_count DESC;
-- SELF JOIN: join a table to itself
-- Find transactions where the amount is greater than
-- the customer's average transaction amount
SELECT t1.transaction_id, t1.amount, avg_amounts.avg_amount
FROM transactions t1
JOIN (
    SELECT account_id, AVG(amount) AS avg_amount
    FROM transactions WHERE status = 'SUCCESS'
    GROUP BY account_id
) avg_amounts ON t1.account_id = avg_amounts.account_id
WHERE t1.amount > avg_amounts.avg_amount;
-- CTE (Common Table Expression) — WITH clause
-- Monthly revenue per customer tier
WITH monthly_revenue AS (
    SELECT
        c.tier,
        DATE_TRUNC('month', t.created_at) AS month,
        SUM(t.amount)                     AS total_amount,
        COUNT(t.transaction_id)           AS tx_count
    FROM transactions t
    JOIN accounts a  ON t.account_id = a.account_id
    JOIN customers c ON a.customer_id = c.customer_id
    WHERE t.transaction_type = 'DEBIT' AND t.status = 'SUCCESS'
    GROUP BY c.tier, DATE_TRUNC('month', t.created_at)
),
tier_rank AS (
    SELECT
        tier, month, total_amount, tx_count,
        RANK() OVER (PARTITION BY month ORDER BY total_amount DESC) AS revenue_rank
    FROM monthly_revenue
)
SELECT * FROM tier_rank WHERE revenue_rank <= 3
ORDER BY month DESC, revenue_rank;

Interview Questions — Topic 1

Q2: What is the difference between WHERE and HAVING? [Easy] ▼
💡 Ideal Answer (Amazon/GS Standard)

WHERE filters rows BEFORE aggregation (GROUP BY). HAVING filters groups AFTER aggregation. You cannot use aggregate functions (SUM, COUNT, AVG) in WHERE — use HAVING. Execution order: FROM → JOIN → WHERE (filter rows) → GROUP BY → HAVING (filter groups) → SELECT → ORDER BY → LIMIT. Example: WHERE status = 'SUCCESS' filters individual rows. HAVING COUNT(*) > 10 filters only groups (accounts) that had more than 10 transactions. You can use both together: WHERE filters the rows fed into GROUP BY, HAVING filters the resulting groups.

Q3: What is a CTE and when would you use it over a subquery? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

CTE (Common Table Expression): named temporary result set defined with WITH clause, readable once per query. Benefits over subqueries: readability (complex logic split into named steps), reusability (reference the same CTE multiple times — a subquery would need to be repeated), recursive queries (impossible with regular subqueries), debugging (test each CTE independently). Use subquery when: simple, used once, inline expression. Use CTE when: result used multiple times, multi-step logic is clearer as named steps, or you need recursion. Performance: in PostgreSQL, each CTE reference is materialized once by default (can be optimized with NOT MATERIALIZED in Postgres 12+).

Q4: What is the difference between UNION and UNION ALL? [Easy] ▼
💡 Ideal Answer (Amazon/GS Standard)

UNION removes duplicates (performs a DISTINCT across both result sets — expensive, requires sort or hash). UNION ALL keeps all rows including duplicates (no deduplication — much faster). Use UNION ALL unless you specifically need to remove duplicates — it avoids the additional sort/hash operation. Common interview mistake: always using UNION when the data is already distinct, paying unnecessary performance cost. Example: combining transactions from two separate partitioned tables — if transaction_ids are unique by design, use UNION ALL.

  • DENSE_RANK() — ties get same rank, NO gaps

  • FIRST_VALUE(), LAST_VALUE(), NTH_VALUE() — frame-aware functions

  • NTILE(n) — divide rows into n equal buckets

Window Functions — The Most-Tested SQL Topic in Senior Interviews

Window functions perform calculations across a set of rows related to the current row — without collapsing them like GROUP BY does:

GROUP BY: reduces 100 transaction rows to 5 account rows. Window function: keeps all 100 rows but adds a column showing each row's rank/running total/previous value within its group. The OVER clause defines the window: PARTITION BY (how to group rows, like GROUP BY), ORDER BY (how to sort within each group), ROWS/RANGE BETWEEN (optional frame — which rows to include in calculation). Once you understand PARTITION BY + ORDER BY, all window functions follow the same pattern. The hardest part is the frame spec — master ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW (running total) and it covers 80% of use cases.

-- ── RANKING FUNCTIONS ──
-- ROW_NUMBER: unique sequential 1,2,3,4 — no ties
-- Use: pagination, deduplication (keep latest row per group)
SELECT
    transaction_id,
    account_id,
    amount,
    ROW_NUMBER() OVER (PARTITION BY account_id ORDER BY created_at DESC) AS rn
FROM transactions
WHERE status = 'SUCCESS';
-- rn=1 is the most recent transaction per account
-- RANK vs DENSE_RANK — when there are ties
-- Scores: 100, 90, 90, 80
-- RANK:       1,  2,  2,  4   (gap after tie — no rank 3)
-- DENSE_RANK: 1,  2,  2,  3   (no gap — next rank is 3)
-- Classic interview Q: Top 3 transactions per account by amount
WITH ranked AS (
    SELECT
        account_id, transaction_id, amount, created_at,
        ROW_NUMBER() OVER (PARTITION BY account_id ORDER BY amount DESC) AS rn
    FROM transactions
    WHERE status = 'SUCCESS'
)
SELECT account_id, transaction_id, amount
FROM ranked
WHERE rn <= 3;
-- Classic interview Q: 2nd highest spend per customer category
WITH category_spend AS (
    SELECT
        a.customer_id,
        t.category,
        SUM(t.amount) AS total_spend,
        DENSE_RANK() OVER (
            PARTITION BY a.customer_id
            ORDER BY SUM(t.amount) DESC
        ) AS spend_rank
    FROM transactions t
    JOIN accounts a ON t.account_id = a.account_id
    WHERE t.status = 'SUCCESS'
    GROUP BY a.customer_id, t.category
)
SELECT customer_id, category, total_spend
FROM category_spend
WHERE spend_rank = 2;
-- ── LAG and LEAD: access adjacent row values ──
-- LAG(column, offset, default): value from N rows BEFORE current row
-- LEAD(column, offset, default): value from N rows AFTER current row
-- Find transactions where amount > 2x the previous transaction for same account
-- (potential fraud signal)
WITH lagged AS (
    SELECT
        transaction_id,
        account_id,
        amount,
        created_at,
        LAG(amount, 1, 0) OVER (
            PARTITION BY account_id
            ORDER BY created_at
        ) AS prev_amount
    FROM transactions
    WHERE status = 'SUCCESS'
)
SELECT transaction_id, account_id, amount, prev_amount,
       ROUND(amount / NULLIF(prev_amount, 0), 2) AS amount_ratio
FROM lagged
WHERE prev_amount > 0 AND amount > prev_amount * 2
ORDER BY amount_ratio DESC;
-- ── RUNNING TOTAL with SUM() OVER ──
-- Running balance per account (chronological)
SELECT
    transaction_id,
    account_id,
    amount,
    transaction_type,
    created_at,
    SUM(
        CASE WHEN transaction_type = 'CREDIT' THEN amount
             ELSE -amount END
    ) OVER (
        PARTITION BY account_id
        ORDER BY created_at
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS running_balance
FROM transactions
WHERE status = 'SUCCESS'
ORDER BY account_id, created_at;
-- ── MOVING AVERAGE: last 7 days spend per account ──
SELECT
    account_id,
    DATE_TRUNC('day', created_at) AS day,
    SUM(amount) AS daily_spend,
    AVG(SUM(amount)) OVER (
        PARTITION BY account_id
        ORDER BY DATE_TRUNC('day', created_at)
        ROWS BETWEEN 6 PRECEDING AND CURRENT ROW  -- 7-day window
    ) AS moving_avg_7d
FROM transactions
WHERE transaction_type = 'DEBIT' AND status = 'SUCCESS'
GROUP BY account_id, DATE_TRUNC('day', created_at)
ORDER BY account_id, day;
-- ── NTILE: split customers into spend quartiles ──
WITH customer_spend AS (
    SELECT a.customer_id, SUM(t.amount) AS total_spend
    FROM transactions t JOIN accounts a ON t.account_id = a.account_id
    WHERE t.transaction_type = 'DEBIT' AND t.status = 'SUCCESS'
    GROUP BY a.customer_id
)
SELECT
    customer_id,
    total_spend,
    NTILE(4) OVER (ORDER BY total_spend DESC) AS quartile
    -- 1 = top 25% spenders, 4 = bottom 25%
FROM customer_spend
ORDER BY total_spend DESC;

Interview Questions — Topic 2

Q5: What is the difference between ROW_NUMBER(), RANK(), and DENSE_RANK()? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

All three assign numbers to rows within a partition ordered by a column. Difference is tie handling. ROW_NUMBER: always unique — ties get different numbers (arbitrary for tied rows). RANK: ties get the same number, next number skips (1,2,2,4 — no 3). DENSE_RANK: ties get the same number, no skipping (1,2,2,3). Use ROW_NUMBER for deduplication (keep exactly one row per group — arbitrary winner). Use RANK when gaps after ties are acceptable (sports leaderboard). Use DENSE_RANK when you need nth-highest without gaps (2nd highest salary per department — with ties, DENSE_RANK=2 means truly 2nd distinct value).

Q6: What is ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW and when do you use it? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

This is the window frame specification for a running total. UNBOUNDED PRECEDING: start of the partition. CURRENT ROW: the current row being processed. Together: include all rows from the start of the partition up to and including the current row. Use for: running balance (sum of all transactions up to now), cumulative user signups, progressive total. Alternative frames: ROWS BETWEEN 6 PRECEDING AND CURRENT ROW (7-row moving average), ROWS BETWEEN CURRENT ROW AND UNBOUNDED FOLLOWING (running total from current row to end). Without explicit frame: ORDER BY defaults to RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW — handles ties differently.

Q7: Write a query to find the transaction immediately before a failed transaction for each account. [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

Use LAG() to access the previous row value: SELECT transaction_id, account_id, amount, status, created_at, LAG(transaction_id) OVER (PARTITION BY account_id ORDER BY created_at) AS prev_transaction_id, LAG(amount) OVER (PARTITION BY account_id ORDER BY created_at) AS prev_amount, LAG(status) OVER (PARTITION BY account_id ORDER BY created_at) AS prev_status FROM transactions WHERE status = 'FAILED'. This gives each failed transaction's immediately preceding transaction details — useful for fraud investigation: was the previous transaction also failed? Was the amount unusual?

🟠 Topic 3: Query Optimization — EXPLAIN, Indexes, Covering Indexes, Index Rules

Checklist

  • EXPLAIN vs EXPLAIN ANALYZE — what each shows, how to read the output
  • B-Tree index: structure, equality and range queries, sort operations
  • Composite index: column order matters — left-prefix rule
  • Covering index: index-only scan — when the index contains all needed columns
  • Index selectivity: high vs low cardinality — when indexes hurt
  • Partial index: index on a subset of rows — powerful for status columns
  • Common index killers: functions on columns, implicit type casts, OR conditions

The Left-Prefix Rule — The Most Important Index Concept

A composite index can only be used from left to right — you cannot skip columns:

Index on (account_id, status, created_at). Can use: WHERE account_id = 1 (leftmost). WHERE account_id = 1 AND status = 'SUCCESS' (left two). WHERE account_id = 1 AND status = 'SUCCESS' AND created_at > '2024-01-01' (all three). Cannot use: WHERE status = 'SUCCESS' (skips account_id). WHERE created_at > '2024-01-01' (skips both). Can partially use: WHERE account_id = 1 AND created_at > '2024-01-01' — uses index for account_id equality, but cannot use created_at because status is skipped. Design rule: put equality columns first (highest cardinality first within equality columns), then range columns last.

-- EXPLAIN ANALYZE output interpretation (PostgreSQL)
EXPLAIN ANALYZE
SELECT t.transaction_id, t.amount, c.name
FROM transactions t
JOIN accounts a  ON t.account_id = a.account_id
JOIN customers c ON a.customer_id = c.customer_id
WHERE t.account_id = 12345
  AND t.status = 'SUCCESS'
  AND t.created_at BETWEEN '2024-01-01' AND '2024-12-31';
-- Output nodes to understand:
-- Seq Scan:        full table scan — NO index used — RED FLAG on large tables
-- Index Scan:      uses index to find rows, then fetches from heap
-- Index Only Scan: ALL needed columns in index — never touches heap — FASTEST
-- Bitmap Index Scan: multiple indexes combined — for OR conditions
-- Hash Join:       good for large unsorted sets
-- Nested Loop:     good when outer set is small, inner has index
-- rows=N (actual): actual rows processed — huge vs estimated = stale statistics
-- cost=N..M:       estimated cost (relative units, not ms)
-- ── INDEX DESIGN for the transactions table ──
-- Most common query pattern: by account + status + date range
-- Composite index: equality first (account_id, status), range last (created_at)
CREATE INDEX idx_tx_account_status_date
    ON transactions (account_id, status, created_at DESC);
-- Covering index: include amount and transaction_type so query never touches heap
CREATE INDEX idx_tx_account_status_covering
    ON transactions (account_id, status, created_at DESC)
    INCLUDE (amount, transaction_type);
-- With INCLUDE: Index Only Scan possible for:
-- SELECT amount, transaction_type FROM transactions
-- WHERE account_id = ? AND status = 'SUCCESS' AND created_at > ?
-- Partial index: only index SUCCESS transactions (skips PENDING/FAILED = 30% of rows)
-- Much smaller index, only used for success queries
CREATE INDEX idx_tx_success_account_date
    ON transactions (account_id, created_at DESC)
    WHERE status = 'SUCCESS';
-- Used by: WHERE account_id = ? AND status = 'SUCCESS' AND created_at > ?
-- NOT used by: WHERE account_id = ? AND status = 'FAILED'
-- ── INDEX KILLERS — these prevent index use ──
-- Function on indexed column → full table scan
-- BAD:  WHERE UPPER(email) = 'USER@EXAMPLE.COM'
-- GOOD: WHERE email = 'user@example.com'  (store lowercase)
-- OR:   CREATE INDEX idx_email_lower ON customers (LOWER(email));
-- Implicit type cast → full table scan
-- BAD:  WHERE account_id = '12345'  (account_id is BIGINT, '12345' is VARCHAR)
-- GOOD: WHERE account_id = 12345
-- Leading wildcard → full table scan
-- BAD:  WHERE reference_id LIKE '%VISA%'  (cannot use B-tree index)
-- GOOD: WHERE reference_id LIKE 'VISA%'   (can use B-tree index)
-- For full text search: use pg_trgm GIN index or full-text search

Interview Questions — Topic 3

Q8: What is a covering index and how does it eliminate heap fetches? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

A covering index contains all columns needed by a query — the query can be satisfied entirely from the index without touching the main table (heap). In PostgreSQL: CREATE INDEX ... INCLUDE (col1, col2) adds non-key columns to the index leaf pages. Result: Index Only Scan — fastest possible query execution. The INCLUDE columns are not used for filtering/sorting but are available for projection. Example: index on (account_id, status) INCLUDE (amount, created_at) — a query selecting amount and created_at WHERE account_id = ? AND status = 'SUCCESS' never reads the heap. Particularly valuable for frequently-run reporting queries on large tables.

Q9: Why does WHERE UPPER(email) = 'X' prevent index use, and how do you fix it? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

When a function is applied to an indexed column, PostgreSQL cannot use the B-tree index — the index stores raw values but the predicate needs transformed values. Fix options: 1) Functional/expression index: CREATE INDEX idx_email_lower ON customers (LOWER(email)) — then query: WHERE LOWER(email) = 'user@example.com'. 2) Application-level convention: always store emails in lowercase, remove UPPER/LOWER from queries. 3) Citext extension: case-insensitive text type with built-in index support. In production: option 2 (store normalized) is simplest, option 1 (expression index) if you can't change existing data. Same applies to type casts — WHERE account_id = '12345' (string '12345' cast to BIGINT) prevents index use.

Q10: What is index selectivity and when does an index hurt rather than help? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

Selectivity: the ratio of distinct values to total rows. High selectivity (user_id, transaction_id): index very effective — quickly narrows to a small number of rows. Low selectivity (status with 3 values like PENDING/SUCCESS/FAILED): index may be counter-productive. For a table of 10M rows where 70% are SUCCESS: a query WHERE status = 'SUCCESS' would use the index to find 7M row pointers — then do 7M random heap fetches. A sequential scan of the entire table is often FASTER. PostgreSQL query planner makes this decision based on table statistics. Partial index on status = 'FAILED' (5% of rows) is highly selective and useful.

🟣 Topic 4: Database Design — Normalization, Schema Patterns, Audit Tables

Checklist

  • 1NF, 2NF, 3NF — what each removes and why it matters
  • When to denormalize: OLTP vs OLAP trade-offs
  • Audit table pattern: created_by, updated_by, created_at, updated_at, soft delete
  • Polymorphic association anti-pattern — and the clean alternatives
  • Surrogate vs natural keys — UUID vs BIGSERIAL — the trade-offs
  • Table partitioning: range partitioning on created_at for large transaction tables

Normalization is about eliminating redundancy. Denormalization is about trading redundancy for read performance. The art is knowing when each applies:

3NF eliminates all transitive dependencies — every column depends only on the primary key. For OLTP (payment processing): normalize to 3NF — updates to customer name only happen in one place, no inconsistency. For OLAP (reporting, analytics): denormalize — pre-join data, store customer_name in the fact table to avoid joins at query time. The transactions table in your schema is already slightly denormalized — storing category directly on transactions (derived from merchant data) avoids a join for the most common query. This is an intentional design decision, not a mistake.

-- ── AUDIT TABLE PATTERN — every production table should have this ──
-- Option 1: Audit columns on the same table
CREATE TABLE transactions (
    transaction_id   BIGSERIAL PRIMARY KEY,
    -- ... business columns ...
    created_at       TIMESTAMP NOT NULL DEFAULT NOW(),
    updated_at       TIMESTAMP NOT NULL DEFAULT NOW(),
    created_by       BIGINT REFERENCES users(user_id),
    updated_by       BIGINT REFERENCES users(user_id),
    is_deleted       BOOLEAN NOT NULL DEFAULT FALSE,  -- soft delete
    deleted_at       TIMESTAMP,
    version          INTEGER NOT NULL DEFAULT 0  -- optimistic locking
);
-- Trigger to auto-update updated_at (PostgreSQL)
CREATE OR REPLACE FUNCTION update_updated_at()
RETURNS TRIGGER AS $$
BEGIN
    NEW.updated_at = NOW();
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;
CREATE TRIGGER trg_transactions_updated_at
    BEFORE UPDATE ON transactions
    FOR EACH ROW EXECUTE FUNCTION update_updated_at();
-- Soft delete: never DELETE rows — set is_deleted = TRUE
-- All queries must filter: WHERE is_deleted = FALSE
-- Partial index for performance:
CREATE INDEX idx_tx_active ON transactions (account_id, created_at DESC)
    WHERE is_deleted = FALSE;
-- Option 2: Separate audit/history table (full change history)
CREATE TABLE transaction_audit (
    audit_id        BIGSERIAL PRIMARY KEY,
    transaction_id  BIGINT NOT NULL,
    operation       CHAR(1) NOT NULL,  -- I=Insert, U=Update, D=Delete
    changed_at      TIMESTAMP NOT NULL DEFAULT NOW(),
    changed_by      BIGINT,
    old_values      JSONB,  -- previous state as JSON
    new_values      JSONB   -- new state as JSON
);
-- ── TABLE PARTITIONING — for the transactions table at scale ──
-- Range partition by created_at — each month is a separate physical partition
CREATE TABLE transactions_partitioned (
    transaction_id BIGSERIAL,
    account_id     BIGINT NOT NULL,
    amount         DECIMAL(18,2) NOT NULL,
    status         VARCHAR(20),
    created_at     TIMESTAMP NOT NULL
) PARTITION BY RANGE (created_at);
CREATE TABLE transactions_2024_01
    PARTITION OF transactions_partitioned
    FOR VALUES FROM ('2024-01-01') TO ('2024-02-01');
-- Partition pruning: queries with WHERE created_at BETWEEN '2024-01-01' AND '2024-01-31'
-- only scan the 2024_01 partition — all other partitions skipped
-- Critical for payment systems with billions of rows

Interview Questions — Topic 4

Q11: What is 3NF and when would you intentionally violate it? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

3NF (Third Normal Form): every non-key column depends ONLY on the primary key — no transitive dependencies (A→B→C where B is not a key). Example violation: storing customer_city and customer_country in transactions — they depend on customer_id (not transaction_id). 3NF fix: remove to customers table. Intentional violation (denormalization): when read performance justifies the redundancy. OLAP reporting queries joining 5 tables at 10M rows/table are slow — store pre-joined denormalized data in a reporting table. E-commerce: store product_name on order_items even though it's in products — because product names can change and you need the historical name at time of purchase. Always document WHY you denormalized.

Q12: What is the difference between a hard delete and a soft delete, and what are the trade-offs? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Hard delete: DELETE FROM transactions WHERE transaction_id = ?. Data is gone — cannot recover without backup. Soft delete: UPDATE transactions SET is_deleted = TRUE, deleted_at = NOW() WHERE transaction_id = ?. Row remains but is logically deleted. Soft delete benefits: auditability (required for financial/compliance systems), recovery, reference integrity (foreign keys from other tables don't break). Soft delete costs: every query must include WHERE is_deleted = FALSE — easy to forget, causes bugs, performance (index must include is_deleted). Mitigation: partial index on is_deleted = FALSE, database-level row security, application-level repository abstraction that always adds the filter.

🔴 Topic 5: Transaction Isolation Levels — Dirty Read, Phantom Read, MVCC

Checklist

  • Isolation problems: dirty read, non-repeatable read, phantom read — with examples
  • Four isolation levels: READ UNCOMMITTED, READ COMMITTED, REPEATABLE READ, SERIALIZABLE
  • PostgreSQL default: READ COMMITTED — what it prevents and what it doesn't
  • MVCC (Multi-Version Concurrency Control) — how PostgreSQL avoids most read locks
  • FOR UPDATE / SKIP LOCKED — pessimistic locking for queue-like patterns

Transaction Isolation — The Problems and Their Solutions

Isolation Level

Dirty Read | Non-Repeatable Read | Phantom Read | Performance

READ UNCOMMITTED

Possible | Possible | Possible | Highest — not used in PostgreSQL

READ COMMITTED (PG default)

Prevented | Possible | Possible | High — good for most OLTP

REPEATABLE READ

Prevented | Prevented | Prevented in PG* | Medium — uses snapshots

SERIALIZABLE

Prevented | Prevented | Prevented | Lowest — full serialization

* PostgreSQL REPEATABLE READ uses MVCC snapshots — also prevents phantom reads (better than SQL standard requires)

Dirty Read: reading uncommitted data from another transaction. Non-Repeatable Read: reading the same row twice, getting different values because another committed transaction updated it between your two reads. Phantom Read: running the same range query twice, getting different rows because another committed transaction inserted/deleted rows matching your WHERE clause. In your VISA payment system: READ COMMITTED is correct for most operations. REPEATABLE READ for: any business logic that reads data multiple times and must see consistent values (e.g., balance check + debit must see same balance). SERIALIZABLE for: transfer between two accounts where you need to prevent concurrent updates.

The Three Isolation Problems:

-- ── MVCC: PostgreSQL keeps multiple row versions ──
-- Writers do NOT block readers
-- Readers do NOT block writers
-- Each transaction sees a consistent snapshot of the database
-- Non-repeatable read scenario at READ COMMITTED (PostgreSQL default):
-- T1: BEGIN; SELECT balance FROM accounts WHERE account_id = 1; -- returns 1000
-- T2: BEGIN; UPDATE accounts SET balance = 500 WHERE account_id = 1; COMMIT;
-- T1: SELECT balance FROM accounts WHERE account_id = 1; -- returns 500 (changed!)
-- READ COMMITTED: each statement sees the latest committed data
-- Fix with REPEATABLE READ:
BEGIN TRANSACTION ISOLATION LEVEL REPEATABLE READ;
-- T1: SELECT balance → 1000 (snapshot taken at transaction start)
-- T2: commits balance = 500
-- T1: SELECT balance again → still 1000 (reads from same snapshot)
COMMIT;
-- ── FOR UPDATE: pessimistic locking ──
-- Prevent another transaction from modifying a row you are about to update
BEGIN;
-- Lock the account row — no other transaction can UPDATE this row until we COMMIT
SELECT balance FROM accounts WHERE account_id = 12345 FOR UPDATE;
-- Now safe to check balance and debit
UPDATE accounts SET balance = balance - 500 WHERE account_id = 12345;
INSERT INTO transactions (account_id, amount, transaction_type, status)
    VALUES (12345, 500, 'DEBIT', 'SUCCESS');
COMMIT;
-- ── SKIP LOCKED: job queue pattern ──
-- Multiple workers picking tasks from a queue table without conflicts
BEGIN;
SELECT * FROM payment_jobs
WHERE status = 'PENDING'
ORDER BY created_at
LIMIT 1
FOR UPDATE SKIP LOCKED;  -- skip rows locked by other workers, grab next available
-- process the job...
UPDATE payment_jobs SET status = 'PROCESSING' WHERE job_id = ?;
COMMIT;
-- Pattern used in: your JP Morgan batch processing, Outbox processors,
-- any polling-based job queue without Redis/Kafka

Interview Questions — Topic 5

Q13: What is a dirty read and which isolation level prevents it? [Easy] ▼
💡 Ideal Answer (Amazon/GS Standard)

Dirty read: Transaction A reads data written by Transaction B that has NOT yet committed. If B then rolls back: A has read data that never officially existed. Prevented by: READ COMMITTED and above. Example: T1 debits account (balance 1000→500, not committed), T2 reads balance as 500, T1 rolls back (balance restored to 1000), T2 makes a decision based on the phantom 500 balance — wrong. PostgreSQL default (READ COMMITTED) prevents dirty reads — you always read only committed data. READ UNCOMMITTED (allows dirty reads) is not even implemented in PostgreSQL — it behaves as READ COMMITTED.

Q14: What is MVCC and how does it allow readers and writers to not block each other? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

MVCC (Multi-Version Concurrency Control): PostgreSQL keeps multiple versions of each row. When a transaction reads a row: it sees the version that was committed at the transaction's snapshot time — not the current in-progress version. When a transaction writes a row: it creates a new version, marks the old version as expired. Readers see old (committed) versions — no lock needed, no waiting for writers. Writers see and lock only the current committed version — no contention with readers. This is why PostgreSQL can have high read throughput even during heavy writes: readers and writers operate on different row versions simultaneously. Trade-off: VACUUM needed to clean up expired row versions — dead tuples accumulate.

HikariCP Configuration — Every Option Matters

Pool exhaustion is not always caused by too-small maximumPoolSize — it is often caused by long-running transactions that hold connections too long:

Scenario: maximumPoolSize=10. A slow query (missing index, locking contention) takes 30 seconds. 10 concurrent requests each acquire a connection. All 10 slow queries are running. 11th request: waits connectionTimeout (default 30s) for a connection — throws SQLTimeoutException. Meanwhile, all 10 threads are blocked. Your Spring Boot service is now effectively dead. The fix is NOT always 'increase pool size' — it is fixing the slow query, adding a statement timeout, and monitoring connection acquisition time. In your JP Morgan work: this scenario likely happened with the N+1 queries before you fixed them with JOIN FETCH.

# HikariCP configuration in Spring Boot application.yml
spring:
  datasource:
    url: jdbc:postgresql://aurora-cluster.rds.amazonaws.com:5432/payments
    username: ${DB_USERNAME}
    password: ${DB_PASSWORD}
    driver-class-name: org.postgresql.Driver
    hikari:
      # Pool size — use formula: (core_count * 2) + effective_spindle_count
      # For m6i.xlarge (4 vCPU) with SSD Aurora: (4*2)+1 = 9, round to 10
      maximum-pool-size: 10
      minimum-idle: 3            # maintain 3 idle connections for fast response
      # Timeout settings
      connection-timeout: 3000   # ms: max wait for a connection from pool (3s)
                                 # if exceeded: SQLTimeoutException
      idle-timeout: 600000       # ms: close idle connections after 10 min
      max-lifetime: 1800000      # ms: force-close connections after 30 min
                                 # prevents connections from going stale
      keepalive-time: 60000      # ms: send keepalive ping every 60s
      # Connection validation
      connection-test-query: SELECT 1  # validate connections before giving to pool
                                       # (not needed for PostgreSQL JDBC4+)
      # Pool name for metrics/logging
      pool-name: PaymentServicePool
      # Statement timeout at DB level (PostgreSQL-specific)
      # Any query running longer than 10s is cancelled by the DB
      data-source-properties:
        socketTimeout: 30           # TCP socket timeout (seconds)
        connectTimeout: 5           # TCP connect timeout (seconds)
        options: "-c statement_timeout=10000"  # PostgreSQL: cancel query after 10s

@Transactional  // acquires connection C1
public void processPayment(PaymentRequest req) {

                pending,
                pool.getActiveConnections(),
                pool.getIdleConnections(),
                pool.getTotalConnections());

Interview Questions — Topic 6

Q15: What is HikariCP and how do you size the connection pool correctly? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

HikariCP is Spring Boot's default JDBC connection pool — maintains a pool of open DB connections for reuse. Opening a new connection takes 100-200ms (TCP handshake, auth, SSL). Pool allows reuse of existing connections. Sizing formula (from HikariCP docs): connections = (core_count * 2) + effective_spindle_count. For m6i.xlarge (4 vCPU) with Aurora SSD: (4*2)+1 = 9, round to 10. Counter-intuitively: FEWER connections often performs better than more — more connections means more context switching at the DB server. Start with formula, measure, adjust. maximumPoolSize is a ceiling — not a target. Set minimumIdle to keep a few warm for immediate response.

Q16: Describe a connection pool exhaustion scenario and how you would debug it. [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

Scenario: All connections in pool are held by slow queries. New requests wait connectionTimeout milliseconds then fail with SQLTimeoutException or HikariPool-N - Connection is not available, request timed out. Debug steps: 1) Check hikaricp.connections.pending metric — threads waiting for connection. 2) Check hikaricp.connections.acquire p99 — time waiting for connection. 3) Find slow queries: pg_stat_activity (PostgreSQL) shows active queries with duration. 4) Check for missing indexes via EXPLAIN ANALYZE on slow queries. 5) Check for locks: pg_locks + pg_blocking_pids(). Root causes: missing index causing full table scans, N+1 queries, long-running transactions, REQUIRES_NEW nested transaction pool deadlock. Fix: address root cause, add statement_timeout as safety net.

Recursive CTE and UPSERT — Two Advanced Patterns You Must Know

-- ── RECURSIVE CTE: hierarchical data ──
-- Example: account hierarchy (corporate accounts with sub-accounts)
-- Or: find all downstream accounts in a referral chain
WITH RECURSIVE account_hierarchy AS (
    -- Anchor member: start with the root account
    SELECT account_id, customer_id, parent_account_id, account_type, 0 AS depth
    FROM accounts
    WHERE account_id = 1000  -- start from root
    UNION ALL
    -- Recursive member: join to find children
    SELECT a.account_id, a.customer_id, a.parent_account_id, a.account_type,
           ah.depth + 1
    FROM accounts a
    INNER JOIN account_hierarchy ah ON a.parent_account_id = ah.account_id
    WHERE ah.depth < 10  -- safety: prevent infinite recursion
)
SELECT account_id, customer_id, depth, account_type
FROM account_hierarchy
ORDER BY depth, account_id;
-- ── UPSERT: INSERT or UPDATE atomically ──
-- Critical for: deduplication tables, idempotent operations, caches
-- Upsert: insert new customer, update tier if already exists
INSERT INTO customers (email, name, tier)
VALUES ('user@example.com', 'Zuber', 'PREMIUM')
ON CONFLICT (email)  -- conflict on unique constraint
DO UPDATE SET
    tier = EXCLUDED.tier,          -- EXCLUDED = the row that was rejected
    updated_at = NOW()
WHERE customers.tier != EXCLUDED.tier;  -- only update if different
-- Upsert for idempotent event processing (your Outbox/Saga deduplication)
INSERT INTO processed_transactions (transaction_id, processed_at, result)
VALUES ($1, NOW(), $2)
ON CONFLICT (transaction_id)
DO NOTHING;  -- if already processed: silently skip
-- Returns 0 rows affected on conflict — your code can check this
-- ── RETURNING clause — get INSERT result ──
INSERT INTO transactions (account_id, amount, transaction_type, status)
VALUES (12345, 500.00, 'DEBIT', 'PENDING')
RETURNING transaction_id, created_at;
-- Returns the generated transaction_id — no separate SELECT needed
-- Spring Data JPA: use @GeneratedValue with IDENTITY strategy for same effect
-- ── BULK INSERT — performance comparison ──
-- Single-row INSERT: 1000 transactions = 1000 round trips to DB
-- Multi-value INSERT: 1 round trip, significant performance improvement
INSERT INTO transactions (account_id, amount, transaction_type, status)
VALUES
    (1, 100.00, 'DEBIT', 'SUCCESS'),
    (2, 200.00, 'CREDIT', 'SUCCESS'),
    (3, 300.00, 'DEBIT', 'SUCCESS');
-- In Spring: use JdbcTemplate.batchUpdate() or Spring Data saveAll()
-- with @Transactional and spring.jpa.properties.hibernate.jdbc.batch_size=50
-- UPDATE with subquery (update account status based on transaction count)
UPDATE accounts
SET status = 'HIGH_VALUE'
WHERE account_id IN (
    SELECT account_id
    FROM transactions
    WHERE status = 'SUCCESS'
      AND created_at > NOW() - INTERVAL '30 days'
    GROUP BY account_id
    HAVING SUM(amount) > 100000
);

Interview Questions — Topic 7

Q17: What is INSERT ... ON CONFLICT DO UPDATE (UPSERT) and when do you use it? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

UPSERT atomically inserts a row or updates it if a unique constraint conflict occurs — no race condition between checking existence and inserting. Syntax: INSERT INTO ... VALUES ... ON CONFLICT (unique_column) DO UPDATE SET col = EXCLUDED.col. EXCLUDED is a special table reference containing the values that were rejected. Use DO NOTHING to silently skip duplicates. Use cases: idempotent event processing (your Outbox/Saga deduplication table), cache invalidation tables, configuration upserts, event deduplication. In your VISA payment system: INSERT INTO processed_transactions ... ON CONFLICT (transaction_id) DO NOTHING ensures idempotent processing even with at-least-once Kafka delivery.

Q18: What is Hibernate batch_size and how does it improve bulk insert performance? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

Without batching: JPA saves 100 entities one by one — 100 separate INSERT statements, 100 round trips to the database. With spring.jpa.properties.hibernate.jdbc.batch_size=50: Hibernate accumulates 50 INSERTs and sends them as a single batch — 2 round trips for 100 entities. Configuration requirements: must use SEQUENCE id generation (not IDENTITY/AUTO) — IDENTITY generation forces Hibernate to flush after each insert to get the generated ID. Also set hibernate.order_inserts=true and hibernate.order_updates=true to group same-entity inserts together. Performance improvement: typically 5-10x for bulk operations. In your payment system: use for bulk transaction import, batch processing jobs.
