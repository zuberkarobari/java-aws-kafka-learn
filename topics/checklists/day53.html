Kafka Scenario Questions

Q1: Your Kafka consumer group has growing lag on partition 2 only, while partitions 0 and 1 have zero lag. What is likely happening and how do you diagnose it? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

Partition-specific lag means the problem is isolated to one consumer thread or one partition's data, not overall throughput. Most likely causes: that partition's consumer thread is stuck on a poison pill message — an event that always throws an exception, preventing the offset from advancing. Or the consumer thread for that partition is slower than the others — perhaps because that partition has unusually large messages or messages requiring more processing. Diagnosis: check the consumer logs for that specific partition's thread — look for repeated exceptions on the same offset. Check message size for partition 2 — unusually large messages would slow processing. If the same offset keeps failing, the message is the poison pill. Fix: configure @RetryableTopic with a DLT so the failing message is moved out of the way after N retries and processing can continue.

Q2: You have acks=all but you are still losing messages when a broker fails. What is the likely configuration gap? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

acks=all means all in-sync replicas (ISR) must acknowledge. But the ISR can contain as few as 1 replica if others have fallen behind. If min.insync.replicas=1 (the default), acks=all with one ISR member (the leader) provides no redundancy. When that leader fails, the acknowledged message is lost because no follower had replicated it. The fix: set min.insync.replicas=2 at the topic or broker level. This requires at least 2 ISR members to acknowledge before the producer gets success. Combine with a replication.factor of at least 3 (two replicas plus the leader). With this configuration, a single broker failure cannot cause data loss because at least one follower always has the latest messages.

Q3: A team asks whether they should use exactly-once semantics for their event processing pipeline. What questions do you ask before recommending? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

I ask four questions. First: what are the consequences of a duplicate? If the consumer is idempotent (storing a result keyed on the event ID), duplicates are harmless and at-least-once is simpler. Second: what are the consequences of losing an event? If loss is unacceptable, exactly-once adds value. Third: what is the throughput requirement? Exactly-once transactions roughly halve throughput — can the pipeline absorb that? Fourth: does the pipeline extend beyond Kafka? Exactly-once within Kafka works well. If the pipeline writes to a database, you need the Outbox pattern or transactional outbox to include the database write in the guarantee. If all four conditions point to exactly-once being needed (duplicates cause real harm, loss is unacceptable, throughput headroom exists, and the sink is Kafka), then recommend it. Otherwise, at-least-once with idempotent consumers is the simpler and safer starting point.

AWS Scenario Questions

Q4: Your payment service on EC2 suddenly cannot write to its S3 bucket, but it was working yesterday. Nothing was deployed. What do you check? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

No deployment change means the code is the same, so the issue is likely IAM or network. First: check CloudTrail for recent IAM policy changes on the EC2 instance role or the S3 bucket policy. A bucket policy change that added a Deny rule would block access even if the instance role has Allow. Second: check if the instance role's policy was changed or detached. Third: check whether the S3 bucket's Block Public Access settings or Object Ownership settings changed — these can affect ACL-based access. Fourth: check CloudWatch for any VPC endpoint changes — if the service was accessing S3 via a VPC endpoint and the endpoint policy changed, that could block traffic. The most common cause: a bucket policy Deny rule added by another team or a security automation tool that is more specific than the instance role's Allow. Deny always wins over Allow in AWS IAM.

Q5: A senior engineer proposes running the payment service on Spot instances to save 70% on EC2 costs. What do you say? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Spot instances are the right tool for interruption-tolerant workloads — batch jobs, data processing, stateless workers that can be replaced in seconds. For a payment service that handles financial transactions, Spot instances are inappropriate. Spot instances receive a 2-minute warning before termination and then the instance is gone. During active payment processing, termination mid-transaction can leave transactions in an inconsistent state, corrupt in-flight Kafka messages, or drop connections that clients are waiting on. The correct cost-saving approach for a payment service: Reserved Instances for baseline capacity (1-year commitment, ~30-40% saving), On-Demand for burst capacity. If the team wants more aggressive savings, Savings Plans provide similar savings with more flexibility. Spot could be used for non-payment workloads in the same account — log processing, analytics, build pipelines.

Q6: You need to give a third-party vendor read access to specific objects in your S3 bucket without giving them an IAM user. How do you do it? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Three approaches depending on requirements. First: pre-signed URLs — generate a time-limited URL for each specific object. The URL includes the credentials and signature, valid for up to 7 days. The vendor downloads the object using the URL with no AWS credentials. This works for ad-hoc, time-limited access to known objects. Second: S3 bucket policy with aws:PrincipalAccount condition — if the vendor has their own AWS account, add a bucket policy that allows their account's ARN to GetObject on specific prefixes. Cross-account access with no IAM user needed. Third: AWS IAM Identity Center (formerly SSO) — create a permission set and grant it to the vendor. They authenticate via identity federation rather than long-term IAM credentials. For a third-party vendor without an AWS account, pre-signed URLs are the simplest. For a vendor with their own AWS account, bucket policy with their account ARN.

Security Scenario Questions

Q7: A developer asks: why do we need to validate the 'aud' claim in a JWT if we already check the signature? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

The signature proves the token was issued by a trusted identity provider with a valid private key. It does not prove the token was intended for your service. Consider a multi-service system: the mobile app gets a JWT from the auth server scoped to the mobile-api audience. That JWT has a valid signature. If your payment-service does not check the aud claim, an attacker who obtains a mobile-api JWT can use it against payment-service. The payment-service trusts the signature and lets them in, even though the token was never meant for that service. Audience validation enforces that a token issued for Service A cannot be used against Service B, even if both services trust the same identity provider.

Q8: You are code reviewing a PR and see that the developer added X-User-Id as an accepted input header in the API gateway JWT filter. They are reading it from the request and passing it downstream. What is the problem? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

The developer is trusting user-supplied input for the user's identity. Any caller can set X-User-Id: admin-123 in their request and the service will treat them as admin-123. The JWT filter's job is to validate the JWT and then inject the authenticated user ID as a trusted header — overwriting any client-supplied value. The correct implementation: remove any incoming X-User-Id, X-Roles, or similar headers from the external request before the JWT filter runs, then inject the gateway-validated values after JWT validation succeeds. The test that catches this: send a request with X-User-Id: admin in the header, verify the downstream service does not see that value, verify it sees the authenticated user's ID from the JWT instead.

Q9: How would you implement service-to-service authentication between your payment service and a fraud service you do not control? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

The answer depends on what the fraud service accepts. If they support OAuth2 Client Credentials: register the payment service as a client with the fraud service's auth server, get a client ID and secret, use Spring's OAuth2AuthorizedClientManager to fetch and cache a scoped token, send it in the Authorization header on every fraud service call. This is the standard approach. If the fraud service uses API keys: store the key in AWS Secrets Manager, fetch it at startup via the instance role, add it to WebClient headers. Never hardcode it. If the fraud service uses mTLS: generate a client certificate signed by a CA the fraud service trusts, configure WebClient with the certificate in an SSLContext. The most important part regardless of method: secrets come from Secrets Manager, not environment variables or config files. Rotation should be automatic where possible.
