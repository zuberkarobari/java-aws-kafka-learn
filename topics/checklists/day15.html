Part 1 — Rapid Recall

Rules:

Cover the right column. Read the question. Say the answer out loud. Then uncover and check. No partial credit — either you knew it instantly or you did not.

Question

Answer

ArrayList default capacity and growth multiplier?

10. Grows by ×1.5 rounded down: newCap = old + (old>>1). 10→15→22→33...

HashMap: what does h ^ (h>>>16) achieve?

Spreads high 16 bits into low 16 bits so bucket index (which only uses low bits) is influenced by the full hash. Reduces collisions.

Treeify threshold in HashMap?

8 nodes in a bucket → convert linked list to Red-Black tree. Untreeify at 6.

ConcurrentHashMap read — is there a lock?

No. Reads use volatile reads of Node.val — zero locking. Fully concurrent.

The compound operation trap in ConcurrentHashMap?

get() + put() is not atomic — race condition between the two calls. Use computeIfAbsent / merge / putIfAbsent instead.

fail-fast vs fail-safe iterator — one example each?

fail-fast: ArrayList (throws CME if modCount changes). fail-safe: CopyOnWriteArrayList (snapshot, never CME).

EnumSet internal storage?

Single long bitmask (RegularEnumSet, ≤64 constants). All ops are single bitwise operation — O(1).

Bill Pugh Singleton — why is it thread-safe without volatile?

The inner static Holder class is loaded by JVM class loading only when getInstance() is called. JVM guarantees class initialisation is atomic.

Factory Method vs Abstract Factory — one-line difference?

Factory Method creates one type of product. Abstract Factory creates a family of related products that must be consistent with each other.

Decorator vs Subclassing — the key advantage?

Decorator is dynamic — compose behaviour at runtime by wrapping. Subclassing is static — decided at compile time. Decorators are also more composable.

Spring self-invocation proxy trap?

CAP theorem — the real practical choice?

P (partition tolerance) is mandatory. The real choice during a partition: C (return error) or A (return stale data).

Cache-aside — what happens on a write?

Update the database, then delete (invalidate) the cache entry. Next read loads fresh from DB.

Thundering herd problem — one fix?

Redis SET NX as a per-key lock: only one request refreshes the cache, others wait for the populated value.

Consistent hashing — why is it better than hash % N?

Adding/removing a node remaps only 1/N of keys instead of nearly all keys. Virtual nodes ensure even distribution.

Hotspot in sharding — one cause?

Using timestamp or date as the shard key: all new writes hit the latest shard. Fix: use high-cardinality, uniform keys (UUID-based IDs).

Token bucket vs leaky bucket — burst handling?

Token bucket allows bursts (accumulates tokens during idle). Leaky bucket smooths traffic to a constant rate regardless of burst.

Dual-write problem — what is it?

Saving to DB and publishing to Kafka are two separate operations. A crash between them leaves them inconsistent.

Outbox pattern — what provides atomicity?

Writing the event to the outbox table in the same DB transaction as the business data. The DB transaction commits both or neither.

CQRS read model — why is it eventually consistent?

The projection updating the read model runs after the event is published — there is a small window where the write has committed but the read model has not yet updated.

Part 2 — 60-Question Blitz

Rules:

Day 11 — Collections (Questions 1–15)

1. Time complexity: ArrayList.get(int), add(E), add(int,E), remove(int)?

get: O(1). add(E) end: O(1) amortised. add(int,E): O(n) shift. remove(int): O(n) shift.

2. Arrays.asList() — can you add elements? Can you modify elements?

Cannot add/remove (UnsupportedOperationException). Can set() — modifies the original array too.

3. HashMap resize trigger?

Resizes at size > capacity × loadFactor (default 0.75). At capacity 16: resize on 13th entry.

4. LinkedHashMap: what does accessOrder=true do?

Each get() moves the accessed entry to the tail. Enables LRU eviction with removeEldestEntry.

5. TreeSet equality — the hidden trap?

Uses compareTo/compare for equality, not equals(). If compareTo returns 0, element is treated as duplicate even if equals() says false.

6. PriorityQueue: is iteration sorted?

No. Iteration does not give sorted order. Only poll() removes in sorted (min-heap) order.

7. Why is java.util.Stack deprecated in favour of ArrayDeque?

Stack extends Vector which synchronises every method even in single-threaded use. ArrayDeque is unsynchronised and faster.

8. Three safe ways to remove elements during List iteration?

1) Iterator.remove(). 2) removeIf(predicate). 3) stream().filter().collect() into new list.

9. CopyOnWriteArrayList: O(?) per write, O(?) per read?

Write: O(n) — copies entire array. Read: O(1) — lock-free snapshot.

10. What does Collections.unmodifiableList() NOT protect against?

Changes made through the original mutable list reference are visible through the unmodifiable view.

11. ConcurrentHashMap: what lock is used for an occupied bucket write?

Synchronised on the first Node (first element) of that bucket only. Other buckets are unaffected.

12. EnumMap vs HashMap — key difference?

EnumMap uses a dense array indexed by ordinal(). No hashing, no collision. O(1) direct lookup, dramatically less memory.

13. What is modCount in ArrayList?

A counter incremented on every structural modification. Iterator stores expectedModCount at creation. Mismatch → ConcurrentModificationException.

14. LinkedBlockingQueue vs ArrayBlockingQueue — one structural difference?

LinkedBlockingQueue has separate head and tail locks — producers and consumers proceed concurrently. ArrayBlockingQueue has one shared lock.

15. When would you use WeakHashMap?

When you need a cache where entries expire automatically when the key has no other strong references — prevents memory leaks.

Day 12 — Design Patterns (Questions 16–30)

16. Singleton: which implementation is serialisation-safe without extra code?

Enum singleton. JVM guarantees one instance and handles serialisation automatically.

17. Why does the naive double-checked locking fail without volatile?

CPU can reorder: assign reference before initialisation completes. A second thread sees non-null but partially-constructed object. volatile adds a memory barrier.

18. What problem does the Builder pattern specifically solve?

Telescoping constructors (unreadable many-argument constructors) and partially-constructed objects. Makes required vs optional fields explicit.

19. Abstract Factory vs Factory Method — one sentence each?

Factory Method: creates one type, lets subclasses decide the concrete class. Abstract Factory: creates a family of related objects that must be used together.

20. Prototype: why is Cloneable problematic?

clone() is protected in Object, requires casting, does a shallow copy by default. Mutable fields are shared between original and copy. Copy constructor is cleaner.

21. Adapter pattern: what does it adapt?

An existing interface that the client cannot use directly into the interface the client expects. Makes incompatible interfaces work together.

22. Decorator vs inheritance — the practical advantage?

Compose behaviour at runtime by wrapping objects. Add logging + metrics + retry to any service without 3x subclasses.

23. What does a Facade pattern specifically hide?

The complexity of a subsystem. Exposes a simplified interface. The subsystem services still exist — the facade just orchestrates them.

24. Spring @Transactional self-invocation — the underlying reason?

Spring wraps @Beans in proxies. Calling this.method() bypasses the proxy entirely, so the transaction advice never runs.

25. Observer pattern: what is @TransactionalEventListener for?

Fires the listener only after the current transaction commits successfully. Prevents side effects (e.g., sending emails) when the transaction later rolls back.

26. Strategy pattern: how does Spring @Qualifier implement it?

Multiple implementations of the same interface = multiple strategies. @Qualifier tells Spring which one to inject. Switch strategies by changing config.

27. Template Method: what does 'final' on the template method enforce?

Subclasses cannot override the overall algorithm structure. They can only fill in the specific abstract steps.

28. Command pattern: how does Kafka implement it at scale?

A Kafka message IS a Command object serialised onto a topic. Queued, retryable, replayable, with the log as audit trail.

29. Proxy: what is the difference between JDK dynamic proxy and CGLIB?

JDK proxy: works only for interfaces, uses reflection. CGLIB: subclasses the concrete class, used when no interface exists. Spring auto-chooses.

30. Open/Closed Principle: which two patterns implement it best?

Strategy (add new algorithm without modifying the context) and Observer (add new listener without modifying the publisher).

Day 13 — System Design (Questions 31–45)

31. Horizontal scaling requirement — what must the application be?

Stateless. Any in-memory session state must be externalised to Redis or a database before horizontal scaling is possible.

32. L4 vs L7 load balancer — one concrete difference?

L4 routes by IP/TCP port, no payload inspection. L7 routes by HTTP path, headers, cookies. AWS NLB = L4. AWS ALB = L7.

33. IP hash load balancing — the main drawback?

34. CAP: which systems choose CP, which choose AP?

CP: Zookeeper, etcd, HBase. AP: Cassandra, DynamoDB (default), Couchbase.

35. PACELC: what does the 'EL' part add?

During normal operation (no partition): trade-off between low Latency (serve local data) and strong Consistency (wait for all replicas).

36. Write-through caching — one advantage and one disadvantage?

Advantage: cache always consistent with DB. Disadvantage: every write pays double latency; caches data that may never be read.

37. Write-behind caching — when would you use it?

Write-heavy, loss-tolerant workloads. App writes to cache only, cache asynchronously flushes to DB. Risk: data loss if cache crashes.

38. Redis Sorted Set: which rate limiting algorithm does it implement?

Sliding window log. Scores are request timestamps. Remove scores older than the window, count remaining.

39. Consistent hashing: how many keys move when one node is added (N nodes)?

Approximately 1/N of all keys. Only keys between the new node and its predecessor on the ring move.

40. Database sharding: why is timestamp a bad shard key?

All new writes hit the latest shard. Old shards sit idle. This is the time-series hotspot problem.

41. Token bucket: what does burst capacity mean?

The maximum number of tokens the bucket can hold. Requests during idle periods accumulate tokens up to this cap, allowing a short burst.

42. Idempotency key: what TTL would you set and why?

24-48 hours. Long enough to cover retry windows. Short enough to not grow the store indefinitely.

43. Back-pressure via Kafka: what is the signal?

Consumer lag. If lag is growing, consumers are slower than producers. Add consumer instances or reduce producer rate.

44. HikariCP pool size formula?

(CPU cores × 2) + effective disk spindles. For a 4-core SSD instance: 9. Bigger pool adds contention, not throughput.

45. Thundering herd: what is probabilistic early expiration?

Randomly expire the cache slightly before TTL for a fraction of requests, so the cache refreshes before everyone misses at once.

Day 14 — Microservices Patterns (Questions 46–60)

46. Why does 2PC fail in microservices?

Requires a coordinator to hold distributed locks across all services from prepare to commit. Slow or unavailable services block the entire transaction.

47. Saga: what replaces atomicity in the traditional ACID sense?

Compensating transactions. Each step commits locally. If a later step fails, previous steps are undone by new forward compensation transactions.

48. Choreography Saga: how does compensation propagate?

The failing service publishes a failure event. Previous services listen for it and run their own compensating transaction.

49. Orchestration Saga: why is it easier to debug than choreography?

The orchestrator persists its current step and status. You query one table to know exactly where the Saga is and what has run.

50. CQRS read model: what is the consistency window?

The time between when the write commits and when the projection updates the read model. Typically milliseconds with Kafka; could be seconds under load.

51. Event Sourcing: how is current state read?

Replay all events from the event log (or from the latest snapshot) and apply each to an in-memory aggregate object.

52. Event Sourcing snapshot: when is it taken?

Every N events (typically 50-100). Reduces replay cost — load snapshot then only events since it.

53. Event Sourcing: can you delete an event?

No. The event log is immutable and append-only. This is what gives you the full audit trail. To 'undo' something, append a compensating event.

54. Outbox: why must consumers be idempotent?

The relay provides at-least-once delivery. If it publishes but fails to mark the event as published, it publishes again on the next poll.

55. Debezium: what does it read from PostgreSQL?

The Write-Ahead Log (WAL). Every committed row change is streamed to Kafka without any application code changes.

56. Distributed trace: what is the traceId?

A globally unique 128-bit ID assigned to a request at the entry point, propagated to all downstream services via HTTP and Kafka headers.

57. W3C trace context header name?

traceparent. Format: 00-{traceId}-{parentSpanId}-{flags}.

58. API Gateway: name three things that belong there and two that do not.

59. @TransactionalEventListener vs @EventListener — one concrete difference?

@TransactionalEventListener fires only after the transaction commits. @EventListener fires immediately when the event is published, regardless of transaction outcome.

60. CQRS + Event Sourcing: are they the same thing?

No. CQRS separates read/write models. Event Sourcing is about how state is stored on the write side. They complement each other but are independent patterns.

Q1: Walk me through how HashMap works internally, from putting a key-value pair to retrieving it, including what happens when there are hash collisions. [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

On put(key, value): first compute h = key.hashCode(), then apply the spreading function h ^ (h >>> 16) — this XORs the high 16 bits into the low 16 so that poor hash codes which only differ in high bits still distribute across buckets. Then compute bucket index as (n-1) & hash, where n is always a power of 2 — this is equivalent to hash % n but uses a fast bitwise AND. Check the bucket: if empty, insert a new Node. If occupied, walk the linked list (or Red-Black tree) comparing hash first (cheap integer equality) then equals() (correct semantic equality). On collision: append to the list. If the list reaches 8 nodes, convert to a Red-Black tree (treeification) — this caps worst-case lookup at O(log n) instead of O(n), and defends against hash collision DoS attacks. On get(key): same hash chain, same traversal. Resize: when size exceeds capacity × 0.75 (load factor), create a new array of double capacity and rehash all entries.

Q2: Tell me about a time you had to design something for resilience in a distributed system. What did you build and what failure modes did you handle? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

In the Deloitte JP Morgan project, we had a CloudWatch log lookup feature that called the AWS SDK and a downstream compliance service. The compliance service had intermittent timeouts — when it was slow, the thread pool would fill with waiting threads and eventually our own service would start timing out on unrelated requests. I added three things: a Resilience4j CircuitBreaker on the compliance service client so that after 50% of calls failed in a 10-second window, the circuit opened and we returned a cached or degraded result fast instead of hanging. A TimeLimiter with a 2-second deadline on each call — after that it was treated as a failure and the circuit state updated. And a Bulkhead — a separate fixed thread pool for compliance calls so that even if they all blocked, the main thread pool for user-facing requests was unaffected. I also added the Outbox pattern for events that needed to flow to Kafka — we had seen cases where our app crashed after writing to the DB but before publishing, leaving the downstream Kafka consumers out of sync. Wrapping the DB write and outbox row in the same transaction fixed that. These were not academic decisions — each came from a production incident we had observed.

Q3: What is the difference between Saga choreography and orchestration, and how would you choose between them for a payment transfer flow? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

Choreography: services react to events with no central coordinator. Service A publishes an event, Service B listens and reacts, B publishes another event, C listens. Loose coupling, easy to add a new step without modifying existing services. The problem is observability — when step 4 of a 6-step flow fails, you have to correlate events across 4 different Kafka topics to understand what happened. Orchestration: a central orchestrator drives each step and persists its state. You can always look at the orchestrator's state table and know exactly where the Saga is. For a payment transfer flow — ValidateFunds, DebitSource, CreditDestination, NotifyUser — I would use orchestration. Transfers are financial operations where the compensation logic matters a lot and I need to be able to answer 'did the debit happen before the credit failed?' from a single source of truth. Choreography is fine for simpler 2-3 step flows, but for anything financial with real compensation consequences, I want the clarity of an orchestrator. In production I would use Temporal — it gives durable execution, automatic retry with backoff, and full workflow history out of the box.

Q4: Explain CQRS. When would you recommend it on a project and when would you tell the team it is not worth the complexity? [Medium] ▼
💡 Ideal Answer (Amazon/GS Standard)

CQRS separates the write model from the read model. The write model enforces domain invariants — account balance cannot go negative, order cannot be cancelled after shipment. The read model is optimised for query patterns — denormalised, pre-aggregated, shaped for what dashboards and reports need. They are kept in sync via domain events consumed by projections. I would recommend it when: the read and write patterns are genuinely different and the normalised write schema is expensive to query for dashboards; when you need to scale reads and writes independently; or when you have multiple read models for different consumers — mobile app, admin dashboard, reporting system — that all need different shapes of the same data. I would say it is not worth it when: the service is mostly CRUD with no complex domain logic, when the team is small and maintaining two models doubles their workload for little gain, or when eventual consistency on reads is politically unacceptable to the business. Most services should start without CQRS and add it when a specific read performance or scalability problem makes it clearly worth the cost.

Q5: You are seeing p99 latency spikes on your payment service every 4 hours. How do you debug this? [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

The 4-hour regularity is a strong signal — it is almost certainly a scheduled task rather than a random failure. My first step is to look at the CloudWatch metrics for that 4-hour window: CPU, memory, GC pause time, DB connection pool wait time, and Kafka consumer lag. If GC pause time spikes at the same time as latency — that is likely full GC, which means a memory leak or a large allocation in the hot path. I would look at what changed 4 hours before each spike and check for any @Scheduled tasks, cron jobs, or scheduled Lambda triggers running on that interval. If it is not GC, I would look at the thread pool metrics — are threads stuck waiting on something? I would pull distributed traces for requests during the spike window in X-Ray and look for which span is slow. Common culprits at regular intervals: database index rebuild jobs, log rotation, token refresh batch jobs that suddenly acquire a global lock, or a Kafka consumer rebalance triggered by a heartbeat timeout. Once I have the slow span, I look at what that span is waiting for — DB lock, external API, or internal resource — and fix the root cause. The 4-hour pattern is the clue: fix the scheduled job or the resource it contends for.

Q6: What is the Outbox pattern? Walk me through implementing it in a Spring Boot service, and explain what happens if the relay process crashes. [Hard] ▼
💡 Ideal Answer (Amazon/GS Standard)

The problem: I need to save a payment to the database and publish a PaymentCompletedEvent to Kafka as part of the same logical operation. If I save to DB and then publish to Kafka, a crash between those two steps leaves them inconsistent — the DB has the payment but downstream services never received the event. The Outbox pattern: write the event to an outbox table in the same database transaction as the payment. The DB transaction guarantees atomicity — either both rows commit or neither does. A separate relay service (I use a @Scheduled method at 500ms intervals) queries for unpublished outbox rows, publishes each to Kafka, and marks the row as published. If the relay crashes after publishing to Kafka but before marking the row as published, the row will be picked up on the next poll and published again — this is at-least-once delivery. Consumers must be idempotent: they use the event's own ID to detect and skip duplicates. The production alternative to a polling relay is Debezium, which reads PostgreSQL's write-ahead log directly and streams changes to Kafka in near real-time without any polling overhead or additional outbox table. I would choose the polling relay for simplicity in smaller services and Debezium when I need near real-time delivery and have the operational capacity to run it.
