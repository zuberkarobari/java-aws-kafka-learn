Day 5 of 30 IAM — Identity & Access Management Policy evaluation engine Production security patterns Credential attacks & defences
IAM is not a permission checkbox system. It is a distributed policy evaluation engine that runs on every single AWS API call — billions of times per day. Understanding the evaluation logic explains why access is granted or denied in every scenario, including the counterintuitive ones that break production systems and get engineers fired.
What IAM actually is at the infrastructure level
A globally distributed, strongly consistent authorization system evaluated on every API call

IAM is a globally distributed authorization system. Every AWS API call — whether from the console, CLI, SDK, or any AWS service — passes through IAM for authentication and authorization before anything happens. There are no exceptions. When you run aws s3 ls, your CLI authenticates with IAM, IAM evaluates whether your identity has s3:ListAllMyBuckets permission, and only then does S3 return the bucket list. The S3 service itself never touches your credentials — IAM handles all authorization and passes a go/no-go decision to S3.

IAM is one of the few AWS services that is truly global — it does not exist per-region. IAM users, roles, and policies you create are replicated across all AWS regions and are immediately available everywhere. This is different from S3 (regional), EC2 (AZ-scoped), or RDS (AZ-scoped). IAM uses strong consistency — a policy change is visible immediately across all regions, not eventually consistent.

Why IAM strong consistency matters in production: If you revoke a permission from an IAM role at 2:00 PM, any API call made using that role at 2:00:01 PM will be denied — no propagation delay. This is critical for incident response. If you discover a compromised IAM role, attaching a deny-all policy is immediately effective globally. Contrast this with rotating a secret (which takes time to propagate across all systems using it). IAM policy changes are the fastest lever for containing a security incident.

IAM's internal architecture: IAM stores policy data in DynamoDB-like replicated storage across all AWS regions. When an API call is made, the IAM authorization service receives the request context (caller identity, action being requested, resource ARN, request conditions like source IP and time), loads all applicable policies for that identity, and runs the policy evaluation algorithm — all in microseconds. The authorization decision is cached briefly at the service level but IAM changes take effect within seconds everywhere.

IAM is not a firewall. It does not block network traffic. It blocks API calls. An attacker who has network access to your EC2 but no IAM credentials cannot make AWS API calls. An attacker who has IAM credentials but is not inside your VPC can make API calls from anywhere on the internet. Network security (VPC, Security Groups) and IAM security are independent, complementary layers — neither is sufficient alone.

The IAM policy evaluation algorithm — the exact logic
Six steps, evaluated in order. Understanding this explains every access denied error you will ever see.

When IAM evaluates whether an API call is allowed, it follows a deterministic algorithm. Every "Access Denied" error in AWS is explained by one of these steps. Senior engineers can identify which step failed from the error message and context.

1
Is there an explicit DENY in any applicable policy? IAM collects all applicable policies: identity-based policies attached to the caller, resource-based policies on the target resource, SCPs from AWS Organizations, permission boundaries, and session policies. If ANY policy contains an explicit Deny statement matching this action on this resource — the request is denied immediately. No further evaluation. Explicit Deny overrides everything else, including explicit Allow. You cannot Allow your way out of an explicit Deny.
2
Does an SCP (Service Control Policy) deny the action? If the account is in an AWS Organization, SCPs apply as a permission boundary for the entire account. An SCP cannot grant permissions — it can only restrict what is possible within the account. If the SCP does not allow the action (either explicitly denies it or simply doesn't include it), the request is denied even if the IAM identity has explicit Allow. The SCP is evaluated before identity policies.
3
Is this a resource-based policy with an explicit Allow for this principal? If the target resource has a resource-based policy (e.g. S3 bucket policy, KMS key policy, Lambda resource policy) that explicitly allows the caller's principal, the request is allowed — even if the caller's IAM identity policy has no explicit allow. Resource-based policies with explicit principal Allows can grant access across accounts and even to services.
4
Is there a permission boundary on this IAM entity? Permission boundaries are IAM managed policies attached to an IAM user or role that set the maximum permissions that entity can ever have. If a permission boundary is attached and the requested action is not allowed in the boundary policy, the request is denied — even if the entity's identity policy explicitly allows it. Used to safely delegate IAM permission management to developers without risking privilege escalation.
5
Is there an identity-based policy with an explicit Allow? The caller's IAM user policies, group policies, and role policies are evaluated. If any of these explicitly allow the action on the resource, and no prior step denied it, the request is allowed.
6
Default: implicit deny. If no explicit Allow was found and no explicit Deny was triggered, the request is implicitly denied. The default in IAM is deny everything — you must explicitly grant every permission. This is the correct security default.
The most counterintuitive consequence of this algorithm: Explicit Deny always wins. If you attach an AdministratorAccess policy to an IAM user (allows everything), but also attach a policy with an explicit Deny for s3:DeleteObject — the user cannot delete S3 objects. The Deny wins even over AdministratorAccess. This is how "break glass" deny policies work: create a deny-all policy and attach it immediately during an incident to neutralize a compromised role faster than you can delete or modify the existing allow policies.
The implicit deny confusion — "I didn't deny it, why is it denied?" A new engineer attaches a policy granting s3:GetObject and s3:PutObject to an IAM user. The user tries to run aws s3 ls (which calls s3:ListAllMyBuckets) and gets Access Denied. The engineer is confused — "I didn't deny ListAllMyBuckets." But they didn't Allow it either. IAM defaults to deny. Every action the identity needs must be explicitly allowed. There is no "allow everything except what I've denied" — it's "deny everything except what I've allowed."
The 6 policy types and when each one applies
Most engineers know 2 of the 6. The others explain half of all production IAM failures.

1. Identity-based policies — attached to IAM users, groups, or roles. Define what that identity can do. The most common policy type. Can be managed (AWS-managed or customer-managed) or inline (embedded directly in the identity, cannot be reused).

2. Resource-based policies — attached to AWS resources (S3 bucket policies, KMS key policies, Lambda resource policies, SQS queue policies, etc.). Define who can access this resource. Support cross-account access — a bucket policy can grant access to a principal from a different AWS account. Not all services support resource-based policies; EC2, RDS, and most compute services do not.

3. Permission boundaries — an IAM managed policy attached to a user or role that sets the maximum permissions that entity can ever have, regardless of what identity-based policies say. If the boundary allows s3:GetObject but the identity policy allows s3:*, the entity can only do s3:GetObject. The effective permission is the intersection of the identity policy and the boundary. Used to safely delegate IAM management: a developer can create and manage their own IAM roles, but cannot create roles with permissions beyond what their own permission boundary allows — preventing privilege escalation.

4. Organizations Service Control Policies (SCPs) — policies attached at the AWS Organization, OU (Organizational Unit), or account level. Apply to all principals in the account or OU. Cannot grant permissions — can only restrict. An SCP that doesn't include s3:DeleteBucket means no principal in that account, including root, can delete S3 buckets — regardless of what IAM policies say. Used for account-wide guardrails: prevent resource creation outside approved regions, prevent disabling CloudTrail, prevent spending above a threshold.

5. Access control lists (ACLs) — legacy S3 and VPC mechanism. AWS recommends against using ACLs. S3 ACLs were deprecated as the preferred mechanism after the introduction of bucket policies and Block Public Access. Understanding ACLs exists to explain why old S3 buckets may have unexpected access patterns.

6. Session policies — policies passed inline when assuming a role or federating via web identity. They further restrict the permissions available in the session — they cannot expand beyond what the role itself allows. Used with AWS STS AssumeRole calls to create least-privilege temporary sessions: even if the role allows full S3 access, you can pass a session policy that restricts the session to only reading from one specific bucket key prefix.

Effective permissions = intersection, not union: Your effective permissions are not the union of all applicable policies — they are constrained by every limiting layer. You can think of it as: (Identity policy ∩ Permission boundary ∩ SCP ∩ Session policy) with resource-based policy grants potentially adding access from the resource side. Every restricting layer must allow the action. One layer saying "no" or "nothing" is enough to deny access.
Cross-account access — the AND logic that everyone gets wrong
Why your IAM policy saying "allow" is not enough when crossing account boundaries

Cross-account access is the most commonly misunderstood IAM scenario. The evaluation logic changes in a critical way when the requesting principal is from a different AWS account than the resource.

Same-account access: If a principal in Account A accesses a resource in Account A, either the identity-based policy OR the resource-based policy can grant access. They are OR'd. If the identity policy allows it and no explicit deny exists, access is granted — even if there's no resource-based policy on the resource.

Cross-account access: If a principal in Account A accesses a resource in Account B, BOTH the identity-based policy in Account A AND the resource-based policy in Account B must explicitly allow the access. This is AND logic. One side allowing is not sufficient.

Why AND logic for cross-account: Account B's resource owner should not be at the mercy of Account A's IAM configuration. If cross-account access worked with just Account A's allow, then any Account A admin could grant their principals access to Account B's resources by simply writing an allow policy — without Account B's resource owner knowing or consenting. The AND requirement means Account B must explicitly opt in by creating a resource policy that allows Account A's principal. Both sides must agree. This is the correct security model for inter-account access.
The most common cross-account debugging scenario: Team A's IAM role has s3:GetObject on Account B's bucket (written by Team A's AWS admin). Team A's developer runs aws s3 cp and gets Access Denied. Team A's admin confirms "the IAM policy is correct." The developer raises a ticket with Team B saying "your bucket is broken." Team B checks and confirms the bucket has no bucket policy. That's the problem. Without a bucket policy in Account B that explicitly allows Team A's role ARN, cross-account access is denied — regardless of what Team A's IAM policy says. Fix: Team B adds a bucket policy allowing arn:aws:iam::ACCOUNT_A_ID:role/role-name to perform s3:GetObject on their bucket.

The role assumption pattern for cross-account access: Rather than resource-based policies on every resource, the cleaner cross-account pattern is role assumption. Account B creates an IAM role with a trust policy allowing Account A to assume it. Account A's principal calls STS AssumeRole to get temporary credentials for Account B's role. All subsequent API calls use Account B's credentials — so every resource in Account B sees a same-account access request. This bypasses the AND requirement and requires no resource policies. This is how your Deloitte project likely connects to JP Morgan's AWS environment — role assumption across account boundaries.

IAM Users, Groups, and Roles are not equivalent constructs with different names. They have fundamentally different credential models, lifecycle characteristics, and security properties. Using the wrong one for the wrong use case is the root cause of most IAM security incidents.
IAM Users — long-lived credentials and why they're dangerous
The original IAM construct — still used for humans and legacy systems, increasingly avoided

An IAM User is an identity with long-lived credentials: a username and password for console access, and/or an access key ID + secret access key for programmatic access. "Long-lived" means the credentials remain valid until you explicitly rotate or delete them — they don't expire automatically.

Access keys consist of two parts: an Access Key ID (like AKIAIOSFODNN7EXAMPLE — always starts with AKIA for regular access keys) and a Secret Access Key (a 40-character string). The access key ID identifies which key is being used; the secret access key signs the request. Both are needed to authenticate. An IAM User can have up to two access keys simultaneously — this supports rotation without downtime (create new, update systems, delete old).

The access key in source code — the most common AWS security incident: A developer creates an IAM user for their Spring Boot application and puts the access key in application.properties: aws.accessKeyId=AKIA... and aws.secretAccessKey=xxx. They commit to GitHub. Within minutes, automated bots scanning GitHub for AWS credential patterns find the keys. Within hours, the attacker spins up hundreds of GPU instances for cryptocurrency mining. AWS bill reaches $50,000. AWS detects the anomaly and alerts the account owner, but the damage is done. This is not hypothetical — it happens daily. AWS has automated scanners monitoring GitHub for credential patterns. You will receive an email from AWS before you notice it yourself.

When IAM Users are appropriate: Human console access with password and MFA (though AWS IAM Identity Center — formerly SSO — is preferred for humans in enterprise settings). Legacy systems that cannot assume roles and require static credentials (increasingly rare). CI/CD systems that cannot use OIDC-based role assumption (Jenkins, GitHub Actions, etc. all support OIDC now — static keys for CI/CD are legacy).

When IAM Users are NOT appropriate: Applications running on EC2, Lambda, ECS, or any AWS compute — use IAM roles instead. Service-to-service communication — use roles. Cross-account access — use role assumption. The only valid use case for static access keys in applications is truly legacy systems with no mechanism to support role assumption.

Access key ID prefix tells you what type of credential it is: AKIA = long-term access key for IAM User. ASIA = temporary credential from STS (assumed role). AROA = IAM Role ID (not a credential). AIDA = IAM User ID. AGPA = IAM Group ID. When you see ASIA in logs or error messages, you know you're looking at temporary credentials from a role assumption — this matters for incident response and debugging.
IAM Roles — the correct model for everything that isn't a human
Temporary credentials, automatic rotation, no long-lived secrets — the security model that scales

An IAM Role is an identity that can be assumed by a principal (user, service, or another account). Roles have no long-lived credentials. When a principal assumes a role, AWS STS (Security Token Service) issues temporary credentials: a temporary access key ID (starts with ASIA), a temporary secret access key, and a session token. These credentials expire — default 1 hour, configurable up to 12 hours for most roles, up to 36 hours for cross-account roles. After expiry, the credentials stop working automatically and must be refreshed.

A Role has two policy components: a trust policy (who is allowed to assume this role — the resource-based policy on the role itself) and permission policies (what the role can do once assumed — the identity-based policies attached to it).

Why temporary credentials are fundamentally more secure: If a static access key is leaked, it remains valid indefinitely until you explicitly rotate it. If temporary credentials are leaked, they expire automatically after at most 12 hours. An attacker who steals temporary credentials from your EC2's metadata service has a maximum window of the session duration (typically 1 hour) to use them. After that, the credentials are worthless — no action needed by you. For long-term static keys, you must detect the compromise, identify which systems use the key, update all of them, then delete the old key — a process that takes hours to days during which the attacker still has valid credentials.

Trust policies — the most overlooked security boundary: The trust policy defines who can call STS:AssumeRole on this role. A common mistake: trust policy with Principal * (all AWS principals) or arn:aws:iam::*:root (all AWS accounts). This means any AWS account can assume your role — if they have credentials to call AssumeRole. This is not immediately catastrophic (they still need to call AssumeRole explicitly) but it violates least-privilege and creates an attack surface. Trust policies should be as specific as possible: exactly which account, service, or user should be allowed to assume this role.

The confused deputy attack via over-permissive trust policy: Your production role has trust policy allowing the EC2 service to assume it, plus an extra condition-less trust for arn:aws:iam::123456789:root (your account root). An attacker who compromises any IAM user in your account (not even an admin) can call AssumeRole on your production role from their compromised identity — escalating to production-role permissions. Overly broad trust policies are privilege escalation vectors. Every trust policy entry must have a clear justification and ideally conditions (aws:SourceAccount, aws:SourceArn) to prevent confused deputy scenarios.
Role assumption mechanics — what STS actually does
The exact sequence from AssumeRole call to temporary credentials to authorized API call

Role assumption is the cornerstone of secure AWS architecture. Understanding the exact mechanics helps you design systems correctly and debug credential issues systematically.

Step 1 — AssumeRole call: The requesting principal (IAM User, another Role, EC2 instance profile, Lambda execution role) calls sts:AssumeRole with the target role ARN. IAM evaluates: does the caller have permission to call sts:AssumeRole on this role? Does the target role's trust policy allow this principal? If both checks pass, STS generates temporary credentials.

Step 2 — STS generates temporary credentials: STS creates three values: a temporary access key ID (ASIA...), a temporary secret access key, and a session token (a long, opaque string). The session token contains encrypted metadata about the assumed role, the original caller, the session duration, and any session policies. All three values are returned to the caller. The session token is mandatory — API calls using role credentials must include it.

Step 3 — Using temporary credentials: Any subsequent API call signed with these temporary credentials must include all three values: access key ID, secret access key (for request signing), and session token (in the x-amz-security-token header or as a query parameter). If the session token is missing, the API call fails with an InvalidClientTokenId error even though the access key is valid.

How the AWS SDK handles this automatically: When your Spring Boot app runs on an EC2 with an IAM instance profile attached, the AWS SDK's DefaultCredentialsProvider automatically: queries the EC2 instance metadata service (IMDS) for the role's temporary credentials, caches them, refreshes them before expiry (5 minutes before). You never write credential code. Never call AssumeRole manually. Never store the session token. The SDK does all of this transparently. This is the correct pattern — your application code should be completely credential-agnostic.

Role chaining: An assumed role can assume another role — this is called role chaining. If Role A assumes Role B, and Role B has permission to call sts:AssumeRole on Role C, the session that assumed Role B can then assume Role C. Each hop creates a new temporary session. The maximum session duration for role chaining is capped at 1 hour, regardless of the individual role's MaxSessionDuration setting. This catches engineers who set up cross-account pipelines with role chains longer than one hop and wonder why credentials expire faster than expected.

The session token requirement failure: A developer manually captures temporary credentials from the instance metadata service and pastes them into their local AWS CLI config to debug something from their laptop. They put the access key ID and secret access key but forget the session token. Every CLI command fails with "InvalidClientTokenId" even though the keys look valid. The SDK and CLI require all three values for assumed role credentials. The session token field is not optional — it's the cryptographic proof that this temporary key was issued by STS rather than being a compromised static key.
Instance Metadata Service (IMDS) — how EC2 gets credentials and why it's an attack target
The link between IAM roles and EC2 — and why IMDSv2 is mandatory in production

The Instance Metadata Service (IMDS) is a special HTTP endpoint available on every EC2 instance at http://169.254.169.254. This link-local IP address is not routable — it can only be accessed from within the EC2 instance itself, not from other instances or the internet. IMDS provides instance metadata (instance ID, AMI ID, AZ, security groups) and, critically, temporary IAM role credentials when an instance profile is attached.

To get credentials from IMDS: curl http://169.254.169.254/latest/meta-data/iam/security-credentials/ROLE-NAME returns a JSON with AccessKeyId, SecretAccessKey, Token, and Expiration. The AWS SDK makes this call automatically and caches the result.

SSRF → IMDS → credential theft — a real attack class: Server-Side Request Forgery (SSRF) is a vulnerability where an attacker tricks your application into making HTTP requests to internal URLs on behalf of the attacker. If your Spring Boot app has an endpoint that fetches a URL provided by the user (e.g. a link preview feature, a webhook validator, or an image downloader), an attacker sends the URL http://169.254.169.254/latest/meta-data/iam/security-credentials/. Your app fetches it, gets the role name, fetches the credentials, and returns them in the response. The attacker now has valid AWS credentials for your EC2's IAM role. This is exactly how the Capital One breach worked in 2019 — SSRF against a misconfigured WAF on EC2, credentials extracted from IMDS, attacker accessed S3 buckets containing customer data. 100 million records exposed.

IMDSv1 vs IMDSv2: IMDS version 1 (IMDSv1) allows any HTTP GET to the metadata endpoint — no authentication. IMDS version 2 (IMDSv2) requires a two-step process: first PUT request to get a session token (with a TTL header), then GET requests with the session token in a header. This two-step process prevents SSRF attacks because SSRF exploits typically only follow GET redirects — a PUT request for the session token requires the attacker's application to support non-GET HTTP methods, which most SSRF vectors don't.

Enforce IMDSv2 on every EC2 instance — non-negotiable in production: When launching an EC2 instance, set HttpTokens to "required" — this disables IMDSv1 entirely. Existing instances: run aws ec2 modify-instance-metadata-options --instance-id i-xxx --http-tokens required. At the account level, use an AWS Config rule or SCP to detect or prevent IMDSv1 usage. The AWS SDK v2 and all modern AWS tools support IMDSv2 automatically. If you have legacy code that directly queries IMDS with curl-style GET requests, update it to add the IMDSv2 session token step. This is not optional for anything with a public-facing interface.
Service-linked roles — IAM roles that AWS services manage for you
Understanding these prevents confusion when you see roles you didn't create in your account

Service-linked roles are IAM roles predefined by AWS services. They are automatically created when you use certain AWS services for the first time. They have exactly the permissions that service needs to call other AWS services on your behalf — no more, no less. You cannot modify the trust policy (the service itself must be the only principal). You cannot delete a service-linked role while the service that uses it still exists.

Examples: AWSServiceRoleForElasticLoadBalancing — created when you create your first ALB, used by ELB to call EC2 APIs to register and deregister targets. AWSServiceRoleForAutoScaling — created when you create your first Auto Scaling Group, used by ASG to launch and terminate EC2 instances, attach ENIs, etc. AWSServiceRoleForRDS — created when you create your first RDS instance, used by RDS to call EC2 and VPC APIs for network configuration.

Why you need to understand service-linked roles: When you look at your IAM console and see 20+ roles you didn't create, these are service-linked roles. Engineers unfamiliar with them often think they've been compromised or that something is wrong with their account. They're not a security risk — they're tightly scoped to the specific service's needs and cannot be assumed by arbitrary principals. However, you should audit them periodically. If a service-linked role for a service you don't use is present (e.g. AWSServiceRoleForRekognition when you've never used Rekognition), it might indicate a service was used by an unauthorized party at some point.
Service-linked roles in Terraform: When writing Terraform for AWS services, service-linked roles are typically created automatically by the first resource of that type. However, if you're creating IAM resources in a separate Terraform run from the service resources, you may need to explicitly create service-linked roles in your Terraform: aws_iam_service_linked_role resource. Failing to do so causes Terraform to fail when the service tries to create the role and gets an "already exists" error if the role was previously created.
IAM policy JSON is not just configuration. Every field has specific semantics that determine whether access is granted or denied. A single wrong character — a missing wildcard, an incorrect ARN format, a condition key typo — silently changes what the policy does. Read policies, don't just write them.
IAM Policy JSON — anatomy of every field
Every field has precise semantics — understanding each one prevents subtle security misconfigurations

An IAM policy is a JSON document with a Version and one or more Statement blocks. Here is a complete, annotated policy:

// This policy allows reading from a specific S3 prefix // and writing logs to CloudWatch, but denies all deletes { "Version": "2012-10-17", // Always use this date. It's not a year — it's the policy language version. // "2008-10-17" is the older version — lacks Conditions support. "Statement": [ { "Sid": "AllowS3Read", // Statement ID — optional, used for documentation and debugging. // Appears in CloudTrail when this statement is the matching one. "Effect": "Allow", // "Allow" or "Deny". No other values accepted. "Action": [ // What API actions this statement covers. "s3:GetObject", // Exact action: service:ActionName (case-insensitive for the action) "s3:ListBucket" // s3:* = all S3 actions. s3:Get* = all Get actions. ], "Resource": [ // Which resources this statement applies to. "arn:aws:s3:::my-bucket", // The bucket itself (for ListBucket) "arn:aws:s3:::my-bucket/*" // Objects in the bucket (for GetObject) // Both needed — ListBucket operates on bucket, GetObject on objects. ] }, { "Sid": "DenyAllDeletes", "Effect": "Deny", // This Deny wins over any Allow, including AdministratorAccess. "Action": "s3:DeleteObject", // Can be a string (single) or array (multiple). "Resource": "*", // All resources. Deny statements often use * to be comprehensive. "Condition": { // Optional: restricts when this statement applies. "StringNotEquals": { "aws:PrincipalTag/Role": "DataAdmin" // Only deny if the caller does NOT have the Role=DataAdmin tag. // DataAdmin can delete; everyone else cannot. } } } ] }
The Version field is not the year — it's a syntax version: Always use "2012-10-17". The older "2008-10-17" version lacks support for policy variables like ${aws:username} in resource ARNs. If you write a policy using ${aws:username} with the 2008 version, the variable is not substituted — it becomes a literal string, silently breaking the policy. AWS will accept the 2008 version without warning — the bug is silent and hard to detect.
The ListBucket vs GetObject resource ARN mistake: A common policy error — the engineer writes the policy to allow s3:GetObject and s3:ListBucket, with Resource: arn:aws:s3:::my-bucket/* (objects only). The user can GetObject fine but cannot ListBucket. Why? s3:ListBucket operates on the bucket itself, not objects. The resource must be arn:aws:s3:::my-bucket (without the /*). s3:GetObject operates on objects — resource must be arn:aws:s3:::my-bucket/*. You need both resource ARNs in the policy. This causes one of the most cryptic S3 access errors: "Access Denied" on listing but "success" on direct object gets — which makes no sense until you understand the action-to-resource mapping.
Resource ARNs — the addressing system for every AWS resource
ARN format, wildcards, and the account ID that engineers forget to include

An ARN (Amazon Resource Name) uniquely identifies every AWS resource. Format: arn:partition:service:region:account-id:resource-type/resource-id

arn:aws:s3:::my-bucket # S3 bucket (no region, no account — S3 is global namespace) arn:aws:s3:::my-bucket/path/to/object.pdf # Specific S3 object arn:aws:s3:::my-bucket/* # All objects in bucket arn:aws:iam::123456789012:role/my-role # IAM role (no region — IAM is global) arn:aws:iam::123456789012:user/zuber # IAM user arn:aws:ec2:ap-south-1:123456789012:instance/i-abc123 # EC2 instance arn:aws:rds:ap-south-1:123456789012:db:my-database # RDS instance arn:aws:lambda:ap-south-1:123456789012:function:my-func # Lambda function arn:aws:sqs:ap-south-1:123456789012:my-queue # SQS queue arn:aws:kms:ap-south-1:123456789012:key/key-uuid # KMS key * # All resources (use with care — only in deny statements)
Wildcards in ARNs — powerful but dangerous: The * wildcard in ARN resource fields matches any string of characters. arn:aws:s3:::* = all S3 buckets. arn:aws:ec2:ap-south-1:123456789012:instance/* = all EC2 instances in your account in ap-south-1. arn:aws:iam::123456789012:role/app-* = all roles with names starting with "app-". The ? wildcard matches exactly one character. Use the narrowest ARN that covers the resources you intend to grant access to. A policy that says arn:aws:s3:::* grants access to every S3 bucket in the universe, not just yours — S3 names are globally unique but the * doesn't scope to your account.
The missing account ID privilege escalation: An engineer writes an IAM policy with Action: sts:AssumeRole and Resource: arn:aws:iam::*:role/admin-role. The wildcard on the account ID means this policy allows assuming any role named "admin-role" in ANY AWS account, not just their own. If any other AWS account happens to have a role named "admin-role" with a trust policy that allows all AWS accounts (common in misconfigured accounts), this identity can now assume admin-role in that external account. Always include the specific account ID in role ARNs in policies. Use the ${aws:PrincipalAccount} condition key to dynamically scope to the caller's account.
IAM Conditions — the precision tool that separates good policies from great ones
Restricting access based on IP, time, MFA, tags, VPC, and dozens of other dimensions

Conditions are optional blocks in policy statements that restrict when the statement applies. Without conditions, a statement applies whenever the action and resource match. With conditions, the statement only applies when all conditions are also true.

A condition has an operator, a condition key, and a value. The condition key is a global key (available in all services) prefixed with aws:, or a service-specific key prefixed with the service name (e.g. s3:prefix).

// Allow S3 access ONLY from within your VPC "Condition": { "StringEquals": {"aws:SourceVpc": "vpc-12345678"} } // Require MFA for all IAM actions except the session itself "Condition": { "BoolIfExists": {"aws:MultiFactorAuthPresent": "true"} } // Restrict EC2 launches to only specific instance types "Condition": { "StringEquals": {"ec2:InstanceType": ["t3.micro", "t3.small"]} } // Only allow access during business hours (UTC) "Condition": { "DateGreaterThan": {"aws:CurrentTime": "2024-01-01T09:00:00Z"}, "DateLessThan": {"aws:CurrentTime": "2024-01-01T18:00:00Z"} } // Allow S3 access only from specific IP ranges (your office + VPN) "Condition": { "IpAddress": {"aws:SourceIp": ["203.0.113.0/24", "198.51.100.0/24"]} } // Tag-based access: only allow accessing resources you own "Condition": { "StringEquals": {"aws:ResourceTag/Owner": "${aws:username}"} }
BoolIfExists vs Bool — the condition operator that engineers get wrong: If you use "Bool": {"aws:MultiFactorAuthPresent": "true"}, this condition evaluates to false when MFA is not present in the request context — blocking access for API calls that don't include MFA. But API calls using access keys (programmatic access from CLI or SDK) never include MFA context — the aws:MultiFactorAuthPresent key is absent in the request context. Absent key ≠ false for Bool — Bool with an absent key evaluates to false, which may not be what you intended. Use BoolIfExists when you want "if this key is present in the request, it must have this value" — it skips evaluation if the key is absent. Use Bool when you want to require the key to be present and true (e.g. requiring MFA for console access where the key is always present).
ABAC — Attribute-Based Access Control with tags: Instead of creating a new IAM policy for every new resource, use resource tags and condition keys to create policies that automatically apply to correctly tagged resources. Example: Policy that allows developers to access any resource tagged with Environment=dev and Team=${aws:PrincipalTag/Team}. When you create a new dev S3 bucket or EC2 instance with those tags, your developers automatically have access without any policy changes. This is ABAC — it scales to thousands of resources and hundreds of developers without policy explosion. Used extensively in large organisations to manage IAM at scale.
Critical policy patterns every engineer must know
Production-tested patterns for the scenarios you will encounter in your first year

Pattern 1 — Least privilege for a Spring Boot app on EC2 (what your Deloitte app needs):

{ "Statement": [ { "Effect": "Allow", "Action": ["s3:GetObject", "s3:PutObject"], "Resource": "arn:aws:s3:::app-data-bucket/*" }, { "Effect": "Allow", "Action": "s3:ListBucket", "Resource": "arn:aws:s3:::app-data-bucket", "Condition": {"StringLike": {"s3:prefix": "app-data/*"}} }, { "Effect": "Allow", "Action": ["secretsmanager:GetSecretValue"], "Resource": "arn:aws:secretsmanager:ap-south-1:ACCOUNT:secret:app/prod/*" }, { "Effect": "Allow", "Action": ["logs:CreateLogGroup", "logs:CreateLogStream", "logs:PutLogEvents"], "Resource": "arn:aws:logs:ap-south-1:ACCOUNT:log-group:/app/prod*" } ] }

Pattern 2 — SCP to prevent resources outside approved regions:

{ "Effect": "Deny", "Action": "*", "Resource": "*", "Condition": { "StringNotEquals": { "aws:RequestedRegion": ["ap-south-1", "ap-south-2"] } } }
This SCP denies all actions in all regions except Mumbai and Hyderabad. Attach at the OU level. Any attempt to launch resources in us-east-1 or eu-west-1 is denied at the account level — regardless of the individual IAM user's permissions. IAM global services (IAM itself, Route 53, CloudFront) automatically exempt themselves from region-based SCPs.

Pattern 3 — Permission boundary for delegated IAM management:

// Attach this as a permission boundary to developer IAM roles. // Developers can do anything permitted by their identity policies, // but cannot exceed what this boundary allows. // This means a developer cannot create a role with more permissions than they have. { "Effect": "Allow", "Action": "*", "Resource": "*", "Condition": { "StringEquals": { "iam:PermissionsBoundary": "arn:aws:iam::ACCOUNT:policy/DeveloperBoundary" } } } // Combined with this deny: prevents removing their own boundary { "Effect": "Deny", "Action": ["iam:DeleteUserPermissionsBoundary", "iam:DeleteRolePermissionsBoundary"], "Resource": "*" }
AWS STS, federated identity, IAM Identity Center, and IAM Access Analyzer are the tools that separate IAM beginners from engineers who can architect secure multi-account, multi-team organisations. These topics appear in principal engineer and solutions architect interviews.
AWS STS — Security Token Service mechanics
The service that issues all temporary credentials — and the four ways to use it

STS is the AWS service that issues temporary credentials. Every time a role is assumed, STS generates the temporary access key, secret, and session token. STS is a global service with regional endpoints — you should use the regional endpoint (e.g. sts.ap-south-1.amazonaws.com) rather than the global endpoint to reduce latency and improve reliability.

AssumeRole — called by IAM users, roles, or applications to assume another IAM role. Returns temporary credentials valid for the role's configured duration. The caller must have sts:AssumeRole permission on the target role ARN, and the target role's trust policy must allow the caller.

AssumeRoleWithWebIdentity — called by applications authenticating via an external identity provider using OIDC (OpenID Connect). The application exchanges an OIDC token from Google, GitHub, or any OIDC provider for temporary AWS credentials. This is the mechanism behind Kubernetes service accounts with IAM roles (IRSA — IAM Roles for Service Accounts) and GitHub Actions OIDC.

AssumeRoleWithSAML — called during SAML-based federation with corporate identity providers (Active Directory, Okta, etc.). Enterprises use this to allow employees to log into the AWS console using their corporate credentials without creating IAM users.

GetSessionToken — used specifically to get temporary credentials that include MFA authentication. An IAM user calls GetSessionToken with their MFA code. The returned temporary credentials have MFA context (aws:MultiFactorAuthPresent = true). Subsequent API calls using these credentials satisfy MFA-required conditions in policies.

Why use regional STS endpoints: The global STS endpoint (sts.amazonaws.com) is located in us-east-1. Every AssumeRole call from your ap-south-1 EC2 that uses the global endpoint makes a round trip to the US and back — adding 150-300ms latency to every credential refresh. Regional endpoint (sts.ap-south-1.amazonaws.com) serves the request from Mumbai — <5ms. For Lambda functions that assume roles frequently (e.g. every invocation), this latency difference is significant. The AWS SDK v2 automatically uses regional endpoints when configured with the region. Always configure the region in your SDK.
OIDC-based role assumption — no static keys for CI/CD or Kubernetes
How GitHub Actions, GitLab CI, and Kubernetes pods get AWS credentials without any stored secrets

OpenID Connect (OIDC) allows external identity providers to issue tokens that AWS STS accepts. The workflow: the external system (GitHub Actions, a Kubernetes pod) authenticates with its own identity provider and receives an OIDC token (a JWT). The system presents this OIDC token to AWS STS AssumeRoleWithWebIdentity. STS validates the token against the OIDC provider's public keys, checks the role's trust policy conditions, and issues temporary credentials. No static AWS credentials are stored anywhere.

GitHub Actions + OIDC — the correct CI/CD pattern (replaces static access keys):
1. Create an OIDC Identity Provider in IAM: token.actions.githubusercontent.com
2. Create an IAM role with trust policy allowing the OIDC provider and condition scoping to your specific repo: token.actions.githubusercontent.com:sub = repo:your-org/your-repo:ref:refs/heads/main
3. In your GitHub Actions workflow: permissions: id-token: write + aws-actions/configure-aws-credentials@v3 with role-to-assume
4. GitHub Actions gets a fresh OIDC token per run, exchanges it for temporary AWS credentials, workflow deploys to AWS, credentials expire when the run completes.
No secrets stored in GitHub. Credentials automatically expire. Every deployment is authenticated to a specific repo and branch. This is the architecture your CI/CD should use — replacing any setup that stores AKIA access keys as GitHub Secrets.
IRSA — IAM Roles for Service Accounts on Kubernetes (EKS): When running Kubernetes on AWS EKS, pods need AWS credentials to call S3, DynamoDB, SQS, etc. Without IRSA, you'd inject credentials via environment variables or ConfigMaps — static keys inside Kubernetes. With IRSA: each pod's ServiceAccount is annotated with an IAM role ARN. EKS injects an OIDC token into the pod as a projected volume. The AWS SDK inside the pod automatically exchanges this token for temporary role credentials via AssumeRoleWithWebIdentity. Each pod has exactly the AWS permissions its ServiceAccount is configured for — different pods can have different IAM roles in the same cluster. No static credentials anywhere in Kubernetes.
IAM Identity Center — the right way to manage human access at scale
Why creating IAM Users for every developer in every account doesn't scale and what replaces it

AWS IAM Identity Center (formerly AWS Single Sign-On) is the recommended way to manage human access to multiple AWS accounts in an AWS Organization. Instead of creating IAM users in each account, you define permission sets centrally and assign them to accounts and users.

The problem it solves: A company with 50 engineers and 20 AWS accounts (dev, staging, prod for multiple teams) would need 50×20 = 1,000 IAM users if each engineer had an account in each environment. Managing password rotations, MFA, and access reviews across 1,000 IAM users is operationally impossible. When an engineer leaves, you must delete their user in 20 accounts — and inevitably someone forgets one.

How it works: Connect Identity Center to your identity provider (Active Directory, Okta, Google Workspace). Define Permission Sets (collections of IAM policies — essentially IAM roles). Assign users or groups to accounts with specific permission sets. When a user logs in via the Identity Center portal (or the AWS CLI with aws sso login), they get temporary credentials for their assigned accounts and permission sets. One login, access to all assigned accounts. One deprovisioning action removes access from all accounts simultaneously.

Connecting to your Deloitte context: At a large consultancy like Deloitte, engineers access client AWS environments without having permanent IAM users in the client's account. IAM Identity Center (or equivalent federation) allows Deloitte engineers to authenticate with Deloitte's corporate identity (Okta, Active Directory) and get temporary AWS credentials for the client's account based on their role. When a consultant rotates off the project, removing them from the Deloitte identity provider removes all their AWS access automatically — no manual cleanup in each client account.
Permission sets and role mapping: A permission set in Identity Center corresponds to an IAM role created in each account it's assigned to. The role name follows the format: AWSReservedSSO_PermissionSetName_randomhex. When a user assumes a permission set, they're assuming this role in that account — getting temporary credentials valid for the configured session duration (1-12 hours). CloudTrail records the role assumption with the user's Identity Center username, providing a full audit trail of who accessed what account when.
IAM Access Analyzer — continuous validation of your IAM posture
Automated detection of unintended public and cross-account access — run this before and after every IAM change

IAM Access Analyzer continuously monitors IAM policies, S3 bucket policies, KMS key policies, Lambda function policies, SQS queue policies, and Secrets Manager secrets to identify resources that are accessible from outside your AWS account or AWS Organization. It uses automated reasoning — formal mathematical proof — to determine whether a policy grants access to external principals, not just pattern matching.

What it finds: S3 buckets accessible from other accounts or publicly. IAM roles with trust policies allowing external accounts. KMS keys usable by external principals. Lambda functions that can be invoked cross-account. SQS queues accessible from outside your org. For each finding, it tells you: which external principal has access, which action they can perform, and which policy statement grants the access.

Two usage modes:
External access analysis: Continuously monitors your account for resources accessible from outside. Free tier includes external access analysis. Any unintended external access generates a finding. Every finding should be archived (if intended) or remediated (if not).
Unused access analysis: Identifies IAM users and roles that have not been used within a configurable period. If a role hasn't made any API calls in 90 days, it appears as an unused access finding. Use this to identify zombie roles (roles from decommissioned services) and over-provisioned access that should be revoked. This tier has a per-IAM-entity cost.
Policy validation: Before deploying any IAM policy, run it through Access Analyzer's policy validation: aws accessanalyzer validate-policy --policy-document file://policy.json --policy-type IDENTITY_POLICY. It checks for: overly broad wildcards, policies that don't follow security best practices, actions that don't exist (typos), resource ARN format errors, and known security anti-patterns. Integrate this into your Terraform CI/CD pipeline as a pre-deploy check. A policy with a typo in an action name (e.g. s3:GetObiect instead of s3:GetObject) silently grants nothing for that action — the policy looks right but doesn't work. Policy validation catches this before deployment.
The stale admin role finding you shouldn't archive: Access Analyzer shows a finding: an IAM role with AdministratorAccess has not been used in 180 days. The engineer archives it saying "it's our emergency break-glass role." Six months later, the company has an incident and an engineer uses the break-glass role. Access Analyzer shows it was used once in the last year. Is that suspicious or the intended break-glass? Without proper procedures, you cannot tell. Break-glass roles should: have a Condition requiring MFA, log to a dedicated CloudWatch alarm that alerts security team immediately on any use, be stored in a separate emergency credentials vault with documented procedure, and be tested quarterly. Unused ≠ unnecessary — but it must be managed intentionally.
IAM misconfiguration is the #1 cause of AWS security incidents. Not zero-days, not sophisticated attacks — misconfigurations. Every pattern below represents a real attack that has compromised real production environments. Understanding attack patterns is the only way to design defences that actually work.
Credential exposure — how AWS secrets are leaked and stolen
The attack surface is larger than engineers think — and the exploitation is faster than you expect

Git repository exposure: The most common vector. Access keys committed to public or private repositories. Automated bots scan GitHub, GitLab, and Bitbucket for credential patterns (AKIA followed by 16 alphanumeric characters). Private repos are scanned too if the attacker gets repository access. AWS has its own partnership with GitHub that alerts you if credentials are found in public repos — but this detection happens after the exposure, not before. Prevention: git-secrets or Gitleaks as pre-commit hooks blocking commits containing credential patterns. AWS Secrets Manager rotation eliminates static credentials entirely.

IMDS credential theft via SSRF: Covered in detail in the entities section. Any public-facing application with an HTTP request capability (URL fetchers, webhooks, image downloaders, link previewers) is vulnerable if it doesn't validate and restrict the URLs it fetches. The fix beyond IMDSv2: validate that no user-provided URLs resolve to 169.254.169.254 or other private IP ranges before fetching. Use an allowlist of permitted domains rather than a blocklist.

Environment variable exposure: Applications log their environment variables during startup (common debugging practice) or expose them via actuator endpoints. Spring Boot Actuator's /env endpoint exposes all environment variables including AWS credentials. Ensure Actuator's sensitive endpoints require authentication. Never log environment variables in production. Use Secrets Manager and fetch secrets programmatically rather than injecting them as environment variables.

What happens in the 10 minutes after a credential is exposed on GitHub: Minute 0 — developer commits access key. Minute 1-2 — automated scanners (both malicious and AWS's own) detect the pattern in the new commit. Minute 3 — attackers run aws sts get-caller-identity to verify the credentials work. Minute 4 — run aws iam list-roles, aws iam list-policies to understand what they can do. Minute 5 — spin up EC2 instances in all available regions for mining. Minute 7 — attempt to create new IAM users with admin access for persistence. Minute 10 — AWS's abuse detection triggers and may temporarily suspend the account. By this point, $5,000-50,000 in compute charges have been committed. The git history shows the credentials were exposed for 10 minutes.
Privilege escalation — getting more permissions than you should have
17 known IAM privilege escalation paths — every one of them uses legitimate AWS APIs

Privilege escalation in AWS does not require exploiting vulnerabilities. Every escalation path uses legitimate, documented AWS APIs — the problem is having the permissions to call them in the wrong combination. Security researcher Spencer Gietzen documented 17 privilege escalation paths in AWS IAM. Here are the most important ones:

iam:CreatePolicyVersion: If you have permission to create new versions of an existing IAM policy, you can create a version that grants AdministratorAccess, then set it as the default version. You've upgraded a restricted policy to admin without creating new resources — only modifying an existing one.

iam:AttachRolePolicy on a role you can assume: If you have iam:AttachRolePolicy on a role, and you can assume that role, you can attach AdministratorAccess to the role, then assume it. You now have admin.

iam:PassRole + ec2:RunInstances: iam:PassRole allows you to pass an IAM role to an EC2 instance when launching it. If you can pass a high-privilege role to a new EC2 instance, and you can SSH into that instance (or use SSM), you can retrieve the high-privilege role's credentials from IMDS. You've escalated from your restricted identity to the high-privilege role.

lambda:CreateFunction + lambda:InvokeFunction + iam:PassRole: Create a Lambda function, pass a high-privilege role to it, invoke the function with code that calls sts:GetCallerIdentity (or anything else). The function runs with the high-privilege role's credentials.

The permission boundary defence against escalation: Attach permission boundaries to every IAM user and role that has any IAM write permissions. The boundary limits the maximum permissions to a pre-approved set — even if the entity manages to attach AdministratorAccess to itself or its roles, the boundary limits what can actually be done. Permission boundaries are the correct technical control for IAM privilege escalation, not just procedural controls like "engineers shouldn't have iam:*".
iam:PassRole is often over-granted: A common IAM policy anti-pattern: allowing iam:PassRole with Resource: *. This means the identity can pass ANY role to any service. If there's a high-privilege role anywhere in the account, this identity can pass it to EC2, Lambda, or ECS and use it. Correct: scope iam:PassRole to specific roles: arn:aws:iam::ACCOUNT:role/allowed-roles/*. Or better: only allow passing roles that have a specific permission boundary attached (using a condition on the boundary ARN). iam:PassRole without restrictions is one of the most dangerous permissions to grant.
Lateral movement — using legitimate access to reach unintended resources
How attackers move from a compromised EC2 to production databases and S3 buckets

Once an attacker has valid IAM credentials (via any of the exposure methods), their goal is lateral movement — using those credentials to access resources beyond their intended scope. The mechanics are all legitimate API calls.

Reconnaissance phase: The first actions after getting credentials are always reconnaissance — understanding what the credentials can do. sts:GetCallerIdentity (who am I?), iam:GetAccountAuthorizationDetails (dump all IAM policies and roles), ec2:DescribeInstances (what EC2s exist?), s3:ListAllMyBuckets (what buckets exist?). These read-only actions are rarely restricted, so attackers typically get full reconnaissance even with limited credentials.

Moving from EC2 role to S3: If the compromised EC2's role has overly broad S3 permissions (s3:* on *), the attacker reads all buckets, finds interesting ones (database exports, credential backups, customer data), and exfiltrates. The S3 access looks like legitimate application activity in CloudTrail — same role, same region, similar time patterns. Detection requires behavioral analytics, not just rule-based alerts.

Moving from EC2 to RDS: Network-level: if the compromised EC2 has the app-sg Security Group, it can reach RDS on port 3306. The attacker installs a MySQL client, finds the DB endpoint from environment variables or config files, and connects using credentials also found in config files or Secrets Manager (if the role has secretsmanager:GetSecretValue). This is why: (1) never log database credentials, (2) use IAM database authentication for RDS instead of static passwords, (3) scope Secrets Manager access to specific secret ARNs, not wildcard.

The defences that actually work against lateral movement:
Least privilege — every role has only the minimum permissions needed. Reduces blast radius when a role is compromised.
IMDSv2 — prevents SSRF-based credential theft.
VPC Endpoints with bucket policies restricting access to specific VPC — even with valid credentials, S3 access is denied if not from the VPC.
CloudTrail + GuardDuty — GuardDuty detects anomalous API patterns: credentials used from unusual geo, high-volume reconnaissance activity, known malicious IPs. Alert within minutes of unusual activity.
SCP deny-delete on CloudTrail — prevents attackers from disabling audit logging to cover their tracks. The first thing sophisticated attackers do is try to disable CloudTrail.
Detection and incident response — what to do when IAM is compromised
The first 15 minutes after detecting a compromised IAM credential determine how bad the damage gets

Detection signals: AWS GuardDuty findings: UnauthorizedAccess:IAMUser/TorIPCaller (credential used from Tor exit node), UnauthorizedAccess:IAMUser/ConsoleLoginSuccess (IAM user login from unusual location), Recon:IAMUser/MaliciousIPCaller (API calls from known malicious IP). CloudTrail: High volume of DescribeInstances, ListBuckets, or GetSecretValue calls from a single credential in a short window. Unusual IAM actions: CreateUser, AttachUserPolicy, CreateAccessKey — these are privilege escalation attempts.

Immediate containment — in order:

1. Attach an inline deny-all policy to the compromised entity immediately. This is faster than deleting the policy or the entity. aws iam put-user-policy --user-name COMPROMISED_USER --policy-name EmergencyDenyAll --policy-document '{"Statement":[{"Effect":"Deny","Action":"*","Resource":"*"}]}'. This takes effect globally in seconds. The attacker's subsequent API calls immediately fail.

2. Deactivate the access key (for IAM users): aws iam update-access-key --user-name USER --access-key-id AKIA... --status Inactive. This invalidates the specific key without affecting the user.

3. Revoke active sessions for IAM roles: Attach an AWSRevokeOlderSessions policy to the role — this adds a Deny condition for any sessions issued before the current time, effectively invalidating all existing sessions. Temporary credentials issued before the policy change are invalid; new assumptions of the role get the new restriction.

4. Check for persistence mechanisms: Did the attacker create new IAM users? New access keys on existing users? New IAM roles with backdoor trust policies? Run aws iam get-credential-report and review all user access key ages. Check CloudTrail for CreateUser, CreateRole, CreateAccessKey events in the last 24 hours.

Why the deny-all inline policy is faster than deleting: Deleting an IAM user or role that's actively being used risks breaking legitimate systems that depend on it. An inline deny-all policy neutralises the attacker while you investigate without disrupting legitimate access (which also gets denied, but that's an acceptable trade-off during active incident response — you can remove the policy after investigation). This is the "break glass in emergency" response and is faster to execute than any other containment option.

Q1 An engineer says "I gave my EC2 AdministratorAccess but it still can't write to this S3 bucket." Walk me through every possible reason and how you'd diagnose each. Hard▼

This is a layered IAM debugging problem. AdministratorAccess grants everything in the identity policy — but the IAM evaluation algorithm has multiple layers that can override it.

1. Explicit Deny somewhere in the chain. Check: is there an explicit Deny for s3:PutObject on the EC2's role? On any policy attached to the role? On the S3 bucket policy? An explicit Deny overrides AdministratorAccess. Run aws iam simulate-principal-policy with the role ARN, action s3:PutObject, and the bucket ARN — the simulator shows which policy statement is the deciding factor.

2. SCP preventing S3 writes. If the account is in an AWS Organization, check SCPs applied to the account or OU. An SCP that says "Deny s3:PutObject except from specific VPCs" overrides the EC2's AdministratorAccess. Check via AWS Organizations console or aws organizations list-policies-for-target. SCPs cannot be viewed from within the target account — you need access to the management account.

3. Permission boundary on the role. Does the EC2's IAM role have a permission boundary attached? If the boundary policy doesn't include s3:PutObject, the role cannot write to S3 regardless of AdministratorAccess. Check: aws iam get-role --role-name ROLE_NAME and look for PermissionsBoundary field.

4. S3 bucket policy explicitly denying the role or its account. The bucket might be in another account (cross-account). Or the bucket policy has an explicit Deny for this role ARN or account. Or the bucket policy has a condition like aws:SourceVpc that's not satisfied (the EC2 is calling S3 via the internet through NAT Gateway, not through a VPC Endpoint). Check the bucket policy directly.

5. S3 Block Public Access restricting cross-account access. If the bucket is in another account and Block Public Access is enabled with RestrictPublicBuckets, cross-account access via bucket policy may be restricted. Usually affects anonymous access, but worth checking if the bucket is in an account you don't control.

6. The role isn't actually attached. Verify the EC2 instance actually has the role attached (not just that the role exists). aws ec2 describe-instances --instance-ids i-xxx --query 'Reservations[0].Instances[0].IamInstanceProfile'. The instance profile must be attached, not just the role created.

Why this answer works: It covers all six IAM evaluation layers in order — explicit deny, SCP, permission boundary, resource policy, cross-account, and the practical "is it actually attached" check. This systematic approach demonstrates that you understand the evaluation algorithm and can apply it to debugging, not just recite it theoretically.
Q2 Explain IAM roles vs IAM users. Why should applications never use IAM users? Medium▼

IAM Users have long-lived credentials — an access key ID and secret that remain valid indefinitely until explicitly rotated or deleted. IAM Roles use temporary credentials issued by STS — they expire automatically (typically in 1 hour), are refreshed automatically by the SDK, and leave no long-term secrets to manage, rotate, or accidentally expose.

Applications running on AWS — whether EC2, Lambda, ECS, or EKS — should never use IAM Users for four reasons. First, temporary credentials cannot be exfiltrated usefully. If an attacker steals a role's temporary credentials from IMDS, they have at most 1 hour before those credentials expire. Stealing a static access key gives the attacker permanent access until you detect and rotate it. Second, there is no secret to store. The EC2 instance profile or Lambda execution role makes credentials available to the SDK through the metadata service — no credentials in code, config files, environment variables, or CI/CD secret stores. Third, automatic rotation happens at the platform level — you don't write rotation logic. Fourth, the principle of least privilege is easier to enforce per-role than per-user when you have many services.

In my Deloitte project, all Spring Boot microservices on EC2 use IAM instance profiles. The Spring Boot app's application.properties has no AWS credentials. The AWS SDK's DefaultCredentialsProvider automatically queries IMDS, gets temporary credentials for the attached role, and refreshes them every hour. The application code is completely credential-agnostic — it works locally (using ~/.aws/credentials or environment variables) and in production (using IMDS) without code changes.

Why this answer works: It explains the credential model difference (long-lived vs temporary) and then gives four concrete technical reasons rather than vague "security best practice" language. The personal project context grounds it in real experience. The DefaultCredentialsProvider detail shows you know how the SDK actually works.
Q3 What is a permission boundary and give a real scenario where you'd use one? Hard▼

A permission boundary is an IAM managed policy attached to an IAM user or role that sets the maximum permissions that entity can ever have, regardless of what identity-based policies say. The effective permission is the intersection of the identity policy and the boundary — the boundary cannot grant permissions, only restrict them. If the boundary allows s3:GetObject but the identity policy allows s3:*, the entity can only do s3:GetObject.

The canonical scenario: you want to give developers the ability to create and manage their own IAM roles for their services — without being able to create roles with more permissions than they themselves have (privilege escalation). Without permission boundaries, a developer with iam:CreateRole and iam:AttachRolePolicy could create a role with AdministratorAccess and assume it — bypassing all access restrictions.

The solution: create a DeveloperBoundary policy that defines the maximum permissions any developer-created role can have. Require developers to attach this boundary when creating roles (enforced via a condition on iam:CreateRole: iam:PermissionsBoundary = arn:...DeveloperBoundary). Also add a Deny for iam:DeleteRolePermissionsBoundary so developers can't remove the boundary. Now developers can create roles freely — but all roles they create are limited to DeveloperBoundary permissions. They cannot create a role that exceeds their own access level.

This is the pattern used in multi-team AWS platforms where a platform team grants development teams self-service IAM capabilities without losing security control. At a company like Deloitte with many project teams in shared AWS accounts, this pattern prevents any one team from accidentally or intentionally creating over-privileged roles.

Why this answer works: The privilege escalation scenario makes the abstract concept concrete. The implementation details (the condition on CreateRole, the Deny on DeleteBoundary) show you know how to actually implement it, not just define it. The organizational context (multi-team platforms) shows you understand when this matters in practice.
Q4 How would you secure an AWS account after discovering it was compromised? Hard▼

This is an incident response question testing whether you have a structured process or would panic and start randomly deleting things.

Immediate containment (first 5 minutes): Identify the compromised entity from GuardDuty findings or CloudTrail. Attach an inline deny-all policy to that entity immediately — takes effect globally in seconds, neutralises the attacker while you investigate. Do not delete the entity yet — you need it for forensics and may need to keep it intact for audit purposes.

Scope assessment (next 30 minutes): Pull the CloudTrail event history for the compromised entity for the last 24-48 hours. Look for: CreateUser, CreateRole, CreateAccessKey (persistence), ListBuckets, GetObject (data exfiltration), RunInstances (cryptocurrency mining), AttachUserPolicy, AttachRolePolicy (privilege escalation). Run aws iam get-credential-report to see all users and their last activity. Check for new IAM users or roles created by the attacker for persistence. Check for EC2 instances launched in all regions — attackers use all regions for mining.

Eradication: Delete any attacker-created resources (IAM users, roles, EC2 instances). Rotate all credentials that could have been observed by the attacker: access keys for any IAM user who used the same environment, RDS passwords if the DB was accessible, any secrets the compromised role could have read via Secrets Manager. Enable MFA delete on S3 buckets if not already enabled. Verify CloudTrail is not disabled or modified.

Recovery and hardening: Enable GuardDuty if not already (should have been on before the incident). Enable Security Hub. Require IMDSv2 on all EC2 instances. Audit all IAM policies for overly broad permissions using IAM Access Analyzer. Apply permission boundaries to all developer roles. Implement SCP denying sts:AssumeRole from outside approved IP ranges if the attack came from external credentials. Add CloudWatch alarms for any IAM changes going forward.

Post-incident: AWS may suspend the account for policy violations (fraudulent charges). Contact AWS Support with the incident details — they may reduce charges and restore the account if you can demonstrate the compromise was external. File an abuse report with any attacker infrastructure details.

Why this answer works: It follows a structured IR process (contain, assess, eradicate, recover) rather than random actions. The specific commands show operational knowledge. The GuardDuty and SCP hardening steps show you learn from incidents rather than just fixing the immediate problem. The AWS Support step is practical knowledge that most engineers don't think about.
Q5 What is the difference between IAM policy evaluation for same-account vs cross-account access? Medium▼

The evaluation logic changes fundamentally when the requesting principal is from a different account than the resource owner.

Same-account: Either the identity policy OR the resource policy can grant access — they are OR'd together. If the IAM role has s3:GetObject in its identity policy and there's no bucket policy, access is granted. If there's a bucket policy allowing this role and no identity policy, access is also granted. You only need one side to allow it, as long as there's no explicit deny.

Cross-account: BOTH the identity policy in the requesting account AND the resource policy in the owning account must explicitly allow the access. This is AND logic. If Account A's IAM role has s3:GetObject in its identity policy but Account B's bucket has no bucket policy (or a bucket policy that doesn't mention Account A), access is denied. Account A's resource owner cannot unilaterally grant access to Account B's resources — Account B's resource owner must explicitly opt in via the resource policy.

The reason for this distinction: the AND requirement ensures mutual consent for cross-account access. Without it, Account A's admin could grant their users access to any resource in Account B simply by writing an allow policy — without Account B's knowledge or consent.

The practical implication: when debugging cross-account access denials, always check both sides. "My IAM policy says allow" is not sufficient — you also need "the resource policy says allow for my account/role." The most common debugging mistake is checking only one side and concluding "everything looks correct" when the other side is missing the explicit allow.

Why this answer works: The OR vs AND logic framing is precise and memorable. The consent rationale explains WHY the rules are different, not just what they are. The debugging implication connects the abstract concept to the real-world consequence every engineer eventually encounters.
Today's lab makes the IAM evaluation algorithm visceral — you'll observe explicit deny overriding allow, observe cross-account logic, observe permission boundaries in action, and set up GitHub Actions OIDC to eliminate static credentials from your CI/CD pipeline.
1
Observe explicit Deny overriding AdministratorAccess
Create an IAM user with AdministratorAccess. Then attach an inline deny policy for s3:ListAllMyBuckets. Observe that the user cannot list buckets even with AdministratorAccess. This makes the evaluation algorithm real.
# Create test user aws iam create-user --user-name iam-lab-user # Give full admin access aws iam attach-user-policy \ --user-name iam-lab-user \ --policy-arn arn:aws:iam::aws:policy/AdministratorAccess # Create access key for testing aws iam create-access-key --user-name iam-lab-user # Test: with AdminAccess, this should work AWS_ACCESS_KEY_ID=NEWKEY AWS_SECRET_ACCESS_KEY=NEWSECRET \ aws s3 ls # Should succeed # Now add an explicit Deny for S3 listing aws iam put-user-policy \ --user-name iam-lab-user \ --policy-name DenyS3List \ --policy-document '{"Statement":[{"Effect":"Deny","Action":"s3:ListAllMyBuckets","Resource":"*"}]}' # Test again: should now fail with Access Denied AWS_ACCESS_KEY_ID=NEWKEY AWS_SECRET_ACCESS_KEY=NEWSECRET \ aws s3 ls # Access Denied — Deny wins over AdminAccess
This is the most important IAM concept to observe directly. Deny always wins. No policy can override an explicit Deny. Clean up: aws iam delete-user-policy --user-name iam-lab-user --policy-name DenyS3List to restore access.
2
Create an IAM role with a trust policy and assume it
Create an IAM role with a trust policy allowing your IAM user to assume it. Observe the AssumeRole call, the temporary credentials (ASIA prefix), and the session token requirement.
# Get your account ID ACCOUNT_ID=$(aws sts get-caller-identity --query Account --output text) # Create trust policy allowing your lab user to assume this role cat > trust-policy.json << EOF { "Version": "2012-10-17", "Statement": [{ "Effect": "Allow", "Principal": { "AWS": "arn:aws:iam::${ACCOUNT_ID}:user/iam-lab-user" }, "Action": "sts:AssumeRole" }] } EOF # Create the role aws iam create-role \ --role-name lab-assumed-role \ --assume-role-policy-document file://trust-policy.json # Attach S3 read policy to the role aws iam attach-role-policy \ --role-name lab-assumed-role \ --policy-arn arn:aws:iam::aws:policy/AmazonS3ReadOnlyAccess # Allow the lab user to call AssumeRole aws iam put-user-policy \ --user-name iam-lab-user \ --policy-name AllowAssumeRole \ --policy-document "{\"Statement\":[{\"Effect\":\"Allow\",\"Action\":\"sts:AssumeRole\",\"Resource\":\"arn:aws:iam::${ACCOUNT_ID}:role/lab-assumed-role\"}]}" # Now assume the role using the lab user's credentials AWS_ACCESS_KEY_ID=NEWKEY AWS_SECRET_ACCESS_KEY=NEWSECRET \ aws sts assume-role \ --role-arn "arn:aws:iam::${ACCOUNT_ID}:role/lab-assumed-role" \ --role-session-name "lab-session" # Observe: AccessKeyId starts with ASIA (temporary), includes SessionToken
The ASIA prefix on the returned AccessKeyId confirms these are temporary credentials from STS. The SessionToken is the third required value. Without it, any API call fails even with valid ASIA keys.
3
Use the IAM Policy Simulator to understand evaluation
Use the IAM Policy Simulator to test what your role can and cannot do. This is the tool for debugging access issues without having to make actual API calls and guess why they fail.
ACCOUNT_ID=$(aws sts get-caller-identity --query Account --output text) # Simulate whether the lab role can do s3:GetObject aws iam simulate-principal-policy \ --policy-source-arn "arn:aws:iam::${ACCOUNT_ID}:role/lab-assumed-role" \ --action-names "s3:GetObject" \ --resource-arns "arn:aws:s3:::any-bucket/*" # Simulate whether it can do s3:DeleteObject (should be denied) aws iam simulate-principal-policy \ --policy-source-arn "arn:aws:iam::${ACCOUNT_ID}:role/lab-assumed-role" \ --action-names "s3:DeleteObject" \ --resource-arns "arn:aws:s3:::any-bucket/*" # Simulate multiple actions at once aws iam simulate-principal-policy \ --policy-source-arn "arn:aws:iam::${ACCOUNT_ID}:role/lab-assumed-role" \ --action-names "s3:GetObject" "s3:PutObject" "s3:DeleteObject" "s3:ListBucket" \ --resource-arns "arn:aws:s3:::any-bucket/*"
The simulator tells you "allowed" or "implicitDeny" or "explicitDeny" for each action. This is your primary tool for debugging IAM before touching real resources. Use it before every IAM policy deployment.
4
Observe the IMDSv2 token requirement
On your Day 2 EC2 instance (start it from stopped state), test IMDSv1 and IMDSv2 access. Then enforce IMDSv2 and observe that IMDSv1 is rejected. This is the Capital One breach prevention step.
# Start your Day 2 EC2 instance if stopped aws ec2 start-instances --instance-ids YOUR_INSTANCE_ID --region ap-south-1 # SSH into the instance, then run: # IMDSv1 — works currently (insecure) curl -s http://169.254.169.254/latest/meta-data/instance-id # IMDSv2 — get session token first, then use it TOKEN=$(curl -s -X PUT \ "http://169.254.169.254/latest/api/token" \ -H "X-aws-ec2-metadata-token-ttl-seconds: 21600") curl -s http://169.254.169.254/latest/meta-data/instance-id \ -H "X-aws-ec2-metadata-token: $TOKEN" # Now enforce IMDSv2 (from your local machine): aws ec2 modify-instance-metadata-options \ --instance-id YOUR_INSTANCE_ID \ --http-tokens required \ --region ap-south-1 # SSH back in and test IMDSv1 — should now fail with 401 curl -s http://169.254.169.254/latest/meta-data/instance-id # Returns 401 Unauthorized — IMDSv1 is disabled # IMDSv2 still works TOKEN=$(curl -s -X PUT "http://169.254.169.254/latest/api/token" \ -H "X-aws-ec2-metadata-token-ttl-seconds: 21600") curl -s http://169.254.169.254/latest/meta-data/instance-id \ -H "X-aws-ec2-metadata-token: $TOKEN"
You've now reproduced the defence that would have prevented the Capital One breach. IMDSv1 returns 401 after enforcing required tokens. IMDSv2 still works. Stop the instance when done.
5
Validate a policy with IAM Access Analyzer
Write a policy with intentional issues — a wildcard Principal, a missing resource ARN, an unused action. Run it through Access Analyzer policy validation and observe the findings. Integrate this into your mental workflow before every policy deployment.
cat > bad-policy.json << 'EOF' { "Version": "2012-10-17", "Statement": [ { "Effect": "Allow", "Action": ["s3:GetObject", "s3:GetObiect"], "Resource": "arn:aws:s3:::my-bucket" }, { "Effect": "Allow", "Action": "s3:ListBucket", "Resource": "arn:aws:s3:::my-bucket/*" } ] } EOF aws accessanalyzer validate-policy \ --policy-document file://bad-policy.json \ --policy-type IDENTITY_POLICY \ --region ap-south-1 # Observe findings: # - s3:GetObiect doesn't exist (typo — silently grants nothing) # - s3:ListBucket on /* is wrong (should be on the bucket, not objects) # These are exactly the silent bugs that break production and take hours to debug
The validator catches the typo (s3:GetObiect) and the wrong resource for ListBucket. Both would be silently ignored at runtime — no error, just behaviour that doesn't match expectations. Run this on every policy before deployment.
6
Clean up lab resources
Clean up the IAM resources created today. IAM users and roles are free, but leaving test users with admin access is a security risk.
ACCOUNT_ID=$(aws sts get-caller-identity --query Account --output text) # Delete inline policies aws iam delete-user-policy --user-name iam-lab-user --policy-name DenyS3List aws iam delete-user-policy --user-name iam-lab-user --policy-name AllowAssumeRole # Detach managed policy aws iam detach-user-policy \ --user-name iam-lab-user \ --policy-arn arn:aws:iam::aws:policy/AdministratorAccess # Delete access key (get the key ID first) aws iam list-access-keys --user-name iam-lab-user aws iam delete-access-key \ --user-name iam-lab-user \ --access-key-id AKIA_YOUR_KEY_ID # Delete user aws iam delete-user --user-name iam-lab-user # Clean up role aws iam detach-role-policy \ --role-name lab-assumed-role \ --policy-arn arn:aws:iam::aws:policy/AmazonS3ReadOnlyAccess aws iam delete-role --role-name lab-assumed-role
Never leave test IAM users with admin access in your account. A compromised test user has the same blast radius as a compromised production user if both have AdministratorAccess.
Day 5 — IAM mechanics understood at depth
Evaluation order
1. Explicit Deny (wins always). 2. SCP. 3. Resource policy allow. 4. Permission boundary. 5. Identity policy allow. 6. Implicit deny (default).
Explicit Deny
Overrides everything including AdministratorAccess. Cannot Allow your way out of a Deny. Use for emergency containment.
Cross-account logic
Same account: OR (either side can grant). Cross-account: AND (both identity policy AND resource policy must allow). Most common IAM debugging trap.
AKIA vs ASIA
AKIA = long-term IAM User key. ASIA = temporary STS credential. ASIA always requires a SessionToken — missing it fails even with valid keys.
Permission boundaries
Sets maximum permissions — intersection with identity policy. Cannot grant beyond the boundary. Used to safely delegate IAM management without privilege escalation.
SCPs
Org-level policy. Cannot grant — can only restrict. Applies to all principals including root. Check when account-level access is denied despite valid IAM policies.
IMDSv2
Requires PUT for session token before GET. Prevents SSRF-based credential theft (Capital One breach vector). Enforce with --http-tokens required on every EC2.
Role assumption
Trust policy = who can assume. Permission policy = what role can do. Both must align. Role chaining caps at 1hr session regardless of role's MaxSessionDuration.
OIDC for CI/CD
GitHub Actions, GitLab CI, and Kubernetes can assume IAM roles via OIDC — no static AKIA keys stored anywhere. Credentials auto-expire per job/pod.
Access Analyzer
Validates policies before deploy. Detects external access. Identifies unused credentials. Run validate-policy on every IAM policy before deployment.
Privilege escalation
17 known paths. All use legitimate APIs. iam:PassRole + ec2:RunInstances is most common. Defend with permission boundaries on all IAM-managing roles.
Incident containment
Attach inline deny-all policy immediately — faster than deleting. Effective globally in seconds. Investigate before eradicating — preserve forensic evidence.
Non-negotiable IAM rules for production
1. Applications on AWS compute (EC2, Lambda, ECS) must use IAM roles — never static access keys.
2. Enforce IMDSv2 (HttpTokens=required) on every EC2 instance.
3. Never commit AWS credentials to source code or config files — use git-secrets pre-commit hook.
4. Run aws accessanalyzer validate-policy on every IAM policy before deployment.
5. Use IAM Access Analyzer to continuously monitor for unintended external access.
6. Replace GitHub Actions static credentials with OIDC role assumption.
7. Apply permission boundaries to all roles that have iam:* permissions.
8. Scope iam:PassRole to specific role ARNs — never Resource: *.
9. Enable GuardDuty in every account — detects compromised credentials in minutes.
10. SCP: deny cloudtrail:StopLogging and cloudtrail:DeleteTrail at org level — attackers can't cover tracks.
