What EC2 actually is: EC2 (Elastic Compute Cloud) is AWS's virtual machine service. You rent a slice of a physical server in Amazon's data centre — choosing the OS, CPU, RAM, and storage — and SSH into it just like any Linux box. Your Spring Boot app runs on EC2 exactly like it would on any Linux server, except you provision it in 60 seconds instead of waiting weeks for hardware.
What is EC2?
Elastic Compute Cloud — rented virtual machines on AWS infrastructure

EC2 is the foundation of most AWS deployments. When you launch an EC2 instance, AWS spins up a virtual machine on physical hardware in the data centre of your chosen AZ. You get root-level access over SSH, install whatever you want, and run any application — exactly like a rented VPS (Virtual Private Server), but with far more flexibility and integrations.

The "Elastic" in EC2 means two things: you can resize it (change instance type) and you can add or remove instances programmatically. You start with one t2.micro for development, and your Auto Scaling Group adds ten m5.large instances during peak traffic — then removes them when load drops. You pay per second of running time.

Your Spring Boot app on EC2: You SSH into the instance, install Java 17 with sudo yum install java-17, upload your JAR via SCP or CodeDeploy, and run it with java -jar app.jar. From there, it behaves exactly like running on your Deloitte office server — except AWS handles the physical hardware, power, cooling, and network.

EC2 instances live inside a VPC (Virtual Private Cloud — Day 4 topic). They can have a public IP (reachable from the internet) or private IP only (reachable only within the VPC). A Security Group acts as a firewall controlling which traffic reaches the instance.

IaaS — you manage the OS upwards Pay per second of running time
AMI — Amazon Machine Image
Your server's blueprint — OS + pre-installed software snapshot

An AMI is a template that defines what your EC2 instance starts with. It contains the operating system, any pre-installed software, configuration, and storage volume snapshots. When you launch an EC2 instance, you always pick an AMI — it's the first thing the wizard asks you.

AWS provides hundreds of public AMIs: Amazon Linux 2023, Ubuntu 22.04 LTS, Windows Server, Red Hat, etc. The AWS Marketplace offers pre-configured AMIs with software like Jenkins, Nginx, or Java already installed. And you can create your own custom AMI from a running instance — capturing everything you've installed and configured.

Why custom AMIs matter for you: Imagine you've set up an EC2 with Java 17, your app's environment variables, and your Spring Boot JAR. Instead of repeating this setup every time you launch a new instance, create an AMI from it. Now every new instance in your Auto Scaling Group launches pre-configured in 60 seconds — no setup scripts needed. This is exactly how production deployments scale.

AMI IDs are region-specific. The same Amazon Linux 2023 AMI has a different ID in ap-south-1 vs us-east-1. When scripting deployments with Terraform (which you already know), always reference the correct region's AMI ID.

Custom AMI = your golden server image AMI IDs are region-specific
Key pairs — SSH access to your instance
Public/private key authentication — the only way in

EC2 instances don't have passwords by default. Access is via SSH key pairs. AWS stores the public key on the instance; you keep the private key (.pem file) on your machine. When you SSH in, your private key proves your identity without transmitting a password.

You create a key pair in the EC2 console before launching an instance, or import your existing SSH public key. AWS injects the public key into the instance's ~/.ssh/authorized_keys file at launch. The private .pem file is downloaded once — if you lose it, you cannot SSH into the instance again.

How to SSH into your EC2 instance:
chmod 400 my-key.pem — fix permissions (SSH rejects world-readable keys)
ssh -i my-key.pem ec2-user@<public-ip> — connect (Amazon Linux uses ec2-user, Ubuntu uses ubuntu)
The default username depends on the AMI: Amazon Linux = ec2-user, Ubuntu = ubuntu, RHEL = ec2-user

In production, you often don't SSH directly — you use AWS Systems Manager Session Manager instead (no open port 22, no key pair needed). But for learning, key pair SSH is the standard approach and something every interviewer will ask about.

Never lose your .pem file chmod 400 before every SSH
Security Groups
EC2's virtual firewall — stateful, inbound + outbound rules

A Security Group is a virtual firewall attached to your EC2 instance (and other resources like RDS, ELB). It controls inbound and outbound traffic using rules. Each rule specifies: protocol (TCP/UDP/ICMP), port range, and source/destination (IP CIDR or another Security Group).

Security Groups are stateful: if you allow inbound TCP port 8080, the corresponding outbound response is automatically allowed — you don't need a separate outbound rule for the response traffic. This is different from Network ACLs (Day 4) which are stateless.

Typical Security Group for a Spring Boot API server:
Inbound: port 22 (SSH) from your IP only — 115.246.x.x/32
Inbound: port 8080 (Spring Boot) from the ALB's Security Group
Inbound: port 443 (HTTPS) from 0.0.0.0/0 (all internet)
Outbound: all traffic to 0.0.0.0/0 (allow all outbound — this is the default)
Never open port 22 to 0.0.0.0/0 in production. That's inviting brute-force attacks.

Security Groups reference each other — your RDS instance's Security Group can allow inbound port 3306 only from your EC2's Security Group ID. This means only your EC2 can reach your database, not the public internet. This is one of the most important security patterns in AWS.

Stateful firewall — response traffic auto-allowed SGs reference each other — key security pattern
Elastic IP
A static public IP address that survives instance stop/start

By default, an EC2 instance gets a new public IP every time it starts. If you stop and start your instance overnight, its public IP changes — breaking any DNS records, Postman collections, or firewall rules pointing to the old IP. An Elastic IP (EIP) solves this: it's a static public IP you allocate from AWS and associate with your instance. The IP stays the same even if you stop/start or even replace the instance.

EIPs are free as long as they're attached to a running instance. If you allocate an EIP but don't attach it, AWS charges a small fee (~$0.005/hr) — this discourages hoarding IPs. When you're done with a lab, always release your EIP to avoid surprise charges.

When do you actually need an EIP? In production, you usually put a Load Balancer in front of EC2 — the ALB has its own DNS name (not an IP), so EIPs become unnecessary for web traffic. EIPs are most useful for: a dev/test EC2 you SSH into regularly, a NAT Gateway (Day 4), or a legacy system that requires a fixed IP for third-party firewall whitelisting.

For your GetWork app or personal projects: allocate an EIP to your dev EC2 on Day 2 so your SSH alias works every session without updating the IP.

Free when attached to running instance Release when not in use — or get charged

Instance shuts down — like turning off a PC.

EBS root volume is preserved. Your OS, files, app, everything on disk survives.

RAM (in-memory state) is lost.

Gets a new public IP on restart (unless EIP).

Can be restarted. Safe for dev instances you don't use overnight.

Instance is deleted permanently.

Real-world horror story pattern: Engineer stops an instance thinking it's safe. Two weeks later, terminates it thinking "it's already off, nothing to lose." The EBS volume with 6 months of app logs and config gets deleted. No snapshot. This is why good engineers always: (1) create a snapshot before any potentially destructive action, (2) set up automated EBS snapshots via AWS Backup, and (3) separate persistent data onto a separate EBS volume with DeleteOnTermination=false.
EC2 is asked in every AWS interview. Questions range from basic definition to deep operational questions about networking, storage, and failure scenarios. Every answer below ties back to your Deloitte/JP Morgan/GetWork experience.
Q1 What is EC2 and how does it differ from a traditional server? Always asked Easy▼

EC2 (Elastic Compute Cloud) is AWS's virtual machine service. It provides resizable compute capacity — you pick an OS, CPU, RAM, and storage configuration and get a fully functional Linux or Windows server running in a specific AWS Availability Zone within minutes. You have root-level SSH access and can install any software.

Key differences from traditional servers: Speed — a new server in 60 seconds vs weeks for physical hardware procurement. Elasticity — resize from 1 vCPU to 96 vCPUs with a stop/start. Add 10 instances in minutes with Auto Scaling, remove them when load drops. Billing — pay per second of running time; stop an instance and billing stops. No maintenance — AWS handles hardware failures; if a physical host degrades, AWS migrates your instance to healthy hardware. You do still manage the OS and everything above it.

In my current Deloitte project, our Spring Boot microservices run on EC2 instances within a VPC, fronted by an Application Load Balancer. We use EC2 over Lambda because our services are long-running processes with persistent TCP connections to Kafka — not suitable for the stateless, short-lived Lambda execution model.

Tip: The last paragraph is gold — explaining WHY you chose EC2 over Lambda shows architectural thinking. Interviewers love candidates who can articulate trade-offs, not just define services.
Q2 What is an AMI and why would you create a custom one? Medium▼

An AMI (Amazon Machine Image) is a snapshot-based template that defines the starting state of an EC2 instance. It contains the OS, pre-installed software, configuration files, and the root EBS volume snapshot. Every EC2 launch requires choosing an AMI — AWS provides hundreds of public ones (Amazon Linux, Ubuntu, Windows), and you can also create your own.

You'd create a custom AMI when you want a "golden image" — a pre-configured server that every new instance in your fleet starts from. Example: you configure an EC2 with Java 17, your Spring Boot JAR, your Prometheus agent, your CloudWatch agent, and all environment variables. You create an AMI from it. Now your Auto Scaling Group launches new instances from this AMI — fully configured in 60 seconds, no bootstrap scripts needed. When you update your app, you bake a new AMI and roll it out.

This is the "immutable infrastructure" pattern — instead of patching running servers, you build a new AMI and replace instances. Netflix pioneered this. It eliminates configuration drift (instances diverging from their intended state over time) and makes rollbacks trivial — just point the ASG at the previous AMI.

Tip: Mentioning "immutable infrastructure" and "configuration drift" signals you think like a senior engineer or DevOps practitioner. These are concepts that appear in Deloitte-level consulting discussions about cloud architecture.
Q3 What is the difference between stopping and terminating an EC2 instance? Tricky Medium▼

Stop is like shutting down a PC — the instance powers off, EBS root volume is preserved, RAM state is lost, and you are not billed for compute time while stopped. You still pay for EBS storage. The instance retains its instance ID, IAM role, security groups, and private IP address. It gets a new public IP when restarted unless you've assigned an Elastic IP. It can be restarted at any time.

Terminate is permanent deletion. The instance is destroyed. By default, the root EBS volume is also deleted (controlled by the DeleteOnTermination attribute, which defaults to true). Any additional EBS volumes you've attached are preserved by default (their own DeleteOnTermination is false by default). The instance cannot be recovered — it disappears from your console within minutes. Billing stops immediately upon termination.

The practical implication: stop your dev instance at night to save compute costs, but your EBS data survives. Terminate at the end of a project when you've confirmed nothing valuable remains. Before terminating any instance in production, always: create an EBS snapshot, verify the AMI is up to date, and confirm no unreplicable data exists on instance storage (ephemeral storage is always deleted on stop/terminate).

Tip: The instance store vs EBS distinction is a follow-up trap. Instance store (ephemeral storage) is physically attached to the host — data is lost on stop, hibernate, OR terminate. EBS survives stop. Knowing this earns points in senior-level interviews.
Q4 What is a Security Group and how is it different from a Network ACL? Medium▼

A Security Group is a virtual firewall at the instance level. It controls inbound and outbound traffic using allow rules only (you cannot write deny rules). Security Groups are stateful — if an inbound connection is allowed, the response traffic is automatically allowed outbound without a separate rule. Security Groups are associated with EC2 instances, RDS instances, and load balancers. A single instance can have multiple Security Groups.

A Network ACL (NACL) is a firewall at the subnet level — it applies to all resources in a subnet. NACLs are stateless — you must explicitly allow both inbound AND outbound for every connection, including the response. NACLs support both allow and deny rules, evaluated in order by rule number.

Key differences: Security Groups operate at instance level (fine-grained), NACLs at subnet level (broad). SGs are stateful (easier to manage), NACLs are stateless (more complex). SGs allow-only, NACLs allow + deny. In practice, most security is implemented via Security Groups. NACLs are used for broader subnet-level controls — blocking a range of IPs, or as a secondary defence layer. For your Spring Boot EC2: you always configure a Security Group. NACLs are usually left at default unless you have specific subnet-level requirements.

Tip: The stateful vs stateless distinction is the key thing interviewers want to hear. Draw it out: "If port 8080 is allowed inbound in a SG, the TCP ACK back to the client is automatically allowed — I don't need an outbound rule. With a NACL, I need both an inbound rule for port 8080 AND an outbound rule for ephemeral ports 1024-65535 for the response." That level of detail is impressive.
Q5 What are the different EC2 purchasing options and when would you use each? Hard▼

On-Demand — pay per second, no commitment. Most expensive per hour but maximum flexibility. Use for: unpredictable workloads, development/testing, short-lived jobs. Default choice when you're unsure.

Reserved Instances (RI) — 1 or 3-year commitment for a specific instance type and region. Up to 72% cheaper than on-demand. Standard RIs are locked to instance type; Convertible RIs allow changing instance family. Use for: steady-state production workloads — your always-on Spring Boot API servers. At JP Morgan scale, RIs save millions annually on compute.

Spot Instances — bid on unused EC2 capacity. Up to 90% cheaper than on-demand. AWS can reclaim with 2-minute warning when capacity is needed elsewhere. Use for: fault-tolerant batch jobs, data processing, CI/CD build agents, stateless workers. Never for primary database servers or stateful applications. AWS Spot Fleet manages a pool of Spot instances, automatically replacing reclaimed ones.

Savings Plans — flexible discount model. Commit to a dollar amount per hour (e.g. $10/hr) for 1 or 3 years. Applies across any instance type, region, and even Lambda/Fargate. More flexible than RIs. Preferred by AWS now over Reserved Instances for most use cases.

Dedicated Hosts — physical server entirely dedicated to you. Required for some software licences tied to physical cores (Oracle, Windows Server). Most expensive. JP Morgan and other regulated institutions sometimes use these for compliance/isolation requirements.

Tip: A complete answer covers at least on-demand + reserved + spot and when to use each. Mentioning Savings Plans shows you keep up with AWS updates (introduced 2019). Mentioning Dedicated Hosts in the context of compliance is a senior-level touch that resonates in fintech interviews.
Q6 How would you deploy a Spring Boot application on EC2 from scratch? Medium▼

Full end-to-end deployment of a Spring Boot app on EC2 involves these steps:

1. Launch instance: Choose Amazon Linux 2023 AMI, t3.small instance type, place in a public subnet with a Security Group allowing port 22 (SSH from your IP) and port 8080 (app traffic from ALB or all internet for dev). Create or select a key pair.

2. Connect and configure: ssh -i key.pem ec2-user@<public-ip>. Install Java: sudo dnf install java-17-amazon-corretto -y. Verify: java -version.

3. Deploy the JAR: From your local machine: scp -i key.pem target/app.jar ec2-user@<ip>:~/. Back on EC2: java -jar app.jar --server.port=8080 &. The & runs it in background.

4. Make it a service: Create a systemd unit file so the app restarts on reboot and on crash. This is production-grade. sudo vim /etc/systemd/system/myapp.service with ExecStart pointing to the JAR.

5. Verify: curl http://localhost:8080/actuator/health — should return UP. Then from your browser: http://<ec2-public-ip>:8080/actuator/health.

For production, this manual process would be replaced by CodeDeploy (automating the SCP + service restart) or a container deployed via ECS (Day 12).

Tip: Mentioning systemd service configuration is what separates a senior from junior answer. Anyone can run java -jar &. Knowing it needs to be a managed service that restarts on failure shows production experience.
Q7 What is an Elastic IP and when do you need one? Easy▼

An Elastic IP is a static, public IPv4 address you allocate from AWS and assign to an EC2 instance. Unlike a default public IP (which changes every time the instance stops and starts), an Elastic IP remains the same — it stays associated with your account until you explicitly release it.

You need an EIP when: your EC2's IP must remain stable across stop/restart cycles (for DNS records, firewall whitelisting, or SSH configuration); when you need to quickly remap traffic from a failed instance to a standby by moving the EIP; or when a NAT Gateway needs a fixed outbound IP (for SaaS vendors that whitelist by IP).

When you don't need an EIP: if your EC2 is behind a Load Balancer (the ALB has a DNS hostname, not an IP — EIP irrelevant). If you're using Route 53 with health checks and automatic failover. Most modern production architectures use ALB + Route 53 and don't use EIPs on individual EC2 instances at all.

Cost note: EIPs are free when attached to a running instance. AWS charges ~$0.005/hr for an EIP that's allocated but not attached — to discourage hoarding public IPs. Always release EIPs after lab sessions.

Tip: The "when you don't need it" part of the answer shows maturity. Knowing that ALB+Route53 makes EIPs irrelevant for web traffic is a real-world insight that junior engineers often miss.
Q8 What is user data in EC2 and how does it work? Medium▼

User data is a script that EC2 runs automatically on an instance's first boot. You provide it at launch time in the "Advanced details" section of the EC2 wizard, or in your Terraform/CloudFormation config. It runs as root, once, during the very first start-up — not on subsequent reboots (by default).

User data is a bash script (on Linux) or PowerShell (on Windows) that bootstraps the instance: installing packages, configuring the OS, downloading and starting your application. This is how you automate the manual steps from Q6 — the SSH + install + deploy flow.

Example user data for a Spring Boot server: update packages, install Java 17, download your JAR from S3, create a systemd service, and start it. Every time Auto Scaling launches a new instance from your AMI, this script runs and your app is live within 2–3 minutes — no manual intervention.

Important: user data runs as root. Failures in user data scripts are silent unless you check the logs at /var/log/cloud-init-output.log. If your instance launches but your app isn't running, that's the first place to look. This is a real debugging skill interviewers sometimes probe with "my EC2 launched but my app isn't running — how do you debug it?"

Tip: Knowing the log path /var/log/cloud-init-output.log for debugging user data is a concrete detail that signals hands-on experience. Also mention that you can force user data to run again on reboot by adding --reboot to cloud-init — sometimes asked in advanced interviews.
Q9 EC2 vs Lambda — how do you decide which to use for a backend service?Senior Hard▼

This is an architectural trade-off question with no single right answer — the correct answer depends on the workload characteristics.

Use EC2 when: the workload is long-running (> 15 minutes per request — Lambda's hard limit). The service maintains persistent connections — WebSockets, persistent Kafka consumers, database connection pools. The app needs custom OS-level configuration. You have predictable, steady-state traffic where Reserved Instance pricing makes EC2 cheaper. The team is more comfortable with traditional server operations.

Use Lambda when: the workload is event-driven and short-lived — process an S3 file upload, respond to an API request, handle a queue message. Traffic is spiky or unpredictable — Lambda scales from 0 to thousands of concurrent executions automatically. You want zero server management — no OS patching, no capacity planning. Cost model favours it: Lambda's free tier is 1M requests/month — a lightly used API costs essentially nothing.

For my current JP Morgan project: we use EC2 for our Spring Boot microservices because they run persistent Kafka consumers with connection pools, require WebFlux reactive streams, and have steady predictable load — making Reserved Instances cost-effective. We use Lambda for lightweight event-driven tasks like processing Bitbucket webhook events and triggering Jira ticket updates — these are infrequent, short-lived, and perfect for serverless.

Tip: This is a favourite senior/architect-level question. The key is knowing the 15-minute Lambda limit, the connection pool issue (Lambda is stateless — no persistent DB connection), and the cost break-even point. Your JP Morgan Kafka context is a perfect real-world justification for EC2 — use it every time.
Today's lab goal: Launch your first real EC2 instance in ap-south-1, SSH into it, install Java 17, and run a Spring Boot JAR on it. By the end, you'll have a live application accessible from your browser via the instance's public IP.
1
Launch a t2.micro EC2 instance
Go to EC2 console → Launch instance. Name it "day2-lab". Choose Amazon Linux 2023 AMI (free tier eligible). Instance type: t2.micro (free tier). Create a new key pair — name it "day2-key", RSA, .pem format. Download it. Network settings: allow SSH from My IP, allow HTTP (port 80) and Custom TCP port 8080 from anywhere (for today's lab). Storage: 8GB gp3 (default). Launch.
EC2 Console → Launch instance Name: day2-lab AMI: Amazon Linux 2023 Type: t2.micro (free tier) Key pair: create new → day2-key.pem (download it) SG: SSH from My IP, port 8080 from 0.0.0.0/0 Storage: 8GB gp3 → Launch instance
Wait ~1 minute for the instance to reach "Running" state and pass status checks before trying to SSH.
2
SSH into the instance
Go to your EC2 instance in the console, note the Public IPv4 address. Open a terminal. Navigate to where you downloaded day2-key.pem. Fix the key permissions (SSH will refuse to connect otherwise). Then connect.
chmod 400 ~/Downloads/day2-key.pem ssh -i ~/Downloads/day2-key.pem ec2-user@<PUBLIC-IP> # You should see: # Amazon Linux 2023 # [ec2-user@ip-172-xx-xx-xx ~]$
If you see "Permission denied (publickey)" — your key permissions are wrong. Run chmod 400 again. If "Connection timed out" — the Security Group doesn't allow port 22 from your IP.
3
Install Java 17 on the instance
Once inside the EC2, update the package manager and install Amazon Corretto 17 (AWS's free, production-grade OpenJDK distribution — the same one you'd use in production at Deloitte).
sudo dnf update -y sudo dnf install java-17-amazon-corretto -y java -version # Expected: openjdk version "17.x.x"
Amazon Corretto is what AWS recommends for Java on EC2. It's fully compatible with OpenJDK and used heavily in enterprise deployments.
4
Upload and run a Spring Boot JAR
From your local machine, SCP your Spring Boot JAR (use your Task Manager or any existing project — or create a quick demo). Then back on EC2, run it on port 8080.
# On LOCAL machine — upload the JAR: scp -i ~/Downloads/day2-key.pem \ target/myapp.jar \ ec2-user@<PUBLIC-IP>:~/ # On EC2 — run the app: java -jar ~/myapp.jar --server.port=8080 & # Verify it's running: curl http://localhost:8080/actuator/health
If you don't have a JAR ready, create a minimal Spring Boot app with Spring Initializr (start.spring.io) — just add Spring Web and Actuator, build it locally with mvn package, and upload the JAR.
5
Test from your browser
Open your browser and hit the EC2's public IP on port 8080. You should see your Spring Boot app responding from an actual AWS server in Mumbai.
http://<EC2-PUBLIC-IP>:8080/actuator/health # Should return: {"status":"UP"}
This is your first live app on AWS. Screenshot this — it's worth saving. The instance is running in ap-south-1 (Mumbai), serving requests from Amazon's data centre.
6
Make the app a systemd service (production practice)
Running the JAR with & means it dies if you close the SSH session and won't restart on reboot. Make it a proper Linux service with systemd — this is how production Spring Boot deployments work on EC2.
sudo tee /etc/systemd/system/myapp.service <<EOF [Unit] Description=My Spring Boot App After=network.target [Service] User=ec2-user ExecStart=/usr/bin/java -jar /home/ec2-user/myapp.jar SuccessExitStatus=143 Restart=always RestartSec=10 [Install] WantedBy=multi-user.target EOF sudo systemctl daemon-reload sudo systemctl enable myapp sudo systemctl start myapp sudo systemctl status myapp
Now the app starts automatically on reboot and restarts if it crashes. This is production-grade EC2 deployment without containers.
7
Stop the instance (do NOT terminate — we'll use it tomorrow)
When you're done, stop the instance (not terminate) to avoid compute charges overnight. The EBS volume and your app setup survive. Tomorrow's labs may reference this instance.
# Option 1: From console EC2 → Instances → select day2-lab → Instance state → Stop # Option 2: From CLI aws ec2 stop-instances \ --instance-ids <your-instance-id> \ --region ap-south-1
Stopped instances don't incur compute charges. You still pay ~$0.0008/hr for the 8GB EBS volume — less than a rupee per day. Safe to leave for 30 days.

Bootstrap script on first boot. Runs as root once. Debug: /var/log/cloud-init-output.log. Powers auto-scaling.
On-demand vs RI vs Spot
On-demand = flexible, most expensive. RI = 1-3yr commit, 72% cheaper. Spot = 90% cheaper, can be reclaimed.
Instance store
Ephemeral physical storage. Lost on stop/terminate. Not EBS. Only for temp files / scratch space.
systemd service
Run Spring Boot as a Linux service. Survives reboot, auto-restarts on crash. Production-grade EC2 deployment.
