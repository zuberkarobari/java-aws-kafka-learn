Containerising your Spring Boot application is not just packaging it differently. It fundamentally changes how the application is deployed, scaled, and operated. The Dockerfile you write, the base image you choose, the image layering strategy you use, and the runtime configuration model you adopt all determine whether your containers are efficient, secure, and observable — or slow to build, large, and opaque.
Docker fundamentals you must understand to use ECS correctly
Layers, image registries, container runtime, and what changes when you move from EC2 to containers

A Docker container is a process running in an isolated namespace on a shared host kernel — not a virtual machine. The container has its own filesystem (from the image), its own network namespace (its own IP address and port space), and its own process namespace (PID 1 in the container is not PID 1 on the host). The host kernel is shared — containers on the same host use the same kernel version. This is the fundamental difference from EC2 VMs (which have their own virtualised kernel via the Nitro hypervisor).

Docker images and layers: A Docker image is a stack of read-only filesystem layers. Each instruction in a Dockerfile creates a new layer. When a container starts, Docker adds a thin writable layer on top. Layers are content-addressed (identified by SHA256 hash of their contents) and are cached — if layer N hasn't changed, Docker reuses it. This is why layer ordering in your Dockerfile matters enormously for build performance.

What changes when you move from EC2 to containers: On EC2, your Spring Boot JAR runs as a systemd service directly on the host OS. On ECS, your Spring Boot JAR runs inside a container that is started by the ECS agent (on EC2) or by AWS Fargate's managed infrastructure. The application code is identical. What changes: the deployment unit is a container image rather than a JAR file, the configuration model shifts from EC2 user data to task definition environment variables and secrets, log collection goes from CloudWatch Agent to container log drivers, scaling is service-level (ECS service) rather than ASG-level, and health checks are defined in the task definition rather than ALB configuration files. The networking model changes most significantly — containers have their own IP addresses, service-to-service communication happens over the container network, and security groups attach to ENIs at the task level (in Fargate's awsvpc networking mode).

What stays the same: The Spring Boot application code. The RDS MySQL database. The S3 integration. The Secrets Manager credential fetching pattern. The ALB with target groups. The IAM role for AWS access. The VPC networking. The fundamental security model. Containerisation changes the runtime and deployment mechanism — it does not change the application architecture or the AWS services it integrates with.

Why the pom.xml-first copy pattern matters for build time: Docker layer caching is based on whether the instruction and its input have changed. If you COPY the entire project directory first, any source code change (even a comment) invalidates the dependency download layer, and mvn dependency:go-offline re-downloads all 200+ dependencies on every build — taking 2-3 minutes per build in CI. By copying only pom.xml first and running dependency:go-offline as its own layer, the cache hit on this layer is high: dependencies only re-download when pom.xml changes (adding/updating dependencies). Typical CI build time: 5-8 minutes without this optimisation, 90 seconds with it (cache hit on the 2GB dependency layer).
UseContainerSupport — the JVM flag that prevents OOM kills: Before Java 10, the JVM read the host's total memory to size the heap, ignoring cgroup (container) limits. A container limited to 512MB running on a 16GB host would try to allocate an 8GB heap — immediately OOM-killed by the kernel. UseContainerSupport (default since Java 10 with Amazon Corretto 8u191+) makes the JVM read cgroup memory limits. MaxRAMPercentage=75.0 means the heap uses 75% of the container's memory limit. A 512MB container gets a 384MB heap. A 2GB container gets a 1.5GB heap. The same Docker image works correctly on any container size — you control the heap by setting the container's memory limit in the task definition, not by hardcoding JVM flags.
Image layering strategy — small images, fast builds, efficient registry storage
Layer ordering, .dockerignore, and the Spring Boot layered JAR that makes container deployments fast

The layer caching principle: Docker caches each layer by the hash of the layer's inputs (the Dockerfile instruction + the files it copies). When any input changes, that layer and all subsequent layers are invalidated. Ordering your Dockerfile instructions from "changes least" to "changes most" maximises cache hits.

Spring Boot layered JAR — why fat JAR in a container is wrong: A traditional Spring Boot fat JAR packages all dependencies (100-200MB of library JARs) + your application classes (a few MB) into a single 100-200MB JAR file. In Docker, this entire JAR is a single layer. Every time you change one line of application code, the entire 200MB layer is pushed to ECR and pulled from ECR on every new container deployment. With the Spring Boot layered JAR approach (enabled by spring-boot:build-image or the Dockerfile above): the 200MB of library dependencies is a separate layer that rarely changes. Your 2MB application code is the top layer that changes on every deployment. ECR push is 2MB instead of 200MB. Container start pulls 2MB of new layers (rest from cache). Deployment time drops from 3-4 minutes to 20-30 seconds.

# .dockerignore — exclude files that trigger unnecessary layer invalidation .git .gitignore target/ # Maven build output — rebuilt inside Docker anyway *.md *.txt .idea/ *.iml .mvn/wrapper/maven-wrapper.jar Dockerfile* docker-compose* # Don't include test files — they're not in the production image src/test/
Why .dockerignore is not optional: Without .dockerignore, COPY . . copies everything including the .git directory (potentially hundreds of MB), target/ (previous build outputs), IDE files, and local configuration. The COPY instruction hashes the entire source tree to determine cache validity — if .git changes (every commit changes the git index), the cache is invalidated on every commit regardless of whether source code changed. With .dockerignore: only relevant source files are considered for cache invalidation. Build times are consistent. The .git directory alone can invalidate every build because git metadata changes on every operation.

Alpine vs non-Alpine base images: amazoncorretto:17-alpine produces images ~200MB. amazoncorretto:17 (Amazon Linux based) produces images ~500MB. Alpine uses musl libc instead of glibc — this can cause issues with some native libraries that link against glibc. For Spring Boot applications using pure Java (no JNI, no native libraries), Alpine is safe and significantly smaller. For applications using native libraries (some Kafka clients, certain cryptography libraries), use the non-Alpine base to avoid subtle runtime failures.

Container security — hardening your image and runtime
Non-root user, read-only filesystem, no new privileges — the container security baseline

Container security has multiple layers. The Dockerfile-level controls (covered above: non-root user, minimal base image) are the first layer. The runtime controls are equally important.

Non-root USER in Dockerfile (already in the production Dockerfile above): By default, containers run as root (UID 0) inside the container. Even though container isolation prevents root in the container from being root on the host (with standard runtime configuration), running as root inside is a bad practice. If an attacker exploits your Spring Boot application and escapes to the container namespace, they have root in the container. Running as a non-root user limits blast radius. ECS supports requiring non-root via task definition settings.

Read-only root filesystem: In the ECS task definition, set readonlyRootFilesystem: true. This makes the container's filesystem read-only except for explicitly defined volume mounts. If an attacker compromises the container, they cannot write malicious files to the filesystem, cannot install tools, and cannot modify application binaries in-place. Spring Boot needs write access to: /tmp (for temporary files), the working directory for log files (if writing to filesystem — prefer logging to stdout). Mount these as volumes even with read-only root filesystem.

# Task definition security configuration (in the containerDefinitions section) { "user": "1000:1000", // Non-root UID:GID matching the Dockerfile USER "readonlyRootFilesystem": true, "privileged": false, // Never true — gives container full host access "linuxParameters": { "capabilities": { "drop": ["ALL"], // Drop ALL Linux capabilities "add": [] // Add back only what's needed (usually nothing for Spring Boot) }, "initProcessEnabled": true // ECS init process manages zombie PID cleanup }, "mountPoints": [ {"sourceVolume": "tmp", "containerPath": "/tmp", "readOnly": false} ] }
The Spring Boot app that fails with readonlyRootFilesystem=true: Common failure modes when enabling read-only root filesystem: Tomcat tries to write to its working directory (set server.tomcat.basedir=/tmp/tomcat), Spring writes temporary files during startup (set java.io.tmpdir to /tmp in JAVA_OPTS), Hibernate generates proxy class files (configure Hibernate to use in-memory class generation), Log4j2/Logback writes to a log file path (configure logging output to stdout/stderr only — ECS captures stdout). Test with read-only filesystem in your local Docker environment before deploying to ECS: docker run --read-only --tmpfs /tmp getwork-api:latest
ECS (Elastic Container Service) is AWS's container orchestration platform. It schedules containers onto infrastructure, manages their lifecycle, integrates them with ALB, handles service discovery, and provides the control plane for rolling deployments. Understanding how ECS makes scheduling decisions, how the ECS agent works on EC2 launch type, and how Fargate abstracts away the infrastructure layer is what makes ECS deployments predictable rather than mysterious.
ECS architecture — clusters, services, tasks, and the control plane
The four-level hierarchy and how each level relates to the others

ECS has four levels of abstraction, each with distinct responsibilities:

Cluster — the logical grouping of infrastructure. A cluster represents a pool of compute capacity (EC2 instances or Fargate capacity) and the namespace within which services and tasks run. One cluster can contain multiple services. You can have separate clusters for prod, staging, and dev — or one cluster for all environments with service-level isolation. Clusters have no inherent cost — they are logical constructs. Cost comes from the EC2 instances or Fargate tasks running within them.

Task Definition — the blueprint for how to run a container (or group of containers). Specifies: which Docker image to pull, how much CPU and memory to allocate, which ports to expose, environment variables and secrets, IAM execution role, logging configuration, health check, network mode, and storage volumes. A task definition is versioned — each update creates a new revision. Think of it as the equivalent of a Launch Template but for containers.

Task — a running instance of a task definition. A task is one or more containers running together on the same host, sharing network namespace and storage volumes. Tasks are ephemeral — they start, run, and stop. A task that fails is terminated; ECS does not automatically restart individual failed tasks (services do that).

Service — the long-running abstraction that maintains a desired count of tasks from a task definition. A service says "always have 3 running instances of the v47 task definition." If a task fails or is stopped, the service launches a replacement. Services integrate with the ALB (register/deregister tasks from target groups as they start and stop), manage rolling deployments (replace tasks one by one during a deployment), and support auto-scaling (scale the number of tasks based on CloudWatch metrics).

The task definition is not the same as the service — a critical distinction: A task definition can be used by multiple services. Your getwork-api task definition might power the getwork-api-service (the main API), a separate getwork-batch-service (for batch processing jobs), and one-off tasks (database migration runs). The task definition defines what runs; the service defines how many, how long, and with what scaling. When you deploy a new version, you update the task definition (new revision with new image tag), then update the service to use the new task definition revision — this triggers ECS's rolling deployment.
EC2 launch type — the ECS agent and how tasks land on instances
What runs on each EC2 instance in an ECS cluster and how the scheduler decides where to place tasks

In the EC2 launch type, you manage a pool of EC2 instances (an Auto Scaling Group) that serve as the underlying compute for ECS tasks. Every EC2 instance in an ECS cluster runs the ECS agent — a Docker container that runs alongside your application containers, communicating with the ECS control plane. The ECS agent reports available capacity (CPU, memory, ports) and receives scheduling decisions from ECS.

Task placement: When ECS places a task on a cluster's EC2 instances, it evaluates placement constraints (must be on a t3 instance, must be in AZ ap-south-1a) and placement strategies (spread across AZs for HA, or bin-pack onto fewest instances for cost efficiency). Placement is only relevant for EC2 launch type — Fargate handles its own placement transparently.

Container instance capacity: Each EC2 instance registers its available CPU and memory with ECS. ECS reserves CPU units and memory from this pool for each task. A t3.medium (2 vCPUs = 2048 CPU units, 4GB RAM) can host multiple tasks as long as their combined resource reservations don't exceed the instance's capacity. If a new task can't fit on any existing instance, ECS triggers an ASG scale-out event (via capacity providers) to add more EC2 instances.

ECS CPU units — the non-obvious unit: ECS measures CPU in "CPU units" where 1 vCPU = 1024 CPU units. A task definition reserving 512 CPU units gets half a vCPU guaranteed. CPU reservation is a soft limit in Docker — the container can use more CPU than reserved if the host isn't busy, but is throttled to the reserved amount when the host is under load. Memory is a hard limit — a container exceeding its memory reservation is killed (OOMKilled). Set CPU reservation low enough to allow flexible scheduling but high enough to guarantee adequate performance. Set memory hard limit conservatively — include headroom for Spring Boot's off-heap memory (Metaspace, thread stacks, direct buffers) beyond the heap.
The OOMKill that looks like a JVM crash: A Spring Boot container is configured with 512MB memory limit. JAVA_OPTS sets MaxRAMPercentage=75.0 → heap = 384MB. At load, the JVM also uses: Metaspace (~100MB for Spring Boot's class loading), thread stacks (~1MB per thread × 200 threads = 200MB), native memory for Netty/NIO buffers. Total JVM memory: 384MB heap + 100MB Metaspace + 200MB thread stacks = 684MB — 33% over the 512MB container limit. The kernel OOMKills the container process. ECS registers it as an exit code 137 (SIGKILL). The container is restarted immediately — and OOMKilled again. Constant restart loop. Logs show nothing (the JVM was killed before it could log). Fix: increase container memory limit to 1024MB for this workload, or reduce thread count. Rule of thumb for Spring Boot containers: memory limit = heap size × 2.5 for MVC (thread-per-request), or heap size × 1.5 for WebFlux (fewer threads).
ECS vs EKS vs Fargate — when to use each
The honest trade-offs between managed Kubernetes, ECS, and serverless containers
ECS + Fargate
Who it's for: Teams that want containers without managing infrastructure or Kubernetes.

Operations: No nodes to manage, no Kubernetes version upgrades, no etcd. AWS handles everything below the task level.

Best for: Microservices, APIs, batch jobs, scheduled tasks. Anything where the team wants to focus on the application, not the platform.

Limit: AWS-specific API. Harder to move to another cloud. Less ecosystem than Kubernetes (fewer community tools).
ECS + EC2
Who it's for: Teams needing container orchestration but wanting control over the underlying EC2 instances (custom AMIs, GPU instances, specific instance types, Spot optimisation).

Operations: You manage the EC2 cluster (patching, scaling). ECS manages container scheduling on top.

Best for: GPU workloads (ML inference), applications requiring custom kernel modules, very cost-sensitive workloads using Spot.

Limit: More operational overhead than Fargate.
EKS
Who it's for: Teams with Kubernetes expertise, multi-cloud ambitions, or complex service mesh requirements.

Operations: AWS manages the control plane (etcd, API server). You manage worker nodes (EC2 or Fargate via EKS Fargate).

Best for: Large engineering organisations with existing Kubernetes investment, teams using Istio/Linkerd service mesh, workloads requiring advanced scheduling.

Limit: Most complex to operate. Control plane costs ~$72/month. Significant learning curve.

It has a low operational overhead, integrates natively with every AWS service you've learned (ALB, CloudWatch, IAM, Secrets Manager, ECR), and handles 95% of production container workloads. For your JP Morgan project at Deloitte, the existing infrastructure is likely ECS — learning ECS Fargate is directly applicable to your day job. For GetWork: Fargate scales to zero cost when no tasks are running (unlike EC2 which has fixed hourly costs), is perfect for a growing application, and gives you all the deployment and scaling capabilities without managing servers. ECS Service Discovery — how services find each other
AWS Cloud Map integration and the three patterns for container-to-container communication

In ECS, each running task gets its own IP address (in awsvpc network mode — the production mode for Fargate). Unlike EC2 where you have a fixed set of instances with known IPs, container IPs are ephemeral — tasks start and stop constantly, and each time they start they may get a different IP. Service discovery is the mechanism for finding other services by name rather than by IP.

Pattern 1 — ALB-based service discovery (recommended): Each ECS service has an ALB target group. Other services call each other via the ALB's DNS name. Simple, uses existing infrastructure, integrates with health checks and connection draining. For internal communication within the VPC, use a private ALB (internal scheme) or the same ALB with different path-based routing rules. This is the most operationally straightforward approach and leverages everything you already know about ALBs.

Pattern 2 — AWS Cloud Map (service mesh light): Cloud Map is AWS's managed service registry. ECS services can automatically register with Cloud Map when tasks start and deregister when they stop. Other services query Cloud Map via DNS or the API to discover healthy instances. DNS-based discovery: auth-service.getwork.local resolves to the current IPs of all healthy auth-service tasks. More direct than ALB (no load balancer hop) but requires the calling service to handle load balancing and retry logic itself.

Pattern 3 — Amazon ECS Service Connect (modern approach): Service Connect (released 2022) is a managed service mesh built into ECS. It configures a proxy sidecar container alongside your application container, handling service discovery, load balancing, circuit breaking, and observability between services without requiring an ALB or manual Cloud Map configuration. Think of it as a simplified Istio service mesh built into ECS. For new microservice deployments starting from scratch: Service Connect is the recommended pattern. For existing applications with ALB-based routing: the migration cost may not justify switching.

awsvpc network mode — why it's the only correct choice for production Fargate: Fargate only supports awsvpc network mode. In awsvpc mode, each task gets its own ENI with its own IP address in your VPC subnets. Security Groups apply at the task level — each service can have its own Security Group with exactly the inbound/outbound rules it needs. Task-level network isolation is the correct security model: the job-service and the payment-service each have their own Security Groups even if they run on the same Fargate infrastructure. Contrast with bridge network mode (EC2 launch type only): containers share the host's network namespace with port mapping. Security Groups apply at the EC2 instance level — all containers on the same EC2 have the same Security Group rules. This is the fundamental networking difference that makes awsvpc the only choice for multi-tenant or security-sensitive ECS deployments.
Fargate is the "serverless containers" runtime for ECS and EKS. You define what to run (task definition) and how much to run (service desired count), and Fargate provides the infrastructure. No EC2 instances to patch, no cluster capacity to manage, no AMIs to maintain. Understanding how Fargate charges, how it handles cold starts, and how its networking model differs from EC2 determines whether it's the right fit for each workload.
How Fargate actually works under the hood
The microVM isolation model, cold start mechanics, and what "serverless" means operationally

Fargate runs each task in a dedicated microVM — a lightweight virtual machine that provides hardware-level isolation between tasks from different customers (and between your own tasks for security). This is similar to how Lambda uses Firecracker microVMs. The microVM boots when a task is scheduled, runs your containers, and is destroyed when the task stops. You never see the microVM — it's an implementation detail that provides the isolation guarantee.

What Fargate manages for you: The microVM lifecycle, the host OS security patching (there's no EC2 instance for you to patch), container runtime installation and maintenance, the ECS agent, network configuration (ENI attachment, IP assignment, security group configuration), and the storage for the container's ephemeral filesystem.

Fargate cold start timing: When ECS starts a new Fargate task, the sequence is: microVM boot (~1-3 seconds) → ENI attachment and IP assignment (~10-15 seconds) → image pull from ECR (~10-60 seconds, depending on image size) → container startup (your application initialisation time). Total time from "ECS decided to start a task" to "task is passing health checks": 60-120 seconds typically. The image pull time is the dominant variable. A 500MB fat-JAR-based image takes 60 seconds to pull. The same application as a 200MB layered image takes 20-30 seconds. For Sprint Boot, your JVM startup + Spring context initialisation adds 30-60 seconds on top. Warm pool equivalent for Fargate: ECS Service Auto Scaling has a "warm" tier (Fargate Spot or provisioned capacity) — but cold starts are the primary scaling latency concern for Fargate.

Fargate Spot: AWS offers Fargate Spot at 70% discount — the same as EC2 Spot, but for Fargate tasks. The trade-off: AWS can interrupt Fargate Spot tasks with 2-minute warning when it needs capacity back. Suitable for: batch processing, stateless workers, CI/CD jobs, tasks that tolerate interruption. Not suitable for: user-facing APIs, payment processing, anything requiring guaranteed availability. The ECS service can mix Fargate (on-demand) and Fargate Spot in a configurable ratio — the same mixed fleet pattern as EC2 On-Demand + Spot.

GetWork architecture with Fargate: Main API service: Fargate on-demand (guaranteed availability, 2 tasks minimum). Background job workers (matching algorithm, notification sending): Fargate Spot (70% cheaper, can tolerate occasional interruption and retry). Database migration jobs: Fargate on-demand one-off tasks triggered by deployments. This mixed strategy reduces the cost of batch/background work by 70% while keeping the user-facing API on reliable on-demand capacity.
Fargate networking — awsvpc and the ENI management that ECS handles
How each Fargate task gets its own VPC identity, Security Group, and IP address

Every Fargate task runs with awsvpc network mode — the only supported mode. When ECS starts a Fargate task, it attaches a new ENI to the task's microVM, assigns a private IP address from the specified subnet, and configures the specified Security Groups on that ENI. From the VPC's perspective, each Fargate task looks like an EC2 instance with its own network identity.

Practical implications: Each Fargate task consumes one ENI from your VPC's ENI limit. Each task consumes one IP address from your subnet's available CIDR range. If you run 100 Fargate tasks in a /24 subnet (254 usable IPs), you'll exhaust the subnet's IPs quickly (especially when considering your EC2 instances, RDS instances, and ALB ENIs also consuming IPs). Plan your subnet CIDR sizes with the maximum number of Fargate tasks in mind. For a service with max 50 tasks: ensure the subnet has at least 100 available IPs (2x headroom for scaling and rolling deployments).

The Fargate task that can't pull its image because of VPC routing: ECS Fargate tasks in private subnets need network access to ECR to pull images. ECR is a public-facing service — without a NAT Gateway or an ECR VPC Interface Endpoint, Fargate tasks in private subnets cannot pull images and fail with "CannotPullContainerError: ImagePullError". Solution: either use a NAT Gateway (which you have from Day 4) or create VPC Interface Endpoints for ECR (com.amazonaws.ap-south-1.ecr.api and com.amazonaws.ap-south-1.ecr.dkr) and S3 Gateway Endpoint (ECR uses S3 for image layer storage). The VPC Endpoint approach eliminates NAT Gateway costs for image pulls — significant for large images pulled frequently.

Security Group configuration for Fargate tasks: With awsvpc, each task has its own Security Groups. Your getwork-api ECS service should use a Security Group (ecs-api-sg) that: allows inbound 8080 from alb-sg (the ALB's Security Group — same as the EC2 approach from Days 7-8). Allows inbound from other ECS services that need to call it directly (if using Service Connect or Cloud Map). Allows outbound to db-sg on 3306 (RDS). Allows outbound HTTPS to the internet (for ECR image pull and Secrets Manager calls — through NAT Gateway or Interface Endpoint). This is the same Security Group chaining pattern from Day 4, applied at the task level.

Fargate pricing — understanding vCPU-hours and GB-hours
How Fargate charges differ from EC2 and the cost calculation for a production Spring Boot service

Fargate charges per second for CPU and memory consumed by running tasks. Billing stops when the task stops — unlike EC2 where you pay for the running instance even when CPU is idle. The billing granularity is per second after the first minute.

Fargate pricing in ap-south-1 (Mumbai): vCPU: $0.04945 per vCPU-hour. Memory: $0.00541 per GB-hour. A task with 0.5 vCPU and 1GB memory running continuously: vCPU cost = 0.5 × $0.04945 = $0.02473/hr. Memory cost = 1 × $0.00541 = $0.00541/hr. Total: ~$0.030/hr per task = ~$21.60/month per continuously-running task.

Comparison with EC2: A t3.micro EC2 instance in ap-south-1: $0.0116/hr = ~$8.35/month. But one t3.micro can typically run 4-6 small Spring Boot tasks simultaneously (bin-packing). With ECS EC2 launch type and bin-packing: $8.35/month for the EC2 ÷ 4 tasks = ~$2.09/task/month. Fargate is ~10x more expensive per task when EC2 is fully packed. The premium is for operational simplicity (no EC2 management). Whether that premium is justified depends on your team's operational capacity and the cost of engineering time to manage EC2 clusters.

When Fargate is cheaper than EC2: Fargate charges only when tasks are running. For batch jobs that run for 2 hours per day: Fargate cost = $0.030/hr × 2 hrs/day × 30 days = $1.80/month. EC2 cost for the equivalent instance running 24/7: $8.35/month. Fargate is 4.6x cheaper for this batch job because you only pay for the 2 hours it runs, not the 22 hours it's idle. The crossover point: if a Fargate task runs more than ~8 hours per day continuously, EC2 launch type (properly bin-packed) is cheaper. For always-on production API tasks: EC2 launch type is economically superior. For intermittent or batch tasks: Fargate is cheaper and simpler.

Secrets and environment variables in task definitions
The two ways to inject sensitive configuration — and why the secrets array is better than environment variables for credentials

ECS task definitions support two methods for passing configuration to containers:

environment array — plain key-value pairs. Values are stored in the task definition JSON in plaintext. Visible in the ECS console, in CloudTrail, and to anyone with ecs:DescribeTaskDefinition permission. Use for: non-sensitive configuration (region, spring profile, JVM flags, feature flags). Never use for credentials, API keys, or any sensitive value.

secrets array — references to Secrets Manager ARNs or Parameter Store paths. The ECS execution role fetches the secret value at task start time and injects it as an environment variable into the container. The secret value is never stored in the task definition JSON — only the ARN reference is stored. The actual value is fetched at runtime and exists only in the container's memory. Anyone inspecting the task definition sees the ARN (which tells them the secret exists) but not the value.

// Task definition secrets — referencing individual JSON keys from Secrets Manager "secrets": [ { // Reference the entire secret string as one env var "name": "DB_SECRET_JSON", "valueFrom": "arn:aws:secretsmanager:ap-south-1:ACCOUNT:secret:/getwork/prod/database" }, { // Reference a specific key from a JSON secret using the :: syntax // format: secretARN:json-key:: (two trailing colons are required) "name": "DB_HOST", "valueFrom": "arn:aws:secretsmanager:ap-south-1:ACCOUNT:secret:/getwork/prod/database:host::" }, { "name": "DB_PASSWORD", "valueFrom": "arn:aws:secretsmanager:ap-south-1:ACCOUNT:secret:/getwork/prod/database:password::" }, { // Parameter Store reference (cheaper than Secrets Manager for non-rotated config) "name": "APP_BUCKET", "valueFrom": "arn:aws:ssm:ap-south-1:ACCOUNT:parameter/getwork/prod/app-bucket" } ]
Secrets are injected once at task start — rotation doesn't propagate automatically: When ECS starts a task, it fetches secret values at that moment and injects them as environment variables. If you rotate a secret in Secrets Manager (e.g., rotate the database password), running tasks still use the old value — they were injected at start time, not fetched dynamically. The new password is only used when tasks are replaced (new deployment, scale-in/scale-out). Rolling deployment after secret rotation is the required procedure: rotate the secret in Secrets Manager → trigger a new ECS deployment → new tasks start with the new value → old tasks terminate after connection draining → all tasks now use the new secret. This is why some teams prefer the Spring Boot application to fetch secrets from Secrets Manager SDK at runtime (dynamic fetch) rather than relying on ECS injection (static inject at start). Both patterns work; choose based on how frequently you rotate secrets.

ECS Service configuration — the key settings that determine deployment behaviour
Desired count, deployment configuration, ALB integration, and the deployment circuit breaker
# Create a production ECS service via CLI aws ecs create-service \ --cluster getwork-cluster \ --service-name getwork-api \ --task-definition getwork-api:7 \ # Specific revision — not :latest --desired-count 2 \ --launch-type FARGATE \ --network-configuration "awsvpcConfiguration={ subnets=[subnet-private-1a,subnet-private-1b], securityGroups=[sg-ecs-api], assignPublicIp=DISABLED}" \ # Private subnets — no public IP needed --load-balancers "targetGroupArn=TG_ARN, containerName=getwork-api, containerPort=8080" \ --health-check-grace-period-seconds 120 \ # Grace period for Spring Boot startup --deployment-configuration " deploymentCircuitBreaker={enable=true,rollback=true}, minimumHealthyPercent=100, maximumPercent=200" \ --enable-ecs-managed-tags \ --propagate-tags TASK_DEFINITION \ --region ap-south-1

minimumHealthyPercent=100 and maximumPercent=200: During deployment, ECS keeps 100% of tasks healthy and can launch up to 200% of desired count. For desired=2: always keep 2 running tasks, and can launch up to 4 total. Deployment strategy: start 2 new tasks (now at 4 total), wait for them to be healthy, stop 2 old tasks (back to 2). Zero downtime — no traffic loss. This is the safest deployment strategy but requires paying for 4 tasks briefly during deployment.

minimumHealthyPercent=50 and maximumPercent=150: Alternative for cost efficiency. Can take one task down before new one is ready. For desired=2: can drop to 1 running task during deployment. Brief capacity reduction but cheaper (only 3 tasks max during deployment instead of 4).

The deployment circuit breaker — automatic rollback when deployments fail: With deploymentCircuitBreaker.enable=true, rollback=true, ECS monitors new tasks launched during a deployment. If more than a threshold number of tasks fail to enter RUNNING state and pass their health check within the configured grace period, ECS automatically rolls back the deployment to the previous task definition revision. This requires zero operator intervention — a bad image that fails to start triggers an automatic rollback without any manual action. Without this: a bad deployment that fails all new task launches leaves the service in a degraded state indefinitely, with the old tasks still (correctly) being stopped as new (broken) ones fail to replace them. The circuit breaker is not enabled by default — enable it for every production service.
The health check grace period that causes constant restarts: The health-check-grace-period-seconds is the time ECS waits after a task starts before considering ALB health check failures as a reason to stop the task. If grace period is too short (10 seconds) and Spring Boot takes 60 seconds to initialise, the ALB health check fails during startup, ECS marks the task as unhealthy, stops it, and starts a new one — which also takes 60 seconds to start, fails, and is stopped. Constant restart loop. Set grace period to at least 1.5× your application's startup time. For Spring Boot on Fargate: 90-120 seconds is appropriate.
ECS rolling deployments — how service updates work
The exact sequence when you update a service to a new task definition revision

When you update an ECS service to point to a new task definition revision (new image, new environment variable, new resource allocation), ECS performs a rolling deployment. The exact sequence with minimumHealthyPercent=100, maximumPercent=200, desired=2:

1. ECS launches 2 new tasks using the new task definition. Now 4 tasks total (2 old, 2 new). ALB target group has 4 registered targets.

2. New tasks go through PENDING → RUNNING → health check evaluation. ALB checks /actuator/health/readiness every 15 seconds (3 healthy checks needed = 45 seconds minimum).

3. When 2 new tasks are healthy (passing ALB health check), the ALB begins routing traffic to them. All 4 tasks receive traffic at this point.

4. ECS stops the 2 old tasks — connection draining applies (deregistration delay, 60 seconds by default). During drain, old tasks accept no new requests but complete in-flight ones.

5. After drain completes, old tasks stop. 2 new tasks remain. Deployment complete.

Total time: ~2-3 minutes (Fargate task start + image pull + Spring Boot startup + health check evaluation + connection drain). Compare to EC2 rolling deployment (Day 8 Instance Refresh): similar total time but fewer moving parts in ECS.

The ECS deployment notification to Slack/PagerDuty: Use EventBridge to catch ECS deployment events and notify your team. The event pattern: {"source":["aws.ecs"],"detail-type":["ECS Service Action"],"detail":{"eventName":["SERVICE_DEPLOYMENT_COMPLETED","SERVICE_DEPLOYMENT_FAILED"]}}. Route to an SNS topic → Slack webhook Lambda. Deployments completing successfully or failing (and triggering circuit breaker rollback) are both actionable events that your team should know about in near-real-time.

ECS auto scaling vs ALB health check vs circuit breaker — three different mechanisms, one flow: ALB health check: determines whether a specific task receives traffic (individual task health). Amazon ECR (Elastic Container Registry) is the private Docker registry for your container images. It integrates with ECS, IAM, and CloudWatch. Understanding how ECR stores images, how cross-account access works, and how lifecycle policies prevent unbounded storage costs is the operational foundation for a working ECS deployment pipeline.
ECR mechanics — image storage, authentication, and the push/pull flow
How images are stored, how the AWS CLI authenticates to ECR, and what a repository policy controls

ECR is a private, encrypted Docker registry hosted within your AWS account. Each repository stores images for one service (e.g., getwork-api). Images are stored as a set of content-addressed layers (same structure as Docker's local image store) in S3 — immutable, versioned, and durable. ECR uses IAM for authentication — Docker login is replaced with a temporary credential exchange.

Authentication flow: To push/pull from ECR, you first authenticate Docker to ECR with a temporary password: aws ecr get-login-password | docker login --username AWS --password-stdin ACCOUNT.dkr.ecr.ap-south-1.amazonaws.com. The get-login-password call exchanges your AWS credentials for a 12-hour Docker authentication token. After authentication, Docker push/pull commands work with the ECR hostname. ECS tasks use the execution role to authenticate — the ECS agent calls GetAuthorizationToken on behalf of your task, which is why that permission is required in the execution role.

The ECR pricing model — it's the same as S3: ECR charges $0.10/GB-month for private repository storage. A 500MB image stored for one month: $0.05. Data transfer to ECS tasks in the same region: free. Data transfer to different regions: standard S3 transfer rates. For a single microservice with weekly deployments, keeping 10 image versions: 10 × 500MB = 5GB = $0.50/month. At scale with 20 services and 50 versions each: 1TB = $100/month. Lifecycle policies that automatically delete old images are essential to control this cost.

ECR repository policy vs IAM policy: ECR repositories have resource-based policies (like S3 bucket policies). They control which principals can push/pull from the repository. Required for cross-account access: if your CI/CD pipeline runs in Account A but ECS runs in Account B, the ECR repository policy in Account A must allow Account B's execution role to pull. Within the same account: IAM policies on the execution role are sufficient — no repository policy required.

Image tagging strategy — making deployments traceable and rollbacks possible
Why :latest is dangerous and what to use instead for production deployments

Docker image tags are mutable pointers — the :latest tag on a repository can point to different images over time. Using :latest in your ECS task definition means: when you deploy a new image, :latest points to the new image. But existing running tasks with :latest in their task definition are not automatically updated — ECS only pulls the image when starting a new task. A running task pinned to :latest in the task definition may actually be running whatever image was :latest at task start time, not today's :latest. This creates ambiguity: which image version is running?

The correct tagging strategy:

# In CI/CD pipeline — tag with build number or git commit SHA IMAGE_TAG="${BITBUCKET_BUILD_NUMBER}" # Monotonically increasing — sortable # Or: IMAGE_TAG="${BITBUCKET_COMMIT:0:7}" # Short git SHA — traceable to commit ECR_REPO="ACCOUNT.dkr.ecr.ap-south-1.amazonaws.com/getwork-api" # Build and push with specific tag docker build -t ${ECR_REPO}:${IMAGE_TAG} . docker push ${ECR_REPO}:${IMAGE_TAG} # Also tag as latest (for human convenience — not used in task definitions) docker tag ${ECR_REPO}:${IMAGE_TAG} ${ECR_REPO}:latest docker push ${ECR_REPO}:latest # Update task definition to use the specific tag NEW_TASK_DEF=$(aws ecs describe-task-definition \ --task-definition getwork-api \ --query 'taskDefinition' --output json | \ jq --arg image "${ECR_REPO}:${IMAGE_TAG}" \ '.containerDefinitions[0].image = $image | del(.taskDefinitionArn,.revision,.status,.requiresAttributes,.placementConstraints,.compatibilities,.registeredAt,.registeredBy)') NEW_REVISION=$(aws ecs register-task-definition \ --cli-input-json "$NEW_TASK_DEF" \ --query 'taskDefinition.revision' --output text) # Update service to new revision aws ecs update-service \ --cluster getwork-cluster \ --service getwork-api \ --task-definition getwork-api:${NEW_REVISION} \ --force-new-deployment \ --region ap-south-1
Why immutable image tags matter for rollbacks: With tag-per-build: to roll back to build 142 from build 145, simply update the service to use task definition revision pointing to image tag :142. The image at tag :142 has never changed — it's always the same image. With :latest: the image at :latest changes every deployment. To roll back, you must rebuild the previous version (or push it as :latest again). Rollback time: seconds (update task definition pointer) vs minutes (rebuild and push image). For a financial services application at JP Morgan where rollback speed is a P0 metric, immutable versioned tags are non-negotiable.
ECR lifecycle policies — automatic image cleanup to control storage costs
Rules that automatically delete old images and prevent ECR storage from growing unboundedly

Without lifecycle policies, every image pushed to ECR is retained forever. A service with daily deployments accumulates 365+ images per year. Each image: 200-500MB. 365 × 300MB = ~109GB = ~$10.90/month per service just for old images. With 20 services: $218/month in obsolete image storage.

# Lifecycle policy — keep only the last 10 images per repository aws ecr put-lifecycle-policy \ --repository-name getwork-api \ --lifecycle-policy-text '{ "rules": [ { "rulePriority": 1, "description": "Keep last 10 tagged images", "selection": { "tagStatus": "tagged", "tagPrefixList": ["v", "build-", "sha-"], "countType": "imageCountMoreThan", "countNumber": 10 }, "action": {"type": "expire"} }, { "rulePriority": 2, "description": "Remove untagged images after 1 day", "selection": { "tagStatus": "untagged", "countType": "sinceImagePushed", "countUnit": "days", "countNumber": 1 }, "action": {"type": "expire"} } ] }' --region ap-south-1
Lifecycle policies run asynchronously: ECR evaluates lifecycle rules daily, not immediately when a new image is pushed. After pushing image #11, the lifecycle policy won't delete image #1 for up to 24 hours. This means during the evaluation window, you may briefly have 11 images. Plan storage estimates accordingly. The policy also doesn't protect images tagged with :latest from being deleted if :latest happens to be the oldest in the count-based policy — pin :latest separately with a "keep images tagged with 'latest'" rule at lower priority if you want to preserve it.
The lifecycle policy that deletes an image that ECS is actively running: A lifecycle policy deletes image tag :build-100. But an ECS task that was started last week is still running with image :build-100 pinned in its task definition. ECS doesn't check if an image is in use before deleting it. The running task is unaffected (the image layers are already pulled into the container's filesystem). But if that task restarts (crash, scale-in/scale-out), ECS tries to pull :build-100 from ECR — it's gone. Task fails to start with CannotPullContainerError. Service enters crash loop. Mitigation: ensure your lifecycle policy keeps more versions than your service would ever roll back to (keep last 10 if you'd never roll back more than 10 deployments). Or use image digests (immutable SHA-based references) in task definitions instead of tags.

Q1 Explain the difference between executionRoleArn and taskRoleArn in an ECS task definition. What permissions does each need? Medium▼

These are two different IAM roles serving different identities in the ECS lifecycle, and confusing them is the most common ECS IAM mistake.

executionRoleArn is assumed by the ECS infrastructure — the ECS agent on EC2 or Fargate's control plane — to perform the setup steps before your application code runs. It needs: ECR permissions to pull the container image (ecr:GetAuthorizationToken, ecr:BatchGetImage, ecr:GetDownloadUrlForLayer), CloudWatch Logs permissions to create log groups and write log streams (logs:CreateLogGroup, logs:CreateLogStream, logs:PutLogEvents), and Secrets Manager / SSM Parameter Store permissions to fetch values that are referenced in the task definition's secrets array (secretsmanager:GetSecretValue, ssm:GetParameter). Your application code never runs under the execution role — it's done with its work by the time your container starts.

taskRoleArn is assumed by the code running inside your container — your Spring Boot application. It's equivalent to the EC2 instance role. It needs whatever AWS API permissions your application requires at runtime: s3:PutObject for image uploads, secretsmanager:GetSecretValue for runtime secret fetching (distinct from ECS injection), cloudwatch:PutMetricData for custom metrics, SQS permissions if your app reads from a queue. The task role credentials are available via IMDS (169.254.170.2 — the ECS metadata endpoint, distinct from EC2's 169.254.169.254).

The wrong pattern: giving the execution role your application's permissions (S3, SQS, etc.) or giving the task role ECR and CloudWatch Logs permissions meant for the execution role. Both are security violations of least-privilege. For the execution role, use the AWS managed policy AmazonECSTaskExecutionRolePolicy as a starting point and add only your secrets references. For the task role, mirror your EC2 instance role's policy.

Why this answer works: Clear mechanism explanation for each role, specific permission examples for each, the "your application code never runs under execution role" clarification, and the wrong pattern identification.
Q2 Your Fargate service keeps deploying new tasks but they never become healthy. The service is in a crash loop. How do you diagnose this? Hard▼

This is a systematic diagnosis problem. The symptom — tasks start but never pass health checks — has several root causes, each diagnosed differently.

Step 1: Check ECS service events. aws ecs describe-services --cluster CLUSTER --services SERVICE --query 'services[0].events[0:5]'. The events give plain English descriptions: "Service was unable to place a task" (capacity problem), "Task failed ELB health check" (app not responding correctly), "CannotPullContainerError" (ECR auth or networking issue), "stopped reason: Essential container in task exited" (the container crashed at startup).

Step 2: If container crashed — check stopped task logs. aws ecs describe-tasks --cluster CLUSTER --tasks TASK_ID shows the stop reason and exit code. Exit code 1: application error, check CloudWatch Logs. Exit code 137: OOMKilled — container exceeded memory limit. Exit code 139: segfault. Exit code 1 with log context: check aws logs tail /getwork/ecs/api --since 10m for the exception that crashed Spring Boot at startup.

Step 3: If image pull failed — check execution role permissions and network connectivity. "CannotPullContainerError" means ECR authentication failed or the Fargate task cannot reach the ECR endpoint. Verify: execution role has ecr:GetAuthorizationToken. The private subnet has a NAT Gateway or ECR VPC Interface Endpoints. The task's security group allows outbound HTTPS (443).

Step 4: If container is running but health check fails — test the health endpoint directly. Use ECS Exec (if enabled) to shell into the container: aws ecs execute-command --cluster CLUSTER --task TASK_ID --container getwork-api --interactive --command "/bin/sh". Inside the container: curl http://localhost:8080/actuator/health/readiness. See the actual response. Common issues: Spring Boot is still initialising (startup takes longer than grace period), database connection failure at startup is returning 503 on readiness, wrong health check path configured in the task definition.

Step 5: Check if the deployment circuit breaker fired. If the deployment already auto-rolled back, the service events will show "deployment circuit breaker condition triggered." Check the previous healthy task definition revision to understand what changed.

Why this answer works: The structured five-step approach with specific CLI commands at each step shows systematic debugging experience. The exit code interpretation (137 = OOM, 139 = segfault, 1 = app error) demonstrates operational depth. The ECS Exec tip shows you know how to get inside a Fargate container for live debugging.
Q3 How does Fargate networking work? Why does each task need its own Security Group? Medium▼

Fargate uses awsvpc network mode exclusively. When ECS starts a Fargate task, it attaches a dedicated ENI to the task's microVM and assigns it a private IP address from the specified subnet. The task's network identity — IP address and Security Groups — belongs to the ENI, not to any underlying host. From the VPC's perspective, each Fargate task is indistinguishable from an EC2 instance.

Each task having its own Security Group is the key security benefit of awsvpc. In the traditional Docker bridge network mode (EC2 launch type), all containers on the same EC2 instance share the host's Security Groups. If the api-service and the admin-service run on the same EC2, they share the same Security Group rules. You can't allow inbound on port 8080 for api-service from the ALB without also allowing inbound to admin-service from the same ALB.

With awsvpc/Fargate: the api-service has Security Group ecs-api-sg allowing port 8080 from alb-sg. The admin-service has Security Group ecs-admin-sg allowing port 8081 from admin-alb-sg. Different services have completely isolated Security Group rules even if their underlying Fargate microVMs happen to run on the same AWS infrastructure. This is the principle of least privilege applied at the network level to containers — each service exposes exactly the ports it needs to exactly the sources that should reach it.

The operational implication: if you have 50 Fargate tasks across 5 services, you need 5 Security Groups (one per service), not 50. The Security Group attaches to the ENI type for the task family, and all tasks of that family share the same Security Group. When the task stops and its ENI is removed, the Security Group remains for reuse by the next task of that family.

Why this answer works: The Security Group isolation comparison between bridge mode (shared host SG) and awsvpc (per-task SG) is the substantive answer. The specific example with api-service and admin-service makes the isolation benefit concrete. The operational implication (5 SGs for 50 tasks) shows you've thought about scale.
Q4 Why is a multi-stage Docker build important for production? Explain the Spring Boot layered JAR approach. Easy▼

Multi-stage builds solve two problems: image size and build tool isolation.

Without multi-stage: a single Dockerfile that installs Maven, downloads dependencies, compiles the code, and runs the JAR includes Maven (hundreds of MB), the JDK (vs just the JRE needed for runtime), all build tools, and build-time dependencies — all in the final image that runs in production. A 500MB image for a small Spring Boot app, and the image contains Maven which has no business being in a production server.

With multi-stage: the builder stage (FROM amazoncorretto:17 AS builder) installs Maven, downloads dependencies, and compiles. The runtime stage (FROM amazoncorretto:17-alpine) contains only the JRE and the compiled application. The build tools never make it to the final image. Image size drops from 500MB to 150-200MB. Fewer components in the image means fewer CVEs found by security scanning.

The Spring Boot layered JAR goes further. A standard fat JAR packages all 200MB of library dependencies + 2MB of application code into one layer. Every code change (even one line) replaces the entire 200MB layer in ECR — 200MB push and pull on every deployment. Layered JAR extraction creates separate Docker layers: one for stable library dependencies (200MB, rarely changes), one for snapshot dependencies (5-10MB, changes occasionally), and one for application code (2MB, changes every deployment). Now a typical code-change deployment pushes 2MB to ECR instead of 200MB. Deployment time for the image pull step: 5 seconds instead of 60 seconds. In a fleet of 50 Fargate tasks all pulling the new image simultaneously, this difference is enormous.

Why this answer works: The specific size numbers (500MB → 150MB for multi-stage, 200MB → 2MB push for layered JAR) make the benefit concrete. The 50-task scaling implication shows you think about the fleet, not just one container.

Day 12 — ECS & Docker on AWS mechanics
Multi-stage Dockerfile
Builder stage installs Maven/JDK + builds. Runtime stage has JRE + app only. No build tools in production image. 500MB → 150MB. Spring Boot layered JAR makes code-change deploys 2MB instead of 200MB.
pom.xml-first layer caching
COPY pom.xml first, then RUN mvn dependency:go-offline. Deps layer cached unless pom.xml changes. Source code layer rebuilds only on code changes. Cuts CI build from 8 min to 90s on cache hit.
UseContainerSupport
JVM reads cgroup memory limits (not host RAM) for heap sizing. Required since Java 10. MaxRAMPercentage=75.0 uses 75% of container memory limit for heap. Scale heap by setting container memory limit — not hardcoded JVM flags.
executionRoleArn vs taskRoleArn
Execution role: used by ECS agent to pull ECR image, write CloudWatch logs, inject secrets at task start. Task role: used by your Spring Boot code for S3, Secrets Manager, SQS calls. Never combine — different principals, different permissions.
awsvpc network mode
Every Fargate task gets its own ENI, private IP, and Security Groups. Task-level network isolation. Required for Fargate. Security Groups apply per-task, not per-host. Each task consumes 1 ENI from VPC limits.
Deployment circuit breaker
Set deploymentCircuitBreaker.enable=true,rollback=true on every production service. Auto-rolls back deployment if new tasks fail to become healthy. Without it: broken deployments leave service in degraded state indefinitely.
OOMKilled = exit code 137
Container exceeded memory hard limit. JVM overhead (Metaspace + threads + NIO buffers) beyond heap size. Rule: memory limit = heap × 2.5 (MVC) or heap × 1.5 (WebFlux). Set ExitOnOutOfMemoryError to crash fast.
Fargate cold start
microVM boot (~3s) + ENI attachment (~15s) + ECR image pull (~20-60s) + Spring Boot startup (~30-60s) = 90-120s total. Layered JAR reduces image pull. Health check grace period must exceed this total.
ECR lifecycle policy
Keep last 10 tagged images, delete untagged after 1 day. Without policy: 365+ images/year per service. 20 services × 500MB × 365 builds = hundreds in storage costs. Set on every ECR repository at creation.
:latest tag danger
:latest is mutable — changes every push. Use build-NUMBER or git-SHA tags in task definitions. Rollback = update task definition to point to previous tag. Immutable tags make rollback a 10-second operation.
Fargate vs EC2 cost
Fargate: ~$21/month per continuous task (0.5 vCPU, 1GB). EC2 launch type (bin-packed): ~$2/task/month. Fargate 10× more expensive for always-on workloads. But Fargate is cheaper for batch/intermittent tasks that don't run 24/7.
CannotPullContainerError
ECR auth failure: check executionRole has ecr:GetAuthorizationToken on *. Network: Fargate in private subnet needs NAT Gateway or ECR Interface Endpoints + S3 Gateway Endpoint to reach ECR.
