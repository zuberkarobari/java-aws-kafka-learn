What is AWS?
Amazon Web Services — the world's largest cloud platform

AWS is Amazon's platform of on-demand cloud services — you rent computing, storage, databases, networking, and more from Amazon's data centres worldwide, paying only for what you use, billed per second or per request. There's no hardware to buy, no physical server room to manage, and no upfront commitment.

Think of it like electricity. You don't own a power plant just because you need electricity at home — you plug in and pay for what you use. AWS is the same thing but for computing infrastructure. When your Spring Boot app runs on "AWS", it's physically running on a server in an Amazon warehouse. You just never see or touch that machine.

AWS vs on-premise (your JP Morgan context): On-premise means buying physical servers, racking them in a data centre, hiring sysadmins to maintain them, and waiting weeks for hardware procurement. AWS eliminates all of that. At your Deloitte project, instead of JP Morgan buying servers, they're renting capacity from AWS and paying per second of use.

AWS currently has 200+ services. You don't need all of them. The 20 core services cover 95% of real-world applications. That's what this 30-day plan teaches.

Regions
The geographic unit where your app lives

A Region is a distinct geographic area, like a country or a city, where AWS has built a cluster of data centres. AWS has 33+ Regions worldwide. Each Region is completely independent — services, data, and networking in one Region don't automatically connect to another.

When you launch any AWS resource — an EC2 instance, an RDS database, an S3 bucket — you are always launching it inside a specific Region. The region you choose affects three things: latency (how fast is your app for end users), cost (prices vary by region), and data residency (legal requirement to store data inside a country's borders).

Most important regions to know:
ap-south-1 → Mumbai (use this for all your labs — lowest latency from Bangalore)
us-east-1 → N. Virginia (AWS's oldest and cheapest region, where new features launch first)
eu-west-1 → Ireland (EU data residency for GDPR compliance)
ap-southeast-1 → Singapore (Southeast Asia)

One gotcha that every beginner hits: the AWS console defaults to us-east-1. You launch an EC2 instance and then can't find it — because you're now looking at ap-south-1. Always verify the region dropdown in the top-right of the console before doing anything.

Always verify your region first ap-south-1 = Mumbai
Availability Zones (AZs)
Physical redundancy inside a region

Each Region is divided into 2–6 Availability Zones. An AZ is one or more physical data centres with completely independent power supply, cooling systems, and networking. Mumbai (ap-south-1) has three AZs: ap-south-1a, ap-south-1b, and ap-south-1c.

Why does this matter? If a physical disaster — a fire, a flood, a power outage — takes down one AZ, the other AZs in the same Region are unaffected. AZs are close enough to each other that network latency between them is less than 1ms, so your app can span multiple AZs without noticeable performance impact.

The deployment rule: Never put everything in one AZ. A production-ready Spring Boot app runs EC2 instances in at least two AZs. If 1a fails, 1b keeps serving traffic. This is called high availability (HA) and it's something you'll implement from Day 8 onwards.

In your Deloitte/JP Morgan project, when AWS CloudWatch monitors your infrastructure, it's monitoring resources that exist inside specific AZs. Understanding AZs is why you set alarm thresholds per-AZ rather than just per-region.

AZs = high availability Sub-1ms latency between AZs
Edge Locations
Speed for your end users — not where your app runs

Edge Locations are 400+ points of presence scattered across the globe — in cities like Chennai, Hyderabad, Delhi, London, New York, and hundreds more. They are NOT data centres where you run your app. They are caching and DNS servers owned by AWS that sit as close as possible to your end users.

Two AWS services use Edge Locations: CloudFront (the CDN — it caches your static files at edge locations so users get them from a nearby cache instead of travelling all the way to your Mumbai region) and Route 53 (DNS — responds to domain name lookups from the nearest edge instead of a single central server).

Example: Your GetWork app serves job listing images stored in S3 in Mumbai. Without CloudFront, a user in Delhi makes a network round-trip to Mumbai (~1500km). With CloudFront, AWS caches those images at the Delhi edge location — the user's request never leaves their city. Page load could drop from 400ms to 40ms.

The key distinction interviewers test: Edge Locations are for content delivery and DNS — not for running compute workloads. You can't deploy your Spring Boot app "to an Edge Location".

400+ worldwide CloudFront + Route 53 only
IAM and the root account
Identity and Access Management — the permission system

IAM (Identity and Access Management) is AWS's permission system. It controls who can do what to which AWS resources. IAM is a global service — it applies across all Regions. Everything in AWS has an identity, and IAM decides what that identity is allowed to do.

IAM has four main entities. Users are real people with long-lived credentials (username + password for console, or access key + secret for CLI). Groups are collections of users that share the same permissions. Roles are temporary identities assumed by services (like your EC2 instance assuming a role to access S3 — no long-lived keys needed). Policies are JSON documents that define exactly what actions are allowed or denied.

The root account problem: When you first create an AWS account, you log in as "root" — the superuser. Root has unrestricted access to absolutely everything, including billing, account closure, and support settings. There is no way to restrict root with IAM policies. If an attacker steals root credentials, they can delete every resource, rack up millions in charges, and you cannot stop them. The rule is: enable MFA on root immediately, create an IAM admin user, and never use root again for daily work.

For your Spring Boot app on EC2: never paste AWS access keys into your application.properties file or environment. Instead, attach an IAM Role to the EC2 instance. The app running on that EC2 automatically inherits the role's permissions — no static keys, no secrets to leak.

Never use root for daily work Use IAM Roles for services, not users
The Shared Responsibility Model
What AWS secures vs what you secure

The Shared Responsibility Model is one of the most important concepts in AWS — and a guaranteed interview question. It defines a clean line between what AWS is responsible for securing and what you are responsible for securing.

AWS is responsible for the security OF the cloud: physical data centre security, hardware, networking, hypervisor, and the software running managed services (e.g. the MySQL engine in RDS, the execution environment in Lambda). You cannot audit this, but AWS has SOC 2, ISO 27001, and other certifications proving they take it seriously.

You are responsible for security IN the cloud: your IAM user permissions, your security group rules, whether your S3 bucket is public or private, encryption of your data, your application code, and OS patching on EC2 instances.

The line moves depending on the service: With EC2 (IaaS), you manage the OS — AWS only manages the hypervisor below it. With RDS (managed database), AWS manages the database engine patches — you manage the data and access. With Lambda (serverless), AWS manages everything below your code — you just write the function.

Real example from your world: At JP Morgan, if you accidentally leave a CloudWatch log group with sensitive financial data accessible to all IAM users — that's your misconfiguration, not AWS's failure. AWS gave you the tools to restrict it. The Shared Responsibility Model is why security audits always include an IAM review.

Q1 What is AWS and why would a company choose it over on-premise servers? Easy▼

AWS is Amazon's cloud computing platform offering 200+ on-demand services — virtual machines (EC2), object storage (S3), managed databases (RDS), networking (VPC), and more — accessible via API, CLI, or console. You pay only for what you use, billed per second or per request, with no upfront hardware costs.

Companies choose AWS over on-premise for several reasons. Speed: you can provision a database in 5 minutes vs weeks for physical hardware procurement. Elasticity: scale from 2 to 2000 servers in minutes and scale back down when load drops — you pay for what you use. Reliability: AWS operates 33+ regions with multiple AZs per region, making it far easier to build highly available systems than owning physical infrastructure. Managed services: RDS handles database patching, S3 handles replication — operational burden shifts to AWS.

At my current Deloitte engagement with a JP Morgan client, we run the application stack on AWS using EC2, CloudWatch for monitoring, S3 for storage, and Terraform for infrastructure provisioning — replacing what would otherwise require significant physical infrastructure investment on the client side.

Interview tip: Always close with your project. "At JP Morgan..." immediately elevates a generic answer into a credible one. Interviewers are looking for engineers who've actually used the technology, not just read about it.
Q2 Explain the difference between a Region, Availability Zone, and Edge Location. Very common Medium▼

These are three different levels of AWS's physical infrastructure, each serving a different purpose.

Region is a geographic area — like Mumbai (ap-south-1) or N. Virginia (us-east-1) — containing multiple isolated data centres. Every AWS resource you create (EC2, RDS, S3 bucket) lives in a specific Region. Choosing a Region affects latency to your users, cost, and data residency laws like GDPR. Regions are completely isolated from each other — data does not replicate between regions unless you explicitly configure it.

Availability Zone is a physically separate data centre (or cluster of data centres) within a Region, with independent power, cooling, and networking. Mumbai has three AZs. They're close enough for sub-1ms latency between them, but isolated enough that a failure in one doesn't affect others. You deploy applications across multiple AZs to achieve high availability — if one AZ's power grid fails, your app keeps running in other AZs.

Edge Location is a point of presence used exclusively by CloudFront (CDN) and Route 53 (DNS). There are 400+ edge locations worldwide — in cities that don't have full AWS Regions. They cache content and respond to DNS queries close to end users, reducing latency. You cannot deploy compute workloads (EC2, RDS) to edge locations — they are purely for delivery and DNS.

Interview tip: The interviewer wants to hear that AZs are your HA strategy and Edge Locations are for CDN/DNS — not for running apps. If you mix them up, it signals shallow understanding. Draw this out on a whiteboard if given the chance.
Q3 What is the AWS Shared Responsibility Model? Always asked Medium▼

The Shared Responsibility Model defines the security boundary between AWS and the customer. AWS is responsible for the security OF the cloud — physical data centres, hardware, hypervisor, networking, and the managed service software (like the RDS database engine). You are responsible for security IN the cloud — your IAM permissions, security group rules, OS patching on EC2, encryption of your data, and your application code.

The dividing line shifts based on which service you use. With EC2 (IaaS), you manage the OS — AWS only manages the hypervisor beneath. With RDS (managed PaaS-like), AWS manages the engine and patches — you manage the data and who can access it. With Lambda (serverless), AWS manages everything except your function code and IAM permissions.

Practical example: If you misconfigure an S3 bucket to be publicly readable and customer data gets leaked — that's your responsibility, not AWS's. AWS provided the tools (bucket policies, ACLs, S3 Block Public Access) to prevent it. Similarly, if an attacker physically breaks into an Amazon data centre — that's AWS's responsibility entirely.

Interview tip: This question is almost always asked in security-focused rounds and cloud architect roles. The S3 public bucket example is vivid and memorable. For senior roles, also mention the service-level shift (IaaS vs PaaS vs Serverless) — it shows depth.
Q4 Why should you never use the AWS root account for daily operations? Easy▼

The root account is the email and password used to create the AWS account. It has unrestricted access to every AWS service and every account setting — including billing, support plan changes, account closure, and reserved instance purchases. Critically, IAM policies cannot restrict what root can do. There is no permission boundary you can apply to root.

If root credentials are compromised, an attacker can delete all resources, disable CloudTrail audit logs to cover their tracks, transfer the account, or spin up thousands of expensive GPU instances for crypto mining — racking up hundreds of thousands of dollars in charges. AWS will hold you responsible.

Best practice: as soon as you create the AWS account, enable MFA on root, create an IAM admin user for daily use, and lock the root credentials away. For ongoing work, use IAM users with appropriate permissions or IAM roles for programmatic access. For your Spring Boot EC2 deployments, use IAM instance roles — never embed root access keys or any static credentials in your code or configuration files.

Interview tip: Mention IAM roles for EC2 specifically. Many junior engineers know "don't use root" but don't know the IAM Role alternative for application-to-AWS communication. Mentioning it demonstrates senior-level awareness.
Q5 What is the difference between IaaS, PaaS, and SaaS? Where does EC2 fit? Easy▼

These are the three cloud service models, distinguished by how much of the stack the provider manages versus you.

IaaS (Infrastructure as a Service) — you get raw compute, storage, and networking. You manage everything from the OS upwards. EC2 is IaaS. You choose the OS (Amazon Linux, Ubuntu), install Java, deploy your JAR, patch the OS. AWS manages only the hypervisor and physical hardware below.

PaaS (Platform as a Service) — the provider manages the OS and runtime. You focus on code and data. AWS Elastic Beanstalk is PaaS — upload your Spring Boot JAR and it handles EC2 provisioning, load balancer configuration, and auto-scaling automatically. RDS is PaaS for databases — AWS manages the MySQL engine, you manage your schema and queries.

SaaS (Software as a Service) — you use the application directly. No infrastructure visible. AWS WorkMail is SaaS. Salesforce is SaaS. You pay for seats and use the product — no servers, no runtimes, nothing to manage.

Within AWS, the spectrum exists: EC2 = IaaS. ECS Fargate = closer to PaaS (AWS manages the underlying VM for containers). Lambda = furthest towards PaaS/serverless. The more managed the service, the less control you have — but also less operational overhead.

Interview tip: The Elastic Beanstalk + RDS as PaaS examples within AWS are often missed. It shows you understand the cloud service model spectrum is not just between cloud providers but between services within AWS itself.
Q6 What are the different cloud deployment models? Which would you recommend for a fintech app? Medium▼

Public cloud — infrastructure owned and operated by a provider (AWS, Azure, GCP), shared among many tenants but isolated via virtualisation. Cost-effective, globally available, no hardware management. Your EC2 instance shares physical hardware with another company's EC2 — but the hypervisor fully isolates them.

Private cloud — dedicated infrastructure for one organisation. More control and stronger compliance posture, but you manage the hardware. AWS Outposts lets you run AWS hardware on-premises — bridging public and private. Expensive to operate.

Hybrid cloud — combination of public + private. A bank runs sensitive transaction processing on-premise (compliance) but uses AWS for analytics, dev/test environments, and non-sensitive workloads. AWS Direct Connect creates a private, low-latency fibre link between your data centre and AWS.

Multi-cloud — using AWS + Azure + GCP simultaneously. Reduces vendor lock-in, increases resilience, but adds operational complexity significantly.

For a fintech app like JP Morgan's systems: hybrid cloud is the typical answer. Core banking systems and sensitive customer data often remain on-premise for regulatory and compliance reasons (RBI mandates, SEBI data localisation requirements). Non-sensitive services, analytics, and developer tooling move to AWS. This is exactly the architecture at my current engagement.

Interview tip: Mention the JP Morgan / fintech context explicitly. RBI and SEBI data localisation requirements are real constraints that justify hybrid cloud for Indian fintech. This level of context is what separates a senior engineer's answer from a junior one.
Q7 What are the 6 pillars of the AWS Well-Architected Framework? Hard▼

The Well-Architected Framework is AWS's collection of architectural best practices, organised into 6 pillars. AWS provides a free Well-Architected Tool in the console to audit your architecture against these pillars. Knowing all 6 signals architect-level thinking.

1. Operational Excellence — automate operations, monitor systems, continuously improve. In practice: use Terraform/CloudFormation instead of manual console clicks, use CloudWatch alarms, do post-incident reviews. Principle: run and monitor production like it's a product, not an afterthought.

2. Security — protect data and systems at every layer. IAM least-privilege, encryption at rest and in transit, CloudTrail audit logging, GuardDuty threat detection. Principle: never trust, always verify.

3. Reliability — recover from failures, scale to meet demand, test recovery. Multi-AZ deployments, Auto Scaling, regular chaos engineering, RDS Multi-AZ, S3 cross-region replication. Principle: everything fails eventually — design for recovery, not just prevention.

4. Performance Efficiency — use the right service for the right job. Lambda for sporadic event-driven work, EC2 for steady long-running processes, ElastiCache for read-heavy workloads, CloudFront for globally distributed content. Principle: don't over-provision or under-provision.

5. Cost Optimisation — eliminate waste. Reserved Instances for steady-state workloads (up to 72% savings), Spot Instances for batch jobs (up to 90% savings), right-sizing EC2, deleting unused resources. Principle: pay only for what you need.

6. Sustainability (added 2021) — minimise environmental impact. Use managed services (more energy-efficient than self-managed), right-size resources, avoid idle infrastructure. Principle: the most sustainable server is the one you don't run.

Interview tip: You don't need to recite all 6 with perfect detail unprompted. But if asked, knowing all 6 names and giving one real sentence each is impressive. The interviewer is checking if you think architecturally — beyond just writing code.
Q8 How would you design a highly available architecture on AWS for a backend API? Senior-level Hard▼

High availability means the system continues operating even when individual components fail. The core strategy is: eliminate every single point of failure by redundancy across Availability Zones.

Compute layer: Run EC2 instances in at least two AZs, placed in an Auto Scaling Group. The ASG maintains a minimum instance count and launches replacements automatically if an instance fails health checks. An Application Load Balancer sits in front, distributing traffic across healthy instances in both AZs. If an entire AZ fails, the ALB stops routing to it and all traffic flows to the surviving AZ.

Database layer: Use RDS with Multi-AZ enabled. AWS creates a synchronous standby replica in a second AZ. If the primary fails, AWS automatically promotes the standby — typically within 1–2 minutes. For read-heavy workloads, add Read Replicas in additional AZs.

Storage layer: S3 is already highly available by design — AWS replicates data across a minimum of three AZs within the Region automatically. You get 11 nines of durability for free.

DNS layer: Route 53 with health checks. If your primary Region becomes unhealthy, Route 53 can failover to a secondary Region using latency-based or failover routing policies.

For my Hotel Microservices system: each service runs in two AZs behind an ALB, each service's RDS is Multi-AZ enabled, and SQS (Kafka equivalent on AWS) decouples services so a slow payment service cannot cascade and crash the booking service.

Interview tip: Structure your answer as layers: compute → database → storage → DNS. Then mention RTO and RPO: "The question is always what your Recovery Time Objective and Recovery Point Objective are. Multi-AZ RDS gives sub-2min RTO and near-zero RPO." That phrase alone marks you as senior-level.
Q9 What is IAM and what is the difference between a User, Group, and Role? Medium▼

IAM (Identity and Access Management) is AWS's global permission system. It controls which identities can perform which actions on which AWS resources. Every API call to AWS — whether from the console, CLI, or your Spring Boot app — is authenticated and authorised through IAM.

User — represents a real person. Has long-lived credentials: a password for console login and/or an access key + secret for CLI/SDK access. You create a user for yourself (the human developer) or for a CI/CD system that needs long-lived access. Should have MFA enabled. Avoid sharing user credentials between people.

Group — a collection of Users. Attach a policy to a Group and all users in it inherit those permissions. For example: a "Developers" group with EC2 and S3 read/write access. A "DBAdmins" group with RDS full access. Managing permissions at the Group level is far easier than managing per-User policies.

Role — a temporary identity that can be assumed by an AWS service (EC2, Lambda, ECS), another AWS account, or a federated identity (Google, Okta). Unlike a User, a Role has no long-lived credentials — it generates temporary security tokens (STS). This is how your EC2 instance gets permission to read from S3 without you embedding AWS keys in your application.properties file. The EC2 assumes the role, gets a temporary token, makes the S3 API call, token expires.

Interview tip: The Role explanation is where most candidates stumble. "The EC2 assumes a role at runtime and gets a temporary token — no static keys in the application" is a senior-level answer. Explicitly mentioning STS (Security Token Service) as the mechanism shows depth.
Q10 What is the AWS Free Tier and what are its limits? What happens if you exceed them? Easy▼

The AWS Free Tier is a set of services available at no charge for the first 12 months after account creation, plus some services that are always free. Key limits: EC2 — 750 hours/month of t2.micro or t3.micro (Linux), sufficient for one instance running 24/7. S3 — 5GB storage, 20,000 GET requests, 2,000 PUT requests per month. RDS — 750 hours/month of db.t2.micro or db.t3.micro, 20GB storage. Lambda — 1 million requests and 400,000 GB-seconds of compute per month, permanently free (not just 12 months). CloudWatch — 10 custom metrics, 5GB log ingestion per month, permanently free at this tier.

If you exceed the free tier, AWS charges automatically to your registered credit card at standard rates. A forgotten EC2 instance running for a month costs around $8–15 on t2.micro in ap-south-1. Accidentally leaving an NAT Gateway running is more costly — about $35/month just in fixed charges. This is why billing alerts are non-negotiable before starting any lab.

Best practice for learners: always terminate (not just stop) EC2 instances after lab sessions, delete unused RDS instances, and set a zero-spend budget alert so you get an email the moment any charge appears.

Interview tip: This question tests cloud cost awareness — a real-world skill. Knowing that NAT Gateway has a fixed hourly cost (not just data transfer) signals production experience. Mentioning the zero-spend alert shows you think defensively about infrastructure costs.
