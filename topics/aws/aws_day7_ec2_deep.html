User data is not a simple startup script. It is the entry point into cloud-init — a complete instance configuration framework used by every major Linux distribution. Understanding cloud-init's execution phases, module system, and failure modes is what separates engineers whose automation works reliably from those whose EC2 instances randomly come up misconfigured and nobody knows why.
cloud-init — what it is and how it relates to user data
The full picture behind the "startup script" abstraction

cloud-init is an open-source multi-distribution package that handles early cloud instance initialisation. It is pre-installed on Amazon Linux 2023, Ubuntu, RHEL, and every other major AWS AMI. When an EC2 instance boots for the first time, cloud-init is the process that fetches your user data from IMDS, interprets it, and executes it. User data is the input to cloud-init — cloud-init is the engine that processes it.

cloud-init supports multiple input formats. The format is determined by the first line of the user data content:

#!/bin/bash — a shell script. Executed directly. Most common format for simple bootstrapping.
#cloud-config — a YAML document following cloud-config syntax. Declarative — you describe the desired state (packages to install, files to create, users to add) and cloud-init translates it to commands. More structured and idempotent than shell scripts.
Content-Type: multipart/mixed — multiple parts combined, allowing both a cloud-config document and a shell script in one user data payload.

How cloud-init fetches user data — and the IMDSv2 interaction: cloud-init queries IMDS to get user data: http://169.254.169.254/latest/user-data. On instances where IMDSv2 is required (which all your instances should be after Day 5), cloud-init correctly handles the two-step IMDSv2 token request. The cloud-init version matters here: versions before 18.4 do not support IMDSv2. If you build an AMI with an old cloud-init version and then enforce IMDSv2, your user data will never execute on new instances — they'll boot with no user data applied, silently. Amazon Linux 2023 ships with a current cloud-init version. Ubuntu 20.04 LTS ships cloud-init 20.x — IMDSv2 compatible. Always check cloud-init version when working with custom AMIs that might have older software.

The instance data hierarchy cloud-init reads: cloud-init reads from multiple IMDS paths: user data (your script), meta-data (instance ID, AZ, hostname, IAM role name), and vendor-data (AWS-specific configuration injected by the hypervisor — things like the SSH public key from your key pair). The SSH public key injection you rely on when you create an EC2 with a key pair is cloud-init reading the key from http://169.254.169.254/latest/meta-data/public-keys/0/openssh-key and writing it to ~ec2-user/.ssh/authorized_keys. If cloud-init fails, SSH access also fails — because the key was never written.

#!/bin/bash = shell script, #cloud-config = YAML declarative cloud-init before 18.4 doesn't support IMDSv2 — check version in custom AMIs
cloud-init execution phases — the exact order everything runs
Five phases, each running at a different point in the boot process — knowing this prevents race conditions

cloud-init execution is split into five phases, each corresponding to a systemd service that runs at a specific point during boot. Understanding the phases is critical for writing user data that doesn't fail due to dependencies not being ready yet.

Phase 1 — cloud-init-generator (very early boot): Runs as a systemd generator before networking is up. Determines if this is a first boot, subsequent boot, or if cloud-init is disabled. Reads the kernel command line. Almost nothing you care about happens here.

Phase 2 — cloud-init local (before networking): Runs before the network is configured. Fetches data sources that don't require network (local disk only in this phase). Sets up the hostname using metadata from the data source that doesn't need network access (for EC2, the hostname is derived from the private IP address). This is why your instance gets its hostname before network is up.

Phase 3 — cloud-init network (after networking, before services): Network is up. cloud-init fetches user data from IMDS. Applies network configuration. Runs the init modules: sets timezone, locale, writes files, creates users, adds SSH authorized keys. This is where SSH key injection happens — the key is written to authorized_keys during this phase.

Phase 4 — cloud-config (after init modules): Runs the config modules: installs packages (yum install, apt-get install), runs commands, configures services, writes configuration files. This is the main execution phase for most user data content.

Phase 5 — cloud-final (last): Runs the final modules: executes your shell scripts (if using #!/bin/bash), runs puppet/chef/ansible if configured, sends completion signals (used by CloudFormation cfn-signal). Your shell script runs in this final phase — meaning if your script depends on something installed in Phase 4, it works correctly because Phase 4 always precedes Phase 5.

Why the phase order matters for your Spring Boot deployment: If you use a #cloud-config user data document that installs Java (Phase 4 — package installation) and then runs a command to start your Spring Boot JAR (also Phase 4 — runcmd), the runcmd executes after the package installation in the same phase. But if you use a shell script (#!/bin/bash) that does yum install java-17 and then java -jar app.jar, both happen in Phase 5 — sequentially, as written. Either approach works for simple cases. The issue arises when you mix formats: a cloud-config document and a separate shell script in a multipart user data — the cloud-config runs first (Phase 4) and the shell script runs second (Phase 5), which is often the right order for "install dependencies then run app."
The "user data ran but packages aren't installed" race condition: An engineer writes a user data script that installs packages and immediately runs them, but the script completes in seconds while package installation takes 2-3 minutes due to package manager initialization. This happens when the script launches a background process (yum install &) and the script exits before installation completes. When the ASG's health check runs before installation finishes, the instance appears healthy (SSH works, user data script returned 0), but the application isn't running. Solution: never background-execute package installation in user data scripts. Block until installation completes. Use set -e so the script fails fast if any command fails.

Debugging user data failures — the systematic approach
The four log locations and the exact commands to diagnose every user data failure

User data failures are silent by default — the instance boots, SSH works, but the application is not running. Without knowing where to look, engineers spend hours randomly checking things. There are four specific locations that tell the complete story.

# LOG LOCATION 1: cloud-init main log — execution timeline sudo cat /var/log/cloud-init.log # Shows: every phase, every module, timestamps, and critical errors # Look for: "Traceback" (Python exception), "FAILED", or "Error" # The last few lines tell you where execution stopped # LOG LOCATION 2: cloud-init output log — your script's stdout/stderr sudo cat /var/log/cloud-init-output.log # Shows: everything your script printed to stdout or stderr # This is the log you see your echo statements in # If this file is empty, your script never ran (look at cloud-init.log) # LOG LOCATION 3: user-data.log (if script redirects there) sudo cat /var/log/user-data.log # Only exists if your script redirects output there (as in the template above) # More readable than cloud-init-output.log because it's only your script's output # LOG LOCATION 4: systemd journal for the app service sudo journalctl -u getwork-app --no-pager -n 100 # Shows Spring Boot application logs after the service started # If user data ran fine but app failed, the error is here # CHECK: Did cloud-init complete successfully? sudo cloud-init status --long # Outputs: status: done (success) or status: error (with details) # This is the fastest first check — one command tells you pass/fail # CHECK: What exact user data did the instance receive? # (Requires IMDSv2 token on instances with --http-tokens required) TOKEN=$(curl -sf -X PUT http://169.254.169.254/latest/api/token \ -H "X-aws-ec2-metadata-token-ttl-seconds: 21600") curl -sf http://169.254.169.254/latest/user-data \ -H "X-aws-ec2-metadata-token: $TOKEN" # Prints the exact user data the instance received — confirms it was set correctly # CHECK: Re-run user data manually for debugging (without rebooting) # cloud-init normally runs user data once. The "user data is base64-encoded" debugging trap: When you pass user data via the CLI using --user-data file://script.sh, the CLI automatically base64-encodes it. When you view user data via the console, it shows the decoded version. But if you pass --user-data fileb://script.sh (binary), the CLI does NOT encode it — and if the content is already base64-encoded, the instance receives double-encoded data. cloud-init tries to execute base64 gibberish, fails silently, and your script never runs. Always use file:// (text mode — auto-encodes) not fileb:// (binary mode — no encoding) for user data scripts.
SSM Run Command — the alternative to user data for post-launch configuration: User data runs once on first boot and is immutable. If you need to push a configuration change to running instances (e.g. update an environment variable, rotate a configuration file), SSM Run Command is the mechanism. You send a command to one or many instances, it executes, and the output is available in SSM. No SSH required, no port 22 open, full audit trail in CloudTrail. This is the production mechanism for operational tasks on running fleets.
AMIs are the unit of deployment for EC2-based systems. The difference between teams that deploy reliably in 60 seconds and teams that spend 10 minutes bootstrapping every instance is entirely in how they build and manage AMIs. Understanding AMI internals — how EBS snapshots underpin them, how the block device mapping works, how AMI sharing and copying work — is what allows you to design immutable infrastructure correctly.
How AMIs actually work — the EBS snapshot relationship
An AMI is not a disk image file. It is a metadata document pointing to one or more EBS snapshots.

When most engineers hear "AMI," they picture a large disk image file stored somewhere. The reality is different and important. An AMI is a metadata object in AWS's registration system that contains: a name, description, architecture, virtualisation type, root device type, and most critically — a block device mapping that specifies one or more EBS snapshots.

When you create an AMI from a running EC2 instance using aws ec2 create-image, AWS does the following: creates EBS snapshots of all volumes attached to the instance, registers a new AMI that points to those snapshots in its block device mapping, and returns the AMI ID. The AMI itself contains no data — the data lives in the EBS snapshots. The AMI is a pointer.

What this means for launch speed: When you launch an EC2 instance from an AMI, AWS does NOT copy the entire EBS snapshot to a new volume before booting the instance. EBS uses a copy-on-write mechanism. The new volume is created instantly, pointing to the same underlying data as the snapshot. The instance boots immediately. As the OS reads data that hasn't been explicitly written, it fetches from the snapshot storage. As the OS writes data, it writes to the new volume's private storage. This is why an EC2 instance can boot in under 60 seconds from a 100GB AMI — it doesn't wait to copy 100GB first. The catch: the first access to each block of data that hasn't been read yet has slightly higher latency as it's fetched from snapshot storage. This is called the "cold storage" penalty. For latency-sensitive applications, pre-warm new volumes using fio or dd to read every block before the instance enters the load balancer target group.

Block device mapping in an AMI: The block device mapping defines which EBS snapshot maps to which device. A typical Spring Boot server AMI might have: /dev/xvda → snapshot-001 (8GB, root volume, gp3), /dev/xvdb → snapshot-002 (50GB, application data volume, gp3). When the instance launches, each mapped device is instantiated as an EBS volume from its snapshot. Additional devices can be added at launch time that are NOT in the AMI (e.g., an ephemeral instance store volume). Devices in the AMI that you specify as "NoDevice" are suppressed — this lets you launch from an AMI without creating one of its defined volumes.

The AMI deregistration / snapshot deletion ordering mistake: You deregister an old AMI you no longer need. The deregistration succeeds. But the EBS snapshots that the AMI pointed to are NOT automatically deleted — they continue to exist and you continue to be charged for them. Many teams deregister old AMIs and discover months later they're paying for hundreds of orphaned snapshots. The correct sequence: first copy all instance IDs that were launched from this AMI (to confirm no instances depend on it), then deregister the AMI, then explicitly delete the EBS snapshots it pointed to. AWS provides scripts for this cleanup, and EC2 Image Builder manages this lifecycle automatically if you use it.
AMI baking strategy — the spectrum from vanilla to fully baked
Four approaches to AMI construction, each with different trade-offs for boot time, flexibility, and maintenance cost

The "right" AMI strategy depends on your deployment frequency, team size, and tolerance for boot-time latency. These are the four main approaches in order of increasing pre-baking:

Approach 1 — Plain OS AMI + full user data bootstrapping. Start from the AWS-provided Amazon Linux 2023 AMI. User data script installs all software (Java, your JAR, monitoring agents, etc.) on every launch. Boot time: 5-10 minutes. Advantages: always uses latest package versions, no AMI maintenance required. Disadvantages: slow scaling — Auto Scaling cannot add capacity quickly because new instances take 10 minutes to become ready. Suitable for: dev environments, non-time-sensitive scaling, teams without a CI/CD pipeline for AMI baking.

Approach 2 — Partially baked AMI + minimal user data. Pre-install Java, monitoring agents, and OS-level tools. Bake this into an AMI. User data fetches your application JAR from S3 and writes environment-specific configuration. Boot time: 90-120 seconds. Advantages: fast enough for Auto Scaling, most static dependencies pre-installed. Disadvantages: AMI must be updated when Java version changes or agents need patching.

Approach 3 — Fully baked AMI (immutable infrastructure). The application JAR, all dependencies, all configuration templates, and even environment-specific config (injected at bake time) are in the AMI. User data does minimal work — perhaps set the hostname and register with a service registry. Boot time: 60-90 seconds. Netflix pioneered this pattern. Advantages: fastest possible scaling, complete reproducibility (every instance that launches from AMI v47 is identical), rollback is simply pointing the ASG at AMI v46. Disadvantages: every application change requires baking a new AMI. CI/CD pipeline must include AMI baking as a step.

Approach 4 — Container-based AMI. The AMI only contains Docker and your container runtime (ECS agent or similar). The application runs as a Docker container pulled from ECR at launch time. Boot time: 2-3 minutes (AMI boot + Docker pull). This decouples the OS-level AMI from the application, letting you update the application independently of the AMI. This is essentially ECS on EC2, covered on Day 12.

The recommended approach for GetWork and your Hotel Microservices: Approach 2 with a baking pipeline. Maintain a base AMI with Java, CloudWatch agent, SSM agent, and common tools pre-installed. In your CI/CD pipeline: on every main branch merge, bake a new AMI with the latest JAR embedded, update the Launch Template to reference the new AMI, and trigger a rolling update of the Auto Scaling Group. New instances boot in 90 seconds rather than 10 minutes, your application version is encoded in the AMI ID (enabling rollback), and the base AMI is refreshed monthly for security patches.
Packer — the standard tool for AMI baking: HashiCorp Packer is the industry-standard tool for building AMIs programmatically. A Packer template defines: the source AMI, a list of provisioners (shell scripts, Ansible playbooks, file uploads), and the target AMI name and tags. Packer launches a temporary EC2 instance, runs your provisioners, creates an AMI from the result, and terminates the temporary instance. The entire process takes 5-10 minutes and produces a tagged, versioned AMI. Since you already use Terraform at Deloitte, adding Packer to your pipeline is natural — they're from the same company and integrate seamlessly.
EC2 Image Builder — AWS-native AMI baking pipeline
Automated, scheduled, tested AMI builds with built-in vulnerability scanning

EC2 Image Builder is AWS's managed service for building, testing, and distributing AMIs. Instead of maintaining Packer templates and a build EC2 instance, Image Builder provides a GUI and API-driven pipeline with automated security scanning, cross-region distribution, and AMI lifecycle management built in.

Image Builder components:

Image recipe — defines the base AMI (parent image) and the build components to apply. Components are versioned documents containing install scripts, configuration commands, and validation tests. AWS provides hundreds of pre-built components (install latest Java, configure CIS hardening, install CloudWatch agent). You write custom components for your application-specific steps.

Infrastructure configuration — specifies which EC2 instance type to use for building (a larger instance type speeds up the build), the VPC/subnet, IAM role, security groups, and whether to terminate the build instance on failure (useful for debugging — keep it running to inspect state).

Distribution configuration — where the finished AMI is copied. Can distribute to multiple regions and accounts simultaneously. Critical for multi-region architectures: your build runs in ap-south-1 and the AMI is automatically copied to ap-southeast-1 and us-east-1.

Image pipeline — ties everything together with a schedule (build nightly) or trigger (build on new base AMI release). Runs the build, runs the test components (integration tests against the new AMI), and distributes on success.

Inspector integration — vulnerability scanning in the AMI build: EC2 Image Builder integrates with Amazon Inspector to scan the built AMI for known CVEs before distribution. If Inspector finds critical vulnerabilities, the pipeline can be configured to fail the build rather than distributing a vulnerable AMI. This prevents your fleet from ever running instances with known exploitable vulnerabilities — a security control that would be expensive and complex to build yourself. Combined with the scheduled nightly rebuild, every new instance launches from an AMI that was scanned for CVEs in the last 24 hours.
AMI lifecycle management — the orphaned snapshot problem solved: EC2 Image Builder tracks which AMIs it created and can automatically deregister old AMIs and delete their snapshots based on a retention policy (keep the last 5 versions). This eliminates the orphaned snapshot billing problem entirely. When you manage AMIs manually, you often forget to clean up old ones. Image Builder's lifecycle policies do this automatically. For a team releasing 2-3 new AMIs per day, without lifecycle management you'd accumulate thousands of AMIs and snapshots within a year.
AMI sharing, copying, and region mechanics
AMI IDs are region-specific — and what that means for multi-region deployment automation

AMI IDs are unique within a region but not globally. The same AMI has a different ID in each region it exists in. ami-0abcdef1234567890 in ap-south-1 is a completely different AMI from ami-0abcdef1234567890 in us-east-1 — or more likely, the second ID doesn't exist in us-east-1 at all. This is a constant source of automation failures when engineers hardcode AMI IDs.

The hardcoded AMI ID in Terraform that breaks multi-region deployments: An engineer writes a Terraform module with ami = "ami-0abcdef1234567890". This works for ap-south-1. Six months later, the team needs to deploy to ap-southeast-1 for Singapore users. The Terraform apply fails: "InvalidAMIID.NotFound". The AMI doesn't exist in Singapore. The correct approach: use an aws_ami data source that dynamically looks up the latest AMI matching your criteria (owner, name filter, architecture) in whatever region Terraform is deploying to. Never hardcode AMI IDs.
# Correct Terraform pattern — dynamic AMI lookup data "aws_ami" "app_ami" { most_recent = true owners = ["self"] # Your account's AMIs filter { name = "name" values = ["getwork-app-*"] # Name prefix from your baking pipeline } filter { name = "state" values = ["available"] } filter { name = "tag:version" values = ["${var.app_version}"] # Specific version from CI/CD } } # Use in Launch Template resource "aws_launch_template" "app" { image_id = data.aws_ami.app_ami.id # ...rest of template }

AMI copying across regions: aws ec2 copy-image --source-image-id ami-xxx --source-region ap-south-1 --region ap-southeast-1 --name "getwork-app-v47". This creates a new AMI in the destination region with new EBS snapshots (the underlying snapshot data is copied). The copy is asynchronous — the destination AMI state is "pending" until the snapshot copy completes. For large AMIs or many regions, copying can take 15-30 minutes. EC2 Image Builder's distribution configuration handles this automatically and in parallel.

EBS is not just "disk storage for EC2." It is a network-attached block storage service with distinct volume types, each optimised for different I/O patterns. Choosing the wrong EBS volume type causes performance degradation that looks like an application problem. Understanding IOPS, throughput, burst credits, and the gp2-to-gp3 economics is directly relevant to your Spring Boot deployments.
EBS architecture — why it's network-attached and what that means
EBS is not local disk. It is a distributed storage service accessed over a dedicated network fabric.

EBS (Elastic Block Store) volumes are not physically inside the EC2 host server. They are stored in a separate, dedicated storage infrastructure and accessed over a dedicated high-performance network built into the Nitro card. This network is separate from the instance's data network — EBS traffic does not compete with your application's network traffic (unless the instance is not EBS-optimised, which is true of older instance types).

The practical implication: EBS latency is network latency, not disk latency. A local NVMe SSD has sub-100µs latency. EBS gp3 has typical latency of 1-5ms (single-digit milliseconds) because packets must travel from the EC2 host to the EBS storage nodes and back. For most application workloads this is fine — a Spring Boot app reading config files or writing logs doesn't notice 3ms disk latency. For I/O-intensive database workloads (MySQL on EC2 vs RDS, or self-hosted Redis), this latency matters significantly.

EBS Multi-Attach — one volume, multiple instances: EBS io1 and io2 volumes support Multi-Attach — a single EBS volume can be simultaneously attached to up to 16 EC2 instances within the same AZ. This is an advanced feature designed for clustered applications that manage their own write locking (like clustered databases, Oracle RAC). It is NOT a general-purpose shared filesystem — without application-level coordination, multiple instances writing to the same blocks causes data corruption. For shared filesystem needs, use EFS (Elastic File System — a managed NFS) or FSx instead. Multi-Attach is for specific software that knows how to coordinate writes at the block level.

EBS encryption: EBS volumes can be encrypted at rest using AWS KMS (either the default EBS CMK or a customer-managed key). Encryption/decryption happens transparently on the Nitro card — zero CPU overhead on the instance, no performance impact. You cannot add encryption to an existing unencrypted volume — you create a snapshot, copy the snapshot with encryption enabled, then create a new volume from the encrypted snapshot. AWS allows setting account-level default encryption — every new EBS volume is encrypted by default. This should be enabled in every production account. Combined with an SCP preventing creation of unencrypted volumes, you have a hard control that ensures all block storage is encrypted.

The EBS volume and AZ lock-in: An EBS volume exists in exactly one Availability Zone. It can only be attached to an EC2 instance in the same AZ. If your EC2 in ap-south-1a has a data volume attached, and that EC2 terminates, you can start a new EC2 in ap-south-1a and reattach the volume. But you cannot attach that volume to an EC2 in ap-south-1b — different AZ, different physical storage infrastructure. This becomes a problem for HA designs. If you store stateful data on an EBS volume (not in a managed service like RDS or S3), and that AZ fails, the data is inaccessible until the AZ recovers. Solution: use EBS snapshots (stored in S3, AZ-independent) for persistent data backup, or use RDS/ElastiCache/DynamoDB which handle AZ redundancy for you.

IOPS vs throughput — they are not the same metric: IOPS (Input/Output Operations Per Second) measures how many read/write operations can complete per second, regardless of size. Throughput (MB/s) measures how many bytes can transfer per second. For small random I/O (4KB database writes — reading an index, writing a record), IOPS is the bottleneck. For large sequential I/O (streaming 1MB blocks — video processing, log files, Hadoop), throughput is the bottleneck. A database on gp3 hitting IOPS limits exhibits high I/O wait and slow query times. A video transcoding workload on gp3 hitting throughput limits shows high CPU wait on I/O with slow processing. Diagnose using CloudWatch's VolumeReadOps/VolumeWriteOps (IOPS) and VolumeReadBytes/VolumeWriteBytes (throughput) metrics.
gp2 vs gp3 — why you should migrate every gp2 volume you have
Same price, but gp3 gives you independent IOPS and throughput control that gp2 cannot

gp3 replaced gp2 as AWS's recommended general-purpose SSD volume in December 2020. They have the same price per GB ($0.08/GB/month in ap-south-1), but gp3 has a fundamentally better design.

gp2 — the legacy model
IOPS is linked to volume size: 3 IOPS per GB, minimum 100, maximum 16,000. A 100GB gp2 volume gets 300 IOPS. To get 3,000 IOPS you need a 1,000GB volume — even if you only use 50GB of data.

Burst credits: Volumes below 1,000GB earn burst credits when IOPS is below baseline, and spend credits when above baseline (up to 3,000 IOPS). A small volume can burst to 3,000 IOPS temporarily. When credits are exhausted, IOPS drops to baseline. This burst behavior is unpredictable — performance varies with workload history, not just current load.

Maximum throughput: 250 MB/s, regardless of IOPS provisioned.
gp3 — the correct modern choice
IOPS is independent of size. Every gp3 volume baseline: 3,000 IOPS and 125 MB/s throughput, regardless of size. A 1GB gp3 gets the same baseline as a 1,000GB gp3. No burst credits — performance is consistent.

Additional IOPS and throughput can be purchased independently: up to 16,000 IOPS at $0.005/IOPS/month above 3,000, and up to 1,000 MB/s throughput at $0.04/MB-s/month above 125.

You provision exactly the IOPS your workload needs, paying only for what you use. No need to over-provision volume size just to get IOPS.
The gp2 to gp3 migration calculation: A Spring Boot app has a 200GB gp2 root volume (bought large for headroom). It gets 600 IOPS (200 × 3). The app actually needs 2,000 IOPS. With gp2, you'd need to resize to 667GB to get 2,000 IOPS, even though you only need 200GB of space — paying for 467GB of unused disk just to get more IOPS. With gp3: keep the 200GB volume, provision 2,000 IOPS explicitly. Pay for 200GB ($16/month) + 0 additional IOPS charge (2,000 < 3,000 baseline). The gp3 volume is cheaper and has predictable performance. AWS now allows you to modify any gp2 volume to gp3 in-place without detaching or stopping the instance — zero downtime migration.
The in-place volume type modification: aws ec2 modify-volume --volume-id vol-xxx --volume-type gp3. The modification happens online — the volume remains attached and the instance continues running. During the modification (which can take minutes to hours for large volumes), I/O performance may fluctuate slightly. After modification, the volume has the new type's characteristics. You can modify type, size, IOPS, and throughput in the same operation. You can increase size but NOT decrease it — another reason to start small and scale up rather than provision large volumes.
EBS optimisation patterns for Spring Boot on EC2
The specific settings and monitoring metrics that determine whether EBS is helping or hurting your application

EBS-optimised instances: Modern EC2 instance types (all Nitro-based instances: t3, m5, c5, r5, and newer) are EBS-optimised by default — they have dedicated bandwidth between the CPU and EBS storage that doesn't compete with network traffic. Older instances (t2, m4, c4) are not EBS-optimised by default — EBS and network traffic share the same NIC, causing contention. If you're still running t2 instances (avoid for production), explicitly enable EBS optimisation in the instance settings. The cost is negligible.

EBS CloudWatch metrics to monitor for your Spring Boot app:

VolumeReadOps / VolumeWriteOps — IOPS being used. Compare against your volume's provisioned IOPS. If this is consistently near the limit, you'll see increased latency.
VolumeQueueLength — How many I/O requests are waiting because the volume can't process them fast enough. Should be near 0 for a healthy volume. Consistently > 1 means your IOPS is the bottleneck.
VolumeThroughputPercentage — For io1/io2: what percentage of provisioned throughput is being used.
BurstBalance — For gp2: percentage of burst credits remaining. If this is consistently at 0, your workload exceeds gp2 baseline and you should migrate to gp3 with provisioned IOPS.

The EBS volume pre-warming technique: New EBS volumes created from snapshots have all blocks in a "cold" state — first access to each block has higher latency than subsequent accesses (the copy-on-write fetch from snapshot storage). For latency-sensitive production applications (databases, Redis on EC2), pre-warm the volume before accepting traffic: sudo fio --filename=/dev/xvda --rw=read --bs=128k --iodepth=32 --ioengine=libaio --direct=1 --name=volume-initialize. This reads every block, forcing them all into the "warm" (local copy) state. After completion, every subsequent block access has full-speed EBS performance. Do this in your user data script after volume mount, before starting the application.
The EBS throughput limit masquerading as CPU bottleneck: A data processing Spring Boot service processes 500MB log files from EBS. CPU is at 60%, which seems fine — engineers increase CPU and it doesn't help. Root cause: the EBS volume is a gp3 with 125MB/s baseline throughput. Reading 500MB takes 4 seconds. The CPU is idle for most of that time, waiting for disk reads. CloudWatch shows VolumeReadBytes near 125MB/s (the ceiling). Increasing CPU does nothing — the bottleneck is disk throughput. Fix: provision higher throughput on the gp3 volume ($0.04/MB-s above 125) or switch to st1 HDD (500MB/s throughput, specifically designed for streaming large files).

If you are still using Launch Configurations in Terraform or the console, you are using deprecated functionality that AWS has stopped adding features to. Launch Templates support versioning, partial specification, inheritance, and every modern EC2 feature that Launch Configurations do not. Every Auto Scaling Group should use a Launch Template.
Launch Template mechanics — what it defines and how it's used
The complete specification that determines what every EC2 instance launched from it looks like

A Launch Template is a versioned document that specifies the complete configuration for launching EC2 instances. Every field is optional — a template can specify only the fields you want to control and leave the rest for the caller to specify at launch time. This partial specification model is what makes Launch Templates composable.

Key fields in a Launch Template: AMI ID, instance type, key pair name, security group IDs, subnet ID (or left unspecified for ASG to determine), EBS volume configuration (size, type, IOPS, encryption, delete-on-termination), IAM instance profile, user data (base64-encoded), instance tags, metadata options (IMDSv2 enforcement), network interfaces, placement (Availability Zone, Placement Group), capacity reservation, monitoring (detailed CloudWatch), and Spot instance options.

The critical Launch Template fields for production security: Always set these in every Launch Template:
MetadataOptions.HttpTokens = required — enforces IMDSv2 (prevents SSRF credential theft, Day 5)
MetadataOptions.HttpPutResponseHopLimit = 1 — prevents containers inside the instance from querying IMDS (IMDS response can only be forwarded once — to the EC2 itself, not to a container inside it)
BlockDeviceMappings[].Encrypted = true — EBS volumes encrypted at rest
BlockDeviceMappings[].DeleteOnTermination = true (for root), false (for data volumes) — ensures data volumes survive instance termination

Launch Template versioning — the deployment and rollback mechanism
How versions enable zero-downtime deployments and instant rollbacks in an Auto Scaling Group

Launch Templates support versioning. Each change to a Launch Template creates a new version. Versions are numbered sequentially (1, 2, 3...) and two special aliases exist: $Latest (always the most recent version) and $Default (whichever version you designate as the default). An Auto Scaling Group is configured to use a specific version, $Latest, or $Default.

The versioning-based deployment workflow:

1. CI/CD pipeline bakes a new AMI (getwork-app-v48), tags it appropriately.
2. Pipeline creates a new Launch Template version with the new AMI ID:
aws ec2 create-launch-template-version --launch-template-id lt-xxx --source-version 5 --version-description "v6 - app v1.2.3" --launch-template-data '{"ImageId":"ami-new"}'
3. Pipeline updates the ASG to use the new version and triggers an instance refresh:
aws autoscaling start-instance-refresh --auto-scaling-group-name getwork-asg --preferences '{"MinHealthyPercentage":90}'
4. ASG performs a rolling replacement — terminates old instances and launches new ones with the new version, maintaining at least 90% capacity throughout.
5. If the new version has issues: aws autoscaling cancel-instance-refresh stops the rollout. Point the ASG back to the previous Launch Template version. Trigger another refresh to roll back. Done.

The $Default version alias for safe rollbacks: Configure your ASG to use the $Default Launch Template version (not $Latest). Your CI/CD pipeline creates a new version but does NOT change the default. The deployment is staged: first deploy to a canary group (10% of traffic) using the new version explicitly, monitor metrics for 10 minutes, then set the new version as default and trigger a full rollout. If canary fails, the default version is still the old version — the rest of the fleet is unaffected. Using $Latest in the ASG is dangerous because any change to the template immediately affects new instances launched during scaling events, without a staged rollout.
Launch Template inheritance — reducing duplication across environments: Launch Templates support inheritance: a child template can specify a parent template version to inherit from. Fields in the child override the parent; fields not specified in the child are inherited. This lets you maintain a base template (AMI, security hardening, IMDSv2 settings, common tags) and environment-specific templates (dev uses t3.micro, prod uses m5.large; different security groups, different user data for environment-specific config). Changing the base template propagates to all children on next use. This reduces configuration drift between environments — a security setting added to the base template automatically applies to all environments.
Instance type requirements — attribute-based instance selection
Describe what you need, not what type you want — AWS picks the optimal instance automatically

Traditional Launch Templates and configurations specify an exact instance type: t3.small. This creates fragility — if t3.small is unavailable in the target AZ due to capacity constraints, the launch fails. For Auto Scaling Groups using Spot Instances (where capacity availability varies by instance type and AZ), specifying a single type is almost always wrong.

Launch Templates support attribute-based instance type selection: instead of specifying t3.small, you specify requirements: minimum 2 vCPUs, minimum 2GB RAM, Intel or AMD architecture, burstable instances allowed. AWS selects from all instance types in the region that satisfy your requirements. For an ASG with Spot pricing, this allows AWS to fulfil your request from whichever instance type has available capacity and the best current Spot price — automatically.

Mixed instance ASG — the correct pattern for Spot + On-Demand: For most production workloads you want a mix: a baseline of On-Demand instances (always available, predictable cost) plus additional Spot capacity that scales out during high load (cheap but interruptible). Configure the ASG with InstancesDistribution: OnDemandBaseCapacity (the minimum On-Demand count — say 2), OnDemandPercentageAboveBaseCapacity (% of additional instances that are On-Demand — say 0%, meaning all scale-out instances are Spot), and SpotAllocationStrategy (price-capacity-optimized — balances Spot price with availability). This gives you reliable baseline capacity with cheap Spot scale-out. For your GetWork app: 2 On-Demand instances for baseline, Spot for additional capacity during peak hours.
EC2 Hibernate — preserving RAM state across stop/start cycles
The mechanism, the requirements, and when it actually makes sense

EC2 Hibernate is a power state between running and stopped. When you hibernate an instance, the OS writes the contents of RAM to the EBS root volume, then powers off. When you start the hibernated instance, the EBS root volume is read back into RAM and the OS resumes exactly where it left off — same processes, same network connections (if they haven't timed out), same application state.

Hibernate vs Stop: Stop: instance powers off, RAM state lost, EBS data preserved, new public IP on restart (unless EIP). Hibernate: RAM written to EBS, instance powers off, RAM state restored on start, EBS data preserved, same IP reassigned. From the application's perspective, hibernate is like waking a laptop from sleep — the Spring Boot app is still running when the instance restarts. Stop is like a hard reboot — the Spring Boot app must restart from scratch.

Requirements for Hibernate: Must be enabled at instance launch (cannot enable hibernate on a running instance). The root EBS volume must be large enough to hold the RAM contents (if you have 8GB RAM, you need >8GB free on the root volume). Root volume must be encrypted (hibernate writes RAM to disk — RAM may contain sensitive data). Not all instance types support hibernate (general purpose, compute, memory optimised families do; GPU instances do not). Maximum hibernate duration: 60 days.

When Hibernate is actually useful: Hibernate is genuinely valuable for: developer workstations running on EC2 (stop paying overnight, resume exactly where you left off — IDE open, browser tabs, running processes). JVM warm-up time is significant for large Spring Boot applications — a fresh start takes 2-3 minutes to JIT-compile hot paths and warm caches. A hibernated instance resumes in 30-60 seconds with a fully warmed JVM. For Auto Scaling Groups in production, hibernate is generally not used — scaling decisions are made based on fleet health, and Spot interruptions don't support hibernate for all instance types.
The hibernate + Spot interruption interaction: AWS gives 2 minutes of warning before reclaiming a Spot instance. EC2 can attempt to hibernate the instance on interruption (if hibernate is enabled and the instance type supports it). If the Spot instance is successfully hibernated, you can potentially resume it later if capacity becomes available. However, AWS does not guarantee that the hibernated Spot instance will be available for resumption — it may never come back. For stateful workloads, relying on Spot + hibernate for state preservation is risky. For stateless workloads (most Spring Boot microservices), Spot interruption just terminates the instance and the ASG replaces it — hibernate is irrelevant.

Elastic Network Interfaces (ENIs) — EC2 networking from first principles
What an ENI actually is, why instances can have multiple, and what breaks when you move them

An Elastic Network Interface is a virtual network card in a VPC. Every EC2 instance has at least one ENI (eth0) — the primary network interface. ENIs are separate from EC2 instances: they exist independently, can be detached from one instance and attached to another, and persist after instance termination if configured to do so.

What an ENI contains: A private IPv4 address (assigned from the subnet CIDR), optionally one or more secondary private IPv4 addresses, an optional public IPv4 address (if the subnet has auto-assign public IP or an Elastic IP is attached), an optional IPv6 address, one or more Security Group associations, a MAC address (unique, persistent — doesn't change when ENI is moved between instances), and a source/destination check flag.

The MAC address — the source of software licence persistence: Some enterprise software (Oracle Database, certain security tools) licences are tied to a MAC address. This was originally designed for on-premise servers where the MAC never changed. On EC2, if you terminate an instance, its MAC is gone — the software licence breaks. The ENI solution: create an ENI specifically to carry the licenced MAC address, and detach/reattach it between instances. The MAC address stays with the ENI, not the EC2 instance. Your Oracle licence sees a consistent MAC address across instance replacements. This is one of the real use cases for Dedicated Hosts at JP Morgan — dedicated instances with predictable MAC addresses for compliance with Oracle's licensing terms.

Secondary private IP addresses and the load balancer pattern: An ENI can have multiple private IP addresses from the same subnet. This is used for: running multiple SSL virtual hosts on one EC2 (each virtual host bound to a different IP), a primary/secondary failover pattern (the primary instance holds the secondary IP; on failure, the secondary IP is moved to the standby instance, preserving any IP-based routing), and ECS on EC2 where each container task gets its own IP address (from secondary IPs on the instance's ENI pool).

The source/destination check — the reason NAT instances fail: By default, every ENI has "Source/Destination Check" enabled. This means the Nitro card drops any packet where the source IP is not one of the ENI's assigned IPs (outbound) or the destination IP is not one of the ENI's assigned IPs (inbound). This is the correct behaviour for normal instances — it prevents IP spoofing. But for NAT instances and network appliances that forward traffic on behalf of other instances (where the source/destination IP is not their own), this check must be disabled. If you're implementing a custom NAT instance (rare today with NAT Gateway) and traffic isn't forwarding, source/destination check is almost certainly the cause.
Instance store NVMe — ephemeral local storage and why it's different
Physically attached NVMe SSDs with sub-100µs latency — and the data loss guarantee you must accept

Certain EC2 instance types (i3, i3en, d2, h1, im4gn, is4gen) come with instance store volumes — physically attached NVMe SSDs that are part of the EC2 host server hardware. These are fundamentally different from EBS in three ways: they have sub-100µs latency (vs 1-5ms for EBS), they have much higher IOPS and throughput than any EBS volume, and they are completely ephemeral — data is permanently lost when the instance stops, hibernates, or terminates.

Performance comparison: An i3.xlarge has 1 × 950GB NVMe SSD with approximately 210,000 read IOPS and 100,000 write IOPS. A gp3 EBS volume costs extra to provision above 16,000 IOPS. The performance difference is roughly 10x for random I/O. For workloads that are I/O-bound and can tolerate data loss (because data is replicated elsewhere), instance store is the highest-performance option.

The use cases where instance store makes sense despite the data loss risk: Apache Kafka brokers — Kafka data is replicated across multiple brokers. If one broker loses its instance store (instance terminated), the data is recovered from replicas. But the remaining replicas must handle more load during recovery — use a Placement Group (Partition) to ensure replicas are on different hardware. Redis clusters — Redis Cluster mode replicates across nodes. Instance store provides the highest cache throughput. On eviction or failure, the cache repopulates from the database. Hadoop HDFS datanodes — HDFS replicates data blocks across multiple nodes by design. Instance store provides the highest local storage throughput. Any buffering or staging layer that holds data temporarily — the data is read from a durable source and written to a durable destination; the buffer itself can be lost and recreated.
The instance store data loss incident — the most predictable data disaster in AWS: A team runs a database on an i3 instance using instance store for performance. The EC2 host has hardware maintenance required by AWS. AWS sends a retirement notice (14 days) and a scheduled stop event. The team is unaware or forgets. The instance is stopped for host maintenance. Instance store data is gone. The database is gone. No backup was taken because "we have high performance storage." This happens regularly. The rule: never store any data on instance store that isn't either replicated elsewhere or reproducible from another source. Instance store is explicitly for ephemeral, replicated, or reconstructable data.
Dedicated Hosts — physical server tenancy for compliance and licensing
When sharing a physical server with other AWS customers is not acceptable

By default, EC2 instances run on shared physical hardware — you and other AWS customers have instances running on the same physical server, isolated by the Nitro hypervisor. For most workloads, this is fine and is the foundation of the Shared Responsibility Model. For certain compliance and licensing requirements, shared tenancy is not acceptable.

Tenancy options:

Shared tenancy (default): Your instances run on physical hosts shared with other AWS customers. Nitro ensures complete isolation. Most cost-effective.

Dedicated Instances: Your instances run on physical hardware dedicated to your AWS account — no other customer's instances on the same physical server. But AWS manages which physical servers your instances land on; you don't control placement. Per-instance hourly surcharge (~$0.04-0.10/hr depending on instance type) plus a per-region fee of $2/hr when any Dedicated Instances are running.

Dedicated Hosts: You get an entire physical server allocated to your account. You control exactly which instances run on it. You can see the host ID, sockets, cores, and capacity. You can launch instances with specific placements. This is required for: Oracle Database (Oracle's licensing model charges per physical socket — on a Dedicated Host you know exactly how many sockets you're paying for), Windows Server with Bring Your Own License (BYOL), and regulatory requirements in financial services (some regulations require knowing the physical hardware your data runs on).

Dedicated Hosts at JP Morgan — the real use case: At a financial institution like JP Morgan, certain regulatory frameworks (may include MAS Technology Risk Guidelines in Singapore, certain RBI requirements for India) may require knowing the physical isolation of production systems. Dedicated Hosts satisfy this requirement — JP Morgan can document that trading systems run on specific physical hardware in specific AWS AZs. The Oracle Database licensing angle is equally relevant — Oracle licenses by physical socket, and JP Morgan almost certainly runs Oracle (being a large financial institution). Dedicated Hosts let them accurately count Oracle licences. When interviewing for roles at fintech companies, understanding Dedicated Hosts as a compliance requirement (not just a performance one) shows maturity.
Nitro Enclaves — isolated compute for sensitive data processing
The architecture that processes credit card data, health records, and cryptographic keys without any human able to access the processing environment

Nitro Enclaves are isolated virtual machines created within an EC2 instance using the Nitro hypervisor. A Nitro Enclave is created by partitioning some of the parent EC2 instance's CPU cores and memory into a completely isolated environment. The enclave has no persistent storage, no external networking (except a vsock connection to the parent instance), and critically — no SSH access, no shell, no way to debug or inspect what's happening inside.

The isolation is enforced by hardware. Not even the root user on the parent EC2 can access the enclave's memory or processes. Not even AWS engineers can access the enclave. The only communication channel is a vsock (virtual socket) between the parent instance and the enclave — and you control what goes in and what comes out.

The Nitro Enclaves use case — why "no access" is the security property you want: Consider processing credit card numbers (PCI-DSS compliance requires minimising who can see cardholder data). With a Nitro Enclave: the encrypted credit card number is sent to the enclave via vsock. Inside the enclave (which no human can inspect), the number is decrypted using a KMS key that the enclave's cryptographic attestation is authorised to use. The processing (charge, tokenise) happens inside. Only the result (transaction ID, token) exits the enclave. The actual credit card number is never visible outside the enclave — not to the application, not to the EC2 admin, not to AWS. The KMS key policy uses the enclave's attestation document (a cryptographically signed proof of what code is running in the enclave) to decide whether to decrypt. If the enclave's code is tampered with, the attestation changes, and KMS refuses to decrypt.

Relevance to your JP Morgan project: Financial data processing — card numbers, account numbers, authentication secrets — is exactly the domain where Nitro Enclaves are used in production financial services. While you may not be implementing Nitro Enclaves directly, understanding that they exist and what problems they solve is the kind of architectural awareness that distinguishes senior engineers in fintech interviews.

Q1 Your EC2 instance booted but your application isn't running. User data was supposed to start it. Walk me through your complete diagnostic process. Hard▼

This is a layered diagnostic problem. I'd check in order from the most general to the most specific.

First: cloud-init status. sudo cloud-init status --long. If status is "error", the cloud-init system itself failed — the issue is in the user data script or cloud-init configuration. If status is "done", cloud-init completed successfully — the issue is post-startup (the application crashed after starting).

If cloud-init errored: cloud-init-output.log. sudo cat /var/log/cloud-init-output.log. This contains everything the user data script printed. I'm looking for error messages, failed commands, package installation failures. If this file is empty, the script never ran — which means cloud-init failed before reaching the script execution phase. Then I'd check sudo cat /var/log/cloud-init.log for the cloud-init internal error.

Common causes at this stage: Package manager lock file (another process has the yum/apt lock — user data waits indefinitely); IMDSv2 not supported by cloud-init version (user data was never fetched); user data over 16KB uncompressed limit (user data is truncated); network not up when script tries to download dependencies (ordering issue — script runs before network is ready); base64 encoding issue (file:// vs fileb:// mismatch from Day 6).

If cloud-init succeeded but app not running: systemd journal. sudo journalctl -u getwork-app --no-pager -n 50. This shows the application's startup logs. Common causes: JAR not found (S3 download failed silently), Java not found (installation succeeded but PATH not updated), port already in use, missing environment variables, database connection failure at startup, out of memory on instance type too small for the JVM heap.

Final check: did the user data actually contain what I expected? Query IMDS: curl -sf http://169.254.169.254/latest/user-data -H "X-aws-ec2-metadata-token: $(curl -sf -X PUT http://169.254.169.254/latest/api/token -H X-aws-ec2-metadata-token-ttl-seconds:21600)". This prints the exact user data the instance received — confirms no encoding issue, confirms the correct version was deployed.

Why this answer works: It follows the actual debugging sequence (cloud-init status → output log → systemd journal → IMDS verification) rather than random guessing. It names specific failure modes at each step rather than vague "check the logs." The IMDSv2 syntax for the final IMDS query shows you know Day 5 security requirements and apply them even in debugging scenarios.
Q2 What is an AMI and explain what happens at the hardware level when you launch an EC2 instance from one. Medium▼

An AMI is a metadata object — not a disk image file — that contains a description of the AMI (name, architecture, virtualisation type) and a block device mapping that points to one or more EBS snapshots. The AMI itself contains no data; the data lives in the EBS snapshots. This distinction is important for understanding what happens at launch.

When you launch an EC2 instance from an AMI, AWS does not copy the EBS snapshots to create new volumes before booting. EBS uses copy-on-write (CoW). A new EBS volume is created instantly, pointing to the same underlying data as the snapshot through a CoW mapping table. The instance boots immediately — it doesn't wait for gigabytes to be copied. As the OS reads data during boot, each block is fetched from the snapshot storage on first access and then cached locally in the volume. As the OS writes data, those writes go directly to the new volume's private storage — not back to the snapshot, which remains unmodified.

The practical implication is that EC2 instances boot quickly (under 60 seconds) even from large AMIs. The trade-off is that first-time block access during or after boot has slightly higher latency (the "cold storage" penalty) as blocks are fetched from snapshot storage. For latency-sensitive production workloads — databases especially — pre-warming the volume by reading every block before the application starts eliminates this penalty.

When the instance terminates, the EBS volume created from the AMI is deleted (if DeleteOnTermination is true, the default for root volumes). The original AMI and its backing snapshots are unaffected — the snapshot was only ever a reference point, never modified.

Why this answer works: Most engineers say "AMI is a template for launching EC2." This answer explains the CoW mechanism that explains boot speed, the "cold storage" latency penalty, and the snapshot immutability — showing you understand the storage architecture, not just the UI concept.
Q3 When would you choose gp3 vs io2 for an EBS volume? What specific metrics would you use to decide? Medium▼

The decision is based on the workload's I/O characteristics and whether gp3's maximum capabilities are sufficient.

gp3 is the right choice for the vast majority of workloads. It provides up to 16,000 IOPS and 1,000 MB/s throughput, with consistent performance (no burst credits). Spring Boot applications, development databases, log storage, application data — all of these are well within gp3's capabilities. gp3 is also significantly cheaper than io2 per IOPS.

io2 (specifically io2 Block Express) is appropriate when: you need more than 16,000 IOPS (io2 Block Express goes to 256,000 IOPS), you need sub-millisecond latency (io2 Block Express has single-digit microsecond latency vs gp3's 1-3ms), you require 99.999% durability (io2) vs 99.9% (gp3) — important for mission-critical databases, or you need Multi-Attach (io1/io2 only).

The metrics I'd use to decide: Start with gp3 and monitor. If VolumeQueueLength is consistently greater than 1 (requests queuing for the volume), IOPS is the bottleneck. Check VolumeReadOps + VolumeWriteOps — if approaching 16,000/s, you've hit gp3's maximum and io2 is needed. If throughput (VolumeReadBytes + VolumeWriteBytes) is consistently near 1,000 MB/s, gp3 is saturated on throughput and io2's 4,000 MB/s ceiling applies. For latency: use VolumeTotalReadTime / VolumeReadOps to compute average read latency. If this is consistently above 3ms under normal load, you may benefit from io2's lower latency.

For a typical Spring Boot application: gp3 with 3,000-6,000 IOPS is almost always sufficient. For a high-transaction database (MySQL handling 10,000+ TPS): evaluate with metrics, likely need 10,000-30,000 IOPS which may require io2.

Why this answer works: It gives a clear decision framework — default to gp3, use metrics to justify io2. The specific CloudWatch metrics and the calculation for average latency show operational depth. The TPS example connects the abstract metrics to a real workload scenario.
Q4 What are Launch Templates and why should you always use them instead of Launch Configurations? Easy▼

A Launch Template is a versioned specification for launching EC2 instances, defining AMI, instance type, security groups, IAM profile, EBS configuration, user data, and dozens of other settings. It is the modern replacement for Launch Configurations, which were deprecated in 2022 (AWS stopped accepting new Launch Configuration creation for certain features and stopped adding new features to them).

The key advantages of Launch Templates over Launch Configurations: versioning — each change creates a numbered version, enabling rollbacks; you can point an ASG at a previous version to instantly revert a deployment. Partial specification — a Launch Template can specify only some fields, with remaining fields specified at launch time or inherited from a parent template. This enables environment-specific overrides (dev uses t3.micro, prod uses m5.large) from the same base template. Support for modern EC2 features — Spot Mixed Instances Policy, attribute-based instance type selection, capacity reservations, Dedicated Hosts, and all new AWS features are only available via Launch Templates. A Launch Configuration cannot use these.

In practice: any new Auto Scaling Group should use a Launch Template. Any existing ASG using a Launch Configuration should be migrated — it's an in-place migration that doesn't require replacing instances. Q5 Explain EC2 instance store. When is it the right choice, and when is it absolutely the wrong choice? Medium▼

Instance store volumes are NVMe SSDs physically attached to the EC2 host server, providing sub-100µs latency and extremely high IOPS (210,000+ for i3 instance types). They are ephemeral — data is permanently lost when the instance stops, hibernates, or terminates, including during scheduled host maintenance by AWS.

Right choice: Workloads where data is replicated across multiple nodes and the loss of one node's local data is tolerable. Apache Kafka brokers in a replicated cluster — if one broker loses its instance store, the data is recovered from replicas on other brokers. Distributed caches like Redis Cluster — cache eviction and repopulation from a backing database handles the loss. Hadoop HDFS datanodes — HDFS replication (typically 3x) ensures data survives node loss. High-throughput buffer/staging layers where data is read from a durable source and written to a durable destination. The common thread: the data exists elsewhere, and the instance store is used purely for performance, not durability.

Absolutely wrong choice: Any single-node database (MySQL, Postgres, Oracle) without external replication — the database is lost on instance stop or host maintenance. Any stateful application data that isn't replicated. Any data that must survive instance termination. Any use case where you would be unhappy receiving an AWS host maintenance notification with 14 days warning and losing everything on that volume.

The failure mode that causes incidents: instance store is used for a database because "we have high performance storage." AWS sends a host retirement notice (hardware maintenance). Instance is stopped. Instance store data is gone. No backup exists. This happens to teams regularly who don't read AWS notifications carefully.

Why this answer works: The right/wrong framing with specific examples (Kafka, Redis, Hadoop vs single-node MySQL) is concrete. The AWS retirement notice failure story makes the risk visceral — this isn't a theoretical risk, it's a documented operational failure pattern.

Day 7 — EC2 deep mechanics
cloud-init phases
5 phases. Shell scripts run in Phase 5 (cloud-final). cloud-config modules run in Phase 4. SSH key injection in Phase 3. Check status: cloud-init status --long.
User data debugging
/var/log/cloud-init.log (engine), /var/log/cloud-init-output.log (script output), journalctl -u service, IMDS query for what was received.
AMI is a pointer
AMI = metadata + block device mapping → EBS snapshots. No data in the AMI itself. EBS uses CoW at launch — no snapshot copy, instance boots instantly.
AMI region-specific IDs
AMI IDs are unique per region. Never hardcode. Use aws_ami data source in Terraform with owner + name filter. Copy AMIs explicitly between regions.
AMI deregistration cleanup
Deregistering AMI does NOT delete backing snapshots. Delete snapshots explicitly. EC2 Image Builder manages this automatically with retention policies.
gp3 vs gp2
gp3: IOPS independent of size, 3000 IOPS + 125MB/s baseline, 1000MB/s max. gp2: 3 IOPS/GB, burst credits, 250MB/s max. Migrate all gp2 to gp3 in-place.
EBS VolumeQueueLength
The metric that shows IOPS bottleneck. Should be near 0. Consistently >1 means requests queuing — increase IOPS or switch volume type.
EBS AZ lock-in
EBS volume exists in one AZ. Can only attach to EC2 in same AZ. Snapshots are AZ-independent (stored in S3). Use snapshots for cross-AZ data recovery.
Launch Templates vs LC
Launch Configurations are deprecated. Launch Templates: versioned, partial spec, supports all modern EC2 features. Always use Launch Templates.
ENI and MAC address
ENI has persistent MAC. Software licences tied to MAC — keep ENI across instance replacements. Source/Destination Check must be disabled for NAT instances.
Instance store vs EBS
Instance store: physically attached NVMe, sub-100µs, data lost on stop/terminate. Only for replicated/ephemeral data. Never for single-node databases.
Dedicated Hosts
Entire physical server for your account. Required for Oracle BYOL (socket-based licensing) and some financial compliance frameworks. More expensive than shared.
Non-negotiable rules for EC2 deployments
1. Every user data script starts with: set -euo pipefail and output redirected to a log file.
2. Never put secrets in user data — fetch from Secrets Manager or Parameter Store in the script.
3. Every Launch Template sets HttpTokens=required (IMDSv2 enforced) and HttpPutResponseHopLimit=1.
4. Every EBS volume in Launch Templates: VolumeType=gp3, Encrypted=true.
5. Never hardcode AMI IDs — use aws_ami data source with name filter in Terraform.
6. When deregistering an AMI, manually delete the backing EBS snapshots afterwards.
7. Pre-warm EBS volumes created from large snapshots for latency-sensitive production workloads.
8. Use Launch Templates (not Launch Configurations) for all new Auto Scaling Groups.
9. Instance store: only for data that is replicated elsewhere or fully ephemeral.
10. Debug user data with: cloud-init status --long, then the appropriate log file.
