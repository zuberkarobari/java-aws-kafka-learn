The AWS CLI is not a GUI wrapper. It is a complete Python application that constructs HTTP requests, signs them with AWS Signature Version 4, sends them to AWS service endpoints, and deserialises the JSON responses. Understanding what the CLI does at each step explains every authentication error, every rate limit, and every unexpected behaviour you will encounter running it in production scripts.
What actually happens on every AWS CLI call
The six-step sequence from your terminal to AWS and back

When you run aws s3 ls, the CLI executes a precise sequence. Understanding this sequence is what makes you able to debug authentication errors, construct equivalent HTTP calls for testing, and understand why certain operations are slow or fail under specific conditions.

Step 1 — Command parsing. The CLI parses your command into a service (s3), operation (ls, mapped internally to ListBuckets), and parameters. The CLI uses a service model — a JSON file describing every API operation for every service — to validate your input and know what API to call.

Step 2 — Credential resolution. The CLI runs the DefaultCredentialsProvider chain (covered in detail on the next tab). It resolves which credentials to use: environment variables, profile from config files, IAM role from EC2 metadata service, etc. If no credentials are found, the command fails before any HTTP request is made.

Step 3 — Request construction. The CLI serialises your command into an HTTP request: URL, headers, and optionally a body. For aws s3 ls this becomes a GET request to https://s3.ap-south-1.amazonaws.com/ with specific headers and query parameters.

Step 4 — SigV4 signing. The CLI signs the request using AWS Signature Version 4 (SigV4). This adds an Authorization header containing a cryptographic signature of the request, the credentials used, the timestamp, and the service. AWS uses this signature to authenticate the request and verify it hasn't been tampered with in transit.

Step 5 — HTTP request and response. The signed HTTP request is sent to AWS. AWS validates the signature, evaluates IAM policies, and returns a response — typically JSON. Error responses also come as JSON with a specific error code and message.

Step 6 — Response formatting. The CLI parses the JSON response and formats it per your --output setting: JSON (raw), text (tab-separated), table (ASCII table), or YAML. The default is JSON for most commands.

Why this matters for debugging: When you get "An error occurred (InvalidClientTokenId) when calling the ListBuckets operation: The security token included in the request is invalid" — the problem is at Step 2 or Step 4. The token is missing, expired, or wrong. When you get "An error occurred (AccessDenied)" — the request reached AWS, authentication succeeded (Step 4 passed), but IAM denied the operation (Step 5). When you get "Could not connect to the endpoint URL" — DNS resolution or network connectivity failed at Step 5. Each error maps to a specific step in the sequence.
The --debug flag — your most powerful CLI diagnostic tool: Add --debug to any failing CLI command and the CLI prints every step: credential source, the constructed HTTP request, the exact URL and headers (with the Authorization header), the response headers, and the full response body. This is the first tool to reach for when a CLI command fails unexpectedly. aws s3 ls --debug 2>&1 | head -100 — the 2>&1 redirects debug output (which goes to stderr) to stdout so you can pipe it.

AWS Signature Version 4 — how every request is authenticated
The cryptographic protocol that proves every API call came from a valid identity and wasn't tampered with

AWS Signature Version 4 (SigV4) is the authentication protocol used by every AWS SDK and CLI. Every HTTP request to AWS must be signed. The signature is a HMAC-SHA256 hash computed from the request contents — if any part of the request changes in transit, the signature becomes invalid and AWS rejects the request. This is both authentication (who sent this) and integrity protection (was this modified in transit).

The four components of SigV4:

1. Canonical request. The request is normalised — URL-encoded path, sorted query parameters, lowercase header names, and a SHA256 hash of the request body (even if empty). This normalisation ensures the same request always produces the same canonical form regardless of how the HTTP client formats it.

2. String to sign. A string is constructed from: the algorithm name (AWS4-HMAC-SHA256), the timestamp (format: YYYYMMDDTHHMMSSZ), the credential scope (date/region/service/aws4_request), and the SHA256 hash of the canonical request.

3. Signing key derivation. A derived signing key is computed by chaining four HMAC operations: HMAC(HMAC(HMAC(HMAC("AWS4" + secret_key, date), region), service), "aws4_request"). This means the signing key is specific to the date, region, and service — a key derived for ap-south-1/s3 on 2024-03-15 cannot be used to sign requests for us-east-1/ec2 or for a different date.

4. Authorization header. The signature is computed as HMAC(signing_key, string_to_sign). The Authorization header is constructed with the algorithm, credentials, signed headers list, and the signature.

Why the timestamp matters — and the clock skew error: The canonical request includes the current timestamp. AWS validates that the timestamp in the request is within ±15 minutes of AWS's authoritative time. If your system clock is more than 15 minutes off, every API call fails with "Signature not yet current" or "Signature expired" — even with perfectly valid credentials. This is the cause of mysterious authentication failures in containers and VMs where the system clock drifts. Fix: sync the system clock with NTP. In containers: ensure the host's clock is synced; container clocks inherit from the host.
The SigV4 pre-signed URL expiry trap: Pre-signed S3 URLs use a query-parameter version of SigV4 where the expiry time is baked into the URL as X-Amz-Expires. The signature covers the expiry time — you cannot extend a pre-signed URL's expiry by editing the URL parameter. An attacker who intercepts a pre-signed URL cannot extend its validity. But there is a subtlety: the URL is valid from the signing time, not from when it's first used. A URL signed at 2PM with 1-hour expiry expires at 3PM — even if the recipient doesn't use it until 2:58PM. If generation and delivery takes time (email, async queue), the recipient may have a URL that's almost expired before they use it. Always generate pre-signed URLs with sufficient expiry for the delivery chain, but not so long they become a security risk.
Endpoint resolution — how the CLI knows where to send requests
Regional vs global endpoints, VPC endpoints, and the FIPS endpoint requirement

The CLI must resolve the correct HTTPS endpoint for each service and region before constructing a request. The endpoint format is typically service.region.amazonaws.com. This seems simple but has several important nuances.

Regional vs global services: Most services have regional endpoints: ec2.ap-south-1.amazonaws.com, s3.ap-south-1.amazonaws.com, rds.ap-south-1.amazonaws.com. Global services — IAM, STS (when using global endpoint), Route 53, CloudFront — use endpoints without a region component: iam.amazonaws.com, route53.amazonaws.com. Calling a regional endpoint with the wrong region returns resources for that region only — a common confusion when an engineer creates an EC2 in ap-south-1 but has their CLI configured for us-east-1 and can't find the instance.

S3 endpoint duality: S3 has two endpoint styles. Path-style: s3.ap-south-1.amazonaws.com/bucket-name/key. Virtual-hosted style: bucket-name.s3.ap-south-1.amazonaws.com/key. Virtual-hosted is the modern default and required for many features. Path-style was deprecated by AWS in 2019 (though still works for existing buckets). The CLI uses virtual-hosted style by default. If you have a bucket name with dots (e.g. my.bucket.name), virtual-hosted style breaks SSL certificate validation because my.bucket.name.s3.amazonaws.com doesn't match the wildcard cert *.s3.amazonaws.com. Avoid dots in bucket names.

VPC Endpoint DNS override: When an Interface VPC Endpoint is created for a service, it registers private DNS names that override the public endpoint within the VPC. From an EC2 inside the VPC, s3.ap-south-1.amazonaws.com resolves to the VPC Endpoint's private IP instead of S3's public IP. The CLI doesn't need any special configuration — DNS resolution handles the routing transparently. This requires enableDnsHostnames and enableDnsSupport to be true on the VPC (Day 4 topic). If DNS-based VPC Endpoint routing isn't working, check these VPC settings first.

The --endpoint-url override: Any CLI command accepts --endpoint-url https://custom-endpoint.com to override the default endpoint. Use cases: LocalStack (local AWS emulation for testing), MinIO (S3-compatible storage), AWS API Gateway custom endpoints, and testing against specific regional endpoints. In automated scripts, be wary of hardcoded endpoint URLs — they break when switching between environments. Use environment variables or config profiles instead.

The dual-stack endpoint and IPv6: AWS services increasingly support dual-stack endpoints (both IPv4 and IPv6): s3.dualstack.ap-south-1.amazonaws.com. If you enable IPv6 on your VPC and EC2 but your CLI or SDK uses IPv4-only endpoints, traffic still goes via IPv4. If your security group or NACL only allows IPv6 outbound (rare but possible), IPv4 API calls will fail. Check your endpoint configuration if you're in a dual-stack environment with unexpected connectivity issues.
Pagination — the silent data truncation problem
Why AWS APIs return partial results and how to get everything — the bug in almost every AWS script

Most AWS list/describe APIs return paginated results. If you have 1,000 EC2 instances and call aws ec2 describe-instances, you might get only 100. The response includes a NextToken (or Marker for older APIs, or NextPageToken) — a cursor pointing to the next page of results. To get all results, you must follow the pagination chain.

Why APIs paginate: Returning 10,000 EC2 instances in one HTTP response would be a 10MB+ JSON payload. API Gateway and service quotas would reject it. Client memory consumption would spike. Pagination lets clients receive manageable chunks. The page size varies by service and operation — typically 100-1000 items per page.

The --no-paginate flag vs --paginate (default): The AWS CLI v2 automatically handles pagination for you with --paginate (the default). When you run aws ec2 describe-instances, the CLI makes multiple API calls behind the scenes, following the NextToken chain, and returns all results concatenated into one response. This is convenient but dangerous for very large result sets — you might be waiting for 50 sequential API calls and don't know it. Add --no-paginate to get only the first page, or use --page-size N to control page size, or use --max-items N to limit the total number of items returned across all pages.
The most common bug in AWS scripts — incomplete data: A script runs aws ec2 describe-instances --query 'Reservations[*].Instances[*].InstanceId' and processes the results. It works perfectly in dev (5 instances). In production (1,200 instances), it silently processes only the first 1,000. Nobody notices because the script doesn't fail — it just misses 200 instances. The fix: use the default pagination (no --no-paginate) or explicitly use aws ec2 describe-instances --no-paginate | ... only when you know the result set is small. Better: use --output json | jq or the SDK's paginator APIs which handle pagination correctly by default.
The correct way to paginate in scripts:
Option 1 — Let the CLI auto-paginate (default, sufficient for most cases):
aws ec2 describe-instances --query 'Reservations[*].Instances[*].InstanceId' --output text

Option 2 — SDK Paginator (correct for production code in Spring Boot):
The AWS SDK v2 for Java has Paginator classes: DescribeInstancesIterable. These return lazy iterators — they fetch the next page only when you iterate past the last item of the current page. Memory-efficient, automatically handles token chaining, and throws exceptions on individual page failures rather than silently truncating.
Retries, throttling, and exponential backoff
Why AWS throttles you and how the SDK's retry logic works — and when it doesn't save you

AWS services have API rate limits — maximum calls per second per account per region per API. When you exceed a rate limit, the service returns an HTTP 429 (Too Many Requests) with error code ThrottlingException or RequestLimitExceeded. The limits vary by service and operation. EC2 DescribeInstances: ~100 calls/second. IAM API calls: much lower, around 1-10/second for write operations. S3 PUT/GET: very high (5,500 GET/s and 3,500 PUT/s per prefix).

The AWS SDK v2 retry behaviour: By default, AWS SDK v2 retries throttled requests (429s) and transient errors (500s, 503s) using exponential backoff with jitter. The retry delay doubles with each attempt (1s, 2s, 4s, 8s...) and adds a random jitter to prevent the "thundering herd" problem — where many clients retry simultaneously at the same backoff interval and all hit the service at once. Default: 3 retry attempts. The maximum retry count and base delay are configurable.

Why jitter is essential — not just exponential backoff: Without jitter: 100 Lambda functions all hit a rate limit at the same time. They all wait 1 second and retry simultaneously — hitting the rate limit again. They wait 2 seconds and retry — hitting it again. The service remains overwhelmed because all clients are in lockstep. With jitter: each client waits base_delay × 2^attempt × random(0,1). The randomness spreads the retries across the backoff window. The service sees a distributed retry pattern instead of synchronized waves. The AWS SDK implements "full jitter" which distributes retries uniformly across the entire backoff window. Understanding this is what makes you dangerous in distributed systems interviews.

Retry budgets and when retries hurt: Retries work for transient throttling but make things worse in some scenarios. If you're making 100 API calls in a Lambda function and all are throttled due to sustained overload, retrying within the same Lambda invocation ties up execution time and memory. Lambda has a maximum execution time of 15 minutes — a function retrying aggressively across many operations can exhaust this limit. Better pattern for high-volume operations: use SQS as a buffer. Put jobs into SQS, process via Lambda with reserved concurrency limits. If Lambda is throttled, SQS holds the messages and Lambda processes them when capacity is available. This is the correct cloud-native pattern — queuing absorbs bursts rather than relying on in-process retries.

The adaptive retry mode and why you should enable it: AWS SDK v2 introduced "adaptive" retry mode (in addition to "standard" and "legacy"). Adaptive mode uses a token bucket algorithm — each successful API call earns tokens, each retry costs tokens. When the token bucket is empty, the SDK slows down requests rather than retrying aggressively. This effectively implements client-side rate limiting that prevents the SDK from overwhelming a throttled service. Enable it via AWS_RETRY_MODE=adaptive environment variable or in the SDK configuration. For applications making many concurrent AWS API calls (your Spring Boot WebFlux app with reactive S3 calls), adaptive mode prevents cascading throttling failures that standard retry mode can cause.
The DefaultCredentialsProvider chain is the mechanism that makes your Spring Boot app work on your laptop with ~/.aws/credentials, on EC2 with an instance role, in Lambda with an execution role, and in GitHub Actions with OIDC — all without changing a single line of application code. Understanding exactly what it checks and in what order prevents credential bugs that take hours to debug.
The DefaultCredentialsProvider chain — complete resolution order
Eight locations checked in strict order — the first one found wins

The AWS SDK's DefaultCredentialsProvider checks credential sources in a specific order. The first source that returns valid credentials wins — no further sources are checked. Understanding this order explains every "why is my app using the wrong credentials" mystery.

1
Java system properties (SDK only) — aws.accessKeyId, aws.secretAccessKey, optionally aws.sessionToken. Set via JVM flags: -Daws.accessKeyId=AKIA.... Almost never used in production — system properties are visible in process listings. Useful for testing.
2
Environment variables — AWS_ACCESS_KEY_ID, AWS_SECRET_ACCESS_KEY, optionally AWS_SESSION_TOKEN. The most common mechanism for CI/CD pipelines, Docker containers, and ECS task definitions. Checked before config files — environment variables override everything below them.
3
AWS_PROFILE environment variable or default profile in ~/.aws/credentials — The credentials file at ~/.aws/credentials contains named profiles in INI format. If AWS_PROFILE is set, that named profile is loaded. Otherwise the [default] profile is used. This is the "developer laptop" credential source.
4
~/.aws/config file (config, not credentials) — The config file contains additional profile settings including role assumption configurations: role_arn, source_profile, mfa_serial. When a profile in config has a role_arn, the SDK automatically assumes that role using the source profile's credentials. This is how developers assume production roles from their laptop.
5
Container credentials (ECS) — When running inside an ECS task, the AWS_CONTAINER_CREDENTIALS_RELATIVE_URI environment variable is set by ECS. The SDK fetches credentials from http://169.254.170.2${AWS_CONTAINER_CREDENTIALS_RELATIVE_URI}. This is how ECS tasks assume their task role — the ECS agent serves the role's credentials at this internal endpoint.
6
EC2 Instance Metadata Service (IMDS) — The SDK queries http://169.254.169.254/latest/meta-data/iam/security-credentials/ROLE-NAME. This is how EC2 instances use their attached IAM instance profile. The SDK caches credentials and refreshes them 5 minutes before expiry. This step only applies when running on an EC2 instance — on a laptop or in a container without an instance profile, the IMDS endpoint doesn't exist.
The most common credential chain bug — environment variable bleed: A developer sets AWS_ACCESS_KEY_ID in their shell for a one-time test. Three weeks later, that environment variable is still set (persisted in their .bashrc or tmux session). Their Spring Boot app starts locally, the SDK finds environment variables at step 2, and uses those old credentials — never reaching the config file profile they intended. The app seems to work but is hitting the wrong account/role. aws sts get-caller-identity is the fastest diagnostic — always run this to confirm which identity your code is actually using. Also run aws configure list to see which source each configuration value came from.
Config file internals — the two files engineers confuse
~/.aws/credentials vs ~/.aws/config — different formats, different purposes, different behaviour

AWS CLI uses two separate files. They have overlapping functionality that causes constant confusion.

~/.aws/credentials — contains only authentication credentials (access keys). INI format. Profile headers have no prefix. Credentials here are for static IAM user keys:

# ~/.aws/credentials [default] aws_access_key_id = AKIAIOSFODNN7EXAMPLE aws_secret_access_key = wJalrXUtnFEMI/K7MDENG/bPxRfiCYEXAMPLEKEY [deloitte-dev] aws_access_key_id = AKIAI44QH8DHBEXAMPLE aws_secret_access_key = je7MtGbClwBF/2Zp9Utk/h3yCo8nvbEXAMPLEKEY

~/.aws/config — contains configuration settings AND can contain credentials. INI format but profile headers are prefixed with "profile " (except [default]). Can contain role assumption, region, output format, MFA, and credential settings:

# ~/.aws/config [default] region = ap-south-1 output = json [profile deloitte-prod] # "profile" prefix required for non-default role_arn = arn:aws:iam::123456789:role/prod-deploy-role source_profile = deloitte-dev # uses credentials from [deloitte-dev] in ~/.aws/credentials region = ap-south-1 mfa_serial = arn:aws:iam::111222333:mfa/zuber # require MFA to assume this role duration_seconds = 3600 [profile jp-morgan-sandbox] role_arn = arn:aws:iam::987654321:role/consultant-role source_profile = deloitte-dev external_id = deloitte-engagement-12345 # required by JP Morgan's trust policy region = ap-south-1
The external_id in cross-account role assumption: JP Morgan's IAM role trust policy likely requires an external_id condition. This is a secret value that Deloitte and JP Morgan agreed on — it prevents the "confused deputy" attack. Without external_id: any AWS account could potentially configure a role to assume JP Morgan's role (if they knew the role ARN). With external_id: even knowing the role ARN, you can't assume the role without also knowing the external_id that JP Morgan requires. This is a standard pattern in cross-account consulting engagements. The external_id goes in your ~/.aws/config profile under the client's role.
The "profile " prefix mistake: In ~/.aws/config, all non-default profiles must have the prefix "profile " in the section header: [profile my-profile]. In ~/.aws/credentials, profiles have no prefix: [my-profile]. If you write [my-profile] in ~/.aws/config (missing the "profile " prefix), the CLI doesn't recognise it as a profile. Running aws --profile my-profile s3 ls fails with "The config profile (my-profile) could not be found." This is one of the most common CLI setup mistakes because the format is inconsistent between the two files — a historical quirk of the CLI design.
Named profiles — managing multiple AWS accounts and roles
The correct way to handle dev/staging/prod, multiple clients, and cross-account access from one machine

Most engineers working at consultancies or with multiple AWS accounts need to switch between them constantly. Named profiles let you maintain multiple credential sets and switch between them per-command or per-session.

# Switch for a single command aws s3 ls --profile deloitte-prod # Switch for an entire shell session export AWS_PROFILE=deloitte-prod aws s3 ls # uses deloitte-prod profile automatically # Override region for a specific command aws ec2 describe-instances --profile deloitte-prod --region us-east-1 # See which profile and credentials are currently active aws configure list # Output shows: Name | Value | Type | Location for each setting # Type tells you: env, config-file, credentials-file, or None # Verify the identity for the current profile aws sts get-caller-identity # Returns: UserId, Account, Arn — always run this first to confirm you're in the right account
Profile setup for a consulting engineer at Deloitte:
~/.aws/credentials: one entry — your personal Deloitte IAM user credentials (long-lived, used only as the source for role assumption).
~/.aws/config: one entry per client/environment — each with role_arn pointing to the client account's role, source_profile pointing to your Deloitte user, and the external_id JP Morgan gave Deloitte. You never directly use long-lived credentials for client work — you always assume the client's role via STS. All client API calls use temporary credentials that expire and don't persist in logs or history.
aws-vault — the secure alternative to plaintext config files: ~/.aws/credentials stores access keys as plaintext on your disk. If your laptop is stolen or your filesystem is compromised, all your AWS credentials are exposed. aws-vault is an open-source tool that stores credentials in your OS keychain (macOS Keychain, Linux secret service, Windows Credential Manager) — encrypted at rest. You run aws-vault exec deloitte-dev -- aws s3 ls; aws-vault retrieves the credentials from the keychain, sets them as environment variables for that subprocess, then clears them. Credentials never exist on the filesystem. For anyone working on client AWS accounts with sensitive data, aws-vault is the professional-grade credential management tool.
Environment variables — the complete set and their precedence
Every AWS-related environment variable, what it does, and when you'd set it

AWS CLI and SDK respect a comprehensive set of environment variables. These override config file settings and are the primary mechanism for credential injection in CI/CD systems, containers, and serverless environments.

# Core credentials — all three required for assumed role credentials AWS_ACCESS_KEY_ID=AKIA... # or ASIA... for temporary credentials AWS_SECRET_ACCESS_KEY=... AWS_SESSION_TOKEN=... # REQUIRED when using ASIA temporary credentials # Region and output configuration AWS_DEFAULT_REGION=ap-south-1 # Overrides config file region AWS_DEFAULT_OUTPUT=json # json | text | table | yaml # Profile selection AWS_PROFILE=deloitte-prod # Selects named profile from config/credentials AWS_CONFIG_FILE=/path/to/config # Override default ~/.aws/config location AWS_SHARED_CREDENTIALS_FILE=/path # Override default ~/.aws/credentials location # Endpoint overrides AWS_ENDPOINT_URL=http://localhost:4566 # Override all service endpoints (LocalStack) AWS_ENDPOINT_URL_S3=http://minio:9000 # Override specific service endpoint # SDK behaviour AWS_RETRY_MODE=adaptive # legacy | standard | adaptive AWS_MAX_ATTEMPTS=5 # Maximum retry attempts AWS_CA_BUNDLE=/path/to/cert.pem # Custom CA bundle for HTTPS (corporate proxies) # Container/ECS credential injection (set by ECS agent automatically) AWS_CONTAINER_CREDENTIALS_RELATIVE_URI=... # ECS task role endpoint path # Disable credential chain steps (useful for debugging) AWS_EC2_METADATA_DISABLED=true # Skip IMDS check (useful on non-EC2 to avoid timeout)
AWS_EC2_METADATA_DISABLED for faster startup on non-EC2: When running locally or in Docker, the SDK still tries to query http://169.254.169.254 as step 6 of the credential chain. On non-EC2 systems, this address isn't reachable and the connection times out — adding 1-2 seconds to the credential resolution time. Set AWS_EC2_METADATA_DISABLED=true in local development environments to skip this step and speed up SDK initialisation. In your Spring Boot application's local dev profile (application-dev.properties), set this to prevent slow startup when running locally.
AWS_CA_BUNDLE for corporate proxy environments: At Deloitte, your network traffic may go through a corporate proxy with SSL inspection. This proxy terminates and re-signs HTTPS connections with the corporate CA certificate. The AWS SDK doesn't trust the corporate CA by default — every AWS API call fails with "SSL certificate verification error." Solution: export the corporate CA certificate bundle and set AWS_CA_BUNDLE=/path/to/corporate-ca-bundle.pem. This tells the SDK to trust that CA for all HTTPS connections. This is an extremely common issue in enterprise environments and a complete mystery to engineers who don't know about it.
The AWS SDK v2 for Java is not just a REST client wrapper. It has a sophisticated architecture: pluggable credential providers, HTTP client abstraction (Netty or Apache), request interceptors, response handlers, waiters, paginators, and async execution models. Using it correctly with Spring Boot WebFlux requires understanding this architecture — using it incorrectly causes thread pool exhaustion, connection pool starvation, and silent data truncation.
AWS SDK v2 architecture — the major components
Understanding the layers that sit between your code and the AWS API

The AWS SDK v2 for Java was a complete rewrite from v1, designed around immutability, non-blocking I/O, and modular architecture. Every client in SDK v2 is immutable after construction — you create the client once at application startup and share it across all requests.

Service clients: Each AWS service has a dedicated client: S3Client, DynamoDbClient, SqsClient, etc. Each client manages: credential resolution, endpoint determination, request serialisation, HTTP transport, response deserialisation, retries, and error handling. Service clients are thread-safe and expensive to create — always create them as singletons (in Spring, as @Bean).

HTTP client: SDK v2 has a pluggable HTTP layer. By default it uses the URLConnection-based synchronous client. For production applications: use software.amazon.awssdk:netty-nio-client for async clients (S3AsyncClient, DynamoDbAsyncClient) and software.amazon.awssdk:apache-client for sync clients when you need connection pooling configuration. The default HTTP client has very limited connection pool size — a production high-throughput application will exhaust it and see connection timeouts.

The SDK request execution pipeline: Every SDK API call goes through a pipeline of execution interceptors before and after the HTTP call. You can add custom interceptors to: add headers to every request (correlation IDs, tracing headers), log request/response metadata, implement custom retry logic, add circuit breaker behaviour. This is how you integrate AWS SDK calls with your existing Spring Boot interceptors and observability stack without modifying every call site.
Creating SDK clients correctly in Spring Boot:
Never create SDK clients per-request — they're expensive (establish connection pools, load service models). Create once as Spring Beans:
Inject AwsCredentialsProvider (DefaultCredentialsProvider.create()) — don't hardcode credentials.
Inject the region from configuration: Region.of(awsRegion) where awsRegion comes from @Value("${aws.region}").
Configure the HTTP client with appropriate connection pool size for your traffic volume.
Use @Bean with @Scope("singleton") — this is Spring's default for @Bean but be explicit.
Shutdown the client in @PreDestroy: s3Client.close() — SDK clients hold resources (thread pools, connections) that must be released on application shutdown.
Sync vs async SDK clients — the critical choice for WebFlux
Why using the wrong client type in a reactive application causes performance catastrophe

AWS SDK v2 provides synchronous and asynchronous variants of every service client. S3Client (sync) blocks the calling thread until the response arrives. S3AsyncClient (async) returns a CompletableFuture immediately and completes it when the response arrives, without blocking any thread.

The WebFlux + blocking SDK client catastrophe: Your Spring Boot WebFlux application uses Netty as the server. Netty runs on a small pool of event loop threads (typically 2×CPU cores — 4-8 threads on a typical EC2). When a request arrives, a Netty event loop thread handles it. If you call S3Client.getObject() (blocking) inside a WebFlux handler, that event loop thread blocks waiting for S3 to respond (say, 200ms). During those 200ms, that thread cannot process any other requests. With 4 event loop threads and 200ms S3 latency, your maximum throughput is 4/0.2 = 20 requests/second — regardless of how many CPU cores or how much RAM you have. Under load, all event loop threads are blocked on S3, new requests queue up, latency spikes, the application appears hung. This does not throw an exception or log a warning — it silently destroys performance.
The correct pattern — S3AsyncClient with WebFlux:
// WRONG — blocks the Netty event loop thread @GetMapping("/file/{key}") Mono<byte[]> getFile(@PathVariable String key) { // This blocks! The S3Client.getObject() call parks this thread until S3 responds ResponseBytes<GetObjectResponse> response = s3Client.getObjectAsBytes( r -> r.bucket(bucket).key(key)); return Mono.just(response.asByteArray()); } // CORRECT — non-blocking, uses S3AsyncClient @GetMapping("/file/{key}") Mono<byte[]> getFile(@PathVariable String key) { return Mono.fromFuture( s3AsyncClient.getObjectAsBytes(r -> r.bucket(bucket).key(key)) ).map(ResponseBytes::asByteArray); // Returns immediately, Mono completes when S3 responds // No thread blocked during the wait } // If you must use sync S3Client in WebFlux, offload to a bounded scheduler: @GetMapping("/file/{key}") Mono<byte[]> getFileSafe(@PathVariable String key) { return Mono.fromCallable( () -> s3Client.getObjectAsBytes(r -> r.bucket(bucket).key(key)).asByteArray() ).subscribeOn(Schedulers.boundedElastic()); // Offloads blocking call to a dedicated thread pool, not event loop }

Connection pooling — the invisible bottleneck
Why default connection pool settings cause production failures under load

Every AWS SDK client maintains a pool of HTTPS connections to AWS service endpoints. Creating a new HTTPS connection requires a TCP handshake + TLS handshake — typically 50-200ms. For an application making thousands of S3 calls per second, connection reuse is essential. The SDK's connection pool handles this: connections are reused across requests, reducing per-request latency to the actual API call time.

Default connection pool sizes: The AWS SDK v2's default Apache HTTP client: 50 connections. Default Netty NIO client: 50 connections per hostname. For high-throughput applications these defaults are too small. A Spring Boot API handling 500 concurrent requests, each making one S3 call, needs 500 connections to S3 simultaneously. With 50 connections, 450 requests wait for a connection to become available — adding queuing latency on top of the actual S3 latency.

How to size the connection pool: Formula: pool_size = (peak_concurrent_requests × avg_s3_latency_ms) / 1000. If your app handles 500 concurrent requests, each S3 call takes 50ms: 500 × 0.05 = 25 connections minimum. Double for safety: 50. This matches the default — fine for moderate traffic. For high-traffic apps: profile your actual S3 call concurrency under peak load, measure connection wait time in the HTTP client metrics, and increase maxConcurrency (Netty) or maxConnections (Apache) until connection wait time is negligible.
Connection pool exhaustion failure pattern: Application works fine at low traffic. Under load, response times gradually increase. Logs show no errors. CloudWatch shows S3 latency is normal (10-20ms). But application latency is 500ms+. Why? The connection pool is saturated — requests queue waiting for a connection. The 500ms is mostly wait time in the connection pool, not S3 latency. Diagnosis: enable HTTP client metrics, look for connection wait time or pool exhaustion metrics. Fix: increase connection pool size, or reduce concurrency at the application level (rate limiting, request queuing), or switch to S3AsyncClient with larger Netty concurrency which is more efficient with connections under high load.
JMESPath is the query language used by the AWS CLI --query flag. Every AWS engineer who writes automation scripts uses it, and almost all of them use only 10% of its capability — resulting in scripts that pipe through grep, awk, and sed when a single --query expression would be cleaner, faster, and more portable. Mastering JMESPath is the difference between a 20-line shell script and a 2-line CLI command.
JMESPath fundamentals — navigating nested JSON
Dot notation, array indexing, wildcards, and flattening
# Basic dot notation — navigate nested structure aws ec2 describe-instances --query 'Reservations[0].Instances[0].InstanceId' # Gets InstanceId of the first instance in the first reservation # Array wildcard — get a field from every element aws ec2 describe-instances --query 'Reservations[*].Instances[*].InstanceId' # Returns [[id1, id2], [id3]] — nested arrays (one per Reservation) # Flatten with [] — collapse nested arrays into a single list aws ec2 describe-instances --query 'Reservations[*].Instances[*].InstanceId | []' # Returns [id1, id2, id3] — flat list # The | [] at the end is the flatten operator # Navigate deeper structures aws ec2 describe-instances --query 'Reservations[*].Instances[*].{ ID: InstanceId, Type: InstanceType, State: State.Name, IP: PublicIpAddress, AZ: Placement.AvailabilityZone } | []' # Returns a list of objects with renamed fields — much cleaner than full JSON
The output format and JMESPath interaction: --output json with --query returns JSON. --output text with --query returns tab-separated values — perfect for piping into bash variables or awk. --output table with --query returns an ASCII table — readable but not machine-parseable. For scripts: always use --output text when extracting a single value into a shell variable. INSTANCE_ID=$(aws ec2 describe-instances --query 'Reservations[0].Instances[0].InstanceId' --output text)
Filtering with conditions — the most powerful JMESPath feature
Filter arrays by field values using [?condition] syntax
# Filter instances by state — only running instances aws ec2 describe-instances --query \ 'Reservations[*].Instances[?State.Name==`running`].InstanceId | []' # [?condition] filters the array to elements matching the condition # Note: string values in conditions use backticks `running`, not quotes # Filter by tag value — find instances tagged Environment=production aws ec2 describe-instances --query \ 'Reservations[*].Instances[?Tags[?Key==`Environment`&&Value==`production`]].{ID: InstanceId, Type: InstanceType} | []' # Filter S3 buckets created after a date aws s3api list-buckets --query \ 'Buckets[?CreationDate>`2024-01-01`].Name' # Find IAM users with no MFA device aws iam get-account-summary --query 'SummaryMap.AccountMFAEnabled' # For per-user MFA check, use list-virtual-mfa-devices and join with list-users # Find security groups with 0.0.0.0/0 on port 22 (dangerous open SSH) aws ec2 describe-security-groups --query \ 'SecurityGroups[?IpPermissions[?FromPort==`22`&&IpRanges[?CidrIp==`0.0.0.0/0`]]].{Name: GroupName, ID: GroupId}' # This is a real security audit query — finds every SG with public SSH open
The backtick vs quote confusion in JMESPath conditions: In JMESPath, string literals in filter conditions must use backticks: [?State.Name==`running`]. Quotes (==`"running"`) produce a JSON string with literal quotes inside the JMESPath expression — it doesn't match anything. Numbers do NOT use backticks: [?Port==`22`] — wait, ports ARE backtick-wrapped because they're JSON literals in JMESPath. This is confusing: JSON numbers in JMESPath conditions also use backticks. The rule: anything that is a JSON literal value (string, number, boolean, null) in a JMESPath condition uses backtick notation. Only identifiers (field names, function names) are unquoted.

Robust scripting patterns — the difference between scripts that break production and scripts that don't
Error handling, idempotency, and --cli-input-json for complex commands
#!/bin/bash # Production-grade AWS script template set -euo pipefail # -e: exit on error -u: error on undefined var -o pipefail: pipe failures caught # Constants — never hardcode in script body REGION="ap-south-1" BUCKET="${APP_BUCKET:-my-default-bucket}" # Use env var with default # Logging function — timestamps every line log() { echo "[$(date -u +%Y-%m-%dT%H:%M:%SZ)] $*" >&2; } # Error handler — prints line number on failure trap 'log "ERROR: Script failed at line $LINENO"' ERR # Idempotent S3 bucket creation — safe to run multiple times create_bucket_if_not_exists() { local bucket=$1 if aws s3api head-bucket --bucket "$bucket" --region "$REGION" 2>/dev/null; then log "Bucket $bucket already exists — skipping" else log "Creating bucket $bucket" aws s3api create-bucket \ --bucket "$bucket" \ --region "$REGION" \ --create-bucket-configuration LocationConstraint="$REGION" log "Bucket $bucket created" fi } # Safe: handles "already exists" instead of failing create_bucket_if_not_exists "$BUCKET" # --cli-input-json for complex commands — avoids shell quoting hell cat > /tmp/launch-params.json << 'EOF' { "ImageId": "ami-0abcdef1234567890", "InstanceType": "t3.micro", "KeyName": "my-key", "SecurityGroupIds": ["sg-12345678"], "SubnetId": "subnet-12345678", "TagSpecifications": [{ "ResourceType": "instance", "Tags": [{"Key": "Name", "Value": "app-server"}, {"Key": "Environment", "Value": "production"}] }] } EOF aws ec2 run-instances --cli-input-json file:///tmp/launch-params.json
set -euo pipefail — why every AWS script must start with this: Without set -e: if aws ec2 stop-instances fails, the script continues to the next line and deletes the EC2 snapshot you were about to create as a backup. Without set -u: a typo in a variable name ($BUCKET_NAEM instead of $BUCKET_NAME) silently becomes an empty string — you run aws s3 rm s3:///critical-file.zip which might succeed and delete something unexpected. Without pipefail: aws ec2 describe-instances | grep "running" — if the aws command fails, grep sees empty input, finds no matches, exits 1, but because the pipe's exit code is the last command (grep), the overall exit is 1 — but the error from aws is silently lost. Always use set -euo pipefail.
Waiters and polling — waiting for AWS state transitions correctly
The right way to wait for EC2 to start, S3 buckets to exist, and deployments to complete

AWS operations are asynchronous. You call aws ec2 run-instances and get an instance ID back — but the instance is in "pending" state, not "running". You cannot SSH in yet. You need to wait. The naive approach — sleeping for 60 seconds — is fragile (sometimes it's faster, sometimes slower). The correct approach is using waiters.

# CLI waiters — poll with intelligent backoff until condition is met INSTANCE_ID=$(aws ec2 run-instances \ --image-id ami-xxx --instance-type t3.micro \ --query 'Instances[0].InstanceId' --output text) log "Waiting for instance $INSTANCE_ID to pass status checks..." aws ec2 wait instance-status-ok --instance-ids "$INSTANCE_ID" log "Instance is ready for SSH" # Common waiters you'll use constantly aws ec2 wait instance-running --instance-ids i-xxx # State = running aws ec2 wait instance-stopped --instance-ids i-xxx # State = stopped aws ec2 wait instance-terminated --instance-ids i-xxx # State = terminated aws s3api wait bucket-exists --bucket my-bucket # Bucket is accessible aws rds wait db-instance-available --db-instance-id my-db # RDS is available aws elbv2 wait load-balancer-available --load-balancer-arns arn:xxx aws cloudformation wait stack-create-complete --stack-name my-stack aws ecr wait image-scan-complete --repository-name my-repo --image-id imageTag=latest # Waiter timeout: default 600 seconds (10 minutes), then fails with error # For longer operations (RDS restore from snapshot): use custom polling loop wait_for_rds() { local db_id=$1 local max_attempts=60 # 60 × 30s = 30 minutes max local attempt=0 while [ $attempt -lt $max_attempts ]; do status=$(aws rds describe-db-instances \ --db-instance-identifier "$db_id" \ --query 'DBInstances[0].DBInstanceStatus' --output text) [ "$status" = "available" ] && { log "RDS ready"; return 0; } log "RDS status: $status (attempt $((attempt+1))/$max_attempts)" sleep 30 ((attempt++)) done log "ERROR: Timed out waiting for RDS" return 1 }
Parallel execution — processing 100 instances in 10 seconds instead of 100
GNU parallel and xargs for concurrent AWS operations with rate limiting
# Process many resources in parallel with xargs # Stop all running instances in parallel (instead of sequential loop) aws ec2 describe-instances \ --filters "Name=instance-state-name,Values=running" \ "Name=tag:Environment,Values=dev" \ --query 'Reservations[*].Instances[*].InstanceId | []' \ --output text | \ xargs -P 10 -I {} aws ec2 stop-instances --instance-ids {} # -P 10: up to 10 parallel processes -I {}: replace {} with each input line # GNU parallel — more control over rate limiting # Install: brew install parallel (Mac) or apt-get install parallel (Linux) aws ec2 describe-instances \ --query 'Reservations[*].Instances[*].InstanceId | []' \ --output text | tr '\t' '\n' | \ parallel --jobs 5 --delay 0.2 \ aws ssm send-command \ --instance-ids {} \ --document-name "AWS-RunShellScript" \ --parameters commands=["sudo yum update -y"] # --jobs 5: max 5 concurrent --delay 0.2: 200ms between job launches (avoid throttling) # Bulk tag resources — add tags to many resources efficiently # EC2 allows up to 1000 resource IDs per create-tags call INSTANCE_IDS=$(aws ec2 describe-instances \ --query 'Reservations[*].Instances[*].InstanceId | []' \ --output text | tr '\t' ' ') aws ec2 create-tags \ --resources $INSTANCE_IDS \ --tags Key=CostCenter,Value=Engineering Key=ManagedBy,Value=Terraform
Rate limiting awareness in parallel scripts: Running 100 parallel AWS CLI processes simultaneously means 100 simultaneous API calls. EC2 DescribeInstances allows ~100 calls/second — 100 parallel calls is fine. But IAM write operations are limited to ~1-10/second. Running 100 parallel aws iam attach-role-policy calls triggers throttling and retries that slow the script down more than sequential execution would have. Match parallelism to the API's actual rate limit. Use --delay in parallel or sleep between xargs batches for throttle-prone APIs.

You run it on your laptop, point your CLI or SDK at http://localhost:4566, and it behaves like real AWS for most operations — without making real API calls, incurring real costs, or requiring network access. Ideal for development, integration tests, and CI pipelines.

Q1 Your Spring Boot application on EC2 suddenly cannot call S3. The IAM role hasn't changed. What do you check? Hard▼

This is a systematic debugging question. The symptom — working before, broken now, no IAM changes — points to several possible causes that I'd check in order.

First: verify the identity and credentials. SSH into the EC2 and run aws sts get-caller-identity. This tells me: are there valid credentials? Is it using the expected role? If this fails with "Unable to locate credentials", the instance profile is detached or IMDS is unreachable. If it returns the wrong role, there's an environment variable override in effect (AWS_ACCESS_KEY_ID set somewhere).

Second: verify IMDS is accessible. curl -s http://169.254.169.254/latest/meta-data/iam/security-credentials/. If this returns nothing or 401 (if IMDSv2 was just enforced), the instance can't get credentials. Check if IMDSv2 was recently enforced but the application's SDK version doesn't support it.

Third: test S3 connectivity directly from the instance. aws s3 ls s3://bucket-name. If this works from CLI but fails from the application, the issue is the application's SDK configuration — likely it's using hardcoded credentials or a different credential source than the instance role.

Fourth: check for a bucket policy change. A change to the S3 bucket policy could have added an explicit Deny or removed an allow for this role. aws s3api get-bucket-policy --bucket bucket-name. Even though the IAM role didn't change, the bucket policy (resource policy) is the other half of the access evaluation — and it may have changed.

Fifth: check VPC network connectivity. Did someone add a NACL rule or modify a Security Group outbound rule? If the EC2 can't reach S3's IP addresses (or the S3 VPC Endpoint if one exists), API calls time out before IAM even evaluates them. VPC Flow Logs would show REJECT entries for outbound traffic to S3's IP range.

Sixth: check if a KMS key policy changed. If the S3 bucket uses SSE-KMS encryption, the EC2 role also needs kms:GenerateDataKey and kms:Decrypt on the KMS key. A KMS key policy change could have removed these permissions — manifesting as an S3 access failure even though the IAM role hasn't changed.

Why this answer works: It covers the full diagnostic chain — credentials, IMDS, direct testing, bucket policy, network, and KMS. The KMS key policy point is the one most engineers miss — it's a real production scenario where S3 access breaks due to KMS changes, not IAM or S3 changes directly.
Q2 Explain the AWS SDK credential chain and how you'd configure it differently for local development vs EC2 production. Medium▼

The DefaultCredentialsProvider checks credential sources in order, returning on the first one found: Java system properties → environment variables → ~/.aws/credentials (default or AWS_PROFILE named profile) → ~/.aws/config with role_arn → ECS container credentials endpoint → EC2 IMDS.

Local development configuration: I use named profiles in ~/.aws/config that assume the appropriate IAM role. My base credentials (IAM user for my Deloitte account) are in ~/.aws/credentials. The config file has profiles for each environment — dev, staging, prod — each with a role_arn pointing to the account and external_id for client accounts. When I run the application locally, I set AWS_PROFILE=dev-environment. The SDK finds the config profile, sees a role_arn, calls STS AssumeRole using my base credentials, and gets temporary role credentials. My application uses the same code path as production — just different credentials.

EC2 production configuration: The EC2 instance has an IAM instance profile attached — the least-privilege role for that service. No environment variables are set for credentials. No credentials files exist on the EC2. The SDK's credential chain eventually reaches IMDS at step 6, fetches the role's temporary credentials, and caches them. The application code is identical to local — the DefaultCredentialsProvider handles the difference transparently. I set AWS_EC2_METADATA_DISABLED=false explicitly in production to be clear, and AWS_EC2_METADATA_DISABLED=true in local development profiles to skip the IMDS timeout on non-EC2 machines.

The key principle: application code should never have any credential configuration hardcoded. The environment provides credentials through the chain. This makes the same JAR deployable to local, CI, staging, and production without modification.

Why this answer works: It explains the chain mechanically, then shows you understand that the same credential chain is used differently in different environments — and that good SDK usage means the application is completely environment-agnostic. The AWS_EC2_METADATA_DISABLED detail shows operational depth.
Q3 What is AWS SigV4 and why does it matter beyond "it signs requests"? Medium▼

SigV4 is AWS's request authentication and integrity protocol. Every API call to AWS is signed using HMAC-SHA256 with a key derived from the caller's secret access key, the date, the target region, and the target service. The signature covers the full request — method, URI, headers, and body hash.

Beyond authentication, SigV4 provides three important security properties. First, request integrity — if any part of the request is modified in transit (man-in-the-middle attack), the signature becomes invalid and AWS rejects it. An attacker cannot change the action you're requesting, the resource you're targeting, or any parameter without invalidating the signature. Second, replay attack prevention — the signature includes a timestamp and AWS requires it to be within 15 minutes of current time. A captured valid request cannot be replayed after 15 minutes. Third, scope restriction — the derived signing key is specific to the date, region, and service. A key used to sign an S3 request in ap-south-1 cannot be used to sign an EC2 request in us-east-1. Even if an attacker captures a signed request, they cannot use the signature to call different services or regions.

The practical implications: the 15-minute clock skew limit means systems with drifted clocks break mysteriously. The region-scoped signing means the region must be configured correctly before any API call — the signature will fail if you sign for us-east-1 but send to ap-south-1. Pre-signed URLs include all signature components in the URL itself, making them self-authenticating but also meaning any modification to the URL invalidates it.

Why this answer works: It goes beyond "it authenticates requests" to explain the three security properties (integrity, replay prevention, scope restriction). The practical implications — clock skew, region configuration, pre-signed URL immutability — show you've worked with this in real systems, not just read the documentation.
Q4 Why would you use AWS SDK v2 AsyncClient instead of the sync client in a Spring Boot WebFlux application? What goes wrong if you use the sync client? Hard▼

Spring Boot WebFlux uses Netty as its HTTP server, running on a small pool of event loop threads — typically 2×CPU cores, so 4-8 threads on a standard EC2 instance. Event loop threads are designed for non-blocking work: they receive requests, execute fast synchronous logic, return Mono/Flux to be scheduled, and move on. They must never block.

If you use S3Client (sync) inside a WebFlux handler, the call blocks the event loop thread while waiting for S3 to respond — typically 10-200ms. During that time, the thread cannot accept or process any other requests. With 4 event loop threads and 100ms average S3 latency: maximum throughput is 4/0.1 = 40 requests/second before threads are exhausted. All subsequent requests queue up, timeout, or fail. The application appears completely hung under moderate load. This produces no exception, no warning, no obvious error — just gradually increasing latency until the application stops responding. It is one of the hardest performance bugs to diagnose because everything appears correct when looking at individual requests.

S3AsyncClient returns CompletableFuture immediately without blocking any thread. The response is handled by Netty's IO threads when the response arrives from S3. Bridging to Reactor: Mono.fromFuture(s3AsyncClient.getObjectAsBytes(r -> ...)). The event loop thread schedules this and immediately continues to the next request. When S3 responds, the Mono completes and the continuation runs on a Reactor scheduler. Thread utilisation is proportional to actual computation, not waiting.

If you must use the sync S3Client in WebFlux (e.g., for a library that doesn't support async), offload to Schedulers.boundedElastic(): Mono.fromCallable(() -> s3Client.getObjectAsBytes(...)).subscribeOn(Schedulers.boundedElastic()). This runs the blocking call on a dedicated elastic thread pool, not the event loop. The event loop is never blocked. The elastic pool grows to handle concurrency. This is the escape hatch but not the ideal solution — S3AsyncClient with proper concurrency configuration is always preferred.

Why this answer works: The arithmetic (4 threads × 100ms latency = 40 req/s maximum) makes the performance impact concrete and memorable. The diagnosis difficulty point shows you understand why this class of bug is dangerous. The boundedElastic escape hatch shows you know the workaround when you can't change the client type.

aws configure list # Shows: name, value, type (env/config-file), location for each setting # Verify: region = ap-south-1, credentials from profile or env vars aws sts get-caller-identity # Shows: Account ID, UserID, ARN # Verify: it's YOUR account, not a client's account # Enable CLI auto-complete (one-time setup) complete -C aws_completer aws echo 'complete -C aws_completer aws' >> ~/.bashrc # Set default output and region for this session export AWS_DEFAULT_REGION=ap-south-1 export AWS_DEFAULT_OUTPUT=json
These two commands are your "am I pointing at the right account?" check. Run them before any destructive operation. The --debug flag on sts get-caller-identity shows you the full SigV4 signing process if you want to see it.

JMESPath queries against your real infrastructure
Run these queries against your Day 4 VPC and Day 2 EC2 instance. Observe how JMESPath transforms raw AWS JSON into exactly the data you need. These exact queries will appear in your daily work.
# List all VPCs with their CIDR and name aws ec2 describe-vpcs --query \ 'Vpcs[*].{ID: VpcId, CIDR: CidrBlock, Default: IsDefault, Name: Tags[?Key==`Name`].Value | [0]}' \ --output table # List all subnets with type (public/private) and AZ aws ec2 describe-subnets --query \ 'Subnets[*].{ID: SubnetId, CIDR: CidrBlock, AZ: AvailabilityZone, Public: MapPublicIpOnLaunch, Name: Tags[?Key==`Name`].Value | [0]}' \ --output table # Find all EC2 instances with their state, type, and IPs aws ec2 describe-instances --query \ 'Reservations[*].Instances[*].{ ID: InstanceId, Type: InstanceType, State: State.Name, PrivateIP: PrivateIpAddress, PublicIP: PublicIpAddress, Name: Tags[?Key==`Name`].Value | [0]} | []' \ --output table # List all security groups with inbound rule count aws ec2 describe-security-groups --query \ 'SecurityGroups[*].{Name: GroupName, ID: GroupId, VPC: VpcId, Rules: length(IpPermissions)}' \ --output table # SECURITY AUDIT: Find any security group with 0.0.0.0/0 on SSH aws ec2 describe-security-groups --query \ 'SecurityGroups[?IpPermissions[?ToPort==`22` && IpRanges[?CidrIp==`0.0.0.0/0`]]] .{Name: GroupName, ID: GroupId}' \ --output table
Save the security audit query as a shell alias — run it regularly against your accounts. Any result means you have port 22 exposed to the internet. That's a finding you should fix immediately.

Debug the credential chain with --debug
Use --debug on a real command to see the full SigV4 signing process, credential source, and HTTP request. This is what you'll do when a command fails unexpectedly.
# See the full request pipeline for a simple command aws s3 ls --debug 2>&1 | head -80 # Look for: # - "Found credentials in..." — which chain step found credentials # - "Calculating signature using..." — SigV4 signing happening # - "http request: GET https://..." — the actual HTTP request # - The Authorization header with the signature # Deliberately cause a credential error to see what it looks like AWS_ACCESS_KEY_ID=FAKEKEYID123 \ AWS_SECRET_ACCESS_KEY=fakeSecretKey \ aws s3 ls --debug 2>&1 | tail -20 # See: "InvalidClientTokenId" error, observe at what step it fails # Test the clock skew error (simulated) # On your EC2, if you've never seen this: # sudo date -s "$(date -d '20 minutes ago' +%Y-%m-%d\ %H:%M:%S)" # aws s3 ls -- will fail with "Signature not yet current" # sudo ntpdate pool.ntp.org -- fix clock # aws s3 ls -- works again
The --debug output is verbose but invaluable. The line "Found credentials in..." tells you exactly which credential source is being used. Redirect to a file for complex debugging: aws s3 ls --debug 2>debug.log && grep "Found credentials" debug.log
4
Observe pagination — verify the CLI auto-paginates correctly
Create enough S3 objects to trigger pagination. Observe the difference between paginated and non-paginated results. This makes the pagination mechanics visceral.
# Create 5 test objects in your Day 3 bucket for i in $(seq 1 5); do echo "Object $i content" | aws s3 cp - s3://YOUR-BUCKET/pagination-test/obj-$i.txt done # List with auto-pagination (default) -- gets all objects aws s3api list-objects-v2 \ --bucket YOUR-BUCKET \ --prefix pagination-test/ \ --query 'Contents[*].Key' # List with page size 2 -- forces multiple API calls behind the scenes aws s3api list-objects-v2 \ --bucket YOUR-BUCKET \ --prefix pagination-test/ \ --page-size 2 \ --query 'Contents[*].Key' # Still returns all 5 -- CLI auto-paginates across 3 pages of 2 items each # List with --no-paginate and --page-size 2 -- only first page aws s3api list-objects-v2 \ --bucket YOUR-BUCKET \ --prefix pagination-test/ \ --page-size 2 \ --no-paginate \ --query 'Contents[*].Key' # Returns only 2 items -- THIS is the truncation bug in scripts # Clean up test objects aws s3 rm s3://YOUR-BUCKET/pagination-test/ --recursive
The last command demonstrates exactly how pagination bugs manifest in scripts — you get partial data with no error. If this were a production script processing S3 object keys, 3 out of 5 objects would be silently skipped.

Set up LocalStack for offline AWS development
Install and start LocalStack. Run basic S3 and SQS operations against it. This sets up your offline development environment for all future days.
# Install LocalStack CLI pip install localstack --break-system-packages # Start LocalStack (requires Docker running) localstack start -d # Or with Docker directly: docker run -d \ --name localstack \ -p 4566:4566 \ -e SERVICES=s3,sqs,dynamodb,secretsmanager \ localstack/localstack:3.0 # Wait for it to be ready localstack wait # Add localstack profile to AWS config aws configure --profile localstack set region ap-south-1 aws configure --profile localstack set output json # Test: create a bucket in LocalStack aws s3 mb s3://local-test-bucket \ --endpoint-url http://localhost:4566 \ --profile localstack # Or configure endpoint_url in your profile (AWS CLI v2): # Add to ~/.aws/config under [profile localstack]: # endpoint_url = http://localhost:4566 # Then: aws s3 ls --profile localstack (no --endpoint-url needed) # Upload, list, download echo "LocalStack test" | aws s3 cp - s3://local-test-bucket/test.txt \ --endpoint-url http://localhost:4566 aws s3 ls s3://local-test-bucket --endpoint-url http://localhost:4566 # Create SQS queue and send a message aws sqs create-queue --queue-name test-queue \ --endpoint-url http://localhost:4566 aws sqs send-message \ --queue-url http://localhost:4566/000000000000/test-queue \ --message-body "Hello from LocalStack" aws sqs receive-message \ --queue-url http://localhost:4566/000000000000/test-queue
LocalStack is now your development AWS environment. You can run all future lab exercises against it before trying on real AWS, eliminating costs and the risk of misconfiguring real resources during learning.

Write a Phase 1 infrastructure audit script
Combine everything from Days 1-5 into one audit script that gives you a complete picture of your AWS account's state. This is something you'll run at the start of every project and every security review.
cat > aws-audit.sh << 'SCRIPT' #!/bin/bash set -euo pipefail REGION="${AWS_DEFAULT_REGION:-ap-south-1}" echo "=== AWS Account Audit: $(date -u) ===" echo "Account: $(aws sts get-caller-identity --query Account --output text)" echo "Region: $REGION" echo "" echo "--- VPCs ---" aws ec2 describe-vpcs --query \ 'Vpcs[*].{ID:VpcId,CIDR:CidrBlock,Default:IsDefault}' \ --output table echo "--- Running EC2 Instances ---" aws ec2 describe-instances \ --filters "Name=instance-state-name,Values=running" \ --query 'Reservations[*].Instances[*].{ID:InstanceId,Type:InstanceType,AZ:Placement.AvailabilityZone,IP:PrivateIpAddress} | []' \ --output table echo "--- S3 Buckets ---" aws s3api list-buckets --query 'Buckets[*].{Name:Name,Created:CreationDate}' --output table echo "--- SECURITY: Open SSH (0.0.0.0/0 on port 22) ---" aws ec2 describe-security-groups --query \ 'SecurityGroups[?IpPermissions[?ToPort==`22`&&IpRanges[?CidrIp==`0.0.0.0/0`]]].{Name:GroupName,ID:GroupId}' \ --output table echo "--- Unattached EBS Volumes (you are paying for these) ---" aws ec2 describe-volumes \ --query 'Volumes[?State==`available`].{ID:VolumeId,Size:Size,AZ:AvailabilityZone}' \ --output table echo "--- NAT Gateways (check costs) ---" aws ec2 describe-nat-gateways \ --filter "Name=state,Values=available" \ --query 'NatGateways[*].{ID:NatGatewayId,State:State,Subnet:SubnetId}' \ --output table SCRIPT chmod +x aws-audit.sh ./aws-audit.sh
Save this script. Run it at the start of every lab session to see what's running and what's costing money. The unattached EBS volumes and NAT Gateways sections will save you from surprise AWS bills.
Day 6 — CLI & SDK mechanics
Phase 1 complete: Cloud Foundations. Days 7–13 begin application deployment.
CLI request pipeline
Parse → credential resolution → construct request → SigV4 sign → HTTP → format response. --debug shows every step.
SigV4 signing
HMAC-SHA256 over canonical request. Covers full request — modification invalidates signature. 15-min clock skew limit. Region+service scoped.
Credential chain (order)
1. Java props → 2. Env vars → 3. ~/.aws/credentials → 4. ~/.aws/config role_arn → 5. ECS endpoint → 6. EC2 IMDS
Config file quirk
~/.aws/credentials: [profile-name]. ~/.aws/config: [profile profile-name]. Missing "profile " prefix = profile not found error.
Pagination default
CLI auto-paginates by default. --no-paginate gives first page only. --page-size controls page size. SDK paginators do this correctly in Java.
Exponential backoff + jitter
SDK retries throttled (429) and transient (5xx) errors. Jitter prevents thundering herd. Use adaptive retry mode for high-concurrency apps.
S3AsyncClient in WebFlux
Never use blocking S3Client on Netty event loop threads. S3AsyncClient returns CompletableFuture → bridge with Mono.fromFuture(). Or offload blocking to Schedulers.boundedElastic().
SDK clients are singletons
Expensive to create (pools, service models). Create once as @Bean. Close in @PreDestroy. Thread-safe. Never create per-request.
JMESPath tags pattern
Tags[?Key==`Name`].Value | [0] — extracts a specific tag value. Returns null if tag absent rather than erroring.
LocalStack
Docker-based AWS emulator. Community: S3, SQS, DynamoDB, Lambda. Configure with --endpoint-url or per-profile endpoint_url in ~/.aws/config.
set -euo pipefail
Every AWS script must start here. -e: exit on error. -u: fail on undefined vars. -o pipefail: pipe errors caught. Prevents silent data corruption.
AWS_EC2_METADATA_DISABLED
Set to true in local dev to skip IMDS timeout (saves 1-2s startup). False/unset on EC2. Speeds up SDK initialisation outside AWS.
