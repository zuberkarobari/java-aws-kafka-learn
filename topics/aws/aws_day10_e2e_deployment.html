Request flow — what happens on every API call end to end
Tracing a single HTTP request from the mobile app through every layer to the database and back

Tracing a single request from a GetWork user submitting a job posting illustrates how all five layers interact. Following one request through the full stack also reveals every possible failure point.

Step 1 — DNS resolution. The GetWork mobile app (React Native, Firebase on frontend) calls https://api.getwork.app/api/v1/jobs. The device resolves api.getwork.app via Route 53 — which returns the ALB's DNS name. The ALB's DNS name resolves to one of the ALB's public IPs (ENIs in ap-south-1a or 1b public subnets).

Step 2 — TLS termination at ALB. The HTTPS connection is established with the ALB. TLS handshake uses the ACM certificate attached to the HTTPS listener. The ALB decrypts the request. The ALB's inbound Security Group (alb-sg) allows TCP 443 from 0.0.0.0/0.

Step 3 — ALB routing decision. The ALB evaluates listener rules. The POST to /api/v1/jobs matches the path rule /api/v1/* → forwards to the getwork-api-tg target group. The ALB selects a healthy EC2 instance from the target group (round robin or least-outstanding-requests). The ALB makes a new TCP connection to the instance's private IP on port 8080, forwarding the request with X-Forwarded-For: [client IP] and X-Forwarded-Proto: https headers. The app-sg Security Group on the EC2 allows TCP 8080 from alb-sg.

Step 4 — Spring Boot request handling. The request arrives at the Spring Boot application on port 8080. The application reads the X-Forwarded-* headers (because server.forward-headers-strategy=native is configured). JWT validation occurs. The request reaches the JobController. The controller calls JobService.createJob() with @Transactional — HikariCP provides a connection to RDS from the write pool.

Step 5 — RDS MySQL write. The JDBC connection goes to the RDS primary endpoint. The EC2's app-sg is the source; the db-sg Security Group allows TCP 3306 from app-sg. The INSERT is executed, committed, and acknowledged. If Multi-AZ is enabled, the write has already been synchronously replicated to the standby before MySQL returns success. HikariCP returns the connection to the pool.

Step 6 — S3 upload (if job has images). If the job posting includes images, the Spring Boot app calls S3AsyncClient.putObject(). The request goes to S3 via the VPC Gateway Endpoint — no NAT Gateway, no public internet. The S3 bucket policy allows PutObject from the VPC Endpoint ID. The IAM instance role has s3:PutObject on the bucket ARN.

Step 7 — Response. Spring Boot returns HTTP 201 Created with the job ID. The ALB forwards the response to the client. The client's HTTP connection to the ALB closes. The backend connection from ALB to EC2 returns to the ALB's connection pool (HTTP/1.1 keep-alive) for potential reuse.

The complete security boundary at each step: Client → ALB: Security Group alb-sg allows 443 from internet. ALB → EC2: Security Group app-sg allows 8080 from alb-sg (SG reference, not CIDR). EC2 → RDS: Security Group db-sg allows 3306 from app-sg. EC2 → S3: VPC Gateway Endpoint + bucket policy restricts S3 access to VPC Endpoint. EC2 → Secrets Manager: VPC Interface Endpoint + IAM role with secretsmanager:GetSecretValue. Every hop has an explicit allow — deny by default at every boundary. No wildcards, no CIDR open rules for internal traffic.

Failure modes map — what breaks when each component fails
Knowing the blast radius of every failure is what separates reactive engineers from proactive ones

For each component in the architecture, what is the user impact when it fails, how is it detected, and how is it recovered?

Single EC2 instance fails
Impact: ALB stops routing to the failed instance within 30 seconds (2 failed health checks × 15s interval). Requests are redistributed to remaining healthy instances. No user-visible impact if ≥2 instances are healthy. Recovery: ASG replaces terminated instance automatically. Detection: ALB HealthyHostCount alarm, ASG activity log.
All EC2 instances in one AZ fail (AZ outage)
Impact: ALB routes all traffic to remaining AZ's instances. If ASG has instances in both AZs (always configure this), impact is absorbed. Capacity is halved until ASG rebalances. Recovery: ASG launches replacement instances in the healthy AZ, then rebalances when the failed AZ recovers. Detection: CloudWatch — sudden drop in ALB target count, ASG instance count alert.
RDS primary instance fails
Impact: 20–120 seconds of database unavailability. Write operations fail during this window. HikariCP retries re-establish connections to the new primary after DNS propagates. Recovery: Automatic Multi-AZ failover. JVM DNS TTL=5s ensures fast reconnection. Resilience4j retry handles transient failures. Detection: CloudWatch RDS event notification, application error rate spike.
S3 service degradation
Impact: Image uploads fail. Core API functionality (job CRUD, worker management) continues because S3 is only used for media storage, not primary data. Recovery: S3 degradation is AWS-managed and self-heals. Retry failed uploads. Detection: S3 SDK errors in application logs, custom CloudWatch metric for S3 operation failures.
Secrets Manager unreachable
Impact: Application cannot start (cannot fetch database credentials at startup). Existing running instances are unaffected — they fetched secrets at their last startup. New instances fail to launch, ASG may enter a crash loop. Recovery: Use VPC Interface Endpoint for Secrets Manager to avoid public internet dependency. Cache the secret value in application memory after first fetch. Detection: EC2 instance launch failures in ASG activity log, cloud-init errors.
The Secrets Manager unavailability design pattern: If Secrets Manager is temporarily unreachable when your Spring Boot app starts, the startup fails. Under an ASG with health check grace period, the ASG terminates the failed instance and tries again — potentially in a loop if Secrets Manager remains unreachable. Solution: implement Secrets Manager retrieval with retry logic (3 attempts, exponential backoff) at application startup. If all retries fail, log a clear error and exit with a non-zero code. The ASG activity log will show the startup failure clearly. Additionally, use a VPC Interface Endpoint for Secrets Manager — this routes all Secrets Manager API calls through the VPC private network, eliminating public internet dependency and providing more reliable connectivity.

The initialization-fail-timeout=-1 safety valve and its implication: Without this setting, if Secrets Manager is unreachable when Spring Boot starts, the DataSource bean creation fails and the application refuses to start. This seems safe but creates a problem: during an ASG rolling deployment, if a new instance launches while Secrets Manager has a brief API hiccup, the instance fails to start, the ASG marks it unhealthy, terminates it, and immediately tries another. If the hiccup lasts several minutes, the ASG can burn through its entire pool of instances trying to launch healthy ones, leaving zero capacity. With initialization-fail-timeout=-1, the DataSource starts without a pre-validated connection. The first actual database query will fail and trigger HikariCP's retry logic — gracefully degrading rather than refusing to start. The trade-off: startup errors are deferred to the first query rather than detected at launch.

spring.jpa.open-in-view=false — the setting that prevents the most common connection leak: Open Session in View (OSIV) is enabled by default in Spring Boot. It keeps a JPA session (and therefore a database connection from HikariCP) open for the entire duration of an HTTP request — including during response serialisation. If your Jackson serialiser lazily loads related entities during JSON serialisation (after the service method has returned), OSIV provides the connection for that. The problem: a connection is held for the full HTTP request duration, not just the database operation duration. Under load, this exhausts the connection pool. With OSIV disabled, lazy loading during serialisation throws LazyInitializationException — which is the correct behavior that forces you to eagerly load all needed data in the transaction. This catches lazy loading bugs at development time rather than manifesting as production connection pool exhaustion.
Custom health indicators — ALB health checks that reflect real readiness
Implementing readiness and liveness health indicators that the ALB uses to make intelligent routing decisions
// DatabaseHealthIndicator.java — included in readiness check @Component public class DatabaseHealthIndicator implements HealthIndicator { private final DataSource dataSource; @Override public Health health() { try (Connection conn = dataSource.getConnection(); Statement stmt = conn.createStatement()) { ResultSet rs = stmt.executeQuery("SELECT 1"); rs.next(); return Health.up() .withDetail("response", rs.getInt(1)) .withDetail("pool.active", getActiveConnections()) .withDetail("pool.max", getMaxConnections()) .build(); } catch (SQLException e) { return Health.down() .withException(e) .withDetail("message", "Database unavailable") .build(); } } } // ApplicationReadinessIndicator.java // Spring Boot 2.3+ supports Readiness state — use this for ALB health check @Component public class ApplicationReadinessIndicator implements ApplicationReadinessCheck { private final DatabaseHealthIndicator dbHealth; private final S3HealthIndicator s3Health; // Called by /actuator/health/readiness // Returns ACCEPTING_TRAFFIC or REFUSING_TRAFFIC @Override public ApplicationAvailability.State check() { Health dbStatus = dbHealth.health(); // Only db health gates readiness — S3 degradation doesn't block all traffic return dbStatus.getStatus().equals(Status.UP) ? ReadinessState.ACCEPTING_TRAFFIC : ReadinessState.REFUSING_TRAFFIC; } } // GracefulShutdown — connects to server.shutdown=graceful // When SIGTERM received, Spring Boot stops accepting new requests, // completes in-flight requests up to spring.lifecycle.timeout-per-shutdown-phase, // then closes HikariCP pool and exits. // The ALB deregistration delay (60s) must be >= the shutdown timeout (30s) // to ensure the ALB stops sending new requests before the app stops accepting them.
The graceful shutdown sequence and why the timing matters: When the ASG terminates an EC2 instance (scale-in or rolling deployment), the following sequence must happen in order: (1) ASG lifecycle terminate hook fires → notifies the ALB to start connection draining (deregistration delay = 60s). (2) During draining, ALB stops sending new requests to this instance but completes in-flight ones. (3) After deregistration completes, AWS sends SIGTERM to the Spring Boot process. (4) Spring Boot's graceful shutdown accepts no new requests, completes in-flight requests within 30s, closes the HikariCP pool, and exits. The deregistration delay (60s) must be greater than the graceful shutdown timeout (30s). If they're reversed — SIGTERM arrives before draining completes — the app stops accepting requests while the ALB is still trying to send them.

SSM Session Manager — SSH without port 22 open: The SSMSessionManager permissions block allows AWS Systems Manager Session Manager to open a shell into your EC2 instance. Instead of SSH (which requires port 22 open in your Security Group and a key pair), Session Manager establishes a session via the SSM agent and AWS's control plane — no inbound port required. The EC2 Security Group has no port 22 rule. Engineers connect via: aws ssm start-session --target i-instanceid. All session activity is logged to CloudTrail. For your Deloitte project: this is the required access mechanism for instances in private subnets where you have no direct network path for SSH, and it satisfies audit requirements because every command is logged.
The cloudwatch:PutMetricData without namespace condition — a silent over-permission: Many engineers write Action: cloudwatch:PutMetricData, Resource: * without a namespace condition. This allows the EC2 instance to write metrics into any CloudWatch namespace — including AWS-managed namespaces like AWS/EC2, AWS/RDS. A bug in application code or a compromised instance could write misleading data into AWS's own monitoring namespaces, causing false alarms or masking real issues. The namespace condition restricts metric writes to your application's own namespace. Apply this pattern to all CloudWatch permissions.
Why resource ARN scoping matters — the blast radius calculation
The difference between a compromised instance having access to one bucket versus all buckets in your account

Consider two IAM policies for S3 access:

Over-permissive: Action: s3:*, Resource: * — grants all S3 operations on all buckets. If this EC2 instance is compromised via SSRF or supply chain attack, the attacker has full access to every S3 bucket in your account — production data, backups, other application data, CloudTrail logs.

Least-privilege: Action: [s3:GetObject, s3:PutObject], Resource: arn:aws:s3:::getwork-app-data/* — grants only the specific operations the application needs on the specific bucket it uses. If this instance is compromised, the attacker can only access job posting images and upload files — not your RDS snapshots, other application's data, or account-wide data.

The IAM Access Analyzer validation step for every policy: Before attaching any policy, run it through IAM Access Analyzer's policy validation: aws accessanalyzer validate-policy --policy-document file://policy.json --policy-type IDENTITY_POLICY. This catches: over-broad wildcards (s3:* when you only need GetObject), typos in action names (s3:GetObiect silently grants nothing), incorrect resource ARN formats, and known privilege escalation patterns. Make this a mandatory pre-commit check in your Bitbucket pipeline for any IAM policy changes. A 30-second CLI call prevents silent IAM misconfigurations that take hours to diagnose in production.

The Condition block on cloudwatch:PutMetricData: Restricts metric writes to the GetWork/API namespace. This means the application cannot pollute other namespaces and cannot forge metrics in AWS's own service namespaces (AWS/EC2, AWS/RDS) to mask real infrastructure issues.

The Condition block on s3:ListBucket: Restricts the prefix scan to uploads/* and exports/*. The application cannot enumerate all objects in the bucket — only the paths it legitimately needs to list. This limits the data surface area if credentials are compromised.

VPC Endpoints and their resource policies — the final security layer
How VPC Endpoints restrict which buckets and secrets can be accessed, even from instances with broad IAM permissions

VPC Endpoints add a second permission boundary that's evaluated independently of IAM. Even if an EC2 instance's IAM role is over-permissive, VPC Endpoint policies can restrict what that identity can access when going through the endpoint.

S3 Gateway Endpoint policy: Restrict the S3 Gateway Endpoint to only allow access to your specific buckets. This means any EC2 in the VPC using the Gateway Endpoint (which is all traffic to S3 from private subnets — it's free, so you route everything through it) can only access the whitelisted buckets. An attacker who compromises an EC2 with a broad IAM policy still cannot access other accounts' S3 buckets or other buckets in your account through this endpoint.

The VPC-bound S3 bucket policy — the most underused security control: Adding this Deny condition to your S3 bucket policy means the bucket is inaccessible from anywhere except your VPC. Stolen IAM credentials used from an attacker's laptop cannot access the bucket because the request doesn't come through your VPC Endpoint. This one policy change elevates your S3 security from "protected by IAM credentials" to "protected by both IAM credentials AND network location." For a financial services project like your JP Morgan engagement, this is a standard requirement, not an optional hardening step.

Bitbucket OIDC — why oidc: true eliminates the biggest CI/CD security risk: Without OIDC, you'd store AWS_ACCESS_KEY_ID and AWS_SECRET_ACCESS_KEY as Bitbucket repository variables — static, long-lived credentials that could be exposed if the Bitbucket account is compromised. With oidc: true, Bitbucket generates a fresh OIDC token for each pipeline step. The pipeline exchanges this token for temporary AWS credentials by assuming an IAM role. The temporary credentials expire when the pipeline step completes. No static keys anywhere. The IAM role's trust policy specifies the exact Bitbucket workspace and repository allowed to assume it. This is the same OIDC mechanism covered on Day 5 for GitHub Actions — now applied to your actual Bitbucket pipeline at Deloitte.

Log retention policy — set it or pay indefinitely: By default, CloudWatch Log Groups retain logs forever and charge you per GB stored. When something breaks in this five-layer architecture, knowing which layer to investigate first prevents the 2AM spiral of checking random things for 3 hours. This systematic troubleshooting guide walks from the outermost layer (client) to the innermost (database), identifying the specific signals at each layer that narrow down the root cause.
The systematic diagnostic flow — outer to inner
Start with the outermost observable symptom and work inward — never start in the middle

Step 1 — What does the user see? 5xx error (server problem), 4xx error (client or auth problem), timeout (slow response), or unexpected data (logic/data integrity issue). This determines the severity and likely layer.

Step 2 — ALB layer check. Check ALB access logs or CloudWatch: is the error originating at the ALB (HTTP 502, 503) or forwarded from the target (5xx from your app)? HTTP 502 = ALB received an invalid response from your EC2. HTTP 503 = ALB has no healthy targets. HTTP 504 = ALB gateway timeout (your app took too long). Each error code tells you something specific about which layer failed.

Step 3 — Target group health. aws elbv2 describe-target-health --target-group-arn TG_ARN. Any targets showing "unhealthy" means the Spring Boot app is failing health checks. Check the reason: "Response code mismatch" (app returning unexpected status), "Connection error" (app not listening on port 8080), "Timeout" (app responding slowly).

Step 4 — Application logs. aws logs tail /getwork/app --since 10m --filter-pattern "ERROR". Recent errors reveal the exception type and stack trace. Database connection errors, S3 SDK errors, OOM errors — all visible here without SSH.

Step 5 — RDS check. aws rds describe-db-instances --db-instance-identifier getwork-db --query 'DBInstances[0].DBInstanceStatus'. Check CloudWatch RDS metrics: CPU, connections, free storage, replica lag. Performance Insights for slow query identification.

Step 6 — If all else looks normal — CloudTrail. Recent IAM permission errors, unexpected API calls, configuration changes that could have caused the issue. More on this in the next section.

The specific ALB error codes and what they mean for your stack: 502 Bad Gateway: your Spring Boot app returned a malformed HTTP response, or closed the connection unexpectedly. Usually: app crashed mid-response, or Tomcat configuration issue. Check application logs for FATAL errors. 503 Service Unavailable: all targets are unhealthy. Check target group health. Usually: Spring Boot failing to start (dependency injection failure, database unreachable at startup with initialization-fail-timeout not set to -1). 504 Gateway Timeout: app didn't respond within the ALB's idle timeout (default 60 seconds). Your Spring Boot endpoint is taking >60 seconds. Usually: very slow database query, external API call without timeout, or deadlock. Check application logs and Performance Insights for long-running queries.
The 10 most common production failures and their root causes
Patterns from real production incidents — recognise the symptom, apply the fix
1. "Connection refused" on ALB health check
Root cause: Spring Boot not listening on port 8080. Either startup failed (check cloud-init-output.log and journalctl for exception) or configured on wrong port. Fix: check service status, check app logs for startup exception.
2. HikariCP "Connection is not available, request timed out"
Root cause: Connection pool exhausted. Either too many concurrent requests, long-running transactions, or @Transactional spanning external API calls. Fix: check hikaricp.connections.pending metric, reduce transaction scope, increase pool size temporarily.
3. "Access Denied" on S3 calls after deployment
Root cause: IAM policy was updated and the ARN changed (new bucket name, new environment). Check the IAM role's policy against the actual bucket ARN. Run aws accessanalyzer validate-policy on the updated policy to catch typos.
4. Spring Boot app starts then immediately crashes, loops
Root cause: Flyway migration failure (schema mismatch between code and DB), missing environment variable, or failed health check causing ASG to terminate and replace repeatedly. Check cloud-init-output.log and journalctl for the startup exception.
5. RDS Multi-AZ failover: app never reconnects
Root cause: JVM DNS cache set to -1 (infinite). Fix: ensure -Dnetworkaddress.cache.ttl=5 is in JVM flags. Verify HikariCP keepalive-time is set. Restart the application instances if the fix wasn't in place before the failover.
6. "Too many connections" error from MySQL
Root cause: Ghost connections from crashed app instances exhausted max_connections. The wait_timeout is too high (8 hours default) — old connections persist. Fix: set wait_timeout=300 in parameter group. Short-term: manually kill idle connections on the RDS instance.
7. ALB returning 460 to some clients
Root cause: JWT token + other headers exceeding 16KB ALB header size limit. Fix: reduce JWT payload (remove unnecessary claims), or pass token in request body for relevant endpoints. Check which clients are sending large tokens — usually after a JWT schema change.
8. New instances launched by ASG are never healthy
Root cause: Either the AMI has the wrong app version (stale AMI ID in Launch Template), the S3 bucket with the JAR is inaccessible, or the database schema migration fails on startup. Check: verify the Launch Template has the correct AMI ID, verify the instance role can access S3 and Secrets Manager, check user data logs on the failing instance.
9. Intermittent 5xx errors only affecting ~10% of requests
Root cause: One of the EC2 instances is unhealthy but not failing health checks (app is responding to health endpoint but not to actual requests — e.g., out of memory or database connection exhaustion on that specific instance). Check per-instance metrics in CloudWatch, check logs on each instance separately using the instance ID log stream filter.
10. Secrets Manager can't be reached at startup in new AZ
Root cause: VPC Interface Endpoint for Secrets Manager only exists in one subnet/AZ. When ASG launches an instance in the other AZ, the instance can't reach the Interface Endpoint. Fix: create Interface Endpoints in every private subnet/AZ that ASG instances can launch in.

Q1 Design the AWS infrastructure for a production Spring Boot REST API serving 10,000 requests per day. Walk me through every component and why you chose it. Hard▼

10,000 requests per day averages about 7 requests per second — low volume. But the design should be correct for production scale-up, not just optimal for current traffic.

Network layer: Custom VPC in ap-south-1 with public subnets (for ALB) and private subnets (for EC2 and RDS) in two AZs. S3 Gateway Endpoint and Secrets Manager Interface Endpoint in both private subnets — eliminating NAT Gateway charges for these services. NAT Gateway in only one AZ for OS updates (cost optimisation at low traffic — add the second AZ NAT when traffic justifies it).

Compute: Auto Scaling Group with t3.small instances, min=1, desired=2, max=6. Two instances for HA across AZs. Launch Template with IMDSv2 enforced, gp3 encrypted EBS, the application IAM role as instance profile. Target tracking scaling on ALBRequestCountPerTarget=800 — even at 7 req/s, the policy provides reactive scaling if traffic grows without manual intervention.

Load balancing: Application Load Balancer in public subnets with HTTPS listener (ACM certificate, HTTP-to-HTTPS redirect on port 80). Target group with 15-second health check interval, 3 healthy threshold, 60-second deregistration delay, Slow Start Mode 120s for JVM warmup. Health check path: /actuator/health/readiness.

Database: RDS MySQL 8.0 on db.t3.micro (cost-appropriate at this scale) with Multi-AZ enabled — data loss is unacceptable regardless of traffic volume. Custom parameter group with wait_timeout=300, slow_query_log enabled. gp3 storage with autoscaling to 100GB. Performance Insights enabled (free tier). Credentials in Secrets Manager — fetched at application startup via the instance IAM role.

Storage: S3 bucket for user uploads (job images, documents) with versioning enabled, server-side encryption (SSE-S3), and a bucket policy requiring VPC Endpoint access. Lifecycle policy: move objects > 90 days old to S3 Glacier Instant Retrieval.

Security: Security Group chaining throughout — internet → alb-sg (443), alb-sg → app-sg (8080), app-sg → db-sg (3306). No port 22 open — SSM Session Manager for administrative access. IAM policy on the instance role scoped to specific resource ARNs. Block Public Access enabled on S3. IMDSv2 required on every EC2.

Observability: CloudWatch Agent shipping Spring Boot logs to /getwork/app log group. Custom business metrics via Micrometer to CloudWatch namespace GetWork/API. Alarms on 5xx error rate, target group health, RDS connections, and disk space. CloudTrail enabled.

Why this answer works: It starts with the traffic context (10K/day = low), makes cost-appropriate choices (t3.micro, single NAT GW), but doesn't cut corners on correctness (Multi-AZ, IMDSv2, Secrets Manager). Every choice has a brief justification. The answer demonstrates that you understand production requirements across all five layers, not just one or two.
Q2 A user reports they submitted a form and can see a success message, but when they refresh the page the data is gone. What could cause this and how do you investigate? Hard▼

This is a data consistency or routing bug. There are four possible causes, in order of likelihood.

Most likely — Read Replica replication lag (if using replicas): The write went to the primary and succeeded. The success response was returned. The next page load is a read — if read traffic is routed to a Read Replica and the replica lags by even 2-3 seconds, the new record hasn't replicated yet. The user sees empty data. Investigation: check CloudWatch ReplicaLag metric on the replica. Check application logs — is the "view my submission" endpoint using @Transactional(readOnly=true)? If yes, it's routing to the replica. Fix: for the "view immediately after submit" pattern, always read from the primary within the same user session for a configurable "freshness window" after a write.

Second most likely — Transaction rollback without error surfaced to user: The write appeared to succeed (no exception thrown to the controller) but was rolled back by the database or rolled back during a subsequent operation in the same transaction. Investigation: check application logs for any rolled-back transaction markers. Check RDS slow query log for long transactions that might have been killed by wait_timeout. Enable Spring's transaction event logging temporarily.

Third — Multiple instances with in-memory state discrepancy: If the application has in-memory caching (an @Cacheable annotation using a local Caffeine cache rather than a distributed cache like Redis), the write invalidates the cache on Instance A, but the subsequent read goes to Instance B which still has the old cached data. Investigation: check if @Cacheable is used on the affected entity's read methods. Fix: use a distributed cache (ElastiCache Redis) instead of local cache, or disable caching for user-specific data that must be immediately consistent.

Fourth — Application bug where the write is conditional and silently skipped: The form data triggers a conditional write that fails its precondition, returns 200 anyway (bad error handling), and the data is never written. Investigation: add database write audit logging for the affected entity — log every write attempt with the data. Check the write path for conditional logic that might silently skip the write.

Why this answer works: The answer doesn't jump to "it's a bug" or "it's a database issue." It systematically walks through four concrete causes with specific investigation steps for each. The Read Replica lag explanation shows architectural awareness of the read/write routing pattern. The in-memory cache scenario shows knowledge of common Spring Boot caching pitfalls.
Q3 How do you handle secrets in a Spring Boot application deployed on EC2? What's wrong with environment variables for secret injection? Medium▼

The correct pattern for secrets on EC2 is AWS Secrets Manager fetched at runtime using the instance's IAM role. The application fetches the secret via the SDK (DefaultCredentialsProvider uses IMDS to get temporary role credentials), caches the value in memory, and uses it for the connection pool or HMAC signing. No secrets in code, no secrets in config files committed to version control, no secrets passed via any external mechanism.

Environment variables for secrets have three specific problems. First, they're visible in process listings: cat /proc/PID/environ on Linux shows every environment variable of a running process to any user with sufficient permissions. On a shared host (unlikely on EC2 but possible) or via a container escape, environment variables leak. Second, they're visible in the AWS console to anyone with ec2:DescribeInstances permission — the Launch Template user data and the ECS task definition show environment variables. Third, rotating a secret requires updating the environment variable configuration and restarting the application — there's no rotation mechanism built in. With Secrets Manager, rotation can be automated and picked up at the next application startup without any code changes.

The one legitimate use of environment variables for configuration: non-sensitive runtime configuration that changes between environments — like the AWS region, the application profile, the log level, or the ALB DNS name. These aren't secrets. They're configuration. The 12-factor app methodology is correct that configuration belongs in environment variables. The mistake is treating secrets as just another form of configuration and putting them there too.

Why this answer works: The three specific problems with environment variables (process listing, console visibility, rotation difficulty) are concrete rather than vague "security concerns." The distinction between configuration (environment variables are fine) and secrets (Secrets Manager) shows nuanced understanding of the 12-factor methodology, not a blanket rejection of environment variables.

Create the EC2 instance IAM role with least-privilege permissions
Create the IAM role that EC2 instances will use to access S3, Secrets Manager, SSM, and CloudWatch. This role is attached to the Launch Template as an instance profile.
ACCOUNT_ID=$(aws sts get-caller-identity --query Account --output text) # Create the IAM role with EC2 trust policy aws iam create-role \ --role-name getwork-app-role \ --assume-role-policy-document '{ "Version": "2012-10-17", "Statement": [{ "Effect": "Allow", "Principal": {"Service": "ec2.amazonaws.com"}, "Action": "sts:AssumeRole" }] }' --region $REGION 2>/dev/null || echo "Role already exists" # Create the application permissions policy cat > /tmp/app-policy.json << POLICY { "Version": "2012-10-17", "Statement": [ {"Effect":"Allow","Action":["s3:GetObject","s3:PutObject","s3:DeleteObject"], "Resource":"arn:aws:s3:::getwork-app-data/*"}, {"Effect":"Allow","Action":"s3:ListBucket", "Resource":"arn:aws:s3:::getwork-app-data"}, {"Effect":"Allow","Action":"secretsmanager:GetSecretValue", "Resource":"arn:aws:secretsmanager:ap-south-1:${ACCOUNT_ID}:secret:/getwork/prod/*"}, {"Effect":"Allow","Action":["ssm:GetParameter","ssm:GetParameters","ssm:GetParametersByPath"], "Resource":"arn:aws:ssm:ap-south-1:${ACCOUNT_ID}:parameter/getwork/prod/*"}, {"Effect":"Allow","Action":["logs:CreateLogGroup","logs:CreateLogStream","logs:PutLogEvents"], "Resource":"arn:aws:logs:ap-south-1:${ACCOUNT_ID}:log-group:/getwork/*:*"}, {"Effect":"Allow","Action":["ssmmessages:CreateControlChannel","ssmmessages:CreateDataChannel", "ssmmessages:OpenControlChannel","ssmmessages:OpenDataChannel", "ec2messages:AcknowledgeMessage","ec2messages:SendReply","ec2messages:GetMessages"], "Resource":"*"} ] } POLICY # Create and attach the policy aws iam put-role-policy \ --role-name getwork-app-role \ --policy-name getwork-app-policy \ --policy-document file:///tmp/app-policy.json # Create instance profile (required to attach role to EC2) aws iam create-instance-profile \ --instance-profile-name getwork-app-profile 2>/dev/null || echo "Profile exists" aws iam add-role-to-instance-profile \ --instance-profile-name getwork-app-profile \ --role-name getwork-app-role 2>/dev/null || echo "Role already in profile" echo "IAM role and instance profile created"

Create production-grade user data and update the Launch Template
Write the user data script that: fetches app version from SSM, downloads the JAR from S3, writes the systemd service with correct Spring profile and JVM flags, and verifies health before completing.

Create ALB, Target Group, and ASG — the complete load-balanced deployment
Create the ALB and ASG using your Day 8 commands as a template. The difference from Day 8: the instances now have the correct IAM role attached and the full user data that starts the application service.
VPC_ID=$(aws ec2 describe-vpcs \ --filters "Name=tag:Name,Values=getwork-vpc" \ --query 'Vpcs[0].VpcId' --output text --region $REGION) PUB_SUBNETS=$(aws ec2 describe-subnets \ --filters "Name=vpc-id,Values=$VPC_ID" "Name=map-public-ip-on-launch,Values=true" \ --query 'Subnets[*].SubnetId' --output text --region $REGION) PUB_1A=$(echo $PUB_SUBNETS | tr ' ' '\n' | head -1) PUB_1B=$(echo $PUB_SUBNETS | tr ' ' '\n' | tail -1) PRIV_SUBNETS=$(aws ec2 describe-subnets \ --filters "Name=vpc-id,Values=$VPC_ID" "Name=map-public-ip-on-launch,Values=false" \ --query 'Subnets[*].SubnetId' --output text --region $REGION) PRIV_1A=$(echo $PRIV_SUBNETS | tr ' ' '\n' | head -1) PRIV_1B=$(echo $PRIV_SUBNETS | tr ' ' '\n' | tail -1) ALB_SG=$(aws ec2 describe-security-groups \ --filters "Name=group-name,Values=alb-sg" \ --query 'SecurityGroups[0].GroupId' --output text --region $REGION) # Create ALB ALB_ARN=$(aws elbv2 create-load-balancer \ --name getwork-alb \ --subnets $PUB_1A $PUB_1B \ --security-groups $ALB_SG \ --scheme internet-facing --type application \ --query 'LoadBalancers[0].LoadBalancerArn' \ --output text --region $REGION 2>/dev/null || \ aws elbv2 describe-load-balancers --names getwork-alb \ --query 'LoadBalancers[0].LoadBalancerArn' --output text --region $REGION) echo "ALB: $ALB_ARN" ALB_DNS=$(aws elbv2 describe-load-balancers \ --load-balancer-arns $ALB_ARN \ --query 'LoadBalancers[0].DNSName' --output text --region $REGION) echo "ALB DNS: $ALB_DNS" # Day 10 — End-to-end deployment integration
Phase 2 Application Deployment complete. Days 11–13 begin: Route 53, CloudFront, ECS, ElastiCache, DynamoDB.
Configuration hierarchy
Secrets Manager for credentials. Parameter Store for config. application.properties for defaults. Environment variables for profile activation. Never credentials in properties or env vars.
Graceful shutdown timing
ALB deregistration delay ≥ Spring lifecycle.timeout-per-shutdown-phase. Otherwise SIGTERM arrives while ALB is still sending requests. Correct order: drain → SIGTERM → shutdown.
OSIV disabled
spring.jpa.open-in-view=false. Prevents holding DB connections during response serialisation. Forces eager loading of all needed data in the transaction — eliminates lazy-load connection pool exhaustion.
SSM Session Manager
Access EC2 instances without port 22 or SSH key. Requires ssmmessages permissions in IAM role + SSM agent on instance. All commands logged to CloudTrail. Required for private subnet instances.
VPC-bound S3 bucket
Bucket policy with Deny unless aws:SourceVpce = your endpoint ID. Stolen IAM credentials cannot access the bucket from outside the VPC. Most important S3 security control for internal data.
IAM policy validation
aws accessanalyzer validate-policy before every policy deployment. Catches typos, over-broad wildcards, wrong resource ARN formats. Run in CI/CD pipeline as mandatory pre-deploy gate.
ALB error codes map
502 = malformed response from backend. 503 = no healthy targets. 504 = backend timeout. 460 = header size limit exceeded (JWT too large). Each maps to a specific layer and fix.
Business metrics vs infra metrics
Infrastructure green ≠ application healthy. Instrument critical user paths (job creation, payment) with custom CloudWatch metrics. Alarm on business outcome failure, not just infrastructure thresholds.
Read-after-write consistency
User submits → reads from replica → stale data. Fix: route reads that follow a write in the same session to the primary. AbstractRoutingDataSource detects readOnly=true transactions automatically.
Log retention policies
Set on every CloudWatch Log Group. Default: infinite (billing forever). Production: 30 days app logs, 90 days errors, 1 year audit logs. Terraform module to enforce across all groups.
Initialization fail timeout
HikariCP initialization-fail-timeout=-1 prevents startup failure when DB is briefly unreachable. Without it: Secrets Manager hiccup → app refuses to start → ASG crash loop → zero capacity.
CloudTrail for incidents
When application logs show no cause: CloudTrail shows the IAM, SG, or RDS change that happened before the incident. Always the first place to look for "it was fine, then it stopped."
