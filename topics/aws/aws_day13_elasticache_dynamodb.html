ElastiCache for Redis — what it actually is
ElastiCache is a managed in-memory data store. Redis is the engine. Understanding how Redis stores data in RAM, why it's orders of magnitude faster than RDS, and what happens on node failure is the foundation for using it correctly.
in-memory microsecond latency caching layer

Redis stores all data in RAM. A GET or SET operation requires no disk I/O — the data is already in memory. This is why Redis delivers sub-millisecond latency (typically 100–500µs) compared to RDS MySQL's 1–10ms for a simple lookup. The difference is not incremental — it is structural.

ElastiCache is AWS's managed Redis service. It runs Redis on EC2 instances inside your VPC, handles patching, automatic failover, CloudWatch metrics, and backup. You cannot SSH into ElastiCache nodes. You connect via the Redis protocol on port 6379 from your Spring Boot application.

How Redis handles a GET command internally When your Spring Boot app sends GET job:123: Redis computes a 64-bit hash of the key, uses the lower bits to locate the hash table slot, traverses the slot's linked list to find the key, and returns the value. The entire operation: one memory read, one hash computation, one pointer dereference. On modern hardware this takes ~200 nanoseconds. No network disk round-trip. No query parser. No query planner. No row deserialisation. This is what makes Redis fundamentally different from a relational database — it is not a simpler database, it is a completely different data access model.

ElastiCache runs Redis inside your VPC's private subnets. Your EC2 or Fargate tasks connect to the ElastiCache endpoint on port 6379. The Security Group on the ElastiCache cluster must allow inbound 6379 from your application's Security Group — the same chaining pattern you built on Day 4.

ElastiCache supports Redis 7.x — the current stable version. Always use the most recent supported version. Redis versions below 6 lack ACL authentication (Role-Based Access Control). Redis 7 added multi-part transactions and improved cluster mode stability.

The ElastiCache node that isn't in your VPC The most common first-time ElastiCache setup failure: the cache is created in the wrong subnet group (or no subnet group), and your application can't reach it. ElastiCache requires a subnet group — a collection of subnets, like an RDS subnet group. The nodes are placed in those subnets. If you create the cluster without specifying a subnet group, it goes into the default VPC. Your Spring Boot app in the custom VPC from Day 4 cannot reach it. Always create a dedicated ElastiCache subnet group using your private subnets before creating the cluster.

Redis is not a simple key-value store. It supports multiple data structure types, each with different operations and use cases. Choosing the right type for each use case determines whether Redis helps or becomes a bottleneck.

String
The fundamental type. Can hold text, JSON, integers, or binary data (up to 512MB).

GetWork use: Cache a serialised job listing as JSON. SET job:123 {json} EX 300 — expires in 300 seconds.

Operations: GET, SET, INCR (atomic counter), GETSET (swap and return old), SETNX (set if not exists — distributed lock foundation).
Hash
A map of field → value pairs, all under one key. Like a row in a database table.

GetWork use: Cache worker profile fields. HSET worker:42 name "Ravi" city "Mumbai" rating "4.8". Read one field with HGET, all fields with HGETALL.

Advantage over String: Update individual fields without serialising/deserialising the entire object.
List
Ordered list, supports O(1) push/pop from both ends. Works as a queue or stack.

GetWork use: Recent job activity feed. LPUSH activity:worker:42 "Applied to job 789". Read last 20 with LRANGE activity:worker:42 0 19.

Caution: LRANGE on a large list is O(N). Cap list size with LTRIM after each LPUSH.
Sorted Set
Members with a float score, sorted by score. O(log N) for inserts/reads by rank.

GetWork use: Top-rated workers in a city. Score = rating. ZADD workers:mumbai 4.8 "worker:42". Read top 10: ZREVRANGE workers:mumbai 0 9 WITHSCORES.

Also used for: Rate limiting (score = timestamp), leaderboards, scheduled task queues.
Set
Unordered collection of unique strings. O(1) add/remove/check membership.

GetWork use: Track which workers have viewed a job. SADD job:123:views worker:42. Check: SISMEMBER job:123:views worker:42. Count: SCARD job:123:views.

Set operations: SUNION (union), SINTER (intersection), SDIFF (difference).
Stream
Append-only log with consumer groups. Similar to Kafka topics but inside Redis.

GetWork use: Real-time notification queue. XADD notifications * type job_match workerId 42.

For GetWork's Kafka context: Streams are Redis-native alternative to Kafka for low-volume event sourcing. Your Deloitte Kafka experience directly applies.

Redis is primarily an in-memory store. If the Redis node restarts, all data is lost — unless persistence is configured. ElastiCache supports two Redis persistence mechanisms.

RDB (Redis Database Backup): Point-in-time snapshots of the entire dataset at configured intervals. Fast to restore (load one file). Data between the last snapshot and the crash is lost. Good for cache use cases where some data loss is acceptable.

AOF (Append-Only File): Every write command is appended to a log file. On restart, Redis replays the log. More durable (near-zero data loss) but slower and produces larger files. For a pure cache (data also lives in RDS), AOF is usually unnecessary overhead.

When you need persistence vs when you don't For GetWork's use case — caching job listings that live primarily in RDS — persistence is optional. If Redis crashes, the cache is empty; the next request to the Spring Boot app misses the cache and reads from RDS. Performance degrades temporarily but no data is lost. The cache repopulates over the next few minutes as requests come in. This is the correct design for a read-aside cache. Only enable AOF persistence for Redis use cases where Redis is the primary store (session state, real-time leaderboards, counters) — not for caches backed by a database.
The AOF file that grows unboundedly AOF records every write command. After months of operation on a write-heavy cluster, the AOF file can be hundreds of GB. Redis rewriting (AOF compaction) runs periodically but can cause a brief performance spike during rewrite. In ElastiCache, you don't manage the AOF file directly — but enabling AOF persistence on a large, write-heavy cluster does increase storage requirements and costs. Benchmark with and without AOF before enabling it on any cache that has significant write volume.

Redis evicts keys from memory when the memory limit (maxmemory) is reached. ElastiCache sets maxmemory to the instance's available RAM. Choosing the right eviction policy determines whether Redis evicts intelligently or causes cache stampedes.

allkeys-lru (recommended for cache)
Evicts the Least Recently Used key from all keys when memory is full. The most appropriate policy for a general-purpose cache.

Your Spring Boot cache: recently accessed job listings stay warm; old unseen listings are evicted first. The cache naturally contains the working set your users actually access.
volatile-lru
Evicts LRU only from keys that have a TTL set. Keys without TTL are never evicted.

Use when you have a mix: some data is permanent (session data, critical counters) and some is cached with TTL. The permanent data survives memory pressure; the TTL-bearing cache data is evicted.
noeviction
Returns an error when memory is full. Writes fail. Application must handle the error.

Never use for a cache. Appropriate only if Redis is a primary store where you must not silently lose data — but even then, monitor memory aggressively to avoid hitting the limit.
allkeys-lfu (Redis 4+)
Evicts Least Frequently Used — keys accessed the fewest times overall, not just recently.

Better than LRU for access patterns where old but frequently referenced data (e.g. a popular static job category) should not be evicted just because it was last accessed 10 minutes ago.
Monitor CacheEvictions in CloudWatch A rising CacheEvictions metric means your Redis instance is running out of memory and evicting valid cached data. This causes cache miss rate to increase, which increases load on RDS. First response: increase the ElastiCache node size. Second response: review your cache TTLs — are you caching objects that could expire sooner? Third response: review what you're caching — are large, rarely-accessed objects consuming disproportionate memory?
Cluster mode vs Replication Group
The two ElastiCache Redis deployment modes have fundamentally different scaling models. Choosing wrong means either unnecessary cost or a configuration that can't scale when you need it to.
Replication Group (non-cluster mode)
Structure: One primary node + up to 5 read replicas. All nodes hold the entire dataset.

Scaling: Vertical only (upgrade node size). Read scale via replicas (route reads to replica endpoint).

Failover: Automatic — replica is promoted to primary within 30–60 seconds. DNS updates.

Use when: Dataset fits in one node's memory. You want simple primary/replica HA. Most Spring Boot applications start here.

Cost: Pay for each node. 1 primary + 2 replicas = 3 nodes.
Cluster Mode Enabled
Structure: Data sharded across 1–500 shards. Each shard has 1 primary + up to 5 replicas.

Scaling: Horizontal — add shards to increase total memory and throughput. Each shard holds a partition of the keyspace.

Sharding: Redis uses consistent hashing with 16,384 hash slots distributed across shards. Key routing: CLUSTER KEYSLOT key.

Use when: Dataset exceeds single node memory, or you need horizontal write scaling.

Limitation: Multi-key commands (MGET, MSET) must hash to the same slot — use hash tags: {user:42}:profile forces same slot.
Failover mechanics — what happens to your Spring Boot connections When the primary node fails, ElastiCache promotes a replica to primary within 30–60 seconds. The endpoint DNS record is updated. Your Spring Boot application (using Lettuce, the default Redis client in Spring Data Redis) handles this through topology refresh — Lettuce continuously refreshes the cluster topology from the Redis nodes themselves. On failover, the next topology refresh (every 60 seconds by default) returns the new primary. Configure spring.redis.lettuce.cluster.refresh.period=30s to refresh every 30 seconds. Without this, Spring Boot may continue trying to write to the old primary (now a replica that rejects writes) for up to 60 seconds after failover.
The recommended setup for GetWork Start with a replication group: one cache.t4g.small primary node ($0.034/hr in ap-south-1) + one replica in a different AZ for HA. Total: ~$49/month for both nodes. Dataset fits easily — job listings for Hyderabad and Mumbai construction workers will be well under the 1.37GB node memory. Enable cluster mode only when the working set exceeds ~1GB or you need horizontal write throughput that exceeds a single node's ~100,000 operations/second. For most applications that start with a cache, a two-node replication group is the correct starting point.
The "MOVED" error in cluster mode that breaks naïve clients In cluster mode, if you try to access a key that belongs to a shard not connected to your current client connection, Redis returns MOVED 1234 10.0.1.25:6379 — telling you which node has slot 1234. Non-cluster-aware clients treat MOVED as an error. Lettuce and Jedis are cluster-aware — they automatically redirect to the correct shard. But if you use a raw Redis client or a library without cluster support, you'll see MOVED errors that look like access failures. Always verify your Redis client library explicitly supports cluster mode before deploying to a cluster-mode ElastiCache.

Primary endpoint vs Reader endpoint ElastiCache replication groups expose two endpoints. The primary endpoint always routes to the current primary node — use this for writes and for the general Spring Boot Redis connection. The reader endpoint load-balances across all replicas — only use this for read-only operations (explicitly, via a secondary Redis template). In most Spring Boot applications, using only the primary endpoint is correct — the replica only becomes active on failover. The reader endpoint is an optimisation for read-heavy scenarios where replica read capacity is explicitly needed.
Java — Spring Cache abstraction
@Configuration
@EnableCaching
public class CacheConfig {

    @Bean
    public RedisCacheConfiguration cacheConfiguration() {
        return RedisCacheConfiguration.defaultCacheConfig()
            .entryTtl(Duration.ofMinutes(10))          // Default TTL
            .serializeValuesWith(RedisSerializationContext
                .SerializationPair.fromSerializer(
                    new GenericJackson2JsonRedisSerializer())) // JSON not Java serialisation
            .disableCachingNullValues();                // Don't cache null results
    }

    @Bean
    public RedisCacheManagerBuilderCustomizer cacheManagerCustomizer() {
        return builder -> builder
            .withCacheConfiguration("jobs",
                RedisCacheConfiguration.defaultCacheConfig()
                    .entryTtl(Duration.ofMinutes(5)))     // Jobs expire in 5 min
            .withCacheConfiguration("workers",
                RedisCacheConfiguration.defaultCacheConfig()
                    .entryTtl(Duration.ofMinutes(30)))    // Worker profiles: 30 min
            .withCacheConfiguration("categories",
                RedisCacheConfiguration.defaultCacheConfig()
                    .entryTtl(Duration.ofHours(24)));     // Static categories: 1 day
    }
}

// Service layer — annotations handle all cache logic transparently
@Service
public class JobService {

    @Cacheable(value = "jobs", key = "#id")             // Cache on GET
    public JobResponse getJob(Long id) {
        return jobRepository.findById(id)                 // Only called on cache miss
            .map(JobResponse::from)
            .orElseThrow(() -> new NotFoundException());
    }

    @CachePut(value = "jobs", key = "#result.id")      // Update cache on write
    public JobResponse updateJob(UpdateJobRequest req) {
        Job job = jobRepository.save(new Job(req));
        return JobResponse.from(job);                     // Also updates cache
    }

    @CacheEvict(value = "jobs", key = "#id")            // Remove from cache on delete
    public void deleteJob(Long id) {
        jobRepository.deleteById(id);
    }

    @Cacheable(
        value = "jobs",
        key = "'search:' + #city + ':' + #category",   // Composite key
        unless = "#result.isEmpty()"                     // Don't cache empty results
    )
    public List<JobResponse> searchJobs(String city, String category) {
        return jobRepository.findByCityAndCategory(city, category)
            .stream().map(JobResponse::from).toList();
    }
}
Java serialisation as the cache value serialiser — a silent time bomb Spring Cache defaults to Java serialisation for Redis values. This works until you deploy a new version of your application that changes a serialised class (adds a field, renames a field). Old cached values with old class structure cannot be deserialised by the new code — every cache hit throws a deserialisation exception, causing cache misses and falling back to the database. Worse: the stale values persist until their TTL expires. Fix: always use GenericJackson2JsonRedisSerializer as shown above. JSON is forward/backward compatible — extra fields are ignored by Jackson. Add @JsonIgnoreProperties(ignoreUnknown = true) to your response classes.

Lettuce (Spring Boot's default Redis client) uses a single shared connection by default — all threads in your Spring Boot application share one Redis connection. Under high concurrency, this serialises all Redis operations through one TCP connection, causing head-of-line blocking. Enable connection pooling explicitly.

Sizing the Lettuce connection pool Pool size formula: consider your Tomcat thread pool (default 200 threads) and how many can be simultaneously executing Redis operations. For a typical API: at most 20–30% of threads are waiting on Redis simultaneously. Pool size = 16 is usually sufficient for most Spring Boot APIs. For WebFlux (non-blocking): Lettuce's async model allows one connection to handle many concurrent operations. Pool of 4–8 is typically enough for WebFlux + Redis. Add the Apache commons-pool2 dependency to enable Lettuce pooling — it doesn't activate without the dependency on the classpath.

Cache patterns — read-aside, write-through, and TTL strategy
The pattern you choose determines data freshness, consistency guarantees, and failure behaviour. Most engineers know read-aside. Understanding when to use write-through and how to design TTLs prevents the subtle consistency bugs that cache misuse introduces.
Read-aside (lazy loading)
Flow: App reads cache → miss → app reads DB → app writes to cache → return.

Good: Only caches data that's actually requested. Simple. Cache can be cleared without affecting correctness.

Bad: First request after TTL expires hits DB (cache miss spike on TTL expiry). Stale data window = TTL duration.

Use for: Job listings, worker profiles, search results. Most Spring Boot caches.
Write-through
Flow: App writes to cache AND DB in the same operation. Cache always current.

Good: No stale data. Read always hits cache (after first write). No thundering herd.

Bad: Every write has cache latency overhead. Cache may contain data that's never read (waste). Failed cache write must be handled.

Use for: User session data, real-time counters, data that's both read and written frequently.
Write-behind (write-back)
Flow: Write to cache immediately, write to DB asynchronously later.

Good: Lowest write latency. Database absorbs batched writes.

Bad: Cache failure before async DB write = data loss. Complex to implement correctly.

Use for: High-frequency counters (view counts), analytics aggregates. Not for financial transactions.
The thundering herd on TTL expiry Your most popular search query — "construction workers in Mumbai" — has its cache entry expire at 12:00:00. At 12:00:01, your ALB is handling 500 concurrent requests. All 500 miss the cache simultaneously, all query RDS, and all try to write the result back to Redis at the same time. RDS handles 500 simultaneous identical queries. This is a thundering herd. Solutions: (1) Cache stampede protection — use a distributed lock (Redis SETNX) so only one request rebuilds the cache entry while others wait. (2) Probabilistic early expiration — before the TTL expires, a fraction of requests proactively refresh the cache in the background, never letting it go cold. (3) Jittered TTLs — instead of TTL=300, use TTL=300 + random(0,60). This spreads expiry across a 60-second window, preventing synchronized stampedes across multiple cache keys.
TTL design for GetWork Job listings that rarely change: 5 minutes TTL. Popular search results (city + category): 2 minutes + 0–30s jitter. Worker profiles: 30 minutes (profiles change infrequently). Static data (job categories, city list): 24 hours. User session tokens: match session timeout (e.g. 2 hours). Never set TTL=0 (no expiry) for cached DB results — orphaned cache entries after DB deletion become stale data that never clears.
DynamoDB — not just a key-value store
DynamoDB is AWS's fully managed distributed NoSQL database. It is not a replacement for RDS. It is a fundamentally different data model optimised for single-digit millisecond performance at any scale, horizontal partitioning, and predictable performance without capacity planning.
NoSQL single-digit ms at any scale serverless

DynamoDB stores data in partitions — horizontal slices of your table distributed across thousands of SSDs in multiple AZs. Each partition holds up to 10GB of data and handles up to 3,000 read units and 1,000 write units per second. DynamoDB automatically splits partitions when they grow beyond these limits — you never configure servers, shards, or cluster topology.

The data model: every item (row) in a DynamoDB table must have a primary key. The primary key is either a partition key alone, or a composite of partition key + sort key. DynamoDB uses the partition key to route the item to a specific storage partition. Items with the same partition key but different sort keys are stored together — sorted by sort key — enabling range queries within a partition.

How DynamoDB routes a read — the path from your query to the data Your Spring Boot app calls dynamoDb.getItem(tableRequest). DynamoDB computes a consistent hash of the partition key value and maps it to a partition. The partition is replicated across three AZs. A strongly consistent read goes to the partition leader; an eventually consistent read goes to any replica. The entire operation — hash computation, routing, storage read, network return — completes in 1–5ms regardless of whether the table has 100 items or 100 billion items. This is not possible with a relational database, which requires query planning, index scans, and join execution that scale with data volume.

What DynamoDB is excellent at: Session state storage (get/put by session ID, automatic TTL-based expiry), rate limiting and counters (atomic INCR, conditional writes), product catalogues with millions of items (get by ID, near-instant), time-series data with partition by entity and sort by timestamp, IoT ingestion pipelines, and any workload with a known access pattern that fits within a partition key + sort key model.

What DynamoDB struggles with: Ad-hoc queries across multiple attributes (DynamoDB's query capabilities are limited to the primary key and GSIs — no WHERE clause on arbitrary fields), complex joins, analytical workloads requiring full table scans, and schemas that change frequently and unpredictably.

DynamoDB is not a drop-in for RDS A common architectural mistake: "our system is getting slow, let's switch to DynamoDB." DynamoDB's performance advantage is only realised when your access patterns are designed around its key model. Migrating a relational application's data model directly to DynamoDB and querying it with scan operations produces worse performance than the original RDS at higher cost. The decision to use DynamoDB must come at the beginning of designing a feature's data model — not as a performance escape hatch after a relational model is already built.

DynamoDB has two capacity modes that determine how you pay for read and write throughput.

On-Demand capacity (recommended to start)
How it works: DynamoDB automatically scales to handle any traffic level. No capacity planning. You pay per request: $1.25 per million writes, $0.25 per million reads (ap-south-1).

When to use: New tables where traffic is unpredictable. Workloads with large traffic spikes. Tables accessed infrequently. GetWork's early stage.

Cost at scale: At 10M writes/day = $12.50/day. At this scale, provisioned may be cheaper.

Switching: Can switch to provisioned once you understand your traffic pattern.
Provisioned capacity
How it works: You set read capacity units (RCUs) and write capacity units (WCUs). Pay per RCU/WCU per hour — regardless of actual utilisation. 1 RCU = 1 strongly consistent 4KB read/s. 1 WCU = 1 write/s up to 1KB.

Auto Scaling: Enable DynamoDB Auto Scaling to adjust provisioned capacity in response to traffic. Set min and max RCU/WCU. 70% utilisation target is typical.

Cost at scale: Cheaper than on-demand for predictable, sustained traffic. 100 WCU = $0.065/hr = ~$47/month.

Throttling: Requests above provisioned capacity are throttled (not queued) — your application gets a ProvisionedThroughputExceededException.
Hot partition throttling with provisioned capacity You provision 1,000 WCUs across your table. DynamoDB distributes this across partitions. If your workload writes disproportionately to one partition key (all writes going to partition key "Mumbai" while other city partitions are idle), that one partition gets throttled even though your overall WCU usage is below provisioned. The fix is key design — not provisioning more capacity. Partition key must be high-cardinality and evenly accessed. Common anti-pattern: using a date as a partition key, causing all today's writes to flood a single partition while yesterday's partition is idle.

DynamoDB offers two consistency models for reads — with significant performance and cost implications.

Eventually consistent reads — the default. DynamoDB reads from any of the three replicas. The replica may be slightly behind the leader (usually milliseconds, occasionally a few seconds). Half the cost of strongly consistent reads. Appropriate for: read-after-write scenarios where the reader is a different request than the writer, analytics, cached lookups.

Strongly consistent reads — DynamoDB reads from the partition leader, guaranteeing the latest data. Twice the cost of eventually consistent. Appropriate for: read-after-write in the same request chain (user creates a session, immediately reads it back), financial record lookups, any scenario where stale data causes a user-visible error.

When eventual consistency bites you in practice A worker submits their profile update at 14:00:00. The write succeeds. At 14:00:00.100 (100ms later), the profile page does a GET to display the confirmation. With eventual consistency, the GET may return the old data because the replica hasn't caught up yet. The worker sees "Your name is still the old name" and thinks their update failed. They submit again. You now have a duplicate update event to handle. For any read that directly follows a write in the same user interaction, use strongly consistent reads or use write-through caching so the latest data is in Redis immediately after the write.
DynamoDB key design — the skill that determines everything
DynamoDB's performance, cost, and query capability are entirely determined by your primary key design. A well-designed key model serves all your access patterns efficiently. A poorly designed one requires either full table scans (expensive, slow) or excessive GSIs (complex, costly).

The cardinal rule of DynamoDB design: identify all access patterns before designing the schema. Unlike relational databases where you design the schema and then write queries, in DynamoDB the access patterns drive the schema — not the other way around.

GetWork access patterns to identify first:

1. Get a specific job by ID (job detail page)
2. List all jobs in a city sorted by post date (browse jobs)
3. Get a worker's profile by ID
4. List all jobs a specific worker applied to
5. Get a session by session token (authentication)
6. Check rate limit counter for an IP address

Single-table design — fitting multiple entity types into one table Advanced DynamoDB practitioners use a single table for multiple entity types. The partition key encodes the entity type: JOB#123, WORKER#42, APPLICATION#789. Sort key encodes the relationship: item PK=JOB#123 SK=METADATA is the job itself; item PK=JOB#123 SK=APPLICATION#WORKER#42 is an application for that job by that worker. A single query to PK=JOB#123 returns the job AND all its applications. This is the most space-efficient design but requires deep DynamoDB expertise. For engineers new to DynamoDB, start with one table per entity type and consolidate later.
DynamoDB — GetWork session table design
// Table: getwork-sessions
// Access pattern: Get session by token (authentication on every request)
// TTL: sessions expire automatically via DynamoDB TTL attribute

// Key design:
// PK: sessionToken (String) — high cardinality, even distribution
// SK: none — single item per partition key

// Sample item:
{
  "sessionToken": "eyJhbGciOiJIUzI1NiJ9...",  // PK
  "workerId": 42,
  "role": "WORKER",
  "createdAt": "2024-03-15T10:30:00Z",
  "expiresAt": 1710513000          // Unix epoch — DynamoDB TTL attribute
                                          // DynamoDB auto-deletes expired items
}

// Table: getwork-rate-limits
// Access pattern: Check and increment request count per IP per minute
// PK: ipAddress#minuteWindow  e.g. "192.168.1.1#202403151430"
// SK: none
// TTL: minuteWindow + 2 minutes (auto-expire rate limit windows)

// Atomic increment with conditional expression:
UpdateItemRequest.builder()
  .tableName("getwork-rate-limits")
  .key(Map.of("pk", AttributeValue.fromS(ip + "#" + window)))
  .updateExpression("ADD requestCount :inc SET expiresAt = :ttl")
  .conditionExpression("requestCount < :limit OR attribute_not_exists(requestCount)")
  .expressionAttributeValues(Map.of(
    ":inc", AttributeValue.fromN("1"),
    ":limit", AttributeValue.fromN("100"),
    ":ttl", AttributeValue.fromN(String.valueOf(expiry))
  ))
  .build();
// If condition fails → ConditionalCheckFailedException → rate limit exceeded
// This entire operation is atomic — no race condition across concurrent requests
DynamoDB TTL — free automatic item expiry DynamoDB TTL is a built-in feature that automatically deletes items when a specified Unix epoch timestamp attribute is in the past. No Lambda, no cron, no DELETE queries. Deletions happen within 48 hours of expiry (usually within minutes for normal table sizes). Items are not immediately deleted at expiry — they remain readable for up to 48 hours. For session tokens where security requires hard expiry, enforce TTL in your application logic (check expiresAt > now()) in addition to relying on DynamoDB's TTL cleanup. Enable TTL with: aws dynamodb update-time-to-live --table-name T --time-to-live-specification Enabled=true,AttributeName=expiresAt
Reads, writes, and Global Secondary Indexes
The DynamoDB operations that matter for a Spring Boot application — GetItem, Query, PutItem, UpdateItem with conditions — and GSIs for access patterns that don't fit the primary key.

GetItem — retrieve a single item by exact primary key. O(1). The fastest operation. Always prefer GetItem over Query when you know the full primary key.

Query — retrieve multiple items with the same partition key, optionally filtering by sort key. Returns items sorted by sort key. Supports filter expressions on non-key attributes (but note: filters are applied after reading — they reduce returned items but not RCUs consumed). Only use filter expressions to reduce network transfer, not to reduce read cost.

Scan — read every item in the table. Expensive. Consumes RCUs proportional to table size. The operation that makes DynamoDB slow and expensive when used for general queries. Never use Scan in a production hot path. Acceptable for: administrative data exports, full-table analytics (run off-peak), seeding test data.

PutItem — create or replace an item. Use a condition expression (attribute_not_exists(pk)) to prevent overwriting existing items — essential for insert-only semantics.

UpdateItem — update specific attributes of an existing item. Does not require reading the item first. Supports atomic ADD (increment), SET, REMOVE, and DELETE operations. Supports conditional expressions for optimistic locking.

Global Secondary Indexes (GSIs) — query by non-key attributes A GSI creates an alternate view of your table with a different partition key and sort key. DynamoDB projects (copies) specified attributes from the base table into the GSI. The GSI is maintained automatically as items are written to the base table. Reads from a GSI behave like reads from the base table: GetItem and Query. GSI writes consume WCUs from the base table. GSI reads consume RCUs from the GSI's allocated capacity (for provisioned mode) or are charged per-request (on-demand).

GetWork example: base table workers has PK=workerId. GSI: PK=city, SK=rating. Query: "give me workers in Mumbai sorted by rating descending" — a query impossible on the base table but trivial on this GSI. Up to 20 GSIs per table. Each GSI costs additional storage (projected attributes) and write capacity.
The GSI that can't be queried because of uneven key distribution You create a GSI with PK=status (ACTIVE, INACTIVE, PENDING). Your table has 10 million workers, 9.8 million of which are ACTIVE. The "ACTIVE" partition in your GSI holds 9.8 million items. A Query on this GSI for status=ACTIVE returns a result set so large it requires many paginated calls and is functionally equivalent to a Scan. This is a hot partition problem in the GSI. GSI partition keys must have the same high-cardinality property as base table partition keys. Use composite GSI keys to increase cardinality: PK = city+status (e.g. "Mumbai#ACTIVE") instead of status alone.

        table.putItem(session);
    }

    public void delete(String token) {
        table.deleteItem(r -> r.key(k -> k.partitionValue(token)));
    }
}

Interview prep
ElastiCache and DynamoDB questions test architectural judgement — knowing when to use each service, what the failure modes are, and how they integrate with a Spring Boot application.
Q1 Your RDS database is CPU-bound. Should you add Redis caching or scale up the RDS instance? How do you decide? Hard ▼

This depends on why the RDS CPU is high. High CPU can come from two distinct sources and the fix differs for each.

If CPU is high due to repeated identical reads: The same queries are executing over and over — user profile lookups, job listing fetches, popular search results. Adding Redis caching eliminates these repeated queries. The RDS CPU drops because it's no longer executing queries that Redis can serve. This is the correct use case for caching — it changes the load profile, not just the capacity.

If CPU is high due to complex or long-running queries: The queries are slow because of missing indexes, inefficient joins, or large aggregations. Adding Redis in front of these doesn't help — cache hits only occur after the first execution. A cache miss still causes the expensive query. Scaling up RDS gives the query more CPU to work with but doesn't address the underlying inefficiency. The correct fix: add indexes, optimise queries with EXPLAIN, and use read replicas to distribute the analytical workload.

The diagnostic: check RDS Performance Insights. If the top SQL queries are identical short queries appearing thousands of times per minute (repeated reads), caching helps. If the top queries are unique slow queries with high per-execution CPU, caching doesn't help — you need query optimisation and potentially read replicas.

In practice: do both. Cache the repeated reads (reduces 80% of query volume), and simultaneously optimise the expensive queries. Neither alone is as effective as both together.

Why this works: distinguishes between load reduction (caching's strength) and query optimisation (indexing's strength). Shows you know how to diagnose the actual problem rather than applying a blanket "add cache" answer.
Q2 What is the thundering herd problem with cache TTL expiry? How do you prevent it? Medium ▼

When a heavily-accessed cache entry expires, multiple concurrent requests miss the cache simultaneously. All of them query the database to rebuild the cache. The database receives a spike of identical queries all at once — multiplied by however many application instances are running. This is the thundering herd: a brief moment where what was previously handled by one cache read becomes dozens or hundreds of database queries.

For GetWork, this is realistic: "construction workers available in Mumbai today" is a popular search cached with TTL=300. At expiry, 200+ concurrent users might all miss the cache in the same second.

Prevention approach 1 — Jittered TTLs: Instead of TTL=300, use TTL=300 + random(0,60). Entries for different keys expire at different times within a 60-second window. This doesn't prevent thundering herd for a single key but prevents multiple popular keys from expiring simultaneously.

Prevention approach 2 — Probabilistic early refresh: Before the TTL expires, a small percentage of requests proactively refresh the cache entry in the background. The cache never goes cold. Spring doesn't have this built-in; implement with a custom @Cacheable aspect that checks remaining TTL.

Prevention approach 3 — Cache stampede lock: When a cache miss occurs, acquire a distributed lock (Redis SETNX with short TTL) before querying the database. Only the lock holder queries the database and rebuilds the cache. Other concurrent requests either wait briefly or return stale data. This guarantees only one DB query per cache miss regardless of concurrency.

Why this works: names the problem precisely, explains the mechanics with a concrete GetWork example, and gives three concrete solutions with their trade-offs.
Q3 When would you use DynamoDB instead of RDS for a new feature in a Spring Boot application? Hard ▼

DynamoDB is the right choice when the access pattern is simple, the throughput requirement is extreme, and the data fits a key-value or key-sortedlist model.

Session storage: Every API request authenticates by looking up a session token. Millions of lookups per day. Always by exact token. No joins, no aggregations. DynamoDB GetItem is 1–3ms and scales horizontally without any configuration. RDS for session storage requires a dedicated table, index on token, and connection pool management — at high request volume, the session lookup table becomes a bottleneck.

Rate limiting: Check and increment a counter per IP per time window atomically. DynamoDB's conditional UpdateItem with ADD is an atomic operation — no race conditions, no distributed locking needed. In RDS, implementing an atomic read-modify-write requires either a transaction (expensive) or application-level locking (complex).

Event logs or activity feeds: Append-only logs accessed by entity ID sorted by time. DynamoDB: partition key = entity ID, sort key = timestamp. Query "last 50 events for worker 42" is a single Query with LIMIT=50. In RDS, this requires a table with a composite index and a SELECT with ORDER BY LIMIT — functional but adds connection pool pressure at high volume.

I would keep in RDS: anything requiring complex queries (job search with multiple filters), financial transactions (strong ACID guarantees), reports and analytics (complex aggregations, JOINs). The rule: if the schema has more than 2-3 access patterns and they vary, RDS is more maintainable. If there are 1-2 access patterns and they're simple gets by key, DynamoDB is likely the better fit.

Why this works: gives concrete use cases rather than abstract criteria, and explicitly states what stays in RDS — showing balanced architectural judgement rather than "DynamoDB is better."
Q4 What is a DynamoDB hot partition and how do you detect and fix it? Medium ▼

A hot partition occurs when a disproportionate share of read or write traffic is routed to one DynamoDB partition because the partition key values are not evenly distributed in access frequency. Each DynamoDB partition handles at most 3,000 read units and 1,000 write units per second. If one partition key value receives more traffic than the partition limit, requests are throttled — even if the overall table's provisioned capacity is sufficient.

Detection: CloudWatch metric ConsumedReadCapacityUnits and ConsumedWriteCapacityUnits at the table level will show normal consumption, while individual requests return ProvisionedThroughputExceededException. The table-level metrics look fine because the overall table isn't saturated — only one partition is. Enable CloudWatch Contributor Insights for DynamoDB to see which specific partition key values are generating the most traffic. This makes hot partitions immediately visible.

Fix — key design: The root cause is always a low-cardinality or skewed-access partition key. Fixes: (1) Add a suffix to the partition key to spread load: MUMBAI#0, MUMBAI#1, ..., MUMBAI#9. Randomly select or derive the suffix. Now "Mumbai" traffic is spread across 10 partitions. (2) Use a higher-cardinality partition key: instead of city (few values), use workerId (millions of values). Fetch per-city data via a GSI. (3) Use write sharding: append a random number to the partition key on writes and scatter-gather on reads.

Why this works: explains the partition model (3,000 RCU / 1,000 WCU per partition), the diagnostic subtlety (table-level metrics look fine), and three concrete key design fixes.

Day 13 cheatsheet
ElastiCache + DynamoDB production mechanics at a glance.
Redis latency
100–500µs for GET/SET. All data in RAM — no disk I/O. Compare: RDS 1–10ms. Use Redis for data that must be accessed thousands of times per second.
allkeys-lru eviction
Correct eviction policy for a pure cache. Evicts least recently used when memory full. Watch CacheEvictions in CloudWatch — rising count = time to scale.
Lettuce topology refresh
Set spring.data.redis.lettuce.cluster.refresh.period=30s. Without this, Lettuce may keep writing to a demoted replica for up to 60s after failover.
JSON not Java serialisation
Always use GenericJackson2JsonRedisSerializer. Java serialisation breaks on class structure changes between deployments — cached values become unreadable until TTL expires.
commons-pool2 required
Lettuce pool configuration (spring.redis.lettuce.pool.*) is silently ignored without the Apache commons-pool2 dependency on the classpath.
Thundering herd
Multiple concurrent cache misses hit DB simultaneously. Prevent with: jittered TTLs (TTL + random(0,30)), Redis SETNX lock before DB query, probabilistic early refresh.
DynamoDB partition limits
Each partition: 3,000 RCU/s + 1,000 WCU/s. Hot partition = throttling even when overall table capacity is sufficient. Fix is always key design, not more capacity.
Scan = expensive
DynamoDB Scan reads every item in the table. Consumes RCUs proportional to table size. Never use in a production hot path. Detect with CloudWatch ConsumedReadCapacityUnits spikes.
DynamoDB TTL
Enable on expiresAt Unix epoch attribute. Auto-deletes expired items within 48hrs. Free. Always do application-level TTL check too — deleted items remain readable for up to 48hrs after expiry.
GSI key cardinality
GSI partition keys must be high-cardinality — same rule as base table. Using low-cardinality values (status, city) as GSI PK creates hot partitions in the index.
executionRoleArn
Only needs ECR + CloudWatch Logs + Secrets Manager for injection. Does NOT need your application's DynamoDB/S3/SQS permissions — those go in taskRoleArn. Never combine.
On-demand vs provisioned
On-demand: pay per request, no capacity planning, auto-scales. Provisioned: cheaper at sustained high throughput, throttles on burst. Start on-demand, switch to provisioned when traffic stabilises.
Production rules for ElastiCache + DynamoDB
1Always use a Redis cluster subnet group — never rely on default VPC placement for ElastiCache.
2Enable at-rest encryption and transit encryption on every ElastiCache cluster. Required for any regulated environment.
3Use GenericJackson2JsonRedisSerializer — never Java serialisation. Add @JsonIgnoreProperties(ignoreUnknown=true) to cached classes.
4Set topology refresh period ≤ 30 seconds on Lettuce for replication group HA failover recovery.
5Add commons-pool2 dependency explicitly — Lettuce pooling is silently disabled without it.
6DynamoDB: enable TTL before the table receives significant data. TTL cannot be retroactively applied to existing items efficiently.
7Use Contributor Insights to detect hot partitions in DynamoDB. Enable at table creation for production tables.
8DynamoDB IAM: never grant dynamodb:Scan to application roles. If Scan is needed, run it via a separate admin role.
9Design cache keys hierarchically: prefix:entity:id. Enables pattern-based invalidation: SCAN prefix:jobs:* for job cache flush.
10Monitor ElastiCache: CacheHitRate > 80% for healthy cache. CacheEvictions rising = resize. CurrConnections near max = connection pool leak.
