The S3 object model — not a filesystem
The single biggest conceptual mistake engineers make with S3

S3 is a flat key-value store, not a filesystem. This distinction sounds academic until it causes production bugs. A bucket contains objects. Each object is identified by a key — a string up to 1024 bytes. The key invoices/2024/march/inv-001.pdf looks like a folder path, but there is no "invoices" folder and no "2024" folder. Those slashes are just characters in the key string. The S3 console draws them as folders for human readability, but underneath there is no directory hierarchy whatsoever.

Why this matters — listing performance. If you list objects with prefix invoices/2024/, S3 must scan its index for all keys that begin with that string. With 50 million objects in a bucket, this can be slow and expensive — you pay per LIST request and there's latency. Contrast this with a database where you'd have an indexed column.

The partition key problem: S3 internally partitions objects across its storage fleet based on the key prefix. Before 2018, if all your keys started with the same prefix (like a timestamp: 2024-03-15/image001.jpg), all those objects landed on the same storage partition — creating a hot spot and throttling at ~3,500 PUT/s and 5,500 GET/s per prefix. Engineers deliberately randomised key prefixes to distribute load. In 2018, AWS improved S3's partitioning to be automatic. But per-prefix limits technically still apply — this is exactly the kind of depth that impresses AWS interviewers.

An object has: a key, a value (bytes — up to 5TB), system metadata (Content-Type, ETag, Last-Modified), up to 10 user-defined metadata key-value pairs, a version ID (if versioning enabled), and an access control list. The ETag is the MD5 hash for single-part uploads. For multipart uploads, the ETag is a hash of hashes — which trips engineers who use ETag for content integrity checks.

Flat keyspace — no real folders Per-prefix request limits exist
How a PUT actually works inside S3
What happens between your API call and the 200 OK response

When your Spring Boot app calls s3.putObject(), the following happens internally — and knowing this explains why S3 is durable, why it achieved eventual consistency historically, and why large uploads are slow to return.

Step 1 — Request routing: Your request hits an S3 front-end server. S3 determines which storage partition owns the key based on bucket name + key hash. It routes internally to the appropriate storage node cluster.

Step 2 — Data write: S3 writes the object data to multiple storage nodes simultaneously across at least three Availability Zones. S3 uses erasure coding — not simple 3x replication. The data is split into chunks and encoded so that if any subset of chunks is lost, the original can be reconstructed. More storage-efficient than storing 3 full copies.

Step 3 — Index update: S3 updates its distributed metadata index to record the key exists and its storage location. From 2020 onwards, this index update is synchronous with the PUT response. Before 2020, it was asynchronous — the source of eventual consistency.

Why the 200 OK timing matters: S3 only returns 200 OK after the object is written across multiple AZs AND the index is updated. A successful PUT guarantees the object is durable before you get the response. Never fire-and-forget a PUT. If you get a 200 OK, the object is durably stored. If you get 5xx, retry with exponential backoff — the write may have partially succeeded, but S3's guarantees mean a retry is always safe.
200 OK = durably stored across 3+ AZs Erasure coding, not 3x replication
The consistency model — what changed in 2020
The most misunderstood S3 concept and a guaranteed senior interview topic

Before December 2020, S3 had eventual consistency for overwrite PUTs and DELETEs. If you PUT a new version at an existing key, then immediately GET it — you might get the old version. If you DELETE an object and immediately LIST, the deleted object might still appear. This wasn't a bug — it was a documented trade-off for global scale.

Real production failure pattern (pre-2020): A deployment pipeline uploads a new config file to S3, then immediately reads it back to validate. The read returns the old config — index hasn't propagated. The pipeline concludes the upload failed and retries — uploading a third copy, now confused about which version is authoritative. Engineers worked around this with cache-busting keys (including a UUID in the key so each upload is brand-new, never an overwrite) or DynamoDB as a coordination layer to track which version was authoritative.

In December 2020, AWS delivered strong read-after-write consistency for all S3 operations — PUTs, DELETEs, LIST operations. After a successful PUT, any subsequent GET returns the new object. Guaranteed globally, for free, with no performance cost.

Why you still need to know the old model: Any codebase written before December 2020 may have workarounds — random prefixes, DynamoDB coordination, retry delays. When you join a team and see these patterns, you need to know what problem they were solving, and whether they were solving multiple problems simultaneously. Blindly removing them could introduce subtle bugs.
Strong consistency from Dec 2020 Pre-2020 workarounds exist in old code — understand why
How S3 achieves 11 nines of durability
99.999999999% — what this number actually means and how it's engineered

S3 Standard promises 99.999999999% (11 nines) durability. Concretely: if you store 10 million objects, you'd expect to lose one object every 10,000 years. S3 stores data across at least three AZs using erasure coding. AWS continuously runs bit rot detection — periodically reads every object, verifies checksums, and automatically heals corrupt data from parity chunks.

Durability vs Availability — the distinction engineers confuse: Durability means your data is not lost. Availability means your data can be accessed right now. S3 Standard gives 11 nines durability but only 99.99% availability (four nines) — up to 52 minutes of unavailability per year. Your data still exists during unavailability — you just can't access it temporarily. S3 One Zone-IA drops to 99.5% availability because it doesn't replicate across AZs.

What S3 durability does NOT protect against: Accidental deletion (you delete the object — S3 faithfully executes it). Overwrites without versioning. Bucket deletion. Ransomware using your own credentials. For protection against these: S3 Versioning, S3 Object Lock (WORM), MFA Delete, and bucket replication to a separate account the primary account cannot delete from.

Durability failure engineers miss: A Spring Boot bug deletes an object instead of updating it. S3 faithfully deletes it. The engineer checks AWS docs, sees "11 nines durability", and is confused why the object is gone. Durability protects against S3's infrastructure failing — not against your own code. Versioning + Object Lock is mandatory for critical data.
Durability ≠ Availability — know the difference Enable versioning — 11 nines doesn't protect from your own deletes
Multipart upload — when and why it matters
The mechanism behind large file uploads and why you must use it in production

S3 has a single-PUT limit of 5GB. For objects larger than 5GB, multipart upload is mandatory. AWS recommends multipart for anything above 100MB. Your application splits the object into parts (minimum 5MB each). Each part uploads independently and in parallel. If one part fails, only that part retries — not the entire file.

Why this matters for GetWork: If workers upload video proof-of-work (500MB–2GB), a single-PUT means if the network drops at 90%, the entire upload restarts. With multipart, only the failed part retries. Additionally, uploading 8 parts simultaneously at 10MB each = 80MB/s effective throughput vs 10MB/s sequential. The AWS Java SDK's TransferManager does this automatically above a configurable threshold.

The lifecycle policy trap: If a multipart upload is initiated but never completed (app crashes mid-upload), the parts accumulate and you are charged for their storage even though there's no usable object. AWS does not automatically clean these up. Engineers discover this when their S3 bill is mysteriously higher than expected — terabytes of incomplete multipart parts. Fix: create an S3 Lifecycle rule to abort incomplete multipart uploads after 7 days. This should be on every bucket you ever create.

Production failure: A pipeline uploads 50GB log archives with a single PUT. Works for months. One day the upload exceeds the pipeline's timeout — process is killed at 80% complete. No object is stored (single-PUT is atomic — partial data not accessible). The pipeline reports success (bad error handling). Next day's job fails because yesterday's archive doesn't exist. Fix: TransferManager with multipart + explicit success validation.
Use multipart above 100MB Set lifecycle rule to clean incomplete uploads — every bucket
Presigned URLs — the right way to give temporary access
Share private S3 objects without making them public or exposing credentials

A presigned URL is a time-limited URL granting temporary access to a private S3 object — or allowing upload — without the recipient needing AWS credentials. Generated by your backend using your credentials, baked into the URL's signature, then given to the client who uses it directly against S3. Your backend is not in the upload/download data path.

How the signature works: The URL contains bucket, key, expiry timestamp, your access key ID, and an HMAC-SHA256 signature computed from all those parameters using your secret key. S3 recomputes the signature using the access key ID to look up the secret key from IAM. Signatures match + expiry not passed = access granted. Your secret key never leaves your server.

Presigned PUT URLs — the pattern GetWork needs: Naive approach: client → your backend → backend uploads to S3. Every upload goes through your EC2, wasting bandwidth and compute. Correct pattern: client requests upload URL → backend generates presigned PUT URL with 5-minute expiry → client uploads directly to S3 → S3 stores the file. Your backend never touches the file bytes. This offloads all upload traffic to S3's infrastructure, which is built for exactly this.

Expiry and the clock skew problem: Presigned URLs expire at a specific UTC timestamp. Mobile device clocks drift. Always generate presigned URLs server-side. Set 15 minutes for user-facing uploads, 5 minutes for automated backend operations.

Security failure: A developer generates a presigned URL with 7-day expiry "to be safe" for a document with sensitive financial data. URL is sent via email. User forwards the email. Anyone with the forwarded email has 7-day access. Presigned URLs are as sensitive as the data they expose — treat them like passwords. Use the shortest expiry that works. Always HTTPS. Log S3 access via CloudTrail to detect unexpected access patterns.
Client uploads direct to S3 — your backend not in data path Short expiry + HTTPS always — treat like a password

How S3 permission evaluation actually works
The evaluation order that determines what gets denied — most engineers get this wrong

1. Explicit DENY? If any policy explicitly denies the action, the request is denied immediately. No other policy can override an explicit deny. This is the most powerful rule.

2. Explicit ALLOW? If any policy explicitly allows the action AND there is no explicit deny, the request is allowed. The allow can come from either IAM identity policy or bucket policy — they are OR'd together.

3. Neither — implicit deny. The default is deny. No explicit allow = denied.

Cross-account access — the critical nuance: If the requester is from a different AWS account, BOTH the IAM policy (requester's account) AND the bucket policy (bucket's account) must allow the action. This is AND logic, not OR. A cross-account request needs explicit permission on both sides. Forgetting this causes the most common S3 access debugging session: "My IAM policy says allow, why is S3 denying?" — because the bucket policy in the other account has no allow for your account.
The confused deputy scenario: Your EC2 instance has an IAM role allowing s3:GetObject on all buckets (*). An attacker compromises your EC2 and uses the instance metadata service to steal the role credentials. They can now read every S3 bucket in your account. Solution: Bucket policies explicitly restrict access to specific IAM roles. VPC endpoints with bucket policies deny all access except from the VPC. SCPs at the AWS Organizations level add account-wide guardrails.
Bucket policies — the right way to control access
JSON resource policies attached to the bucket — more powerful than ACLs

A bucket policy is a JSON resource policy attached to the S3 bucket. Unlike IAM policies (identity-based), bucket policies are resource-based — they define who can access this specific resource. They can grant access to other AWS accounts, services, and anonymous users.

Two policies you should put on every production bucket:
Enforce HTTPS only — Deny all requests where aws:SecureTransport = false. Without this, data can be intercepted in transit. Standard compliance requirement.
Restrict to VPC only — Deny all requests where aws:SourceVpc is not your VPC ID. Only resources inside your VPC can access the bucket — not even IAM users accessing from outside. Used for highly sensitive data.
The wildcard principal mistake — the root cause of every S3 data breach: Setting Principal to "*" with Action s3:GetObject makes every object publicly readable to the entire internet. This exact configuration caused almost every major S3 data breach. AWS added Block Public Access specifically to prevent this — but if BPA is disabled and a wildcard policy exists, your data is exposed to the public internet with no further authentication required.
S3 Block Public Access — the account-level safety net
Four independent settings that override bucket policies and ACLs

BlockPublicAcls: Blocks new ACLs that grant public access. Does NOT remove existing public ACLs.

IgnorePublicAcls: Makes S3 ignore ALL public ACLs entirely — even existing ones. Stronger than BlockPublicAcls.

BlockPublicPolicy: Prevents new bucket policies granting public access. Rejects Principal "*" policies. Does not affect existing policies.

RestrictPublicBuckets: Restricts public and cross-account access even if a public policy somehow exists. The nuclear option.

AWS default since April 2023: All four BPA settings are enabled BY DEFAULT on all new S3 buckets. Any policy or ACL trying to grant public access is blocked or ignored. You must consciously opt out to make a bucket public. The friction is intentional — AWS is making you confirm public exposure is deliberate.
Account-level vs bucket-level confusion: BPA can be set at account level (applies to all buckets) and bucket level. Account-level is stronger — it overrides bucket-level. Many teams enable BPA at account level for compliance, then are confused when a specific bucket they need to make public can't be. Solution: disable BPA at the specific bucket level for that bucket, keep account-level BPA on for all others.
S3 encryption — at rest and in transit
Three options with very different key management and cost implications

SSE-S3: AWS manages keys entirely. Free, automatic, enabled by default since January 2023. Sufficient for most use cases. Not acceptable when regulations require customer-controlled keys.

SSE-KMS: Keys in AWS KMS. You control rotation, key policies, get CloudTrail audit of every key usage. Every S3 object GET triggers a KMS API call — costs $0.03/10,000 requests with 5,500 calls/second rate limits. Required for PCI-DSS and similar financial compliance frameworks.

SSE-C: You send the encryption key with every PUT and GET. AWS encrypts/decrypts and never stores the key. If you lose the key, data is permanently inaccessible. Highest control, highest operational risk.

KMS throttling failure: A media company stores video thumbnails with SSE-KMS. Their pipeline processes 10,000 videos simultaneously — each GET triggers a KMS decrypt call. KMS default limit is 5,500 calls/second. They hit the limit, get ThrottlingException errors, blame S3 — but S3 is fine, KMS is the bottleneck. Fix: enable S3 Bucket Keys (caches the data key for the bucket, reducing KMS calls by up to 99%) or request a KMS limit increase.
These are the actual patterns used in production systems — not textbook examples. Each maps to a real decision you will face when building GetWork, Hotel microservices, or at Deloitte with JP Morgan.
S3 static website hosting
Host a full React/Angular frontend with zero servers

S3 can serve static content directly. Enable "Static website hosting" on the bucket, configure index.html and error document, and S3 gives you a website endpoint. S3 alone has limitations: no HTTPS on the S3 website endpoint (HTTP only), no custom domain, no global performance.

The production-ready architecture — S3 + CloudFront: CloudFront sits in front of S3, provides HTTPS (ACM certificate — free), custom domain via Route 53, global edge caching (400+ locations), and AWS Shield Standard (DDoS protection — free). CloudFront serves from edge cache — your S3 bucket gets almost zero direct traffic, reducing costs and latency dramatically.

SPA routing problem: React Router handles routing client-side. When a user navigates directly to https://getwork.app/jobs/123, S3 tries to find an object at key jobs/123. It doesn't exist — S3 returns 403 or 404. Fix: configure CloudFront with a custom error response — intercept 403 and 404 responses from S3 and return index.html with HTTP status 200. The browser loads index.html, React Router reads the URL path, renders the correct page. This catches almost every first-time deployer.

Cache invalidation after deployment: CloudFront caches your S3 files at edge locations. After deploying a new React build, old cached files continue serving until TTL expires. Solution: use content-addressed filenames (Webpack/Vite adds a hash — main.a3b4c5.js). Only index.html needs a CloudFront invalidation after each deploy. Create an invalidation for /index.html as the last step of your CI/CD pipeline. Invalidating /* (all files) costs money per path — only invalidate what changed.
S3 event-driven pipeline
S3 as the trigger point for an entire processing workflow

S3 emits events when objects are created, deleted, or restored. These trigger Lambda functions, SQS queues, or SNS topics — making S3 a natural trigger for processing pipelines.

GetWork worker document processing pattern: Worker uploads ID document → S3 PUT triggers Lambda → Lambda calls AWS Rekognition (ID document analysis) + Textract (text extraction) → results stored in DynamoDB → SNS notification to admin → admin reviews in dashboard. The entire pipeline is event-driven, runs only when triggered, scales to zero when idle. You pay only for Lambda invocations, not always-on servers.

Event delivery and the idempotency requirement: S3 event notifications are delivered at least once — the same event can be delivered twice in rare cases. If your Lambda is triggered twice for the same object, it must be idempotent. For document processing: check DynamoDB before processing to see if the document ID was already handled. If yes, skip. Prevents charging a worker twice or creating duplicate records.

The missing event trap: S3 events are asynchronous — in rare edge cases, an event may not be delivered. For critical pipelines where every file must be processed, don't rely solely on S3 events. Use S3 Inventory (daily manifest of all objects) as a reconciliation mechanism — compare the inventory against your processing records to find objects that were uploaded but never triggered an event. Run this reconciliation daily as a batch job.

S3 + SQS for production safety: Instead of triggering Lambda directly, S3 sends events to SQS, and Lambda polls SQS. SQS buffers events if Lambda is throttled. Multiple Lambda functions can consume from the same queue concurrently. This is the production-safe pattern for high-volume S3 processing.

Spring Boot + S3 — the right integration pattern
How to integrate S3 with your existing Spring Boot apps correctly

Use AWS SDK v2 for Java. Never hardcode credentials. Use IAM instance roles on EC2 (SDK picks them up automatically from the instance metadata service) — never credentials in application.properties.

S3Client vs S3AsyncClient — a critical decision for WebFlux: Your Spring Boot app uses WebFlux (reactive) given your Deloitte project. Use S3AsyncClient which returns CompletableFuture — bridge to Reactor with Mono.fromFuture(). Using the blocking S3Client inside a WebFlux handler blocks the Netty event loop thread — destroying your reactive throughput. This is a real mistake causing silent performance degradation in reactive Spring Boot apps that is extremely hard to diagnose.
The upload pattern for GetWork (presigned PUT):
1. Worker app: POST /api/upload-url?filename=photo.jpg
2. Backend validates session, generates presigned PUT URL with 10-minute expiry
3. Backend creates DynamoDB record: {docId, workerId, status: PENDING, s3Key}
4. Backend returns presigned URL to worker app
5. Worker app PUTs file directly to S3 using presigned URL
6. S3 event triggers Lambda → Lambda updates DynamoDB to UPLOADED
7. Admin dashboard queries DynamoDB for UPLOADED documents
Your backend never receives file bytes. Your EC2 just manages metadata and URLs.
Thread pool exhaustion failure: A Spring Boot MVC app (not WebFlux) uses blocking S3Client inside a REST endpoint to upload files. Under load, concurrent requests each block a Tomcat thread while waiting for S3. Tomcat's default 200-thread pool exhausts — requests queue, latency spikes, service appears down. Fix: offload S3 operations to a dedicated @Async executor, or switch to S3AsyncClient with CompletableFuture, or use presigned URLs so S3 interaction happens entirely on the client side.
S3 Cross-Region Replication (CRR)
When and why you replicate across regions — and what it does NOT do

S3 CRR automatically copies objects from a source bucket in one region to a destination bucket in a different region. Replication is asynchronous — typically seconds to minutes delay. Requires versioning enabled on both buckets.

What CRR does NOT do — the misconceptions that cause incidents: CRR does not replicate existing objects — only new objects written after replication is configured. 10TB of existing data is not copied. Use S3 Batch Operations for a one-time initial sync. CRR does not replicate delete markers by default — deleting an object in the source does NOT automatically delete in the destination. This is intentional — the replica is a safety net against accidental deletes. CRR does not provide strong consistency across regions — the replica lags by seconds to minutes.
The replication lag compliance failure: A financial services team sets up CRR and tells their compliance officer data is "replicated in real-time to the EU region". The auditor tests this — objects appear in the EU replica 45 seconds after being written. The compliance requirement specifies "real-time replication". S3 CRR is asynchronous and does not meet "real-time" requirements. Fix: either re-negotiate the compliance definition or write to both regions simultaneously in your application (synchronous dual-write), accepting the performance and complexity cost.
Same-Region Replication (SRR) — the underused pattern: Replicates within the same region. Use case: replicate from production bucket to a separate account's bucket for audit log isolation — the audit account can read but not write to production, preventing tampering. Use case: aggregate logs from multiple source buckets into a single destination. SRR has same mechanics as CRR but without cross-region data transfer costs.

Q1 S3 claims 11 nines of durability. Can you actually lose data in S3? Walk me through every way it can happen. Hard▼

Yes, absolutely — and this is one of the most important things to understand about S3. The 11 nines durability guarantee refers specifically to S3's infrastructure not losing your data due to hardware failure or bit rot. AWS's internal systems are extremely robust at preserving stored data. However, there are several ways data is legitimately lost that the durability guarantee does not protect against.

Deletion by your application or your team. If your code calls s3:DeleteObject, or a developer runs the wrong CLI command, S3 faithfully executes the delete. This is not an infrastructure failure — you asked S3 to delete, and it did. Without versioning, this is permanent. With versioning enabled, a delete operation adds a delete marker — the object isn't actually removed. You can restore it by removing the delete marker.

Overwrites without versioning. PUT a new version of an object at the same key without versioning enabled — the old version is gone. S3 doesn't keep history unless you explicitly enable versioning.

Expired lifecycle policies. A lifecycle policy configured to delete objects after 30 days will delete them — exactly as instructed. This has caused real data loss when engineers forget lifecycle rules exist. Fix: S3 Object Lock (WORM) for data that must never be deleted regardless of lifecycle rules.

Compromised AWS credentials. An attacker with IAM credentials with s3:DeleteObject can delete every object programmatically. S3 has no way to distinguish legitimate deletes from malicious ones at the API level. Fix: S3 Object Lock in Compliance mode — even the root account cannot delete locked objects. MFA Delete. Bucket replication to a separate account where the primary account has no delete permissions.

Bucket deletion. If the bucket is deleted, all objects inside are gone. You can't delete a non-empty bucket via console — but you can programmatically. Fix: SCPs to deny s3:DeleteBucket for production accounts.

Why this answer works: It reframes the question correctly — durability is about infrastructure, not protecting you from yourself. Every failure mode has a concrete mitigation, showing defence-in-depth thinking. The S3 Object Lock detail signals you know AWS's advanced security features.
Q2 Explain the S3 consistency model and what changed in 2020. How did engineers design around the old model? Hard▼

Before December 2020, S3 had a split consistency model. New object PUTs had read-after-write consistency — PUT a brand-new key and immediately GET it, you'd always get the new object. But overwrite PUTs and DELETEs had only eventual consistency. Overwrite an existing object and immediately read it — you might get the old version. Delete an object and immediately list — the deleted object might still appear. This was a documented property of S3's distributed architecture: index updates propagated asynchronously.

The reason for this design: at S3's scale, strong consistency would require synchronous coordination across all storage nodes before returning a response — massively increasing latency and reducing throughput. Eventually consistent systems trade correctness windows for performance. S3 prioritised throughput; the consistency window was typically milliseconds to a few seconds. But at high request rates, it was observable.

Engineers designed around eventual consistency in several ways. Cache-busting keys — instead of overwriting config/settings.json, write config/settings-v42.json and store the version pointer in DynamoDB. DynamoDB provided strong consistency; S3 provided storage. DynamoDB coordination layer — a DynamoDB item tracked which S3 key was the current authoritative version. Time-delayed reads — pipelines waited 5 seconds after a write before reading. Brittle but common.

In December 2020, AWS delivered strong read-after-write consistency for all S3 operations — PUTs, DELETEs, and LIST — for all buckets, with no performance degradation and no additional cost. AWS redesigned S3's metadata subsystem to make index updates synchronous with the PUT/DELETE response.

The implication for existing codebases: these workarounds are technically unnecessary for consistency. However, many workarounds — DynamoDB coordination, versioned keys — provide other benefits (DynamoDB gives atomic operations and transactions; versioned keys enable rollback). Don't blindly remove them without understanding whether they were solving multiple problems simultaneously.

Why this answer works: Explains the why behind the old design (performance vs consistency trade-off), demonstrates distributed systems fundamentals, shows real engineering workarounds, and concludes with practical advice about legacy code — the reasoning of a senior engineer who has maintained long-lived systems.
Q3 Your S3 bill is unexpectedly 5x higher this month. Walk me through your diagnosis process. Hard▼

I'd approach this systematically, eliminating cost categories one at a time.

Step 1 — Cost Explorer breakdown. Open AWS Cost Explorer, filter to S3, group by usage type. S3 has independent cost dimensions: storage (GB-months per class), requests (PUT, GET, LIST — priced per 1,000), data retrieval (per-GB on IA and Glacier), data transfer out (per-GB to internet or other regions), replication (CRR transfer), and Intelligent-Tiering monitoring fees. The breakdown shows which dimension spiked.

If storage spiked: Check if versioning is enabled and objects are being overwritten without a lifecycle rule deleting old versions — versions accumulate silently. Check for incomplete multipart uploads. Use S3 Storage Lens for per-bucket, per-class breakdown.

If request costs spiked: Something is making far more API calls. Common causes: application doing LIST operations in a loop (bad pagination — calling ListObjects repeatedly without using the continuation token), CloudFront cache miss rate increased (more GETs hitting S3 origin — check CloudFront cache hit ratio metric), or an S3 event trigger Lambda that's failing and being retried in a loop.

If data retrieval spiked: Data was moved to IA or Glacier, but access patterns didn't actually decrease. The IA retrieval trap — storage cost fell, but retrieval fees exceeded the savings.

If data transfer out spiked: A new analytics job is reading raw data from S3 to a local machine instead of using Athena, or a CloudFront distribution has a bug causing large files to be served without range requests, or CRR is replicating large volumes to another region.

After identifying root cause: set up S3 Storage Lens with advanced metrics, configure S3 cost allocation tags, and create a CloudWatch alarm on estimated S3 charges so any future spike triggers an alert within 24 hours.

Why this answer works: Demonstrates detailed knowledge of S3's pricing model (most engineers don't know all the dimensions), structured debugging approach, and closes with prevention — not just fixing the immediate issue. This is the senior engineer diagnostic mindset interviewers are probing for.
Q4 How would you design S3 access for a multi-tenant SaaS where tenants must never access each other's data? Hard▼

Multi-tenant S3 isolation has several valid approaches with different trade-offs.

Option 1 — One bucket per tenant. Each tenant gets their own S3 bucket. IAM policies restrict each tenant's role to only their bucket. Strongest isolation — a misconfigured policy on tenant A's bucket cannot affect tenant B's. However, AWS has a default limit of 100 buckets per account (raiseable, but still a constraint). For a SaaS with thousands of tenants, doesn't scale.

Option 2 — One bucket, per-tenant key prefix with IAM conditions. All tenants share one bucket. Each tenant's data stored under tenant-{tenantId}/ prefix. IAM policies use s3:prefix condition to restrict access to their prefix. Scales to millions of tenants in one bucket. Risk: a misconfigured IAM policy or key-generation bug could allow cross-tenant access. Use AWS IAM Access Analyzer to continuously validate policies.

Option 3 — Separate AWS account per tenant. Enterprise SaaS. Complete account-level isolation. Your application account assumes a cross-account role to read/write to their bucket when needed. Managed via AWS Organizations and Control Tower. Used by enterprise-grade SaaS platforms with strict compliance requirements. Complex to operate at scale.

Option 4 — Presigned URLs with server-side enforcement. Your backend generates presigned URLs for specific objects. Tenant authenticates to your backend, backend validates their identity and which objects they're authorised to access, then generates a presigned URL for only that specific object. S3 never knows about tenants — isolation is enforced by your application layer. Risk: your application's authorisation logic is the security boundary. A bug breaks isolation.

For GetWork: Option 2 (single bucket, per-worker prefix) with presigned URLs for all client access. The backend enforces that a worker's session token can only generate presigned URLs for their own workers/{workerId}/ prefix. Simple, scales to millions of workers, application layer provides the authorisation boundary.

Why this answer works: Multiple options with trade-offs rather than one "correct" answer — architectural maturity. Tied to real context (GetWork). Mention of IAM Access Analyzer shows knowledge of AWS tooling beyond core services.
Today's lab makes you observe things — the consistency model, the versioning mechanism, the presigned URL flow, the lifecycle policy — so the mechanics become visceral, not theoretical.
1
Create a bucket with production-grade settings from the start
Create a bucket in ap-south-1 with versioning enabled. Never create a bucket for real data without versioning. Create a separate logging bucket first. Enable server access logging targeting the logging bucket.
aws s3api create-bucket \ --bucket zuber-learning-$(date +%s) \ --region ap-south-1 \ --create-bucket-configuration LocationConstraint=ap-south-1 aws s3api put-bucket-versioning \ --bucket YOUR-BUCKET-NAME \ --versioning-configuration Status=Enabled aws s3api get-bucket-versioning --bucket YOUR-BUCKET-NAME
The $(date +%s) appends a Unix timestamp for global uniqueness. Note the bucket name — used throughout this lab.
2
Observe versioning — the mechanism that saves your data from yourself
Upload a file. Overwrite it. Observe both versions exist. Delete it — observe what actually happens with versioning enabled. This is the difference between data loss and data safety.
echo "Version 1 content" > test.txt aws s3 cp test.txt s3://YOUR-BUCKET/test.txt echo "Version 2 content" > test.txt aws s3 cp test.txt s3://YOUR-BUCKET/test.txt # See BOTH versions — not just the latest aws s3api list-object-versions \ --bucket YOUR-BUCKET --prefix test.txt # "Delete" the object aws s3 rm s3://YOUR-BUCKET/test.txt # List again — a delete MARKER was added, not a real delete aws s3api list-object-versions \ --bucket YOUR-BUCKET --prefix test.txt # Restore: delete the delete marker (get VersionId from above) aws s3api delete-object \ --bucket YOUR-BUCKET \ --key test.txt \ --version-id DELETE-MARKER-VERSION-ID
S3 delete with versioning = delete marker added. The data is NOT gone. Remove the delete marker and the object reappears. Without versioning, that first delete would be permanent and unrecoverable.
3
Test strong consistency — verify the 2020 guarantee yourself
Write a script that uploads an object and immediately reads it back in a loop. Every read should return the latest version. Think about what this would have returned before December 2020.
for i in {1..10}; do echo "content-version-$i" > temp.txt aws s3 cp temp.txt s3://YOUR-BUCKET/consistency-test.txt RESULT=$(aws s3 cp s3://YOUR-BUCKET/consistency-test.txt -) echo "Wrote: content-version-$i | Read: $RESULT" done
Every line should show wrote and read matching. Pre-2020, some iterations would have returned the previous version. You are now observing the December 2020 guarantee directly.
4
Generate and use a presigned URL — the exact pattern your apps will use
Generate a presigned GET URL. Access it from your browser with zero AWS credentials. Generate a presigned PUT URL and upload using curl. This is the GetWork worker upload pattern.
# Presigned GET URL (60 second expiry) aws s3 presign s3://YOUR-BUCKET/test.txt \ --expires-in 60 --region ap-south-1 # Open the URL in browser — no login required # Presigned PUT URL via Python SDK python3 -c " import boto3 s3 = boto3.client('s3', region_name='ap-south-1') url = s3.generate_presigned_url( 'put_object', Params={'Bucket': 'YOUR-BUCKET', 'Key': 'via-presigned.txt'}, ExpiresIn=300 ) print(url) " # Upload directly to S3 using the URL — no AWS credentials in curl curl -X PUT -d "Hello from presigned PUT" "PASTE-URL-HERE" aws s3 cp s3://YOUR-BUCKET/via-presigned.txt -
The presigned PUT succeeded without any AWS credentials in the curl command. The signature baked into the URL is sufficient. This is how 100M-user apps handle file uploads without routing through their servers.
5
Write a real bucket policy — enforce HTTPS only (production requirement)
Apply a bucket policy that denies all non-HTTPS requests. This is a standard compliance requirement at JP Morgan and every other regulated enterprise.
cat > policy.json << 'EOF' { "Version": "2012-10-17", "Statement": [{ "Sid": "DenyNonHTTPS", "Effect": "Deny", "Principal": "*", "Action": "s3:*", "Resource": [ "arn:aws:s3:::YOUR-BUCKET", "arn:aws:s3:::YOUR-BUCKET/*" ], "Condition": { "Bool": {"aws:SecureTransport": "false"} } }] } EOF aws s3api put-bucket-policy \ --bucket YOUR-BUCKET --policy file://policy.json aws s3api get-bucket-policy --bucket YOUR-BUCKET
AWS CLI always uses HTTPS so your normal operations still work. This policy denies any request coming over plain HTTP — a standard PCI-DSS and SOC 2 requirement.
6
Configure a lifecycle policy — automate cost management and clean up the multipart trap
Create a lifecycle policy with storage class transitions AND the incomplete multipart upload cleanup rule that should be on every S3 bucket you ever create.
cat > lifecycle.json << 'EOF' { "Rules": [ { "ID": "TransitionLogs", "Status": "Enabled", "Filter": {"Prefix": "logs/"}, "Transitions": [ {"Days": 30, "StorageClass": "STANDARD_IA"}, {"Days": 90, "StorageClass": "GLACIER"} ] }, { "ID": "CleanIncompleteMultipart", "Status": "Enabled", "Filter": {"Prefix": ""}, "AbortIncompleteMultipartUpload": {"DaysAfterInitiation": 7} } ] } EOF aws s3api put-bucket-lifecycle-configuration \ --bucket YOUR-BUCKET \ --lifecycle-configuration file://lifecycle.json aws s3api get-bucket-lifecycle-configuration --bucket YOUR-BUCKET
The CleanIncompleteMultipart rule should be on EVERY S3 bucket. Without it, abandoned multipart uploads silently accumulate and inflate your bill — invisible in the console until you run S3 Storage Lens.
Day 3 — concepts understood at depth
Flat keyspace
No real folders. Slashes are characters. Affects listing performance and partition key hot spots.
PUT mechanics
200 OK = written across 3+ AZs via erasure coding. Not 3x replication. Index updated synchronously post-2020.
Strong consistency
From Dec 2020. All ops strongly consistent. Pre-2020 workarounds exist in old code — know what they were solving.
11 nines durability
Protects against AWS infrastructure failures ONLY. Your deletes, overwrites, and compromised credentials can still lose data.
Versioning
Delete = delete marker. Overwrite = new version, old survives. Enable on every bucket with data you cannot recreate.
Multipart upload
Mandatory above 5GB, recommended above 100MB. Incomplete parts = billed. Set lifecycle to abort after 7 days.
Presigned URL
Client uploads direct to S3. Backend never touches file bytes. Short expiry. Treat like a password.
SSE-KMS + Bucket Keys
KMS per object = throttle risk at scale. Bucket Keys reduce KMS calls by 99%. Always enable with SSE-KMS.
Storage class minimums
Standard-IA: 30-day min. Glacier: 90 days. Deep Archive: 180 days. Early delete = charged for full minimum.
Block Public Access
4 independent settings. Account-level overrides bucket-level. Enabled by default since April 2023.
Permission evaluation
Explicit DENY always wins. Cross-account = BOTH IAM policy AND bucket policy must allow (AND logic).
ETag caveat
Single-part upload = MD5 of data. Multipart = hash of hashes. Do NOT use ETag for content integrity on multipart objects.
Non-negotiable rules for every S3 bucket
1. Enable versioning on every bucket containing data you cannot recreate.
2. Set lifecycle rule to abort incomplete multipart uploads after 7 days — every bucket.
3. Deny non-HTTPS requests with a bucket policy.
4. Block Public Access at account level. Opt out per-bucket only when deliberate.
5. Enable SSE-S3 or SSE-KMS. Enable Bucket Keys if using KMS.
6. Never set the access log target as the same bucket — infinite recursion.
7. For sensitive data: Object Lock + CloudTrail Data Events + replication to a separate account.
8. Never route file bytes through your EC2 — use presigned URLs for client uploads.
9. Never use ETag for content integrity on objects uploaded with multipart.
