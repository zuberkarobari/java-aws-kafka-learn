CI/CD on AWS — what we're building and why
A CI/CD pipeline is the automated path from a git commit to a running production deployment. Without it, every release is a manual, error-prone, anxiety-inducing event. With it, deployments are boring — which is exactly what production should be.

Each stage is a gate. A failure in any stage stops the pipeline and leaves production untouched. The earlier a failure is caught, the cheaper it is to fix. A test failure in the Build stage costs seconds. A failure caught in production validation after a bad deploy costs real users real downtime.
AWS-native pipeline (CodePipeline)
Built entirely from AWS services. CodePipeline orchestrates, CodeBuild compiles and tests, CodeDeploy pushes to EC2/ECS.

Best for: Teams already using AWS-managed source (CodeCommit) or want full AWS integration (IAM, CloudWatch, EventBridge native events).

Cost: $1/active pipeline/month + CodeBuild minute charges (~$0.005/min for default environment).
Bitbucket Pipelines → AWS (your stack)
Bitbucket handles orchestration and build. AWS OIDC provides temporary credentials. Direct SDK/CLI calls deploy to ECS, EC2 ASG, Lambda.

Best for: Teams already on Bitbucket (like your Deloitte project). Less AWS vendor lock-in. Familiar YAML-based config.

Cost: Bitbucket minutes (~50 free/month, then $10/month for more) + zero AWS pipeline cost.
Why this day matters for your JP Morgan project At Deloitte, your code goes through a CI/CD pipeline before reaching JP Morgan's environment. Understanding CodePipeline and CodeDeploy explains what happens in that pipeline — how builds are triggered, where artifacts are stored, how deployments are gated, and why rollbacks sometimes don't happen automatically when they should. When a deployment breaks production and the pipeline doesn't auto-rollback, you need to know where to look.
CodePipeline — the orchestrator
CodePipeline is the workflow engine. It doesn't build, test, or deploy — it coordinates the services that do. Understanding its stage/action model, artifact passing, and approval gates explains every CodePipeline behaviour you'll encounter.

A CodePipeline pipeline is a series of stages. Each stage contains one or more actions. Stages run sequentially. Within a stage, actions can run in parallel (same run order number) or sequentially (different run order numbers). A stage is complete only when all its actions succeed.

CodePipeline is a state machine. It doesn't poll for changes — it responds to events. A push to the main branch in CodeCommit (or an S3 object upload, or an EventBridge rule) triggers a pipeline execution. The execution moves through stages; each action invokes an AWS service (CodeBuild, CodeDeploy, Lambda, ECS) or waits for a human approval.

How CodePipeline knows when source code changes For CodeCommit sources: EventBridge detects the commit and triggers the pipeline automatically. For S3 sources: an S3 event notification triggers the pipeline when the specified object changes. For GitHub/Bitbucket connections: AWS CodeStar Connections creates a webhook in the external source provider. When you push to the configured branch, the webhook fires EventBridge, which triggers the pipeline. The connection must be manually authorised in the AWS console before webhooks work — a common first-time setup step that engineers miss.
The pipeline that runs on every commit, including hotfix branches By default, a CodePipeline source stage triggers on the configured branch. If you configure it on "main", only merges to main trigger the pipeline. But if you accidentally configure it on all branches (using a wildcard trigger pattern), every feature branch push triggers a production deployment attempt. Lock the trigger: configure the source action to trigger only on the specific production branch. Use separate pipelines (or pipeline conditions) for different environments — one pipeline for staging (triggers on develop branch), one for production (triggers on main or version tags only).

Each action in a stage has a provider (which AWS service executes it), an action category (Source, Build, Test, Deploy, Approval, Invoke), and an optional run order (determines parallel vs sequential execution within a stage).

Source actions
Provider: CodeCommit, S3, GitHub, Bitbucket (via CodeStar Connections), ECR.

Downloads the source code or artifact into a pipeline artifact for downstream stages. The artifact is stored in a CodePipeline-managed S3 bucket.

ECR source: Triggers when a new image is pushed to a specified ECR repository — useful for pipelines that deploy pre-built container images.
Build actions
Provider: CodeBuild, Jenkins.

Receives the source artifact, runs commands (compile, test, package), produces output artifacts. The buildspec.yml file (committed in the repo or defined inline) tells CodeBuild what to run.

Multiple build actions in the same stage run in parallel — useful for running unit tests and integration tests simultaneously to reduce total pipeline time.
Deploy actions
Provider: CodeDeploy, ECS, CloudFormation, Elastic Beanstalk, S3.

Takes the build artifact and deploys it to the target environment. For ECS: updates the task definition and triggers a service update. For EC2: CodeDeploy coordinates the rolling deployment across instances.

Multiple deploy actions with different run orders: deploy to staging (run order 1), then deploy to production (run order 2) — sequential environment promotion.
Approval actions
No provider — just a manual gate. The pipeline pauses until someone approves or rejects in the console, via CLI, or via a custom Lambda.

Use for: Production deployments that require human sign-off. Pre-deployment validation window. Compliance checkpoints in regulated environments (your JP Morgan project).

Auto-reject after configurable timeout (1 day default).

Artifacts are the outputs of one stage that become the inputs of the next. CodePipeline stores artifacts in an S3 bucket it creates automatically (or one you specify). Artifacts are versioned — each pipeline execution has its own artifact set, identified by the execution ID.

Source artifact: The source stage downloads your code and packages it as a ZIP file. This ZIP is the artifact passed to the Build stage. The ZIP contains your entire repository content.

Build artifact: The Build stage (CodeBuild) compiles your code, runs tests, packages the JAR or Docker image, and uploads an output artifact (the JAR, a Docker tag, or a deployment specification) to the S3 artifact bucket.

Why artifacts are stored in S3, not passed directly between stages CodePipeline stages can run in different accounts or regions. Storing artifacts in S3 with cross-account access policies allows the pipeline to span account boundaries — a source in a developer account, a build in a build account, a deployment into a production account. Even for single-account pipelines, S3-based artifact storage provides: immutability (the build artifact for execution #47 is always available for audit or re-deployment), parallelism (multiple stages can read the same artifact simultaneously), and durability (pipeline failure doesn't lose the artifact). The artifact bucket has a KMS encryption key — ensure your CodeBuild role has kms:GenerateDataKey and kms:Decrypt on that key, or artifact operations fail silently.
The artifact that's too large and silently fails CodePipeline has a 3GB limit per artifact. A build artifact containing a fat JAR (200MB) plus all test reports, coverage reports, and generated documentation can exceed this if you're not careful. The build stage completes successfully but the artifact upload to S3 fails silently — downstream stages receive an empty or partial artifact. The deploy stage then fails with a confusing error about missing files. Always inspect what you're actually packaging as the output artifact. In your buildspec.yml, the artifacts: files: section controls what's included. List exactly what downstream stages need — not "**/*" which includes everything.

Approval actions are the human-in-the-loop gates in a pipeline. They're particularly important in regulated environments like financial services where deployment to production requires sign-off from a release manager or security reviewer.

CLI — approve or reject a pipeline action
# The pipeline pauses at the approval action and sends an SNS notification
# The notification contains: pipeline name, stage name, action name, token

# Approve: pipeline continues to the next stage
aws codepipeline put-approval-result \
  --pipeline-name getwork-pipeline \
  --stage-name production-deploy \
  --action-name manual-approval \
  --result summary="Reviewed deployment plan - approved" \
  --token "YOUR_APPROVAL_TOKEN" \   # Token from the SNS notification
  --region ap-south-1

# Reject: pipeline stops, production is not touched
aws codepipeline put-approval-result \
  --pipeline-name getwork-pipeline \
  --stage-name production-deploy \
  --action-name manual-approval \
  --result summary="Regression detected in staging - rejecting" \
  --token "YOUR_APPROVAL_TOKEN" \
  --status Rejected \
  --region ap-south-1
Automating approval with Lambda — the right way for most teams Pure manual approvals slow down deployments and create bottlenecks when the release manager is unavailable. A better pattern: the approval action sends an SNS notification to a Lambda function. The Lambda runs automated validation (smoke tests against staging, error rate check in CloudWatch, security scan results). If all checks pass, Lambda auto-approves via the CodePipeline API. If any check fails, Lambda rejects and sends a detailed alert to Slack. Humans only intervene when the automated checks fail — turning approval into exception-based oversight rather than routine gate-keeping.
CodeBuild — running your build in the cloud
CodeBuild is a managed build service that compiles your code, runs tests, and produces artifacts — in a fresh, isolated environment for every build. Understanding the buildspec, environment configuration, caching, and the VPC integration explains every build failure you'll encounter.

The buildspec.yml file is the complete specification for what CodeBuild does during a build. It is committed into the repository and defines phases, environment variables, artifacts, and caching. Every phase runs sequentially; a non-zero exit code in any phase fails the build.

buildspec.yml — production Spring Boot pipeline
version: 0.2

env:
  variables:
    JAVA_HOME: "/usr/lib/jvm/java-17-amazon-corretto"
    MAVEN_OPTS: "-Xmx2g -XX:+TieredCompilation -XX:TieredStopAtLevel=1"
  parameter-store:           # Fetch from SSM at build start (not secrets)
    APP_BUCKET: "/getwork/prod/app-bucket"
    ECR_REPO: "/getwork/prod/ecr-repo"
  secrets-manager:           # Fetch secrets — never echo these to logs
    SONAR_TOKEN: "/getwork/build/sonar-token:value"

phases:
  install:
    runtime-versions:
      java: corretto17
    commands:
      - echo "Build started on $(date)"
      - java -version
      - mvn -version

  pre_build:
    commands:
      # Authenticate to ECR before build (needed for Docker layer cache pulls)
      - aws ecr get-login-password --region $AWS_DEFAULT_REGION |
          docker login --username AWS --password-stdin $ECR_REPO
      # Validate IAM policies if any changed
      - |
        for f in $(git diff --name-only HEAD~1 | grep '.json$' | grep 'policy'); do
          echo "Validating policy: $f"
          aws accessanalyzer validate-policy \
            --policy-document file://$f \
            --policy-type IDENTITY_POLICY \
            --query 'findings[?findingType==`ERROR`]' \
            --output text | grep -q . && exit 1 || true
        done

  build:
    commands:
      # Run tests with coverage
      - mvn test jacoco:report -B
      # Fail if coverage below 70%
      - mvn jacoco:check -B
      # Package (skip tests — already ran above)
      - mvn package -DskipTests -B
      # Build Docker image with build number tag
      - IMAGE_TAG=$CODEBUILD_BUILD_NUMBER
      - docker build -t $ECR_REPO:build-$IMAGE_TAG -t $ECR_REPO:latest .
      # Run container security scan
      - docker push $ECR_REPO:build-$IMAGE_TAG
      # Wait for scan and fail on CRITICAL CVEs
      - |
        for i in $(seq 1 20); do
          STATUS=$(aws ecr describe-image-scan-findings \
            --repository-name getwork-api \
            --image-id imageTag=build-$IMAGE_TAG \
            --query 'imageScanStatus.status' --output text 2>/dev/null)
          [ "$STATUS" = "COMPLETE" ] && break; sleep 10
        done
        CRITICAL=$(aws ecr describe-image-scan-findings \
          --repository-name getwork-api \
          --image-id imageTag=build-$IMAGE_TAG \
          --query 'imageScanFindings.findingSeverityCounts.CRITICAL // `0`' \
          --output text)
        [ "$CRITICAL" -gt 0 ] && echo "CRITICAL CVEs found: $CRITICAL" && exit 1

  post_build:
    commands:
      # Tag as latest only after passing all checks
      - docker push $ECR_REPO:latest
      # Write the image tag for downstream deployment stage
      - printf '{"imageTag":"%s"}' $IMAGE_TAG > imagedefinitions.json
      - echo "Build $IMAGE_TAG completed successfully"

artifacts:
  files:
    - imagedefinitions.json    # Tells ECS deploy which image to deploy
    - appspec.yml             # CodeDeploy deployment specification
    - scripts/**/*            # Lifecycle hook scripts
  discard-paths: no

cache:
  paths:
    - '/root/.m2/**/*'    # Maven dependency cache
The build that prints your secrets to the build log Any echo or set -x (xtrace) in a phase that uses environment variables can print secret values to the build log. CodeBuild build logs are stored in CloudWatch Logs and are visible to anyone with codebuild:GetBuildLog or cloudwatch:GetLogEvents. Never echo environment variables set from secrets-manager or parameter-store. Avoid set -x in phases that execute commands using secrets. CloudWatch Logs can also be disabled per-build project to prevent log storage of sensitive build output — consider this for builds that handle credentials.

Each CodeBuild build runs in a fresh, isolated Docker container. The environment is defined by the compute type (CPU and memory) and the build image (the Docker image used as the build environment).

AWS-managed images
Pre-built by AWS with common tools. Available for Amazon Linux 2023, Ubuntu.

aws/codebuild/amazonlinux2-x86_64-standard:5.0 — latest AL2023 with Java 17, Docker, Node 18, Python 3.11, Go 1.21.

Use runtime-versions in buildspec to select specific tool versions within the image. No need to apt-get install common tools.
Custom build images
Your own Docker image from ECR. Used when: proprietary tools, specific tool versions not in managed images, faster builds via pre-installed dependencies.

A custom image with Maven dependencies pre-loaded cuts build time by 2-3 minutes for large projects — the dependency download phase is eliminated because the image already contains them.

Must be in ECR (private) or Docker Hub (public). CodeBuild pulls the image before the build starts.
BUILD_GENERAL1_SMALL
3 GB RAM, 2 vCPUs. $0.005/min. Sufficient for unit tests on small Spring Boot projects.
BUILD_GENERAL1_MEDIUM
7 GB RAM, 4 vCPUs. $0.010/min. Recommended for most Spring Boot builds with Docker. Enough RAM to run the JVM, Maven, and Docker simultaneously.
Docker in Docker — why CodeBuild needs privileged mode Building Docker images inside a CodeBuild environment (Docker in Docker) requires the CodeBuild project to run in privileged mode. This gives the build container access to the host Docker daemon. Privileged mode is a security consideration — the build container has elevated access. Mitigate by: using AWS-managed images (not arbitrary Docker Hub images) for builds, restricting the CodeBuild service role to exactly the permissions it needs, enabling CloudTrail logging for CodeBuild API calls.

By default, CodeBuild starts with a clean environment every build — Maven downloads all 200+ dependencies from the internet each time. For a large Spring Boot project, this can take 3-4 minutes per build. Caching stores the Maven local repository between builds, reducing download time to near-zero on cache hits.

CodeBuild — cache configuration options
# Option 1: S3 cache (buildspec.yml cache section)
# CodeBuild uploads /root/.m2 to S3 after build, downloads before next build
cache:
  paths:
    - '/root/.m2/**/*'         # Maven dependencies
    - '/root/.gradle/caches/**/*' # Gradle cache (if using Gradle)
    - '/root/.npm/**/*'         # npm cache (if building frontend)

# Configure S3 cache in CodeBuild project (not buildspec):
aws codebuild update-project \
  --name getwork-build \
  --cache type=S3,location=getwork-build-cache/maven \
  --region ap-south-1

# Option 2: Docker layer cache (for Docker build performance)
# Pull the previous image before building to populate Docker's layer cache
pre_build:
  commands:
    - docker pull $ECR_REPO:latest || true   # || true: don't fail if no image yet
build:
  commands:
    - docker build --cache-from $ECR_REPO:latest -t $ECR_REPO:build-$IMAGE_TAG .
    # --cache-from: Docker reuses layers from previous build if they haven't changed
    # Application layer (2MB) always rebuilds. Dependency layer (200MB) is cached.
Caching impact: 8-minute build → 90-second build A typical large Spring Boot project cold build: 3min dependency download + 2min compilation + 2min tests + 1min Docker build = 8 minutes. With Maven S3 cache (cache hit): 0min dependency download + 2min compilation + 2min tests + 30sec Docker build (layer cache) = 4.5 minutes. With custom build image (deps pre-baked): 0min deps + 1.5min compile + 90sec tests + 20sec Docker = ~3.5 minutes. Total cache savings: ~55% reduction in build time = 55% reduction in CodeBuild costs. For a team doing 50 builds per day, this matters significantly.

By default, CodeBuild runs outside your VPC — in AWS's managed infrastructure. For builds that need to access private resources (internal test databases, private API endpoints, ElastiCache for integration tests), you must configure VPC access.

When VPC access is enabled, CodeBuild attaches an ENI to your VPC in the specified subnets with the specified Security Groups. The build environment can reach private resources. The Security Group for the CodeBuild ENI must be allowed by the target resource's Security Group (same chaining pattern as Days 4 and 8).

VPC-enabled CodeBuild that can't reach the internet You enable VPC access for CodeBuild to access your private RDS for integration tests. The build now runs in your private subnet — which has no internet access (private subnet, no NAT Gateway in the build subnet). The build fails immediately trying to download Maven dependencies from the internet. Two solutions: (1) configure VPC access in a subnet with NAT Gateway access, or (2) use private subnets for CodeBuild but configure an S3 Gateway Endpoint and CodeArtifact (AWS's Maven proxy) for dependency resolution — fully private build with no internet required. Solution 2 is the correct enterprise pattern for build pipelines that must not access the internet.
CodeDeploy — managing application deployments
CodeDeploy automates application deployments to EC2, Lambda, and ECS. It handles the rollout mechanics — in-place, blue-green, canary — and the lifecycle hooks that give you control over every step of the deployment process.

CodeDeploy has three key concepts: Application (a named group of deployments — one per service), Deployment Group (defines where to deploy — which EC2 instances or ECS service — and how — in-place or blue-green), and Deployment (a single deployment execution — push revision X to the deployment group).

The revision is the thing being deployed — for EC2: a ZIP file on S3 containing your JAR and an appspec.yml. For ECS: a task definition JSON or an ECS task definition ARN. For Lambda: a new function version.

In-place deployment (EC2)
CodeDeploy stops the application on each instance, installs the new revision, restarts the application — in rolling batches.

Rollback: Re-deploy the previous revision. Not instant — requires another deployment.

Use when: Stateful applications where you can't have multiple versions simultaneously, or when cost of parallel infrastructure isn't justified.
Blue-green deployment (EC2 or ECS)
New (green) instances are launched with the new revision. Traffic is shifted from blue to green via ALB rule change. Blue remains available for instant rollback.

Rollback: Shift traffic back to blue — takes seconds. No re-deployment needed.

Use when: Zero-downtime requirement, production services where rollback speed is critical.
CodeDeploy's relationship to what you already know The ASG Instance Refresh from Day 8 and CodeDeploy rolling deployment are different mechanisms for the same goal. Instance Refresh is native to ASG and simpler to configure. CodeDeploy provides lifecycle hooks (BeforeInstall, AfterInstall, ApplicationStart, ValidateService) that give finer-grained control over what happens before and after the new version starts — without requiring Lambda or EventBridge. For teams already using CodePipeline, CodeDeploy is the natural deploy stage. For teams using Bitbucket Pipelines, Instance Refresh via CLI is simpler and doesn't require CodeDeploy setup.

The appspec.yml file defines how CodeDeploy installs the revision and what lifecycle hook scripts to run at each stage of the deployment. For EC2 deployments, it lives in the root of your deployment bundle alongside the application files.

appspec.yml — EC2 Spring Boot deployment
version: 0.0
os: linux

files:
  - source: /app.jar              # Source in the deployment bundle
    destination: /opt/app         # Destination on the EC2 instance
  - source: /config/
    destination: /opt/app/config

permissions:
  - object: /opt/app/app.jar
    owner: appuser
    group: appgroup
    mode: "644"

hooks:
  BeforeInstall:              # Before CodeDeploy copies files
    - location: scripts/stop_server.sh
      timeout: 60
      runas: root

  AfterInstall:              # After files copied, before app starts
    - location: scripts/install_dependencies.sh
      timeout: 120

  ApplicationStart:          # Start the application
    - location: scripts/start_server.sh
      timeout: 180
      runas: root

  ValidateService:           # Health check after start — fail = rollback
    - location: scripts/validate_service.sh
      timeout: 120

---
# scripts/validate_service.sh
#!/bin/bash
set -e
for i in $(seq 1 12); do
  if curl -sf http://localhost:8080/actuator/health/readiness > /dev/null; then
    echo "Application healthy"
    exit 0
  fi
  sleep 10
done
echo "Validation failed — triggering rollback"
exit 1
The ValidateService script that always exits 0 A developer writes a ValidateService script that runs a health check but always exits 0 (success) regardless of the result. CodeDeploy considers the deployment successful. The application is actually down — the health check silently failed. Every deployment goes green in the console while production is broken. Always use set -e at the top of lifecycle scripts and return non-zero exit codes for actual failures. CodeDeploy only triggers an automatic rollback when a lifecycle script exits with a non-zero code. A script that exits 0 unconditionally defeats the entire purpose of the ValidateService hook.

For EC2 in-place deployments, CodeDeploy needs the CodeDeploy agent installed on every EC2 instance. The agent polls CodeDeploy for new deployments, downloads the revision from S3, and executes the appspec.yml lifecycle hooks locally.

Deployment configurations — HalfAtATime, OneAtATime, AllAtOnce CodeDeploy provides three built-in deployment configurations for EC2 in-place. AllAtOnce: deploys to all instances simultaneously — fastest but causes downtime if the deployment fails. HalfAtATime: deploys to 50% of instances at a time — the remaining 50% continues serving traffic. OneAtATime: deploys to one instance at a time — slowest but safest; only one instance is at risk at any given time. For production: HalfAtATime balances speed and availability. If you have 10 instances, 5 are always serving traffic during deployment. Custom configurations allow you to specify exact percentages.

For ECS deployments, CodeDeploy integrates with the ALB to perform blue-green deployments at the container level. This is different from the ECS native rolling deployment (from Day 12) — CodeDeploy gives finer-grained traffic shifting control.

appspec.yml — ECS deployment (CodeDeploy)
version: 0.0
Resources:
  - TargetService:
      Type: AWS::ECS::Service
      Properties:
        TaskDefinition: <TASK_DEFINITION>   # Replaced by CodePipeline
        LoadBalancerInfo:
          ContainerName: getwork-api
          ContainerPort: 8080
        PlatformVersion: LATEST

Hooks:
  # Lambda function invoked at each hook point
  - BeforeAllowTraffic: arn:aws:lambda:...:function:ValidateGreenDeployment
  - AfterAllowTraffic: arn:aws:lambda:...:function:CleanupBlueDeployment

---
# Traffic shifting configurations for ECS CodeDeploy:
# CodeDeployDefault.ECSAllAtOnce — shift 100% traffic immediately
# CodeDeployDefault.ECSLinear10PercentEvery1Minutes — 10% more per minute
# CodeDeployDefault.ECSLinear10PercentEvery3Minutes — 10% more per 3 minutes
# CodeDeployDefault.ECSCanary10Percent5Minutes — 10% for 5 min, then 100%
#
# The linear and canary options are production-correct: traffic shifts
# gradually, with CloudWatch alarms able to trigger rollback at any point
ECSCanary10Percent5Minutes — the deployment config for production APIs This shifts 10% of traffic to the new (green) ECS task set for 5 minutes. If CloudWatch alarms (configured in the deployment group) trigger during those 5 minutes, CodeDeploy automatically rolls back by shifting 100% traffic back to blue — taking seconds. If alarms stay OK after 5 minutes, CodeDeploy shifts the remaining 90% and completes the deployment. This is the safest ECS deployment option: small blast radius (only 10% of users see the new version), fast rollback (seconds), and automatic gate (alarm-triggered rollback without human action).
Deployment strategies — choosing the right risk profile
Every deployment strategy is a trade-off between deployment speed, rollback speed, blast radius, and cost. Senior engineers are expected to choose the right strategy for each type of change — not to use the same strategy for every deployment.

The relationship between change risk and deployment strategy:

Low risk
Rolling
Dependency updates, config changes, minor features
Medium risk
Canary
API changes, new features, performance-sensitive updates
High risk
Blue-Green
Breaking changes, schema migrations, major version bumps
Rolling
Mechanism: Replace instances/tasks in batches, maintaining minimum healthy capacity throughout.

Rollback time: Slow — requires another full deployment cycle.

Blast radius: Grows as rollout progresses — 50% of traffic hits new version after half the fleet is replaced.

Cost overhead: Zero — no additional infrastructure.

Use when: Low-risk routine deployments (dependency updates, minor feature flags, configuration changes).
Canary
Mechanism: Route small % (5-10%) to new version. Monitor for N minutes. Progressive increase if healthy.

Rollback time: Fast — shift weight back to 0% within seconds.

Blast radius: Small — only 10% of users see the new version during validation.

Cost overhead: Small — canary fleet is a fraction of production fleet.

Use when: New API behaviour, changes affecting user experience, A/B testing.
Blue-Green
Mechanism: Full parallel fleet for new version. Instant traffic cutover. Old fleet held for rollback.

Rollback time: Instant — switch ALB target group or DNS back to blue.

Blast radius: Full — once cutover, 100% of traffic is on new version.

Cost overhead: Large — double fleet cost during validation window.

Use when: High-risk releases, database migrations, need instant rollback capability.
The version compatibility window — valid for ALL strategies Any strategy that doesn't do an instant, simultaneous swap of all instances (i.e. rolling and canary) creates a window where two versions run simultaneously. During this window, both versions share: the same RDS database, the same Redis cache, the same SQS queues. V2 must be able to read data written by V1 and write data that V1 can read. This means database migrations during rolling/canary deployments must be additive (add columns, don't remove or rename them) until V1 is completely gone. This constraint is not optional — it's the fundamental requirement for zero-downtime deployments with a shared database.
Rollback — making it fast, reliable, and tested
A rollback that takes 20 minutes isn't a rollback — it's a second deployment in the wrong direction. The goal is to design your deployment pipeline such that rollback is as fast and reliable as a deployment forward, with as little human decision-making required under pressure.

There are four rollback approaches, ordered from fastest to slowest:

1
Traffic shift rollback — seconds
Blue-green or canary: shift ALB weights back to 0% green. The old version (blue) never stopped. No deployment, no image pull, no startup time. The fastest possible rollback — but only available when the old version infrastructure is still running.
2
Task/service rollback — 2-3 minutes
Update ECS service to previous task definition revision, or trigger ASG Instance Refresh with previous Launch Template version. The container image is already in ECR or the AMI is already baked — rollback is as fast as a new deployment forward. This is the correct EC2/ECS rollback mechanism.
# ECS rollback — point service at previous task definition PREV_REVISION=$((CURRENT_REVISION - 1)) aws ecs update-service \ --cluster getwork-cluster \ --service getwork-api \ --task-definition getwork-api:$PREV_REVISION \ --force-new-deployment \ --region ap-south-1 # EC2 ASG rollback — Instance Refresh with previous LT version aws autoscaling start-instance-refresh \ --auto-scaling-group-name getwork-asg \ --desired-configuration "LaunchTemplate={LaunchTemplateId=$LT_ID,Version=$PREV_VERSION}" \ --region ap-south-1
3
Automated rollback — triggered by alarms
CodeDeploy deployment groups can be configured to automatically roll back when specific CloudWatch alarms fire. The rollback starts within minutes of the alarm triggering — no human action required. This is the production pattern: deploy, watch your error rate alarm, let the system roll back automatically if the alarm fires.
# Configure automatic rollback on alarm in CodeDeploy deployment group aws deploy update-deployment-group \ --application-name getwork-app \ --current-deployment-group-name getwork-prod \ --auto-rollback-configuration \ enabled=true,events=DEPLOYMENT_FAILURE,DEPLOYMENT_STOP_ON_ALARM \ --alarm-configuration \ enabled=true,alarms=[{name=getwork-api-5xx-rate-high}] \ --region ap-south-1
4
Git revert + redeploy — slowest
Revert the commit in git, push to main, let the pipeline re-run. Total time: pipeline execution time (8-15 minutes). This is the worst rollback option — avoid it. It's only necessary when the rollback itself introduces a new bug that wasn't in the previous version (rare). Using this as the primary rollback strategy is a sign of a pipeline that doesn't preserve immutable deployment artifacts.
The rollback you never test that fails when you need it most At 2AM during an incident, under pressure, is the wrong time to discover that your rollback procedure doesn't work. Make rollback testing part of your regular deployment practice: once per sprint, deliberately deploy a "broken" version (a version that fails health checks), verify the automatic rollback triggers and completes within your RTO target, and document the result. Teams that test rollback regularly roll back in 3 minutes at 2AM. Teams that don't test rollback spend 45 minutes figuring out why the rollback process they thought they had doesn't work.

Repository variables: any repo-specific non-sensitive values.
Environment-specific variables (marked secured): any sensitive overrides per environment.
Never store: AWS credentials (use OIDC instead), database passwords, API keys — these go in Secrets Manager and are fetched at runtime by the application, not injected into the pipeline.
OIDC credential exchange — eliminating static AWS keys from CI/CD
Storing AWS_ACCESS_KEY_ID and AWS_SECRET_ACCESS_KEY in your CI/CD system is a critical security risk. OIDC (OpenID Connect) eliminates this: the CI/CD system authenticates with its own identity provider and exchanges a short-lived token for temporary AWS credentials. No stored credentials. No rotation burden. No credential leak risk.

The OIDC flow for Bitbucket Pipelines (same concept applies to GitHub Actions, GitLab CI, and any OIDC-capable CI system):

1
Configure the OIDC Identity Provider in AWS IAM
Register Bitbucket's OIDC provider URL with AWS IAM. This tells AWS to trust tokens issued by Bitbucket as valid identities.
# One-time setup — create the OIDC provider in AWS IAM aws iam create-open-id-connect-provider \ --url https://api.bitbucket.org/2.0/workspaces/YOUR_WORKSPACE/pipelines-config/identity/oidc \ --client-id-list ari:cloud:bitbucket::workspace/YOUR_WORKSPACE_ID \ --thumbprint-list "a031c46782e6e6c662c2c87c76da9aa62ccabd8e" \ --region ap-south-1 # Get the provider ARN (needed for the trust policy) aws iam list-open-id-connect-providers \ --query 'OpenIDConnectProviderList[*].Arn' --output text

The trust policy specifies which Bitbucket workspace and repository is allowed to assume this role. This prevents any other Bitbucket repository from using this role.
OIDC_PROVIDER_ARN="arn:aws:iam::ACCOUNT:oidc-provider/api.bitbucket.org/..." WORKSPACE_ID="your-bitbucket-workspace-uuid" REPO_UUID="your-repo-uuid" # Trust policy — only your specific repository can assume this role cat > deploy-trust-policy.json << POLICY { "Version": "2012-10-17", "Statement": [{ "Effect": "Allow", "Principal": {"Federated": "$OIDC_PROVIDER_ARN"}, "Action": "sts:AssumeRoleWithWebIdentity", "Condition": { "StringEquals": { "api.bitbucket.org/2.0/workspaces/$WORKSPACE_ID/pipelines-config/identity/oidc:aud": "ari:cloud:bitbucket::workspace/$WORKSPACE_ID" }, "StringLike": { "api.bitbucket.org/2.0/workspaces/$WORKSPACE_ID/pipelines-config/identity/oidc:sub": "{$REPO_UUID}:*" } } }] } POLICY aws iam create-role \ --role-name getwork-bitbucket-deploy \ --assume-role-policy-document file://deploy-trust-policy.json # Attach only the permissions needed for deployment # ecr:* (push images), ecs:* (update services), codedeploy:* (create deployments) aws iam attach-role-policy \ --role-name getwork-bitbucket-deploy \ --policy-arn arn:aws:iam::aws:policy/AmazonECS_FullAccess # Scope down in production
3
Exchange OIDC token for AWS credentials in the pipeline
In the pipeline step, Bitbucket injects the OIDC token. The AWS SDK reads it via the AWS_WEB_IDENTITY_TOKEN_FILE environment variable and automatically exchanges it for temporary credentials.
# In bitbucket-pipelines.yml step (oidc: true must be set) - export AWS_ROLE_ARN="arn:aws:iam::ACCOUNT:role/getwork-bitbucket-deploy" - export AWS_WEB_IDENTITY_TOKEN_FILE=$(mktemp) - echo "$BITBUCKET_STEP_OIDC_TOKEN" > $AWS_WEB_IDENTITY_TOKEN_FILE - export AWS_DEFAULT_REGION=ap-south-1 # The AWS SDK now automatically: # 1. Reads the OIDC token from $AWS_WEB_IDENTITY_TOKEN_FILE # 2. Calls sts:AssumeRoleWithWebIdentity # 3. Returns temporary credentials (ASIA... prefix) # 4. Uses those credentials for all subsequent AWS CLI/SDK calls # Verify it works aws sts get-caller-identity # Returns: role/getwork-bitbucket-deploy — not a user, not AKIA keys
The temporary credentials expire when the pipeline step completes. Even if a log file or artifact were to expose the credentials, they'd be worthless minutes later. This is the key security advantage over stored AKIA keys, which remain valid indefinitely until manually rotated.
Scope down the deploy role aggressively The deploy role should have exactly what the pipeline needs: ECR permissions to push images, ECS permissions to update task definitions and services, optionally CodeDeploy permissions. It should NOT have: IAM permissions (can't create new roles), RDS permissions (can't modify databases), S3 Delete on production data buckets. Run the pipeline once with AdministratorAccess to see what it needs, then replace with a minimal policy. AWS Access Advisor shows which services the role actually used — a perfect basis for scoping down.

The jq task definition update pattern — what it does and why The pipeline reads the current task definition JSON, updates only the image field using jq, strips AWS-generated fields that aren't valid on register (taskDefinitionArn, revision, status, etc.), and registers a new revision. This approach preserves all other task definition fields (resource limits, environment variables, secrets, health check, log configuration) and only changes the image tag. An alternative — hardcoding the entire task definition JSON in the pipeline — means any change to task definition configuration requires a pipeline code change, not just a task definition update. The jq approach is more maintainable.
Interview prep
CI/CD interview questions test whether you understand deployment pipelines as systems — with failure modes, security boundaries, and the trade-offs between speed, safety, and cost.
Q1 A deployment just broke production. Error rate spiked to 15%. Walk me through exactly what you do in the next 10 minutes. Hard ▼

The first 10 minutes of a production incident are about containment — stopping the bleeding before investigation. Don't investigate while users are impacted if you can stop the impact first.

Minute 0–2 — Rollback immediately, investigate later. If you have a known working previous version and the deployment just happened, roll back now. Don't wait to understand the cause. aws ecs update-service --cluster getwork-cluster --service getwork-api --task-definition getwork-api:PREV_REVISION --force-new-deployment. For ASG: trigger Instance Refresh with previous Launch Template version. The rollback is the fastest path to stopping user impact.

Minute 2–5 — Verify rollback is working. Watch ECS services-stable or Instance Refresh status. Monitor ALB 5xx count in CloudWatch — it should drop within 2-3 minutes as new tasks with the old version start receiving traffic and the failing new tasks are drained.

Minute 5–8 — Confirm recovery. Check ALB HealthyHostCount and 5xx rate. When 5xx drops below 1%, the rollback has taken effect. Post to incident channel: "Rolled back to build N. Service recovered. Error rate now [X]%. Investigating root cause."

Minute 8–10 — Begin investigation with preserved evidence. The failed tasks are stopped but CloudWatch Logs has their output. Query the log stream for the failed tasks: aws logs tail /getwork/ecs/api --since 30m --filter-pattern "ERROR". Look for the first error that appeared after the deployment. Was it a startup failure (database connection, Flyway migration error)? An NullPointerException in new code? An environment variable missing from the task definition?

After recovery — don't deploy again until you understand the failure. The failed build tag remains in ECR. After the postmortem, deploy a fixed version as a new build, not a re-run of the failed build.

Why this works: prioritises user impact reduction over investigation. Gives the exact commands for rollback. Describes the observable signals for confirming recovery. The "investigate after rollback" discipline is what separates experienced incident responders from engineers who investigate while users are impacted.
Q2 What is OIDC authentication for CI/CD and why is it better than storing AWS access keys in pipeline secrets? Medium ▼

OIDC (OpenID Connect) allows a CI/CD system to authenticate using its own identity provider and exchange a short-lived token for temporary AWS credentials. No AWS access keys are stored anywhere.

The flow: Bitbucket authenticates the pipeline step with its own OIDC provider and issues a signed JWT (the OIDC token) identifying the workspace, repository, and branch. The pipeline exchanges this token for temporary AWS credentials by calling sts:AssumeRoleWithWebIdentity — the same mechanism used by EC2 instance profiles, Lambda execution roles, and EKS IRSA. The temporary credentials (ASIA prefix) expire when the pipeline step completes — typically within minutes.

Why OIDC is better than stored keys: Stored AKIA access keys are long-lived — they remain valid until explicitly rotated. If a key is exposed (in a log file, in a pipeline artifact, through a CI system compromise), the attacker has permanent access until you discover and rotate the key. OIDC credentials expire automatically — a leaked OIDC-derived credential is worthless minutes after it was generated. There's nothing to rotate because there are no static credentials. The IAM role trust policy additionally restricts which specific Bitbucket workspace and repository can assume the role — even with a valid OIDC token from a different repository, an attacker cannot assume your deploy role.

At Deloitte with JP Morgan: Enterprise security policies often prohibit storing cloud credentials in CI/CD systems due to the risk of credential exposure through pipeline logs, build artifacts, or system compromise. OIDC satisfies this requirement: the pipeline never has long-lived credentials, the credentials are scoped to the pipeline execution, and the trust boundary is cryptographically enforced rather than relying on the CI/CD system's secret storage security.

Why this works: explains the mechanism (not just "it's more secure"), the specific risk mitigated (long-lived key exposure), and connects it to the enterprise context you'll be asked about at Deloitte/JP Morgan interviews.
Q3 When would you choose a canary deployment over blue-green for a new API feature in Spring Boot? Medium ▼

The choice depends on what you want to validate and what the blast radius of a failure should be.

Choose canary when: The new feature changes user-facing behaviour and you want to validate it with real production traffic before full rollout. You're not confident that your test suite covers all edge cases — real user requests from Mumbai construction workers will exercise paths your tests didn't. You can afford 30-60 minutes for the progressive rollout. The blast radius should stay small — only 10% of users affected if the new version has a bug. Canary also gives you A/B test capability: compare conversion rates or engagement metrics between the 10% on the new version and 90% on the old version.

Choose blue-green when: The change is high-risk (database schema migration, major feature rewrite, third-party integration change) and you need instant rollback capability. You want 100% validation coverage before any real traffic hits the new version — you deploy green, run full regression tests against the green ALB endpoint directly, and only then cut traffic. You can't afford the 30-60 minute canary window because the current version has a critical bug that needs immediate replacement. The cost of running double fleet for 1-2 hours is acceptable.

For the GetWork job creation API refactor: I'd use canary. The change is an API refactor (same functionality, internal restructuring). I want to validate that the refactor doesn't affect real job creation flows. Route 10% of traffic to the new version for 30 minutes. Monitor job creation success rate and latency. If both look identical between old and new traffic, progressively increase. If not, instant rollback — flip the weight back to 0%.

Why this works: The answer isn't a rule ("always use canary for APIs") but a decision framework based on risk, blast radius, and validation requirements. The specific GetWork example with a concrete monitoring signal (job creation success rate) shows you know how to instrument the validation, not just set up the infrastructure.
Q4 How do you ensure a database migration runs safely during a zero-downtime deployment? Hard ▼

Database migrations during zero-downtime deployments require the expand-contract pattern because during any rolling or canary deployment, both V1 and V2 of the application run simultaneously against the same database.

The fundamental constraint: V2 must be able to run against the database schema that V1 wrote, and V1 must be able to run against any database state that V2 produces. This eliminates: renaming columns (V1 can't find the old column name), deleting columns (V1 reads a column that no longer exists), changing column types incompatibly, and adding non-nullable columns without defaults.

The expand-contract pattern in three deployments:

Deployment 1 (expand): Add the new column/table. Keep the old column. V2 writes to both old and new columns. V1 reads from old column (unchanged). Both versions work against the migrated schema. Deploy with rolling or canary — both versions are compatible.

Deployment 2 (migrate data): If necessary, backfill the new column from old column data. This can be a one-off batch job or Flyway migration, not a code deployment. Run during a low-traffic window.

Deployment 3 (contract): Update V2 to read from new column only. Stop writing to old column. Deploy. Old column is now unused.

Cleanup: In a fourth deployment (or out-of-band migration): drop the old column. Now it's gone permanently.

Using Flyway or Liquibase: each migration is additive. Migrations tagged as V2__ run when the V2 application starts. If a rolling deployment runs V1 and V2 simultaneously, V2 applies the migration when its JVM starts. V1 must continue to work against the migrated schema — hence the expand (not replace) approach.

Why this works: names the pattern (expand-contract), explains why simple rename/delete breaks zero-downtime, and gives the three-deployment sequence with what each version does. The Flyway integration point shows you've implemented this, not just described it.

Day 15 cheatsheet
CI/CD production mechanics at a glance.
CodePipeline role
Orchestrator only — doesn't build, test, or deploy. Coordinates CodeBuild, CodeDeploy, Lambda, and manual approvals. $1/active pipeline/month.
Pipeline artifact S3
CodePipeline stores artifacts in S3 between stages. Artifacts are versioned per execution. KMS-encrypted — CodeBuild role needs kms:Decrypt or artifact operations fail silently.
Buildspec phases
install → pre_build → build → post_build. Non-zero exit in any phase fails the build. Never echo secrets to build logs. set -x in phases with secrets prints credentials to CloudWatch Logs.
CodeBuild caching
S3 cache for Maven/Gradle deps: cold 8min → warm 4.5min. Docker layer cache (--cache-from ECR:latest): dep layer reused, only app layer rebuilt. 55%+ build time reduction.
appspec.yml ValidateService
MUST exit non-zero on failure. A script that always exits 0 makes every deployment succeed in CodeDeploy regardless of application health. Use set -e + curl health check + explicit exit 1.
ECSCanary10Percent5Minutes
Route 10% to green for 5 minutes. If CloudWatch alarm fires → auto-rollback to blue in seconds. Production-correct ECS deployment config for APIs where blast radius must be limited.
OIDC = no stored credentials
CI/CD exchanges short-lived OIDC token for temporary AWS creds. Credentials expire when step completes. Trust policy scoped to specific repo. Satisfies enterprise "no static keys in CI" requirement.
Rollback priority order
1. Traffic shift (seconds) — blue-green/canary. 2. Task/service rollback (2-3min) — ECS previous revision. 3. Auto-rollback via alarm. 4. Git revert + redeploy (slowest). Never use 4 as primary rollback.
Expand-contract migrations
Rolling/canary deployments run V1+V2 simultaneously. Database migrations must be additive only: add columns (don't rename/delete). Three-deployment pattern: expand → migrate → contract.
jq task def update
Read current task def → update image field → strip AWS-generated fields → register new revision. Preserves all other configuration. Don't hardcode full task def in pipeline — it drifts.
CodeBuild VPC access
Required to reach private RDS/ElastiCache for integration tests. But VPC = no internet. Need NAT Gateway in build subnet OR CodeArtifact proxy for Maven. Plan network topology before enabling.
Test rollback quarterly
Deliberately deploy a broken version. Verify rollback triggers automatically (or manually) within your RTO target. Document the result. 2AM is the wrong time to discover your rollback procedure is broken.
Production pipeline rules
1Use OIDC for CI/CD AWS authentication. No static AKIA keys in pipeline secrets — ever.
2Scope the deploy IAM role to exactly the permissions the pipeline needs. Run once with broad access, use Access Advisor to scope down.
3Use immutable versioned image tags (build-N, git-SHA). Never deploy :latest in production — rollback requires knowing which exact image was previously running.
4Gate on ECR vulnerability scan before deploying. Fail the pipeline on CRITICAL CVEs. This is a compliance requirement in financial services.
5Always validate after deployment: health check endpoint + error rate check in a 5-minute window. Auto-rollback if error rate exceeds threshold.
6appspec.yml ValidateService scripts must exit non-zero on failure. Test with a deliberately broken version before relying on it in production.
7Enable deployment circuit breaker on ECS services. Automatic rollback when tasks fail to become healthy prevents extended outages from bad deployments.
8Database migrations in rolling/canary deployments: additive only. Expand-contract pattern across three deployments. Never rename or delete columns while old version is still running.
9Run full pipeline on every pull request against a staging environment. Production deployments should be boring because you already validated the same code in staging.
10Test rollback quarterly. A rollback procedure you've never exercised is not a rollback procedure — it's wishful thinking.
