CloudWatch Metrics — the data model
Every number AWS publishes about your infrastructure — CPU, network bytes, request count — flows through CloudWatch Metrics. Understanding how metrics are stored, resolved, and aggregated explains every confusing behaviour you'll encounter when building alarms and dashboards.
data points periods statistics resolution

A CloudWatch metric is identified by three things: a namespace (e.g. AWS/EC2), a metric name (e.g. CPUUtilization), and a set of dimensions (key-value pairs that further identify the resource, e.g. InstanceId=i-abc123). These three together uniquely identify a metric time series.

Data points are individual measurements with a timestamp and value. AWS services publish data points on a schedule — EC2 publishes every 5 minutes by default (detailed monitoring: every 1 minute). Each data point is stored with the exact timestamp of measurement. CloudWatch stores raw data points for 3 hours, then aggregates them into 1-minute rollups (kept 15 days), then 5-minute rollups (kept 63 days), then 1-hour rollups (kept 455 days).

Why the storage model matters for your dashboards When you query a metric over a 2-week period, you must use at least a 5-minute period — the raw 1-second data doesn't exist anymore. When you query a metric from 3 months ago, you can only get 1-hour resolution. If you build a dashboard that queries the last 30 days with a 1-minute period, CloudWatch will silently return no data for the older half of the range (where 1-minute rollups have expired) and show data for the recent half. This looks like a gap in your metrics rather than a resolution mismatch. Always match the query period to the data's available resolution.
The metric that disappears after 15 minutes Standard resolution metrics published by your application (e.g. via PutMetricData) with a timestamp more than 2 hours in the past are rejected by CloudWatch. A batch job that runs for 3 hours and tries to backfill metrics with old timestamps will silently drop any data point with a timestamp older than 2 hours. Always publish metrics with the current timestamp, not the timestamp of when the event occurred.

When you query a metric, you choose a period (the time bucket size) and a statistic (how to aggregate data points within each bucket). These two choices determine what you see.

Statistics
Average — mean of all data points in the period. Use for: CPU utilisation, cache hit rate. Masks spikes if data is noisy.

Sum — total of all data points. Use for: request count, error count, bytes transferred. Tells you "how much total" in that window.

Maximum — highest single value. Use for: latency (you care about the worst case), queue depth peaks.

Minimum — lowest single value. Use for: free memory (minimum free memory over the period is the most concerning).

p99, p95, p50 — percentile statistics. Require extended statistics. Use for: latency (p99 latency tells you what 99% of users experienced).
Period selection guide
Real-time dashboards — 1 minute period. Requires detailed monitoring on EC2 (enabled per instance).

Operational dashboards — 5 minutes. Matches standard EC2 publish frequency. Sufficient for most monitoring.

Trend analysis — 1 hour or 1 day. Long-term patterns, capacity planning, weekly/monthly reports.

Alarms — match period to how quickly you need to react. 1-minute alarm period for latency. 5-minute for CPU. 1-hour for disk usage.
Average CPU is almost always the wrong statistic for an alarm You have 10 EC2 instances. One is at 100% CPU (saturated, users on that instance see timeouts). Nine are at 10% CPU. Average: 19%. Your "average CPU > 70%" alarm doesn't fire. Use Maximum CPU across all instances, or use per-instance alarms. Better yet: use ALBRequestCountPerTarget (from Day 8) which directly measures what users experience, not what instances are doing internally.

Standard resolution metrics are published at 1-minute minimum granularity. Most AWS service metrics are standard resolution. You can publish custom metrics at standard resolution.

High resolution custom metrics can be published at 1-second granularity via PutMetricData with StorageResolution=1. High-resolution metrics cost more ($0.30/metric/month vs $0.10) and alarms on high-resolution metrics can fire in as little as 10 seconds.

When to use high-resolution metrics Payment processing: detect a spike in payment failures within 10 seconds rather than waiting for the 1-minute standard resolution. Trading systems: detect price anomalies or order failures in near-real-time. Health check failover: detect an ALB target going unhealthy in 10 seconds instead of 60. For GetWork and most Spring Boot applications, standard resolution (1-minute) alarms are sufficient. High-resolution adds cost and complexity that is rarely justified outside financial services or real-time systems.

CloudWatch retains metric data at decreasing resolution over time. This is automatic — you don't configure it:

3–15 days1-minute rollupsShort-term trend analysis, incident postmortems

Metrics Streams for long-term retention If you need metric data beyond 15 months (regulatory requirement, multi-year capacity planning), CloudWatch Metrics Streams can continuously stream all metrics to S3 (via Kinesis Data Firehose) in Parquet or JSON format. Query the archived metrics from Athena. This is the correct architecture for organisations with multi-year data retention requirements — not increasing CloudWatch's retention limit, which cannot be extended.
Namespaces and dimensions
Namespaces partition metrics by service. Dimensions are the key-value filters that identify which specific resource a metric belongs to. Mastering dimensions is what allows you to write queries and alarms that target exactly the right resource.

The AWS service metric namespaces you'll use most often in a Spring Boot deployment:

AWS/EC2EC2 instance metricsCPUUtilization, NetworkIn/Out, StatusCheckFailed, EBSReadOps
AWS/ApplicationELBALB metricsRequestCount, HTTPCode_Target_5XX_Count, TargetResponseTime, HealthyHostCount
AWS/AutoScalingASG metricsGroupInServiceInstances, GroupDesiredCapacity, GroupPendingInstances

Dimensions are the key-value pairs that narrow a metric to a specific resource. The same metric name (CPUUtilization) exists in the AWS/EC2 namespace for every EC2 instance in your account. The InstanceId dimension (InstanceId=i-abc123) identifies which specific instance's CPU you're looking at.

Multiple dimension combinations for the same metric represent different aggregation levels. For ALB: RequestCount with dimension LoadBalancer=app/getwork-alb/abc123 gives total ALB request count. Adding TargetGroup=targetgroup/getwork-api-tg/xyz narrows it to a specific target group. Both are valid metrics — different aggregation granularity.

The dimension mismatch that makes your alarm always insufficient You create an alarm on AWS/ApplicationELB HTTPCode_Target_5XX_Count without specifying any dimensions. You expect it to alarm when any target group returns 5xx. But CloudWatch aggregates across all load balancers and all target groups — a low 5xx rate on one target group is diluted by zero 5xx on all other resources. Your alarm doesn't fire even when your API target group has a 10% error rate. Always specify the LoadBalancer and TargetGroup dimensions when alarming on ALB metrics. The metric name alone without dimensions is nearly useless for precise alarming.
Custom metrics — measuring what AWS can't see
Infrastructure metrics (CPU, memory, network) tell you about resources. Custom business metrics (jobs created per minute, payment success rate, active worker sessions) tell you whether your application is actually working. A system with perfect infrastructure metrics can still be completely broken from a user perspective.

The simplest way to publish a custom metric is directly via the AWS SDK's PutMetricData API. This is appropriate for one-off scripts, batch jobs, and Lambda functions.

Java — direct PutMetricData
// Direct SDK call — useful for batch jobs, Lambda, one-off instrumentation
public class MetricsPublisher {
    private final CloudWatchClient cw = CloudWatchClient.builder()
        .region(Region.AP_SOUTH_1)
        .credentialsProvider(DefaultCredentialsProvider.create())
        .build();

    public void publishJobCreated(String city, int count) {
        cw.putMetricData(r -> r
            .namespace("GetWork/API")                 // Your namespace
            .metricData(MetricDatum.builder()
                .metricName("JobsCreated")
                .value((double) count)
                .unit(StandardUnit.COUNT)
                .timestamp(Instant.now())
                .dimensions(                              // Max 30 dimensions per metric
                    Dimension.builder().name("City").value(city).build(),
                    Dimension.builder().name("Environment").value("production").build()
                )
                .storageResolution(1)               // 1=high-res(1s), 60=standard(1min)
                .build())
        );
    }
}
PutMetricData costs — the bill that surprises teams at scale CloudWatch charges $0.01 per 1,000 metric data points published. A Spring Boot service publishing 5 custom metrics every second: 5 × 60 × 60 × 24 = 432,000 data points/day = $4.32/day = ~$130/month for one service. Ten microservices: $1,300/month just for custom metrics. Mitigation: publish every 60 seconds (not every second), batch multiple data points per API call (up to 1,000 per call), and use Embedded Metric Format (EMF) which is free up to the first 10 custom metrics per EMF log event.

For Spring Boot applications, Micrometer is the preferred instrumentation layer. It provides a vendor-neutral API for counters, timers, and gauges, with backends for CloudWatch, Prometheus, Datadog, and others. Adding the micrometer-registry-cloudwatch2 dependency automatically publishes all Spring Boot actuator metrics plus your custom metrics to CloudWatch.

Java — Micrometer with CloudWatch backend
// pom.xml
<dependency>
  <groupId>io.micrometer</groupId>
  <artifactId>micrometer-registry-cloudwatch2</artifactId>
</dependency>

// CloudWatch registry config — publishes every 1 minute
@Bean
public CloudWatchMeterRegistry cloudWatchRegistry() {
    CloudWatchConfig config = key -> switch(key) {
        case "cloudwatch.namespace" -> "GetWork/API";
        case "cloudwatch.step"      -> "PT1M";   // Publish every 60 seconds
        case "cloudwatch.batchSize" -> "20";    // Up to 20 metrics per API call
        default -> null;
    };
    return new CloudWatchMeterRegistry(config, Clock.SYSTEM,
        CloudWatchAsyncClient.builder()
            .region(Region.AP_SOUTH_1).build());
}

// Instrumentation in your service — zero AWS SDK calls in business logic
@Service
public class JobService {
    private final Counter jobsCreated;
    private final Timer jobCreationLatency;
    private final Gauge activeJobCount;

    public JobService(MeterRegistry registry, JobRepository jobRepo) {
        this.jobsCreated = Counter.builder("getwork.jobs.created")
            .tag("env", "production")
            .register(registry);

        this.jobCreationLatency = Timer.builder("getwork.jobs.creation.duration")
            .publishPercentiles(0.5, 0.95, 0.99)
            .register(registry);

        // Gauge: reads the value when Micrometer publishes (not on every call)
        Gauge.builder("getwork.jobs.active.count", jobRepo, JobRepository::countActive)
            .register(registry);
    }

    public Job createJob(CreateJobRequest req) {
        return jobCreationLatency.recordCallable(() -> {
            Job job = jobRepository.save(new Job(req));
            jobsCreated.increment();
            return job;
        });
    }
}
Micrometer publishes to CloudWatch every 60 seconds by default — this is correct The PT1M step means your metrics arrive in CloudWatch once per minute, batched. This keeps PutMetricData costs manageable (60× fewer API calls than per-second publishing) while giving you 1-minute granularity in dashboards and alarms. For 20 custom metrics × 1 publish/minute × 60 min/hr × 24 hr = 28,800 data points/day = $0.29/day per service — affordable at scale.

Embedded Metric Format (EMF) is a way to publish CloudWatch Metrics by embedding structured metric data inside CloudWatch Logs. The advantage: CloudWatch Logs Insights automatically extracts the metrics and creates CloudWatch metric time series — at no additional PutMetricData cost. You pay only for the CloudWatch Logs ingestion and storage, not for metric publishing.

EMF is available as a first-class feature in the CloudWatch agent and as a library (aws-embedded-metrics for Node.js/Python). For Java, you can implement EMF by writing structured JSON logs with the _aws.CloudWatchMetrics key, which CloudWatch Logs automatically processes.

JSON — EMF log format
// A single log line containing EMF-formatted metrics
// CloudWatch Logs extracts this and creates metric data points automatically
{
  "_aws": {
    "Timestamp": 1710513000000,
    "CloudWatchMetrics": [{
      "Namespace": "GetWork/API",
      "Dimensions": [["City", "Environment"]],
      "Metrics": [
        {"Name": "JobsCreated", "Unit": "Count"},
        {"Name": "JobCreationLatency", "Unit": "Milliseconds"}
      ]
    }]
  },
  "City": "Mumbai",
  "Environment": "production",
  "JobsCreated": 5,
  "JobCreationLatency": 142,
  "requestId": "abc-123"     // additional log context — searchable in Insights
}
EMF is the recommended approach for Lambda functions Lambda functions are charged per invocation and duration. Making PutMetricData SDK calls adds latency (15–50ms network call) to every Lambda invocation. EMF avoids this: write a structured JSON log line (microseconds), Lambda's logging mechanism ships it to CloudWatch Logs asynchronously after the function returns, and CloudWatch extracts the metrics. Zero added latency to function duration. Zero PutMetricData API calls. Metrics appear in CloudWatch within 1–2 minutes of log ingestion.
Metric math — deriving new metrics without publishing them
Metric math lets you create derived metrics by combining existing ones with mathematical expressions — directly in CloudWatch dashboards and alarms, without writing any code or publishing additional data points. This is the correct way to build composite signals like error rate percentage or cache miss ratio.

Metric math expressions use a simple syntax where each referenced metric gets a label (m1, m2, etc.) and you write an expression using those labels and built-in functions. The result is a virtual metric that only exists in the context of the dashboard widget or alarm you're building.

CloudWatch Metric Math — essential expressions
# Error rate percentage — 5xx as % of total requests
# m1 = HTTPCode_Target_5XX_Count (Sum, 1 min)
# m2 = RequestCount (Sum, 1 min)
expression: (m1 / m2) * 100
# Alarm: expression > 1 for 3 consecutive minutes = error rate > 1%

# Cache hit rate percentage
# m3 = CacheHits (Sum), m4 = CacheMisses (Sum)
expression: (m3 / (m3 + m4)) * 100
# Target: > 80%

# Sum across multiple regions or target groups
# m5 = RequestCount in ap-south-1, m6 = RequestCount in ap-southeast-1
expression: SUM([m5, m6])

# Running average of last N data points (smoothing noisy metrics)
expression: RUNNING_SUM(m1) / RUNNING_SUM(m2)

# Anomaly detection band (covered in the Anomaly Detection section)
expression: ANOMALY_DETECTION_BAND(m1, 2)
# Creates upper/lower bounds. 2 = 2 standard deviations from expected

# Fill gaps in sparse metrics (avoid sawtooth graphs)
expression: FILL(m1, 0)
# Returns 0 for periods with no data instead of null (missing data points)

# Rate (data points per second from a counter)
# Useful when you have a cumulative counter and want the rate of change
expression: RATE(m1)
Dividing by zero when request count is 0 The error rate expression (m1 / m2) * 100 throws a divide-by-zero when m2 (request count) is 0 — during off-peak hours when no requests are flowing. CloudWatch treats this as an error and the alarm may transition to INSUFFICIENT_DATA state. Protect against it: IF(m2 > 0, (m1 / m2) * 100, 0). Returns 0% error rate when there are no requests, rather than an undefined value that confuses alarms.
CloudWatch Alarms — how they actually work
Alarms are not simple threshold triggers. They have three states, a complex evaluation model with datapoints-to-alarm semantics, and nuanced missing-data handling that determines whether a quiet period produces false alarms or silently hides real problems.

A CloudWatch Alarm has three states: OK (metric is within the threshold), ALARM (metric has breached the threshold), and INSUFFICIENT_DATA (not enough data points to evaluate — occurs at creation or when data stops flowing).

State transitions trigger actions. You can configure separate actions for: entering ALARM state (send a page), exiting ALARM state (send recovery notification), and entering INSUFFICIENT_DATA state (send an alert — this is often skipped but represents a real signal: your monitoring has gone blind).

Alarm evaluation period vs evaluation window The alarm has a period (how often to evaluate — 60 seconds, 300 seconds) and an evaluation period (how many periods to look at before deciding). The threshold condition is checked against each data point in the evaluation window. Combined with the datapoints-to-alarm setting, these three numbers determine how quickly the alarm responds and how sensitive it is to transient spikes.

The datapoints-to-alarm (M out of N) setting is the most important alarm configuration that engineers routinely set incorrectly. It defines how many of the N evaluation periods must breach the threshold before the alarm fires.

1 out of 1 (default)
A single data point above the threshold triggers the alarm immediately.

Problem: Any brief spike (a single slow request, a momentary CPU blip) triggers the alarm. You get paged at 3AM for a 15-second anomaly that self-resolved. Alert fatigue follows.

Use when: Zero tolerance — payment failures, complete service outages. One failure is one too many.
3 out of 5 (production sweet spot)
3 of the last 5 evaluation periods must breach threshold.

Why this works: Transient spikes (1-2 periods) don't trigger the alarm. Sustained problems (3+ consecutive periods) do. The alarm is still responsive (fires within 3 evaluation periods of a real problem starting) but resistant to false positives.

Use for: Latency alarms, CPU alarms, error rate alarms. Most production operational alarms.
The recommended alarm config for GetWork's API error rate Metric: HTTPCode_Target_5XX_Count / RequestCount * 100 (metric math expression). Period: 60 seconds. Evaluation periods: 5. Datapoints to alarm: 3. Threshold: > 1%. This means: the alarm fires when the error rate exceeds 1% in 3 of the last 5 minutes. A single bad minute (deployment, brief DB hiccup) doesn't trigger. 3+ consecutive bad minutes definitely triggers. Recovery requires the expression to be below 1% in 3 of 5 consecutive evaluation periods — so the alarm clears only when the situation is definitively resolved, not just temporarily better.

When a metric has no data points for an evaluation period, CloudWatch must decide how to treat that missing data. The four options have very different implications:

missing = notBreaching
Missing data counts as below the threshold. The alarm stays OK during quiet periods.

Use for: Metrics where no data means nothing is happening — error count (no errors = no data = good). Request count during maintenance windows.
missing = breaching
Missing data counts as above the threshold. The alarm enters ALARM state if data stops.

Use for: Heartbeat monitors — if your application stops publishing metrics, you want to know. Critical health checks where "no data" is itself a sign of failure.
missing = ignore
Missing periods are ignored — the alarm state doesn't change, and the missing period doesn't count toward datapoints-to-alarm.

Use for: Sporadic metrics that are published irregularly. Scheduled batch jobs.
missing = insufficient data
Missing data transitions the alarm to INSUFFICIENT_DATA state.

Default behaviour. Results in frequent INSUFFICIENT_DATA state for sparse metrics. Not usually what you want for operational alarms — it creates noise.
The alarm that fires every night at 2AM for no reason Your error rate alarm uses the default missing data = insufficient_data. At 2AM, traffic drops to near-zero. Some 1-minute evaluation periods have no request count data. The metric math expression returns null. The alarm transitions to INSUFFICIENT_DATA. CloudWatch's INSUFFICIENT_DATA action fires. Your on-call engineer gets paged. The "error" is just low traffic, not an actual problem. Fix: set missing data treatment to notBreaching for rate-based alarms. Missing data on an error rate means "we couldn't calculate the rate" — not "there were errors."

Alarm actions are triggered when the alarm state changes. The primary action type is SNS (Simple Notification Service) — the alarm publishes a message to an SNS topic, which then fans out to subscribers: email, SMS, Lambda, SQS, HTTP endpoints (PagerDuty, Slack webhooks, OpsGenie).

CLI — create a production alarm with SNS action
# Create SNS topic for alerts
ALERT_TOPIC=$(aws sns create-topic --name getwork-alerts \
  --query TopicArn --output text --region ap-south-1)

# Subscribe your PagerDuty/Slack webhook endpoint to the topic
aws sns subscribe --topic-arn $ALERT_TOPIC \
  --protocol https \
  --notification-endpoint https://events.pagerduty.com/integration/YOUR_KEY/enqueue

# Create the alarm — 5xx error rate > 1% for 3/5 minutes
aws cloudwatch put-metric-alarm \
  --alarm-name "getwork-api-5xx-rate" \
  --alarm-description "API 5xx error rate exceeds 1%" \
  --namespace "AWS/ApplicationELB" \
  --metric-name "HTTPCode_Target_5XX_Count" \
  --dimensions Name=LoadBalancer,Value=app/getwork-alb/abc123 \
               Name=TargetGroup,Value=targetgroup/getwork-api-tg/xyz \
  --statistic Sum \
  --period 60 \
  --evaluation-periods 5 \
  --datapoints-to-alarm 3 \
  --threshold 5 \
  --comparison-operator GreaterThanOrEqualToThreshold \
  --treat-missing-data notBreaching \
  --alarm-actions $ALERT_TOPIC \
  --ok-actions $ALERT_TOPIC \
  --region ap-south-1
Alarm as an Auto Scaling trigger — the connection from Day 8 CloudWatch Alarms are also the trigger mechanism for EC2 Auto Scaling step scaling policies. When CPU > 80% alarm fires → ASG adds 2 instances. When CPU < 30% alarm fires for 10 consecutive minutes → ASG removes 1 instance. This is the same alarm infrastructure — you configure --alarm-actions arn:aws:autoscaling:region:account:scalingPolicy/policy-id instead of an SNS ARN. Target tracking scaling (from Day 8) creates and manages these alarms internally on your behalf.
Composite alarms — reducing alert noise with logic
A composite alarm uses boolean logic (AND, OR, NOT) to combine multiple individual alarms into a single alarm that only fires when the combination of conditions is true. This is how you reduce alert fatigue from correlated alarms.

Consider what happens during an RDS failover: the error rate alarm fires, the latency alarm fires, the connection count alarm fires, and the EC2 CPU alarm fires (instances work harder handling errors). You receive 4 pages simultaneously for what is a single root cause event.

A composite alarm solves this: ALARM("api-5xx-rate") AND ALARM("rds-failover-in-progress") → one page with context, rather than four separate pages. The composite alarm fires only when both conditions are true, and only sends one notification.

CLI — composite alarm for correlated failures
# Composite alarm: page only when BOTH error rate is high AND latency is high
# (eliminates false pages when one metric briefly spikes without the other)
aws cloudwatch put-composite-alarm \
  --alarm-name "getwork-api-degraded" \
  --alarm-description "API degraded: high errors AND high latency" \
  --alarm-rule "ALARM(\"getwork-api-5xx-rate\") AND ALARM(\"getwork-api-p99-latency\")" \
  --alarm-actions $ALERT_TOPIC \
  --region ap-south-1

# Composite alarm: page when API is down for any reason
aws cloudwatch put-composite-alarm \
  --alarm-name "getwork-api-down" \
  --alarm-rule "ALARM(\"getwork-alb-healthy-hosts-zero\") OR \
               (ALARM(\"getwork-api-5xx-rate\") AND ALARM(\"getwork-api-no-requests\"))" \
  --alarm-actions $PAGERDUTY_TOPIC \
  --region ap-south-1

# Suppress child alarm notifications — only composite alarm pages
# Set action-suppressor on child alarms pointing to composite alarm
aws cloudwatch put-metric-alarm \
  --alarm-name "getwork-api-5xx-rate" \
  ... # same as before
  --actions-suppressor "getwork-api-degraded" \  # suppress when composite handles it
  --actions-suppressor-wait-period 10 \
  --actions-suppressor-extension-period 30
The three-tier alarm architecture for GetWork Tier 1 (individual alarms, no SNS actions): CPU, memory, disk — these monitor individual resources and feed into composite alarms. They are visible in the console but don't page anyone directly.
Tier 2 (composite alarms, Slack): sustained CPU > 80% AND memory > 80% → post to #ops-alerts Slack. Non-urgent, investigate during business hours.
Tier 3 (composite alarms, PagerDuty): ALB healthy hosts = 0 OR (error rate > 5% AND duration > 5 min) → wake someone up. These represent actual user impact.
CloudWatch Logs — centralised logging architecture
CloudWatch Logs is the centralised logging service for your AWS infrastructure. Understanding the log group, log stream, and retention model — and why log groups cost money you're probably not thinking about — is necessary for running a production system without a surprise bill.

The hierarchy: Log Group → Log Stream → Log Events.

A log group is a container for related log streams. Typically one per application or service component. Retention policy, metric filters, and subscription filters are configured at the log group level. Examples: /getwork/app, /getwork/ecs/api, /aws/lambda/getwork-auth.

A log stream is a sequence of log events from a single source. In ECS: each task instance has its own log stream (prefixed with the task ID). In Lambda: each concurrent execution environment has its own stream. In EC2: each instance has its own stream within the log group.

A log event is a single log entry: a timestamp and a message string. CloudWatch Logs Insights can parse structured (JSON) messages into fields for querying.

CloudWatch Logs pricing — the component of your AWS bill nobody reviews Ingestion: $0.50/GB. Storage: $0.03/GB-month. Filter extraction to metrics: $0.10/GB scanned. For a production Spring Boot app logging at INFO level: roughly 1–5 GB/day = $0.50–$2.50/day in ingestion alone = $15–$75/month just for log ingestion. Multiply by 10 microservices: $150–$750/month. Log retention defaults to Never Expire. Every log event stored since creation accumulates storage charges forever at $0.03/GB/month. After 12 months: $0.03 × 12 months × monthly volume = significant recurring cost. Set retention policies. Use compression-friendly log formats. Filter out DEBUG logs in production. These three actions typically reduce log costs by 60–80%.

Retention policies determine how long CloudWatch keeps log events before automatically deleting them. Without a retention policy, logs are kept forever at $0.03/GB/month.

CLI — set retention on all log groups
# Recommended retention periods by log type
# Application logs (debug/info): short retention
aws logs put-retention-policy \
  --log-group-name /getwork/app \
  --retention-in-days 30 --region ap-south-1

# Application error logs: medium retention
aws logs put-retention-policy \
  --log-group-name /getwork/errors \
  --retention-in-days 90 --region ap-south-1

# CloudTrail security audit logs: long retention (1 year)
aws logs put-retention-policy \
  --log-group-name /aws/cloudtrail/getwork \
  --retention-in-days 365 --region ap-south-1

# Set retention on ALL log groups (run this as a periodic audit)
aws logs describe-log-groups \
  --query 'logGroups[?!retentionInDays].logGroupName' \
  --output text --region ap-south-1 | \
tr '\t' '\n' | while read GROUP; do
  echo "Setting 30-day retention on: $GROUP"
  aws logs put-retention-policy \
    --log-group-name "$GROUP" \
    --retention-in-days 30 --region ap-south-1
done
Enforce retention with AWS Config or Terraform Manually setting retention is error-prone — any new log group created automatically (by Lambda, ECS, etc.) has no retention policy until someone notices. Use an AWS Config rule to detect log groups without retention policies and alert or auto-remediate. In Terraform, every aws_cloudwatch_log_group resource should include retention_in_days — make it a required field in your Terraform module for log groups. This prevents the "accumulated 2 years of CloudWatch Logs storage from Lambda functions nobody knew was running" bill.

Metric filters parse log events in real-time and publish CloudWatch metric data points when the filter pattern matches. This converts log events into metrics without changing your application code — useful for creating alarms based on specific log patterns.

CLI — create metric filters from Spring Boot logs
# Filter pattern: count ERROR-level log events as a metric
aws logs put-metric-filter \
  --log-group-name /getwork/app \
  --filter-name ErrorCount \
  --filter-pattern "{ $.level = \"ERROR\" }" \
  --metric-transformations \
    metricName=ApplicationErrors,metricNamespace=GetWork/App,metricValue=1,unit=Count \
  --region ap-south-1

# Filter for OutOfMemoryError specifically
aws logs put-metric-filter \
  --log-group-name /getwork/app \
  --filter-name OOMErrors \
  --filter-pattern "OutOfMemoryError" \
  --metric-transformations \
    metricName=OOMErrors,metricNamespace=GetWork/App,metricValue=1,unit=Count \
  --region ap-south-1

# Filter for specific exception class in JSON logs
# JSON log format: {"level":"ERROR","exception":"NullPointerException",...}
aws logs put-metric-filter \
  --log-group-name /getwork/app \
  --filter-name NullPointerExceptions \
  --filter-pattern "{ $.exception = \"NullPointerException\" }" \
  --metric-transformations \
    metricName=NullPointerExceptions,metricNamespace=GetWork/App,metricValue=1,unit=Count \
  --region ap-south-1
Metric filters are the bridge between logs and alarms without code changes You have a legacy service that logs payment failures as ERROR Payment failed: {orderId}. You can't easily change the code to publish a CloudWatch metric directly. Metric filter: pattern ERROR Payment failed → metric PaymentFailures count. Create an alarm on that metric. You get payment failure alerting without touching the application. This is a powerful operational capability for observing systems you can't modify.

Subscription filters stream log events from a log group to another destination in near real-time: a Lambda function, a Kinesis Data Stream, or a Kinesis Data Firehose (for S3 archival or OpenSearch). This is the architecture for centralised logging, SIEM integration, and long-term log archival.

→ Lambda function
Process log events in real-time. Use cases: parse logs and extract structured data, enrich with additional context (add account ID, environment), forward to a third-party logging service (Datadog, Splunk), auto-create PagerDuty incidents from specific log patterns.

Invoked per log batch (typically every 5 seconds or when batch reaches 1MB). Lambda has 15-minute max execution — suitable for near-real-time processing, not slow batch analytics.
→ Kinesis → S3
Long-term archival for logs beyond CloudWatch's retention window. Kinesis Data Firehose buffers logs (60 seconds or 128MB) and writes to S3 in GZIP-compressed format. S3 stores indefinitely at $0.023/GB/month (vs $0.03/GB/month in CloudWatch Logs).

S3-archived logs can be queried with Athena using a Glue catalog — cost-effective analytics on historical logs without loading into a database.
CloudWatch Logs Insights — querying your logs
Logs Insights is a query language for searching and aggregating log data across multiple log groups simultaneously. It replaces grepping log files and becomes your primary tool for incident investigation and performance analysis.

Logs Insights queries are charged per GB of data scanned ($0.0057/GB). Use time range filters and specific log group selections to limit scan scope. A query scanning 100GB costs $0.57 — manageable, but querying your entire log history for 12 months of a busy service could cost tens of dollars per query.

Logs Insights — essential query patterns
/* Find all ERROR logs in the last hour */
fields @timestamp, @message
| filter @message like /ERROR/
| sort @timestamp desc
| limit 100

/* Count errors by exception type (requires JSON logging format) */
fields exception, level
| filter level = "ERROR"
| stats count(*) as errorCount by exception
| sort errorCount desc
| limit 20

/* p99 latency of job creation endpoint over the last 6 hours */
fields @timestamp, duration, endpoint
| filter endpoint = "/api/v1/jobs" and method = "POST"
| stats pct(duration, 99) as p99, pct(duration, 95) as p95,
        avg(duration) as avg, count(*) as requests
  by bin(5m)   /* Group into 5-minute buckets */
| sort @timestamp desc

/* Find requests from a specific user that resulted in errors */
fields @timestamp, userId, endpoint, statusCode, message
| filter userId = "42" and statusCode >= 400
| sort @timestamp desc

/* Find slow requests (> 2 seconds) and their distribution */
fields @timestamp, endpoint, duration, traceId
| filter duration > 2000
| stats count(*) as slowRequests by endpoint
| sort slowRequests desc

/* Identify which EC2 instance is generating errors (multi-stream query) */
fields @logStream, @message
| filter level = "ERROR"
| stats count(*) as errors by @logStream
| sort errors desc
/* @logStream = "instance/i-abc123" — reveals which instance has the most errors */
Querying multiple log groups simultaneously Logs Insights supports querying up to 50 log groups in a single query. Select all service log groups and find which service generated a specific error. This is the key capability for incident investigation in microservices architectures — you don't need to query each service's logs separately to trace a failing request across services.
parse and regex are powerful but expensive The parse command extracts fields from unstructured log lines using regex or glob patterns. parse @message "* user=* endpoint=*" as level, user, endpoint. It works, but applying regex to every log line in a large time window scans more data and costs more than querying pre-parsed JSON fields. If you use JSON logging (as recommended in Day 12's ECS section), Logs Insights automatically recognises JSON fields without any parse command — reducing query cost and complexity significantly.
CloudWatch Agent — OS-level visibility
Standard CloudWatch EC2 metrics (published by the hypervisor) don't include memory usage, disk utilisation, or process-level CPU. The CloudWatch Agent runs inside the EC2 instance and publishes these OS-level metrics that the hypervisor can't see.

What standard EC2 metrics don't include: memory utilisation (the hypervisor doesn't know what your OS is doing with RAM), swap usage, disk space percentage (it knows disk IOPS but not how full the disk is), per-process CPU, open file descriptors. All of these require the CloudWatch Agent running inside the instance with access to /proc and other OS interfaces.

Anomaly detection — alarms that learn your baseline
Standard threshold alarms require you to know what "normal" looks like and set a static number. Anomaly detection learns your metric's normal behaviour (including daily and weekly patterns) and alarms when behaviour deviates significantly — without you needing to choose a threshold.

CloudWatch Anomaly Detection uses a machine learning model to establish a baseline band for a metric — the expected range of values based on historical data. The model accounts for: time-of-day patterns (lower traffic at 3AM), day-of-week patterns (lower traffic on Sundays), seasonal trends. The band is visualised as shaded region around the metric line on your dashboard — the metric is "normal" when it stays within the band.

You configure how many standard deviations constitute an anomaly. A value of 2 means the alarm fires when the metric is more than 2 standard deviations from the expected value. Higher values (3, 4) produce fewer false alarms but may miss subtle anomalies.

When anomaly detection outperforms threshold alarms Request count on Monday mornings: peaks at 1,000 req/s. Same metric at 3AM Saturday: baseline is 50 req/s. A static threshold alarm at "below 100 req/s" fires every Saturday night and every Sunday, creating noise. An anomaly detection alarm fires when request count is significantly below what's expected for that specific time — it knows Saturday 3AM should have 50 req/s, so 50 req/s is fine. But if the model expects 800 req/s at Monday 9AM and you're seeing 200 req/s, that's a real anomaly worth paging about.
CLI — create anomaly detection model and alarm
# Create anomaly detection model for ALB request count
aws cloudwatch put-anomaly-detector \
  --namespace "AWS/ApplicationELB" \
  --metric-name "RequestCount" \
  --dimensions Name=LoadBalancer,Value=app/getwork-alb/abc123 \
  --stat "Sum" \
  --configuration ExcludedTimeRanges=[{StartTime=2024-01-01T00:00:00Z,EndTime=2024-01-03T00:00:00Z}] \
  --region ap-south-1
# ExcludedTimeRanges: exclude holiday periods from the training data

# Create alarm that fires when metric is BELOW the expected band
# (request count drop = possible outage, not a metric threshold exceedance)
aws cloudwatch put-metric-alarm \
  --alarm-name "getwork-request-count-anomaly" \
  --alarm-description "Request count significantly below expected baseline" \
  --metrics '[
    {"Id":"m1","MetricStat":{"Metric":{"Namespace":"AWS/ApplicationELB",
      "MetricName":"RequestCount",
      "Dimensions":[{"Name":"LoadBalancer","Value":"app/getwork-alb/abc123"}]},
      "Period":300,"Stat":"Sum"}},
    {"Id":"e1","Expression":"ANOMALY_DETECTION_BAND(m1,2)","Label":"Expected"}
  ]' \
  --comparison-operator LessThanLowerOrGreaterThanUpperThreshold \
  --threshold-metric-id "e1" \
  --evaluation-periods 3 \
  --datapoints-to-alarm 2 \
  --treat-missing-data notBreaching \
  --alarm-actions $ALERT_TOPIC \
  --region ap-south-1
Anomaly detection requires 2 weeks of training data The ML model needs at least 2 weeks of historical metric data to learn weekly patterns. A model trained on less data produces poor bands and frequent false alarms. If you're setting up anomaly detection for a new service, wait 2–3 weeks before enabling anomaly-based alarms. Use standard threshold alarms during the training period.
Container Insights — ECS and EKS observability
Container Insights provides container-level metrics and logs for ECS and EKS that standard CloudWatch doesn't include — CPU and memory per task, per container, and per service. It also provides a pre-built performance dashboard and automated log collection from containers.

Without Container Insights, ECS service-level metrics (like CPUUtilization in the AWS/ECS namespace) show the average across all tasks — the same problem as average EC2 CPU. Container Insights publishes per-task and per-container metrics, letting you identify which specific task is consuming excessive CPU or is close to its memory limit.

Container Insights for ECS publishes to the ECS/ContainerInsights namespace. Key metrics: CpuUtilized, MemoryUtilized, MemoryReserved, NetworkRxBytes, NetworkTxBytes — all with dimensions for ClusterName, ServiceName, TaskDefinitionFamily.

Interview prep
CloudWatch interview questions test whether you understand observability as a system design discipline — not just metric names and alarm configuration.
Q1 Your on-call engineer is getting paged 20 times per week for alarms that self-resolve within 5 minutes. How do you fix the alerting system? Hard ▼

This is an alert fatigue problem caused by alarms that are too sensitive. The fix is a combination of tuning existing alarms and restructuring the alarm hierarchy.

Step 1 — Audit the offending alarms. For each alarm that fires repeatedly and self-resolves, look at the metric history. Are these 1-minute spikes above threshold followed by immediate recovery? If yes, the datapoints-to-alarm setting is 1/1. Change it to 3/5 — require 3 consecutive breaches in 5 evaluation periods before alarming. A 1-minute spike that self-resolves won't trigger. A 3-minute sustained problem will.

Step 2 — Fix missing data treatment. Alarms transitioning to INSUFFICIENT_DATA during low-traffic periods (nights, weekends) and triggering INSUFFICIENT_DATA actions are a common source of false pages. Set missing data treatment to notBreaching for rate-based alarms (error rate, request count) and ignore for sparse metrics.

Step 3 — Introduce composite alarms. If the engineer is getting 5 separate pages for one correlated failure (RDS failover triggers error rate alarm, latency alarm, connection count alarm simultaneously), consolidate into a composite alarm. The composite alarm fires once; the child alarms are suppressed.

Step 4 — Separate urgency tiers. Not all alarms should page. Create three tiers: informational (write to Slack, no page), warning (Slack with @-mention, page if sustained), critical (PagerDuty, wake someone up). Most operational metrics belong in the first two tiers. Only genuine user-impact conditions belong in the third.

Step 5 — Review the alarm on the right metric. CPU-based alarms on a load-balanced fleet often fire when one instance spikes but users aren't affected (load is spread across 9 others). Switch to user-outcome metrics: p99 latency, error rate percentage. These directly measure user impact. CPU is an infrastructure signal, not a user-impact signal.

Why this works: structured five-step approach covering all root causes. The distinction between infrastructure signals and user-impact signals shows architectural maturity. Mentioning composite alarm suppression shows advanced CloudWatch knowledge.
Q2 How would you monitor a Spring Boot application on ECS to detect a memory leak? Medium ▼

Memory leak detection requires combining multiple monitoring signals because no single metric definitively identifies a leak.

Container Insights MemoryUtilized trend: The most direct signal. With Container Insights enabled, watch ECS/ContainerInsights MemoryUtilized for the ECS service. A memory leak shows as a steady, non-plateauing increase in memory usage over hours or days — not correlated with traffic increase. A step function increase followed by a plateau is normal (JVM heap filling up and stabilising). A linear trend that never plateaus is a leak.

JVM metrics via Micrometer: Expose JVM metrics from Spring Boot Actuator (automatically available with Micrometer). Key metrics: jvm.memory.used{area=heap} (heap usage), jvm.gc.pause (GC pause duration — increasing GC pauses with high heap usage indicates GC is struggling to free memory), jvm.classes.loaded (class loading anomalies can indicate classloader leaks). Publish to CloudWatch. Alarm on GC pause duration > 500ms sustained — this indicates the heap is so full that GC can't free enough space.

Practical detection approach: Create an alarm on MemoryUtilized > 85% of MemoryReserved for the ECS service. This fires before OOMKill (which happens at 100%). When the alarm fires, use ECS Exec (aws ecs execute-command) to shell into the affected container and run a heap dump: jmap -dump:format=b,file=/tmp/heapdump.hprof $(pgrep java). Copy the dump to S3 and analyse with Eclipse Memory Analyzer offline. The heap dump reveals exactly what objects are holding memory unexpectedly.

Why this works: three concrete monitoring signals (Container Insights trend, JVM via Micrometer, GC pause correlation), a specific alarm threshold (85% before OOMKill), and the exact diagnostic steps including ECS Exec and heap dump. This is operational depth, not just theory.
Q3 What is the difference between CloudWatch Logs metric filters and publishing custom metrics via PutMetricData? When would you use each? Easy ▼

Both produce CloudWatch metric data points, but they differ in where the metric extraction happens, the cost model, and what data is available.

Metric filters are configured on a log group. CloudWatch scans log events in real-time and creates a metric data point when the filter pattern matches. No application code change required. The metric value is extracted from the log message (e.g. count matching lines, or extract a numeric field from JSON). Cost: $0.10/GB of log data scanned for filter evaluation. Works on existing log streams without touching the application.

PutMetricData (or Micrometer) is called by the application itself at the point of measurement. The application has full context about what it's measuring — latency in milliseconds, count of a specific event type, gauge value. Richer metadata (dimensions, units). Cost: $0.01/1,000 data points published. Requires application code instrumentation.

Use metric filters when: You can't change the application (legacy system), the metric is derivable from existing log output, you need a quick metric without a deployment cycle, you want to count occurrences of specific log messages (errors, warnings, specific exceptions).

Use PutMetricData/Micrometer when: You're instrumenting a new application, the metric requires application context not available in logs (intermediate calculation results, business logic state), you need higher precision (per-endpoint latency with specific dimensions), you're building dashboards that should work even when logging is disabled.

Why this works: names the mechanism difference (log scanning vs application publishing), the cost model for each, and gives concrete use-case guidance for choosing between them.

DASHBOARD_BODY=$(cat << 'DASH' { "widgets": [ { "type": "alarm", "properties": { "title": "System Status", "alarms": [ "arn:aws:cloudwatch:ap-south-1:ACCOUNT:alarm:getwork-alb-no-healthy-hosts", "arn:aws:cloudwatch:ap-south-1:ACCOUNT:alarm:getwork-api-5xx-rate-high", "arn:aws:cloudwatch:ap-south-1:ACCOUNT:alarm:getwork-rds-connections-high", "arn:aws:cloudwatch:ap-south-1:ACCOUNT:alarm:getwork-rds-low-storage", "arn:aws:cloudwatch:ap-south-1:ACCOUNT:alarm:getwork-app-error-spike" ] }, "x": 0, "y": 0, "width": 24, "height": 3 }, { "type": "metric", "properties": { "title": "ALB Request Count & Error Rate", "metrics": [ ["AWS/ApplicationELB", "RequestCount", "LoadBalancer", "app/getwork-alb/SUFFIX"], ["AWS/ApplicationELB", "HTTPCode_Target_5XX_Count", "LoadBalancer", "app/getwork-alb/SUFFIX"] ], "period": 60, "stat": "Sum", "view": "timeSeries" }, "x": 0, "y": 3, "width": 12, "height": 6 }, { "type": "metric", "properties": { "title": "RDS Connections & CPU", "metrics": [ ["AWS/RDS", "DatabaseConnections", "DBInstanceIdentifier", "getwork-db"], ["AWS/RDS", "CPUUtilization", "DBInstanceIdentifier", "getwork-db"] ], "period": 300, "stat": "Average", "view": "timeSeries" }, "x": 12, "y": 3, "width": 12, "height": 6 } ] } DASH ) aws cloudwatch put-dashboard \ --dashboard-name GetWork-Operations \ --dashboard-body "$DASHBOARD_BODY" \ --region $REGION echo "Dashboard created!" echo "View at: https://ap-south-1.console.aws.amazon.com/cloudwatch/home#dashboards:name=GetWork-Operations"

Day 14 cheatsheet
CloudWatch production mechanics at a glance.
Metric resolution decay
Raw: 3hrs. 1-min rollups: 15 days. 5-min rollups: 63 days. 1-hr rollups: 455 days. Match query period to available resolution or you get empty graphs.
Average CPU is wrong
Fleet average masks saturated individual instances. Use Maximum CPU, or better: ALBRequestCountPerTarget which directly measures user experience.
3/5 datapoints-to-alarm
The production sweet spot. Ignores 1-2 period transient spikes. Fires on 3 consecutive breaches. Reduces false pages by 80%+ vs default 1/1.
Missing data = notBreaching
For rate-based alarms (error rate, request count). Prevents false INSUFFICIENT_DATA alerts during low-traffic periods. Missing rate data = 0 errors, not unknown.
Composite alarms
Boolean AND/OR/NOT of child alarms. One page for correlated failures. Suppress child alarm actions when composite handles the notification.
Divide-by-zero in metric math
Error rate = (5xx / requests) * 100. Returns NaN when requests=0. Use: IF(m2 > 0, (m1/m2)*100, 0). Prevents alarm INSUFFICIENT_DATA state during zero-traffic periods.
Log retention = $ forever
Default: never expire. Set retention on every log group: 30 days (app logs), 90 days (errors), 365 days (audit). Auto-created Lambda/ECS log groups have no retention — audit monthly.
JSON logging → free field queries
Logs Insights auto-parses JSON log fields. No parse command needed. Queries are faster, cheaper, and simpler. Use logstash-logback-encoder or structured logging from Day 12.
EMF for Lambda metrics
Embedded Metric Format: write JSON log lines with _aws.CloudWatchMetrics key. CloudWatch extracts metrics for free. Zero added Lambda execution time vs PutMetricData SDK call.
defaultValue=0 in metric filters
Critical for alarm evaluation. Without it, periods with no matching log events have no data point — alarm transitions to INSUFFICIENT_DATA instead of staying OK.
Anomaly detection training
Needs 2 weeks of data to learn daily/weekly patterns. Use threshold alarms during training period. Accounts for time-of-day and day-of-week patterns automatically.
Container Insights cost
~$0.30/metric/month + log ingestion. 10 services × 15 metrics = $45/month for metrics alone. Enable on production only; use standard ECS metrics for dev/staging.
Production alarm checklist
1Never use 1/1 datapoints-to-alarm for operational alarms. Minimum: 2/3. Recommended: 3/5.
2Set treat-missing-data=notBreaching on all rate-based alarms (error rate, request count). Use breaching only for heartbeat monitors.
3Always specify dimensions on ALB alarms. Undimensioned ALB metrics aggregate across all load balancers and dilute the signal.
4Use composite alarms to reduce correlated failure noise. Never page on individual resource alarms that are caused by one root event.
5Set retention on every log group — including auto-created ones. Run the retention audit script monthly.
6Add defaultValue=0 to every metric filter transformation. Without it, alarm evaluation breaks on zero-activity periods.
7Use JSON structured logging. Logs Insights queries are simpler, faster, and cheaper than unstructured log parsing with regex.
8Alarm on user outcomes first (error rate %, p99 latency), infrastructure second (CPU, memory). Infrastructure alarms are leading indicators; outcome alarms are definitive.
9Store CloudWatch Agent config in SSM Parameter Store. Update agent config without AMI rebuild or deployment.
10Add deployment annotations to CloudWatch dashboards. Instant correlation between deployment events and metric changes.
