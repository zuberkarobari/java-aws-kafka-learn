7. Load Balancer

When one server isn't enough, we add more. But how do we distribute millions of requests evenly without crushing a single machine?

1. Definition

Simple English

Think of a traffic cop standing at a busy intersection during rush hour. Instead of letting all cars jam down one lane, the cop directs traffic evenly across three open lanes so everyone keeps moving quickly.

Technical

A device or software (like AWS ALB or HAProxy) that acts as a reverse proxy, distributing network or application traffic across a cluster of servers to improve responsiveness and availability.

Purpose

To achieve Horizontal Scaling (scaling out). It prevents any single server from becoming a bottleneck (Single Point of Failure) and constantly checks server health to route around dead instances.

2. Internal Working: Routing & Health

The load balancer intercepts the traffic and decides which backend server should handle the request based on an algorithm and health checks.

3. Interview Questions

Answer: A Layer 4 LB (Network Load Balancer) routes traffic based strictly on IP addresses and TCP/UDP ports. It is extremely fast because it doesn't look at the data payload.

A Layer 7 LB (Application Load Balancer) decrypts the HTTPS traffic, reads the HTTP headers, URLs, and cookies. It allows for "smart" routing, e.g., routing requests for `/api/videos` to a specific video microservice, and `/api/users` to the user microservice.

Answer: Sticky Sessions (Session Affinity) is a feature where the LB routes a specific user's requests to the exact same backend server every time. This was used when servers held user session state (like a shopping cart) in their local RAM.

Why avoid them? They defeat load balancing. If one server gets "stuck" with heavy users, it crashes, and those users lose their session data. Modern REST APIs are Stateless. They store session data in a distributed cache like Redis. This means the LB can safely route any request to any server at any time, achieving perfect horizontal scaling.
