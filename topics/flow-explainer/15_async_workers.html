15. Async Workers

Once a task is in Kafka, who actually does the work? Enter the background worker nodes that scale independently to crunch through queues.

1. Definition

Simple English

If the API server is the "Cashier" taking orders, the Async Worker is the "Chef" in the back room actually cooking the meals. They operate completely out of sight from the customer.

Technical

Stateless background processes or microservices that continuously poll a message broker (Kafka/RabbitMQ) for jobs, execute heavy computational or I/O bound logic, and optionally update a database or emit a success event upon completion.

Purpose

To protect the main web servers from CPU exhaustion. Workers allow you to independently scale the parts of your app that do heavy lifting (like video rendering) without scaling the parts that just serve web pages.

2. Consumer Groups & Scaling

If one worker is too slow to process all emails in the queue, Kafka allows you to spin up multiple instances of the worker in a Consumer Group to share the load evenly.

Kafka Topic: email_queue
Partition 0
Partition 1
Partition 2
Consumer Group: email-senders
Worker Node 1
Reads Partition 0
Worker Node 2
Reads Partition 1
Worker Node 3
Reads Partition 2
Kafka guarantees that a partition is read by ONE AND ONLY ONE worker within the same group. This prevents duplicate emails.

3. Interview Questions

Answer: The 4th Worker Node will sit completely idle. In Kafka, a single partition cannot be split between multiple consumers in the same group (to ensure message ordering). The maximum number of active workers in a group is strictly equal to the number of partitions. If you want to process faster, you must first add more partitions to the topic.

Answer: This triggers a Rebalance. Every worker node constantly sends a heartbeat to the Kafka broker. If Worker 2 crashes, the broker notices the missing heartbeat. It takes Partition 1 (which Worker 2 was reading) and assigns it to either Worker 1 or Worker 3. Processing continues seamlessly. When Worker 2 reboots and rejoins, another Rebalance occurs, and Partition 1 is handed back to it.
