20. The Complete Flow

Everything comes together. Trace a single POST /api/orders request from the browser click all the way to a Kafka event, and back to a JSON response.

The Full Request Journey

Scenario: A user clicks "Place Order" on an e-commerce site. Follow every single hop.

1
Browser (Client) ~0ms

User clicks "Place Order". JavaScript calls fetch('POST /api/orders', body). Browser checks HSTS, local cache, and DNS cache — all misses for this dynamic endpoint.

2
DNS Resolution ~10ms

Browser asks the OS DNS resolver for api.shop.com. ISP Resolver queries the Root → TLD → Authoritative (Route 53). Returns an IP pointing to the Cloudflare CDN edge node closest to the user.

3
TCP + TLS Handshake ~30ms

Browser establishes a TCP connection with the CDN edge (SYN → SYN-ACK → ACK). Immediately performs a TLS 1.3 handshake. Certificate is validated. Symmetric session key is negotiated. Channel is secure.

4
WAF + CDN Edge ~2ms

Cloudflare WAF inspects the decrypted HTTP request. Checks IP reputation, rate limiting, OWASP rules. It's a POST to a dynamic endpoint — not cached. WAF passes it. CDN forwards to the origin Load Balancer via an optimized private network.

5
Load Balancer → API Gateway ~1ms

AWS ALB receives the request, performs a health check round-robin, and forwards to one of the three API Gateway pods in the K8s cluster. Spring Cloud Gateway validates the JWT token, applies rate limiting (Redis token bucket), and routes the request to the Order Service based on path predicate /api/orders/**.

6
Spring Boot Application Layer ~5ms

Tomcat accepts the request. DispatcherServlet routes to OrderController.createOrder(). Jackson deserializes the JSON body into OrderRequest DTO. Bean Validation runs (@Valid). Execution delegates to @Service.

7
Redis Cache Check ~1ms

Service checks Redis for cached product catalog and user details. Cache Hit on both — data returned from RAM in ~1ms. No DB query needed for these reads. The @Transactional boundary starts for the write operation.

8
Database Write (PostgreSQL) ~8ms

HikariCP pool provides a connection in microseconds. Hibernate generates INSERT INTO orders VALUES (...). PostgreSQL writes to the WAL (Write-Ahead Log) and confirms the transaction. Hibernate maps the returned row to an Order entity. Transaction commits. Connection returned to pool.

9
Kafka Event Published ~2ms

After the DB commit, kafkaTemplate.send("order-events", orderEvent) appends an event to the Kafka log. This is fire-and-forget (async). The Email Worker and Inventory Worker will consume this event independently — completely decoupled from this HTTP request.

10
HTTP Response to Browser ~total 60ms

Controller maps Order entity to OrderDTO, Jackson serializes to JSON. HTTP 201 Created with body {orderId: "ORD-9981", status: "PROCESSING"}. Response travels back through API Gateway → ALB → CDN Edge → TLS-encrypted over TCP → Browser. JavaScript renders "Order Placed ✓".

Latency Budget Breakdown

Network (Browser ↔ CDN)

~20ms

Physical distance & speed of light. CDN reduces this dramatically vs. hitting origin directly.

TLS Handshake

~10ms

TLS 1.3 reduced from 4 round trips (TLS 1.2) to just 1 round trip. 0-RTT session resumption is even faster.

Database Query

~8ms

Dominant latency factor. Indexes are everything. Avoid N+1 queries — they multiply this by the record count.

Spring Boot Logic

~5ms

Deserialization, validation, business rules, serialization. Rarely the bottleneck if you avoid blocking I/O on the request thread.

Principal Engineer Interview Questions

Answer:
1. Single Database Master: If it crashes, writes are lost. Fix: Use Multi-AZ RDS with automatic failover (standby replica promoted to master in ~30 seconds).

2. Single Redis Instance: If it crashes, the cache is gone and the DB gets flooded. Fix: Redis Cluster with sharding + Redis Sentinel for automatic leader election.

3. Single Kafka Broker: If it crashes, events are lost. Fix: Kafka Cluster with Replication Factor 3. Each partition has 1 leader and 2 followers. If the leader dies, one follower is elected the new leader automatically.

4. Single API Gateway Instance: Fix: Kubernetes Deployment ensures 3 replicas of the Gateway are always running.

Answer: The database almost always breaks first. It is the hardest component to scale horizontally because of the ACID guarantees on writes.

Mitigation plan:
1. First line: Redis absorbs the read spike (cache hit rate should jump to 95%+).
2. Next: Read Replicas share the SELECT load.
3. If writes are still spiking: Kafka acts as a write buffer. Instead of writing directly to DB on every API call, drop the event to Kafka and have a consumer persist at a controlled rate.
4. Finally: Database sharding (partition data by user_id or geography across multiple DB instances).

1. Get the Trace ID: Ask the user for the timestamp and userID. Search Kibana logs for their userID around that time. Find both requests' Trace IDs (T-111 and T-112).

2. Check Zipkin: Pull up both Traces side-by-side. Do they have different Source IPs? Or the same IP? If the same, it may be a client-side double-click. Look at timestamp gap — if < 50ms, it's definitely a browser re-submission.

3. Check API Gateway logs: Did the Gateway see 2 requests, or did it forward one request and it got duplicated downstream? If the Gateway only saw 1 request, the duplication happened in the Order Service.

4. Check DB: Are there 2 separate rows in the `orders` table? Do they have the same `idempotency_key` column value? If yes, the Idempotency Key check failed (bug). If different keys, a genuine double-call happened.

5. Fix: Ensure the OrderController checks a unique client-generated `idempotency_key` per request. Store this key in Redis with a short TTL. If the key exists in Redis on a second call, return the cached `201 Created` response without creating a new DB row.
