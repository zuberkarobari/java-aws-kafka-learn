16. Third-Party APIs

You can't build everything yourself. Here is how modern applications integrate with Stripe, Twilio, and SendGrid without breaking when they go down.

1. Definition

Simple English

Instead of building your own power plant to get electricity, you plug into the city's power grid. Third-party APIs allow you to plug your app into specialized services (like payment processing) that other companies have perfected.

Technical

External Web Services (usually RESTful or GraphQL) provided by third parties. Your Spring Boot application acts as a Client (using `RestTemplate` or `WebClient`) to make HTTP calls to these external servers.

Purpose

To dramatically accelerate development. Handling credit card security (PCI compliance) takes years to build; using Stripe takes hours.

2. The Circuit Breaker Pattern

What happens when Stripe goes down? If your app waits 30 seconds for a timeout, all your Tomcat threads will block, and your entire application will crash. We prevent this using a Circuit Breaker (Resilience4j).

State: CLOSED (Normal)
Requests flow normally from Spring Boot to Stripe. Everything succeeds.
State: OPEN (Failing)
Stripe API is timing out. Resilience4j detects 5 failures in a row. It "Opens the circuit". Future requests fail FAST immediately (without waiting for Stripe) and execute a Fallback Method (e.g., "Payment system down, try later").
State: HALF-OPEN (Testing)
After 60 seconds, the circuit lets exactly 3 requests pass through to Stripe to test if it's back online. If they succeed, it goes back to CLOSED. If they fail, it goes back to OPEN.

3. Interview Questions

Answer: RestTemplate is synchronous and blocking. If you make a call to an API, the thread waits and does nothing until the response comes back.

WebClient (from Spring WebFlux) is asynchronous and non-blocking. When it makes an API call, it immediately releases the thread back to the pool to do other work. When the API responds, an event loop assigns a thread to process the result. WebClient is the modern standard for high-performance microservices.

Answer: If an API call fails with a Network Timeout, you don't know if the request reached Stripe but the *response* failed, or if the request never reached Stripe at all. If you blindly retry, you might charge the user twice.

To handle this, good APIs support Idempotency Keys. You generate a unique UUID for the transaction and pass it in the HTTP Header (e.g., `Idempotency-Key: uuid`). If you retry the exact same request 5 times, Stripe sees the same UUID, realizes it's a retry, and only processes the charge exactly once, returning the cached success response for subsequent retries.
