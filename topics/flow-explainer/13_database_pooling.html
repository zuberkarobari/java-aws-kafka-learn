13. Connection Pooling

Establishing a TCP connection to a database is wildly expensive. Here is how HikariCP saves Spring Boot from buckling under pressure.

1. Definition

Simple English

Imagine renting a car every time you need to go to the grocery store. Filling out the paperwork takes an hour, and the drive takes 5 minutes. A connection pool is like having a fleet of company cars parked outside with the keys in the ignition, ready to go instantly.

Technical

A cache of database connections maintained so that they can be reused when future requests to the database are required. Connection pools reduce the overhead of TCP handshakes and database authentication for every query.

Purpose

To drastically improve performance by eliminating connection latency, and to protect the database from being overwhelmed by limiting the maximum number of concurrent connections.

PostgreSQL typically runs on a server with a fixed number of CPU cores (e.g., 8 cores). If you send 1000 concurrent queries, the DB OS must context-switch between 1000 threads. Context switching requires CPU overhead. The database will spend more time switching contexts than executing queries, leading to complete gridlock.

The golden formula (by PostgreSQL) is: Connections = ((core_count * 2) + effective_spindle_count). Often, a small pool size like 10-20 handles thousands of RPS perfectly because queries take only milliseconds.
