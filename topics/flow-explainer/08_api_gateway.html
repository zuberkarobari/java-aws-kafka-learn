8. API Gateway

The single entry point to your microservices architecture. It handles routing, JWT validation, and rate limiting so your downstream services don't have to.

1. Definition

Simple English

Imagine walking into a massive hospital. Instead of wandering around looking for cardiology, you go to the front reception desk. They check your ID, figure out what you need, and direct you to the right floor. The API Gateway is the reception desk.

Technical

An API management tool that sits between a client and a collection of backend services. It acts as a reverse proxy to accept all API calls, aggregate the various services required to fulfill them, and return the appropriate result.

Purpose

To extract cross-cutting concerns (authentication, rate limiting, CORS, routing) out of individual microservices, centralizing them in one place. Developers can focus on business logic instead of security boilerplate.

2. Internal Working (Spring Cloud Gateway)

When a request hits Spring Cloud Gateway, it evaluates a set of Predicates. If they match, it runs the request through a chain of Filters before routing it to the destination.

3. Interview Questions

Answer: A Load Balancer distributes traffic across multiple instances of the *same* application to prevent overloading (e.g., distributing traffic across 5 identical Order Service instances).

An API Gateway routes traffic to *different* applications based on the URL path (e.g., `/orders` goes to Order Service, `/users` goes to User Service), while also executing application-level logic like JWT validation and rate limiting. In production, you often place a Load Balancer *in front* of the API Gateway, and *another* Load Balancer between the Gateway and the internal microservices.

Answer: Instead of having one massive API Gateway for all clients, the BFF pattern creates multiple tailored gateways depending on the client.

For example, you might have one Gateway tailored for the Mobile App (returns smaller JSON payloads, aggressively caches), and a different Gateway for the Web Dashboard (returns large aggregated payloads). This prevents the Gateway from becoming a bloated monolith itself.
