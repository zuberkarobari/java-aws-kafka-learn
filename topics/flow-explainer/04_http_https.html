4. HTTP & HTTPS

Before the browser sends your password, it must secure the connection. Let's dive into TLS Handshakes, Asymmetric Encryption, and SSL Certificates.

1. Definition

Simple English

HTTP is like sending a postcard in the mail—anyone who handles it (routers, ISPs) can read the message. HTTPS is like putting that postcard in an unbreakable locked box where only the recipient has the key.

Technical

HTTPS is HTTP layered over the TLS (Transport Layer Security) protocol. It provides three guarantees: Encryption (data is unreadable), Data Integrity (data wasn't altered), and Authentication (you are talking to the real server).

Purpose

To prevent Man-in-the-Middle (MitM) attacks where bad actors on public Wi-Fi or compromised routers intercept sensitive data like passwords, credit cards, or session cookies.

2. Internal Working: The TLS Handshake

Immediately after the TCP 3-Way Handshake finishes, the TLS Handshake begins. This is where asymmetric cryptography (Public/Private keys) is used to safely agree on a symmetric session key.

3. Interview Questions

Answer: Asymmetric encryption (Public/Private keys like RSA) is very secure for securely sharing a secret over an untrusted network, but it is mathematically complex and extremely slow.

Symmetric encryption (one shared key like AES) is incredibly fast and efficient.

Therefore, TLS uses the slow Asymmetric method just once during the handshake to securely exchange the Symmetric "Session Key". Once both sides have the Session Key, they use the fast Symmetric method to encrypt the actual HTTP payload.

Answer: A CA (like Let's Encrypt or DigiCert) is a trusted third-party organization that digitally signs a server's SSL certificate to prove its identity. Your OS/Browser comes pre-installed with the root certificates of trusted CAs.

If a CA is compromised, hackers can generate valid SSL certificates for ANY domain (e.g., a fake google.com). When a user visits the hacker's fake site, the browser will show a green padlock because the certificate was signed by a CA the browser trusts. This allows perfect Man-in-the-Middle attacks. Browsers immediately push updates to revoke trust in compromised CAs when this happens.
