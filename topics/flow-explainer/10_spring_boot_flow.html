10. Spring Boot Flow

The request has finally reached our code. How does it traverse the Servlet Container, DispatcherServlet, Controller, Service, and Repository layers?

1. Architectural Flow

Web Layer

The embedded Tomcat server accepts the TCP connection. Filters (like Spring Security) inspect the request. The DispatcherServlet maps the URL to the correct @RestController.

Business Layer

The Controller maps JSON to a DTO and passes it to the @Service. Here, transactional boundaries are started (`@Transactional`), and the core business logic executes.

Data Layer

The Service calls a @Repository (Spring Data JPA). Hibernate translates the method call into a native SQL query, fetches the Data Entity, and maps it back to a Java Object.

3. Interview Questions

Answer: The DispatcherServlet is the "Front Controller" in Spring MVC. It intercepts every incoming HTTP request and acts as the central router. It consults the HandlerMapping to find which @RestController should handle the specific URL, and then uses HttpMessageConverters (like Jackson) to deserialize the JSON body into Java objects before invoking your method.

Answer: Returning JPA Entities directly from the Controller is a major anti-pattern for three reasons:
1. Security: An entity might contain sensitive fields (like a password hash). If you return it directly, you leak data.
2. Coupling: It tightly couples your database schema to your external API contract. If you rename a DB column, your API response changes, breaking clients.
3. Infinite Recursion: If you have Bidirectional Relationships (e.g., User → Orders, Order → User), Jackson will infinitely loop while trying to serialize it to JSON unless you add complex annotations. DTOs provide a clean, flattened contract.
