17. Service Discovery

When microservices spin up and down dynamically in the cloud, their IP addresses constantly change. How do they find each other? Enter Eureka.

1. Definition

Simple English

Imagine an office where employees change desks every day. If you need to talk to Bob, you can't memorize his desk number. Instead, you ask the receptionist (Service Discovery), who knows exactly where everyone is sitting right now.

Technical

A central registry (like Netflix Eureka or Consul) where microservices register their current network locations (IP and Port) at startup. Other services query this registry to dynamically resolve the IP address of the service they need to call.

Purpose

To eliminate hardcoded IP addresses in configuration files. It enables auto-scaling, as new instances register themselves automatically and dead instances are removed when their heartbeats fail.

2. Client-Side Service Discovery

In the Spring Cloud ecosystem, Eureka operates on the client-side. The API Gateway asks Eureka for addresses, and then load balances the request itself.

The only downside is that if a new Order Service instance spins up during the outage, the Gateway won't know about it until Eureka comes back online and the cache is refreshed.

Answer:
Client-Side (Eureka): The client queries the registry to get a list of IPs, then the client itself applies load balancing algorithms to pick an IP and make the call. It gives clients more control but couples them to the registry tech.

Server-Side (Kubernetes Services/AWS ALB): The client just calls a fixed internal DNS name (like `http://order-service`). It hits a load balancer. The load balancer queries the registry, picks an IP, and forwards the request. The client has zero knowledge of the registry or the underlying IPs. This is the modern, preferred approach.
