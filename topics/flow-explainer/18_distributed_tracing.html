System Request Flow

18. Distributed Tracing

When a request fails after traversing 5 different microservices, how do you know which one crashed? Enter Sleuth and Zipkin.

1. Definition

Simple English

Imagine handing a package to FedEx. They put a barcode on it. Every time it changes trucks or cities, it gets scanned. If it gets lost, you look up the barcode to see exactly where it stopped. Tracing puts a barcode on HTTP requests.

Technical

A method used to profile and monitor applications, especially those built using a microservices architecture. It tracks a single request through all its complex interactions across multiple independent systems.

Purpose

To pinpoint performance bottlenecks and debug errors. Without tracing, looking at logs from 10 different servers to find one failed request is like finding a needle in a haystack.

2. Trace IDs and Span IDs

Tools like Spring Cloud Sleuth (Micrometer Tracing) automatically inject headers into every HTTP request. Zipkin collects these to draw visual timelines.

1. API Gateway
Receives GET /order. Generates a new Trace ID (T-123) and a Span ID (S-A). Logs it, then forwards to Order Service.

Headers injected:
`X-B3-TraceId: T-123`
`X-B3-SpanId: S-B`
`X-B3-ParentSpanId: S-A`
2. Order Service
Receives request. Keeps Trace ID (T-123) but uses its new Span ID (S-B). Performs DB logic (Span S-C), then calls Inventory Service.

Headers injected:
`X-B3-TraceId: T-123`
`X-B3-SpanId: S-D`
`X-B3-ParentSpanId: S-B`
3. Inventory Service
Keeps Trace ID (T-123). Uses Span ID (S-D). ERROR OCCURS HERE.

3. Interview Questions

Answer: A Trace ID represents the entire journey of a single user request. It is generated at the Gateway and remains exactly the same as the request bounces across 5 different microservices.

A Span ID represents a single logical operation or "hop" within that journey. Every time the request hits a new microservice, or even executes a new database query within a service, a new Span ID is generated. Spans are linked together via Parent/Child relationships to build a timeline tree.

Answer: While Zipkin provides the visual timeline of how long each span took, you still need to see the actual error logs (stack traces).

Spring Cloud Sleuth injects the [TraceID, SpanID] into the standard SLF4J/Logback logs (e.g., INFO [OrderSvc, T-123, S-B]). Logstash forwards all these logs from all servers into Elasticsearch. When the user reports an error, you find their Trace ID, paste it into the Kibana dashboard search bar, and instantly see every single log line printed by every microservice for that exact request, filtered out from the millions of other logs.
