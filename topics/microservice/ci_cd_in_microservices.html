CI/CD in Microservices

Mastering independent deployability, Immutable Artifacts, GitOps, and Zero-Downtime rollouts.

1. Definition

Simple English

Instead of a massive factory shutting down once a month to release a giant product, a microservices CI/CD pipeline is like having 50 small, independent assembly lines. If the "Tire" factory wants to release a new tire, it does so immediately, automatically testing and shipping it without asking the "Engine" factory to stop.

Technical

Continuous Integration (CI) automates the merging, building, and testing of code. Continuous Delivery/Deployment (CD) automates the release of containerized artifacts (Docker images) into environments (Kubernetes). In microservices, every service must have its own strictly isolated pipeline.

Purpose

To achieve Independent Deployability. The entire architectural benefit of microservices is lost if you must deploy all 50 services together. CI/CD guarantees safe, fast, and automated rollouts of individual domains.

3. Real-World Project Example

The E-Commerce Polyrepo

How isolation prevents system-wide bottlenecks.

Service A: Product Search

Written in Go. The team pushes a performance tweak. Their GitHub Action runs Go tests, builds a Docker image tagged v1.2.0, and deploys it to the K8s cluster. Total time: 3 minutes. They deploy 5 times a day.

Service B: Payment Gateway

Written in Java/Spring Boot. Because it touches money, its pipeline is intentionally slow. It runs heavy integration tests, dynamic security scans (DAST), and deploys using a Canary strategy. Total time: 45 minutes.

The Decoupling Magic

Because their codebases and pipelines are isolated (Polyrepo), the Search team doesn't have to wait 45 minutes for the Payment team's tests to finish just to deploy a UI fix. If the Payment pipeline breaks due to a bad test, the Search team can still deploy successfully.

4. 7 Key Concepts For senior-level depth

1

Monorepo vs Polyrepo

Polyrepo: 1 Service = 1 Git Repo = 1 Pipeline (Standard approach). Monorepo: All services in 1 Git Repo. Requires complex CI tools (like Bazel or Nx) to ensure only the modified microservice is built and deployed.

2

Immutable Artifacts

A core rule: You build a Docker image ONCE, tag it (e.g., `v1.2.3`), and promote that exact same image through DEV -> QA -> PROD. Never rebuild the code for different environments; inject config at runtime.

3

Semantic Versioning

Artifacts should be tagged clearly. MAJOR.MINOR.PATCH. A change in the MAJOR version indicates a breaking API change, warning other microservices that depend on it.

4

Infrastructure as Code (IaC)

Terraform or AWS CDK. The infrastructure required for the microservice (databases, queues, IAM roles) is defined in code and deployed automatically alongside the application.

5

GitOps (ArgoCD/Flux)

Instead of the CI tool running kubectl apply (Push), an agent inside the Kubernetes cluster watches a Git repository containing YAML manifests and pulls the changes to match the desired state.

6

Zero-Downtime Deployments

Achieved via Rolling Updates (replacing pods one by one), Blue/Green (spinning up a parallel environment and flipping traffic), or Canary (routing 5% of traffic to the new version to test for errors).

7

Database Migrations (Flyway/Liquibase)

Database schemas must be version-controlled and executed automatically during CD. Crucially, they must be Backward Compatible. For example, to rename a column, you must add the new column (Release 1), update code to write to both (Release 2), migrate old data, and finally drop the old column (Release 3). Otherwise, Rolling Updates will crash.

Instead, we use Consumer-Driven Contract Testing (e.g., Pact). Service A writes a contract defining what it expects from Service B. Service B's pipeline verifies it doesn't break this contract before deploying, ensuring compatibility without spinning up the whole system.

Answer: Using the :latest tag is a severe anti-pattern in Kubernetes. If a pod crashes and K8s pulls the image again, it might pull a newer, untested version that overwrote the latest tag, causing production bugs. Furthermore, you cannot quickly rollback to a previous version because you don't know what latest was pointing to yesterday. Always use specific, immutable tags (like the Git commit hash).

6. Code Example: GitHub Actions Pipeline

.github/workflows/order-service-ci.yml
Copied! 
name: Order Service Pipeline

on:
  push:
    branches: [ 'main' ]
    # Polyrepo/Monorepo strategy: Only run if code in THIS folder changes
    paths:
      - 'services/order-service/**'

env:
  IMAGE_NAME: company/order-service
  # Use Git Commit Hash for Immutable Tagging
  IMAGE_TAG: ${{ github.sha }}

jobs:
  build-and-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      
      - name: Set up JDK 17
        uses: actions/setup-java@v3
        with:
          java-version: '17'
          distribution: 'temurin'
          cache: maven

      - name: Run Unit Tests & SonarQube Analysis
        run: mvn -B clean verify sonar:sonar -Dsonar.projectKey=order-service
        working-directory: ./services/order-service

  docker-build-push:
    needs: build-and-test
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      
      - name: Login to DockerHub / ECR
        uses: docker/login-action@v3
        with:
          username: ${{ secrets.DOCKER_USERNAME }}
          password: ${{ secrets.DOCKER_PASSWORD }}

      - name: Build and Push Docker Image
        run: |
          docker build -t $IMAGE_NAME:$IMAGE_TAG .
          docker push $IMAGE_NAME:$IMAGE_TAG
        working-directory: ./services/order-service

  deploy-to-k8s:
    # In a modern GitOps flow, this step updates a YAML manifest in a separate git repo.
    # ArgoCD then detects the YAML change and automatically syncs it to Kubernetes.
    needs: docker-build-push
    runs-on: ubuntu-latest
    steps:
      - name: Update Kubernetes Manifests
        run: |
          sed -i "s|image: .*|image: $IMAGE_NAME:$IMAGE_TAG|g" k8s/deployment.yml
          git commit -am "Update Order Service image to $IMAGE_TAG"
          git push

7. Senior Developer Answer

"The fundamental requirement for microservices is independent deployability. If you have 20 microservices but they must all be deployed together in a specific order, you haven't built microservices—you've built a distributed monolith.

To prevent this, every microservice must have its own isolated CI/CD pipeline. When a developer pushes code, the CI phase executes unit tests, performs static analysis, and builds an immutable Docker image tagged uniquely, usually with the Git commit hash.

For the Continuous Deployment phase, I strictly advocate for the GitOps model using tools like ArgoCD. Instead of our CI tool pushing credentials into our Kubernetes cluster to deploy, our pipeline simply updates a YAML manifest in a Git repository. ArgoCD, running securely inside the cluster, detects the change and pulls the new image, applying it via a Rolling Update. This guarantees zero-downtime, ensures our Git repo is the single source of truth for our infrastructure state, and makes instant rollbacks trivial."

8. Quick Revision Notes

CI (Continuous Integration): Automates build, test, and artifact creation on code push.
CD (Continuous Delivery): Automates deployment of the artifact to staging/production.
Independent Deployability: The golden rule. Service A deploys without affecting Service B.
Immutable Artifacts: Build a Docker image once, promote the exact same image to Prod.
GitOps: Infrastructure state is stored in Git. Cluster pulls changes automatically.
DB Migrations: Must be backward compatible so old pods don't crash during a rolling update.
Blue/Green: Stand up new environment, flip router. Fastest rollback.
Canary: Route 5% of traffic to new version to test for errors before 100% rollout.

9. Interview Answer Template

"In a microservices architecture, CI/CD pipelines are absolutely critical to achieving independent deployability. Every microservice must have its own isolated pipeline.

The Continuous Integration phase is triggered upon code push. It runs unit tests, performs static analysis via tools like SonarQube, and packages the application into an immutable Docker image tagged with the Git commit hash.

For Continuous Delivery into Kubernetes, we utilize a GitOps approach with tools like ArgoCD. Rather than pushing code directly to the cluster, the CI pipeline updates a configuration repository. The cluster detects this change and initiates a Zero-Downtime Deployment—usually a Rolling Update or a Canary release. This ensures safe, automated rollouts and makes instant rollbacks trivial if errors occur."
