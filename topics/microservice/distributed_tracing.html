The Pillars of Observability

Distributed Tracing

Mastering Trace IDs, Context Propagation, OpenTelemetry, and tracking requests across decoupled microservices.

1. Definition

Simple English

Imagine tracking a FedEx package internationally. You don't just know it left and arrived; you have a unique Tracking Number that tells you exactly when it passed through London, New York, and Chicago. Distributed Tracing assigns a "Tracking Number" to every user request.

Technical

It is a diagnostic technique that monitors applications built on a microservices architecture. It tracks a single request as it propagates through various services by injecting correlation IDs (Trace ID and Span IDs) into network protocols (HTTP headers, Kafka events).

Purpose

To pinpoint performance bottlenecks, resolve latency issues, and debug failures in highly decoupled architectures. Without it, finding which specific service caused a user's checkout to fail is like finding a needle in a haystack.

2. Internal Working: Traces & Spans

Distributed tracing relies on Context Propagation. The system injects metadata into the thread context (MDC) and passes it downstream via HTTP headers (W3C Trace Context).

The Propagation Journey

Global Trace ID: 4bf92f3577b34da6
API Gateway
Generates Initial Trace ID
Span ID: A1
Parent: null
HTTP Header: traceparent: 00-4bf92f3577b34da6-A1-01
Payment Service
Span ID: B2
Parent: A1
Inventory Service
Span ID: C3
Parent: A1
All services asynchronously send their "Spans" to a central backend (like Jaeger, Zipkin, or Datadog), which reconstructs the tree using the Parent IDs.

3. Real-World Project Example

The 5-Second Checkout Delay

How tracing turns days of debugging into seconds.

The Problem

Users complain the "Place Order" button spins for 5 seconds before failing. The API Gateway logs show a 5000ms latency. Is it the Order Service? The Payment Gateway? A database lock in Inventory? The Fraud Check service? Without tracing, you have to SSH into 4 different servers, grepping logs for timestamps, hoping clocks are perfectly synced.

The Discovery

You open Jaeger/Datadog and search for requests taking > 4000ms. You open the Trace Waterfall. It visually shows:
- Gateway: 5000ms
   ↳ OrderService: 4980ms
      ↳ InventoryService: 15ms
      ↳ FraudService: 4950ms (TIMEOUT)

The Resolution

In 10 seconds, you immediately know the Fraud Service's 3rd-party API is experiencing a partial outage. You trigger a circuit breaker in the OrderService to bypass Fraud temporarily, restoring checkout speeds instantly.

4. 7 Key Concepts For senior-level depth

1

Trace ID

A globally unique identifier (usually 16 or 32 bytes hex) assigned to a single external request. Every log and metric generated by this request, across every microservice, shares this exact ID.

2

Span ID

Represents a single logical unit of work within a Trace (e.g., a DB query, a REST call). Every time a request crosses a network boundary (or a new internal method is wrapped), a new Span ID is generated.

3

MDC (Mapped Diagnostic Context)

A ThreadLocal map used by logging frameworks (Logback, Log4j). Tracing libraries automatically inject the Trace ID and Span ID into the MDC so that every log.info() statement prints the IDs automatically.

4

Context Propagation

The mechanism of moving the IDs between services. For HTTP, it's done via headers (e.g., X-B3-TraceId). For Kafka/RabbitMQ, it's injected into the message headers.

5

W3C Trace Context

Historically, everyone used different headers (Zipkin used B3, Datadog used x-datadog). W3C is the modern, vendor-neutral standard using the traceparent and tracestate headers.

6

OpenTelemetry (OTel)

The CNCF standard that merged OpenTracing and OpenCensus. It provides a unified set of APIs, SDKs, and tooling to generate and export telemetry data (Traces, Metrics, Logs) to any backend.

7

Sampling Strategies

Sending 100% of traces to Jaeger will crash Jaeger and slow down your app. You configure a Sampler (e.g., 10% of traffic, or Adaptive Sampling which captures 1% of successful requests but 100% of failed requests).

import io.micrometer.tracing.Span;
import org.springframework.stereotype.Service;

@Service
public class PaymentService {

    private final Tracer tracer;

    public PaymentService(Tracer tracer) {
        this.tracer = tracer;
    }

    public void processPayment() {
        // 1. Automatic: RestTemplate/WebClient calls automatically inject W3C headers

        Span newSpan = this.tracer.nextSpan().name("calculate-fraud-score");
        try (Tracer.SpanInScope ws = this.tracer.withSpan(newSpan.start())) {
            
            // Add custom tags for Datadog/Jaeger filtering

        } finally {
            // MUST end the span to record duration
            newSpan.end(); 
        }
    }

7. Senior Developer Answer

"Without Distributed Tracing, a microservices architecture is effectively a black box. If a user complains about a slow checkout, grepping through isolated logs across 5 different services is impossible.

To solve this, we rely on the OpenTelemetry (OTel) standard. When a request hits our API Gateway, it generates a unique Trace ID. As the request propagates—whether via synchronous HTTP calls using the W3C Trace Context headers, or via asynchronous Kafka events using record headers—every downstream service extracts that Context.

In Spring Boot 3, this is completely natively integrated via Micrometer Tracing. It seamlessly populates the MDC, ensuring our SLF4J logs are stamped with the Trace ID, allowing instant log correlation in Splunk. Simultaneously, it exports the 'Spans' (timing data) to Jaeger or Datadog, giving us visual waterfall charts to instantly pinpoint latency bottlenecks. Crucially, I always implement probability sampling (e.g., 5%) to ensure the tracing telemetry doesn't overwhelm our network or storage costs."

8. Quick Revision Notes

Trace vs Span: Trace = The entire user journey. Span = A single hop or method execution.
MDC (Mapped Diagnostic Context): ThreadLocal map. Used to inject TraceID into log files automatically.
W3C Trace Context: The universal standard HTTP headers (traceparent) for passing IDs.
OpenTelemetry (OTel): The CNCF standard replacing OpenTracing. Vendor-neutral telemetry.
Micrometer Tracing: The Spring Boot 3 successor to the deprecated Spring Cloud Sleuth.
Sampling: Recording only a percentage of traces (e.g. 10%) to prevent excessive overhead.
The Async Trap: Spawning new threads loses the MDC context. Must use Context Propagation wrappers.
Kafka Tracing: Inject Trace IDs into Kafka Message Headers, not payloads.
