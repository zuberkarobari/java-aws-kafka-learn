Database per Service

Mastering Polyglot Persistence, breaking the Distributed Monolith, and solving the complex challenges of Sagas and CQRS.

1. Definition

Simple English

You don't let your roommate reach directly into your wallet to pay for their stuff; they have to ask you for the money. Microservices shouldn't reach into each other's databases. They must ask for data via APIs.

Technical

An architectural pattern where each microservice's persistent data is private to that service and accessible only via its API. A service's database cannot be accessed directly by other services.

Purpose

To achieve true loose coupling. It prevents a schema change in Service A from crashing Service B. It also allows Polyglot Persistence—using the best database type (SQL, NoSQL, Graph) for each specific service's workload.

2. The Golden Rule of Microservices

Sharing a database between microservices creates a "Distributed Monolith." You get all the network latency of microservices with none of the independent deployment benefits.

3. Real-World Project Example

Modern E-Commerce Backend

How Polyglot Persistence optimizes different domains.

1. Order Service -> PostgreSQL

Orders require strict ACID compliance, complex state transitions, and absolute financial accuracy. PostgreSQL (a relational database) is the perfect tool for this job.

2. Product Catalog -> Elasticsearch / MongoDB

Products have highly variable attributes (a TV has 'resolution', a shirt has 'size'). Relational tables struggle with this. A document database handles flexible schemas, and Elasticsearch provides lightning-fast full-text search capabilities.

3. Recommendation Engine -> Neo4j

To calculate "Users who bought X also bought Y", you need to traverse complex relationships. A Graph Database (Neo4j) is exponentially faster for this than a SQL database executing massive recursive JOINs.

4. The 7 Challenges (And Solutions) For senior-level depth

1

Implementation Options

You don't always need separate physical database servers. You can implement this via: Private Tables (in a shared DB server), Private Schemas (in a shared DB server), or entirely Dedicated DB Servers per service.

2

Data Encapsulation

Service A cannot run a SQL query against Service B's tables. It must call Service B's REST API or subscribe to Service B's Kafka events. This prevents fragile, tangled data dependencies.

3

Distributed Transactions (Sagas)

Challenge: You cannot use ACID transactions (BEGIN/COMMIT) across two databases.
Solution: The Saga Pattern. A transaction is broken into local steps. If one step fails, compensating events (e.g., "Refund Payment") are fired to undo previous steps.

4

Cross-Service JOINs (CQRS)

Challenge: You can't write SELECT * FROM orders JOIN users.
Solution: Command Query Responsibility Segregation (CQRS). A separate service listens to events from both domains and builds a read-optimized, pre-joined "Materialized View" database.

5

API Composition

An alternative to CQRS for simpler JOINs. An API Gateway or "BFF" (Backend for Frontend) makes parallel API calls to the Order Service and User Service, stitches the JSON together in memory, and returns it to the client.

6

Eventual Consistency

Because data is distributed and synchronized via message brokers, the system abandons immediate consistency. Different databases will sync up "eventually." UIs must be designed to handle this (e.g., "Your order is processing...").

7

CDC (Change Data Capture)

To reliably publish events when a service's database updates (avoiding the Dual-Write problem), we use the Transactional Outbox Pattern paired with CDC tools like Debezium. Debezium reads the database transaction log directly and streams changes securely to Kafka.

 * ANTI-PATTERN: The Distributed Monolith (DO NOT DO THIS)
 * OrderService connects directly to UserService's database tables.
 * ------------------------------------------------------------- */
@Service
public class BadOrderService {
    // WRONG: OrderService has a repository mapped to a table owned by UserSvc
    private final UserRepository sharedUserRepository; 
    
    public void createOrder(Order order, Long userId) {
        User user = sharedUserRepository.findById(userId); // TIGHT COUPLING!
        // ...
    }
}


/* -------------------------------------------------------------
 * BEST PRACTICE: Database-per-Service (API / Event Integration)
 * OrderService asks UserService for data via an API client (or Kafka).
 * ------------------------------------------------------------- */
@Service
public class GoodOrderService {
    
    private final OrderRepository orderRepository; // Owns its own DB
    private final UserApiClient userClient; // Feign/WebClient to call UserSvc

    public GoodOrderService(OrderRepository repo, UserApiClient client) {
        this.orderRepository = repo;
        this.userClient = client;
    }

    public void createOrder(Order order, Long userId) {
        // 1. Fetch data via API (UserSvc encapsulates its DB)
        UserDTO user = userClient.getUserDetails(userId); 
        
        if (user.isActive()) {
            // 2. Save to our private database
            orderRepository.save(order);
            
            // 3. Emit Event to Kafka for other services (Saga/Choreography)
            // kafkaTemplate.send("order-created", order);
        }
    }
}

7. Senior Developer Answer

"The entire premise of microservices is independent deployability and fault isolation. If multiple services connect to the same database, you haven't built microservices; you've built a 'Distributed Monolith.' A schema change by Team A will inevitably crash Team B's application.

To solve this, we strictly enforce the Database-per-Service pattern. Every service wholly encapsulates its data. This enables Polyglot Persistence—allowing our Search service to use Elasticsearch while our Billing service uses PostgreSQL.

However, this architectural purity introduces immense data consistency challenges. We can no longer use simple SQL JOINs or ACID transactions. To query across domains, we implement CQRS to build read-optimized, event-driven views. To handle multi-step business flows, we implement the Saga pattern, utilizing Kafka and the Transactional Outbox pattern via Debezium to guarantee that local database commits and inter-service events achieve eventual consistency reliably."

8. Quick Revision Notes

The Rule: A service's database can ONLY be accessed by its own API. No direct DB sharing.
Anti-Pattern: "Distributed Monolith" occurs when DBs are shared, killing independent deployments.
Polyglot Persistence: The ability to mix Postgres, Mongo, and Redis based on the service's specific needs.
Saga Pattern: The replacement for distributed ACID transactions. Uses compensating events to rollback failures.
CQRS: Command Query Responsibility Segregation. Builds read-only database views to solve the "No JOINs" problem.
Eventual Consistency: The trade-off for loose coupling. Data will sync eventually, but not instantly.
Outbox Pattern: Guarantees an event is published to Kafka ONLY if the DB commit succeeds.
Strangler Fig: The strategy for migrating a monolithic DB to DB-per-service piece by piece.

9. Interview Answer Template

"The Database-per-Service pattern is the defining characteristic of a true microservices architecture. It mandates that a service's persistent data can only be accessed through its own API, preventing the severe tight-coupling of the 'Distributed Monolith' anti-pattern. This isolation allows teams to deploy independently and leverage Polyglot Persistence, choosing the optimal database engine for their specific domain.

However, this architectural purity removes the ability to execute cross-database SQL JOINs or distributed ACID transactions.

To overcome these limitations, we must embrace Eventual Consistency. We handle complex reads by implementing CQRS to aggregate data across services into materialized views. For distributed writes, we implement the Saga pattern, orchestrating multi-step business transactions via an event broker like Kafka, using compensating events to gracefully rollback operations if failures occur."
