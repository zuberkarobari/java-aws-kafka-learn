Service Discovery

Understanding Netflix Eureka, dynamic routing, client-side load balancing, and the Self-Preservation mode.

1. Definition

Simple English

It's a dynamic phonebook for your microservices. Instead of hardcoding "Service B lives at IP 192.168.1.5", Service A asks the Service Discovery server, "Hey, what's the current address for Service B?"

Technical

Service Discovery is a pattern where a dedicated registry (like Netflix Eureka or Consul) maintains a real-time list of available service instances and their network locations. Services register on startup and deregister/fail health checks on shutdown.

Purpose

To enable extreme elasticity. In cloud environments, instances scale up/down dynamically, changing IPs constantly. Service Discovery allows clients to find servers without manual configuration updates.

2. Internal Working: Netflix Eureka

The Service Discovery pattern involves three components: The Service Provider, the Service Registry (Eureka Server), and the Service Consumer. Here is how they interact.

Registration & Discovery Flow

Service Provider (Inventory)

1. Starts up: Sends its IP, Port, and Name (INVENTORY-SERVICE) to Eureka.
2. Heartbeat: Sends a ping every 30s saying "I am alive".

Eureka Server (Registry)

Service Name IP & Port
ORDER-SERVICE 10.0.1.5:8080
INVENTORY-SERVICE 10.0.1.6:8081
INVENTORY-SERVICE 10.0.1.7:8081
Maintains in-memory concurrent map of all healthy instances.

Service Consumer (Order)

3. Fetch Registry: Downloads the entire Eureka phonebook and caches it locally.
4. Client Load Balancing: Needs to call Inventory. Looks in local cache, finds 2 IPs. Picks one (Round Robin) and calls it directly.

3. Real-World Analogy

The Modern Yellow Pages

How microservices avoid hardcoded dependencies.

The Monolith Problem

Without Service Discovery, the Order Service code looks like this: restTemplate.getForObject("http://192.168.1.44:8080/api/inventory"). If the Inventory server crashes and is replaced by a new server with IP 192.168.1.88, the Order Service breaks until you manually rewrite the code and restart it.

The Discovery Solution

With Eureka, the Order Service code becomes: restTemplate.getForObject("http://INVENTORY-SERVICE/api/inventory"). When a new Inventory server spins up, it tells Eureka its new IP. Order Service automatically learns the new IP. No code changes, no restarts.

4. 7 Key Points For senior-level depth

1

Client-Side Load Balancing

Eureka does NOT route traffic like an API Gateway. It just gives the IPs to the caller. The caller (using Spring Cloud LoadBalancer or Ribbon) picks an IP and makes a direct point-to-point network call.

2

Self-Preservation Mode

If Eureka stops receiving heartbeats from *many* services at once, it assumes there is a network partition, not that all services died. It stops evicting instances from the registry to prevent catastrophic failure.

3

Local Caching

Eureka Clients fetch the registry and cache it locally. If the Eureka Server goes completely offline, microservices can still communicate using their cached IP addresses.

4

High Availability (HA)

Eureka Servers can run in clusters (Peer Awareness). They replicate their registries with each other. A client can connect to any node in the cluster.

5

Heartbeats (Renewals)

By default, an instance sends a heartbeat every 30 seconds. If Eureka doesn't receive a heartbeat for 90 seconds (3 missed beats), it evicts the instance from the registry.

6

Spring Cloud LoadBalancer

Netflix Ribbon is deprecated. Spring Boot 3 uses Spring Cloud LoadBalancer to intercept `@LoadBalanced` RestTemplates and swap the logical service name with a real IP.

7

Kubernetes vs Eureka

If you deploy to Kubernetes, K8s has built-in service discovery (Kube-DNS) and server-side load balancing (Services). Therefore, deploying Eureka inside Kubernetes is an anti-pattern; you should use `spring-cloud-kubernetes` instead.

6. Code Example

/* --- 1. EUREKA SERVER (The Registry) --- */
import org.springframework.boot.SpringApplication;
import org.springframework.boot.autoconfigure.SpringBootApplication;
import org.springframework.cloud.netflix.eureka.server.EnableEurekaServer;

@SpringBootApplication
@EnableEurekaServer // Turns this boot app into a Registry
public class EurekaServerApplication {
    public static void main(String[] args) {
        SpringApplication.run(EurekaServerApplication.class, args);
    }
}

/* server/application.yml:
server:
  port: 8761
eureka:
  client:
    register-with-eureka: false # Don't register with itself
    fetch-registry: false
*/

/* --- 2. EUREKA CLIENT (Order Service making a call) --- */
import org.springframework.cloud.client.loadbalancer.LoadBalanced;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.web.client.RestTemplate;

@Configuration
public class AppConfig {

    @Bean
    @LoadBalanced // Enables Spring Cloud LoadBalancer to intercept calls
    public RestTemplate restTemplate() {
        return new RestTemplate();
    }
}

// In the Service Class...
@Service
public class OrderService {
    private final RestTemplate restTemplate;

    public OrderService(RestTemplate restTemplate) {
        this.restTemplate = restTemplate;
    }

    public void checkStock() {
        // The URL uses the logical application name, NOT an IP address.
        // The @LoadBalanced interceptor asks Eureka for the IP behind the scenes.
        String url = "http://INVENTORY-SERVICE/api/inventory/status";
        String result = restTemplate.getForObject(url, String.class);
    }
}

7. Senior Developer Answer

"Service Discovery is a foundational pattern for building dynamic, elastically scalable distributed systems. By utilizing Netflix Eureka, we decouple services from static IP configurations. When a microservice scales out, it registers its dynamic IP with Eureka; when a consumer needs to communicate, it asks Eureka for the logical name, receives a cached list of IPs, and utilizes client-side load balancing (via Spring Cloud LoadBalancer) to distribute the load point-to-point without a centralized bottleneck.

A key architectural detail is Eureka's Self-Preservation Mode, which protects the system during network partitions by refusing to evict nodes if heartbeats drop drastically.

However, the landscape is shifting. If our deployment target is Kubernetes, deploying Eureka becomes a redundant anti-pattern. Kubernetes inherently manages service discovery via Kube-DNS and handles load balancing via its native Service abstraction. In those environments, I advocate for removing Eureka entirely and using spring-cloud-kubernetes to seamlessly integrate native K8s service discovery with Spring's @LoadBalanced clients or Feign."

8. Quick Revision Notes

Concept: Dynamic phonebook preventing hardcoded IPs.
Heartbeats: Services ping registry every 30s to stay 'UP'.
Self-Preservation: Stops evictions during network partitions.
Local Cache: Clients cache the registry so they survive Eureka downtime.
Client-Side LB: Consumer directly routes traffic, not a central proxy.
Config: @EnableEurekaServer for server, @EnableDiscoveryClient for nodes.
@LoadBalanced: Intercepts RestTemplate/WebClient to swap names for IPs.
Kubernetes: Don't use Eureka if deploying to K8s; use Kube-DNS.
