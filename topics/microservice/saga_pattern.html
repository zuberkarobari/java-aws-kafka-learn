Saga Pattern

Mastering Data Consistency across microservices: Choreography vs. Orchestration, Compensating Transactions, and Eventual Consistency.

1. Definition

Simple English

Imagine booking a vacation. You book a flight, then a hotel, then a rental car. If the rental car fails, you can't magically "rollback" the flight you already bought. You have to actively call the airline and say "cancel my ticket." That's a Saga.

Technical

The Saga pattern is an architectural pattern for managing distributed transactions. Instead of a single ACID transaction locking multiple databases, a Saga is a sequence of local transactions. Each local transaction updates the database and publishes an event to trigger the next local transaction.

Purpose

To maintain data consistency across multiple microservices without using synchronous, blocking protocols like Two-Phase Commit (2PC). If one step fails, the Saga executes Compensating Transactions to undo the preceding steps.

2. Internal Working: The Two Approaches

There are two ways to coordinate a Saga: Choreography (decentralized, event-driven) and Orchestration (centralized, command-driven). Both handle failures using Compensating Transactions.

3. Real-World Project Example

E-Commerce Order Fulfillment

Handling distributed failures gracefully.

1. Order Service (Success)

User clicks "Buy". The Order Service creates an order in PENDING state, commits the transaction to its PostgreSQL DB, and emits an OrderCreated event.

2. Payment Service (Success)

Listens to the event. Charges the credit card $100. Commits the payment record to its MongoDB. Emits a PaymentProcessed event.

3. Inventory Service (FAILURE & COMPENSATION)

Listens to the payment event. Tries to reserve the item, but it's out of stock. It emits an InventoryFailed event.

The Rollback: The Payment Service hears this and triggers a Refund (Compensating Transaction 1). The Order Service hears this and marks the Order as FAILED (Compensating Transaction 2).

4. 7 Key Concepts For senior-level depth

1

Local vs Global Transactions

A Saga abandons Global (Distributed) Transactions. Instead, each step is a perfectly isolated Local Transaction with strict ACID guarantees within its own database.

2

Compensating Transactions

You cannot execute a SQL `ROLLBACK` across microservices. If Step 3 fails, you must execute explicit business logic to "undo" Step 2 (e.g., If Step 2 was 'Charge $50', the compensation is 'Refund $50').

3

The Pivot Transaction

The point of no return in a Saga. If the Pivot transaction succeeds, the entire Saga is guaranteed to eventually succeed. (e.g., Verifying Inventory). Everything before it is compensatable.

4

Retryable Transactions

Transactions that occur after the Pivot transaction. They are guaranteed to succeed eventually, so if they fail (e.g., network error sending a receipt), they must be infinitely retried until they succeed.

5

Idempotency is Mandatory

Because network failures cause message brokers to retry deliveries (At-Least-Once delivery), every step in a Saga MUST be idempotent. Refunding $50 twice should only actually deduct $50 once.

6

Choreography (Pros & Cons)

Pros: Simple, no single point of failure, highly decoupled. Cons: Hard to monitor. If a Saga spans 10 services, tracking the current state of an order requires checking 10 databases or complex distributed tracing.

7

Orchestration (Pros & Cons)

Pros: Centralized logic. You have one state machine (the Orchestrator) that knows exactly where the transaction is. Easy to monitor and rollback. Cons: Single point of failure. Creates tighter coupling because the orchestrator must know about the APIs of all participating services.

6. Code Example: Choreography via Kafka

import org.springframework.kafka.core.KafkaTemplate;
import org.springframework.kafka.annotation.KafkaListener;
import org.springframework.stereotype.Service;

/* --- MICROSERVICE 1: ORDER SERVICE --- */
@Service
public class OrderService {

    public void createOrder(Order order) {
        // 1. Save local state (PENDING)
        order.setStatus("PENDING");
        repository.save(order);
        
        // 2. Trigger next step in Saga
        kafkaTemplate.send("order-created", new OrderCreatedEvent(order));
    }

    // Listen for COMPENSATION requests
    @KafkaListener(topics = "payment-failed")
    public void compensateOrder(PaymentFailedEvent event) {
        Order order = repository.findById(event.getOrderId());
        order.setStatus("CANCELLED");
        order.setReason("Payment Failed: " + event.getReason());
        repository.save(order);
        System.out.println("Saga Compensated: Order Cancelled.");
    }
}

/* --- MICROSERVICE 2: PAYMENT SERVICE --- */
@Service
public class PaymentService {

    @KafkaListener(topics = "order-created")
    public void processPayment(OrderCreatedEvent event) {
        try {
            // Attempt local transaction
            paymentGateway.charge(event.getUserId(), event.getAmount());
            
            // Saga continues forward...
            kafkaTemplate.send("payment-successful", new PaymentSuccessEvent(event));
            
        } catch (Exception e) {
            // Local transaction failed. Trigger COMPENSATION backward.
            kafkaTemplate.send("payment-failed", 
                new PaymentFailedEvent(event.getOrderId(), e.getMessage()));
        }
    }
}

7. Senior Developer Answer

"In a true Database-per-Service architecture, distributed transactions via Two-Phase Commit (2PC) are an anti-pattern. They create tight coupling, severe latency bottlenecks, and block resources. Instead, we use the Saga Pattern to maintain data consistency through Eventual Consistency.

A Saga breaks a global transaction down into a sequence of isolated local ACID transactions. If any local step fails, the system executes Compensating Transactions to reverse the preceding steps. For simpler flows, I prefer Choreography, where services react to domain events via Kafka. It's highly decoupled, though tracking the state requires robust Distributed Tracing. For complex state machines—like order fulfillment spanning 5+ services—I prefer Orchestration. We use a central orchestrator component to issue commands and handle failures explicitly.

Regardless of the approach, two technical practices are non-negotiable: First, to avoid the dual-write problem, events must be published using the Transactional Outbox pattern. Second, because message brokers guarantee at-least-once delivery, every downstream consumer and compensating transaction must be strictly idempotent to safely absorb network retries."

8. Quick Revision Notes

The Core Concept: A sequence of local ACID transactions updating isolated databases.
Why not 2PC? Two-Phase Commit is slow, locks resources, and isn't supported by NoSQL.
Compensation: Semantic "undo" operations (e.g., refunding a payment) when a step fails.
Choreography: Decentralized. Services emit events, others listen. Good for simple flows.
Orchestration: Centralized. A state machine explicitly commands services. Good for complex flows.
Idempotency: All steps must be safe to retry, as Kafka ensures At-Least-Once delivery.
Outbox Pattern: Required to guarantee a DB commit and a Kafka event publish occur atomically.
Pivot Transaction: The point of no return. Once successful, the Saga cannot be compensated.

9. Interview Answer Template

"In a Database-per-Service architecture, maintaining data consistency requires moving away from synchronous distributed transactions (like Two-Phase Commit) and adopting Eventual Consistency via the Saga Pattern.

A Saga breaks a business flow into a sequence of isolated local transactions. Each service updates its own database and publishes an event (via a broker like Kafka) to trigger the next step. If a downstream service fails—for example, an inventory check fails after a payment has been processed—the system executes a series of Compensating Transactions to reverse the preceding steps.

We can implement this via Orchestration (using a central controller) or Choreography (decentralized event listening). In either implementation, we must heavily utilize the Transactional Outbox pattern to solve the dual-write problem and ensure all consumer endpoints are idempotent to handle message retries gracefully."
