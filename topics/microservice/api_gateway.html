API Gateway

Mastering Spring Cloud Gateway: Reactive routing, cross-cutting concerns, rate limiting, and the BFF pattern.

1. Definition

Simple English

It's the ultimate front desk receptionist and bouncer for your microservices. Instead of clients talking to 50 different microservices on 50 different IPs, they talk ONLY to the Gateway. The Gateway checks their ID, applies limits, and routes them to the right room.

Technical

An API Gateway is a reverse proxy and the single entry point into a distributed system. Spring Cloud Gateway is built on Project Reactor and Spring WebFlux (Netty), offering a non-blocking, reactive architecture for high-throughput routing, filtering, and API composition.

Purpose

To centralize Cross-Cutting Concerns. It prevents every microservice from having to implement its own JWT validation, CORS rules, Rate Limiting, SSL termination, and malicious traffic filtering.

2. Internal Working: Reactive Routing

Spring Cloud Gateway uses three core concepts: Routes (the destination), Predicates (the if-condition to match a route), and Filters (modifying the request/response). It executes entirely on an asynchronous, non-blocking Netty server.

3. Real-World Project Example

Global Protection & Routing

How an enterprise protects its backend cluster.

1. Path Rewriting & Routing

A mobile app calls https://api.company.com/v1/orders/123. The Gateway strips the /v1 and routes the request to http://order-service:8080/orders/123, hiding internal infrastructure structure from the outside world.

2. Rate Limiting (DDoS Protection)

To protect the legacy Inventory DB, the Gateway applies a Redis-backed Token Bucket filter to the /inventory route, limiting requests to 100 per second per User IP. If exceeded, it returns HTTP 429 (Too Many Requests).

3. JWT Verification Offloading

Instead of making 50 different microservices download public keys and verify JWTs, the Gateway does it once. If the JWT is valid, the Gateway extracts the userId and injects it as an HTTP header (X-User-Id: 99) before routing to the microservice.

4. 7 Key Purposes For senior-level depth

1

Dynamic Routing & Load Balancing

It integrates with Service Registries (like Eureka/Consul). When routing to lb://ORDER-SERVICE, the gateway dynamically resolves healthy IPs and load-balances the traffic.

2

Cross-Cutting Concerns

Centralizes logic that applies everywhere: SSL Termination, CORS configuration, centralized logging/correlation IDs, and Security (AuthN/AuthZ).

3

Rate Limiting

Prevents abuse. Spring Cloud Gateway integrates with Redis to apply the Token Bucket Algorithm, limiting requests per IP, user, or endpoint.

4

Resilience (Circuit Breaking)

Integrates with Resilience4j. If a downstream service is struggling, the gateway can "trip the circuit" and redirect traffic to a fallback route or return a fast failure.

5

BFF Pattern

Backend For Frontend. You can deploy multiple gateways tailored for specific clients (one for Mobile, one for Web), optimizing payloads and routes for each specific client type.

6

Reactive Architecture

Unlike Netflix Zuul 1.x (which blocked threads), SC Gateway runs on Netty. This non-blocking, async architecture handles tens of thousands of concurrent connections with minimal memory overhead.

7

Payload Modification

Using the ModifyRequestBodyGatewayFilterFactory, the gateway can intercept the JSON body of a request or response, mutate the data (e.g., masking credit card numbers), and pass it along on the fly.

# application.yml configuration for Spring Cloud Gateway
spring:
  cloud:
    gateway:
      routes:
        # Route 1: Order Service via Service Discovery (Eureka)
        - id: order-service-route
          uri: lb://ORDER-SERVICE # 'lb' triggers load balancing
          predicates:
            - Path=/api/orders/**
          filters:
            # Strip the /api prefix before sending downstream
            - StripPrefix=1 
            # Add a custom header
            - AddRequestHeader=X-Routed-By, SC-Gateway

        # Route 2: Inventory Service with Rate Limiting
        - id: inventory-service-route
          uri: lb://INVENTORY-SERVICE
          predicates:
            - Path=/api/inventory/**
          filters:
            - name: RequestRateLimiter
              args:
                redis-rate-limiter.replenishRate: 10 # Tokens added per second
                redis-rate-limiter.burstCapacity: 20 # Max burst limit
            # Circuit Breaker Integration
            - name: CircuitBreaker
              args:
                name: inventoryCircuitBreaker
                fallbackUri: forward:/fallback/inventory # Route if service fails

# --- Global Custom Filter (Java) ---
@Component
public class LoggingGlobalFilter implements GlobalFilter, Ordered {
    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        log.info("Incoming request: {}", exchange.getRequest().getPath());
        // Execute Pre-logic, proceed down chain, then execute Post-logic
        return chain.filter(exchange).then(Mono.fromRunnable(() -> {
            log.info("Response code: {}", exchange.getResponse().getStatusCode());
        }));
    }
    
    @Override
    public int getOrder() { return -1; } // High priority
}

7. Senior Developer Answer

"An API Gateway is the edge layer of a microservices architecture. Instead of exposing 50 distinct microservices to the public internet, the Gateway acts as our single ingress point. This allows us to brutally centralize cross-cutting concerns.

In our enterprise setup, we use Spring Cloud Gateway because its underlying reactive Netty server handles massive concurrency without blocking threads, unlike the older Netflix Zuul. We offload JWT signature verification entirely to the Gateway. This means our downstream microservices don't waste CPU cycles validating tokens; they just trust the internal headers the Gateway injects after validation. We also heavily utilize Redis-backed Token Bucket filters at the Gateway layer to protect legacy databases from traffic bursts.

A common architectural debate is API Gateway vs Kubernetes Ingress vs Service Mesh. While a K8s Nginx Ingress handles basic path routing perfectly, it lacks the developer-friendly ecosystem to write custom Java logic (like complex payload mutation or integration with Spring Security). We typically use both: a K8s Ingress exposes the cluster to the internet, which immediately hands off the traffic to Spring Cloud Gateway to execute our complex, domain-specific security and routing logic."

8. Quick Revision Notes

Single Ingress: The one and only public entry point to your cluster.
Reactive/Non-blocking: Built on Project Reactor / Netty for high throughput.
Predicates: "If-conditions" matching paths, headers, or cookies to a Route.
Filters: Pre-filters (mutate request), Post-filters (mutate response).
Rate Limiting: Uses Redis to limit API calls (Token Bucket alg).
Security: Ideal place to validate JWTs and terminate SSL.
BFF Pattern: Creating separate Gateways for Mobile vs Web clients.
Anti-Pattern: DO NOT put business logic or heavy DB calls in the Gateway.
