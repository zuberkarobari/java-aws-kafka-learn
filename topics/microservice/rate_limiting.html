Mastering the algorithms behind API protection, handling DDoS attacks, and distributed token buckets using Redis.

1. Definition

Simple English

Imagine a popular rollercoaster. To prevent the ride from breaking and ensure everyone gets a turn, the operator only lets 10 people through the gate per minute. If you arrive as the 11th person, you are told to "come back later."

Technical

Rate limiting is a technique used to control the rate of traffic sent or received by a network. If the API request count exceeds a predefined threshold defined by the system owner, all excess calls are blocked and the client receives an HTTP 429 status code.

Purpose

To prevent Resource Exhaustion (DDoS attacks), mitigate "Noisy Neighbor" problems in multi-tenant architectures, and control operational costs (preventing infinite loops from racking up cloud bills).

3. Real-World Project Example

Payment Gateway Security

Protecting endpoints from brute-force and unexpected bot traffic.

The Threat

A malicious script discovers your POST /api/checkout endpoint. It starts sending 5,000 requests per second attempting to brute-force credit card numbers (Carding attack). Without limits, this overwhelms your database and racks up massive fees from your actual payment provider.

The Implementation

You configure an API Gateway (like Spring Cloud Gateway) to intercept all traffic. You apply a Redis-backed Token Bucket filter specifically for the /checkout route, keyed by the user's IP Address and Auth Token. You set the capacity to 5 requests, refilling at 1 request per minute.

The Result

A normal user clicking "Submit" twice accidentally is allowed through (using 2 tokens). The bot burns through its 5 tokens in one millisecond. The remaining 4,995 requests are immediately dropped at the Gateway layer with a 429 Too Many Requests, never touching your application servers.

4. The 5 Core Algorithms Crucial for System Design

1

Token Bucket

Tokens are added to a bucket at a fixed rate. Requests consume tokens. If empty, the request is dropped. Pros: Memory efficient, allows for short bursts of traffic. Cons: Requires tuning two parameters (capacity and refill rate).

2

Leaky Bucket

Requests enter a bucket (queue) and are processed ("leak") at a strictly fixed rate. If the queue is full, new requests are dropped. Pros: Smooths out traffic perfectly (used by Shopify). Cons: A burst of old requests fills the queue, blocking fresh requests.

3

Fixed Window Counter

Divides time into fixed windows (e.g., 12:00 to 12:01). Each window has a counter. Pros: Extremely easy to implement (Redis INCR). Cons: The "Spike" problem. An attacker can send 100 requests at 12:00:59 and 100 at 12:01:01, doubling the allowed rate.

4

Sliding Window Log

Instead of a counter, keeps a timestamp of every request in a sorted set (Redis ZSET). When a new request comes, removes old timestamps outside the window, then counts the remainder. Pros: Perfectly accurate. Cons: Terrible memory footprint.

5

Sliding Window Counter

A hybrid approach. It takes the previous Fixed Window's count, weights it based on how far we are into the current window, and adds the current window's count. Pros: Smooths out traffic spikes like the Log, but with the tiny memory footprint of the Fixed Window. Very common in enterprise systems.

Answer: Rate Limiting acts like a brick wall: if you exceed the limit, the server immediately drops the request and returns a 429 error. Throttling acts like a traffic jam: if you exceed the limit, the server slows down your request processing, placing it in a queue to be handled when capacity frees up.

Answer: In a multi-node architecture, if two API Gateways read a Redis counter simultaneously, they might both read '4', increment to '5', and write '5' back, effectively missing a count. To prevent this "Read-Modify-Write" race condition, you must use Redis Lua Scripts. Lua scripts execute atomically in Redis, ensuring the check and increment happen in a single, un-interruptible step.

6. Code Example: Bucket4j

RateLimitFilter.java
Copied! 
import io.github.bucket4j.Bandwidth;
import io.github.bucket4j.Bucket;
import io.github.bucket4j.ConsumptionProbe;
import io.github.bucket4j.Refill;
import jakarta.servlet.*;
import jakarta.servlet.http.HttpServletResponse;
import org.springframework.stereotype.Component;

import java.io.IOException;
import java.time.Duration;

/* Implementing a basic Token Bucket using the industry-standard Bucket4j library */
@Component
public class RateLimitFilter implements Filter {

    private final Bucket bucket;

    public RateLimitFilter() {
        // Rule: Refill 10 tokens every 1 minute
        Refill refill = Refill.intervally(10, Duration.ofMinutes(1));
        
        // Rule: Bucket capacity is 10 tokens maximum
        Bandwidth limit = Bandwidth.classic(10, refill);
        
        this.bucket = Bucket.builder()
                .addLimit(limit)
                .build();
    }

    @Override
    public void doFilter(ServletRequest request, ServletResponse response, FilterChain chain)
            throws IOException, ServletException {

        HttpServletResponse httpResponse = (HttpServletResponse) response;

        // Try to consume 1 token
        ConsumptionProbe probe = bucket.tryConsumeAndReturnRemaining(1);

        if (probe.isConsumed()) {
            // Success: Add headers and proceed
            httpResponse.addHeader("X-Rate-Limit-Remaining", String.valueOf(probe.getRemainingTokens()));
            chain.doFilter(request, response);
        } else {
            // Failed: Bucket empty. Reject request immediately.
            long waitForRefill = probe.getNanosToWaitForRefill() / 1_000_000_000;
            httpResponse.addHeader("X-Rate-Limit-Retry-After-Seconds", String.valueOf(waitForRefill));
            
            // Send HTTP 429 Too Many Requests
            httpResponse.setStatus(429);
            httpResponse.getWriter().write("Too many requests. Please try again later.");
        }
    }
}

7. Senior Developer Answer

"Rate Limiting is fundamentally about protecting backend resources from exhaustion, whether malicious (DDoS) or accidental (noisy neighbors). In modern architectures, we never implement rate limiting at the application level; it belongs strictly at the edge, typically inside the API Gateway.

The industry standard algorithm is the Token Bucket, favored for its memory efficiency and ability to handle sudden, acceptable bursts of traffic. For distributed systems running multiple gateway instances, local in-memory counters fail. We rely on a centralized Redis cluster to maintain state.

A critical engineering detail here is preventing race conditions. Because gateways process thousands of requests concurrently, simple read-modify-write loops in Redis will overwrite each other. To achieve accuracy at scale, we execute the Token Bucket logic inside an atomic Redis Lua Script, ensuring the entire evaluation and deduction happens in a single, thread-safe command. If limits are breached, we fail fast, returning an HTTP 429 along with the standard `X-RateLimit` headers to allow clients to back off gracefully."

8. Quick Revision Notes

Goal: Prevent resource exhaustion (DDoS) and manage operational costs.
HTTP Code: Returns 429 Too Many Requests when blocked.
Token Bucket: Adds tokens at fixed rate. Allows bursts. Very memory efficient.
Leaky Bucket: Smooths out traffic into a fixed flow. Bursts are discarded if queue full.
Fixed Window: Resets counter every minute. Flawed: allows 2x limit at window edges.
Redis + Lua: Required for distributed apps to execute limit checks atomically.
Keys: Usually keyed by User IP, API Key, or User ID.
Location: Should be placed at the Edge (API Gateway), before business servers.

The most common implementation is the Token Bucket algorithm, which provisions tokens at a steady rate and rejects requests when the bucket is empty, returning an HTTP 429 status code. It is highly favored because it handles bursty traffic smoothly while remaining memory-efficient.
