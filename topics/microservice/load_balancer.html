Horizontal Scaling

Load Balancer

Distributing network traffic efficiently across a pool of servers to ensure high availability, reliability, and scale.

1. Definition

Simple English

Imagine a popular supermarket. If everyone goes to one cashier, the line becomes massive and the cashier gets overwhelmed. A load balancer is the store manager standing at the front, directing new customers to whichever cashier is currently free.

Technical

A Load Balancer is a reverse proxy that sits between clients and servers. It receives incoming network or application traffic and intelligently distributes it across a cluster of backend servers using specific algorithms to optimize resource use and maximize throughput.

Purpose

Its primary purpose is to enable **Horizontal Scaling** (adding more servers rather than upgrading one big server) and to guarantee **High Availability** by routing traffic away from failed servers.

2. Internal Working

A Load Balancer operates by continuously monitoring the "health" of the backend servers. When a request arrives, it uses a routing algorithm (like Round Robin) to select a healthy backend node, forwards the request, receives the response, and sends it back to the client.

3. Real-World Project Example

Layer 7 Smart Routing

How modern load balancers do more than just pick a server.

1. SSL Termination

Decrypting HTTPS traffic is CPU-intensive. Instead of making all 10 backend servers do this work, the Load Balancer holds the SSL certificate. It decrypts the incoming HTTPS traffic once, and forwards plain HTTP traffic to the internal servers within the secure private network, saving immense CPU overhead.

2. Path-Based Routing (Layer 7)

An Application Load Balancer (ALB) can read the HTTP URL. If a user requests example.com/api/v1, it routes traffic to the API microservices. If they request example.com/images, it intelligently routes that specific request to a cluster of optimized static-file servers or an S3 bucket.

4. 7 Key Concepts For senior-level depth

1

Layer 4 vs Layer 7

Layer 4 (Transport): Balances based solely on IP and Port (TCP/UDP). Very fast. (e.g., AWS NLB). Layer 7 (Application): Balances based on HTTP headers, URLs, and cookies. Slower but much smarter. (e.g., AWS ALB).

2

Round Robin & Least Connections

Round Robin: Distributes requests sequentially (A, B, C, A, B...). Least Connections: Sends traffic to the server with the fewest currently active connections, better for variable-length requests.

3

IP Hash / Sticky Sessions

Hashes the client's IP address to ensure a specific user is always routed to the same backend server. Required for legacy apps holding local session state, but violates modern stateless microservice design.

4

Health Checks

The LB continuously sends ping requests (e.g., HTTP GET /health) to backend servers. If a server fails to respond 3 times in a row, it is removed from the routing pool until it recovers.

5

Active-Passive HA

To prevent the Load Balancer itself from being a Single Point of Failure (SPOF), you run two LBs. The Active one handles traffic. The Passive one monitors the Active one. If the Active dies, the Passive instantly takes over the IP address (using VRRP/Keepalived).

6

Hardware vs Software LBs

Hardware LBs (F5 BIG-IP) are expensive, physical appliances used in enterprise data centers. Software LBs (NGINX, HAProxy, AWS ELB) run on standard OS instances and are the backbone of modern cloud computing.

7

DNS Load Balancing vs Reverse Proxy LBs

DNS Load balancing (like Route53) returns a different IP address for google.com to different users globally. Reverse Proxy LBs (like NGINX) sit inside the local network and actively proxy the HTTP connection to backend machines. They are often used together.

6. Code Example: NGINX Config

nginx.conf
Copied! 
# Define the pool of backend servers
upstream backend_api_servers {
    # Algorithm: Least Connections (defaults to Round Robin if omitted)
    least_conn;
    
    # The internal private IPs of the actual application servers
    server 10.0.1.10:8080 max_fails=3 fail_timeout=30s;
    server 10.0.1.11:8080 max_fails=3 fail_timeout=30s;
    server 10.0.1.12:8080 max_fails=3 fail_timeout=30s;
    
    # Backup server only used if all primary servers fail
    server 10.0.1.13:8080 backup;
}

server {
    # Listen for HTTP traffic on port 80
    listen 80;
    server_name api.mycompany.com;

    # Layer 7 Routing block
    location / {
        # Forward the request to the upstream pool
        proxy_pass http://backend_api_servers;
        
        # Pass crucial headers to backend (so it knows the original client IP)
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    }
}

7. Senior Developer Answer

"A Load Balancer is critical for transitioning an architecture from vertical scaling to horizontal scaling. By distributing traffic across multiple instances, we ensure high availability and eliminate the application server as a Single Point of Failure.

In practice, we deploy Layer 7 load balancers (like AWS ALB or NGINX) to enable advanced path-based routing and SSL termination, which drastically reduces the CPU overhead on our backend nodes. We typically use a 'Least Connections' algorithm to distribute variable-length API requests efficiently.

Crucially, we must design our backend services to be completely stateless—storing sessions in Redis rather than memory—to avoid the anti-pattern of Sticky Sessions. Finally, to prevent the Load Balancer itself from becoming the SPOF, we deploy them in an Active-Passive High Availability pair using keepalived, ensuring seamless failover at the infrastructure edge."

8. Quick Revision Notes

Core Goal: Horizontal Scaling and High Availability.
SPOF: Single Point of Failure. Mitigated by Active-Passive LBs (Keepalived).
Layer 4 (L4): Transport layer. Fast, blindly routes by IP and Port (TCP/UDP).
Layer 7 (L7): Application layer. Inspects HTTP Headers/URLs for smart routing.
SSL Termination: LB decrypts HTTPS, saving backend CPU cycles.
Health Checks: LB constantly pings servers to avoid routing to dead nodes.
Algorithms: Round Robin (sequential) vs Least Connections (sends to least busy).
Sticky Sessions: IP Hash routing. An anti-pattern compared to stateless backends.
