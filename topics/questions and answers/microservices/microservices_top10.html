01
What are Microservices? How do they differ from Monoliths? What are the trade-offs?
FoundationMust KnowAlways Asked First
▾
This is your opening answer — frame it as a deliberate architectural trade-off, not "microservices are better." Interviewers at senior level want to see you acknowledge the cost.
  • 01
    Definition: Microservices is an architectural style where a large application is decomposed into small, independently deployable services — each owning a single bounded business capability, its own data store, and communicating over lightweight protocols (HTTP/gRPC/messaging). At JP Morgan, the Ops Intelligence system is 12 independently deployable services. At VISA, the payment notification pipeline is a separate service from reconciliation.
  • 02
    Monolith vs Microservices — Core Differences:
    MONOLITH: [UI → Business Logic → Data Access → Single DB] Deploy together, scale together, fail together MICROSERVICES: [Service A] → [Service B] → [Service C] ↓ ↓ ↓ [DB-A] [DB-B] [DB-C] Deploy independently, scale independently
  • 03
    Microservices Advantages: (1) Independent deployment — release Service A without touching Service B. (2) Independent scaling — scale only the reconciliation service during peak settlement, not the entire monolith. (3) Technology heterogeneity — Java for heavy processing, Go for high-throughput APIs. (4) Fault isolation — a bug in the notification service doesn't bring down payment processing. (5) Team autonomy — JP Morgan teams own their service end-to-end.
  • 04
    Microservices Costs (The Honest Answer): (1) Distributed systems complexity — network failures, partial failures, eventual consistency. (2) Operational overhead — 12 services to deploy, monitor, and debug vs 1. (3) Data consistency — no ACID transactions across services. (4) Latency — network calls vs in-process calls. (5) Testing complexity — integration testing 12 services is harder than testing 1 monolith.
  • 05
    When Microservices Make Sense: Large teams (Conway's Law — architecture mirrors team structure), different scaling requirements per domain, different release cadences per team, need for technology flexibility, domains with clear boundaries. When they DON'T make sense: small team (<10 engineers), startup MVP (operational overhead too high), tightly coupled domains where service boundaries are unclear.
  • 06
    JP Morgan Context — Why We Use Microservices: The Ops Intelligence system has 6 teams working on different domains simultaneously. Without microservices, team A's deployment would require coordinating with teams B through F. With independent services, each team deploys 8-10 times per week without affecting others. The operational overhead (Terraform, Jenkins pipelines, CloudWatch dashboards per service) is justified by the deployment independence at this scale.
Always acknowledge trade-offs. Saying "microservices are always better" is a red flag to senior interviewers. The right answer is: "microservices make sense WHEN the team size and domain complexity justify the operational overhead."
02
How do Microservices communicate? REST vs gRPC vs Messaging — when do you use each?
CommunicationSync vs AsyncFrequently Asked
▾
Show you know all three patterns and — critically — when NOT to use each. The sync vs async decision is what separates senior from mid-level answers.
  • 01
    Synchronous REST (HTTP/JSON): Simple, universal, human-readable, stateless. Best for: request-response queries where the caller needs the result immediately, public-facing APIs (external consumers), simple CRUD operations. At JP Morgan: the Event Query API (dashboards querying event history) uses REST — the dashboard needs the data immediately to render. Downside: tight coupling between caller and callee availability.
  • 02
    Synchronous gRPC (HTTP/2 + Protocol Buffers): Binary serialization (3-10x smaller payload than JSON), strongly-typed contracts via .proto files, bidirectional streaming, code generation for all languages. Best for: internal service-to-service calls where performance matters, streaming data, polyglot microservices. Downside: less human-readable, steeper learning curve, no native browser support. At JP Morgan: high-frequency internal enrichment calls use gRPC — 5x less bandwidth than equivalent JSON.
  • 03
    Asynchronous Messaging (Kafka/SQS): Decoupled producer and consumer — producer doesn't wait for or know about consumers. Best for: event notification, fire-and-forget operations, fan-out to multiple consumers, operations where the caller doesn't need the result immediately, temporal decoupling (consumer processes when ready). At VISA: payment events published to Kafka — notification, reconciliation, and audit services all consume independently. Adding a new consumer requires zero producer changes.
    SYNC: Producer ──→ Consumer (waits for response) → tight coupling ASYNC: Producer ──→ [Kafka Topic] ←── Consumer A ←── Consumer B (loose coupling) ←── Consumer C
  • 04
    Decision Framework: 
    Need immediate result?         → REST or gRPC
High-frequency internal calls? → gRPC (performance matters)
External / public API?         → REST (universally supported)
Fire-and-forget notification?  → Kafka/SQS
Multiple consumers same event? → Kafka (consumer groups)
Order guaranteed per entity?   → Kafka (key-based partitioning)
Simple work queue?             → SQS
Need event replay?             → Kafka (retention policy)
  • 05
    Choreography vs Orchestration: In async systems, choreography means each service reacts to events independently — no central coordinator. Orchestration means a central saga orchestrator directs the flow. Choreography: more decoupled, harder to trace. Orchestration: clear audit trail, easier to debug, single point of failure risk. At VISA I used orchestration for the payment reconciliation saga because financial workflows need a traceable audit trail of each step.
  • 06
    The Hybrid Approach (JP Morgan): We use all three. REST for external API (event dashboard). gRPC for internal enrichment service calls (performance-critical, same VPC). Kafka for event processing pipeline (async, multiple consumers, replay). SQS for simple inter-service command passing (trigger report generation). The key insight: no single protocol is right for everything — choose based on the coupling, latency, and reliability requirements of each interaction.

How do you handle distributed transactions across Microservices? Explain the Saga pattern.
Saga PatternDistributed TxSystem Design
▾
This is the most technically deep question in microservices interviews. Show you understand WHY 2PC fails and HOW Saga solves it with a real VISA example.
  • 01
    The Problem — No ACID Across Services: In a monolith, a single DB transaction spans all operations atomically — if one step fails, everything rolls back. In microservices, each service owns its own database. There is no single transaction manager that spans all DBs. If Service A commits and Service B fails, you have an inconsistent distributed state — this is the core challenge of distributed transactions.
  • 02
    Why NOT 2-Phase Commit (2PC): 2PC uses a coordinator that asks all participants to "prepare" (phase 1), then "commit" (phase 2). The problem: if the coordinator crashes between phase 1 and 2, ALL participants hold locks indefinitely — distributed deadlock. In microservices environments with network partitions and independent failure modes, 2PC is a liability, not a solution. It creates tight coupling and availability risk.
  • 03
    The Saga Pattern — Sequence of Local Transactions: A Saga decomposes a distributed transaction into a series of local transactions, each publishing an event to trigger the next step. If any step fails, compensating transactions undo the preceding steps.
    VISA Payment Settlement Saga: Step 1: ingest_payment_event → success → publish PaymentIngested Step 2: match_to_settlement → success → publish SettlementMatched Step 3: apply_to_ledger → FAIL → publish LedgerUpdateFailed ↓ COMPENSATE: Step 2C: unmark_settlement → reversal Step 1C: flag_event_for_review → creates audit record
  • 04
    Choreography vs Orchestration Sagas: Choreography: each service publishes events and subscribes to others — no central coordinator. Decoupled but hard to trace the overall workflow. Orchestration: a Saga Orchestrator directs all steps — service A complete? Tell B to go. B complete? Tell C. The orchestrator maintains state in a DB table. I used orchestration at VISA because financial workflows need a full audit trail of every step in the saga — the orchestrator's state table was the single source of truth for any settlement's progress.
  • 05
    Transactional Outbox — Ensuring Reliable Event Publishing: The biggest risk in a Saga: Service A commits to DB but the event publication to Kafka fails. The Saga is now in an inconsistent state. The Outbox pattern solves this — write the event to an outbox table in the SAME local DB transaction as the business update. A relay service reads the outbox and publishes to Kafka. If the DB commit succeeds, the event WILL be published eventually — guaranteed. 
    @Transactional  // single DB transaction
public void saveAndRecord(Settlement s) {
    settlementRepo.save(s);            // business entity
    outboxRepo.save(new OutboxMessage(/* event data */));
    // If commit succeeds → event guaranteed to publish
}
  • 06
    Compensating Transactions Must Be Idempotent: Compensations may be retried if they fail. Running a compensation twice must produce the same result as running it once. Design every compensation to check current state before applying: "if settlement is not already cancelled, cancel it; if it is already cancelled, return success." This idempotency requirement applies to both forward steps AND compensating steps in a Saga.
Sagas do NOT provide isolation — other sagas can read intermediate state. This is "BASE" (Basically Available, Soft-state, Eventually consistent) rather than ACID. Design your business flows to tolerate brief periods of inconsistency, or add domain-level locking (e.g., PENDING status that prevents concurrent settlement operations on the same entity).

What is Service Discovery? How does it work and how did you implement it?
Service DiscoveryInfrastructureCloud Native
▾
Connect this to your AWS/ECS experience — service discovery in cloud environments is done differently than the classic Eureka approach.
  • 01
    The Problem Service Discovery Solves: In microservices, service instances are ephemeral — they start, stop, scale, and move across hosts constantly. Hardcoding IP addresses breaks immediately. Service Discovery allows services to find each other dynamically by name rather than by fixed IP/port. Two patterns: Client-side discovery (client asks a registry for addresses, then load-balances) and Server-side discovery (infrastructure handles it — load balancer, DNS).
  • 02
    Client-Side Discovery — Eureka (Classic Spring Cloud): Services register with Eureka on startup. Clients query Eureka to get a list of available instances, then use a client-side load balancer (Ribbon/Spring Cloud LoadBalancer) to pick one. Works well but: Eureka is another service to operate, clients must implement discovery logic, works best in VM/bare-metal environments. 
    # Spring Cloud Netflix Eureka (older approach)
@EnableDiscoveryClient  // register with Eureka
# Client-side load-balanced call:
@LoadBalanced
WebClient.builder().baseUrl("http://enrichment-service").build();
  • 03
    Server-Side Discovery — AWS ECS + Route 53 (JP Morgan): In our ECS Fargate deployment, service discovery uses AWS Cloud Map + Route 53 private hosted zones. Each ECS service registers automatically with a DNS name: enrichment-service.ops.local. The Application Load Balancer handles load balancing. Services call each other by DNS name — zero client-side registry logic. AWS manages registration, deregistration, and health checking.
    Service A → DNS lookup "enrichment.ops.local" → AWS Cloud Map returns healthy instance IPs → ALB load-balances across healthy tasks → Service A never knows about individual task IPs
  • 04
    Kubernetes Service Discovery (for context): Kubernetes uses built-in DNS. Each Service object gets a stable DNS name: enrichment-service.namespace.svc.cluster.local. kube-proxy handles load balancing across matching pods. Services call each other by name — fully managed by Kubernetes. No external registry needed. This is why Kubernetes has largely replaced Eureka in modern microservices deployments.
  • 05
    Health Checking and Deregistration: Service discovery only works correctly if unhealthy instances are deregistered promptly. At JP Morgan, ECS health checks hit /actuator/health/readiness every 15 seconds. If 3 consecutive checks fail, ECS removes the task from the target group AND deregisters it from Cloud Map. New requests never reach the unhealthy instance. This automatic deregistration is essential — stale registrations cause cascading failures.
  • 06
    Eureka vs Cloud Map vs Kubernetes DNS — Decision: Use Eureka: legacy Spring Cloud ecosystem, on-premises deployment. Use AWS Cloud Map: ECS-native, fully managed, integrates with ALB/Route 53, minimal ops overhead — our choice at JP Morgan. Use Kubernetes DNS: Kubernetes-native, works out of the box. The trend is toward infrastructure-managed discovery (cloud-native or Kubernetes) over application-managed discovery (Eureka) — less client-side complexity, more reliable.

What is an API Gateway? What responsibilities does it own and how did you configure it?
API GatewayCross-CuttingInfrastructure
▾
The API Gateway is a critical architectural component. Show you know what belongs there vs what belongs in individual services — that boundary matters.
  • 01
    What an API Gateway Is: A single entry point for all client requests to the microservices ecosystem. It sits in front of all services and handles cross-cutting concerns so individual services don't have to. Without an API Gateway, every client would need to know the addresses of all services, and every service would have to implement authentication, rate limiting, and SSL termination redundantly.
    Client → [API Gateway] → /events → EventService → /enrichment → EnrichmentService → /reports → ReportService No cross-cutting code in individual services
  • 02
    API Gateway Responsibilities: (1) Authentication & Authorization — validate JWT tokens, check scopes. (2) Rate Limiting — per-client, per-endpoint throttling. (3) SSL Termination — decrypt HTTPS once at the edge. (4) Request Routing — route /v1/events to EventService v1, /v2/events to EventService v2. (5) Request/Response Transformation — header injection, protocol translation. (6) Logging & Tracing — assign correlation IDs to all incoming requests. (7) Circuit Breaking — at the edge level.
  • 03
    AWS API Gateway at JP Morgan: 
    # Terraform configuration:
resource "aws_api_gateway_rest_api" "ops_api" {
  name = "jpmorgan-ops-intelligence-api"
}
# JWT Authorizer validates every request:
resource "aws_api_gateway_authorizer" "jwt" {
  type                   = "JWT"
  identity_source        = "$request.header.Authorization"
  jwks_uri               = "https://auth.jpmorgan.com/.well-known/jwks"
}
# Usage plans for rate limiting per API key
  • 04
    What Does NOT Belong in the API Gateway: Business logic, domain validation, data transformation specific to one service's domain. The gateway is infrastructure — it should be stateless and domain-agnostic. If your gateway contains business rules, you've made it a service and now you have a deployment dependency problem. Keep it thin.
  • 05
    Spring Cloud Gateway (Code-Level): 
    @Bean
public RouteLocator routes(RouteLocatorBuilder b) {
    return b.routes()
        .route("events-v2", r -> r
            .path("/v2/events/**")
            .filters(f -> f
                .requestRateLimiter(c -> c.setKeyResolver(ipKeyResolver()))
                .addRequestHeader("X-Gateway-Version", "2"))
            .uri("lb://events-service-v2"))
        .build();
}
  • 06
    BFF Pattern (Backend for Frontend): An evolution of the API Gateway pattern — instead of one gateway for all clients, create specialized gateways for each client type. A Mobile BFF returns compact, mobile-optimized responses. A Web BFF returns richer, web-optimized responses. At JP Morgan, the internal operations dashboard (complex aggregated data) has a different BFF from the external API (simpler, stricter rate limits). Each BFF is owned by the team that owns the client it serves.

Explain the Circuit Breaker pattern in Microservices — states, fallbacks, and when it saved production.
Circuit BreakerResilienceProduction Critical
▾
This is the resilience question. Ground it in a real production scenario — what would have happened WITHOUT the circuit breaker is as important as what happened WITH it.
  • 01
    The Cascade Failure Problem It Prevents: In microservices, services call each other. If Service B becomes slow (3s instead of 100ms), all of Service A's threads accumulate waiting for B. Service A's thread pool exhausts. Service A becomes slow too. Now Service C (which calls A) exhausts its threads. The entire system cascades to failure because of one slow downstream service. The circuit breaker breaks this cascade.
    WITHOUT CB: A calls B (slow) → A threads exhaust → C calls A (slow) → cascade WITH CB: A calls B (slow) → circuit OPENS → A returns fallback in <1ms → C calls A → A responds normally → no cascade
  • 02
    Circuit Breaker States (Resilience4j): 
    CLOSED:    Normal operation. Calls pass through.
           Failures counted in sliding window.
           → If failure rate > 50%: trip to OPEN

OPEN:      All calls immediately rejected.
           Fallback returned. Downstream gets breathing room.
           → After 30s wait: move to HALF_OPEN

HALF_OPEN: Allow 3 probe calls through.
           → All 3 succeed: close circuit (service recovered)
           → Any fail: re-open circuit for another 30s
  • 03
    Fallback Strategy at JP Morgan: 
    @CircuitBreaker(name="enrichmentService", fallbackMethod="enrichFallback")
public Mono<Enrichment> enrich(String eventId) {
    return webClient.get().uri("/enrich/{id}", eventId)
        .retrieve().bodyToMono(Enrichment.class);
}
public Mono<Enrichment> enrichFallback(String id, Throwable t) {
    // Try stale Redis cache first, then safe defaults
    return redisCache.get("enrich:"+id)
        .switchIfEmpty(Mono.just(Enrichment.minimal(id)));
}
  • 04
    The VISA Incident — Where It Saved Us: The counterparty API slowed from 100ms to 3,000ms during a third-party maintenance window. Without a circuit breaker, this would have exhausted our notification service's thread pool in ~60 seconds, causing downstream cascade failure to the reconciliation pipeline. With the circuit breaker configured with a 2s slow-call threshold: after 5 consecutive slow calls, circuit opened. Fallback served minimal enrichment. Notification pipeline continued at 85% quality. Counterparty API recovered 20 minutes later, circuit closed automatically. Zero manual intervention.
  • 05
    Slow Call Threshold — Often Missed: Most engineers configure only failure rate threshold. But in financial systems, a slow response is just as dangerous as a failed one. Configure both: 
    resilience4j.circuitbreaker.instances.counterpartyApi:
  failureRateThreshold: 50       # 50% errors → OPEN
  slowCallRateThreshold: 50      # 50% slow calls → OPEN too
  slowCallDurationThreshold: 2s  # anything >2s = "slow"
  • 06
    Circuit Breaker vs Retry — When to Use Each: Retry handles transient failures (network blip, brief overload — retrying makes sense). Circuit Breaker handles sustained failures (service is genuinely down or overloaded — retrying makes it worse by adding more load). Combine both: Retry wraps individual calls (3 retries with backoff), Circuit Breaker monitors aggregate call health and trips when the service is fundamentally unhealthy. Retry inside Circuit Breaker, not the other way around.

Explain CQRS and Event Sourcing — how do they work together and when should you use them?
CQRSEvent SourcingAdvanced Pattern
▾
Know when NOT to use these — that's the senior signal. Many teams apply CQRS/ES unnecessarily and create massive complexity.
  • 01
    CQRS — Command Query Responsibility Segregation: Separate the model for reading data (Query) from the model for writing data (Command). Commands modify state. Queries read state. Why? Reads and writes often have very different requirements — queries may need denormalized views optimized for specific UIs; writes need consistency and validation. Separate models allows each to be optimized independently.
    CQRS Architecture: Client → [Command API] → CommandService → Write DB (normalized) → publishes event → Read DB (denormalized, optimized) Client → [Query API] → QueryService → Read DB (fast, indexed for queries)
  • 02
    Event Sourcing — Store Events, Not State: Instead of storing the current state of an entity (UPDATE events SET status='COMPLETED'), store every event that led to the current state (INSERT INTO event_log: PaymentInitiated, PaymentProcessed, PaymentCompleted). The current state is derived by replaying events. Benefits: complete audit trail, ability to replay history, temporal queries ("what was the account balance on March 1?"), event-driven integration.
  • 03
    CQRS + Event Sourcing Together at VISA: The settlement service used a simplified version: Commands (ApplySettlement, MarkDiscrepancy) were processed and events appended to an event log. Query projections were built from the event log — a real-time settlement summary projection for the dashboard, a separate discrepancy report projection. Adding a new report required only a new projection from the same event log — no new DB columns, no migration: 
    Event Log: [SettlementIngested, MatchedToCounterparty, LedgerApplied, NotificationSent]
     ↓
Projection 1: SettlementSummaryView (for dashboard)
Projection 2: DiscrepancyReportView (for compliance)
Projection 3: AuditTrailView (for regulators)
  • 04
    Eventual Consistency of Read Models: Read projections are updated asynchronously after write events. A user who posts a command will briefly see stale data in the query side. This is acceptable for most use cases ("Your request is processing") but not for all (bank balance shown immediately after deposit must be accurate). Design your UX to handle eventual consistency — show "updating..." states, provide refresh mechanisms, or use synchronous read-after-write for critical paths.
  • 05
    When to Use CQRS: Use CQRS when: read and write loads are vastly different (10:1 read:write ratio), different teams own read vs write concerns, multiple read representations needed for different consumers. Do NOT use CQRS for: simple CRUD with no complex read requirements, small teams where the complexity overhead exceeds the benefit, systems with <100K records where a single well-indexed table serves all queries. CQRS is not a default — it's a solution to a specific scaling problem.
  • 06
    When to Use Event Sourcing: Use it when: you need a complete immutable audit trail (financial systems, healthcare, legal), you need temporal queries ("what was the state at time T?"), you need event replay for rebuilding projections or debugging, you're building event-driven systems where events are the first-class integration mechanism. Do NOT use it: when the full audit trail is not required, when the event schema will change frequently (schema evolution is hard with event sourcing), for simple CRUD domains. At JP Morgan, we use event sourcing for the audit trail (regulatory requirement) but not for all services.

How do you ensure data consistency across Microservices? Explain eventual consistency and compensating transactions.
Data ConsistencyEventual ConsistencyDistributed Systems
▾
This is the hardest problem in microservices — show you've dealt with it in production, not just theory.
  • 01
    The Core Challenge: Each microservice owns its own database. When a business operation spans multiple services (create order → reserve inventory → charge payment → ship), there is no single ACID transaction. Any step can fail after previous steps have committed. You must design for partial failure and inconsistency as a normal operating condition, not an exception.
  • 02
    Eventual Consistency — The Accepted Model: In distributed systems, you trade strong consistency for availability. "Eventually consistent" means: if no new updates are made to an entity, eventually all reads will return the same value. The window of inconsistency might be milliseconds or seconds. Design your system to tolerate brief inconsistencies. At JP Morgan, operational dashboard data may be 1-2 seconds stale — acceptable. Payment settlement amounts are synchronously consistent — not acceptable to be stale.
  • 03
    Idempotency — The Foundation: Every operation that may be retried must be idempotent. In a distributed system with at-least-once delivery, the same message will arrive more than once. Idempotency ensures processing it twice produces the same result as once. Implementation: check a unique event_id before processing. If already processed, return success without re-processing: 
    // Check before processing — atomic with the write operation
if (processedEventRepo.existsById(eventId)) return; // already done
processBusinessLogic(event);
processedEventRepo.save(new ProcessedEvent(eventId));
  • 04
    Compensating Transactions vs Rollbacks: Traditional rollbacks undo a DB operation — only possible within a single transaction. In microservices with committed local transactions, you cannot rollback. Instead, you apply compensating transactions — new operations that create the opposite business effect. "Cancel the order" is a compensation for "create the order." Compensations are domain-meaningful operations, not technical undos.
    VISA Settlement Failure Compensation: Forward: ingest → match → apply_ledger → FAIL Compensate: revert_ledger (new business action, not a SQL ROLLBACK) unmark_settlement (marks record as unmatched) flag_for_review (creates audit record)
  • 05
    Outbox Pattern for Reliable Event Delivery: The dual-write problem: if you write to DB and publish to Kafka separately, one can succeed and the other fail. Solution — write to an outbox table in the same DB transaction as the business update. A Debezium CDC (Change Data Capture) connector or a polling relay reads the outbox and publishes to Kafka. Exactly-once delivery guaranteed: event published if and only if the DB transaction committed.
  • 06
    Domain-Level Locking via Status Fields: A pragmatic approach used at VISA for preventing concurrent modifications: add a status field to entities (PENDING, PROCESSING, COMPLETED, FAILED). A settlement in PROCESSING status is locked — other services attempting to modify it check the status first and reject if PROCESSING. This is application-level locking, not DB locking — works across services because all check the same shared DB status field. Lightweight and effective for low-contention scenarios.

How do you design Microservices for observability? Explain the three pillars and how you implemented them.
ObservabilityMetrics · Logs · TracesProduction
▾
Observability is what separates engineers who write microservices from engineers who OPERATE microservices. Show all three pillars with JP Morgan specifics.
  • 01
    Why Observability is Different in Microservices: In a monolith, you SSH into one server and grep one log file. In microservices, a single user request spans 5-10 services across 30-50 container instances. When something fails, you must be able to: see which service is slow (metrics), understand what happened in detail (logs), trace the full request path (distributed traces). Without all three, production debugging takes hours. With all three, it takes minutes.
  • 02
    Pillar 1 — Metrics (CloudWatch + Micrometer at JP Morgan): The 4 Golden Signals per service: Latency (p99 per endpoint), Traffic (requests/min), Errors (5xx rate), Saturation (pool utilization). Plus business metrics: events_processed/min, kafka_consumer_lag, circuit_breaker_state. CloudWatch Composite Alarms combining multiple signals — single metric alarms generate too much noise. Alert only when BOTH latency p99 > 500ms AND error rate > 2% simultaneously. This reduced false alarms by 70%.
  • 03
    Pillar 2 — Structured Logs (JSON + Correlation ID): Every log line is JSON with standard fields: timestamp, level, service, correlationId, eventId, durationMs, message. The correlation ID is generated at the API Gateway, propagated in every HTTP header and Kafka message header, and added to MDC for every log line. 
    // CloudWatch Logs Insights query:
fields @timestamp, level, service, correlationId, durationMs
| filter correlationId = "req-abc-123"
| sort @timestamp asc
// Full trace across 12 services in 5 seconds — was 20 minutes before
  • 04
    Pillar 3 — Distributed Tracing (Micrometer Tracing + X-Ray): Each request gets a Trace ID at entry. Every service call creates a Span — a timed unit of work with the trace ID. Spans are reported to AWS X-Ray. The result is a service map showing exact latency contribution of each service and DB call. At JP Morgan, X-Ray identified that a missing Aurora index was adding 300ms per request — invisible without tracing because the slow query was in a different service than the symptom appeared in.
  • 05
    MTTD Reduction — The 35% I Achieved: Before: ops team received alerts after users reported issues — MTTD ~20 minutes. After implementing structured logging + composite alarms + X-Ray tracing: MTTD dropped to ~13 minutes (35% reduction). Key changes: (1) Composite alarms fire faster and with less noise. (2) Structured logs let investigators find root cause in 2 minutes instead of 20. (3) X-Ray service map immediately shows which service is the bottleneck — no guessing.
  • 06
    Runbook Links on Every Alert: Technical observability is half the battle. The other half is ensuring on-call engineers know what to do when an alert fires. Every CloudWatch alarm at JP Morgan has a description field linking to a Confluence runbook: detection signals, diagnostic steps, resolution actions, escalation path. An alert without a runbook is an alert that causes a 2 AM panic. An alert with a runbook is an alert that gets resolved in 10 minutes.

What are the 12-Factor App principles? How do they guide Microservices design in production?
12-FactorCloud NativeProduction Best Practices
▾
Don't just list all 12 factors — focus on the 6 most important ones and connect each to a concrete JP Morgan or VISA practice. That's what interviewers remember.
  • 01
    What the 12-Factor App Is: A methodology for building software-as-a-service apps that are portable, scalable, and maintainable. Developed by Heroku engineers, it's now the de facto standard for cloud-native microservices. Every factor addresses a specific class of production problems. The 12 factors are: Codebase, Dependencies, Config, Backing Services, Build/Release/Run, Processes, Port Binding, Concurrency, Disposability, Dev/Prod Parity, Logs, Admin Processes.
  • 02
    Factor 3 — Config (Most Important in Practice): "Store config in the environment." Never hardcode DB URLs, API keys, or service addresses in code. Never commit secrets to version control. All config that differs between environments (dev/staging/prod) must come from environment variables or a config service. At JP Morgan: all secrets in AWS Secrets Manager, all environment-specific config in ECS task environment variables. The same Docker image runs in all 3 environments — only config changes.
  • 03
    Factor 6 — Processes (Stateless): "Execute the app as one or more stateless processes." Services must not store session state in memory between requests. All state goes to external stores (Redis, RDS, S3). This enables horizontal scaling — any instance can handle any request. At JP Morgan: all 6 EventService instances are interchangeable. ECS scales from 2 to 20 tasks during peak with zero coordination overhead because there's no shared in-memory state between instances.
  • 04
    Factor 9 — Disposability: "Maximize robustness with fast startup and graceful shutdown." Services should start fast (be ready quickly for auto-scaling), and shut down gracefully (finish in-flight requests before stopping). At JP Morgan: startup optimized to 8 seconds via Spring Boot lazy initialization + AOT. Graceful shutdown configured with server.shutdown=graceful + 30s timeout. ECS rolling deployments complete without dropping a single request.
  • 05
    Factor 11 — Logs (Treat as Event Streams): "Treat logs as event streams." Services write to stdout only — they do not manage log files or log rotation. The infrastructure (ECS → CloudWatch Logs) handles collection, storage, and routing. At JP Morgan: Spring Boot writes JSON-structured logs to stdout. ECS CloudWatch log driver captures them to log groups. No log configuration inside the service — just System.out.println via SLF4J, the platform handles the rest.

    Factor 10 — Dev/Prod Parity: "Keep development, staging, and production as similar as possible." The biggest source of "works on my machine" bugs is environment drift. At JP Morgan: same Docker image in all environments. Same Terraform modules (different variables). Same Kafka cluster type (different MSK instance size). Local development uses Docker Compose with real Kafka and PostgreSQL containers — not in-memory fakes. When local behaves like prod, production surprises disappear.
