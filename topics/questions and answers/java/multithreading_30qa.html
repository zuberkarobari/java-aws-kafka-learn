What is the difference between a Process and a Thread? Explain Thread lifecycle states.
FundamentalsMust Know
▾
  • 01
    Process vs Thread: A process is an independent program with its own memory space and OS resources. Threads are lightweight units of execution within a process, sharing the same heap, code, and file handles. Thread creation (~microseconds) is far cheaper than process creation (~milliseconds). At JP Morgan, our Spring Boot ECS containers are processes; Tomcat manages hundreds of threads within each.
  • 02
    Thread Lifecycle States:
    NEW           → created, Thread() called, not yet started
RUNNABLE      → start() called, eligible to run (may or may not be on CPU)
BLOCKED       → waiting to acquire a monitor lock (synchronized block)
WAITING       → indefinitely waiting (wait(), join(), LockSupport.park())
TIMED_WAITING → waiting with timeout (sleep(), wait(ms), join(ms))
TERMINATED    → run() completed or exception thrown
  • 03
    BLOCKED vs WAITING — Critical Difference: BLOCKED = competing for a monitor lock held by another thread (auto re-enters RUNNABLE when lock released). WAITING = explicitly waiting for a signal (Object.notify() required). In thread dumps: BLOCKED threads = lock contention problem; WAITING threads = normal coordination pattern.
  • 04
    Daemon Threads: Background threads that don't prevent JVM shutdown. Set with thread.setDaemon(true) BEFORE start(). JVM exits when only daemons remain. Never use daemon threads for tasks that must complete — JVM won't wait for them.
02
Explain the Java Memory Model — visibility, atomicity, and happens-before relationship.
JMMSenior TopicFrequently Asked
▾
  • 01
    The Problem the JMM Solves: CPUs have multi-level caches. Without rules, Thread A writing a variable might not be visible to Thread B reading from its CPU cache copy. The JMM defines when writes by one thread are guaranteed visible to another — the "happens-before" relationship.
  • 02
    Key Happens-Before Rules:
    // 1. Program order: each statement hb next in same thread
// 2. Monitor unlock hb next lock of same monitor
// 3. volatile write hb any subsequent read of same variable
// 4. Thread.start() hb any action in the started thread
// 5. All thread actions hb Thread.join() returning
// 6. Transitivity: A hb B and B hb C → A hb C
  • 03
    Visibility Problem:
    boolean flag = false; // not volatile
// Thread A:
flag = true;          // may stay in CPU register/cache
// Thread B:
while (!flag) {}      // may loop forever — sees stale cached value!
// Fix: volatile boolean flag = false;
  • 04
    Atomicity Problem: count++ is NOT atomic — it's read + increment + write = 3 separate operations. Two threads can both read the same value, increment it, and write back the same result — losing one increment. Fix: AtomicInteger.incrementAndGet() or synchronized.
  • 05
    Reordering: JVM and CPU can reorder instructions for performance as long as the single-thread result is unchanged. But reordering CAN break multi-threaded programs. The JMM's happens-before rules tell the JVM which reorderings are forbidden.

What is volatile? When should you use it and when is it NOT enough?
volatileJMMProduction Critical
▾
  • 01
    What volatile Guarantees: (1) Visibility: every write flushed to main memory; every read fetches from main memory not cache. (2) Ordering: prevents instruction reordering around the access. Does NOT provide atomicity for compound operations like check-then-act.
  • 02
    Classic Use Case — Stop Flag:
    private volatile boolean running = true; // volatile!
// Thread worker:
while (running) processNextEvent(); // reads from main memory every iteration
// Main thread:
running = false; // guaranteed visible to worker immediately
  • 03
    When volatile is NOT Enough — Check-Then-Act:
    volatile int count = 0;
// UNSAFE — volatile guarantees visibility but NOT atomicity of compound op:
if (count < 10) count++; // race condition! read and write not atomic
// Fix: AtomicInteger or synchronized block
  • 04
    Double-Checked Locking — volatile required:
    private static volatile Singleton instance; // volatile MANDATORY
public static Singleton getInstance() {
    if (instance == null) {
        synchronized (Singleton.class) {
            if (instance == null)
                instance = new Singleton(); // volatile prevents partial-init visibility
        }
    }
    return instance;
}
volatile on a reference does NOT make the referenced object's fields volatile. Writes to list contents are not thread-safe even if the list reference is volatile.

What is a race condition and a deadlock? Show examples of each and how to fix them.
Race ConditionDeadlockCritical
▾
  • 01
    Race Condition — VISA Example:
    int balance = 1000;
// Thread A: withdraw 700  |  Thread B: withdraw 700
if (balance >= 700) {    // A checks: true (1000≥700)
                           // B executes: balance=1000-700=300
    balance -= 700;       // A: 300-700=-400 → OVERDRAFT!
}
// Fix: synchronize the check-and-update as one atomic operation
  • 02
    Deadlock — Circular Lock Acquisition:
    // Thread A:                    Thread B:
synchronized(lockA) {         synchronized(lockB) {
    synchronized(lockB) {         synchronized(lockA) {
        // DEADLOCK — A holds A, wants B   // B holds B, wants A
    }                             }
}
  • 03
    Deadlock Prevention — Consistent Lock Order:
    // FIXED: BOTH threads acquire locks in the same order
synchronized(lockA) {
    synchronized(lockB) { /* no circular wait possible */ }
}
  • 04
    Detect Deadlocks in Production:
    // 1. jstack <pid> — look for "Found one Java-level deadlock"
// 2. /actuator/threaddump — same info via HTTP (no SSH needed)
// 3. Programmatic detection:
ThreadMXBean mx = ManagementFactory.getThreadMXBean();
long[] deadlocked = mx.findDeadlockedThreads(); // null if none
// At JP Morgan: scheduled health check calls this, alerts on-call if non-null
  • 05
    Livelock vs Starvation: Livelock: threads keep changing state in response to each other but make no progress (corridor stepping-aside problem). Starvation: a thread never gets CPU time because high-priority threads keep preempting it. Both are detectable in thread dumps — look for threads that are RUNNABLE but making no progress.

Explain AtomicInteger, AtomicReference, CAS, and LongAdder.
AtomicsLock-FreePerformance
▾
  • 01
    Compare-And-Swap (CAS): CAS(variable, expected, newValue): atomically check if variable equals expected; if yes update and return true; if no return false. All Java atomic classes are built on CAS — a single hardware instruction, much faster than locking (no kernel entry).
  • 02
    AtomicInteger:
    AtomicInteger counter = new AtomicInteger(0);
counter.incrementAndGet();             // atomic read-increment-write
counter.compareAndSet(5, 10);          // set 10 only if current == 5
counter.getAndAdd(3);                  // returns old, adds 3
// At JP Morgan: events-processed counter for CloudWatch metrics
  • 03
    AtomicReference — Atomic Swap of Object:
    AtomicReference<Config> ref = new AtomicReference<>(currentConfig);
// Atomically swap entire config object — no locks needed:
ref.compareAndSet(currentConfig, newConfig);
// At VISA: atomic swap of cached counterparty reference on refresh
  • 04
    LongAdder vs AtomicLong: Under high contention, many threads spinning on one AtomicLong creates overhead. LongAdder maintains a per-thread cell array — each thread updates its own cell, sum() combines them. LongAdder wins under high contention; AtomicLong is better when you need exact real-time value.
  • 05
    ABA Problem: CAS checks value equality, not identity. If value goes A→B→A between your read and CAS attempt, CAS succeeds incorrectly. Fix: AtomicStampedReference pairs value with a version stamp that must also match.

What is ThreadLocal? Use cases, dangers, and MDC pattern.
ThreadLocalContext PropagationMemory Leak Risk
▾
  • 01
    What It Does: Each thread has its own independent copy of the variable. No synchronization needed because there is no sharing. Classic uses: DB connection per thread, user context per request, SimpleDateFormat (not thread-safe — one per thread via ThreadLocal).
  • 02
    MDC Pattern (JP Morgan):
    // In WebFilter — set at request entry:
MDC.put("correlationId", correlationId);
MDC.put("eventId", eventId);
// Every log line in this thread now includes these fields automatically
log.info("Processing event"); // → {"correlationId":"abc","eventId":"123","msg":"Processing event"}
// In finally block — MANDATORY:
MDC.clear(); // thread pool reuse leaks MDC values to next request!
  • 03
    Memory Leak — The Biggest Danger: Thread pools REUSE threads. If you set a ThreadLocal value and never call remove(), the next request on the same thread sees the previous request's data. In Spring Security this can mean one user seeing another's auth context — a critical security leak. Always call remove() in a finally block.
  • 04
    ThreadLocal vs Reactor Context: ThreadLocal is thread-bound — doesn't work with WebFlux's event-loop model (one request may switch threads). Spring WebFlux uses Reactor Context for per-request data propagation instead.
ThreadLocal + thread pools = memory leak if remove() is not called. With 200 Tomcat threads handling 1,000 req/s, leaked ThreadLocals accumulate causing subtle bugs and OOM errors.

Explain synchronized — method vs block, static vs instance, reentrancy.
synchronizedMonitorFoundation
▾
  • 01
    Intrinsic Lock (Monitor): Every Java object has an intrinsic lock. synchronized acquires it — only one thread can hold a given object's intrinsic lock at a time. Others trying to acquire the same lock are BLOCKED until it is released.
  • 02
    Instance vs Static synchronized:
    public synchronized void deposit(int amount) { ... }  // locks on 'this'
public static synchronized void log(String msg) { ... } // locks on Class object
// Instance lock ≠ class lock — both can be held simultaneously by different threads
  • 03
    Prefer Private Lock Objects:
    private final Object balanceLock = new Object();
private final Object logLock     = new Object();
public void deposit(int amount) {
    synchronized(balanceLock) { balance += amount; }  // fine-grained
    synchronized(logLock)     { writeLog(amount); }   // can run concurrently
}
// Private lock: external code cannot accidentally synchronize on your lock
  • 04
    Reentrancy: Java intrinsic locks are reentrant — a thread that already holds a lock can acquire it again without deadlocking. This allows synchronized methods to call other synchronized methods on the same object safely.

Explain wait(), notify(), notifyAll() — producer-consumer pattern with correct usage.
wait/notifyProducer-ConsumerClassic
▾
  • 01
    The Rules: wait(), notify(), notifyAll() must be called inside a synchronized block on the same object. wait() releases the lock and suspends. notify() wakes one waiting thread. notifyAll() wakes all. Awakened thread must re-acquire the lock before proceeding.
  • 02
    Bounded Buffer — Producer-Consumer:
    class BoundedBuffer<T> {
    private final Queue<T> q = new LinkedList<>();
    private final int cap;

    synchronized void put(T item) throws InterruptedException {
        while (q.size() == cap) wait();  // WHILE — not IF!
        q.add(item);
        notifyAll();
    }
    synchronized T take() throws InterruptedException {
        while (q.isEmpty()) wait();       // WHILE — not IF!
        T item = q.poll();
        notifyAll();
        return item;
    }
}
  • 03
    Why WHILE not IF — Spurious Wakeups: A thread can wake from wait() without being notified (JVM/OS behavior — "spurious wakeup"). If you use IF, the thread proceeds without re-checking the condition. WHILE re-checks after waking. This is the golden rule — always loop around wait().
  • 04
    Modern Alternative: Use java.util.concurrent classes instead of raw wait/notify. BlockingQueue handles all the synchronization correctly and is far harder to misuse. Only use wait/notify for custom coordination patterns not covered by standard concurrent utilities.

What is a Semaphore? How does it differ from a mutex? Production use case.
SemaphoreRate LimitingConcurrency Control
▾
  • 01
    Semaphore vs Mutex: Mutex (binary semaphore): exactly 1 thread at a time. Counting semaphore: up to N threads simultaneously. Semaphore has no "owner" — one thread can acquire, a different thread can release. Maintain permit count: acquire() decrements (blocks if 0), release() increments.
  • 02
    JP Morgan — Limit Concurrent Enrichment Calls:
    private final Semaphore sem = new Semaphore(10); // max 10 concurrent

public Enrichment callEnrichmentApi(String id) throws InterruptedException {
    sem.acquire();                   // blocks if 10 already in-flight
    try {
        return enrichmentClient.fetch(id);
    } finally {
        sem.release();               // ALWAYS in finally
    }
}
  • 03
    tryAcquire — Non-Blocking:
    sem.tryAcquire()                           // false if no permits available
sem.tryAcquire(100, TimeUnit.MILLISECONDS) // wait up to 100ms
// Circuit-breaker-like: return fallback instead of blocking when overloaded

Explain CountDownLatch and CyclicBarrier — differences and when to use each.
CoordinationCountDownLatchCyclicBarrier
▾
  • 01
    CountDownLatch — One-Time Gate: Initialized with N. Threads call await() to block until count reaches 0. Any thread calls countDown() to decrement. Cannot be reset. Use for: "wait for N tasks to complete before proceeding."
    CountDownLatch latch = new CountDownLatch(3);
// 3 worker threads each call latch.countDown() when done
latch.await(); // main thread blocks until all 3 complete
// At VISA: wait for all 3 settlement partitions to validate before committing
  • 02
    CyclicBarrier — Reusable Meeting Point: N threads each call await() — all block until all N arrive, then released together. Can be reset and reused. Use for: repeated phases where all threads must complete phase N before any starts N+1.
    CyclicBarrier barrier = new CyclicBarrier(4, () -> mergeResults());
// 4 threads each process partition, await() at barrier
// When all 4 arrive: mergeResults() runs, all continue to next phase
  • 03
    Key Differences: CountDownLatch: one-time, counter decremented by any thread, waiting threads don't participate in count. CyclicBarrier: reusable, every participant calls await(), has optional barrier action. If one CyclicBarrier thread throws, the barrier is broken for all.

What are the happens-before guarantees for synchronized, volatile, and final fields?
JMMHappens-BeforeDeep Dive
▾
  • 01
    synchronized: Unlock of a monitor happens-before every subsequent lock of that same monitor. All writes done while holding a lock are visible to any thread that subsequently acquires the same lock. This is why synchronized works for visibility, not just atomicity.
  • 02
    volatile: A write to a volatile variable happens-before every subsequent read of that variable. Additionally — all writes before a volatile write are visible after the volatile read (a "memory fence"). volatile flushes more than just one variable.
  • 03
    final field safety — Safe Publication:
    class ImmutablePoint {
    final int x, y;
    ImmutablePoint(int x, int y) { this.x=x; this.y=y; }
    // After constructor completes, ANY thread reading x or y
    // sees correctly initialized values — NO synchronization needed.
    // This is the JMM guarantee for final fields.
}
  • 04
    this Escape — Common Bug: Never let a reference to this escape the constructor. If another thread observes the reference before the constructor finishes, it may see uninitialized final fields — breaking the final field safety guarantee.

What is ReentrantLock? How does it improve on synchronized?
ReentrantLockLocksAdvanced
▾
  • 01
    ReentrantLock Advantages Over synchronized: tryLock() without blocking, tryLock(timeout), lockInterruptibly() (interruptible wait), fairness setting (FIFO), multiple Condition variables per lock, monitoring (getWaitQueueLength, isHeldByCurrentThread).
  • 02
    Always Use try-finally:
    ReentrantLock lock = new ReentrantLock();
lock.lock();
try {
    // critical section
} finally {
    lock.unlock(); // MUST be in finally — exception will never release the lock
}
  • 03
    Condition Variables — Two-Condition Producer-Consumer:
    ReentrantLock lock = new ReentrantLock();
Condition notFull  = lock.newCondition();
Condition notEmpty = lock.newCondition();
// notFull.signal() wakes ONLY producers — unlike notifyAll() which wakes everyone
// More efficient: no spurious wakeups of wrong thread type
  • 04
    tryLock to Prevent Deadlock:
    if (lock1.tryLock()) {
    try {
        if (lock2.tryLock()) {
            try { /* both held */ } finally { lock2.unlock(); }
        }
    } finally { lock1.unlock(); }
}
// Can't get lock2? Release lock1 and retry — eliminates circular wait

Explain ReadWriteLock and StampedLock. When do they outperform ReentrantLock?
ReadWriteLockStampedLockRead-Heavy
▾
  • 01
    ReadWriteLock — Concurrent Reads: Multiple readers allowed simultaneously; writer needs exclusive access. Perfect for read-heavy data (reads don't conflict).
    // At VISA: counterparty cache — 99% reads, 1% invalidations
ReentrantReadWriteLock rwl = new ReentrantReadWriteLock();
public Counterparty get(String id) {
    rwl.readLock().lock();
    try { return cache.get(id); } finally { rwl.readLock().unlock(); }
}
public void invalidate(String id) {
    rwl.writeLock().lock();
    try { cache.remove(id); } finally { rwl.writeLock().unlock(); }
}
  • 02
    StampedLock — Optimistic Reads (Java 8):
    StampedLock sl = new StampedLock();
long stamp = sl.tryOptimisticRead(); // NO lock acquired
double x = this.x, y = this.y;
if (!sl.validate(stamp)) {           // did a write happen?
    stamp = sl.readLock();            // fallback to proper read lock
    try { x=this.x; y=this.y; } finally { sl.unlockRead(stamp); }
}
// Optimistic: zero lock overhead for reads when writes are rare
  • 03
    Lock Downgrading: Can downgrade write→read (acquire read while holding write, then release write). Cannot upgrade read→write without releasing read first (someone else may have modified meanwhile).

What causes thread contention? How do you diagnose and reduce it in production?
ContentionPerformanceProduction
▾
  • 01
    Common Blocking Causes: Lock contention (synchronized, ReentrantLock), blocking I/O (DB calls, HTTP in blocking mode), Thread.sleep(), wait(), blocking queue operations, class loading (static initializer blocks), GC stop-the-world pauses.
  • 02
    Diagnosis:
    // 1. Thread dump: jstack <pid> — look for chains of BLOCKED threads
// 2. /actuator/threaddump — same via HTTP (no SSH)
// 3. JConsole: live blocked thread count chart
// 4. Micrometer: executor.pool.size, executor.queue.size, executor.active
// 5. AWS X-Ray: identify exact operation consuming wall-clock time
  • 03
    Reducing Contention: Minimize critical section size (hold lock shortest time possible). Lock striping (N locks for N buckets — what ConcurrentHashMap does). Replace synchronized with lock-free atomics (AtomicInteger, ConcurrentHashMap). Use ReadWriteLock for read-heavy data.
  • 04
    JP Morgan Fix: Found a static synchronized method generating correlation IDs — every incoming request competed for it. Replaced with ThreadLocalRandom.current().nextLong() — no locking, each thread has its own generator. Throughput improved 15% under peak load.

Explain ExecutorService — configure ThreadPoolExecutor correctly for production.
ExecutorServiceThread PoolsMust Know
▾
Never use Executors.newCachedThreadPool() or newFixedThreadPool() in production — always configure your own ThreadPoolExecutor with bounded queue.
  • 01
    ThreadPoolExecutor Parameters:
    new ThreadPoolExecutor(
    10,                               // corePoolSize: always-alive threads
    50,                               // maximumPoolSize: max threads
    60, TimeUnit.SECONDS,             // keepAlive for threads above core
    new ArrayBlockingQueue<>(1000),   // BOUNDED queue — CRITICAL!
    new ThreadFactoryBuilder().setNameFormat("enrichment-%d").build(),
    new ThreadPoolExecutor.CallerRunsPolicy() // rejection: natural backpressure
);
  • 02
    Why NOT Executors.newFixedThreadPool(): Uses LinkedBlockingQueue with UNBOUNDED capacity. Under sustained load, tasks queue indefinitely → OOM. At JP Morgan an unbounded queue during a traffic spike caused heap to fill with 4GB of queued tasks. Always use bounded queues.
  • 03
    Rejection Policies:
    AbortPolicy        // throws RejectedExecutionException (default)
CallerRunsPolicy   // caller thread runs the task — natural backpressure ✓
DiscardPolicy      // silently drop — dangerous
DiscardOldestPolicy // drop oldest, add new
  • 04
    Thread Pool Sizing: CPU-bound: N_CPUs threads. I/O-bound (DB, HTTP): much more. Formula: N_threads = N_CPUs × (1 + wait_time/service_time). At JP Morgan with 4 vCPUs, avg DB wait 20ms, service 5ms: 4×(1+4)=20 threads.
  • 05
    Spring @Async — Always Configure Custom Executor:
    @Bean
public Executor taskExecutor() {
    ThreadPoolTaskExecutor e = new ThreadPoolTaskExecutor();
    e.setCorePoolSize(10); e.setMaxPoolSize(50); e.setQueueCapacity(500);
    e.setThreadNamePrefix("async-");
    e.setRejectedExecutionHandler(new ThreadPoolExecutor.CallerRunsPolicy());
    e.initialize(); return e;
}
// @Async without this uses SimpleAsyncTaskExecutor — creates thread per call!

What is ForkJoinPool and work-stealing? When does it outperform a regular thread pool?
ForkJoinPoolWork-StealingParallel Streams
▾
  • 01
    Work-Stealing Architecture: Each thread has its own deque. Threads push/pop from their own deque. When idle, a thread STEALS tasks from the tail of another thread's deque. Minimises contention — no single global queue bottleneck. Better CPU utilisation when tasks vary in size.
  • 02
    RecursiveTask Pattern (Divide-and-Conquer):
    class SumTask extends RecursiveTask<Long> {
    protected Long compute() {
        if (size <= THRESHOLD) return sumDirectly();
        SumTask left  = new SumTask(lo, mid);
        SumTask right = new SumTask(mid, hi);
        left.fork();                        // push to own deque async
        return right.compute() + left.join(); // current thread handles right
    }
}
  • 03
    Parallel Streams Use commonPool — Be Careful:
    // Uses ForkJoinPool.commonPool() — shared with everything!
list.parallelStream().map(this::expensiveOp).collect(toList());
// For I/O-bound parallel work, use a custom ForkJoinPool:
new ForkJoinPool(20).submit(
    () -> list.parallelStream().map(this::ioOp).collect(toList())).get();
  • 04
    When to Use: CPU-bound divide-and-conquer (sorting, tree traversal, parallel stream processing). NOT for I/O-bound tasks — blocking prevents work-stealing benefit. Most business service code: use ThreadPoolExecutor.

How does ScheduledExecutorService work? Production use cases.
SchedulingScheduledExecutorProduction
▾
  • 01
    API:
    ScheduledExecutorService s = Executors.newScheduledThreadPool(2);
s.schedule(task, 5, TimeUnit.SECONDS);              // once after 5s
s.scheduleAtFixedRate(task, 0, 10, TimeUnit.SECONDS);   // every 10s (fixed clock)

  • 02

  • 03
    JP Morgan Production Uses:
    // AWS Secrets Manager refresh every 24h
s.scheduleWithFixedDelay(this::refreshSecrets, 24, 24, TimeUnit.HOURS);
// Kafka consumer lag metric push every 30s
s.scheduleAtFixedRate(this::publishLagMetrics, 0, 30, TimeUnit.SECONDS);
// Outbox relay: poll for unpublished events
s.scheduleWithFixedDelay(outboxRelay::run, 0, 5, TimeUnit.SECONDS);
  • 04
    Spring @Scheduled:
    @Scheduled(fixedDelay = 30000)         // 30s after last completion
@Scheduled(fixedRate = 60000)          // every 60s
@Scheduled(cron = "0 0 2 * * *")      // daily at 2 AM
public void refreshCache() { ... }      // requires @EnableScheduling

Future vs CompletableFuture — what problems does CompletableFuture solve?
CompletableFutureFutureJava 8
▾
  • 01
    Future Limitations:
    Future<String> f = executor.submit(() -> fetchData());
String result = f.get(); // BLOCKS caller thread — defeats async purpose
// No chaining, no callbacks, no combine, no functional exception handling
  • 02
    CompletableFuture — Non-Blocking Pipeline:
    CompletableFuture.supplyAsync(() -> fetchData(), executor)
    .thenApply(data -> transform(data))
    .thenApplyAsync(data -> enrich(data), enrichExecutor)
    .thenAccept(result -> store(result))
    .exceptionally(ex -> { log.error("Failed", ex); return null; });
// Caller thread never blocks — callbacks fire when ready
  • 03
    Combining Multiple Futures (JP Morgan):
    CompletableFuture<A> a = CompletableFuture.supplyAsync(() -> serviceA.get(), exec);
CompletableFuture<B> b = CompletableFuture.supplyAsync(() -> serviceB.get(), exec);
CompletableFuture.allOf(a, b)
    .thenRun(() -> process(a.join(), b.join()));
// a and b run in parallel — latency = max(a,b) not a+b
  • 04
    Always Specify an Executor: supplyAsync without executor uses commonForkJoinPool — shared, potentially starved. Always pass your own: supplyAsync(task, myExecutor).
  • 05
    Exception Handling:
    cf.exceptionally(ex -> defaultValue)          // recover
cf.handle((result, ex) -> ex!=null ? fallback : process(result)) // both cases
cf.whenComplete((r, ex) -> logMetric(r, ex))  // side-effect only

How do you handle exceptions in thread pools and CompletableFuture chains?
Exception HandlingAsync ErrorsProduction Critical
▾
  • 01
    Exceptions in Runnable Are Silently Swallowed:
    executor.submit(() -> { throw new RuntimeException("error"); });
// Exception DISAPPEARS unless you call future.get()
// future.get() wraps it in ExecutionException
  • 02
    UncaughtExceptionHandler — Safety Net:
    ThreadFactory factory = r -> {
    Thread t = new Thread(r);
    t.setUncaughtExceptionHandler((thread, ex) -> {
        log.error("Uncaught in thread {}", thread.getName(), ex);
        metrics.increment("thread.uncaught.exceptions");
    });
    return t;
};
  • 03
    CompletableFuture Exception Propagation:
    cf.supplyAsync(() -> fetch())
  .thenApply(r -> transform(r))   // exception skips remaining stages
  .exceptionally(ex -> {
      log.error("Failed", ex.getCause()); // getCause() unwraps CompletionException
      return defaultValue;
  });
  • 04
    Production Error Boundary Pattern:
    private <T> CompletableFuture<T> withBoundary(Supplier<T> task, T fallback, String op) {
    return CompletableFuture.supplyAsync(task, executor)
        .exceptionally(ex -> {
            metrics.increment("async.errors", Tags.of("op", op));
            log.error("Async op {} failed", op, ex.getCause());
            return fallback;
        });
}

Implement non-blocking retry with exponential backoff using CompletableFuture.
CompletableFutureRetryProduction Pattern
▾
  • 01
    Non-Blocking Retry:
    public <T> CompletableFuture<T> retryAsync(
        Supplier<CompletableFuture<T>> supplier, int retries, long delayMs) {
    return supplier.get().exceptionallyCompose(ex -> {
        if (retries == 0) return CompletableFuture.failedFuture(ex);
        CompletableFuture<T> result = new CompletableFuture<>();
        scheduler.schedule(
            () -> retryAsync(supplier, retries - 1, delayMs * 2)
                    .whenComplete((v, e) -> { if (e!=null) result.completeExceptionally(e); else result.complete(v); }),
            delayMs, TimeUnit.MILLISECONDS);
        return result;
    });
}
  • 02
    In Practice — Use Resilience4j: Building retry from scratch is instructive for interviews but in production always use Resilience4j — handles jitter, exception classification, metrics, and circuit breaker integration out of the box.

How does ConcurrentHashMap work? How is it different from HashMap and Hashtable?
ConcurrentHashMapInternalsMust Know
▾
  • 01
    Three Generations: HashMap: not thread-safe, null keys/values. Hashtable: synchronized on every method — one global lock, terrible performance. ConcurrentHashMap: fine-grained locking — only the affected bucket is locked; all other buckets fully concurrent.
  • 02
    Java 8 Internals: Array of Node buckets. ≤8 entries: linked list. >8: TreeNode (red-black tree, O(log n)). Writes use CAS + synchronized on bucket head only. Reads are lock-free. Resize is incremental and concurrent.
  • 03
    Atomic Compound Operations:
    map.computeIfAbsent("key", k -> new ArrayList<>()).add(event); // atomic
map.merge("eventType", 1L, Long::sum);                         // atomic counter
map.computeIfPresent("key", (k, v) -> updateValue(v));           // atomic update
  • 04
    What It Does NOT Guarantee: Compound operations ACROSS multiple keys are not atomic. if(map.contains(k)) map.put(k,v) — race condition. Use putIfAbsent() or computeIfAbsent() for atomic check-and-insert.
  • 05
    JP Morgan — Per-Event-Type Frequency Counter:
    private final ConcurrentHashMap<String, AtomicLong> counts = new ConcurrentHashMap<>();

    counts.computeIfAbsent(type, k -> new AtomicLong()).incrementAndGet();

ConcurrentLinkedQueue — lock-free thread safety via CAS.
Lock-FreeCAS QueueNon-Blocking
▾
  • 01
    Lock-Free via CAS: Linked-node structure. head and tail are AtomicReference fields. Enqueue: CAS on tail.next. Dequeue: CAS on head. No locks — threads retry CAS if they lose the race. Non-blocking: a thread that fails CAS simply retries, never suspends.
  • 02
    Key Properties: Unbounded. FIFO. size() is O(n) — use isEmpty() instead. No blocking operations — use BlockingQueue if you need waiting producers/consumers. Best for: multi-producer multi-consumer event buses where blocking is unacceptable.
  • 03
    CLQ vs BlockingQueue Decision: ConcurrentLinkedQueue: producers must never block (fire-and-forget, event logging). BlockingQueue: producers should apply backpressure when consumers are slow (thread pool work queues). CLQ grows unboundedly — always monitor size in production.

Implement a thread-safe bounded blocking queue from scratch.
CodingDesignLive Coding Q
▾
Live-coding question asked at PhonePe and Razorpay. Tests deep understanding of ReentrantLock and Condition variables.
  • 01
    ReentrantLock + Two Conditions:
    public class BoundedBlockingQueue<T> {
    private final Object[] items;
    private int head = 0, tail = 0, count = 0;
    private final ReentrantLock lock = new ReentrantLock();
    private final Condition notFull  = lock.newCondition();
    private final Condition notEmpty = lock.newCondition();

    public BoundedBlockingQueue(int cap) { items = new Object[cap]; }

    public void put(T item) throws InterruptedException {
        lock.lock();
        try {
            while (count == items.length) notFull.await(); // WHILE!
            items[tail] = item;
            tail = (tail + 1) % items.length;
            count++;
            notEmpty.signal(); // wake one consumer only
        } finally { lock.unlock(); }
    }

    @SuppressWarnings("unchecked")
    public T take() throws InterruptedException {
        lock.lock();
        try {
            while (count == 0) notEmpty.await();        // WHILE!
            T item = (T) items[head];
            items[head] = null;                        // prevent GC leak
            head = (head + 1) % items.length;
            count--;
            notFull.signal();  // wake one producer only
            return item;
        } finally { lock.unlock(); }
    }
}
  • 02
    Why signal() not signalAll(): Separate Condition variables mean notEmpty.signal() wakes ONLY consumers, notFull.signal() wakes ONLY producers. No spurious wakeups of wrong thread type. More efficient than synchronized+notifyAll() which wakes everyone.

    Immutable = No Synchronization Needed: If state cannot change after construction, there is nothing to synchronize. No race conditions, no visibility problems, no deadlocks. 10 concurrency quick-fire answers for closing interview rounds.
Quick FireClosing RoundMemorize
▾
20–30 seconds each — memorize these. They appear as rapid-fire questions at the end of every concurrency round.
  • Q1
    sleep() vs wait(): sleep() pauses thread, does NOT release locks. wait() pauses thread AND releases the intrinsic lock. wait() must be called inside synchronized; sleep() can be called anywhere.
  • Q2
    Can you start a thread twice? No — calling start() twice throws IllegalThreadStateException. Create a new Thread instance if you need to run the task again.
  • Q3
    Exception in run(): Thread terminates. Exception goes to UncaughtExceptionHandler if set, otherwise default handler prints it. Does NOT propagate to the thread that called start(). Use Future.get() to receive exceptions from submitted tasks.
  • Q4
    Thread.yield(): Hints to scheduler that current thread is willing to yield CPU. Scheduler may ignore it. Doesn't release locks. Rarely useful in production — don't use it.
  • Q5
    Thread.join() happens-before: All actions in thread T happen-before Thread.join(T) returns. After join(), you can safely read any value written by that thread without additional synchronization.
  • Q6
    synchronized vs Lock: synchronized: simpler, JVM-managed, can't interrupt while waiting, no tryLock. Lock: explicit unlock, tryLock with timeout, interruptible waiting, fairness, multiple Conditions. synchronized for simple cases; Lock for advanced control.
  • Q7
    Memory barrier: CPU instruction preventing reordering of memory operations. volatile read/write and lock acquire/release insert memory barriers. Ensures all prior writes are visible after the barrier.
  • Q8
    synchronized inheritance: synchronized is NOT inherited. Overriding method does not automatically get the lock — must explicitly declare synchronized. Common source of subtle thread-safety breaks in subclasses.
  • Q9
    What is Exchanger? Synchronization point where exactly two threads swap objects. Each presents an object and receives the other's. Used for pipeline buffer swapping. Rare in typical backend services.
  • Q10
    Three rules for thread-safe code: (1) Don't share mutable state — use immutable objects or ThreadLocal. (2) When you must share, synchronize ALL accesses — both reads AND writes. (3) Prefer java.util.concurrent over hand-rolled synchronization — BlockingQueue, ConcurrentHashMap, AtomicInteger cover 95% of needs.
