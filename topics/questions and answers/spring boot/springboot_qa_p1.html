What is Spring Boot and how does it differ from the Spring Framework?
CoreMust Know
▾
Frame this as "problem it solves" — Spring Boot's value is eliminating boilerplate configuration.
  • 01
    Core Distinction: Spring Framework is a comprehensive IoC container requiring significant XML or Java configuration to wire components. Spring Boot is an opinionated layer on top providing auto-configuration, embedded servers, starter dependencies, and production-ready defaults — enabling production-grade apps with minimal setup. You get a running app with just a main() method and a starter dependency.
  • 02
    The 3 Core Pillars: (1) Auto-Configuration — scans classpath and conditionally creates beans. Add spring-boot-starter-data-jpa? You automatically get DataSource, EntityManagerFactory, and TransactionManager. (2) Starter POMs — curated dependency sets with compatible versions eliminating dependency hell. (3) Embedded Servers — Tomcat/Netty embedded in the JAR; deployment is just java -jar app.jar.
  • 03
    @SpringBootApplication Unpacked: 
    @SpringBootConfiguration   // marks class as config source
@EnableAutoConfiguration   // triggers auto-config magic
@ComponentScan             // scans current package + sub-packages
// All three composed into @SpringBootApplication
  • 04
    Auto-Configuration Mechanism: @EnableAutoConfiguration reads META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports from every JAR on classpath. Each listed class uses @ConditionalOnClass, @ConditionalOnMissingBean, etc. These conditions fire only when relevant dependencies exist and no user beans override — your beans always take precedence.
  • 05
    Fat JAR Deployment: Spring Boot's Maven/Gradle plugin repackages everything — dependencies, embedded server, custom classloader — into one fat JAR. At JP Morgan, our ECS Fargate tasks run java -jar app.jar directly from the Docker image. No external Tomcat needed. This makes container deployment dramatically simpler than deploying WARs to standalone servers.
  • 06
    JP Morgan Production Value: Opinionated defaults match 90% of production needs. Actuator provides out-of-box observability. Spring Security, Spring Data, Spring Kafka all auto-configure via starters. The team writes business logic — not plumbing code. We provisioned 12 microservices in the JP Morgan engagement without writing a single XML config file.

Explain the Spring Boot startup sequence — from main() to ApplicationReadyEvent.
LifecycleSenior Topic
▾
This proves you understand Spring Boot at the internals level. Walk through each step with confidence.
  • 01
    Step 1 — SpringApplication.run(): Detects application type (SERVLET/REACTIVE/NONE), loads SpringApplicationRunListeners, fires ApplicationStartingEvent. Run listeners handle logging framework initialization, banner printing, and any pre-context setup hooks.
  • 02
    Step 2 — Environment Preparation: Creates ConfigurableEnvironment, loads property sources in priority order: command-line args → System properties → OS env vars → application.properties → profile-specific properties. Fires ApplicationEnvironmentPreparedEvent. This is when spring.profiles.active resolves and all external config is loaded.
  • 03
    Step 3 — ApplicationContext Creation: Creates AnnotationConfigServletWebServerApplicationContext (MVC) or AnnotationConfigReactiveWebServerApplicationContext (WebFlux). Registers the primary @SpringBootApplication class and fires ApplicationContextInitializedEvent.
  • 04
    Step 4 — Bean Definition Loading: Component scan discovers all stereotyped beans. Auto-configuration classes evaluate their @Conditional* annotations. BeanFactoryPostProcessors run — modifying bean definitions before instantiation. @ConfigurationProperties and @PropertySource processed here.
  • 05
    Step 5 — Bean Instantiation & Wiring: Beans instantiated in dependency order. BeanPostProcessors apply AOP proxies — @Transactional, @Cacheable, @Async, Resilience4j annotations all become proxies here. @PostConstruct methods fire. ApplicationRunner / CommandLineRunner beans queued.
  • 06
    Step 6 — Server Start + ApplicationReadyEvent: Embedded Tomcat/Netty starts on configured port. ApplicationStartedEvent fires, then Runners execute, then ApplicationReadyEvent — the signal the app is fully ready for traffic. At JP Morgan, Kafka listeners start consuming only after this event: 
    @EventListener(ApplicationReadyEvent.class)
public void onReady() { kafkaListenerEndpointRegistry.start(); }

How does auto-configuration work? How do you override or exclude it?
Auto-ConfigFrequently Asked
▾
  • 01
    The Conditional System: 
    @ConditionalOnClass(DataSource.class)       // only if class on classpath
@ConditionalOnMissingBean(DataSource.class)  // only if no user bean defined
@ConditionalOnProperty(name="feature.enabled", havingValue="true")
@ConditionalOnWebApplication                // only in web context
  • 02
    Override with Your Own Bean: Define a @Bean in your @Configuration class. The @ConditionalOnMissingBean on the auto-config class backs off. At JP Morgan, I override the default KafkaConsumerFactory with one using JPMorgan's SASL/SSL credentials from AWS Secrets Manager. No exclusion needed — just define yours.
  • 03
    Exclude Auto-Config: 
    @SpringBootApplication(exclude = {DataSourceAutoConfiguration.class})
// OR in application.properties:
spring.autoconfigure.exclude=org.springframework.boot.autoconfigure.security.SecurityAutoConfiguration
  • 04
    Debug Auto-Configuration: Add --debug flag or debug=true in properties. Spring Boot prints a "Conditions Evaluation Report" showing every auto-config and why applied or not. The /actuator/conditions endpoint shows this at runtime — invaluable when a bean you expect isn't being created. I use this every time a Kafka or JPA bean behaves unexpectedly.
  • 05
    Writing Custom Auto-Configuration: Create class annotated with @AutoConfiguration, list it in META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports. Guard with @ConditionalOnClass. This is how I built shared infrastructure configs for the JP Morgan team's common monitoring and Kafka setup — one starter, all services configured identically.
  • 06
    Auto-Configuration Order: Use @AutoConfigureBefore, @AutoConfigureAfter to control sequence. JpaBaseConfiguration must run after DataSourceAutoConfiguration so DataSource bean exists. Order matters when one auto-config bean depends on another — ordering annotations make this explicit and safe.

How does Spring Boot externalize configuration? Explain property precedence and @ConfigurationProperties.
ConfigProduction
▾
  • 01
    Property Source Priority (highest to lowest): (1) Command-line args, (2) OS environment variables, (3) Java System properties, (4) application-{profile}.properties, (5) application.properties, (6) @PropertySource. At JP Morgan, ECS task definitions inject secrets via environment variables — they override the default properties file. This is intentional: prod values never live in source-controlled files.
  • 02
    @ConfigurationProperties — Type-Safe Binding: 
    @ConfigurationProperties(prefix = "jpmorgan.kafka")
@Component
public class KafkaProperties {
    private String bootstrapServers;
    private int maxPollRecords = 50;
    private Duration sessionTimeout = Duration.ofSeconds(45);
    // Getters/setters. YAML: jpmorgan.kafka.bootstrap-servers: ...
}
  • 03
    @Value vs @ConfigurationProperties: @Value("${key}") injects a single property — simple but brittle (no type conversion, no validation). @ConfigurationProperties binds entire prefix to POJO — type-safe, supports @Validated constraints, works with complex types (List, Map, Duration), IDE-friendly. Always prefer @ConfigurationProperties for groups of related properties.
  • 04
    Validation on Config (Fail-Fast): 
    @ConfigurationProperties(prefix = "app")
@Validated
public class AppProperties {
    @NotBlank private String apiKey;
    @Min(1) @Max(100) private int maxRetries;
    // App FAILS TO START if validation fails — catches misconfiguration early
}
  • 05
    Profiles: spring.profiles.active=prod activates application-prod.properties. Beans conditionally loaded with @Profile("prod"). At JP Morgan: separate profiles per environment pointing to different AWS endpoints, Kafka clusters, and log levels. Activated via ECS task environment variables — the same Docker image behaves differently in each environment purely through config.
  • 06
    Relaxed Binding: jpmorgan.kafka.bootstrap-servers, JPMORGAN_KAFKA_BOOTSTRAP_SERVERS, jpmorgan.kafka.bootstrapServers all bind to the same field. This "relaxed binding" means properties work uniformly from .properties files (kebab-case), env vars (UPPER_SNAKE), or Java system properties (camelCase) — matching the natural convention of each source format.

What is Spring AOP? How are @Transactional, @Cacheable, and @Async implemented with it?
AOPInternals
▾
  • 01
    AOP via Proxies: Spring AOP wraps your bean in a proxy (JDK dynamic proxy for interfaces; CGLIB for classes) that intercepts method calls and applies cross-cutting logic. @Transactional, @Cacheable, @Async, Resilience4j annotations all work this way. The bean you inject is the proxy, not the original object.
  • 02
    @Transactional — Proxy Intercepts: Proxy opens transaction → calls real method → commits or rolls back. Critical self-invocation bug: calling a @Transactional method from within the SAME class bypasses the proxy — NO transaction. This is the #1 @Transactional production bug. Solution: inject self (@Autowired ApplicationContext ctx; ctx.getBean(ThisClass.class).method()) or restructure.
  • 03
    @Transactional Propagation Options: 
    @Transactional(propagation = Propagation.REQUIRED)     // default: join or create
@Transactional(propagation = Propagation.REQUIRES_NEW) // always new tx
@Transactional(propagation = Propagation.NESTED)       // savepoint in outer tx
@Transactional(readOnly = true)                        // skip dirty checking
@Transactional(rollbackOn = BusinessException.class)   // rollback checked ex
  • 04
    @Cacheable — Proxy Checks Cache First: On invocation, proxy generates cache key (default: method params), checks cache. Hit → return cached; Miss → invoke method, store, return. Related: @CachePut (always invoke, update cache), @CacheEvict (remove entry). Requires @EnableCaching and a CacheManager bean (Caffeine, Redis). At VISA, counterparty cache hit ratio ~92% reduced DB load by 30%.
  • 05
    @Async — Fire and Forget: Proxy submits method to TaskExecutor and returns immediately. Returns Future<T> or CompletableFuture<T>. Requires @EnableAsync. Same self-invocation caveat — internal calls bypass proxy and run synchronously. Always configure a custom ThreadPoolTaskExecutor — the default SimpleAsyncTaskExecutor creates a new thread per call and is NOT production-safe.
  • 06
    Custom AOP Aspect: 
    @Aspect @Component
public class AuditAspect {
    @Around("@annotation(Auditable)")
    public Object audit(ProceedingJoinPoint pjp) throws Throwable {
        long start = System.currentTimeMillis();
        try { return pjp.proceed(); }
        finally { auditLog(pjp.getSignature().getName(), System.currentTimeMillis()-start); }
    }
}

How do you handle exceptions globally in Spring Boot? Explain @ControllerAdvice and ProblemDetails.
Exception HandlingREST API
▾
  • 01
    @RestControllerAdvice — Global Handler: A class annotated with @RestControllerAdvice intercepts exceptions from any controller and maps them to structured HTTP responses. One place for all exception handling — no try-catch in controllers. Methods annotated with @ExceptionHandler handle specific exception types.
  • 02
    Implementation Pattern (JP Morgan): 
    @RestControllerAdvice
public class GlobalExceptionHandler {
    @ExceptionHandler(EntityNotFoundException.class)
    @ResponseStatus(HttpStatus.NOT_FOUND)
    public ProblemDetail handleNotFound(EntityNotFoundException ex) {
        ProblemDetail pd = ProblemDetail.forStatusAndDetail(
            HttpStatus.NOT_FOUND, ex.getMessage());
        pd.setType(URI.create("https://api.jpmorgan.com/errors/not-found"));
        return pd;
    }
}
  • 03
    RFC 7807 ProblemDetail (Spring Boot 3): Enable with spring.mvc.problemdetails.enabled=true. Returns machine-readable error responses: 
    { "type": "https://api.jpmorgan.com/errors/not-found",
  "title": "Not Found", "status": 404,
  "detail": "Event 123 not found", "instance": "/events/123" }
  • 04
    Validation Error Handling (@Valid): 
    @ExceptionHandler(MethodArgumentNotValidException.class)
public ProblemDetail handleValidation(MethodArgumentNotValidException ex) {
    Map<String,String> errors = ex.getBindingResult().getFieldErrors().stream()
        .collect(toMap(FieldError::getField, FieldError::getDefaultMessage));
    ProblemDetail pd = ProblemDetail.forStatus(400);
    pd.setProperty("fieldErrors", errors);
    return pd;
}
  • 05
    WebFlux Exception Handling: Use @RestControllerAdvice for annotated controllers, or implement WebExceptionHandler for functional routing. In WebClient error handling, use .onErrorMap() and .onErrorResume() — traditional try-catch does NOT work inside reactive pipelines. Exceptions in reactive code must be handled through operators.
  • 06
    Exception Hierarchy Strategy: Define base ApiException with status code + error code. All business exceptions extend it. Global handler has ONE catch-all for ApiException: 
    @ExceptionHandler(ApiException.class)
public ResponseEntity<ProblemDetail> handleApi(ApiException ex) {
    return ResponseEntity.status(ex.getStatus())
        .body(ProblemDetail.forStatusAndDetail(ex.getStatus(), ex.getMessage()));
}// New exception types just extend ApiException — zero handler changes

Explain Spring Boot testing — @SpringBootTest, @WebMvcTest, @DataJpaTest, and mocking strategies.
TestingQuality
▾
  • 01
    Test Slice Strategy: Spring Boot provides "test slices" — partial application contexts that load only the relevant layer. This makes tests fast and focused. Full @SpringBootTest is expensive (loads everything) — use it only for true integration testing. For unit/slice tests, always use the most targeted annotation.
  • 02
    @WebMvcTest — Controller Layer Only: 
    @WebMvcTest(EventController.class)
class EventControllerTest {
    @Autowired MockMvc mockMvc;
    @MockBean EventService eventService; // replaces real bean in context
    @Test
    void test() throws Exception {
        given(eventService.find("1")).willReturn(new EventDto(...));
        mockMvc.perform(get("/events/1")).andExpect(status().isOk());
    }
}
  • 03
    @DataJpaTest — JPA Slice: Loads only JPA components with H2 in-memory DB. Tests actual JPQL queries, entity mapping, pagination. Transactions roll back after each test. I use this to catch N+1 queries and missing indexes before they hit staging — found a missing index that caused full table scans in the VISA reconciliation system during code review.
  • 04
    @MockBean vs @Mock: @Mock (Mockito) creates a mock but does NOT add to Spring context — use in pure unit tests with @ExtendWith(MockitoExtension.class). @MockBean creates a mock AND registers it as a Spring bean replacing any existing bean — use in slice tests. Mixing them up is a frequent source of confusing test failures.
  • 05
    WebFlux Testing — WebTestClient: 
    @WebFluxTest(EventHandler.class)
class EventHandlerTest {
    @Autowired WebTestClient client;

    @Test void test() {

        client.get().uri("/events/1").exchange()
              .expectStatus().isOk().expectBody(EventDto.class).isEqualTo(event);
    }
}
  • 06
    @EmbeddedKafka for Integration Tests: 
    @SpringBootTest
@EmbeddedKafka(partitions = 3, topics = {"ops.events"})
class KafkaConsumerIntegrationTest {
    @Autowired KafkaTemplate<String,String> template;

}

How do you tune embedded Tomcat and Netty for production? Explain graceful shutdown.
Server TuningProduction
▾
  • 01
    Tomcat Thread Pool Tuning: 
    server.tomcat.threads.max=200          # max worker threads
server.tomcat.threads.min-spare=10     # always-warm threads
server.tomcat.accept-count=100         # queue when all threads busy
server.tomcat.connection-timeout=20000 # idle connection timeout
server.tomcat.max-connections=8192     # max simultaneous connections
  • 02
    Netty Tuning (WebFlux — JP Morgan): Netty uses event-loop threads — default: CPUs × 2. For I/O-bound work this is correct. Don't increase this arbitrarily. Instead tune the connection pool and buffer sizes: 
    server.netty.connection-timeout=30s
server.netty.idle-timeout=60s
# Event loop threads = Runtime.getRuntime().availableProcessors() * 2
# Never block event-loop threads — use Schedulers.boundedElastic() for blocking
  • 03
    Tomcat vs Netty Decision: Tomcat: existing team, blocking I/O stack, simple CRUD. Netty: reactive stack (WebFlux), high concurrency, fan-out to multiple downstream services. At JP Morgan the event processing service uses Netty — the 25% p99 latency reduction I delivered required the non-blocking event-loop model. Wrong server for wrong stack negates the architecture benefits.
  • 04
    SSL/TLS Configuration: 
    server.ssl.key-store=classpath:keystore.p12
server.ssl.key-store-password=${SSL_KEYSTORE_PASSWORD}
server.ssl.key-store-type=PKCS12
server.ssl.enabled=true
server.port=8443
# At JP Morgan: certs fetched from AWS Secrets Manager at startup, never in files
  • 05
    Graceful Shutdown — Critical for ECS Rolling Deploy: 
    server.shutdown=graceful
spring.lifecycle.timeout-per-shutdown-phase=30s
    When SIGTERM arrives (ECS task stop), Tomcat/Netty stops accepting new requests, lets in-flight requests complete within 30 seconds, then shuts down. Without this, rolling deployments drop active requests causing 500 errors and failed transactions. This is a non-negotiable production setting.
  • 06
    Port and Context Path: 
    server.port=8080
server.servlet.context-path=/api       # all endpoints prefixed with /api
management.server.port=8081            # actuator on separate port
# Separate management port means actuator not exposed via ALB
# ECS health check hits :8081/actuator/health internally, not externally

Compare constructor injection, setter injection, and field injection — which do you prefer and why?
DIBest Practice
▾
Spring team recommends constructor injection. Know the exact reasons — this is asked at every senior interview.
  • 01
    Constructor Injection (Preferred): 
    @Service
public class EventService {
    private final EventRepository repo;
    private final KafkaTemplate<String,Event> kafka;
    // @Autowired optional since Spring 4.3 for single constructor
    public EventService(EventRepository repo, KafkaTemplate<String,Event> kafka) {
        this.repo = repo; this.kafka = kafka;
    }
}
  • 02
    Why Constructor Injection Wins: (1) Immutability — fields can be final. (2) Mandatory deps explicit — can't create object without all deps. (3) Testability — instantiate directly: new EventService(mockRepo, mockKafka), no Spring context needed. (4) Circular dependency detection — Spring fails at startup, not runtime. (5) No Spring coupling in domain objects — pure Java, no annotations needed.
  • 03
    Field Injection — Avoid: 
    @Autowired private EventRepository repo; // BAD
// Problems: can't be final, can't instantiate without Spring,
// hides dependencies, circular deps discovered at runtime not startup
    Spring team explicitly discourages this. Every senior code review should flag field injection.
  • 04
    Setter Injection — For Optional Deps Only: 
    @Autowired(required = false)
public void setMetricsPublisher(MetricsPublisher m) { this.metrics = m; }
// Use only when dep is truly optional — class works without it
// For mandatory deps, always constructor injection
  • 05
    @Primary and @Qualifier for Multiple Implementations: 
    @Primary @Service
public class NewReconciliationEngine implements ReconciliationEngine {}

@Service("legacyEngine")
public class LegacyReconciliationEngine implements ReconciliationEngine {}
// @Autowired picks @Primary; @Qualifier("legacyEngine") picks specific one
  • 06
    Circular Dependency: A → B → A via constructor injection throws BeanCurrentlyInCreationException at startup. Solutions: (1) Redesign — circular deps usually signal a design flaw. (2) @Lazy on one injection. (3) Setter injection for one. The real fix is always architectural — introduce a third service or use an ApplicationEvent to break the cycle.

Explain Bean Scopes — Singleton, Prototype, Request, Session. Production use cases.
Bean ScopeInternals
▾
  • 01
    Singleton (Default): One instance per ApplicationContext, shared by all callers. Correct for stateless services. Critical: Singleton beans must be thread-safe — accessed concurrently. Never store mutable request-specific state in singleton fields. All JP Morgan services, repositories, and Kafka consumers are singletons — stateless by design.
  • 02
    Prototype: New instance on every getBean() call. Spring does NOT call destruction callbacks on prototypes. Use for stateful, non-shareable objects. Access via Provider<T> to get fresh instances from a singleton: 
    @Autowired Provider<OrderProcessor> processorProvider;
public void process() { processorProvider.get().run(); } // fresh each time
  • 03
    Request Scope: One instance per HTTP request, destroyed after. Ideal for per-request state: current user context, request trace ID, per-request counters. In WebFlux, use Reactor Context instead — request scope is thread-bound and doesn't work with non-blocking event-loop model.
  • 04
    Session Scope: One instance per HTTP session. Useful for shopping cart, user preferences, wizard-state across requests. Beans must be serializable for session replication in clustered environments. In our stateless microservices at JP Morgan, session scope is not used — JWT-encoded state or Redis handles session data.
  • 05
    Prototype in Singleton Pitfall: Injecting prototype into singleton via constructor gives same prototype instance forever. Use Provider<T> or ObjectFactory<T> to get fresh instances. Or use @Lookup method injection which Spring overrides to call getBean() each time.
  • 06
    @PostConstruct and @PreDestroy: @PostConstruct runs after bean fully wired — use for initialization (cache warm-up, connection validation). @PreDestroy runs before bean destroyed — use for cleanup (close resources, deregister listeners). Both work with singleton beans. Prototype beans get @PostConstruct but NOT @PreDestroy — Spring doesn't manage prototype lifecycle after creation.

What are ApplicationEvents? How do you use them for decoupled communication between beans?
EventsDesign Pattern
▾
  • 01
    Observer Pattern Built-In: ApplicationEventPublisher + @EventListener = decoupled bean communication. Publisher doesn't know listeners. At JP Morgan I use this to trigger post-processing (cache invalidation, audit write, metric update) after an event is persisted — without coupling the core service to each downstream action.
  • 02
    Publishing and Listening: 
    // Publisher
publisher.publishEvent(new EventProcessedEvent(e.getId(), Instant.now()));

// Listener (different bean — zero coupling)
@EventListener
public void onProcessed(EventProcessedEvent evt) {
    auditLog.record(evt.eventId(), evt.processedAt());
}
  • 03
    @TransactionalEventListener — Key Pattern: Fires ONLY after the publishing transaction commits successfully. If transaction rolls back, listener doesn't fire — prevents processing a rolled-back event: 
    @TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
public void sendKafkaAfterCommit(EventProcessedEvent evt) {
    kafkaProducer.send(evt); // safe: only fires if DB commit succeeded
}
  • 04
    @Async Events: Default: listener runs synchronously in publisher's thread. Add @Async to run in separate thread pool — fire-and-forget pattern. Enables the publisher to return quickly while listeners handle side effects asynchronously. Combine with @TransactionalEventListener carefully — async runs outside the original transaction.
  • 05
    Conditional Listener: 
    @EventListener(condition = "#event.priority == 'HIGH'")
public void handleHighPriority(OperationalAlert event) {
    alertingService.escalate(event);
}// SpEL condition — fires only for HIGH priority events
// Avoids if-else filtering inside listener bodies
  • 06
    Built-in Application Events: Spring Boot publishes lifecycle events you can listen to: ApplicationReadyEvent (all beans ready — start consumers), ApplicationFailedEvent (startup failed — cleanup resources), ApplicationEnvironmentPreparedEvent (props loaded — validate config). I listen to ApplicationReadyEvent to pre-populate caches and start Kafka consumer containers at JP Morgan.

Explain the N+1 problem in JPA — how do you detect and fix it in production?
JPAProduction Bug
▾
N+1 is the #1 JPA production performance bug. Show you've hit it, diagnosed it, and fixed it at VISA.
  • 01
    What N+1 Is: JPA fetches 1 parent query (N records), then fires N additional queries for each lazy association — N+1 total. At VISA with 5,000 settlement records each having a lazy-loaded Counterparty, this meant 5,001 DB round-trips instead of 1. Each round-trip ~2ms → 10 seconds of unnecessary latency killing the end-of-day SLA.
  • 02

Why did you choose Spring WebFlux at JP Morgan? Explain reactive vs blocking trade-offs.
WebFluxArchitecture Decision
▾
This is your signature answer — connect every reactive concept to the JP Morgan 25% p99 reduction you delivered.
  • 01
    The Problem with Blocking MVC: In Spring MVC, each request consumes a Tomcat thread (default pool: 200). If enriching an event requires calling 3 downstream services (100ms each), that thread is blocked for 300ms doing nothing. Under high load, thread pool exhausts and requests queue — latency spikes at exactly the worst moment (peak traffic).
  • 02
    How WebFlux Solves It: Netty uses an event-loop model with a small fixed thread pool (one thread per CPU core). Instead of blocking on I/O, threads register callbacks and process the next request. When I/O completes, the event loop picks it up. Thousands of concurrent requests on 8 threads — threads are never idle waiting.
  • 03
    JP Morgan Use Case — Parallel Fan-Out: 
    public Mono<EnrichedEvent> enrich(Event event) {
    Mono<ServiceAData> a = serviceA.getData(event.getId());
    Mono<ServiceBData> b = serviceB.getMeta(event.getType());
    return Mono.zip(a, b, (dataA, dataB) -> merge(event, dataA, dataB));
    // a and b subscribed simultaneously — latency = max(a,b) not a+b
    // This specific change gave us the 25% p99 latency reduction
}
  • 04
    The Golden Rule — Never Block the Event Loop: 
    // BAD: blocks event loop, kills all other requests on this thread
public Mono<Result> process() { return Mono.just(legacyBlockingJdbcCall()); }

// GOOD: offload blocking to dedicated thread pool
public Mono<Result> process() {
    return Mono.fromCallable(() -> legacyBlockingJdbcCall())
               .subscribeOn(Schedulers.boundedElastic());
}
  • 05
    When NOT to Use WebFlux: CPU-bound work (no I/O benefit), teams unfamiliar with reactive debugging (stack traces are 15 frames deep in Reactor internals), blocking dependencies throughout the stack, simple CRUD services. I kept the VISA reconciliation system on Spring MVC — the team wasn't ready for reactive and the sequential processing nature didn't benefit from it.
  • 06
    Backpressure — Why It Matters: Reactive Streams spec: subscriber calls request(n) to signal demand, publisher only emits when demand exists. In JP Morgan pipeline: if Aurora write service is slow, Kafka consumer's Flux naturally slows down — avoiding OOM from unbounded buffering. This propagated flow control prevents cascade failures under load.

  • 02
    Redis-Based Sliding Window Rate Limiter: 
    @Component
public class RateLimiterService {
    public boolean isAllowed(String clientId, int limit, long windowMs) {
        String key = "rate:" + clientId + ":" + (System.currentTimeMillis() / windowMs);
        Long count = redisTemplate.opsForValue().increment(key);
        if (count == 1) redisTemplate.expire(key, windowMs, TimeUnit.MILLISECONDS);
        return count <= limit;
    }
}

10 Quick-Fire Spring Boot Questions — rapid answers for interview closing rounds.
Quick FireClosing Round
▾
These are the rapid-fire questions at the end of every Spring Boot interview. Have these 10 answers memorized — 30 seconds each, zero hesitation.
  • 01
    Q: What is the difference between @RestController and @Controller?
    A: @RestController = @Controller + @ResponseBody. Every method return value is serialized to JSON/XML directly into the response body. Use @RestController for REST APIs. Use @Controller when you're returning view names for template rendering (Thymeleaf/FreeMarker) and need some methods to return views and others to return JSON using @ResponseBody selectively.
  • 02
    Q: What is the difference between @PathVariable and @RequestParam?
    A: @PathVariable binds to a URI template variable — /events/{id} where id is extracted from the path. @RequestParam binds to a query string or form parameter — /events?status=PENDING where status is extracted from the query string. Both support required/defaultValue attributes. Use path variables for resource identification, query params for filtering/pagination.
  • 03
    Q: What does @SpringBootTest do and when should you NOT use it?
    A: Loads the FULL ApplicationContext — all beans, embedded server (optionally), all auto-configuration. Use it only for true end-to-end integration tests. Don't use it for unit tests (use @ExtendWith(MockitoExtension.class)), controller tests (use @WebMvcTest), or repository tests (use @DataJpaTest). @SpringBootTest is slow — overusing it makes your test suite take minutes instead of seconds.
  • 04
    Q: How does @Async work and what are its caveats?
    A: Marks a method to execute asynchronously in a TaskExecutor thread pool. Returns void, Future<T>, or CompletableFuture<T>. Requires @EnableAsync. Two critical caveats: (1) Self-invocation bypasses the proxy — the method runs synchronously if called from within the same class. (2) The default SimpleAsyncTaskExecutor creates a new thread per call and is not production-safe — always configure a bounded ThreadPoolTaskExecutor.
  • 05
    Q: What is the difference between Mono.just() and Mono.fromCallable()?
    A: Mono.just(value) wraps an ALREADY-COMPUTED value — the value is evaluated IMMEDIATELY when the Mono is created, not when subscribed. Mono.fromCallable(() -> compute()) wraps a computation — it's lazy, evaluated ONLY when subscribed. If compute() is expensive or has side effects (DB call, API call), always use fromCallable to ensure lazy evaluation. Using just(heavyDbCall()) calls the DB immediately even if no one subscribes.
  • 06
    Q: What is Spring Boot's auto-configuration exclusion order — user beans vs auto-config?
    A: User-defined beans ALWAYS take precedence over auto-configured beans. Spring's auto-configuration uses @ConditionalOnMissingBean — it only creates a bean if no user bean of that type exists in the context. So simply defining your own @Bean DataSource is enough to override the auto-configured DataSource, without any exclusion annotation needed. This is the "opinionated defaults, always overrideable" philosophy of Spring Boot.
  • 07
    Q: What is @ConditionalOnProperty matchIfMissing?
    A: matchIfMissing=true means the condition is treated as matching (bean IS created) even if the property is not defined. matchIfMissing=false (default) means if the property is absent, the condition does NOT match (bean is NOT created). Use matchIfMissing=true to implement "enabled by default, explicitly disable with feature.x=false" — the most common feature flag pattern in Spring Boot.
  • 08
    Q: How do you prevent SQL injection in Spring Data JPA?
    A: Spring Data JPA prevents SQL injection by default when using: (1) Method name query derivation — Spring generates parameterized queries. (2) @Query with :namedParam or ?1 positional parameters — Spring uses prepared statements. (3) Criteria API — parameterized by design. DANGER: using string concatenation in native queries: @Query(value = "SELECT * FROM events WHERE status = '" + status + "'") — never do this. Always use @Param bindings.
  • 09
    Q: What is the difference between @Bean and @Component?
    A: @Component is used on classes you control — Spring discovers them via component scan and manages instantiation. @Bean is used on methods inside @Configuration classes — you control instantiation yourself and return the bean instance. Use @Bean for: third-party library objects you can't annotate (DataSource, ObjectMapper), objects needing complex initialization logic, or objects where you need conditional creation. Use @Component for your own classes.
  • 10
    Q: Describe three things you would do immediately when joining a new Spring Boot codebase at a senior level.
    A: (1) Run with --debug and review the Conditions Evaluation Report — understand what auto-configs are active and why. Check /actuator/beans to see all registered beans. (2) Enable Hibernate statistics and slow query logging in dev — find N+1 problems and missing indexes before they become production incidents. (3) Review the Resilience4j/Circuit Breaker configuration (or absence of it) — every external call needs timeout and fallback. If none exists, that's the first production risk to address.
Final tip: In every Spring Boot interview answer, follow: (1) Concept definition, (2) WHY it was designed that way, (3) Code example, (4) Production use case from JP Morgan or VISA. This 4-part structure takes 90 seconds per answer and consistently distinguishes senior from mid-level candidates.
