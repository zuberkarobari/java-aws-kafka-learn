Valid Parentheses — Check if brackets are properly matched
EasyStackStringAsked Everywhere
▾
Pattern — Stack for Matching Pairs
Push opening brackets onto stack. When a closing bracket appears, check if it matches the top of the stack. If not — invalid. At the end, stack must be empty (all openers matched). Use a Map to store closer→opener mappings cleanly.
P
Problem: Given string s containing only '(', ')', '{', '}', '[', ']', determine if the input string is valid. Every opener must be closed in correct order.

Input: "({[]})" → true    Input: "([)]" → false
1
Why Stack? Parentheses must close in LIFO order — the most recently opened bracket must close first. Stack naturally models LIFO. When we see ), the last unclosed opener must be (. Stack's top always holds "the opener waiting to be closed."
2
Three Failure Cases: (1) Closing bracket doesn't match top of stack → invalid order like "([)]". (2) Closing bracket when stack is empty → extra closer like ")". (3) Stack not empty at end → unmatched opener like "(". Handle all three explicitly.
3
Clean Implementation Using Map: Store {')':'(', ']':'[', '}':'{' }. When we see a closer, its expected opener is map.get(c). Compare with stack top. This avoids multiple if-else chains and is easier to extend to more bracket types.
4
Variations asked in interviews: Minimum brackets to add to make valid (LC 921), Minimum brackets to remove (LC 1249), Score of Parentheses (LC 856). The core stack pattern is the same — knowing the base problem deeply unlocks all variants.
Time
O(n)
Space
O(n)
Pattern

public boolean isValid(String s) {
    Deque<Character> stack = new ArrayDeque<>();
    Map<Character,Character> map = Map.of(
        ')', '(',  ']', '[',  '}', '{'
    );
    for (char c : s.toCharArray()) {
        if (!map.containsKey(c)) {         // it's an opener
            stack.push(c);
        } else {                            // it's a closer
            if (stack.isEmpty() || stack.pop() != map.get(c))
                return false;             // no match
        }
    }
    return stack.isEmpty();                // all openers matched?
}

// Java 8 style — functional but less readable for interviews:
public boolean isValidStream(String s) {
    Deque<Character> stack = new ArrayDeque<>();
    Map<Character,Character> pairs = Map.of(')','(',']','[','}','{');
    return s.chars()
        .mapToObj(c -> (char) c)
        .allMatch(c -> {
            if (!pairs.containsKey(c)) { stack.push(c); return true; }
            return !stack.isEmpty() && stack.pop() == pairs.get(c);
        }) && stack.isEmpty();
}

Min Stack — Stack that retrieves minimum in O(1)
MediumStackDesignClassic Design Q
▾
Pattern — Auxiliary Min Stack
Maintain two stacks: the main stack for actual values, and a min-stack that tracks the current minimum at each level. When we push onto main, also push min(val, minStack.top()) onto minStack. When we pop from main, also pop from minStack. getMin() = minStack top. O(1) all operations.
P
Problem: Design a stack that supports push, pop, top, and getMin — all in O(1) time.

MinStack s = new MinStack();
s.push(-2); s.push(0); s.push(-3);
s.getMin(); → -3   s.pop(); s.top(); → 0   s.getMin(); → -2
1
Core Insight: The minimum at any point depends on what's currently on the stack. When an element is popped, the minimum may change. So we need to remember, for every state of the stack, what the minimum was at that exact level. The minStack stores exactly this — one min entry per main stack entry.
2
Why Not Just One Variable? If you keep a single currentMin variable, popping the minimum element leaves you with no knowledge of the previous minimum. The minStack solves this by recording the minimum at every stack depth. When we pop from main, popping from minStack restores the previous minimum.
3
Space-Optimised Alternative: Instead of two stacks, encode the delta: push val - currentMin when val < currentMin; this encodes the minimum change. Tricky to implement but uses less space. In interviews, the two-stack approach is clearer — prefer clarity over cleverness unless asked for space optimisation.
4
Related: Max Stack, Max Queue: Same pattern — auxiliary stack for max. Max Queue (deque-based) uses a different pattern (monotonic deque). These are natural follow-up questions at Groww and Juspay.
push/pop/top/getMin
O(1)

class MinStack {
    private Deque<Integer> stack    = new ArrayDeque<>();
    private Deque<Integer> minStack = new ArrayDeque<>();

    public void push(int val) {
        stack.push(val);
        // minStack top always = current minimum of main stack
        int min = minStack.isEmpty() ? val : Math.min(val, minStack.peek());
        minStack.push(min);
    }

    public void pop() {
        stack.pop();
        minStack.pop();  // always pop both together
    }

    public int top()    { return stack.peek(); }
    public int getMin() { return minStack.peek(); }
}

// Space-optimised single-stack approach (encode delta):
class MinStackOptimised {
    private Deque<Long> stack = new ArrayDeque<>();
    private long min;

    public void push(int val) {
        if (stack.isEmpty()) { stack.push(0L); min = val; return; }
        stack.push((long) val - min);    // store delta
        if (val < min) min = val;
    }
    public void pop() {
        long top = stack.pop();
        if (top < 0) min -= top;          // restore previous min
    }
    public int top()    { return (int) (stack.peek() < 0 ? min : stack.peek() + min); }
    public int getMin() { return (int) min; }
}
Operations: push(-2), push(0), push(-3), getMin, pop, top, getMin ──────────────────────────────────────────────────────── push(-2): stack=[-2] minStack=[-2] (min=-2) push(0): stack=[0,-2] minStack=[-2,-2] (min stays -2) push(-3): stack=[-3,0,-2] minStack=[-3,-2,-2] (new min=-3) getMin() → minStack.peek() = -3 ✓ pop(): stack=[0,-2] minStack=[-2,-2] (min restored to -2) top() → stack.peek() = 0 ✓ getMin()→ minStack.peek() = -2 ✓
Say: "I maintain a parallel minStack where each position stores the minimum of all elements at or below that depth. Push and pop both stacks together — they stay in sync. getMin() is always O(1) — just peek at minStack top."

Binary Search — Classic implementation and all variants
EasyBinary SearchFoundationTemplate to Memorize
▾
Pattern — Binary Search Template (3 variants)
Binary search works on ANY sorted/monotone space — not just arrays. The key is: identify what you're searching for, define the search space [lo, hi], define the condition that splits the space into "yes" and "no" halves, and decide whether to use exact match or leftmost/rightmost boundary.
P
Problem: Given sorted array nums and integer target, return its index or -1 if not found.
Input: nums=[-1,0,3,5,9,12], target=9 → Output: 4
1
The Off-By-One Problem: Binary search has 3 common bugs: (1) Integer overflow in mid = (lo+hi)/2 — fix: mid = lo + (hi-lo)/2. (2) Infinite loop with while(lo < hi) when update doesn't shrink range. (3) Missing +1 on lo or -1 on hi causing infinite loop. Use the template below religiously.
2
Template 1 — Exact Match: while(lo <= hi). mid = lo+(hi-lo)/2. If found return mid. If target < nums[mid]: hi=mid-1. Else lo=mid+1. Return -1 at end. Use when you need exact match and won't miss target by shrinking past it.
3
Template 2 — Leftmost Boundary (first true): while(lo < hi). If condition(mid) is true: hi=mid. Else: lo=mid+1. Returns lo = first index where condition is true. Use for: find first bad version, first position of element, leftmost target.
4
Template 3 — Rightmost Boundary (last true): while(lo < hi). mid = lo+(hi-lo+1)/2 (ceiling!). If condition(mid): lo=mid. Else: hi=mid-1. Returns lo = last index where condition is true. The ceiling mid prevents infinite loop when lo+1=hi.
Time
O(log n)
Space
O(1)
Pattern
Halving
// Template 1: Exact Match
public int search(int[] nums, int target) {
    int lo = 0, hi = nums.length - 1;
    while (lo <= hi) {
        int mid = lo + (hi - lo) / 2;   // NEVER (lo+hi)/2 — overflow risk
        if      (nums[mid] == target) return mid;
        else if (nums[mid] < target)  lo = mid + 1;
        else                          hi = mid - 1;
    }
    return -1;
}

// Template 2: Leftmost boundary — first occurrence of target
public int firstOccurrence(int[] nums, int target) {
    int lo = 0, hi = nums.length;    // hi = length (open right bound)
    while (lo < hi) {
        int mid = lo + (hi - lo) / 2;
        if (nums[mid] < target) lo = mid + 1;
        else                    hi = mid;       // keep mid as candidate
    }
    return lo < nums.length && nums[lo] == target ? lo : -1;
}

// Template 3: Rightmost boundary — last occurrence of target
public int lastOccurrence(int[] nums, int target) {
    int lo = 0, hi = nums.length - 1;
    while (lo < hi) {
        int mid = lo + (hi - lo + 1) / 2;  // ceiling mid!
        if (nums[mid] <= target) lo = mid;
        else                     hi = mid - 1;
    }
    return nums[lo] == target ? lo : -1;
}

if (idx < 0) System.out.println("Not found, insert at: " + (-idx - 1));

Memorize: mid = lo + (hi-lo)/2 always. The "search space" in binary search is often not the array itself — it's the answer range (e.g., "find minimum speed", "find minimum days"). Recognizing binary search on answer space is what separates medium from hard solvers.

Search in Rotated Sorted Array — Binary search after rotation
MediumBinary SearchRotated ArrayVery Frequently Asked
▾
Pattern — Identify Sorted Half, Then Narrow
In a rotated array, at every midpoint, one half is always normally sorted. Check which half is sorted by comparing nums[lo] with nums[mid]. If target lies in the sorted half, search there. Otherwise search the other half. O(log n) maintained despite rotation.
P
Problem: Given integer array nums sorted in ascending order and then rotated at some pivot, search for target. Return index or -1.

Input: nums=[4,5,6,7,0,1,2], target=0 → Output: 4
1
Key Observation: A rotated sorted array always has ONE sorted half at any midpoint. If nums[lo] <= nums[mid], the left half [lo..mid] is sorted. Otherwise the right half [mid..hi] is sorted. This is the invariant that makes binary search work here.
2
Decision Logic: Once you identify the sorted half, check if target lies within it: if left sorted AND nums[lo] <= target < nums[mid] → search left (hi=mid-1). Otherwise search right (lo=mid+1). Mirror logic for right sorted half.
3
With Duplicates (LC 81): If nums[lo] == nums[mid], we can't determine which half is sorted. Only safe option: increment lo. This degrades worst-case to O(n) for arrays with many duplicates (e.g., [1,1,1,0,1,1]).
4
Common Mistake: Using < instead of <= in the sorted half check, causing off-by-one when lo==mid (single element). Always use nums[lo] <= nums[mid].
Time
O(log n)
Space
O(1)
Pattern
Modified BS
public int search(int[] nums, int target) {
    int lo = 0, hi = nums.length - 1;
    while (lo <= hi) {
        int mid = lo + (hi - lo) / 2;
        if (nums[mid] == target) return mid;

        // Left half [lo..mid] is sorted
        if (nums[lo] <= nums[mid]) {
            if (nums[lo] <= target && target < nums[mid])
                hi = mid - 1;   // target in left sorted half
            else
                lo = mid + 1;   // target must be in right half
        }
        // Right half [mid..hi] is sorted
        else {
            if (nums[mid] < target && target <= nums[hi])
                lo = mid + 1;   // target in right sorted half
            else
                hi = mid - 1;   // target must be in left half
        }
    }
    return -1;
}
nums=[4,5,6,7,0,1,2], target=0 idx: 0 1 2 3 4 5 6 ────────────────────────────────────────────────── lo=0,hi=6: mid=3, nums[3]=7 ≠0 nums[lo=4] <= nums[mid=7] → left [0..3] sorted target=0 NOT in [4..7) → go right: lo=4 lo=4,hi=6: mid=5, nums[5]=1 ≠0 nums[lo=0] <= nums[mid=1]? 0<=1 → left [4..5] sorted target=0 in [0..1)? 0<=0 <1 → go left: hi=4 lo=4,hi=4: mid=4, nums[4]=0 == 0 → return 4 ────────────────────────────────────────────────── Answer: 4 ✓
The hardest part is the boundary conditions. Practice saying out loud: "I check which half is sorted. The sorted half has a predictable range [nums[lo], nums[mid]]. If target falls in that range, I search there. Otherwise I search the other half."

Find Minimum in Rotated Sorted Array — Binary search for pivot
MediumBinary SearchRotated ArrayCompanion to Q14
▾
Pattern — Compare mid with right boundary
The minimum is always in the UNSORTED half. Compare nums[mid] with nums[hi]: if nums[mid] > nums[hi], the right half is unsorted (minimum is somewhere in right) → lo=mid+1. Otherwise left half is unsorted (minimum is in left or at mid) → hi=mid. Loop ends when lo==hi==minimum.
P
Problem: Given a sorted rotated array of unique integers, find the minimum element in O(log n).

Input: nums=[3,4,5,1,2] → Output: 1
Input: nums=[4,5,6,7,0,1,2] → Output: 0
1
Why Compare with hi, not lo? This problem is about finding the "pivot" (rotation point). We compare mid with the right boundary because the rotation creates a drop somewhere — the minimum is where the drop happens. If nums[mid] > nums[hi], the drop is to the right of mid → minimum is in [mid+1..hi]. If nums[mid] <= nums[hi], no drop to the right → minimum is in [lo..mid].
2
Key: hi=mid not hi=mid-1: When nums[mid] <= nums[hi], mid itself could be the minimum. We set hi=mid (keep mid in range) not hi=mid-1 (which might exclude the answer). This is Template 2 (leftmost boundary).
3
No Rotation Case: If the array isn't rotated (e.g., [1,2,3,4,5]), nums[0] <= nums[n-1]. The algorithm handles this naturally — keeps moving hi leftward until lo=hi=0, which is the minimum.
4
With Duplicates (LC 154): If nums[mid]==nums[hi], we can't tell which side the minimum is on. Decrement hi by 1 to safely shrink the range. O(n) worst case with all duplicates.
Time
O(log n)
Space
O(1)
Pattern
Pivot Search
public int findMin(int[] nums) {
    int lo = 0, hi = nums.length - 1;
    while (lo < hi) {                      // lo < hi (not <=)
        int mid = lo + (hi - lo) / 2;
        if (nums[mid] > nums[hi])
            lo = mid + 1;                  // min is in right half
        else
            hi = mid;                       // min is mid or in left half
    }
    return nums[lo];                       // lo == hi == minimum index
}

// With duplicates (LC 154):
public int findMinWithDups(int[] nums) {
    int lo = 0, hi = nums.length - 1;
    while (lo < hi) {
        int mid = lo + (hi - lo) / 2;
        if (nums[mid] > nums[hi])       lo = mid + 1;
        else if (nums[mid] < nums[hi])  hi = mid;
        else                             hi--;   // can't determine — shrink right
    }
    return nums[lo];
}
