01
Two Sum — Find two numbers that add up to a target
Easy ArrayHashMapMost Asked
▾
Pattern
HashMap for O(1) complement lookup. For any number x, the complement is target - x. Store each number's index in a map while iterating — if complement already exists in map, return both indices.
P
Problem: Given an array of integers nums and an integer target, return indices of the two numbers that add up to target. Exactly one solution exists. Cannot use the same element twice.

Input: nums = [2, 7, 11, 15], target = 9
Output: [0, 1] — because nums[0] + nums[1] = 2 + 7 = 9
1
Brute Force O(n²): Two nested loops — check every pair. For each element i, check all j > i. Simple but fails for large inputs (10⁴ elements = 10⁸ operations). Never say this as your final answer — say it as a starting point.
2
Optimal — HashMap O(n): Single pass. For each number at index i, compute complement = target - nums[i]. Check if complement is already in the map. If yes — found the pair. If no — store nums[i] → i in the map and continue. The map answers "have I seen the complement before?" in O(1).
3
Why Store BEFORE Checking? No — Check FIRST, Then Store: Check complement first (to avoid using same element twice), then store current element. If target=6 and nums=[3,3], we need indices [0,1] not [0,0]. Checking before storing prevents accidentally matching an element with itself.
4
Edge Cases: Negative numbers (handled naturally by HashMap), duplicate values in array (handled because we check before storing), target = 0 (works fine), empty or single-element array (no solution — but problem guarantees exactly one solution).
Time
O(n)
Space
O(n)
Pattern
HashMap
// Approach 1: HashMap — O(n) time, O(n) space
public int[] twoSum(int[] nums, int target) {
    Map<Integer, Integer> map = new HashMap<>();
    for (int i = 0; i < nums.length; i++) {
        int complement = target - nums[i];
        if (map.containsKey(complement)) {
            return new int[]{map.get(complement), i};
        }
        map.put(nums[i], i); // store AFTER checking to avoid self-match
    }
    throw new IllegalArgumentException("No solution found");
}

// Java 8 Stream version (shows Java 8 skill — good to mention)
public int[] twoSumStream(int[] nums, int target) {
    Map<Integer, Integer> map = new HashMap<>();
    return IntStream.range(0, nums.length)
        .filter(i -> {
            if (map.containsKey(target - nums[i])) return true;
            map.put(nums[i], i);
            return false;
        })
        .mapToObj(i -> new int[]{map.get(target - nums[i]), i})
        .findFirst()
        .orElseThrow();
}
nums = [2, 7, 11, 15], target = 9 ────────────────────────────────────── i=0: nums[0]=2, complement=9-2=7 map={}, 7 not in map → store: map={2→0} i=1: nums[1]=7, complement=9-7=2 map={2→0}, 2 IS in map! → return [map.get(2), 1] = [0, 1] ────────────────────────────────────── Answer: [0, 1] ✓ (nums[0]+nums[1] = 2+7 = 9)
Interviewers love: "I start with the complement idea — for any nums[i], I need target-nums[i] to have appeared before. HashMap gives O(1) lookup. Single pass — check then store."
02
Best Time to Buy and Sell Stock — Maximum profit from one transaction
Easy ArraySliding MinFintech Favourite
▾
Pattern
Track minimum price seen so far. At each day, profit = current price − minimum price so far. Track maximum profit seen. One pass, O(1) space. This pattern — "track best so far, compute delta" — appears in many array problems.
P
Problem: Given array prices where prices[i] is the stock price on day i. Maximize profit by buying on one day and selling on a later day. If no profit is possible, return 0.

Input: prices = [7, 1, 5, 3, 6, 4]
Output: 5 — Buy day 1 (price=1), sell day 4 (price=6), profit = 5
1
Key Insight: To maximize profit on day i, you want to have bought at the lowest price seen so far (days 0 to i-1). So maintain a running minimum. For each day, compute profit = price[i] - minPrice. Update maxProfit if this is better.
2
Why NOT sort? Sorting destroys time order. You must BUY before SELL — sorting removes this constraint. Many candidates make this mistake.
3
Why NOT two pointers? Two pointers work when both pointers move independently. Here the buy point (minPrice) updates greedily — whenever we see a new minimum, we reset the buy point. It's more naturally modeled as "track running minimum."
4
Follow-Up — Multiple Transactions: If allowed to buy/sell multiple times, greedily collect every upward slope: sum all max(0, prices[i] - prices[i-1]). This is a different problem — mention it proactively.
Time
O(n)
Space
O(1)
Pattern
Running Min
public int maxProfit(int[] prices) {
    int minPrice = Integer.MAX_VALUE;
    int maxProfit = 0;

    for (int price : prices) {
        if (price < minPrice) {
            minPrice = price;           // new buy opportunity found
        } else {
            maxProfit = Math.max(maxProfit, price - minPrice);
        }
    }
    return maxProfit;
}

// Java 8 Stream version — elegant but slightly harder to read
public int maxProfitStream(int[] prices) {
    int[] minPrice = {Integer.MAX_VALUE};
    int[] maxProfit = {0};
    Arrays.stream(prices).forEach(price -> {
        minPrice[0] = Math.min(minPrice[0], price);
        maxProfit[0] = Math.max(maxProfit[0], price - minPrice[0]);
    });
    return maxProfit[0];
}

// Follow-up: Multiple transactions (Leetcode 122)
public int maxProfitMultiple(int[] prices) {
    return IntStream.range(1, prices.length)
        .map(i -> Math.max(0, prices[i] - prices[i - 1]))
        .sum();
}
prices = [7, 1, 5, 3, 6, 4] ────────────────────────────────────────────── day 0: price=7 minPrice=7 profit=0 maxProfit=0 day 1: price=1 minPrice=1 profit=0 maxProfit=0 day 2: price=5 minPrice=1 profit=5-1=4 maxProfit=4 day 3: price=3 minPrice=1 profit=3-1=2 maxProfit=4 day 4: price=6 minPrice=1 profit=6-1=5 maxProfit=5 day 5: price=4 minPrice=1 profit=4-1=3 maxProfit=5 ────────────────────────────────────────────── Answer: 5 ✓ (Buy at 1, sell at 6)
03
Container With Most Water — Maximum area between two heights
Medium Two PointersGreedyMust Know
▾
Pattern — Two Pointers (Shrink from Ends)
Place pointers at both ends. Area = min(height[L], height[R]) × (R - L). Always move the pointer with the SHORTER height inward — because keeping the shorter side and moving the taller side can never increase the area (width decreases AND height is still limited by the shorter side).
P
Problem: Given array height of n non-negative integers representing heights of lines, find two lines that together with the x-axis forms a container with the most water.

Input: height = [1, 8, 6, 2, 5, 4, 8, 3, 7]
Output: 49 — Lines at index 1 (h=8) and 8 (h=7), area = min(8,7) × (8-1) = 7×7 = 49
1
Why Move the Shorter Pointer? Proof by contradiction: If we move the taller pointer inward, width decreases AND the limiting height stays at the shorter side (or gets worse). We can NEVER do better by moving the taller pointer. So the only hope for improvement is moving the shorter pointer — it might find a taller line that overcomes the width reduction.
2
The Loop Invariant: At every step, the maximum area achievable with the current pointer pair has been considered. When L and R meet, all pairs have been evaluated. This greedy proof is what makes two pointers work — it's important to be able to explain WHY it works, not just WHAT it does.
3
Common Mistake: Moving both pointers when heights are equal. Move either one (or either — both are fine). The proof holds because both sides are the bottleneck when equal, and moving either can only maintain or improve.
Time
O(n)
Space
O(1)
Pattern
Two Pointers
public int maxArea(int[] height) {
    int left = 0, right = height.length - 1;
    int maxArea = 0;

    while (left < right) {
        int h = Math.min(height[left], height[right]);
        int area = h * (right - left);
        maxArea = Math.max(maxArea, area);

        // Move the shorter side inward — only way to potentially improve
        if (height[left] < height[right]) left++;
        else right--;
    }
    return maxArea;
}

// Alternative: Functional-ish using IntStream.iterate (Java 9+)
// For Java 8, the imperative version above is cleaner — use it

// Brute force O(n²) — only mention to show you started from basics:
public int maxAreaBrute(int[] height) {
    return IntStream.range(0, height.length)
        .boxed()
        .flatMapToInt(i -> IntStream.range(i + 1, height.length)
            .map(j -> Math.min(height[i], height[j]) * (j - i)))
        .max().orElse(0);
}

Longest Substring Without Repeating Characters — Sliding Window
Medium Sliding WindowHashMapExtremely Frequent
▾
Pattern — Sliding Window (Variable Size)
Maintain a window [left, right] where all characters are unique. Use a HashMap to store each character's last seen index. When a duplicate is found at index i, jump left past the duplicate's last position. The window always contains unique characters — no need to manually shrink one step at a time.
P
Problem: Given a string s, find the length of the longest substring without repeating characters.

Input: s = "abcabcbb"
Output: 3 — "abc" (indices 0-2)
1
The Key Insight — Jump Don't Slide: Instead of moving left one step at a time when a duplicate is found, jump left directly to (lastSeen[char] + 1). This skips all positions that would still contain the duplicate. Result: each character is processed at most twice — O(n) total.
2
Critical Guard — left = max(left, lastSeen[c]+1): When you see a character again, its last-seen index might be to the LEFT of the current window. Without the max(), you'd move left backwards — bug! Always take max to ensure left only moves forward.
3
Alternative — HashSet: Use a Set instead of Map. When duplicate found, keep removing from Set and advancing left until duplicate is removed. Simpler to understand, but worst-case O(2n) instead of O(n). Map approach is more elegant.
4
Space Optimization: Since inputs are ASCII (128 chars), replace HashMap with int[128] array initialized to -1. Array access is faster than HashMap. For Unicode, keep HashMap.
Time
O(n)
Space
O(min(m,n))
Pattern
Sliding Window
// Optimal: HashMap stores last seen index → O(n)
public int lengthOfLongestSubstring(String s) {
    Map<Character, Integer> lastSeen = new HashMap<>();
    int maxLen = 0;
    int left = 0;

    for (int right = 0; right < s.length(); right++) {
        char c = s.charAt(right);
        if (lastSeen.containsKey(c)) {
            // Jump left past the previous occurrence
            // max() ensures left never moves backward
            left = Math.max(left, lastSeen.get(c) + 1);
        }
        lastSeen.put(c, right);
        maxLen = Math.max(maxLen, right - left + 1);
    }
    return maxLen;
}

// Space-optimised: int[128] for ASCII — faster than HashMap
public int lengthOfLongestSubstringAscii(String s) {
    int[] lastSeen = new int[128];
    Arrays.fill(lastSeen, -1);
    int maxLen = 0, left = 0;

    for (int right = 0; right < s.length(); right++) {
        char c = s.charAt(right);
        if (lastSeen[c] >= left) left = lastSeen[c] + 1;
        lastSeen[c] = right;
        maxLen = Math.max(maxLen, right - left + 1);
    }
    return maxLen;
}
s = "abcabcbb" ───────────────────────────────────────────────── right=0: c='a' left=0 window="a" map={a:0} len=1 right=1: c='b' left=0 window="ab" map={b:1} len=2 right=2: c='c' left=0 window="abc" map={c:2} len=3 right=3: c='a' a seen at 0 left=max(0,0+1)=1 window="bca" map={a:3} len=3 right=4: c='b' b seen at 1 left=max(1,1+1)=2 window="cab" map={b:4} len=3 right=5: c='c' c seen at 2 left=max(2,2+1)=3 window="abc" map={c:5} len=3 right=6: c='b' b seen at 4 left=max(3,4+1)=5 window="cb" map={b:6} len=2 right=7: c='b' b seen at 6 left=max(5,6+1)=7 window="b" map={b:7} len=1 ───────────────────────────────────────────────── Answer: 3 ✓ ("abc")
The max() guard is the most common bug. Practice saying: "I take max(left, lastSeen+1) so the window never shrinks backward — only moves forward."
05
Product of Array Except Self — No division allowed
Medium ArrayPrefix ProductRazorpay / Groww Asked
▾
Pattern — Prefix & Suffix Products
For index i, result[i] = product of all elements to the LEFT × product of all elements to the RIGHT. Build a prefix product array (left to right), then multiply by suffix product (right to left) in a second pass. O(n) time, O(1) extra space (output array doesn't count).
P
Problem: Given integer array nums, return array output where output[i] = product of all elements of nums except nums[i]. Must run in O(n), cannot use division.

Input: nums = [1, 2, 3, 4]
Output: [24, 12, 8, 6]
1
Why No Division? Division approach: totalProduct / nums[i]. Breaks when array contains zero (division by zero). The constraint "no division" forces a cleaner, more general solution. Also: interviewers use this constraint to test if you can think beyond the obvious.
2
Prefix Pass (Left to Right): result[i] = product of all elements to the LEFT of i. result[0] = 1 (nothing to the left). result[1] = nums[0]. result[2] = nums[0]*nums[1]. etc.
3
Suffix Pass (Right to Left): Maintain a running rightProduct initialized to 1. For each i from right to left: multiply result[i] by rightProduct, then update rightProduct *= nums[i]. This multiplies in the right side product without a separate suffix array — O(1) extra space.
4
Follow-Up — With Zeros: If the array has exactly one zero, only the position of that zero has a non-zero result (product of all others). If two or more zeros, all results are zero. This can be handled with the same algorithm naturally — the zero product propagates correctly.
Time
O(n)
Space
O(1)*
Pattern
Prefix×Suffix
* O(1) extra space — output array doesn't count
public int[] productExceptSelf(int[] nums) {
    int n = nums.length;
    int[] result = new int[n];

    // Pass 1: Fill result with prefix products (product of all LEFT of i)
    result[0] = 1;
    for (int i = 1; i < n; i++) {
        result[i] = result[i - 1] * nums[i - 1];
    }

    // Pass 2: Multiply by suffix product (product of all RIGHT of i)
    int rightProduct = 1;
    for (int i = n - 1; i >= 0; i--) {
        result[i] *= rightProduct;
        rightProduct *= nums[i];
    }

    return result;
}

When interviewer says "no division" — immediately think Prefix × Suffix. This constraint is the hint. Say: "Without division, I can compute for each index the product of everything to its left, and separately everything to its right. I can do both passes in O(n) using O(1) extra space."
