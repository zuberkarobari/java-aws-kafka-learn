01
What is a Lambda Expression in Java 8? Why was it introduced?
Core ConceptMust Know
▾
This is the starting question of every Java 8 interview. Explain the WHY, not just the syntax.
  • 01
    Definition: A Lambda expression is an anonymous function — a block of code with parameters, a body, and no name. It implements a functional interface (an interface with exactly one abstract method) inline, without needing a concrete class or anonymous inner class. Syntax: (parameters) -> expression or (parameters) -> { statements }.
  • 02
    Why Introduced: Before Java 8, to pass behaviour as an argument, you had to create an anonymous inner class — verbose and hard to read. Java 8 lambdas enable functional programming style: treat functions as first-class citizens, pass behaviour to methods, write concise and expressive code. The primary driver was the Collections API — enabling parallel, pipeline-style processing of data via the Stream API.
  • 03
    Before vs After: 
    // Before Java 8 — Anonymous Inner Class
Runnable r = new Runnable() {
    public void run() { System.out.println("Hello"); }
};

// Java 8 — Lambda
Runnable r = () -> System.out.println("Hello");
  • 04
    Lambda Syntax Rules: (1) If only one parameter, parentheses are optional: x -> x * 2. (2) If body is a single expression, braces and return keyword are optional: (a, b) -> a + b. (3) If body has multiple statements, braces and return are required: (a, b) -> { int sum = a+b; return sum; }.
  • 05
    Variable Capture: Lambdas can capture variables from the enclosing scope but those variables must be effectively final (assigned exactly once and never modified after). This is because lambdas may run in a different thread and mutable shared state would cause race conditions. If you try to modify a captured variable inside a lambda, the compiler gives: "local variables referenced from a lambda must be final or effectively final".
  • 06
    Under the Hood — invokedynamic: Java 8 compiles lambdas using the invokedynamic bytecode instruction (not as anonymous inner classes). At runtime, the JVM dynamically links the lambda to the functional interface. This is more efficient than creating new Class objects for each lambda — the JVM can apply runtime optimizations like inlining and escape analysis that it can't apply to inner class instances.
If asked "what is the difference between lambda and anonymous inner class?" — key point: lambda is NOT syntactic sugar for anonymous inner class. Lambdas use invokedynamic; anonymous classes always create a new .class file. Also, this inside a lambda refers to the enclosing class; this inside an anonymous inner class refers to the inner class itself.

What are Functional Interfaces? Explain Predicate, Function, Consumer, Supplier, and BiFunction.
Functional InterfacesMust Know
▾
Know each built-in FI by its signature: what goes IN and what comes OUT. That's how you remember them.
  • 01
    Functional Interface Definition: An interface with exactly one abstract method, annotated with @FunctionalInterface (optional but recommended — makes the compiler enforce the single-abstract-method contract). It can have any number of default or static methods. Examples: Runnable, Comparator, and all interfaces in java.util.function package.
  • 02
    Predicate<T> — T → boolean: Tests a condition. Use for filtering. 
    Predicate<Integer> isEven = n -> n % 2 == 0;
isEven.test(4); // true

// Combine predicates
Predicate<Integer> isEvenAndPositive = isEven.and(n -> n > 0);
Predicate<Integer> isEvenOrNegative = isEven.or(n -> n < 0);
Predicate<Integer> isOdd = isEven.negate();
  • 03
    Function<T, R> — T → R: Transforms input to output. Use for mapping. 
    Function<String, Integer> strLength = s -> s.length();
strLength.apply("Hello"); // 5

// Compose: andThen (first this, then that) / compose (first that, then this)
Function<Integer, Integer> doubleIt = x -> x * 2;
Function<String, Integer> lengthThenDouble = strLength.andThen(doubleIt);
lengthThenDouble.apply("Hello"); // 10
  • 04
    Consumer<T> — T → void: Accepts input, performs side effect, returns nothing. Use for forEach. 
    Consumer<String> printer = s -> System.out.println(s);
printer.accept("Java 8"); // prints: Java 8

// andThen: chain consumers
Consumer<String> upperPrinter = s -> System.out.println(s.toUpperCase());
printer.andThen(upperPrinter).accept("java"); // java \n JAVA
  • 05
    Supplier<T> — () → T: Takes no input, returns a value. Use for lazy initialization, factory methods. 
    Supplier<String> greeting = () -> "Hello, World!";
greeting.get(); // "Hello, World!"

// Common use: Optional.orElseGet(supplier) — only evaluated if empty
Optional<String> opt = Optional.empty();
opt.orElseGet(() -> fetchFromDB()); // fetchFromDB only called if empty
  • 06
    BiFunction<T, U, R> — (T, U) → R: Two inputs, one output. Also: BiConsumer<T,U> (two inputs, void), BiPredicate<T,U> (two inputs, boolean), UnaryOperator<T> (extends Function, T→T same type), BinaryOperator<T> (extends BiFunction, (T,T)→T). 
    BiFunction<String, Integer, String> repeat =
    (s, n) -> s.repeat(n);
repeat.apply("ab", 3); // "ababab"

What are Method References? Explain all 4 types with examples.
Method ReferencesFrequently Asked
▾
Method references are shorthand for lambdas that call exactly one method. Know all 4 types — interviewers love listing them.
  • 01
    What is a Method Reference: A compact syntax ClassName::methodName that creates a lambda equivalent for any method that matches the functional interface's signature. It's a special kind of lambda — instead of a body, it points to an existing method. There are 4 types: static method reference, instance method reference (on a specific object), instance method reference (on arbitrary object of a type), and constructor reference.
  • 02
    Type 1 — Static Method Reference: ClassName::staticMethod 
    // Lambda equivalent: x -> Integer.parseInt(x)
Function<String, Integer> parser = Integer::parseInt;
parser.apply("42"); // 42

List<String> nums = Arrays.asList("3","1","2");
nums.stream().map(Integer::parseInt).sorted().forEach(System.out::println);
  • 03
    Type 2 — Instance Method on Specific Object: instance::method 
    String prefix = "Hello, ";
Function<String, String> greeter = prefix::concat;
greeter.apply("Zuber"); // "Hello, Zuber"

// Most common: System.out::println
List.of("a","b","c").forEach(System.out::println);
  • 04
    Type 3 — Instance Method on Arbitrary Object of Type: ClassName::instanceMethod 
    // Lambda: s -> s.toUpperCase()
Function<String, String> upper = String::toUpperCase;
upper.apply("hello"); // "HELLO"

// Lambda: (s1, s2) -> s1.compareTo(s2)
List<String> names = Arrays.asList("Zuber","Amreen","Ali");
names.sort(String::compareTo);
  • 05
    Type 4 — Constructor Reference: ClassName::new 
    // Supplier<T> creates objects with no-arg constructor
Supplier<ArrayList> listFactory = ArrayList::new;
ArrayList list = listFactory.get(); // new empty ArrayList

// Function<String, Employee> creates objects with one arg
Function<String, Employee> empFactory = Employee::new;
Employee e = empFactory.apply("Zuber"); // new Employee("Zuber")
  • 06
    When to Use: Use method references when a lambda does nothing but delegate to an existing method without transformation. Rule of thumb: If your lambda is x -> someMethod(x), replace with SomeClass::someMethod. If the lambda has additional logic beyond the method call, keep it as a lambda. Method references make code more readable but only when the reader can recognize the referenced method at a glance.

What are Default and Static methods in interfaces? Why were they added?
Interface ChangesDesign
▾
This was added specifically to allow backward-compatible evolution of the Collections API without breaking existing implementations.
  • 01
    The Problem They Solve: Java 8 needed to add stream(), forEach(), removeIf() to the Collection interface. But if you add abstract methods to an interface, all existing implementations break (millions of classes). Default methods allow adding new methods with a default implementation to existing interfaces without breaking any implementing class. This is the "interface evolution" problem solved.
  • 02
    Default Method Syntax: 
    interface Greeter {
    void greet(String name); // abstract

    default void greetLoudly(String name) { // default
        greet(name.toUpperCase());
    }

    static Greeter formal() { // static factory method
        return name -> System.out.println("Good day, " + name);
    }
}
  • 03
    Diamond Problem Resolution: If a class implements two interfaces that both have a default method with the same signature, it must override the method to resolve ambiguity: 
    interface A { default void hello() { System.out.println("A"); } }
interface B { default void hello() { System.out.println("B"); } }

class C implements A, B {
    public void hello() { A.super.hello(); } // must override, can choose
}
  • 04
    Static Methods in Interfaces: Static methods in interfaces belong to the interface, not to implementing classes or their instances. They cannot be overridden by implementing classes. Used for factory methods and utility methods tightly coupled to the interface contract. Example: Comparator.comparing(), Predicate.not().
  • 05
    Default Methods vs Abstract Classes: Key distinctions — (1) A class can implement multiple interfaces (with default methods) but can only extend one abstract class. (2) Abstract classes can have instance state (fields); interfaces cannot. (3) Abstract classes can have constructors; interfaces cannot. (4) Default methods are meant for interface evolution, not as a replacement for abstract classes when shared state is needed.
  • 06
    Real Example from JDK: Collection.stream(), Collection.forEach(Consumer), List.sort(Comparator), Map.getOrDefault(), Map.forEach(BiConsumer), Map.putIfAbsent(), Iterable.forEach() — these are all default methods added to existing JDK interfaces in Java 8. Without default methods, every ArrayList, LinkedList, HashMap etc. would have needed to be updated separately.

Explain Comparator enhancements in Java 8 — comparing(), thenComparing(), reversed().
ComparatorCoding
▾
Comparator chaining is used in almost every sorting coding question. Know these methods cold.
  • 01
    Comparator.comparing() — Static Factory: 
    // Sort employees by name
List<Employee> emps = getEmployees();
emps.sort(Comparator.comparing(Employee::getName));

// Sort by salary descending
emps.sort(Comparator.comparing(Employee::getSalary).reversed());
  • 02
    thenComparing() — Multi-Field Sort: 
    // Sort by department, then by salary desc within department
emps.sort(
    Comparator.comparing(Employee::getDepartment)
              .thenComparing(Comparator.comparing(Employee::getSalary).reversed())
);
  • 03
    comparingInt / comparingDouble / comparingLong: Avoids boxing overhead for primitive key extractors. 
    // Prefer comparingInt over comparing for int fields
emps.sort(Comparator.comparingInt(Employee::getAge));

// Natural order and reverse order constants
List<String> names = Arrays.asList("Zuber","Ali","Amreen");
names.sort(Comparator.naturalOrder());
names.sort(Comparator.reverseOrder());
  • 04
    nullsFirst / nullsLast: 
    // Handle nulls gracefully — put them first
emps.sort(Comparator.comparing(Employee::getName,
          Comparator.nullsFirst(Comparator.naturalOrder())));
  • 05
    Stream.sorted() with Comparator: 
    // Top 3 highest paid employees
emps.stream()
    .sorted(Comparator.comparingDouble(Employee::getSalary).reversed())
    .limit(3)
    .collect(Collectors.toList());
  • 06
    Custom Key Extractor with Lambda: 
    // Sort strings by their length, then alphabetically
List<String> words = Arrays.asList("banana", "apple", "kiwi", "fig");
words.sort(Comparator.comparingInt(String::length)
                    .thenComparing(Comparator.naturalOrder()));
// [fig, kiwi, apple, banana]
Section 02 Stream API — Internals & Operations Q6–Q14
06
What is the Stream API? Explain lazy evaluation, intermediate vs terminal operations.
Stream CoreMust Know
▾
Lazy evaluation is the most important concept in Stream internals — it's what makes streams efficient and the most common interview trick question.
  • 01
    What is a Stream: A Stream is a sequence of elements supporting sequential and parallel aggregate operations. It is NOT a data structure — it doesn't store data. It's a pipeline that processes data from a source (Collection, array, I/O channel) through a chain of operations. Streams are single-use — once a terminal operation is called, the stream is consumed and cannot be reused.
  • 02
    Lazy Evaluation — The Critical Concept: Intermediate operations (filter, map, etc.) are lazy — they don't execute until a terminal operation is called. The pipeline is built first, then executed optimally when the terminal operation triggers it. 
    Stream<String> stream = List.of("a","bb","ccc")
    .stream()
    .filter(s -> { System.out.println("filter: "+s); return s.length() > 1; })
    .map(s -> { System.out.println("map: "+s); return s.toUpperCase(); });
// Nothing printed yet! No terminal operation called.
stream.findFirst(); // NOW it executes — and only for "bb", stops early!
  • 03
    Intermediate Operations (lazy, return Stream): filter(Predicate), map(Function), flatMap(Function), distinct(), sorted(), sorted(Comparator), peek(Consumer), limit(n), skip(n), mapToInt/Long/Double(). These are lazy — they build the pipeline but do not execute.
  • 04
    Terminal Operations (eager, trigger execution): collect(Collector), forEach(Consumer), reduce(BinaryOperator), count(), findFirst(), findAny(), anyMatch(), allMatch(), noneMatch(), min(Comparator), max(Comparator), toArray(). These are eager — they execute the entire pipeline and return a result.
  • 05
    Short-Circuit Operations: Some operations don't need to process the entire stream — they stop early. findFirst(), findAny(), anyMatch(), allMatch(), noneMatch(), limit(n). This makes combinations like filter().findFirst() very efficient — the filter and findFirst together process only until the first match is found.
  • 06
    Stream Creation Methods: 
    List.of(1,2,3).stream()                          // from Collection
Arrays.stream(new int[]{1,2,3})                // from array
Stream.of("a", "b", "c")                        // from varargs
Stream.iterate(0, n -> n + 1)                  // infinite ordered stream
Stream.generate(Math::random)                   // infinite unordered stream
IntStream.range(1, 11)                          // primitive stream 1..10
IntStream.rangeClosed(1, 10)                   // same as above

Explain map() vs flatMap() with examples. When do you use flatMap?
Stream OpsFrequently Asked
▾
flatMap is asked in almost every interview. The one-liner: map wraps, flatMap flattens.
  • 01
    map() — One-to-One Transformation: map(Function<T, R>) transforms each element to exactly one output. The result is a Stream<R>. 
    List<String> words = List.of("hello", "world");
words.stream()
     .map(String::toUpperCase)
     .collect(Collectors.toList());
// [HELLO, WORLD]
  • 02
    The Problem flatMap Solves — Nested Streams: If the mapping function returns a Stream (or collection), map() produces a Stream<Stream<T>> — nested. flatMap() maps and then flattens one level, producing a Stream<T>. 
    List<List<Integer>> nested = List.of(
    List.of(1,2,3), List.of(4,5), List.of(6)
);

// map gives Stream<Stream<Integer>> — WRONG
nested.stream().map(List::stream); // Stream<Stream> — NOT what we want

// flatMap flattens — gives Stream<Integer>
nested.stream()
      .flatMap(List::stream)
      .collect(Collectors.toList());
// [1, 2, 3, 4, 5, 6]
  • 03
    Classic Example — Split Sentences into Words: 
    List<String> sentences = List.of("Hello World", "Java Eight");
sentences.stream()
         .flatMap(s -> Arrays.stream(s.split(" ")))
         .collect(Collectors.toList());
// [Hello, World, Java, Eight]
  • 04
    flatMap with Optional: A common pattern is to flatMap over Optional to chain optional-producing operations without nested Optionals: 
    Optional<String> opt = Optional.of("hello");
Optional<String> result = opt.flatMap(s -> findUpperCase(s));
// vs opt.map(...) which gives Optional<Optional<String>>
  • 05
    Get All Characters from List of Strings: 
    List<String> names = List.of("Zuber", "Ali");
names.stream()
     .flatMap(name -> name.chars()
                          .mapToObj(c -> (char) c)
                          .map(String::valueOf))
     .distinct()
     .sorted()
     .collect(Collectors.joining(", "));
// A, Z, a, b, e, i, l, r, u
  • 06
    mapToInt, mapToLong, mapToDouble: For primitive streams (no boxing overhead): 
    List<String> words2 = List.of("apple", "banana", "kiwi");
int totalLength = words2.stream()
                         .mapToInt(String::length)
                         .sum(); // 15

double avgLength = words2.stream()
                          .mapToInt(String::length)
                          .average()
                          .orElse(0.0); // 5.0

Explain reduce() operation — how does it work and when do you use it?
Stream OpsTricky
▾
reduce() is the most powerful terminal operation — many other operations (sum, count, min, max) are specialized reduces.
  • 01
    What reduce() Does: reduce() combines all stream elements into a single result by repeatedly applying a BinaryOperator. It takes two forms: with an identity value (always returns T) and without (returns Optional<T> because the stream may be empty). 
    // With identity — returns T (safe, handles empty stream)
int sum = IntStream.rangeClosed(1, 10)
                   .reduce(0, Integer::sum); // 55

// Without identity — returns Optional<T>
Optional<Integer> product = Stream.of(1,2,3,4,5)
    .reduce((a, b) -> a * b); // Optional[120]
  • 02
    Custom Reductions: 
    // Concatenate all strings with comma
String joined = Stream.of("a","b","c")
    .reduce("", (a, b) -> a.isEmpty() ? b : a + "," + b);
// "a,b,c"  (use Collectors.joining for this in practice)

// Find max manually using reduce
Optional<Integer> max = Stream.of(3,1,4,1,5,9)
    .reduce(Integer::max); // Optional[9]
  • 03
    Three-Argument reduce for Parallel Streams: reduce(identity, accumulator, combiner) — the combiner merges partial results from parallel sub-streams. 
    // Sum of string lengths in parallel
int total = Stream.of("Java","Eight","Rocks")
    .parallel()
    .reduce(0,
        (sum, s) -> sum + s.length(),  // accumulator
        Integer::sum);               // combiner
// 14
  • 04
    reduce vs collect: reduce() is for immutable reduction — it combines values into a new value each step. collect() is for mutable reduction — it accumulates into a mutable container (List, Map, StringBuilder). For building collections, always use collect(), not reduce(). Reducing into a list with reduce creates a new list on every step — O(n²) time and memory.
  • 05
    Factorial using reduce: 
    long factorial(int n) {
    return LongStream.rangeClosed(1, n)
                     .reduce(1L, (a, b) -> a * b);
}
factorial(5); // 120
  • 06
    Common Reduce Patterns: 
    // Sum    .reduce(0, Integer::sum)
// Max    .reduce(Integer.MIN_VALUE, Integer::max)
// Min    .reduce(Integer.MAX_VALUE, Integer::min)
// Count .reduce(0, (acc, e) -> acc + 1, Integer::sum)
// Any string concatenation
// .reduce("", String::concat)  ← works but use Collectors.joining instead

What is the difference between findFirst() and findAny()? When would you use each?
Stream OpsParallel
▾
This is a short but tricky question about sequential vs parallel stream behavior. Know the parallel angle.
  • 01
    findFirst() — Ordered, Deterministic: Returns the first element of the stream in encounter order as an Optional<T>. In a sequential stream this is simply the first element. In a parallel stream, even though elements are processed in parallel, findFirst() still guarantees returning the first element in encounter order — this requires coordination between parallel threads, which adds overhead. 
    Optional<Integer> first = Stream.of(1,2,3,4,5)
    .filter(n -> n % 2 == 0)
    .findFirst(); // always Optional[2]
  • 02
    findAny() — Nondeterministic, Faster in Parallel: Returns any element that matches. In a sequential stream it typically returns the first element (same as findFirst). In a parallel stream, whichever thread finds a match first returns it — no coordination needed. This is faster in parallel at the cost of predictability. 
    Optional<Integer> any = Stream.of(1,2,3,4,5)
    .parallel()
    .filter(n -> n % 2 == 0)
    .findAny(); // Could be Optional[2], [4] — not guaranteed
  • 03
    When to Use Which: Use findFirst() when order matters and you need a deterministic result — e.g., finding the oldest record, the first transaction in a batch, the first error in a validation chain. Use findAny() when you just need "any" match and are processing in parallel for performance — e.g., checking if any record satisfies a condition in a large dataset.
  • 04
    anyMatch vs findAny: anyMatch(Predicate) returns a boolean — just tells you if any element matches. findAny() returns Optional<T> — gives you the actual matching element. Use anyMatch for existence checks; use findAny when you need the element itself.
  • 05
    allMatch, noneMatch: 
    List<Integer> nums = List.of(2,4,6,8);
nums.stream().allMatch(n -> n % 2 == 0); // true — all even
nums.stream().noneMatch(n -> n % 2 != 0); // true — none odd
nums.stream().anyMatch(n -> n > 6);       // true — 8 exists
// All three are short-circuit operations
  • 06
    Handling Empty Optional: 
    Optional<Integer> result = stream.filter(pred).findFirst();

result.isPresent();                  // check if value exists
result.get();                        // get value (throws if empty!)
result.orElse(-1);                  // default if empty
result.orElseGet(() -> computeDefault()); // lazy default
result.orElseThrow(() -> new RuntimeException()); // throw if empty
result.ifPresent(v -> process(v));   // action only if present

What are parallel streams? When should you use them and what are the risks?
Parallel StreamsSenior Topic
▾
Parallel streams are a double-edged sword. Show you know the risks — that's what impresses senior interviewers.
  • 01
    How Parallel Streams Work: collection.parallelStream() or stream.parallel() uses the ForkJoinPool.commonPool() to split the source into sub-streams, process them concurrently on multiple threads, and merge results. The number of threads defaults to Runtime.getRuntime().availableProcessors() - 1. 
    long sum = LongStream.rangeClosed(1, 1_000_000)
                     .parallel()
                     .sum(); // faster on multi-core for large datasets
  • 02
    When Parallel Helps: (1) Large data sets (typically >10K elements), (2) CPU-bound operations per element (heavy computation), (3) Operations that can be split and merged independently (stateless, non-interfering lambdas), (4) The sequential stream is a measured bottleneck. For small collections or simple operations, parallel stream overhead (thread coordination, ForkJoin overhead) makes it slower than sequential.
  • 03
    Risk 1 — Shared Mutable State: 
    // WRONG — race condition with parallel stream
List<Integer> results = new ArrayList<>();
Stream.of(1,2,3).parallel().forEach(results::add); // NOT thread-safe!

// CORRECT — use collect() which is thread-safe
List<Integer> safe = Stream.of(1,2,3)
    .parallel()
    .collect(Collectors.toList()); // thread-safe
  • 04
    Risk 2 — Encounter Order: If the source stream is ordered (List), parallel operations preserve order unless you use unordered(). Operations like forEach may print in any order in parallel. Use forEachOrdered() if order matters (but this serializes the terminal operation, reducing parallelism benefit).
  • 05
    Risk 3 — Common ForkJoinPool Saturation: All parallel streams share the JVM's common ForkJoinPool. If one parallel stream runs long blocking operations (I/O, DB calls), it starves other parallel streams in the application. For I/O-bound work, use a custom ForkJoinPool or async APIs (CompletableFuture) instead of parallel streams.
  • 06
    When NOT to Use Parallel Streams: (1) Small collections (<1000 elements), (2) I/O-bound operations — threads block, defeating parallelism, (3) Stateful lambdas that share mutable state, (4) Operations that require strict encounter order, (5) When you need deterministic results from findAny(). Rule: Always measure before switching to parallel — don't assume parallel = faster.

Explain Collectors.groupingBy() — all variants with examples.
CollectorsMost Used
▾
groupingBy is the most important Collector. It comes up in almost every practical Java 8 coding round.
  • 01
    Basic groupingBy — Group by field: 
    // Group employees by department
Map<String, List<Employee>> byDept =
    employees.stream()
             .collect(Collectors.groupingBy(Employee::getDepartment));
// {IT=[e1,e2], HR=[e3], Finance=[e4,e5]}
  • 02
    groupingBy with downstream Collector — Count per group: 
    // Count employees per department
Map<String, Long> countByDept =
    employees.stream()
             .collect(Collectors.groupingBy(
                 Employee::getDepartment,
                 Collectors.counting()
             ));
// {IT=2, HR=1, Finance=2}
  • 03
    groupingBy with averagingDouble — Average per group: 
    // Average salary per department
Map<String, Double> avgSalaryByDept =
    employees.stream()
             .collect(Collectors.groupingBy(
                 Employee::getDepartment,
                 Collectors.averagingDouble(Employee::getSalary)
             ));
  • 04
    groupingBy with mapping — Transform elements in each group: 
    // Department → List of names (not Employee objects)
Map<String, List<String>> namesByDept =
    employees.stream()
             .collect(Collectors.groupingBy(
                 Employee::getDepartment,
                 Collectors.mapping(Employee::getName, Collectors.toList())
             ));
  • 05
    Multi-level groupingBy — Nested grouping: 
    // Group by department, then by gender within department
Map<String, Map<String, List<Employee>>> nested =
    employees.stream()
             .collect(Collectors.groupingBy(
                 Employee::getDepartment,
                 Collectors.groupingBy(Employee::getGender)
             ));
  • 06
    groupingBy with maxBy / minBy: 
    // Highest paid employee per department
Map<String, Optional<Employee>> topEarnerByDept =
    employees.stream()
             .collect(Collectors.groupingBy(
                 Employee::getDepartment,
                 Collectors.maxBy(Comparator.comparingDouble(Employee::getSalary))
             ));

Explain partitioningBy(), joining(), toMap(), and counting() collectors.
CollectorsFrequently Asked
▾
Know each collector's return type. partitioningBy always produces Map<Boolean, List> — that's the key difference from groupingBy.
  • 01
    partitioningBy() — Split into two groups (true/false): 
    // Partition numbers into even and odd
Map<Boolean, List<Integer>> evenOdd =
    Stream.of(1,2,3,4,5)
          .collect(Collectors.partitioningBy(n -> n % 2 == 0));
// {false=[1,3,5], true=[2,4]}

// With downstream: count in each partition
Map<Boolean, Long> count = stream.collect(
    Collectors.partitioningBy(pred, Collectors.counting()));
  • 02
    Collectors.joining() — String concatenation: 
    List<String> names = List.of("Zuber", "Ali", "Amreen");

names.stream().collect(Collectors.joining());
// "ZuberAliAmreen"

names.stream().collect(Collectors.joining(", "));
// "Zuber, Ali, Amreen"

names.stream().collect(Collectors.joining(", ", "[", "]"));
// "[Zuber, Ali, Amreen]"
  • 03
    Collectors.toMap() — Convert to Map: 
    // Name → Salary map
Map<String, Double> salaryMap =
    employees.stream()
             .collect(Collectors.toMap(
                 Employee::getName,    // key
                 Employee::getSalary   // value
             ));

// Handle duplicate keys with merge function
Map<String, Double> withMerge =
    employees.stream()
             .collect(Collectors.toMap(
                 Employee::getDept,
                 Employee::getSalary,
                 Double::sum // merge: sum salaries if same dept key

Section 04 Optional — Null Safety Done Right Q13–Q14
13
What is Optional? What problem does it solve and how do you use it correctly?
OptionalMust Know
▾
Know what Optional is NOT meant for — that's the key interview differentiator.
  • 01
    What it Solves: Optional<T> is a container that may or may not contain a non-null value. It was introduced to reduce NullPointerExceptions and make the possibility of absence explicit in the API. Instead of returning null and hoping the caller checks, a method returning Optional<T> forces the caller to handle the empty case consciously.
  • 02
    Creation: 
    Optional.of("value")          // throws NPE if value is null!
Optional.ofNullable(value)     // safe — handles null, returns empty Optional
Optional.empty()              // explicitly empty Optional
  • 03
    Retrieval — DON'T use get() blindly: 
    // BAD — get() throws NoSuchElementException if empty
String val = opt.get(); // same problem as null.toString()

// GOOD — always handle the empty case
opt.orElse("default")           // default value (always evaluated)
opt.orElseGet(() -> compute())  // lazy — only evaluated if empty
opt.orElseThrow(IllegalStateException::new) // throw custom exception
  • 04
    Chaining with map() and flatMap(): 
    // Safe chaining without null checks
Optional<String> name =
    Optional.ofNullable(user)
            .map(User::getAddress)
            .map(Address::getCity)
            .map(String::toUpperCase);
// Returns Optional.empty() at any null step instead of NPE
  • 05
    filter() on Optional: 
    Optional<String> opt = Optional.of("HELLO");
opt.filter(s -> s.startsWith("H"))
   .map(String::toLowerCase)
   .ifPresent(System.out::println); // "hello"
  • 06
    What Optional is NOT for: (1) Not for method parameters — don't accept Optional as a parameter; use method overloading. (2) Not for fields — Optional is not Serializable, don't use it in entity fields. (3) Not as a collection replacement — use an empty List/Map instead of Optional<List>. (4) Not for primitives — use OptionalInt, OptionalLong, OptionalDouble to avoid boxing.

How does Optional.orElse() differ from Optional.orElseGet()? When does it matter?
OptionalTricky
▾
This is a classic trick question. The answer is about EAGER vs LAZY evaluation — always frame it in terms of performance impact.
  • 01
    orElse() — Eager Evaluation: orElse(T other) evaluates its argument unconditionally, whether the Optional is empty or not. If other is an expensive computation or has side effects, it still runs even when the Optional has a value. 
    opt.orElse(expensiveOperation()); // ALWAYS calls expensiveOperation()
  • 02
    orElseGet() — Lazy Evaluation: orElseGet(Supplier<T>) takes a Supplier which is only evaluated if the Optional is empty. This is the correct choice when the default is expensive to compute, has side effects (DB calls, network requests, object creation). 
    opt.orElseGet(() -> expensiveOperation()); // only calls if opt is empty
  • 03
    Demonstration: 
    Optional<String> opt = Optional.of("present");

// orElse — always evaluates (prints side effect)
opt.orElse(createExpensiveDefault()); // createExpensiveDefault CALLED!

// orElseGet — lazy, skips supplier
opt.orElseGet(() -> createExpensiveDefault()); // NOT called
  • 04
    Rule of Thumb: Use orElse() only for simple constant defaults: orElse("unknown"), orElse(0), orElse(Collections.emptyList()). Use orElseGet() whenever the default involves method calls, object creation, or any computation that has a cost or side effect. In production code, defaulting to orElseGet() is the safer choice — a lambda with a simple return has negligible overhead.
  • 05
    Java 9+ Additions to Optional: ifPresentOrElse(Consumer, Runnable) — act on value if present, or run else action. or(Supplier<Optional>) — if empty, return another Optional from supplier (like orElseGet but returns Optional). stream() — converts Optional to a Stream of 0 or 1 elements, useful for flatMapping. 
    // Java 9 — ifPresentOrElse
opt.ifPresentOrElse(
    v -> System.out.println("Value: " + v),
    () -> System.out.println("Empty")
);
  • 06
    Practical Pattern — findById with Optional: 
    public Optional<Employee> findById(long id) {
    return Optional.ofNullable(repository.get(id));
}

// Caller handles absence explicitly
Employee emp = findById(42L)
    .orElseThrow(() -> new EntityNotFoundException("Employee 42 not found"));

What is CompletableFuture? How does it improve on Future?
CompletableFutureSenior Topic
▾
This is a senior-level Java 8 topic. Frame your answer in terms of the limitations of the old Future API.
  • 01
    Problems with Future: The old Future<T> interface had critical limitations: (1) future.get() blocks the calling thread — no callback mechanism, (2) Cannot chain multiple async operations, (3) No way to combine multiple Futures, (4) Cannot handle exceptions gracefully, (5) Cannot be completed manually. CompletableFuture<T> solves all of these.
  • 02
    Creating CompletableFutures: 
    // supplyAsync — returns a value (Supplier)
CompletableFuture<String> cf =
    CompletableFuture.supplyAsync(() -> fetchDataFromAPI());

// runAsync — fire and forget (Runnable)
CompletableFuture<Void> cf2 =
    CompletableFuture.runAsync(() -> sendEmail());

// With custom executor (don't abuse common pool for I/O)
ExecutorService pool = Executors.newFixedThreadPool(4);
CompletableFuture.supplyAsync(() -> fetch(), pool);
  • 03
    Chaining — thenApply, thenAccept, thenRun: 
    CompletableFuture.supplyAsync(() -> "hello")
    .thenApply(String::toUpperCase)     // transform: Function
    .thenApply(s -> s + "!")
    .thenAccept(System.out::println)   // consume: Consumer, returns Void
    .thenRun(() -> cleanup());          // run after: Runnable
  • 04
    Combining Multiple Futures: 
    CompletableFuture<String> f1 = CompletableFuture.supplyAsync(() -> getUser());
CompletableFuture<String> f2 = CompletableFuture.supplyAsync(() -> getOrder());

// thenCombine — both complete, then combine results
f1.thenCombine(f2, (user, order) -> user + " ordered: " + order);

// allOf — wait for ALL to complete
CompletableFuture.allOf(f1, f2).thenRun(() -> processAll());

// anyOf — continue when FIRST completes
CompletableFuture.anyOf(f1, f2).thenAccept(result -> handle(result));
  • 05
    Error Handling: 
    CompletableFuture.supplyAsync(() -> riskyCall())
    .exceptionally(ex -> {
        log.error("Failed: ", ex);
        return "fallback";         // recovery value
    })
    .handle((result, ex) -> {    // handle both success and error
        if (ex != null) return "error";
        return result.toUpperCase();
    });
  • 06
    thenCompose vs thenApply: thenApply wraps the result: if function returns CompletableFuture, you get CF<CF<T>>. thenCompose flattens it: returns CF<T>. Same relationship as map vs flatMap in streams. 
    // thenCompose — when second CF depends on first result
CompletableFuture.supplyAsync(() -> getUserId())
    .thenCompose(id -> CompletableFuture.supplyAsync(() -> getUser(id)));
// Returns CF<User> not CF<CF<User>>
