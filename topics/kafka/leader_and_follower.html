Distributed Systems

Leader & Follower

Mastering data replication, fault tolerance, In-Sync Replicas (ISR), and failover mechanics in distributed databases and message brokers like Kafka.

1. Definition

Simple English

Imagine a CEO (Leader) and their assistants (Followers). All important decisions (Write requests) must go through the CEO. The assistants immediately write down everything the CEO decides. If the CEO suddenly quits, the most up-to-date assistant is promoted to be the new CEO.

Technical

Also known as Primary-Replica or Master-Slave. It is a replication architecture where one node (the Leader) is designated to handle all write requests for a specific data partition. Other nodes (Followers) passively replicate the data from the leader to maintain exact copies.

Purpose

To provide High Availability (HA) and Fault Tolerance. If the machine hosting the data catches fire, the system automatically fails over to a follower with zero data loss. It also prevents concurrency conflicts since only one node accepts writes.

2. Internal Working: Replication Flow

The complexity of this pattern lies in how data is replicated. Systems must balance Durability (ensuring data isn't lost) with Latency (how fast the client gets a response).

3. Real-World Project Example

AWS Data Center Outage (Kafka)

How the Leader/Follower pattern prevents catastrophic data loss during hardware failures.

The Setup (Replication Factor = 3)

You have a Kafka Topic named financial-transactions. It is configured with a replication factor of 3. Broker A is the elected Leader. Broker B and Broker C are the Followers. All new transactions are written strictly to Broker A, which immediately syncs them to B and C.

The Crash

The AWS Availability Zone hosting Broker A loses power. Broker A dies instantly. Clients attempting to write transactions experience a sudden connection timeout. The system is momentarily headless.

The Failover

The cluster's Controller node notices Broker A stopped sending heartbeats. It looks at the In-Sync Replicas (ISR) list and sees Broker B has an exact, up-to-date copy of the data. The Controller instantly promotes Broker B to be the new Leader. Producer clients automatically fetch the new metadata, discover Broker B is the new boss, and resume writing. Data loss: Zero. Downtime: Milliseconds.

4. 7 Key Concepts For senior-level depth

1

Synchronous vs Asynchronous

Sync: Leader waits for followers to confirm the write before replying to the client (High durability, high latency). Async: Leader writes locally, replies to client, and followers copy later (Low latency, high risk of data loss on crash).

2

ISR (In-Sync Replicas)

In Kafka, a follower is only considered "In-Sync" if it has fetched the latest data from the Leader within a specific time window. Only nodes in the ISR list are eligible to become the new Leader during a failover.

3

High Watermark

The offset (position) of the last message that has been successfully copied to all members of the ISR. Clients (Consumers) are only allowed to read messages up to the High Watermark to ensure they never read uncommitted data.

4

Split-Brain Problem

A catastrophic scenario where a network partition causes the cluster to split in half, and both halves elect their own Leader for the same data. Resolving this usually requires a Quorum (majority vote) or Fencing tokens.

5

Read Replicas (Databases)

In relational databases (like PostgreSQL), you often route all INSERT/UPDATE queries to the Master/Leader, but route heavy SELECT queries to the Slaves/Followers to distribute CPU load (CQRS pattern).

6

Unclean Leader Election

If the Leader dies and NO followers are in the ISR (they were lagging), the system halts. If you enable "Unclean Election," the system picks a lagging follower as the new leader, restoring Availability but guaranteeing Data Loss.

7

CAP Theorem Trade-offs

The Leader/Follower pattern is fundamentally a manifestation of the CAP theorem. Configuring it requires choosing between Consistency (waiting for all followers to sync before acking) vs Availability (acking immediately after the leader writes, risking data loss on failure).

6. Configuration: Kafka Durability

Broker & Producer Configs
Copied! 
# ========================================================
# 1. BROKER CONFIGURATION (server.properties)
# ========================================================

# How many total copies of the data exist (1 Leader + 2 Followers)
default.replication.factor=3

# The minimum number of replicas that must acknowledge a write 
# before it is considered successful (if acks=all is used by producer).
# Setting this to 2 ensures data is written to at least the Leader and 1 Follower.
min.insync.replicas=2

# If the Leader dies and NO followers are in the ISR (they are out of sync),
# should we pick an out-of-sync follower as the new leader?
# false = Cluster halts (Favors Consistency, prevents data loss)
# true = Failover happens (Favors Availability, causes data loss)
unclean.leader.election.enable=false


# ========================================================
# 2. PRODUCER CONFIGURATION (application.yml)
# ========================================================
spring:
  kafka:
    producer:
      # Defines how the Producer interacts with the Leader & Followers
      # acks=0   -> Fire and forget (Fastest, High Data Loss Risk)
      # acks=1   -> Leader writes to disk and acks (Medium Risk)
      # acks=all -> Leader waits for ALL ISR followers to write to disk (Zero Data Loss)
      acks: all

7. Senior Developer Answer

"The Leader/Follower pattern is the architectural bedrock of high availability in distributed stateful systems. In Kafka, topics are divided into partitions, and every partition has a designated Leader broker. All read and write requests are routed to this Leader to guarantee strict ordering and consistency.

To ensure fault tolerance, the Leader passively replicates its data to Follower brokers. A critical mechanism here is the ISR (In-Sync Replicas) list. The Leader monitors which followers are fully caught up with the data log. When a producer writes data with acks=all, the Leader will not acknowledge the write until every node in the ISR has safely written the data to disk.

If the Leader node experiences a hardware failure, the cluster Controller immediately steps in, selects a healthy follower from the ISR, and promotes it to be the new Leader. Because we enforce min.insync.replicas=2 and disable unclean leader election in our enterprise environments, we guarantee zero data loss during these failovers, favoring Consistency over absolute Availability in our CAP theorem trade-offs."

8. Quick Revision Notes

Leader: The primary node. Handles 100% of standard Write and Read requests.
Follower: Passive node. Silently copies data from the Leader to stay up-to-date.
ISR (In-Sync): Followers that are fully caught up with the Leader.
High Watermark: Consumers can only read messages that have been replicated to ALL ISRs.
acks=all: Producer waits for Leader AND all ISRs to save data before continuing.
Unclean Election: Picking an out-of-sync follower as leader. Causes data loss.
Split-Brain: Network issue causing two leaders. Solved by Quorum consensus.
Read Replicas: In DBs, followers handle heavy SELECT queries to offload the Leader.
