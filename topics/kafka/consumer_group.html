Consumer Group

Mastering Partition Assignment, Rebalancing, Offsets, the Group Coordinator, and Static Membership.

1. Definition

Simple English

Imagine a massive pile of letters (a Topic) that needs to be read. One person (a Consumer) can't read them all fast enough. So, you hire a team of readers (a Consumer Group). Kafka ensures that no two people in the team read the exact same letter.

Technical

A Consumer Group is a set of consumers that share the same `group.id`. Kafka treats them as a single logical subscriber. The Kafka cluster dynamically distributes the partitions of a topic evenly among the active consumers in the group.

Purpose

To provide massively scalable, parallel processing of data streams. It guarantees that a message is processed exactly once by the application (group), while allowing the application to scale horizontally by just spinning up more instances.

2. The Golden Rule of Partition Assignment

The most critical rule in Kafka is: A single partition can only be consumed by ONE consumer within the same Consumer Group at a time. This is how Kafka guarantees strict ordering per partition.

3. Real-World Project Example

Chewing Through Lag

How Consumer Groups handle sudden massive traffic spikes (e.g., Ticket Sales).

The Setup & The Spike

You have a topic ticket-orders with 30 partitions. Normally, you run 3 pods of your OrderProcessor microservice (group id: order-app). Each pod processes 10 partitions. Suddenly, Taylor Swift tickets go on sale. Producers dump 500,000 messages into Kafka in one minute. Your 3 pods can only process 10,000 per minute. You have massive Consumer Lag.

Horizontal Scaling (Auto-Scaling)

Your Kubernetes HPA detects the CPU spike and spins up 27 new pods, bringing the total to 30 pods. All 30 pods have the same group.id="order-app".

The Rebalance

Kafka's Group Coordinator notices the new consumers joining. It triggers a Rebalance. It pauses processing momentarily and re-assigns the partitions. Now, each of the 30 pods gets exactly 1 partition. The processing throughput increases 10x instantly, and the lag drops to zero in minutes.

4. 7 Key Concepts For senior-level depth

1

The `group.id`

The unique string identifying the consumer group. If two applications have different group IDs, they act as independent subscribers (Pub/Sub pattern). Both groups will get a copy of every message in the topic.

2

The Group Coordinator

One specific Kafka Broker is designated as the Coordinator for a group. It receives heartbeats from the consumers. If a heartbeat fails, the Coordinator kicks the dead consumer out and triggers a rebalance.

3

Offset Commits

Consumers must periodically tell Kafka "I have successfully processed up to message #42". This is an Offset Commit. It is actually a write to a hidden internal Kafka topic named __consumer_offsets.

4

Rebalancing

The process of reassigning partitions among consumers. It happens when a consumer joins the group, leaves the group (crashes), or when new partitions are added to the topic. During a standard rebalance, all consumption stops ("Stop the World").

5

Consumer Lag

The delta between the last message produced to a partition and the last message committed by the consumer group. High lag means your consumers are too slow. (Calculated as LogEndOffset - CurrentOffset).

6

Auto.Offset.Reset

If a new consumer group starts, it has no committed offsets. This config tells Kafka where to start reading. earliest reads from the absolute beginning of the topic. latest reads only new messages arriving from now on.

7

Static Membership (Preventing Rebalance Storms)

In Kubernetes, pods deploy and restart frequently. By default, a restart triggers a global rebalance. By assigning a group.instance.id to a consumer, it becomes a "Static Member". If it restarts and reconnects within the session timeout, Kafka recognizes it and gives its old partitions back without triggering a rebalance storm.

import org.springframework.kafka.support.Acknowledgment;
import org.springframework.stereotype.Service;
import org.apache.kafka.clients.consumer.ConsumerRecord;

/* Requirements in application.yml:
   spring.kafka.consumer.group-id: inventory-service-group
   spring.kafka.listener.ack-mode: manual_immediate
   spring.kafka.consumer.enable-auto-commit: false
*/

@Service
public class OrderConsumer {

    // 1. Defines the Topic and Group ID.
    // 2. concurrency = "3" tells Spring to spin up 3 independent consumer 
    //    threads inside THIS single JVM instance, all belonging to the same group.
    @KafkaListener(
        topics = "orders-topic", 
        groupId = "inventory-service-group",
        concurrency = "3"
    )
    public void consumeOrder(
            ConsumerRecord<String, String> record, 
            Acknowledgment ack) {
        
        try {
            String orderId = record.key();
            String payload = record.value();
            int partition = record.partition();
            
            System.out.printf("Thread %s processing Order %s from Partition %d%n", 
                Thread.currentThread().getName(), orderId, partition);
            
            // Execute DB logic (e.g. deduct inventory)
            processInventory(payload);
            
            // Manual Commit: Tells the Group Coordinator we are successfully done.
            // This prevents data loss if the app crashes mid-processing.
            ack.acknowledge();
            
        } catch (Exception e) {
            // Handle errors (e.g., send to Dead Letter Queue)
            // Do NOT call ack.acknowledge() if you want Kafka to retry sending this later.
            handleFailure(record, e);
        }
    }

7. Senior Developer Answer

"A Consumer Group is Kafka's mechanism for achieving massive horizontal scalability while maintaining strict message ordering. The golden rule is that a single partition can only be assigned to one consumer within a group at any given time. If we have a topic with 50 partitions, we can scale our consumer application up to exactly 50 instances. Any instances beyond that will sit idle as hot-standbys.

In production, managing the Group Coordinator's rebalance process is critical. When a consumer joins, leaves, or crashes, a rebalance is triggered, temporarily pausing message processing. In environments like Kubernetes where pods restart frequently, this can cause a 'rebalance storm.' We mitigate this by utilizing Static Membership via the group.instance.id property, which allows a restarting pod to reclaim its specific partitions without triggering a group-wide rebalance.

Finally, we never use auto-commits. We utilize manual offset acknowledgment to guarantee 'At-Least-Once' delivery semantics, pairing it with idempotent database operations to ensure that duplicate message deliveries during network retries do not corrupt our system state."

8. Quick Revision Notes

Group ID: Apps with the same group.id share the workload of a topic.
The Golden Rule: 1 Partition = Max 1 Consumer (within the same group).
Idle Consumers: If Consumers > Partitions, the extra consumers sit idle.
Rebalance: Triggered when consumers join/leave. Partitions are reassigned.
Offsets: The read-position. Saved in the internal __consumer_offsets topic.
Consumer Lag: Unprocessed messages. (LogEndOffset - CurrentOffset).
Static Membership: Assigning an Instance ID to prevent rebalances during pod restarts.
Pub/Sub: To have multiple microservices read the same topic, give them DIFFERENT group IDs.
