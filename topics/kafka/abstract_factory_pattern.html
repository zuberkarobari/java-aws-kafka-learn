Creational Design Pattern

Abstract Factory

Mastering the "Factory of Factories": Ensuring product compatibility, enforcing interfaces, and understanding the Open/Closed Principle.

1. Definition

Simple English

Think of buying furniture. You pick a theme, like "Modern" or "Victorian". The Abstract Factory is the catalog: if you pick the Modern catalog, you are guaranteed that the Chair, Sofa, and Table you order will all match each other perfectly.

Technical

A creational design pattern that provides an interface for creating families of related or dependent objects without specifying their concrete classes. It encapsulates a group of individual factories that share a common theme.

Purpose

To enforce compatibility among products. It ensures that a client system always receives a suite of objects designed to work together, isolating the client entirely from the concrete implementations.

2. Internal Working: Family Creation

The pattern is structured around a primary abstract interface (`AbstractFactory`) which declares creation methods for each abstract product type. Concrete factories implement these methods to produce specific product variants.

3. Real-World Project Example

Cross-Platform UI Toolkit

How Java's AWT/Swing or modern UI frameworks handle operating systems.

The Problem: Mixing Styles

If you are building an application that runs on both Windows and macOS, rendering a Windows-style button inside a Mac-style window looks broken. You need to ensure that if the app runs on a Mac, every single UI component generated belongs to the Mac family.

The Abstract Factory Solution

At application startup, the system detects the OS. If it's macOS, it instantiates a MacFactory and passes it to the UI rendering engine. The rendering engine only knows it has a generic GUIFactory. When it calls factory.createButton(), it is guaranteed to receive a perfectly matched Mac button, completely oblivious to the underlying concrete class.

4. 7 Key Concepts For senior-level depth

1

Families of Products

The defining characteristic of Abstract Factory is that it groups related objects together. It doesn't just create one product; it creates a suite of products designed to collaborate.

2

Interface-Based Design

The client code must be strictly programmed to interfaces. It should never use the `new` keyword to instantiate the concrete products or the factories directly.

3

Open/Closed Principle (Pros)

You can introduce new variants (e.g., adding a `LinuxFactory`) without breaking any existing client code, fully respecting the OCP.

4

The "Interface Bloat" (Cons)

Adding a entirely new product type (e.g., adding `createScrollbar()`) requires changing the Abstract Factory interface and updating every single concrete factory implementation.

5

Factory Method Integration

Abstract Factory is often implemented using Factory Methods internally. Each method inside the Abstract Factory (like `createButton`) is technically a Factory Method.

6

Dependency Injection

In modern frameworks like Spring, Abstract Factories are often injected as dependencies into the classes that need to create the object families dynamically at runtime.

7

Singleton Factories

Because concrete factories rarely carry state themselves (they only exist to produce objects), they are almost always implemented as Singletons to save memory.

5. Follow-Up Questions

Answer: The Factory Method pattern is used to create a single product (it delegates instantiation to subclasses). The Abstract Factory pattern is used to create families of related or dependent products. An Abstract Factory is essentially an object that contains multiple Factory Methods.

Answer: Use it when your code needs to work with various families of related products, but you don't want it to depend on the concrete classes of those products (because they might be unknown beforehand, or you want to allow for future extensibility). If you only have one product hierarchy, it is over-engineering.

Answer: Spring's `BeanFactory` is an excellent example of an Abstract Factory. It provides the interface to produce various beans (products) while hiding the concrete instantiation logic from the developer.

6. Code Example

AbstractFactoryDemo.java
Copied! 
/* --- 1. Abstract Products --- */
interface Button { void paint(); }
interface Checkbox { void paint(); }

/* --- 2. Concrete Products (Family A: Mac) --- */
class MacButton implements Button {
    public void paint() { System.out.println("Render Mac Button"); }
}
class MacCheckbox implements Checkbox {
    public void paint() { System.out.println("Render Mac Checkbox"); }
}

/* --- 3. Concrete Products (Family B: Win) --- */
class WinButton implements Button {
    public void paint() { System.out.println("Render Win Button"); }
}
class WinCheckbox implements Checkbox {
    public void paint() { System.out.println("Render Win Checkbox"); }
}

/* --- 4. Abstract Factory Interface --- */
interface GUIFactory {
    Button createButton();
    Checkbox createCheckbox();
}

/* --- 5. Concrete Factories --- */
class MacFactory implements GUIFactory {
    public Button createButton() { return new MacButton(); }
    public Checkbox createCheckbox() { return new MacCheckbox(); }
}
class WinFactory implements GUIFactory {
    public Button createButton() { return new WinButton(); }
    public Checkbox createCheckbox() { return new WinCheckbox(); }
}

/* --- 6. Client Code --- */
class Application {
    private Button button;
    private Checkbox checkbox;

    // The client only knows about abstract interfaces
    public Application(GUIFactory factory) {
        button = factory.createButton();
        checkbox = factory.createCheckbox();
    }

    public void render() {
        button.paint();
        checkbox.paint();
    }
}

7. Senior Developer Answer

"The Abstract Factory pattern is crucial when we need to enforce the creation of mutually compatible families of objects while strictly adhering to the Dependency Inversion Principle. Instead of scattering conditional logic throughout our code to instantiate concrete classes, we abstract the creation process into a specialized factory layer.

This pattern perfectly supports the Open/Closed Principle. If we need to support a new platform or theme, we simply add a new concrete factory implementation without modifying a single line of the core application logic.

However, as a senior developer, I'm highly aware of its primary drawback: interface bloat. If the business requirement changes and we suddenly need to introduce a new product type—say, a `Dropdown`—we are forced to modify the core `AbstractFactory` interface and fundamentally break every concrete factory implementation. Because of this rigidity, I only implement an Abstract Factory when the product families are well-established and highly unlikely to change structurally."

8. Quick Revision Notes

Core Need: Creating families of related, compatible objects.
Structure: One interface (AbstractFactory) defining multiple creation methods.
Benefit (OCP): Easy to add new families (e.g., adding a LinuxFactory).
Drawback: Hard to add new product types (requires changing interface and all classes).
Encapsulation: Hides concrete class names entirely from the client.
Vs Factory Method: Factory Method = 1 product. Abstract Factory = family of products.
Spring Usage: The `BeanFactory` is a high-level implementation of this concept.
Combination: Concrete factories are often implemented as Singletons to save memory.
