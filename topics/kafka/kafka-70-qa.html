🟣 Core Concepts
01
What is Apache Kafka?
Core ConceptMust Know
▾
A distributed streaming platform — understand what makes it different from a traditional message queue.
  • ✦
    Definition: Apache Kafka is a distributed streaming platform that allows for publishing, subscribing to, storing, and processing streams of records in real-time.
  • ✦
    Key distinction: Unlike traditional message queues (RabbitMQ), Kafka acts as an append-only distributed commit log — messages persist after consumption.
  • ✦
    Designed for: High-throughput, fault-tolerant, and scalable data pipelines and real-time streaming applications.
  • ✦
    Use cases: Real-time analytics, event sourcing, microservice decoupling, log aggregation, CDC (Change Data Capture).
💡
Interview tip: Mention the "dumb broker, smart consumer" philosophy — Kafka doesn't track who read what, the consumer manages its own offset.

Six core building blocks — know all of them cold before any Kafka interview.
Producer

Consumer

03
What is a Topic in Kafka?
Core Concept
▾
Think of a Topic like a database table — a named channel where records flow.
  • ✦
    Definition: A Topic is a category or feed name to which records are published. It's a logical name for a stream of records.
  • ✦
    Multi-subscriber: A topic can have zero, one, or many consumers — all can read independently without affecting each other.
  • ✦
    Partitioned: Topics are split into partitions for improved scalability and parallel processing.
  • ✦
    Retention-based: Unlike a queue, messages stay in the topic until the retention policy (time or size) expires.
04
What is a Partition in Kafka?
Core ConceptMust Know
▾
The fundamental unit of parallelism in Kafka — understanding partitions unlocks understanding of everything else.
  • ✦
    Definition: An ordered, immutable sequence of records that is continually appended to — a structured commit log.
  • ✦
    Offset: Each record gets a sequential integer ID called an offset, used by consumers to track position.
  • ✦
    Parallelism: Multiple partitions → multiple consumers can read simultaneously → horizontal scaling.
  • ✦
    Ordering guarantee: Order is guaranteed within a partition only. Use a message key to route related messages to the same partition.
05
What is the role of ZooKeeper in Kafka?
Core ConceptArchitecture
▾
ZooKeeper was Kafka's external coordinator — now being replaced by KRaft (Kafka Raft) in modern versions.
  • ✦
    Responsibilities: Manages broker coordination, maintains configuration, naming, distributed synchronization.
  • ✦
    Cluster state: Tracks which brokers are alive, which topic-partitions exist, and who the leaders are.
  • ✦
    Leader election: Orchestrates partition leader elections when a broker fails.
  • ✦
    KRaft mode (Kafka 3.3+): Kafka now manages its own metadata internally using the Raft consensus protocol — ZooKeeper is deprecated.
06
What is a Broker in Kafka?
Core ConceptArchitecture
▾
The "dumb" server at Kafka's heart — it just stores and serves, consumers do the smart work.
  • ✦
    Definition: A Kafka server running in a Kafka cluster. Multiple brokers form the cluster.
  • ✦
    Receives: Takes messages from producers, assigns offsets, commits to disk.
  • ✦
    Serves: Responds to consumer fetch requests with messages from its stored partitions.
  • ✦
    Leader/Follower: Each partition has one broker as Leader (serves reads/writes) and others as Followers (replicate data).
07
How does Kafka ensure fault tolerance?
Core ConceptArchitecture
▾
Kafka's fault tolerance is built on replication — every partition has multiple copies across brokers.
  • ✦
    Replication Factor: Each partition is replicated across N brokers (configurable). Typically 3 in production.
  • ✦
    Leader/Followers: One broker is the Leader handling all R/W. Others are Followers passively replicating.
  • ✦
    Auto leader election: If the Leader fails, a new Leader is elected from the ISR (In-Sync Replicas) automatically.
  • ✦
    ISR: Only replicas fully caught-up with the leader can participate in elections — no stale data promoted.
08
What is the difference between a Kafka Consumer and a Consumer Group?
Core ConceptConsumers
▾
Individual vs. team — one reads, many share the work.
Consumer
A single application instance reading from one or more topic partitions.
Consumer Group
A set of consumers sharing the same group.id, cooperating to consume a topic. Each partition → exactly one consumer in the group.
  • ✦
    Load balancing: Partitions are divided among consumers in the group — enables parallel processing.
  • ✦
    Rule: Max active consumers in a group = number of partitions. Extra consumers sit idle.
  • ✦
    Pub-sub: Multiple groups each get ALL messages — each group is independent.
09
What is the purpose of the Offset in Kafka?
Core ConceptConsumers
▾
The bookmark — consumers use offsets to remember where they stopped reading.
  • ✦
    Definition: A unique sequential integer assigned to each record within a partition. Think of it as a line number in the log file.
  • ✦
    Consumer responsibility: Unlike RabbitMQ, consumers track their own offset — Kafka doesn't track who read what.
  • ✦
    Storage: Consumer offsets are stored in the internal topic __consumer_offsets.
  • ✦
    Replay: Consumers can reset their offset to re-read messages — a powerful feature for debugging or reprocessing.
10
How does Kafka handle message delivery semantics?
Core ConceptMust Know
▾
Three guarantees — know the trade-offs for each. This is asked in almost every senior interview.
At Most Once
Messages may be lost, never redelivered. Commit offset before processing. Fastest.
At Least Once
Messages never lost, may be duplicated. Commit offset after processing. Default.
Exactly Once
Each message processed exactly once. Requires idempotent producers + Transactions API.
💡
Production choice: Most systems use "At Least Once" with idempotent consumers (deduplication logic) as a practical middle ground.
🟠 Producers
11
What is the role of the Kafka Producer API?
Producers
▾
The Producer API is the write path — it handles partitioning, compression, batching, and retries.
  • ✦
    Publishes: Sends streams of records to Kafka topics.
  • ✦
    Partitioning: Decides which partition each record goes to (by key hash or round-robin).
  • ✦
    Compression: Compresses message batches (gzip, snappy, lz4, zstd).
  • ✦
    Retries: Automatically retries failed sends based on retries and retry.backoff.ms config.
  • ✦
    Delivery guarantees: Configurable via the acks parameter (0, 1, or all).
15
What is the significance of the acks parameter in Kafka producers?
ProducersMust Know
▾
The most important producer config — controls the durability vs. latency trade-off.
acks=0
Fire and forget. No acknowledgment. Lowest latency, highest risk of data loss.
acks=1
Leader writes locally and acks. Risk: leader crashes before followers replicate.
acks=all
All ISRs acknowledge. Strongest durability guarantee. Highest latency.
💡
Production standard: Use acks=all with min.insync.replicas=2 for banks/fintech. For analytics pipelines acks=1 is acceptable.
20
How does Kafka handle message compression?
Producers
▾
Compression is configured at the producer level and applies to whole message batches — not individual records.
gzip
Best compression ratio, higher CPU
snappy
Good balance of speed and compression
lz4
Fastest compression/decompression
zstd
Best ratio with reasonable speed (Kafka 2.1+)
  • ✦
    Compression reduces network transfer size and disk storage. The broker can store messages in their compressed form.
  • ✦
    Consumer decompresses transparently — no application code changes needed.
25
What is the purpose of the Idempotent Producer in Kafka?
ProducersSenior Level
▾
Idempotent producers are the key to exactly-once delivery — they eliminate duplicates from network retries.
  • ✦
    Problem it solves: Without idempotency, a network timeout causes the producer to retry → broker writes the same message twice.
  • ✦
    Mechanism: Each producer gets a unique Producer ID (PID). Each message gets a Sequence Number per partition.
  • ✦
    Broker deduplication: If the broker sees a sequence number it already committed, it discards the duplicate silently.
  • ✦
    Enable it: Set enable.idempotence=true on the producer config. This also forces acks=all.
27
What is the difference between a round-robin partitioner and a key-based partitioner?
Producers
▾
How a producer decides which partition to send a message to — crucial for ordering guarantees.
Round-Robin
No key provided → messages distributed evenly across all partitions cyclically. No ordering guarantee.
Key-Based (murmur2 hash)
Key provided → hash(key) % numPartitions → same key always routes to same partition. Guarantees ordering per key.
💡
Senior answer: For user event ordering (e.g., banking transactions), always use userId as the key to ensure all events for a user land on the same partition and are consumed sequentially.
34
How does Kafka handle message batching?
Producers
▾
Batching is how Kafka achieves massive throughput — group many messages into one network request.
  • ✦
    How it works: The producer buffers messages destined for the same partition and sends them as a single batch.
  • ✦
    batch.size: Maximum bytes in a batch (default 16KB). Larger batches → better throughput, more memory.
  • ✦
    linger.ms: Time to wait for more messages before sending. Default 0ms (send immediately). Increasing this fills larger batches.
  • ✦
    Benefit: Reduces network overhead and improves I/O efficiency dramatically — key to Kafka's speed.
38
How does Kafka handle message key hashing?
Producers
▾
The murmur2 hash determines partition routing — consistent and deterministic.
  • ✦
    Algorithm: Kafka uses the murmur2 hash algorithm by default: partition = Math.abs(murmur2(key)) % numPartitions
  • ✦
    Consistency: Same key → always same partition (as long as partition count doesn't change).
  • ✦
    Why it matters: Critical for ordered processing of related events and local state in stream processing.
  • ✦
    Warning: Adding partitions to a topic breaks key-to-partition mapping — plan your partition count upfront.
41
What is the purpose of the Kafka Quota API?
ProducersOperations
▾
Rate limiting for producers and consumers — prevents a single client from starving the cluster.
  • ✦
    Purpose: Enforce quotas on produce and fetch request rates, preventing any single client from consuming too many broker resources.
  • ✦
    Granularity: Defined per-client-id or per-user. Can limit bytes/sec for produce or consume.
  • ✦
    Enforcement: When a client exceeds its quota, the broker throttles it by delaying responses (not dropping messages).
42
How does Kafka handle producer message acknowledgments (deep dive)?
ProducersSenior Level
▾
Deep dive into acks — understanding the failure modes of each setting is a senior-level question.
  • A0
    acks=0 — Producer doesn't wait at all. Lowest latency. Message may be silently lost if broker goes down. Good for: metrics, logs where loss is acceptable.
  • A1
    acks=1 — Leader writes and acks. Risk: leader crashes before followers replicate → data loss despite ack. Good for: throughput-sensitive use cases.
  • AA
    acks=all — All in-sync replicas acknowledge. No data loss if any ISR survives. Highest latency. Good for: financial transactions, critical events.
⚠️
acks=all alone is not enough — also set min.insync.replicas=2 to prevent a single-replica partition from satisfying "all" with just the leader.
53
What is the purpose of the Kafka producer's Partitioner interface?
Producers
▾
A pluggable interface for custom partition routing logic.
  • ✦
    Default behavior: Hash(key) % numPartitions for keyed records; sticky batching for unkeyed records.
  • ✦
    Custom partitioner: Implement org.apache.kafka.clients.producer.Partitioner to control distribution based on business logic.
  • ✦
    Example: Route "premium" customers to low-latency partitions on dedicated brokers.

How does Kafka handle message delivery timeouts?
Producers
▾
Producers have a total delivery timeout — after which the send is considered failed.
  • ✦
    delivery.timeout.ms — Total time the producer will retry sending before throwing an exception. Encompasses retries and backoffs.
  • ✦
    request.timeout.ms — How long the producer waits for a response per request attempt.
  • ✦
    max.block.ms — How long send() blocks when the buffer is full or metadata is unavailable.
63
What is the purpose of the Kafka producer's max.block.ms parameter?
Producers
▾
Prevents indefinite blocking — sets an upper bound on how long your app will wait.
  • ✦
    Controls: How long send() will block when the producer's send buffer is full, or when requesting metadata.
  • ✦
    On expiry: Throws a TimeoutException — allows the application to fail fast rather than hang.
  • ✦
    Default: 60,000ms (60 seconds). Set lower in latency-sensitive applications.
67
What is the purpose of the Kafka producer's linger.ms parameter?
Producers
▾
The throughput knob — wait a little longer to fill bigger batches.
  • ✦
    linger.ms=0 (default): Send messages immediately. Lowest latency, but smaller batches.
  • ✦
    linger.ms=5: Wait up to 5ms for more messages before sending a batch. More throughput, slight latency increase.
  • ✦
    Trade-off: Higher linger.ms → larger batches → better compression ratio → higher throughput → but higher latency.
  • ✦
    Production tip: Set linger.ms=5-20 for throughput-critical producers like log aggregation pipelines.

How does Kafka support scalability?
ArchitectureSenior Level
▾
Four mechanisms that make Kafka horizontally scalable to millions of events per second.
  • 1
    Partitioning: Topics split across multiple brokers. More partitions = more parallelism.
  • 2
    Sequential I/O: Kafka appends to disk sequentially (like a log file), which is faster than random disk I/O even vs. RAM in some cases.
  • 3
    Zero-Copy: OS-level optimization — data goes directly from disk page cache to network socket, bypassing the JVM heap entirely.
  • 4
    Message Batching: Multiple records bundled per request, reducing network round trips.
13
What is Log Compaction in Kafka?
Architecture
▾
A retention strategy where Kafka keeps only the latest value per key — turning Kafka into a key-value store.
  • ✦
    What it does: Selectively removes records where a more recent update exists with the same key. Guarantees at least the last state for each key.
  • ✦
    vs Time-based retention: Normal topics delete all messages after N days. Compacted topics delete only obsolete key versions.
  • ✦
    Use case: Database changelogs (CDC), user profile updates, config stores — where you only care about the latest state.
  • ✦
    Tombstone: To delete a key entirely, produce a record with a null value (tombstone) — the log cleaner will eventually remove it.
14
How does Kafka handle message ordering?
ArchitectureMust Know
▾
Ordering is per-partition — not global across a topic. This is a common interview gotcha.
  • ✦
    Guaranteed within a partition: Messages sent to a partition are appended in order and consumed in order.
  • ✦
    NOT guaranteed across partitions: If a topic has 3 partitions, global ordering is not maintained.
  • ✦
    Design pattern: Use a consistent message key (e.g., orderId) to route related events to the same partition → per-key ordering guaranteed.
  • ✦
    Caveat: Retries can reorder messages. Use enable.idempotence=true + max.in.flight.requests.per.connection=5 to preserve order with retries.
16
How does Kafka handle data retention?
Architecture
▾
Two retention strategies — time-based or size-based, configured per topic.
Time-based
retention.ms — Delete segments older than N ms. Default: 7 days.
Size-based
retention.bytes — Delete oldest segments when partition exceeds N bytes.
Log Compaction
Keep only the latest value per key — for changelog use cases.
  • ✦
    Deletion is segment-level (whole files), not individual messages — this is why minimum retention granularity = 1 log segment.
18
How does Kafka ensure high availability?
ArchitectureSenior Level
▾
Five pillars of Kafka's high availability — know all of them.
  • 1
    Partition Replication: Each partition copied across N brokers (typically 3).
  • 2
    Auto Leader Election: If a leader broker fails, a new leader is elected from ISR automatically.
  • 3
    Zero-downtime scaling: Add brokers to a cluster and rebalance partitions without stopping Kafka.
  • 4
    min.insync.replicas: Configures minimum ISR count required for a write to succeed — prevents data loss.
  • 5
    KRaft / ZooKeeper: Distributed coordination service manages cluster membership and metadata.
22
How does Kafka handle message size limits?
Architecture
▾
Kafka defaults to 1MB max per message — but this is configurable across multiple layers.
message.max.bytes (broker)
Maximum size the broker will accept. Default 1MB.
max.request.size (producer)
Max size of a produce request. Must align with broker config.
max.partition.fetch.bytes (consumer)
Max bytes fetched per partition per poll.
⚠️
Large messages hurt performance. Best practice: store large payloads in S3/blob storage and put only the reference URL in Kafka.
24
How does Kafka handle data replication?
ArchitectureMust Know
▾
Replication is Kafka's core durability mechanism — understand Leader/Follower/ISR deeply.
  • ✦
    Leader: One broker is designated leader for each partition. All reads and writes go through the leader.
  • ✦
    Followers: Other brokers replicate the leader's log passively. They continuously fetch from the leader.
  • ✦
    ISR (In-Sync Replicas): Set of replicas caught up with the leader within replica.lag.time.max.ms. Only ISR members can become the new leader.
  • ✦
    Replication Factor: Configurable per topic. RF=3 means 1 leader + 2 followers. Can survive 1 broker failure with no data loss.
28
How does Kafka handle message deletion?
Architecture
▾
Kafka never deletes individual messages — it removes whole log segments based on policy.
  • ✦
    Segment-based deletion: The log is divided into segments. Kafka deletes entire segments, never individual records.
  • ✦
    Triggered by: Time (retention.ms) or size (retention.bytes) policies.
  • ✦
    Log compaction alternative: Keeps only the latest value per key, deleting only stale versions.
  • ✦
    Active segment: The current open segment is never deleted — only closed (rolled) segments are eligible.
31
What is the role of the Controller in a Kafka cluster?
ArchitectureSenior Level
▾
One elected broker takes on controller responsibilities — it's the cluster manager.
  • ✦
    Definition: One broker is elected as the Controller — it has extra responsibilities beyond normal broker operations.
  • ✦
    Responsibilities: Elects partition leaders, manages partition distribution, handles topic creation/deletion admin ops.
  • ✦
    Single controller: Only one controller at a time. If it fails, ZooKeeper (or KRaft) elects a new one.
  • ✦
    KRaft: In KRaft mode, the controller runs as a separate process/quorum — more scalable than the ZooKeeper-based approach.
32
How does Kafka ensure data consistency?
ArchitectureSenior Level
▾
Five interlocking mechanisms that together guarantee consistency.
  • 1
    Replication: Each partition copied across multiple brokers.
  • 2
    ISR: Only up-to-date replicas can become leaders — no stale data promoted.
  • 3
    Acknowledgments: acks=all ensures all ISR have the data before confirming to producer.
  • 4
    Atomic partition writes: Writes to a partition are ordered and atomic.
  • 5
    Idempotent producers: Prevent duplicates from network retries.
36
How does Kafka handle message ordering within a partition?
Architecture
▾
Within a partition, order is absolute and immutable — this is the append-only log guarantee.
  • ✦
    Messages are appended to the partition log in the exact order they are received by the leader.
  • ✦
    The log is immutable — once written, records cannot be modified or reordered.
  • ✦
    Consumers always read in the written order — no random access, only sequential reads from an offset.

How does Kafka handle message timestamps?
Architecture
▾
Two types of timestamps — producer time vs broker arrival time. Configurable per topic.
CreateTime
The time the producer created the message. Set by the producer client.
LogAppendTime
The time the broker received and appended the message. Set by the broker.
  • ✦
    Used for: time-based log retention, log compaction, and consumer time-based seeking (offsetsForTimes()).
  • ✦
    Configure via log.message.timestamp.type at the topic or broker level.
46
How does Kafka handle topic deletion?
ArchitectureOperations
▾
Topic deletion is asynchronous — Kafka marks it for deletion, then cleans up.
  • 1
    Topic is marked for deletion in ZooKeeper / KRaft metadata.
  • 2
    Kafka stops accepting produce/consume requests for that topic immediately.
  • 3
    Log segments on disk are asynchronously deleted in the background.
⚠️
Requires delete.topic.enable=true on the broker (default is true in modern Kafka). If false, topics are only marked but never deleted.
50
How does Kafka handle message ordering across multiple partitions?
Architecture
▾
Short answer: it doesn't. Cross-partition ordering is the developer's responsibility.
  • ✦
    No global ordering: Kafka only guarantees ordering within a single partition.
  • ✦
    Pattern 1 — Single partition: Use 1 partition for global ordering. Limits throughput to one consumer.
  • ✦
    Pattern 2 — Key-based routing: Use a partition key so related events go to the same partition. Ordering within a key is guaranteed.
  • ✦
    Pattern 3 — Application sequencing: Include a monotonic sequence number in the message and handle ordering in the consumer.

What is the role of the broker's log.segment.bytes configuration?
ArchitectureOperations
▾
Controls log file segment size — affects retention granularity and partition reassignment speed.
  • ✦
    Definition: Maximum size (bytes) of a single log segment file. Default: 1GB. When reached, a new segment is created.
  • ✦
    Retention granularity: Kafka can only delete entire segments — smaller segments = more precise retention.
  • ✦
    Reassignment speed: Smaller segments are faster to move during partition reassignment.
  • ✦
    Trade-off: Too small → too many open file handles and metadata overhead. Too large → retention is coarser.

How does Kafka handle message consumption patterns?
ArchitectureConsumers
▾
Kafka supports both Queue and Pub-Sub patterns through its consumer group model.
Queue Pattern
Multiple consumers in ONE group. Each message processed by exactly one consumer. Load balanced.
Pub-Sub Pattern
Each consumer in its OWN group. Every consumer gets every message. Broadcast.
💡
This is Kafka's superpower — the same topic can serve both patterns simultaneously by having different consumer groups for different purposes.
68
How does Kafka handle message delivery guarantees (end-to-end)?
ArchitectureMust Know
▾
End-to-end delivery guarantees require coordination between producer config, broker config, and consumer offset management.
  • ✦
    At Most Once: Producer: acks=0. Consumer: commit before processing. Fast but lossy.
  • ✦
    At Least Once: Producer: acks=1/all. Consumer: commit after processing. Default for most systems. May duplicate.
  • ✦
    Exactly Once (EOS): Producer: enable.idempotence=true. Use Kafka Transactions API. Consumer: transactional commits. No loss, no duplicates.
💡
In Spring Kafka, EOS is configured via spring.kafka.producer.transaction-id-prefix and spring.kafka.consumer.isolation-level=read_committed.

What is the role of the Group Coordinator in Kafka?
ConsumersArchitecture
▾
A designated broker that manages all consumer group membership and offset commits.
  • ✦
    What it is: A broker elected to manage a specific consumer group — handles membership and offset tracking.
  • ✦
    Membership: Tracks which consumers are alive (via heartbeats). Detects failures and triggers rebalances.
  • ✦
    Partition assignment: After a rebalance, coordinates which consumer reads which partition.
  • ✦
    Offset storage: Receives and stores consumer offset commits in __consumer_offsets.
26
How does Kafka handle consumer offsets?
ConsumersMust Know
▾
Offset management is at the heart of Kafka's consumer reliability — consumers own their position.
  • ✦
    Per group per partition: Each consumer group maintains its own offset per partition independently.
  • ✦
    Storage: Committed offsets stored in the internal __consumer_offsets topic (replicated, durable).
  • ✦
    Auto commit: enable.auto.commit=true commits the highest fetched offset periodically (every 5s by default). Risk: may commit before processing.
  • ✦
    Manual commit: Use commitSync() or commitAsync() after processing for at-least-once guarantees.
  • ✦
    Replay: Reset offset to re-consume messages — use auto.offset.reset=earliest or the kafka-consumer-groups CLI tool.
39
What is the role of the Kafka Consumer Coordinator?
Consumers
▾
The consumer-side coordinator handles group join, sync, and heartbeat protocols.
  • ✦
    Group join: When a consumer starts, it sends a JoinGroup request to the Group Coordinator broker.
  • ✦
    Leader consumer: One consumer in the group is elected as the "Group Leader" — it runs the partition assignment algorithm and reports results back to the coordinator.
  • ✦
    Sync: Coordinator distributes the assignment from the leader consumer to all group members.
  • ✦
    Heartbeat: Each consumer sends heartbeats to the coordinator to stay in the group.
47
What is the difference between a consumer's poll() and subscribe() methods?
Consumers
▾
Subscribe declares intent, poll does the actual work — you need both in every Kafka consumer.
subscribe(topics)
Registers the consumer to receive messages from a topic. No data fetched yet. Must be called before polling.
poll(duration)
Fetches records from subscribed topics. Drives all consumer mechanics — heartbeats, rebalances, offset commits. Called in a loop.
💡
poll() must be called regularly to maintain group membership. If you don't call it within max.poll.interval.ms, the consumer is considered dead and a rebalance triggers.
49
What is the purpose of the Kafka consumer heartbeat thread?
Consumers
▾
A background thread that keeps the consumer "alive" in the group — without it, rebalancing triggers.
  • ✦
    Purpose: Sends periodic heartbeat signals to the Group Coordinator broker to indicate the consumer is alive.
  • ✦
    Frequency: Configurable via heartbeat.interval.ms (default: 3000ms). Must be less than session.timeout.ms.
  • ✦
    On failure: If no heartbeat received within session.timeout.ms (default: 45s), consumer is declared dead → rebalance triggered.
  • ✦
    Separate thread: Runs independently of the main polling thread since Kafka 0.10.1, allowing longer processing times without missed heartbeats.
52
How does Kafka handle consumer lag?
ConsumersOperations
▾
Consumer lag is a critical production metric — high lag = consumer can't keep up with producers.
  • ✦
    Definition: Lag = (Latest producer offset) − (Last committed consumer offset). High lag means the consumer is falling behind.
  • ✦
    Monitoring: Use kafka-consumer-groups --describe CLI, or AdminClient API, or tools like Grafana + Prometheus Kafka Exporter.
  • ✦
    Remediation: Scale up consumers (add instances up to the partition count), optimize consumer processing logic, or increase partition count.
  • ✦
    Kafka doesn't auto-fix lag — it exposes the data; your infrastructure (auto-scaling, alerting) must respond.
57
What is the role of the consumer's position() method?
Consumers
▾
Returns the offset of the next record that will be fetched — your current read position.
  • ✦
    position(topicPartition) — Returns the offset of the next record to be fetched. Useful for tracking consumption progress.
  • ✦
    committed(topicPartition) — Returns the last committed offset. The difference between position() and committed() is the number of uncommitted (pending) records.
  • ✦
    Gap between position and committed = potential for reprocessing on consumer restart. Keep this gap small.

What is the role of commitSync() and commitAsync() methods?
ConsumersMust Know
▾
Manual offset commit — the right way to ensure exactly the right messages are committed at the right time.
commitSync()
Blocks until offset is committed. Retries on retriable errors. Reliable but slower throughput.
commitAsync()
Non-blocking. Does not retry on failure (to avoid out-of-order commits). Faster. Use a callback for error handling.
💡
Production pattern: Use commitAsync() in the polling loop for performance, and commitSync() in the finally block on shutdown for safety.

How does Kafka handle message consumption across consumer group rebalances?
ConsumersArchitecture
▾
Rebalances cause a consumption pause — understanding this is critical for high-availability systems.
  • 1
    All consumers stop consuming and commit their current offsets.
  • 2
    The Group Coordinator reassigns partitions to the remaining (or new) consumers.
  • 3
    Consumers resume consuming from their newly assigned partitions, starting from the last committed offset.
💡
Minimize rebalances: Use session.timeout.ms and max.poll.interval.ms wisely. Consider Cooperative Sticky Assignor (CooperativeStickyAssignor) in Kafka 2.4+ for incremental rebalances that don't stop all consumption.

What is the role of the consumer's auto.offset.reset configuration?
Consumers
▾
Defines what happens when a consumer has no committed offset — the cold start configuration.
earliest
Start from the very first message in the partition. Good for: new consumers that need to process all history.
latest
Start from new messages only (skip existing). Good for: consumers that only care about real-time events.
none
Throw an exception if no previous offset found. Strict mode — forces explicit offset management.

How does Kafka handle message retrieval for consumers?
ConsumersArchitecture
▾
Kafka uses a pull model — consumers request data, brokers don't push. This is a core design decision.
  • ✦
    Pull model: Consumers make fetch requests to brokers specifying topic, partition, and starting offset.
  • ✦
    Why pull? Consumers control their own rate — a slow consumer won't be overwhelmed. No back-pressure complexity needed.
  • ✦
    max.poll.records — Limits records returned per poll. fetch.min.bytes — Min data before broker responds. fetch.max.wait.ms — Max time broker waits to fill min bytes.
  • ✦
    Lag handling: If consumers fall behind, they simply catch up at their own pace — the log persists until retention expires.

How does Kafka handle message compression at the broker level?
OperationsArchitecture
▾
The broker can accept, decompress, recompress, or pass through messages based on its configuration.
Pass-through
Broker stores messages in original compressed form. Most efficient — no CPU overhead.
Decompress on receipt
Broker decompresses and stores uncompressed. Needed when broker needs to inspect individual records.
Recompress
Broker decompresses and recompresses with a different algorithm. Used when broker.compression.type differs from producer.

What is the role of the broker's log cleaner thread?
OperationsArchitecture
▾
A background thread that performs log compaction on eligible topics.
  • ✦
    What it does: Scans log segments of compaction-enabled topics and removes records where a newer value for the same key exists.
  • ✦
    When it runs: Periodically in the background. Frequency controlled by log.cleaner.backoff.ms.
  • ✦
    Dirty ratio: Compaction is triggered when the "dirty" (uncompacted) ratio of a partition exceeds min.cleanable.dirty.ratio.
  • ✦
    Active segment: The currently-being-written segment is never compacted — only closed segments are eligible.

How does Kafka handle message schema evolution?

  • ✦

    Broker stops accepting new produce requests.
  • 2
